Apart from boot volumes, some other volumes are also considered critical if there are services/roles installed on them. Anyway, to restore registry files you can do this:

  1. Mount the .vhd file as a virtual hard disk (see below). I will assume that you use drive letter V:.
  2. Copy all registry files from V:\Windows\System32\config to some temporary location.
  3. Keep tapping F8 at boot time to boot the machine into Repair Mode / Command Prompt.
  4. Create a backup of your existing registry files. The files could reside on a drive other than C: but it cannot be drive X:.
  5. Copy the restored registry files to where they belong.

And here is how you can mount a .VHD file:

  1. Use Windows Explorer to locate the .vhd (Virtual Hard Disk) file in the folder \WindowsImageBackup on your image disk.
  2. Launch the Disk Manager (diskmgmt.msc).
  3. Click Action, then Attach Virtual Hard Disk.
  4. Use the Browse button to locate the file you found in Step 1 above, then click OK.
  5. Note the new drive letter that Windows assigned. It could be G:. If no drive letter was assigned, right-click the new disk and give it a drive letter.
  6. Use Windows Explorer to open drive G: and retrieve your registry files.
  7. When finished, close Windows Explorer.
  8. Go back to the Disk Manager.
  9. Right-click the new disk on the left side of the panel where it says "Basic" and "Online".
  10. Click "Detach VHD".

tony mcp say:

Well, since Microsoft won't answer the question, I will.

I researched this a lot.  A good starting point for how to do this is the "Associate a file Type or protocol with a specific app using GPO" blog post.  This approach is documented on several other sites, and I confirmed it works on Windows 10 Pro.

Summary
Starting with Windows 8, the open with GPO no longer works.  You must follow this new procedure:

  1. Set the file associations the way you want
  2. Use DISM to export the configuration to an XML file
  3. Optionally, edit the exported file
  4. Place the file in a location accessible by the user.
  5. Enable a different GPO that specifies the XML file.

Problems/Downsides

  • Per-computer GPO, not per-user
  • Settings don't take effect until second logon
  • Monolithic - All or nothing XML file -- only winning GPO will take effect

Alternatives

  • Use DISM to import the configuration
  • Edit the OEMDefaultAssociations.xml file in the system32 directory.  It is applied during logon for new profiles.
