This is very clearly a bug of the small "launcher" module added for Windows 8/8.1 and Server 2012/2012 R2 in order to allow "Live" Microsoft applications to start from the desktop without visiting a web site before (such as Xbox.com) with a browser

That launcher is used to connect to Microsoft Live accounts (Xbox, Office Lice, Essentials, Technet, MSDN) in order to verify the identity of users and check their status (also forcing them to accept some EULA and register a secure account with verified contact addresses or phones)

The games crash even before they have really started. The launcher module just displays a static image of HTML page prior to the connection, and if account validation passes, it will run the software.

The bug is that this launcher module is performing a background HTTP request to online Microsoft Live accounts services, and this requests can complete even before the launcher has initialized its completion routine.

This causes an unhandled exception to ossur because the HTTP query wants to use a callback to inform the client application that the HTTP request is complete.

The bug occurs when you have a fast Internet connection (fibre for example, more than 30 Mbps, and with a roundtrip time not exceeding 40 ms) and a fast PC with multiple cores (very frequent for gamers)

This bug affects in fact ALL games made by Microsoft and adapted for Windows 8 and the Windows Store, but also Skype, several components of Office Live (both of them being used also on Windows Server 2012 or 2012 R2).

That launcher module for the Windows Store will need to be recompiled and corrected in ALL Microsoft applications requiring to subscibe a Microsoft Live account (for various Microsoft brands; Xbox, Technet, MSDN, Office Live, Skype., Azure Clouds..). They ALL have this bug.

Applications in the Windows Store NOT made by Microsoft, and not requiring a Live account do not have this bug ! They cannot be safely launched from the new Metro UI, theur launcher fails before, they just display a static page presenting the software that should be started after verifying status and licences (or updating advertizing channels for free apps).

Note that avertizing is correctly downloaded (before the Live account checking).

Microsoft has simply forgotten the usage of its **own** API to perform asynchronous HTTP requests. The bug however does not affect applciatiosn launched from a website with a browser (the JSON requests are performed in Javascript, and the Javascript engines in all browsers, not just IE but Chrome or Firefox or Opera as well) enforces a correction initialization with a default completion callback (this is standard for JSON requests and XhttpRequests). That bug for asynchronnous HTTP requests also does not occur in Python, or Java (here again there are always default handlers for all HTTP requests).

The bug is typical of applications written in C# (because the CLR does not enforce it and the C# compiler does not check how you use the Windows API), but it also affects applications written in C.

Its origin is a bad device given in the MSDN documentation about HTTP request handlers, which states that a completion routine is only needed in C#; this is plain wrong, it is also needed in C, J#, VB, or other languages, unless the Windows API is encapsulted with a binding library enforcing the rules, like in Javascript or Java or Python.
