How to analyze the log file entries that the Microsoft Windows Resource Checker (SFC.exe) program generates in Windows Vista

How to view the log file The SFC.exe program writes the details of each verification operation and of each repair operation to the CBS.log file. Each SFC.exe program entry in this file has an [SR] tag. The CBS.log file is located in the %windir%\Logs\CBS folder.

Note The Windows Modules Installer service also writes to this log file. (The Windows Modules Installer service installs optional features, updates, and service packs.)

You can search for [SR] tags to help locate SFC.exe program entries. To perform this kind of search and to redirect the results to a text file, follow these steps:
  1. Click Start
    Collapse this imageExpand this image
    , type cmd in the Start Search box, right-click cmd in the Programs list, and then click Run as administrator.

    Collapse this imageExpand this image
    If you are prompted for an administrator password or for a confirmation, type your password, or click Continue.
  2. Type the following command, and then press ENTER:
    findstr /c:"[SR]" %windir%\logs\cbs\cbs.log >sfcdetails.txt

a. user say:

I agree with you - mostly.

Directly's 'independent advisors' are quick to recommend reinstalling from scratch, but that's not a solution, as you explained, it's a do-over. The poster is no closer to understanding what went wrong or how to avoid the same problem in the future.

Moreover, unless you like to use a totally stock install, with no added software and all config settings at their defaults, reinstalling Windows can be a major undertaking that requires a good deal of time and advance preparation if you want to get it right on your first try. Something the IAs forget to mention.

In addition: Reinstalling will effectively remove software that doesn't come from Microsoft Store and replace customized setups with defaults, and posters are often ill-equipped to deal with that. The worst part is that posters are frequently not acquainted with how to backup, so they lose files by reinstalling.

And, of course, if a poster's problem is caused by hardware, reinstalling software won't change anything.

Where I disagree with you a little is when to use the System File Checker and the Deployment Image Service and Management tool. As with a reinstall, these are things that an 'independent advisor' recommends when they don't know what else to say.

The fact is, DISM is used to fix problems that may prevent an update from installing properly (or at all.) If the poster's problem is something else, running DISM is pointless. In addition, DISM replaces missing or damaged files through Windows Update, so if Windows Update itself isn't working properly, DISM won't work and other, more technical procedures are needed to replace damaged or missing files from other sources.

At best, SFC can replace damaged or missing system files if there is a cached copy of the file in question in the Windows directory. It's not a 'fix all' for Windows problems. If a poster's problem is not caused by a protected Windows system file, the poster has wasted their time.

And not only wasted their time: Sometimes SFC returns results that have nothing to do with the poster's problem, or don't really matter. This only confuses posters and can make them waste even more time by chasing their tails.

In the final analysis, the solution to a poster's problem comes from understanding what went wrong and fixing that, not by throwing generic procedures at the wall and hoping that something sticks.

Finally (and I'm ready to shut up now,) when a poster is truly lost and doesn't even know where to begin diagnosing a problem, it's my personal opinion that they are much better served by working with a professional (vs. leaving their computer and saying 'fix it') to solve their problem(s.) That is ultimately more beneficial than back-and-forth in a community forum with people who tell them 'try this' and try that.'
