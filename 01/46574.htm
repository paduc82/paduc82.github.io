alessandro_riolo say:

I have observed an issue with KB4022719 on 7 hosts, all Windows 7.

In my scenario, I was receiving 0x80004005 in the Setup Windows Log as well:

Package KB4022719 failed to be changed to the Installed state. Status: 0x80004005.

After examining the first 4 hosts, looking at the logs, I thought the issues were caused by lack of disk space on the system recovery partition. The clue which led me to think it could have been the lack of space in the system recovery partition was the 0x70 error reported in the CBS log:

BFSVC: Error copying boot files from C:\Windows\boot\PCAT to \\?\GLOBALROOT\Device\HarddiskVolume1\Boot! Last Error = 0x70 "][gle=0x80004005]

The error code 0x70 means ERROR_DISK_FULL, which is self explanative. Over the years, I have seen that connected with the issues due a full system recovery partition in a number of scenarios.

Most of this scenarios have been due the unusual (weird? unexpected?) use of the system recovery partition from a backup solution, Dell AppAssure. There is a well known issue with the AppAssure logs filing up the system recovery partition. As a couple of years ago we moved away from AppAssure (for this class of hosts, we are still using it for some other hosts), we have deleted the AppAssure logs in the vast majority of hosts, but apparently not in this bunch, and for some reason this specific update seems to be using more system recovery space than the recent updates.

I have delete the AppAssure logs from a couple of the hosts, and I then succesfully reapplied KB4022719:

Package KB4022719 was successfully changed to the Installed state.

After applying KB4022719, Windows Update in one of the hosts requested KB3177467, the other KB3042058.
