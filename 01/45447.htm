robert aldwinckle on forums say:

I should call "sfc /scannow" from an elevated command prompt and then run ProcMon and provide the results?

No.  ProcMon needs to be running first.  You are trying to use it to capture and supplement the events involved in the generation of all the diagnostics that the  SFC  (or DISM)  command will be adding into the CBS.log.  

In fact, it might be an idea to reduce the level of tracing first to get used to using ProcMon in that manner.   For example, DISM has a /LogLevel: parameter that we can use to reduce the amount of output that it generates, presumably leaving the essential information if that is enough for someone who can "read between the lines".  But if you were running ProcMon then you could make it easier for yourself to find the events where interesting messages were being generated.  The point is that once you find where in the trace a message that you are interested in was probably written you can use other preceding events as clues to what the message contents are saying, e.g. other registry and file accesses would then be given a clearer context.

The filter that I usually recommend as a preliminary one when all you have is a collection of log messages is:  Operation Is WriteFile.    E.g. before letting ProcMon run, Add that filter and then reproduce whatever it is that you want to trace.  Note that what is traced is the write of the record only, not its content.  So you have to

It's sometimes tedious making sure that you have found the right trace record to match the log message you are interested in but that is mostly a problem for the first one, since others can then be judged relative to that, e.g. by counting and matching lengths quickly.

One other real benefit of having the ProcMon trace is that you can use its Count Occurrences... tool to get a summary list of all the files that were written to and how many writes were done to them.   E.g. count unique Path fields while viewing the Operation Is WriteFile display.  You may find significant diagnostics you weren't aware of that way.   ; )

 

Good luck

Robert

ruwim.b say:

Skype will not "update" itself to 8.37 when 8.38 version is already installed. There must be something peculiar with your computer's system, as e.g. a 3rd party software updater.

Please, run the System Information tool (msinfo32)

Go to Windows Start and in the Run box type msinfo32.exe and press the OK button. This will open the System Information window.  In this window click on File and select Export. Save the msinfo.txt file to your hard drive and post this file on a File Sharing site.
