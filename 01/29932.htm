ken blake - microsoft mvp say:

On Tue, 25 Dec 2012 21:08:34 +0000, John McGuinness897 wrote:

I accidently deleted files from a SDHC card.

Can these files be recovered

An SD or SDHC card is like a hard drive in this respect.

"Deleting" a file doesn't actually delete it; it just marks the space
as available to be used. There are third-party programs that can
sometimes recover deleted files.  Download an undelete program (here's
a good one: http://www.piriform.com/recuva but there are several
others to choose from; do a search).  Ken Blake, Microsoft MVP

vijay a. verma say:
The cause for this problem seems to be third party shell extensions. To get rid of culprit extension, you can download ShellExView. ShellExView is a free third-party application that can be used to manage, disable and enable all of the shell extensions that you have on your computer

Download ShellExView - http://www.nirsoft.net/utils/shexview.zip

Extract the zip file

Launch ShellExView by right-clicking on the application named shexview and clicking on Run as administrator

You will be met with a list of all the shell extensions installed on your computer once the program is done compiling it. Once you see the list, click on Options > Filter by Extension Type > Context Menu.

In the newly compiled list, you will see entries that have pink backgrounds. All of these entries are shell extensions installed on your computer by third-party applications.

Hold down the Ctrl key and click on each of the “pink background” entries to select them.

Once all of the “pink background” entries have been selected, right-click on them and click on Disable Selected Items to disable all of them.

Click on Options > Restart Explorer. Try right-clicking on your Desktop, and File Explorer should no longer crash.

Once you have fixed the problem, next comes identifying the culprit and disabling it for good. To do so, you are going to have to:

Right-click on any one of the “pink background” shell extensions that you disabled and click on Enable Selected Items to enable it.

Click on Options > Restart Explorer, right-click on your Desktop and see if File Explorer

If File Explorer does not crash, keep on repeating steps 1 and 2, enabling a different third-party shell extension every time, until File Explorer crashes and you start experiencing the problem again.

The third-party shell extension you enabled just before the problem returned is the culprit. You can go ahead and enable all of the “pink background” shell extensions you disabled except for this one as this one is the culprit. Keep this shell extension disabled for good – in fact, it is recommended that you uninstall the third-party application that installed this shell extension on your computer altogether.

First the good news:

The chances of there really being something wrong with your hal.dll are minuscule.  

When you see that error, the problem is usually in the c:\boot.ini file which may have gotten messed up by a human being, some accident while trying to fix something else,  or from malicious software.  The more likely issues is that XP just can't find the file where is it being told to look.

The annoying news is that if your system came with XP preinstalled, whatever installation method used to put the XP image (Sysprep usually) may have had an Administrator password and you are unlikely to ever know what it is.  You might be able to call the manufacturer (Dell, Compaq, IBM, etc.) and they can help you.

You can change the Administrator password later though - but first you have to get running again.

You can also then fix your system so that the next time you need to boot into the Recovery Console it will never ask you for the Administrator password again (I like that part).

Photos and Runtime Broker using 30% CPU - Microsoft Community

lucweerts say:

Photos and Runtime Broker using 30% CPU

So I have had this issue for a little while now. It showed up in the CPU usage graph of Afterburner hardware monitor, which I have running on my gaming PC at all times to keep tabs on my system. This CPU graph would be at 5% for a couple of seconds, and then go up to 30% for a couple of seconds, after which it repeats. Note that this is with the system idling. My CPU is an Intel Core i3 6100, so it is not very powerful, but 30% usage during idle is still very high. Also, I am running Windows 10 Home, fully updated. 

So I dug a little deeper, and using Task Manager, found that Runtime Broker is the issue. A quick search found lots of problems with Runtime Broker, mainly about it using 100% and slowing down the system massively. This is not the case, and those solutions did not work in my case. After some more digging on the internet, I learned that Runtime Broker manages the UWP apps, like Edge, Settings, Search and Photos. Most of them have a little leaf shown in Task Manager, to tell you that that app is in a suspended state. 

Photos, however, behaves differently. It has the leaf, but then it disappears. Runtime Broker starts using 30%. A few moments later, Photos gets the leaf again and Runtime Broker goes quiet. 

So, I decided to use End Task on Photos, and lo and behold, Runtime Broker is no longer going up and down. There is just 1 thing: Photos start up again after a little while, and the whole sequence begins again. 

For those who are interested, here is a link to a video, captured with my phone. It shows the up and down of the CPU usage, that Runtime Broker is using CPU and that ending Photos fixes the issue. https://drive.google.com/open?id=1iz9PSsi5IrppoVgZyForP7gYHkYuE_oZ 

This problem persists between Windows updates and even clean installs, so it is not caused by some malicious 3rd party software.

I consider this to be a bug, but there is no good way of reporting bugs to Microsoft. I hope that someone will bring this to Microsoft's attention.

Thank you for your attention.
