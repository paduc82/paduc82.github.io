I thought you were seeing an outline?  Now it sounds like you may be seeing normal functionality and not understanding how it works.  FWIW I think the TK is intentionally ephemeral.  E.g. it closes at the slightest touch of a real keyboard's keys.  Do you have a real keyboard attached when this is happening?  Also, what exactly are you describing by "non-responsive"?  Not coming back when you tap in an input field?  Or not coming back when you try to click

Microsoft Text Input Application (its properties option shows this name: InputApp_cw5n1h2txyewy) 

I don't know what that is or how you started it.  FWIW I tried opening a modern app (OneNote) and using its Search box as an input field.  That made the TK do something different when I started "typing", so it might be what you are seeing, but as soon as I tried to investigate (carefully avoiding using my real keyboard) that overlay disappeared.  

The second process you found  tabtip.exe  is the one that I wanted you to use as a diagnostic.  However, I now think that the  TabletInputService  service (which is an svchost.exe) is the real guts of this thing.  When I use ProcExp (instead of Task Manager) to look at the task structure it is clear that tabtip.exe is started by that service and so killing the service would probably kill both.  In fact, I think I sort of proved that by trying to stop the service.  Note that after the stop attempt that both are running and both have different PIDs.

I don't know what you would need to do to get both back otherwise.  So far, killing tabtip.exe as I described has been sufficient to demonstrate how the process works but I still don't know what exactly the TK window needs when it is open.  It's not just tabtip.exe because normally that process remains running after the TK is closed, by whatever of the multiple ways of closing it is used.
