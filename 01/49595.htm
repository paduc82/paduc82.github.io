      frederik long say:

      I have log on as a standard user account and right click the .cmd , and run as admin. The .cmd script interpreted the %username% parameter as the admin because I have run as admin.

      -> Not quite. There are two cases:

      • You are logged on under an admin account. When you right-click cmd.exe then you get an "elevated" console session that still runs under your current account name.
      • You are logged on under an ordinary account. When you right-click cmd.exe then you get a prompt for and admin account name and its password. The new session will run under this admin account.

      Is there a way to set the %username% parameter as the logged on user, when running the .cmd as admin.

      -> No - see above. Running an elevated session under an ordinary user account would defy the whole principle of admin accounts.

      Either of the two options I gave you in my previous reply would work, with a bit of scripting. A far simpler method would consist of restoring the default permissions for the %userprofile%\AppData\Roaming\Microsoft\Word folder. Users do normally have full access to it. It's my preferred option. Using a scheduled task running under an admin account is a kludge.
