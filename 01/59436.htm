remaker say:

Windows 10 Anniversary Edition: installation failed in the SAFE_OS phase with an error during BOOT operation

I have been tearing my hair out with my Acer V5-571P laptop trying to upgrade from Windows 10 1511 to the Windows 1607 (Anniversary Edition). I even opened case SRX1372701333IDS.

The TL;DR version: The 1607 installer code has a bug and can't cope with some specific GUID disk layouts, especially if they were from a cloned drive with quirky layouts, or from an MBR to GUID transformed system.

Let's review.

My Windows 10 system tried dozens of times to auto update to 1607, and hung every time at the first boot after the files installed. The screen briefly (half second) pops up a DOS window before coming to a screen where the mouse works and you can move the pointer, but the system is otherwise hung.

Once hard rebooted, I get the error message:

0xC1900101 ‚Äì 0x20017
This installation failed in the SAFE_OS phase with an error during BOOT operation.

So, I tried to boot the installer directly from the USB drive directly. And the system hung right away.

Something about the Windows 1607 installer hated my Acer laptop.

SOLUTION: Remove the drive and erase all partitions, then start fresh with a blank drive and the USB based installer. I had to remove the drive since the Windows installer wouldn't boot from USB with the "strange drive" in place; it just hung at the blank screen with the working mouse.

I was set in BIOS to use secure boot, so I couldn't boot an unsigned disk-erasing USB image. I guess I could have turned off secure boot to do the erasing with Ubuntu or DBAN, but I first wanted to swap out and test another drive before I erased my current one.

How did I get the drive into the "strange drive" state?

Well, the system came with Windows 8, configured to MBR boot. I upgraded to Windows 8 and then Windows 10 without incident. I then wanted better performance, so I cloned my install to an SSD and realigned the partition boundaries appropriately. The partitions were set as Recovery, EFI, Free space, Windows, Free Space, "Push Button Recovery." (which I think was the Windows 8 OEM recovery partition. Maybe the presence of multiple recovery partitions was an issue?

Additionally, I had "live converted" the system from MBR to GUID using the Microsoft unsupported method at https://technet.microsoft.com/en-us/library/cc725671(v=ws.11).aspx.

So, something about that collection of steps left my disk organized in such a way that the Windows 1607 installer simply could not cope with it. I'm not curious enough to dive into it much further. I'm happy to provide the troublesome disk image to anyone who cares to dig into it.

With Windows, sometimes you have to burn it all down and start again. I spend a LOT OF TIME resisting that urge, but sometimes you just have to start again.

hal hostetler - mvp-outlook say:
Due to the fragile nature of .PST files, locating them on remote share is "unsupported".  What that means, in this case, is that it will probably work fine, but you run some risk of .PST file corruption which can sneak in due to network issues between you and the remote storage location.  Also, since a .PST file can only be opened by one user at a time, if you should forget to close it on one machine, you won't be able to open it on another until its closed on the first.  Other that that, it should work OK for you.
