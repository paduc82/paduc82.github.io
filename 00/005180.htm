karl timmermans [outlook mvp 2012-2018] say:

Does Outlook 2013 automaticially use the office 2010 PST files or will I have to point it to it during the migration?

If you are installing O'2013 (Office 365) on the same system where Office 2010 currently exists, both versions of Outlook will co-exist (assuming you are using the typical click2run install used by most people as opposed to the standalone .msi installer available only with a MSDN subscription)

Outlook'2013 will make a copy of all '2010 profiles migrating everything accordingly. Once that's done, changes made to an Outlook '2013 profile do not apply to the equivalent '2010 profile and vice versa. In this scenario, Outlook '2013 will have no impact on Outlook '2010.

If you are installing O'2013 on a completely new system, you would need to point to PST files manually (where appropriate) when creating/configuring the Outlook '2013 profile. What default folders get used, infostores are created can depend on the sequence in which your email accounts are created when multiple accounts of different types (POP, Imap, EAS - Outlook.com formerally MAPI using Hotmail connector, Exchange etc) are involved.

Sorry but you can't email an Access form using Outlook.  If you had Access 2007 or above (and Outlook 2007 or above) you could use the newer feature which allows you to send an email using Access/Outlook and having it create a special HTML form or using an InfoPath form (although the receiver would need Infopath on their machine to use it) and then they fill out the form and send it back and Access can update, or add, the information directly into the database.

How to make subform updates affect parent form values? - Microsoft Community

robbob s say:

How to make subform updates affect parent form values?

I have a subform in a main form which has a date field called LastUpdated.  When I update fields in the main form, I update the value of LastUpdated using the Before_Update event (this works, but I'm not sure this is the best way to do this).  I also want to make this work for subforms: if a subform is changed, I want to update the LastUpdated value in the main form.  

I'm having problems making it work using the Before_update event in the subform, but I think the real problem is that my approach is wrong.  There must be a more natural way to do this type of thing...?
I realize this is probably a fairly basic question.

scottgem say:
We need to know more about your structure to help here. What is the main form and what is the subform. Do you have multiple, nested subforms?

Generally you can use the syntax:

Forms!mainformname!LastUpdate = Now()

in the After (or Before) Update event of the subform. 

ken sheridan say: Essentially what you seem to be saying in terms of the underlying reality modelled by the database, is that if an entity (represented by a row in the parent form's table) is amended, or if any entity related to it (represented by a row in the sub-form's table) is amended, then you want that logged.  To me that seems a perfectly reasonable thing to do, and should be achievable relatively easily.  BTW don't confuse an entity with an entity type.  A table models the latter, a row in a table the former.

Keep your current update of the parent form's record via the BeforeUpdate event procedure as it is.  For amendments to the subform's table's records use the subform's AfterUpdate event procedure to execute an SQL statement to update the parent form's current record.  If we assume as an example that the parent form is based on a table Orders and the subform on a table OrderDetails, linked by OrderID then the code would be along these lines:

Dim strSQL As String

' update parent record with current date/time
strSQL = "UPDATE Orders" & _
    " SET  LastUpdate = #"  & Format(Now(),"yyyy-mm-dd hh:nn:ss") & _
    " # WHERE OrderID = " & Me.Parent.OrderID

CurrentDb.Execute strSQL, dbFailOnError

' refresh parent form
Me.Parent.Refresh

Note the use of the ISO standard for date notation of YYYY-MM-DD hh:mm:ss when building the date literal, to ensure international unambiguity.

One thing you might like to think about, however, is what is meant by 'update'.  A row in a table is updated if edited, but that does not necessarily mean any data has changed.  If you want to log actual changes to data then simply using the BeforeUpdate or AfterUpdate event procedures is not enough.  To cater for this you'll find a file ChangedRecord.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo illustrates the use of a RecordHasChanged function which detects real changes to data rather than simple updates.
