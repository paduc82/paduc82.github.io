amall say:

C:\Program Files\Microsoft opens at startup and not MSE

On a couple of installations, I experienced a weird bug where the folder C:\Program Files\Microsoft would open upon startup, but MSE would not open.

Upon inspection of HKEY_LOCAL_MACHINE\Software\Microsoft\Windows\CurrentVersion\Run, I found that the entry for MSE was C:\Program Files\Microsoft Security Essentials\msseces.exe -hide, without quotes.

Windows interprets this path as C:\Program Files\Microsoft (there is no C:\Program folder, so it goes to Program Files), and opens that folder instead of launching MSE.  Placing quotes around the directory as such:

"C:\Program Files\Microsoft Security Essentials\msseces.exe" -hide

rectified the problem.

ken sheridan say: You can drop the PartlsListID and OrginalPLID columns from the table for reasons we'll come to.  As each row represents an entity of type PartsList, the Key column would be more appropriately named PartsListID in fact.  The resulting redundant duplicate row from your sample set can be deleted, so we are left with the following set:

PartsListID           Qty         PartNumber       PartPO      PartPrice
1                          3             xxxxxxxx             xxxxx        $$$$$
2                          2             yyyyyyyy             yyyyy        $$$$$

The binary relationship type between this table and Orders now has to be modelled by a further table, OrderDetails say, like this:

OrderID    PartsListID
200           1
200           2
210           1

This table is 'all key', i.e. its primary key is a composite one made up of the two foreign key columns.  I'm unsure what attribute PartPO represents, and whether it is transitively dependent on the key via PartNumber, and consequently whether table is normalized to Third Normal Form or not.  At first sight I'd assume it represents Part Purchase Order, but I can't quite see how that would fit into the model.  PartPrice is functionally determined by the key of course, as price is not time-independently determined by part number, so is valid in this table.

With this set of correctly normalized tables (ignoring the possible question mark over PartPO for the moment), there is no longer any need to update 'copied' rows, as there no longer are any such rows.

Hello

If the account you want to close is tied with your Microsoft account, please be reminded that deleting it will also delete your entire Microsoft account and if that Microsoft account is also tied to the Skype account you want to keep, that will be deleted as well.

You will also have 2 Skype accounts under one email address, if you have signed in to Skype using your Microsoft credential without linking it to your Skype account. This is due to Microsoft migration and the option to sign in to Skype using Microsoft account.

To prevent from accessing the second account, I suggest to link your Microsoft and Skype accounts. This way, you can sign in to Skype using your email address without directing to a different profile. To link your accounts, please refer to the steps below:

  1. Sign in to the Microsoft website using your Skype name.
  2. Once you see the message "Your Skype account is now a Microsoft account", click Next.
  3. Once you're asked for the email address of the account you want to link your Skype Name, please type your email address.
  4. Follow the steps to continue the merging process.
