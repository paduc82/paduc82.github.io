ken sheridan say: You were on the right track.  Referencing the column's Value property is the correct approach.  In the report design you then have to group the report so that the repeated data is returned once only in a group header, while the individual values of the species are returned in the detail section.

As you have multiple multi-valued fields, however, you may well find it difficult achieving the layout you want.  The problem will be compounded by the fact that not every multi-valued field will have the same number of items per row in the table, so you will almost certainly still get unwanted duplication of values from some, requiring you to further group the report to eliminate these.

In view of these difficulties I would recommend that you do not use multi-valued fields at all.  A multi-valued field is really just a way of representing a many-to-many relationship type, e.g. each site will have one or more species of trees, and each species will be represented in one or more sites.  While it is a superficially simple way to do this it is very limiting.  The traditional way to represent a many-to-many relationship type like this in a relational database  is to model the relationship type by a separate table which resolves it into two one-to-many relationship types.  In broad outline the tables would be like this:

Sites
….SiteID  (PK)
….SiteName
….etc

TreeSpecies
.…TreeSpeciesID  (PK)
….TreeSpecies

and to model the relationship type:

SiteTreeSpecies
….SiteID  (FK)
.…TreeSpeciesID  (FK)
….other columns to represent attributes of the relationship, e.g. quantity of the species at the site

The primary key of this last table is a composite one made up of the two foreign key columns.

Using this time-honoured method of modelling a many-to-many relationship type you can then base a report on the Sites table, and within include a subreport based on a query which joins the SiteTreeSpecies and TreeSpecies tables.  Similarly you can include other subreports in the report to represent other many-to-many relationship types currently represent by multi-valued fields.  Not only does this avoid any unwanted duplication of data, it gives you much greater freedom in the layout of the final report than would be the case with a single grouped report.

  1. Exit Outlook.
  2. Start Registry Editor. To do this: Click Start, and then click Run to open a Run dialog box.

Key: HKEY_CURRENT_USER\Software\Microsoft\Office\x.0\Common

DWORD: BlockHTTPimages

Value: 1

  • With Group Policy:

Key: HKEY_CURRENT_USER\Software\policies\Microsoft\Office\x.0\Common

DWORD: BlockHTTPimages

Value: 1

  1. Right-click the BlockHTTPimages key, and then click Delete.
  2. When you are prompted to confirm the deletion, click Yes.
  3. On the File menu, click Exit to exit Registry Editor.

Note: If the registry value is located under the Policies hive, it may have been configured by Group Policy. If it was configured by Group Policy, it will be restored when this policy is refreshed on the computer. Your administrator must change the policy to change this setting.
