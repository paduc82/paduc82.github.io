For the first issue, the problem is that, until you accept the change, the text contains both the original word (marked as a deletion) and the replacement (marked as an insertion). You can see that if you go to the Review tab and set the Show Markup > Balloons selection to Show All Revisions Inline. For the second issue, I do see that the Find and Replace doesn't highlight the found item on the first line of the next page. I think this has to do with the fact that moving the selection to that instance causes the screen to scroll, placing the the selection at the top of the window. It's a bug, but only a cosmetic one, and I know that Microsoft is reluctant to spend resources to fix things that don't cause data loss.

Expanding on Doug's suggestion, you can wrap a Content Control around the table in two different ways. Both allow you to stop edits and deletions of the content UNLESS the user is clever enough to work out how to remove the content control (eg. downsaves the the document to doc format).

ken sheridan say: You'll find an example in CopyTree.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

This little demo file copies the current record and all referencing rows in two other tables by calling a CopyTree function.  In the function the primary key of the new row  in the referenced  table (Countries) is computed with:

    ' get next CountryID value
    lngNewCountryID = DMax("CountryID", "Countries") + 1

so, to insert a value manually you would need to replace this code with code which gets the new value manually.  This could be from an unbound control in the form from which function is called, or by calling the InputBox function for instance.  If doing this the value should of course be validated before executing the SQL statements, by code which confirms that it is not duplicating an existing value in the table.

The example used in the demo is an absurd one of course, and deliberately so to reflect the fact that a more realistic scenario would not be where the whole tree structured hierarchy is copied, but where the relationship type between the referenced  table and the table referencing it is a many-to-many relationship type, in which case it would be the relationship as represented by a row in the table which models the relationship type which is copied, i.e. the model would not be a tree but a 'convergent graph' in which there can be more than one path between two nodes.
