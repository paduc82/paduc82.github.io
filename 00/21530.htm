MS Project 2010: Suggestion for managing Buffer - Microsoft Community

ladywhitmofgp say:

MS Project 2010: Suggestion for managing Buffer

I am managing projects with MS Project 2010 Professional.    I am able to baseline the project, and save a copy of the baseline to store the original Scheduled Finish Date (at baseline).   I am also able to store the original Critial Path tasks, with all of their projected start and finish dates (at baseline).   

My problem is I haven't figured out how to accommodate for buffer (and the consumption of buffer) for my critical chain tasks.   What I would really like to do is:

  • Create an ability for the project file to calculate an agreed amount of buffer (... say 30%);  buffer would be added only onto the critical chain path tasks.  
  • Once that buffer is calculated (and scheduled... at baseline),  it is SET, and never changes unless I re-baseline.     That means, I have a Project Scheduled Finish Date (unbuffered),  and a Project Buffered End Date (which takes into account the scheduled finish, plus the buffer days..... and this doesn't change).
  • As I am regularly updating, and the project begins to slip,  the buffer amount is "used up" if the slippage impacts the critical path.
  • Thus, I can report the "health" of the project considering the amount of buffer consumption.

Is there a way to do this in MS Project 2010?    Or, is there a macro that can be used to pair with my  mpp file and make that extra calculation(s) for me?     I appreciate any help.

Thanks!

john - project say:

Lady,

Interesting approach. I can't think of anything you could do with the basic application to implement your buffer, other than manually entering a buffer or buffers, but it definitely could be done with VBA.

I assume you are not working to a fixed completion date, since adding any extra buffer time, even to one of the critical path tasks will push the end date out by that much. So the first question you have to ask is, how much total time can I add to the schedule and still meet some end goal. Once that is agreed upon, then you could potentially spread that time, perhaps proportionally among all the critical tasks. Even then, someone has to make a decision as to how much buffer time is appropriate for each critical path task. Just because a task is on the critical path doesn't mean they are all equally important or "critical" to success of the project.

So, yes, it could be done with VBA but there is a lot to consider and the algorithms in the code could get complex in a hurry.

How about a simpler approach of a single buffer task added to the last task in the critical path. If the project slips, manually adjust that buffer task to re-align the plan. This may seem like a crude approach but even though I personally like the challenge of VBA, a manual approach is very straightforward and will, in my opinion, give just as much visibility on the "health" of the project as an sophisticated automated approach.

John
