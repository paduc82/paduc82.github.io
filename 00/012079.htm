scottgem say:

Mike,

I'm going to agree with Roger yet again. Helping you has been a pleasure and we will be here to support you as your project continues.

The main difference between bound and unbound forms is that, with a bound form, Access handles all the I/O between the form and the table. With an unbound form, the user has to write code (VBA), to handle the I/O. The complexity of that code can vary widely depending on the usage and purpose of a form. For example, a form solely for entering new records is fairly easy to code. But a form for retrieving data and populating the unbound controls, can be very complex. The coding involved depends on the table structure and form layout. Personally, I'm lazy and tend to use bound forms wherever I can.

There is only ONE circumstance in Access where a child record can be created with an automatically populated FK. And that's when using a subform. When you correctly link a subform to the main (parent) form, then any records created in the subform have the FK automatically populated. This is one of the reasons I said that subforms are a very powerful feature. But this only works with bound forms. So, in every other circumstance, when you add a record to a child table, you have to make sure the FK is populated correctly.

 

Hope this helps, Scott<> P.S. Please post a response to let us know whether our answer helped or not. Running a macro containing the following code will allow you to select a folder containing the documents and then open each document in that folder and process it:

Dim fd As FileDialog
Dim strDocFolder As String
Dim strDoc As String
Dim doctoclean As Document
Set fd = Application.FileDialog(msoFileDialogFolderPicker)
With fd
    .Title = "Select the folder containing the documents."
    .AllowMultiSelect = False
    .InitialFileName = ""
    If .Show = -1 Then
        strDocFolder = .SelectedItems(1)
    Else
        MsgBox "You have not selected a folder."
        Exit Sub
    End If
End With
strDoc = Dir(strDocFolder & "\*.doc*")
While strDoc <> ""
    Set doctoclean = Documents.Open(strDocFolder & "\" & strDoc)
    With doctoclean
        .RemovePersonalInformation = True
        .RemoveDocumentInformation (wdRDIRemovePersonalInformation)
        .Save
        .Close
    End With
    strDoc = Dir()
Wend

Then the chapter has a Table of Contents (TOC) for that specific chapter that is built from the heading fields and it has the same "f"-switch identifier as the headings and sub-headings.  This is inserted by clicking on Insert>Quick Parts>Fields, then in the Field Names listbox you click on "TOC", then click on the "Table of Contents" button, then click on "Options..." button, then deselect the "Styles" and "Outline levels" checkboxes and click/select/check "Table entry fields".  Then click on OK and dismiss the dialog boxes.  After the field is entered, you type in the same "f"-switch identifier for that chapter (it matches all of the heading "f"-switches).

The idea here is that each chapter's TOC is built from ONLY from the headings and sub-headings of that particular chapter/section.  I assume that is why the "f"-switch identifiers are supposed to be the same.

MY PROBLEM:  Some of the chapter TOCs are incorporating heading/sub-heading field code "f"-switches from prior chapters.  So, for example, let's say I have chapters A, B, C, D, E, F and G.  My "G" chapter TOC is incorporating fields from chapters B, C and D, as well as G.  This should not be happening because the "f"-switch identifiers for chapters B, C, and D are unique to those chapters.

a. user say:

Before we go any further, another thing you could do is to use other alphabets to go beyond the 26-letter limit. e.g. a Greek delta character is not treated as a "d". But I leave you to try that for your self.

If you use the \b bookmark approach, because you already have TC fields where you have used the \f switch to specify a letter, you have to make a choice.

If each chapter is set up so its TC fields all have the same letter, you can either leave that \f switch there, or you could remove all the \f switches in the TC fields and just rely on the \b switch in the TOC.

That's because you can either specify

{ TOC \f \b "hazcom" }

which will pick up all TC fields covered by the hazcom bookmark that *do not* have an \f letter switch, or, say,

{ TOC \f "h" \b "hazcom" } if all the TCs covered by the hazcom field use \f "h"

Unfortunately "\f"  on its own does not just pick up all the TC entries covered by the bookmark, and you cannot do, e.g. \f "a,b" in the ToC.

Even if your message doesnt exceed the Outlook or Exchange Server attachment size limits, there is a chance your recipients email provider might block messages that include large attachments. To help reduce the size of attachments that you send in Outlook, see Reduce the size of pictures and attachments.

In addition to mail provider limits on total message size, sending large attachments through email is not always the best solution.

  • Your message might cause your recipients mailbox to exceed the storage quota    If the recipients mailboxes exceed the storage quota, the recipients could be prevented from receiving other messages.

  • Your message might cause your mailbox to exceed the storage quota    Each message that you send is saved in the Sent Items folder. Sending large attachments to other people counts against your mailbox size quota as well. When you exceed the storage quota, you could be blocked from receiving additional messages.

  • Attachment bloat    When a file is attached to message, it has to be encoded. The encoding process causes attached files to become 1.37 times larger than they are on your computer. For example, if you send a 5 MB file attachment, the attachment is sent as 6.85 MB of data.
