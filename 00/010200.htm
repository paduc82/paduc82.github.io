I guess the problem is not so much that upgrades break stuff (certainly I've seen very little that would break a custom masterpage in many years of customising SharePoint Online). The bigger problem is that you miss out on new stuff if your masterpage is based on an earlier version of Seattle.

The classic case would be the updated suite bar and app tray - without constant maintenance of your custom masterpage, your customers would still be stuck on the old suite bar.

Now, that's not necessarily a bad thing - SharePoint Online can be a turbulent place when things are changing, and using a custom masterpage can insulate from that. But its worth factoring in ongoing maintenance to include new features when they're stable.

bobgreenblatt say:

Yes! You are having the problem because you are using incorrect syntax. By enclosing the arguments in parentheses, you are asking the msgbox function to return a value. Therefore, you need to set a variable to accept the value returned. So, you need to do either:

result=msgbox ("hello"),vbyesno)

Or

msgbox "Hello", vbyesno

In the first case, you can then check the value o result to see if it is vbyes or vbno.

qa guy commelec say:

You should have at least three tables - tblRecipeTitles, tblIngredients, and tblRecipeIngredient.

tblRecipeTitles would have autonumber field RecipeID as primary key and text field for title.

tblIngredients would have autonumber field IngredientID as primary key and text field for name of Ingredients.

tblRecipeIngredient would have autonumber field RecipeIngredientID as primary key and number fields (long integer) RecipeID and IngredientID as foreign keys.  

The primary keys would be joined to the foreign keys one-to-many with referential integrity.

To load the data use a form to enter new recipe and subform to with combo to select ingredients.

To search the data use a form with combo to select ingredient and subform to list the recipes having that ingredient.  The query for the form/subform would be refreshed upon selection of ingredient.

ken sheridan say: A primary key is one or more columns which contain distinct values by which any row in a table can be identified.  In your case RecipeID would be the primary key of the Recipes table and IngredientID would be the primary key of the Ingredients table.  These will normally be autonumbers.

A foreign key is identified as such when you create a relationship between two tables.  In your case when you create a relationship between Recipes and RecipeIngredients the RecipeID column in the latter is a foreign key.  Similarly when you create a relationship between Ingredients and RecipeIngredients the IngredientID column in the latter is a foreign key.  The foreign keys must not be autonumbers of course.

Assuming that each ingredient can only appear once in each recipe IngredientID and RecipeID in combination are what is known as a candidate key of the table.  To define them as such they must be included in a single unique index.  The easiest way to do this is to make the two columns the composite primary key of the table, which you can do easily in table design view.  You can include a separate autonumber column as the primary key if you wish, but it serves no purpose to do so in this situation, merely adding an additional overhead in the maintenance of an unnecessary index.  If you do add such a column as a surrogate key, the two foreign keys are still a candidate key and must be defined as such by their inclusion in a unique index.  It's far simpler therefore to define the two foreign keys as the composite primary key.

The model for my demo database which I posted earlier illustrates exactly this sort of structure.  Have you been able to download the file?  If not you can mail me at kenwsheridan<at>yahoo<dot>co<dot>uk and I can send it to you directly.

graham mayor say:
Figure captions are numbered with SEQ fields and are cross referenced with REF fields. SEQ fields do not contain the number and the cross references do not contain the sequence number. A wildcard search is not going to be of much help here. Without knowing what exactly you are trying to do with these references, it is difficult to suggest a way forward. However you can use vba to step through the cross references

Dim oFld As Field
For Each oFld In ActiveDocument.Fields
    If oFld.Type = wdFieldRef Then
        If InStr(1, oFld.Code, "_Ref") > 0 Then
            oFld.Select
            MsgBox oFld
        End If
    End If
Next oFld

It may simply be that you have edited the document and the ref fields merely need updating. If you want to do that in code, see the example macro at http://www.gmayor.com/installing_macro.htm
