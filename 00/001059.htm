jay freedman say:

I generally agree with Doug's recommendation to use a userform (custom dialog box) instead of legacy form fields to gather the information. I'll add a suggestion about "Section 2" and post-processing.

Since the user doesn't need to see the blocks of text, there is no reason to include it in the document that the user edits. The blocks of text can be stored as building blocks (e.g., AutoText) in the Word template that contains the post-processing VBA. The necessary blocks can be inserted at that time, rather than having to delete the unnecessary ones. The code should also populate document variables with the names of blocks as they're inserted, so the code can avoid adding duplicate blocks.

Regarding the "fragility" of bookmarks: If a user deletes text that contains bookmarks, that also deletes the bookmarks. The start and end markers of bookmarks are invisible unless they're deliberately turned on in the Options, and there's no warning when a bookmark is about to be deleted, so it's nearly inevitable that this will happen. If a macro relies on the presence of a bookmark to operate, that accidental deletion will cause the macro to malfunction.

The Caller ID feature is entirely dependant on the weakest link in your network, in this case the SFR Network operating in the Dominican Republic may provide too many hurldes for this feature to work reliably.  If the call originates from a landline hosted by a telephone switch that is incompatible with Caller-ID, all sorts of problems can occur.

Of course, there's always the possibility that the issue you are having is normal; if you get number XXX-XXX-XXX1 on Caller-ID and you redial it and it's always busy, then it's hard to narrow that issue down to a problem specific to the Caller-ID feature.    If caller XXX-XXX-XXX1 calls and you know this person should have number XXX-XXX-XXX2, then that is a problem to report to your provider.
