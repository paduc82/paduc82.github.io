ken sheridan say: I'd suggest you change your code to open Form2 at an empty new record, and in dialogue mode:

Private Sub AddRecord_Click()

    DoCmd.OpenForm "Form2",DataMode:=acFormAdd, WindowMode:=acDialog
    Forms![Form2]![txtDUF] = Me![DUF]
    ' code to requery Form1 goes here

End  Sub

As Scott says the code to requery Form1 will depend on where it sits as a subform  in relation to the Add Record button.  If the button is in Form1 itself then you simply need:

    Me.Requery

If it's in Form1's parent form then you would need code like this:

   Me.YourSubformControl.Requery

where YourSubformControl is the control in the parent form's Controls collection which houses the subform.  This might or might not have the same name as the source form object.

If Form1 is nested more than one level deep in relation the form in which the Add Record button is located then you'd need to extend the code, but we'd need more detailed information from you on the exact set up of the form/subforms.

The other question is whether you want Form1 to return to its currently selected record after being requeried, or perhaps to the newly inserted record?  This would require some further code as calling the Requery method moves the record pointer to the first record in the form's recordset order.  If you want it to go to the newly inserted record then you'd need some value by which to identify that record so that you can navigate to it.  If it has an autonumber primary key this could be the Max value of the key, though that is not completely reliable as an autonumber guarantees distinct values only.  Usually the next number in sequence is inserted, however, so it will usually be reliable as a basis for identifying the new record.

ken sheridan say:

I would like that when form2 close after saving the data, it returns to Form1 and automatically through code requeries Form1.

The code I posted earlier is intended to do that, but as your Form2 is unbound you won't need to call the DataMode argument.  As the button is in the subform you simply requery it after opening Form2 in dialogue mode.  When a form is opened in dialogue mode code execution is paused until the dialogue form is closed (or hidden).  I did make one error in the code, however.  Because of the pause in code execution you can't assign a value to a control in the dialogue form.  What you have to do is pass the value to the form as the OpenArgs argument of the OpenForm method, so the code would be:

    DoCmd.OpenForm "Form2", WindowMode:=acDialog, OpenArgs:=Me.DUF

Me.txtDUF = Me.OpenArgs

Once you close Form2 code execution in the calling procedure will resume, and the Requery method will be called.

ken sheridan say:

1.  To navigate to the newly added row you need to firstly obtain the value of the primary key in that row.  As this is an autonumber it should be the highest value of that column (though this can't be absolutely guaranteed).  Then you need to find that row in the subform's RecordsetClone. Finally you need to synchronize the subform's Bookmark property with that of the RecordsetClone.  The code for this would be added to the current code, following the Me.Requery line, and would be as follows:

Dim lngAndreID As Long

' find highest AndreID value in table
lngAndreID = DMax("AndreID", "NameOfTheTableGoesHere")

' find row in ReciordsetClone and synchronize bookmarks
With Me.RecordSetClone
    .FindFirst "AndreID = " & lngAndreID
    Me.Bookmark = .Bookmark
End With

2.   The additional code for the DblClick event procedure would be along these lines, but it might need some debugging on your part.  Getting it right without having access to the objects is not always easy:

Dim frmParent as Form
Dim frmSub As Form

Dim frmSubSub As Form

Dim strFilter As String

strFilter = "UtdanningID = " & Me.RecordID

Set frmParent = Forms("00Kartlegging")
'  NB: The following line relies on the names being the those the subform controls
'  i.e. the controls in the parent form/subform which house the subform/subsubform.  These might or
'  might not be the same as their source form objects.

Set frmSub = frmParent.20KartTab22.Form

Set frmSubSub = frmSub.UtdanningDetail.Form
frmSubSub.Filter = strFilter
frmSubSub.FilterOn = True

frmParent.NameOfTabControl.Value = 1
frmSub.NameOfTabControl.Value = 1

As the Value property of a tab control is zero-based 1 is the second page of a tab control.

We have asked numerous different experts and can't seem to get the program to be multiple user friendly.

 

Thanks in advance!

john w. vinson say:

Your database must be "Split" into a Backend containing only the tables, and a Frontend containing the forms, reports, queries and code. Each user should have her own copy of the frontend, linked to the shared backend. See http://www.granite.ab.ca/access/splitapp.htm for details.

Without splitting, you'll not only get this kind of contention, but risk rapid bloating of the database and a considerable risk of irreversible corruption. You certainly cannot have two users opening the database in design view (changing code or forms) at the same time; this restriction started with 2007.

brian tillman [outlook mvp 2007-2019] say:

Open My Computer in Windows Explorer by double-clicking on it.  Click Tools>Folder Options.  Select the File Types tab.  Scroll down until you see "URL:MailTo Protocol".  Select it and click Advanced.  Select "open" and then click Edit.  In the "Application used to perform action" field, change the string "IPM.Note" to "IPM.Appontment". Click OK, then OK, then OK.  The next time you click a MailTo URL, you'll get a New Meeting form, with the address on the MailTo as one of the attendees.

I think this is of limited utility, however.  When I verified it worked, it did work - once.  I did not have Outlook started when I tried it and when I tried it a second time, the New Message form opened.  I checked the association and it had reverted to "IPM.Note" on the command line.  I find that if I leave Outlook running, then the association remains as I change it, but if Outlook is closed when I clicked the URL, Outlook resets the command in the association when it starts.  This is with Outlook 2010, however, so you'll have to check with Outlook 2003.
