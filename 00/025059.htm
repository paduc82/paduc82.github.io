I have found solution last night diggind deep inside nbu file and RFC regarding VCARD file format.

The reason why contacts were not restored is very simple yet very frustrating.
In Poland we use characters that have ASCII codes higher than 127 (7-bits). PC Suite 6.8.13 writes backup file with those characters in binary form. Meanwhile it sets ENCODING flag in VCARD as QUOTED-PRINTABLE. If so those characters should be changed to =XX form rather than binary form.
I wrote simple PHP script (since PHP server was handy;)) that changed all occurences of binary chars in VCARDS. Then all contacts were imported without any problems.

I can post PHP source file here if moderators won't ban me for that ;)

So here is the conclusion: Dear Nokia. When you back up phone with UTF-8 characters you have to convert it to =XX form. At least if you want to be able to restore them back to NokiaOS based phone. From my experiments I can say that Symbian 9 (N71,72) based phones have no problems with this backup file.
(Please refer to: RFC 2426, http://www.ietf.org/rfc/rfc2426.txt for more details in VCARD 3.0 format)

I assume that the same problem may be with VCALENDAR but my calendar items fortunately have no polish characters inside.

I hope this helps other people. I suspect that this problem will only occur on NokiaOS based phones with characters that not belong to US-ASCII (32-127 decimal) character set.

I've made a few tests and unfortunatelly I found that CopyPicture causes the memory leak (I assume that Export use the same routine internally).

If I run this simple sub in XL2010 I can see that the memory usage of EXCEL.EXE grows and never shrinks until I close the whole application.

Sub Test()
  'Press ESC to abort this sub
  Application.EnableCancelKey = xlErrorHandler
  On Error GoTo ExitPoint
  Do
    ActiveChart.CopyPicture
  Loop Until False
ExitPoint:
End Sub

If I run this simple sub in XL2000 it works, the memory grows and shrinks in a stable range. I guess that it works in XL2002 and XL2003 too.

I see 3 possible solutions:

a.) Ignore the problem and restart the application automatically at midnight (shut it down with a macro, use the task scheduler to start it again).

b.) Install VirtualPC on your maschine with Windows and Excel prior to 2007 and run the code inside that machine.

c.) Rewrite the code and use only APIs to copy and write the pictures. But this is a terrible work and you need a lot of code.

Andreas.

e.j. gun say:

Option a.) is one that I use all the time.  I have some workbooks that process data around the clock, and I have found that it's always a good idea to give Excel a break (shut it down) and then give it a fresh start (restart it).  You can use the Workbook_Open event to kick off your macro each time the workbook is opened by the task scheduler, and use the OnTime method of the Application to choose when to shut it down.

 

Sub Workbook_Open

    Application.OnTime TimeValue("23:59:00"), "Shut_Down_Workbook"

End Sub

 

Sub Shut_Down_Workbook

    ThisWorkbook.Close SaveChanges:=True

    Application.Quit  ' Forgot to add this for your case...

End Sub

 

Have the task scheduler start the workbook up every morning at "00:05:00" or something like that - whatever works for your data reporting schedule.

 

Eric
