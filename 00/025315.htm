resourceful say:

Hello. While it is true that you won't be able to sync files using AS/WMDC over WIFI, Windows Mobile does support networking (WIFI) and accessing shared network folders. So, start by creating a shared folder on the PC. Place your file(s) you want to transfer to the phone in there. On the phone you can go two ways:

1) Open IE and type \\computername (computername can be found for the PC. Control Panel>System Properties>Computer name tab) in the address bar. File Explorer will open and you will be prompted for any password. Open the shared folder and tap/hold the file. You should be given the option to cut or copy. Navigate to where you want to place the file on the phone. Select menu>edit>paste.

2) Open File Explorer. Go to menu>open path. Type in the computername. Open the shared folder and select the file. Proceed to cut/copy and paste.

Of course, you can transfer a phone file to the shared folder on the PC the same way. No extra apps. A great feature of Windows Mobile. Hope this helps.

The items in On My Computer are completely separate from your Exchange Server. If you're moving items to the On My Computer folders and not seeing a decrease in Exchange usage, you're probably only copying the messages, not moving them.

When you click a drag a message to the On My Computer folders, you should not see a green plus mark:



This green plus mark is saying that Outlook will create a copy of the message and not move it from the original source.

This will move the message and free up the space on your Exchange server account.

which file extensions should be excluded in mse ? - Microsoft Community

jagrataparui say:

which file extensions should be excluded in mse ?

I think mse takes very long time to scan cause it has no smart scan technology to exclude the files which can't be virus . Mse scans all types of files that takes so much time .So i want to know which files can't be virus and which file extensions should be excluded in mse ?

kosh vorlon say:

HI Jagrata,

 

The word "can't" is a big word.  While there are admittedly some file types today that have not knowingly been successfully infected, each day 70,000 new variants of malware enter the world!  In the past, there were types that "couldn't" be infected and then someone came up with a way to do so and suddenly those file types were vulnerable.  Furthermore, there is no list I'm aware of that specifies "invulnerable" filetypes.  Some believe that most video and picture files cannot be infected, and TXT files either, but I know about variants that can infect JPEG files for example (though I'm not totally familiar with the others).  It once wasn't true for JPEG either, but someone came up with a way.  In time, it is possible if not probable that someone will find a way with the others as well eventually.  It just isn't worth the risk to save a little time.

 

I recommend you exclude only filetypes that are generating "false positive" results (certain files created by Microsoft Visual Studio products can do that - I can't recall one off the top of my head but recently we saw such a case and that's how it was resolved).  Otherwise, let MSE do its job as best as it can to protect you even if it does take a while for scans (especially full scans) to complete.  This way you know you're as protected as MSE can make you (assuming you've also got SpyNet turned on - that is critical to making sure you are as protected as possible).

 

Plus some files are intentionally given the wrong filetype or the real filetype is hidden (for example, sample.txt.vbs may appear as sample.txt but actually be a VB scriprt file that can infect you).  If accessed by the wrong program (intentionally or through some other infection intended solely for the purpose of activating the first infection), this could activate the infection.

 

In short, I strongly suggest you not exclude filetypes based on anything you read or find online about some that "cannot be infected" because while that may or may not be true today, with 70,000 new strains being released daily by some very smart people, it seems inevitable that those "impossibilities" will be overcome and if you've excluded them then you'll be infected by the first one that comes along.  Don't place your system at risk just to save a little time.

 

I hope this helps.

 

Good luck!

A forum like this is better suited to asking and getting answers to specific questions. So I agree with Tom, that you may be better off with a consultant.

The answer is your third option: In a separate table, 1 record per student per test.
