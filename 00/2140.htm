ken sheridan say: As you refer to subforms within the parent form this presupposes that there are rows in a number of related tables.  You cannot therefore duplicate the data without saving it as referential integrity requires that the row in the referenced table (that underlying the parent form) be saved before a row can be inserted into any of the referencing tables (those underlying the subforms).

I'd suggest that you adopt the following strategy:

1.  Firstly ensure that referential integrity and cascade deletes are enforced in the relationships between the parent form's underlying table and its referencing tables, i.e. those underlying the subforms.

2.  Create a duplicate of your parent form, and remove the code which locks the controls.

3.  In the parent form's module, e.g. in the Click event procedure of a button do as follows:

3.1 Grab the value of the form's current record's primary key to a variable.

3.2 Execute an SQL statement which inserts a new row into the parent form's underlying table, using the data from the current record but incrementing the value of the Rev column by 1.

3.3 Execute SQL statements to insert rows into each of the referenced tables.  For each insert from a SELECT statement which selects the columns from the relevant referenced table where the relevant foreign key column matches the value of the variable holding the primary key value of the current row in the referenced table, but again increasing the value of the Rev column by 1 if this is the key.

3.4 Open the new form created in 2 above, setting the WhereCondition argument of the OpenForm method to an expression which filters the form to the new row inserted in 3.2 above, and setting its window mode to acDialog to open the form in dialogue mode, which will force the user to close it before returning to the calling form, and will cause the calling code execution to pause in the meantime.

4.  In the Close event of the new form include code which pops up a message box in which the user is asked to confirm whether to save the new record and its related records.  If the user answers Yes then the rows are automatically saved, but if the primary key of the new row in the referenced table is not the Rev column, but something else, e.g. an autonumber, you'll need to pass the value of this back to the calling form, e.g. by assigning it to a hidden text box control in the original form, so that it can be used in 5.2 below to navigate to the new record.  If, on the other hand, the user answers No, execute an SQL statement which deletes the newly inserted row from the referenced table.  As cascade deletes have been enforced in the relationships the matching rows from the related tables will also be deleted.

5.  Back in the original form, the calling code should then, following the line which opened the new form in dialogue mode (code execution has been paused here, remember), do as follows:

5.1 Requery the parent form to include the new record in its underlying recordset.

5.2 Navigate to the new record in the usual way by finding the record in the form's RecordsetClone, and, if found (i.e. if the user elected to save the new record in 4 above), synchronizing the Bookmark of the form with that of its RecordsetClone.

el_loco---01 say: Here we go again.

If you video is 3GP format, first convert the video to AVI. Use Super or other video encoders to encode into xVid or Divx AVI.

Then use Windows Movie Maker to get the video there and then use the options there to rotate the video and save it in rotated version.

You can also convert the video to MPEG-2 (using Super) then use this converted video and re-encode it to whatever you want, just use the DirectShow button in Super encoder and then select the option to rotate the video there.

I've used both methods and it works.

If the video is MP4. Download and install a little program called MP4CamtoAvi (just google it) run the program (follow the instructions there) and convert it to AVI.

Once it's in AVI format use Windows Movie Maker or Super as mentioned above.

I strongly recommend you get the DivX codec from www.divx.com so that it's installed in your PC and divx codec is Directshow. IF your Windows Media Player can play AVI files then you don't need to do this as it's already installed.
