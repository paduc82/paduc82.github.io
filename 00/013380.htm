rohn007 say:

Those settings are optional (that is why they are in options). The idea is to either totally block, or add an extra step when opening older / "risky" file types so that the user does not accidentally open infected files. 

For example, since the DOC file type allows macros, it is a favorite "macro virus carrier" for hackers. So, if you know you do not use DOC files and do not normally share DOC files with customers, you can block them. That way if you are sent an infected file you do not accidentally "automatically" open it, infecting your machine.

Over the years, MS has blocked more file types by default.

You can turn them on simply by clicking in the check boxes. First you have to decide what the check marks will do. That is set by the 3 radio buttons at the bottom of the dialog:

  • do not allow the checked file types to open at all
  • Open in "protected view", do not execute macros, do not edit
  • Open in "Protected view" allow editing

Protected View displays a yellow warning bar at the top of the editing window

Here is more detailed info about protected view from MS

https://docs.microsoft.com/en-us/previous-versions/office/office-2013-resource-kit/ee857087(v=office.15)

ken sheridan say: If you'll excuse me briefly breaching your closure edict, not everything is documented, and it would be unduly optimistic to expect it to be so.  There will always be syntactical eccentricities which fall through the holes in the bucket.  There are other more important ones of which you won't find any mention.  The Help topic for the ValidationRule property specifies that it must be an expression which returns a Boolean value.  I would regard a function call which does this as an expression, though all the examples given in Help do involve two operands.  These things usually come out in the wash in one way or another, and in most cases a reasonably intelligent developer will spot what needs doing, as you have done in this case, and by doing so have alerted others who might come across the same issue.

What it does reinforce of course is the need to thoroughly test every small detail of the design.

One should not compartmentalise the various technologies too much.  When you call a VBA function in a context like this you are of course using VBA, merely not in the context of a procedure or function.  The same is true if you call a VBA function in a query.  Conversely if you build an SQL statement in VBA and establish a recordset, you are using SQL.  You are also using DAO or ADO of course.  There is a great deal of interconnection between the technologies, but you won't find every last detail documented.
