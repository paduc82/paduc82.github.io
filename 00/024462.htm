jay freedman say:
I assume that the list box and text box you have are "legacy form fields" (from the Legacy Tools button on the Developer tab) and not content controls with the similar functions.

Add the following macro to the form's template (see http://www.gmayor.com/installing_macro.htm). Then go into the Properties dialog of the text box and select the AddNewRow macro as the Exit macro for the box.

Sub AddNewRow()
Dim rg As Range
With ActiveDocument
    If .ProtectionType <> wdNoProtection Then
       .Unprotect
    End If
   
    Set rg = .Tables(1).Range
    rg.Collapse wdCollapseEnd
    ActiveDocument.AttachedTemplate.AutoTextEntries("newrow").Insert _
        Where:=rg, RichText:=True

    .Protect Type:=wdAllowOnlyFormFields, NoReset:=True
End With
End Sub

Finally, select that one table row, press Alt+F3, and assign the following values:

Name: newrow
Gallery: AutoText
Category: (first click Create New Category, then enter) FormRow
Description: Table row with form fields
Save in: (select the name of the form's template)
Options: Insert content only

Click OK. Protect the form's template and save it.

When the user creates a new document based on this template, and then tabs or clicks out of the text box, a copy of the AutoText is added to the end of the table.

jay freedman say:
That behavior indicates that when you first created the row, before you pressed Alt+F3 to make it into an AutoText entry, the row (or the table it was in, if there were other rows) had a text wrapping of Around instead of In Line with Text. You can't work with that, because the new row will never connect to the existing table.

Discard the document you're working with, and open the template (use File > Open instead of File > New). Turn off forms protection. Click in the table in the template's body. On the Table Tools | Layout tab, click the Properties button. Under the "Text wrapping" label in the dialog, which of the two buttons is selected, None or Around?

If it's Around, click None and then click OK. The table will now behave the same way as a picture that has In Line with Text wrapping, that is, like a paragraph of text. You can reposition it vertically by adding Space After to the paragraph before the table, and horizontally by clicking the Center or Right button or setting the Indent from Left measurement in the Table Properties dialog.

Then select the last row of the table, press Alt+F3, and put in the same information as I gave you before. When you click OK in that dialog, you should see a prompt asking whether to redefine the building block; say Yes.

Turn on forms protection again, and save the template (which saves both the new wrapping of the table and the redefined AutoText entry). Then try a new document based on the template.

don cuthbert say:

Yes, I found out that DISM .... /source:wim:... will fail if the install.wim is too compressed. What you can do is export a properly compressed install.wim from it:

DISM /Export-Image /SourceImageFile:install.wim /SourceIndex:1 /DestinationImageFile:compressmax.wim /Compress:max /CheckIntegrity

A /Compress:max WIM file will work as a source for DISM repair. You can name it whatever you want. Add in whatever path you need for the source and destination files. Then sfc /scannow should succeed.

EDIT: This presumes that your Windows 10 disk is the same version as Windows on the computer. You can download the latest public build here: https://www.microsoft.com/en-us/software-download/windows10ISO/
