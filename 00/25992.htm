joeu2004 say:

How to use correlation coefficient in Monte Carlo simulation?

I have always been curious about how to use the correlation coefficient in the compuations of a Monte Carlo simulation.  Can anyone enlighten me by embellishing the following example?

Suppose that F1 and F2 contain the mean and std dev of some normally-distributed random variable.  And suppose I choose to use NORMINV(RAND(),$F$1,$F$2) in F3:F2500 to compute random values [1].

Likewise, suppose I choose to use NORMINV(RAND(),$H$1,$H$2) in H3:H2500 to compute random values for a second normally-distributed random variable with mean and std dev in H1 and H2.

So far, so good.  Now suppose that CORREL(A3:A2500,C3:C2500) is the correlation coefficient for the two random variables based on historical data in A3:A2500 and C3:C2500.

Simply using the two NORMINV formulas does not "ensure" [3] that the simulated data in F3:F2500 and H3:H2500 has a "similar" correlation coefficient.  In my example [4], I purposely chose historical data in A3:A2500 and C3:C2500 so there would be a strongly negative correlation, about -0.6336.  Yet I was able to generate random simulated data in F3:F2500 and H3:H2500 with a (weak) positive correlation [5], about +0.0516.

So how do I incorporate the historical correlation coefficient into the NORMINV formulas [2] to compute simulated random data with a "similar" correlation coefficient [3]?


-----
[1] Disclaimer:  Some people argue that NORMINV(RAND(),...) is a poor way to generate a random normal distribution.  So be it!

[2] If it makes it is easier to incorporate the CORREL value into the random value formulas, note that NORMINV(RAND(),$F$1,$F$2) is equivalent to $F$1+$F$2*NORMSINV(RAND()).  Or use whatever other random value algorithm in Excel or VBA that you prefer that is based on the mean, std dev and correl coeff of two normally-distributed random variables.

[3] We cannot truly ensure that the simulated data has a correlation coefficient that is "similar" to the historical data.  But I expect there is some large probability if we incorporate the historical correlation coefficient appropriately.

[4] If you're interested in working with my example, download "sim correl.xls" from http://www.box.net/shared/k19bcam99pq8zirnkf18.

[5] In fact, I was able to generate many examples with very weak negative and positive correlations; i.e. close to zero on one side or the other.  But in all my many trial simulations, the simulation data correlation coefficient is never close to the historical correlation coefficient.

brian tillman [outlook mvp 2007-2019] say:

Are you looking in the address book interface by clicking To, then entering a name in the Search field or are you simply typing in the recipient field? If the latter, that doesn't reference your contacts in any way. It references a cache of addresses to which you have previously sent mail, whether they are contacts or not. This cache will contain name/address pairs, where the names are whatever showed in the To line when you sent a message to it the last time on a new message or a reply. When you start typing, Outlook will show you names from the cache whose display names or addresses start with the characters you've typed.

Suppose your cache contains entries like this:

Thomas, Dylan <Dylan.Thomas@someaddress.com>

You will see this entry if you start typing either "Tho" or "Dyl", because the display name starts with the first string, while the address starts with the second string. If the entry looked like this however:

Dylan Thomas <Dylan.Thomas@someaddress.com>

then only "Dyl" will find the address. "Tho" won't work because neither field starts with that string.

Of course, if you are using the Address Book interface, then what I've said doesn't apply.

A solution to the "Your network access was interrupted. To - Microsoft Community

scott 12345 say:

A solution to the "Your network access was interrupted. To continue close the database and then open it" error

Today I encountered the vague "Your network access was interrupted. To continue close the database..." error when trying to open an access mde using sql linked tables using the runtime client. This happened after migrating a user profile from one machine to another using the Transwiz tool. I did some reading and the error seems to have many causes, the solution for my case was not documented anywhere so I wanted to share it. 

In c:\users\username\appdata\roaming\microsoft\access there was a file named system. I renamed the file to _bad and relaunched access. Problem resolved. I'm guessing the file contained configuration information which pointed to somewhere on the old machine and needed to be refreshed after moving the profile.

This will not be everyone's solution because it is a vague error but if you're struggling with this error, it is worth a try.

charles kenyon say:

jay freedman say:

If you're using the Columns command on the Page Layout ribbon, that's the wrong tool for the layout you want.

One way to do this is to use a borderless table. The heading goes in the cell on the left, and the text goes in the cell on the right. Each time you need a new heading, start a new row of the table. (Because Word sometimes has difficulty with table rows that continue for many pages, you should start a new row whenever the text runs more than a page or so; just leave the left cell empty at that point.) This method is described in http://wordfaqs.ssbarnhill.com/UsingColumns.htm.

Another way is to create a wide left margin and then put the headings into frames that position in that margin. The frame must be anchored to the first paragraph of the text to which it applies. For how to set this up, see http://wordfaqs.ssbarnhill.com/MarginalText.htm#TrueMarginalText.
