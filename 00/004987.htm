Add this macro to your Normal.dotm template (http://www.gmayor.com/installing_macro.htm ):

Sub AutoOpen()
    Dim strTitle As String
    strTitle = ActiveDocument.BuiltInDocumentProperties(wdPropertyTitle)
    If Len(strTitle) > 0 Then
        ActiveWindow.Caption = strTitle
    Else
        ActiveWindow.Caption = ActiveDocument.FullName
    End If
End Sub

laurie_g say:

How to edit the "Last modified by" info

Where in Office do I set the information that appears in the backstage view of my document? I'd like to edit the "Last modified by" user so that it reads my company name instead of the user name. If I choose the option to remove personal data, it also removes the Author name, which I want to keep.

greg maxey say:

laurie_g,

The data is stored in the Word OpenOfficeXMLFormat file in the docProps\core.xml file.  You can add an .zip extension to the file name and then unzip the file to expose the various parts.  Here is an example of that file where I changed the Last Modified By information:

<?xml version="1.0" encoding="UTF-8" standalone="yes"?>

-<cp:coreProperties xmlns:cp="http://schemas.openxmlformats.org/package/2006/metadata/core-properties" xmlns:dc="http://purl.org/dc/elements/1.1/" xmlns:dcterms="http://purl.org/dc/terms/" xmlns:dcmitype="http://purl.org/dc/dcmitype/" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance">
 <dc:title/>
 <dc:subject/>
 <dc:creator>Gregory K. Maxey</dc:creator>
 <cp:keywords/>
 <dc:description/>
 <cp:lastModifiedBy>Joe Miller</cp:lastModifiedBy>
 <cp:revision>2</cp:revision>
 <cp:lastPrinted>2010-07-01T08:55:00Z</cp:lastPrinted>
 <dcterms:created xsi:type="dcterms:W3CDTF">2010-07-10T13:48:00Z</dcterms:created>
 <dcterms:modified xsi:type="dcterms:W3CDTF">2010-07-10T13:48:00Z</dcterms:modified>
 </cp:coreProperties>
After making the change you can repackage the file and the change appears in the Backstage view.
It seems that there should be a way to make this change using the CustomXMLParts object.  I tried, but I simply don't know enough about XML to figure it out.
Hope someone comes along that does.
Cheers.

ken sheridan say: Splitting the database should not have made any difference.  The conventional model for this type of application would be like this:

Customers----<Orders----<OrderDetails>----Products

i.e., the OrderDetails table models the many-to-many relationship type between Orders and Products by resolving it into two one-to-many relationship types.  OrderDetails would therefore contain two foreign key columns such as OrderID and ProductID, referencing the primary keys of Orders and products respectively.

The usual interface for data entry would be an Orders form in single form view, and embedded within it an OrderDetails subform linked to the parent form on OrderID.

From your description my initial guess would be that the relationship between Orders and OrderDetails is one-to-one, i.e. the foreign key OrderID column in Order Details is either designated as its primary key or is otherwise uniquely indexed.

For an example of this sort of model take a look at InvoicePDF.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

This little demo file is really intended to illustrate how to output a report (invoice in this case) as a PDF file, but it happens to be based on a simple invoicing system so is modelled in the same way.  Take a look at the relationships window to see how the model fits together.

The main invoice form in this case includes a number of subforms as it also caters for VAT calculation and partial payments against an invoice, but the main form and the invoice details subform operate in exactly the same way as would be the case with an orders form/order details subform.
