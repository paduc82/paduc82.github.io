Autosave only really works when and if PPT actually detects that it's about to crash ... typically, the kind of situation where it pops up a message telling you that it's run into an irrecoverable error or the like.

If it simply disappears on you, it didn't catch itself before it fell, so won't offer to recover any autosaved files.

You might want to look at PowerPoint Options | Save to see what the autosave folder is set to, then look there for any files with last-accessed dates around the time of your crash.

Best bet, though, is not to trust it as anything but a sometime last-resort kind of feature.

john w. vinson say:

This is NOT default behavior. Setting focus to the subform ordinarily leaves the mainform unperturbed. The Requery event is the cause of the problem here - it doesn't "requery fields", it actually reruns the main form's Recordsource query (and since it's a brand new query, as far as Access is concerned, positions at the first record).

 

What's the purpose of the requery? Perhaps a less intrusive Recalc would work? Feel free to post the code you're using and explain what you intend to accomplish.

michael gallop say: Hi John,

Thank you for your quick response.

The code for the main form's AfterInsert event is to requery a couple of comboboxes on the main form - not to requery the form itself.  The code is basically [ComboBox].Requery.

Does moving from the main form to the sub-form trigger the main form's AfterInsert event?

Thank you,

Michael

john w. vinson say:

Yes: the mainform record is saved at the instant you set focus to the subform (it has to be, in order for the Master Link Field to have a defined value). This triggers the form's Before and AfterUpdate events (and AfterInsert, if it's on the new record).

 

I don't see why requerying the combo box would requery (and reorder) the form! What happens if you comment out the requery lines? Would you be willing to post the actual code? Maybe someone will see what's going on.
