This little demo file includes, amongst others, the following two functions which are amendments of the original CanShrinkLines function published by Microsoft.  The first, ConcatValues, returns a continuous string in which any separator character(s) can be used between the values; the second, CanShrinkLines separates each value with a carriage return/line feed.  Both suppress Nulls, zero-length strings or a string composed entirely of spaces.

Public Function ConcatValues(strSeparator As String, ParamArray arrVals())
 
    ' Pass this function a character or characters
    ' to be used as the separator followed by the values to be combined
    ' For example: strFullName =
    ' ConcatValues(" ",FirstName,MiddleName,LastName)
    
    Dim X As Integer, strLine As String
    
    For X = 0 To UBound(arrVals)
        If Not IsNull(arrVals(X)) And Trim(arrVals(X)) <> "" Then
          strLine = strLine & strSeparator & arrVals(X)
        End If
    Next
      
    ' remove leading separator character(s)
    ConcatValues = Mid(strLine, Len(strSeparator) + 1)
 
End Function

Public Function CanShrinkLines(ParamArray arrLines())

    ' Pass this function the lines to be combined
    ' For example: strAddress =
    ' CanShrinkLines(Name, Address1, Address2, City, State, Zip)
    Dim X As Integer, strLine As String
    
    For X = 0 To UBound(arrLines)
        If Not IsNull(arrLines(X)) And Trim(arrLines(X)) <> "" Then
          strLine = strLine & vbCrLf & arrLines(X)
        End If
    Next
      
    ' remove leading carriage return/line feed
    CanShrinkLines = Mid(strLine, 3)
 
End Function

john w. vinson say:

There's actually a sneaky trick using a quirk of Access: both the & and + operators concatenate strings, but they treat NULL differently. X & Y will return just the value in X if Y is null; but X+Y will return NULL if either argument is NULL. 

So to get the six address lines concatenated with blanks or new-line characters between them, just use parentheses appropriately:

SELECT ([Address Line 1]) & {Chr(13) + Chr(10) + [Address Line 2]) & {Chr(13) + Chr(10) + [Address Line 3]) & {Chr(13) + Chr(10) + [Address Line 4]) & {Chr(13) + Chr(10) + [Address Line 5]) & {Chr(13) + Chr(10) + [Address Line 6])

Use " " in place of Chr(13) + Chr(10) if you want a blank rather than a newline.

ken sheridan say:

Can I do all 6 address line fields in the same query? If so how? I can get it to work for 1 field but not multiple fields

Microsoft publishes the CanShrinkLines function, which I posted earlier, exactly for this situation.  It concatenates the values with carriage returns/line feeds and suppresses Nulls.  You can see how it works in the debug window:

Addr1 = "42 Gaolgate Street"
Addr2 = Null
Addr3 = "Stafford"
Addr4 = ""
Addr5 = "Staffordshire"

? CanShrinkLines(Addr1,Addr2,Addr3, Addr4,Addr5)
42 Gaolgate Street
Stafford
Staffordshire

 Note that the Null Addr2 and the zero-length string Addr4 have been suppressed.

My variation of the above, ConcatValues, allows you to use the separator character(s) of your choice, e.g. a colon followed by a space:

? ConcatValues(": ",Addr1,Addr2,Addr3, Addr4,Addr5)
42 Gaolgate Street: Stafford: Staffordshire

Perhaps the most important thing to note about these functions is that they do not accept a separate argument for each address field as in Scott's example, but a parameter array (by declaring the argument with the ParamArray keyword).  This means you can pass any number of fields into the argument, e.g. if we omit Addr5:

? CanShrinkLines(Addr1,Addr2,Addr3, Addr4)
42 Gaolgate Street
Stafford

Hi Shane,

The more accurate answer should be the Purges subfolder in the Recoverable Items folder.

The whole process is as follows:
1. When emails are deleted in the deleted items folder, they will go to the Recoverable Items folder.
2. When a user purges an item in the Recoverable Items folder or when the deleted item retention period expires for an item, it's moved to the Purges subfolder in the Recoverable Items folder and marked for permanent deletion. It will be purged from Exchange the next time the mailbox is processed by the Managed Folder Assistant (MFA).

But when a mailbox is placed on Litigation Hold, items in the Purges subfolder are preserved for the hold duration specified by the Litigation Hold. If an indefinite hold is placed on a mailbox, items will never be purged from the Purges subfolder.  So that's the answer.

Let us know if anything is unclear.

graham mayor say:

Normal.dotm is your default template for Word and it should never be read-only, however you will see the quoted error message if there is an unwanted lock file for that file in the folder, orphaned by a Word crash.

It can also occur when a hidden Word version has stalled, which rebooting should address.

Basically, the lock file prevents anyone else from saving to the normal template when it is in use.

Word 2016 is better than earlier versions at handling such an issue, but it is not infallible. You need to close Word then check from the Windows Task manager (CTRL+SHIFT+ESC) and see if WinWord.exe is still running. If it is, close it from the dialog.

Then see http://www.gmayor.com/what_to_do_when_word_crashes.htm  with respect to orphaned lock files and delete any that you find - especially the one relating to the normal template - '~$Normal.dotm'

When reviewer names change to "Author" as you reopen a document, someone has run the Document Inspector on the document and selected the option to remove personal information.
