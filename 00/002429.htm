ken sheridan say: I've taken a look at your file.  The first thing you need to do is correct the relationships.  At present you are relating tables on non-key columns (fields).  Tables should be related on the primary key from one table and the foreign key from another.  I've made an amended copy of you file which does this, but before doing so I changed the primary keys to more meaningful names.  Calling the primary key of every table ID, as you had done, will soon lead to confusion.

The next thing is relating charts and materials.  This is a many-to-many relationship type, and you'd quite correctly created a table Chart Materials to model the relationship, but you'd not got it quite right.  The way such a table models a many-to-many relationship type is by having foreign key columns which reference the primary keys of the other two tables, ChartID and MaterialID in this case.  The table's primary key is a composite one made up of both of these columns.

You also had a lot of redundancy in your tables, e.g. repeating the Description column in several tables.  You only need this once as you it is known from the relationships between the tables, and can be returned in a query which joins the appropriate tables.  If you look at my amended version of your file you'll see it is now only in the Material Type table.  Eliminating redundancy like this is important as it prevents update anomalies, i.e. you can't have the same data entered twice as inconsistent values.  This happens - in one database I found three slightly different versions of my own name as author of technical papers in my own field of work!

You'll get the best picture of the way I've amended the file by looking at the relationships window.  You'll see that the relationships are now all on-to-many and referential integrity is enforced in each case, which protects the integrity of the data by ensuring that a row can only be inserted into a referencing (many side) table if a matching row already exists in the referenced (one side) table.  Also a row cannot be deleted from the referenced table if at least one matching row exists in the referencing table.

The user will be able to select the designer from the combo box's drop down list, but the value of the control and therefore of the field to which it's bound will be the hidden numeric value.  The relevant rows must have been inserted into the Designer table first of course.  We can come back to how you'd add a new designer directly from the charts form later.

The controls for the other foreign key columns would be set up in much the same way.

ken sheridan say:
Where do we go from here?

The next step is to create the subform in which to insert rows into the Chart Materials table, but before that you need to create a query which will be the subform's RecordSource.  For this you join the Chart Materials table to the materials table on the MaterialID columns, and the Materials table to the material Type table on the MatrialTypeID columns.

Add the following fields to the query:  The ChartID,MaterialID and Qty fields from Chart Materials; the
Code and Reorder fields from Materials; and the Material Type and Description fields from material Type.  In SQL view the query will be:

SELECT [Chart Materials].ChartID, [Chart Materials].MaterialID, [Chart Materials].Qty,
[Chart Materials].Needed, Materials.Code, Materials.Reorder,
[Material Type].MaterialType, [Material Type].Description
FROM [Material Type] INNER JOIN (Materials INNER JOIN [Chart Materials]
ON Materials.MaterialID = [Chart Materials].MaterialID)
ON [Material Type].MaterialTypeID = Materials.MaterialTypeID;

Save the query as qryChartMaterials

You can now create the form to be used as the subform in your charts form.  So, create a form in continuous forms view with the above query as its RowSource.  Design the form so that it is one row deep with all the bound controls side by side in the details section, and the label for each control above it in the form header section.  Add controls for each of the fields from the query apart from the ChartID field, which you don't need to show in the subform.  It is in the query merely to provide the link to the charts form.  Use text box controls for each of the fields apart from the MaterialID control for which you will use a combo box set up much as you did with those in the chart form, but in this case showing multiple columns in its drop down list, i.e.

Name:                   cboMaterialID

ControlSource:    MaterialID

RowSource:        SELECT Materials.MaterialID, Materials.Code,
                              [Material Type].MaterialType, [Material Type].Description
                              FROM [Material Type] INNER JOIN Materials
                              ON [Material Type].MaterialTypeID = Materials.MaterialTypeID
                              ORDER BY Materials.Code;

BoundColumn:    1
ColumnCount:     4
ColumnWidths:    0cm;2cm;3cm;3cm
ListWidth:            8cm

Access will convert the dimensions to inches if you are not using metric units.  Experiment with the ColumnWidths dimensions  to get the best fit.  The ListWidth should be the sum of the column widths.  You'll find it easier to do this if you have already entered some data into the Materials and Material Type tables, so that you will see values when you test the control.

The combo box's drop down list will show the code, material type and description columns.  When you select a row from the list the code will show in the combo box (so set its label's caption property to Code) and the other values in the material type and description text boxes in the form.  The reorder value will also show in its control.  So that the user does not inadvertently try to change the values in these last three controls set their Locked property to True (Yes) and their Enabled property to False (No).  The qty and needed values will be manually entered by the user of course. 

You won't be able to save any records in the form at this stage, but you can test it by selecting an item in the combo box and making sure the correct reorder, material type and description values show in the other controls.  Once you are happy that its working correctly save it as fSubChartMaterials.

You can now embed the new form as a subform in the charts form.  To do this open the charts form in design view and drag the fSubChartMaterials form from the navigation panel with the mouse and drop it into the charts form below the controls you put in when you designed the charts form.  Size and position the subform to suit, making it wide enough to accommodate all the columns and deep enough to accommodate a reasonable number of rows.  If a chart includes more materials than the number of rows the subform shows you can scroll down within the subform.

If you have set up the relationships between the tables as in my amended version of your original file you should find that the subform's LinkMasterFields and LinkChildFields properties are both ChartID.  If not set each property to that.  When you open the charts form you will be able to enter as many materials as necessary for the current chart record by entering a new row for each in to the subform.

That will give you the basic set-up for entering data.  There are various refinements you can add, particularly to enable you to enter insert rows into the Materials and Materials Type tables directly from the subform as necessary.  We can come back to this later once you have the basic form/subform working correctly.
