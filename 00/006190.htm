jeeped say:
... I only need the outputted data in simple text format. I will be using it for simple search engine searches ...

If you want one-size-fits-all you will have to use a UDF, but this should serve your purposes.

Note: I hate using quotes within quoted strings, so I am using CHAR(34) to represent a " mark.

     

The formula in B1 is,
=CONCATENATE(CHAR(34),A1,CHAR(34))
... and the formula in B2 is,
=IF(LEN(A2),CONCATENATE(B1," or ",CHAR(34),A2,CHAR(34)),A2)
... filled down as far as necessary to catch the entries in column A.

Some UDF code might look, like,

Public Function fcnMakeSearchString(rSearches As Range) As Variant
    For Each s In rSearches
        If Len(s.Text) Then _
            sSearchString = sSearchString & Chr(34) & s.Text & Chr(34) & " or "
    Next s
    fcnMakeSearchString = Left(sSearchString, Len(sSearchString) - 4)
End Function

Tap Alt+F11 then immediately use the pull-down menus to Insert, Module and paste the above into the new pane titled something like Book1 - Module1 (Code). Tap Alt+Q when done to return to your worksheet. Use as above like,
=fcnMakeSearchString(A1:A3)

joeu2004 say:

It is difficult to comment without a concrete example:  the values, dates and formula that you use.

A mix of positive and negative values should not cause a #VALUE error, as long as the corresponding dates are also in the correct form.  That is true, even if the values and dates are text (!).  Note that the format of the cells does not matter.

However, they might cause a #NUM or #DIV/0 error, or XIRR might return +/-2.98E-09.  Sometimes, that can be corrected by providing a "guess" in the third parameter.

The only time that I see a #VALUE error from XIRR is if Excel cannot interpret the text as a number or date.

[EDIT] XIRR even allows dates in A2:A10 to be earlier than A1 (!), for example.  A1 is still treated as the PV date (i.e, d1).

joeu2004 say:

PS.... I wrote:

The only time that I see a #VALUE error from XIRR is if Excel cannot interpret the text as a number or date.

Do not rely on visual inspection or the cell format in order to determine if Excel should "interpret the text as a number or date".

Instead, use formulas of the form =ISNUMBER(--A1).  If that function returns FALSE, the cell is not numeric (or empty), and Excel cannot interpret the text as a number of date.

If the data was copy-and-pasted from a web page, there might be "invisible" non-breaking spaces (HTML nbsp; ASCII 160).  Confirm with formulas of the form =ISNUMBER(SEARCH(CHAR(160),A1)).
