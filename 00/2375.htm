A DL/ContactGroup is nothing more than a special "contact item" that resides in a contact folder - no more - no less. A difference between a DL in a contact folder and a regular contact item is the item's "messageclass" - Contact item = IPM.Contact (for a standard contact not using a custom form) and DL  IPM.DistList

DL items are just as much part of a contact folder in a PST file as is any other contact item so if it doesn't appear in your old PST file - then it was deleted prior to your backup or there are other issues at play

<IF> the DL was not permanently deleted altogether (in which case everything that follows is completely moot) - then the following are some last chance items to check/do

#1 - The DL was moved another folder - not necessarily a contact folder so you'll need to confirm that

#2 - You can check your contact folders for all items with a message class of IPM.DistList (select one of the list type" views - i.e. phone etc - and customize it by adding the MessageClass column to it and then group by MC just to see what exists

karl timmermans [outlook mvp 2012-2018] say:

The second Contacts folder has a few less contacts in it than my "default." Think I can just delete it? Don't know why I would have two.

Without knowing the actual details of your config in terms of configured email accounts and/or how you came to have 2 PST files - would never suggest deleting anything.

Having said that - what you've described is that you have 2 PST files that are (or at one point <have been> used as the default for a configured account. At a minimum, you have one account tied to your "default" folders. Once a PST file has been designated as a "default" - none of the <default> folders that get created can be deleted (option is greyed out). In your scenario, you either have 2 different email accounts OR you've opened the 2nd PST file from a previous install etc.

To make a long story short, if you only have one email account then you can either copy any data from the 2nd PST file to your primary and then close it (essentially removing it) or just remove it altogether. Comes down to your config and requirements.

We successfully used the following procedure, in Word 2003. 

 NOTE:  All documents must be in the one folder, together with the TOC document. Also,make sure you have hidden text displayed (Tools=>Options=>View.)

  •  Go to Insert => Reference =>Index and Tables. Select Table of Contents.
  • Result: Error message: "No table of contents entries found". This is correct. Now you will supply the entries.
  •  Insert fields for each document
  1. Ctrl+9 to insert field brackets.
  2. Type RD document name.doc between the brackets

  • Close the TOC doc.
  • Open one of the files referenced, save as itself, close.
  • Open the TOC, and try again.

Using a query to truncate a text field - Microsoft Community

ronaldcheshier say:

Using a query to truncate a text field

I have a text field called EmployeeName which contains the names of people that work as interpreters in a non-profit I do volunteer work for.  Typically the names are straight forward like Larry Jones or Mike Hamlin but some of the names specify the employee proficiency level by containing the phrase "_Level 1" after then name, like Mike Hamlin_Level 1". I would like to have a query that cuts off the "_Level 1" part of the text but have been unable to find a query statement that works. The query should, in effect, cut off 8 characters starting from the right so that the names (which have varying lengths) remainin tact.  Any suggestions?

Thanks in advance.

Ronald C. Cheshier

john w. vinson say:

Do you want to permanently change the name in the table to just the name (which I'd recommend), or leave the Larry Jones_Level 1 alone but just have it display as Larry Jones? You can do either; the first would use an Update Query and the second just a standard (SELECT) query.

In either case, the trick is to use the InStr() function to find the position of the _Level. For instance:

Left([yournamefield], InStr([yournamefield], "_Level")-1)

would find the position of the text string _Level and return the  characters up to that point.

This is an example of why good database design insists that fields should be "atomic" - containing only one "fact". You should have separate fields for the name and the proficiency level. In fact, typically one would have fields for FirstName, LastName, and perhaps MiddleName, Title and Suffix, so you could have "Dr. Helen Marie Jacobson" and be able to address her as "Dr. Jacobson" or as "Dear Helen", as appropriate.
