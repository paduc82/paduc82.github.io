    Remove Word From Dictionary - Microsoft Community

    johannswart say:

    Remove Word From Dictionary

    Is it possible to remove words from the normal dictionary with specified spellings?

    For example, in our organisation, the word "feedwater" is written one word; not two words; not hyphenated. And believe me, we have are many many more such words.

    The problem now is, the spell checker does not highlight the other versions up as a "misspellt" word.

    Alternatively, can the custom dictionary be employed to show up "misspellt" words?

    jay freedman say:
    Unfortunately, the contextual speller feature won't help with this task. Its job is to warn users when they've typed the wrong word from a set of homonyms: "Their it is" instead of "There it is", or "Put it hear" instead of "Put it here". The words are correctly spelled but incorrect for the context. Also, the list of words the feature recognizes is not customizable.

    The ordinary spelling feature, and its exclude dictionary, are limited to recognizing single words without spaces, hyphens, or other punctuation. That's why Suzanne said "feedwater" vs. "feed-water" might not be successful.

    An approach that is likely to be successful is a macro add-in provided by Greg Maxey at http://gregmaxey.mvps.org/VBA_Find_And_Replace.htm. With that add-in, you can create a document containing a two-column table, with the word or phrase to find in the left column and the word or phrase to use as the replacement in the right column. (There's an example on Greg's page as a sample Quick List, but the format for a "predefined Word list" is the same.) I've verified that you can use multiple-word phrases and hyphenated terms in either column.

    - The GAPI issue prevents games or emulators to display correctly, believing they are filling a 176 pixels wide screen.

    [quote]Game developers using GAPI - The Game API - (directly or through other libraries) may have problems on the Motorola Q.
    If you call the GAPI routines on the Motorola Q, they will tell you that:
    1) the screen size is width=176 and height=220 (in fact the display on the Motorola Q is Landscape with width=320 and height=240).
    2) GAPI is using a DRAM buffer, i.e. you cannot write directly in the raw frame buffer with GAPI, rather GAPI GXBeginDraw will return you the address of a 176x220 memory buffer, which will be copied in the real display frame buffer when you call GXEndDraw

    ....
    It's very similar to what HP did with GAPI on the iPAQ 3800, and to what Microsoft did with VGA-resolution Pocket PC's, and the result is the same: they made a mistake, and their bad decision is just making it harder on developers, not simpler, as they though.[/quote]

    Copy a combo box across various columns - Microsoft Community

    ericscott5 say:

    Copy a combo box across various columns

    I have created a combo box list in cell A4  using ActiveX Controls. Is there a VBA code to create the same combo box in cells B4:P4?

    ossiemac say:

    Try the following.

    Sub CreateComboBoxes()
        Dim ws As Worksheet
        Dim oleObjCombo As OLEObject
        Dim c As Long
        Dim lngFirstCol As Long
        Dim lngLastCol As Long
        Dim dblTop As Double
        Dim dblLeft As Double
        Dim dblHeight As Double
        Dim dblWidth As Double
        Dim strListFillRng As String
       
        Set ws = Worksheets("Sheet1")       'Edit "Sheet1" to your sheet name
       
        With ws
            lngFirstCol = .Columns("B").Column
            lngLastCol = .Columns("P").Column
           
            'Save dimensions of original ComboBox
            With .OLEObjects("ComboBox1")         'Edit ComboBox1 to your original ComboBox name
                dblTop = .Top
                dblHeight = .Height
                dblWidth = .Width
                strListFillRng = .ListFillRange
            End With
           
            For c = lngFirstCol To lngLastCol
                'Get left position of column where combo to be created
                dblLeft = Columns(c).Left + 1   'Add 1 to ensure Control is created with left inside the column
               
                Set oleObjCombo = .OLEObjects.Add(ClassType:="Forms.ComboBox.1", _
                                                Link:=False, _
                                                DisplayAsIcon:=False, _
                                                Left:=dblLeft, _
                                                Top:=dblTop, _
                                                Width:=dblWidth, _
                                                Height:=dblHeight)
                                               
                'Optional code to rename the ComboBox (Each name must be different)
                'and add a ListFillRange. Other properties can also be set here.
                With oleObjCombo
                    .Name = "MyComboName" & c
                    .ListFillRange = strListFillRng      'Same as original ComboBox
                    '.ListFillRange = "Sheet2!$A$1:$A$10"   'Alternatively directly apply a range
                End With
            Next c
        End With
           
    End Sub

    Sub UpdateCheckBoxes()

    consolidate data on report - Microsoft Community

    bethp4150 say:

    consolidate data on report

    I have multiple users from different departments entering various pieces of information that all have to do with one job.  This results in multiple records in my table.  I want all of those records to combine into one summary line on my report that includes all information about this job.  Basically I want to rolls all of the records up into one.  I presume I do this with a query, but don't know where to start.  Here is detail section of report.

     

    Here is query that report pulls from.

    ken sheridan say:

    Basically I want to rolls all of the records up into one.


    You would do so by joining multiple instances of the table on whatever distinctly identifies the rows in the table which you wish to consolidate into a single row in the query's result table.  I'm guessing from the fact that it is shown in boldface in your report that this is RunName.

    However, the fact that you have to do this points to an incorrect table design.  Users should not be inserting different rows into the table for the same entity, but editing a single row following its insertion by whoever first initiates the entity, presumably the Run, within the database.  If, when currently inserting multiple rows for the same entity, different users can enter different values into the same column, then the table would need to be decomposed into two or more related tables in which multiple rows would be entered in relation to the same run.  Without a more detailed knowledge of the process being modelled I can't be more precise on that point, however.

    A word or two on terminology so as to avoid any confusion as to what is being referred to here:

    1.  An entity is what each row in a table represents, and is an instance of the entity type modelled by the table.

    2.  A row equates to a record.

    3.  A column equates to a field.

    bethp4150 say:

    Thank you, I was afraid of that.  I have multiple users in our production facility and each contributes various pieces of information regarding a particular press run.  For example.  The plate room folks are entering data onto a form that updates the table (what time they received a file, and what time they output the last plate) for a press run.  The press folks are entering data onto a form that updates the same table (press start time, press stop time, issues with the run).  Two other departments are doing the same thing.  Each individual department's entry is a separate record in the table, creating multiple records on the report.  Obviously I want all information to end up in one record.  How do I set up the tables so that happens?

    ken sheridan say: Firstly you need to prevent duplicate rows being inserted for each press run by indexing the RunName column uniquely (no duplicates) in the table design.  You won't be able to do this with the current table, however, as it already contains duplicates so make an empty copy of the table and do so in that.  You can then run an append query which inserts one user's set of values into the new table.  You'll need to work out a way to identify the rows from the table for this, as what it amounts to is inserting the rows originally entered by one user.  This should give you one row in the table per run.  If when executing the append query you get a message saying it can't append some rows, then it means you haven't identified a distinct subset of rows from the original table, and you'll have to think again how to do so.

    You will then need to insert the other users' sets of values into the new table by executing a series of update queries in each of which another distinct subset of rows for one user is selected and used to update the relevant columns in the new table.  The old and new tables should be joined on the RunName column in this query.  Before you execute each update query open it as a datasheet and check that you have got a distinct set of runs returned.  If it does return any run more than once you'll lose data when its executed.

    If all goes well you should end up with all runs represented by a single row each in the new table, with the data from the separate users consolidated into each row.  You can then rename the original table, e.g. by giving it a _bak suffix, and then rename the new table to the original name.  This will now be your operational table both for data entry, and as the basis for reports.

    For data entry, you must use a form, and not enter data directly into the table's datasheet.  In the form add an unbound combo box, cboRunName say, in the form header with RowSource property like this:

    SELECT RunName, 1 AS SortColumn
    FROM [YourTableNameGoesHere]
    UNION
    SELECT "<New Run>", 0
    FROM [YourTableNameGoesHere]
    ORDER BY SortColumn, RunName;

    In its AfterUpdate event procedure put the following code:

    Dim ctrl As Control

    Set ctrl = Me.ActiveControl

    If Not IsNull(ctrl) Then
        If ctrl = "<New Run>" Then
            DoCmd.GotoRecord acForm, Me.Name, acNewRec
        Else
            Me.Filter = "RunName = """ & ctrl & """"
            Me.FilterOn = True
        End If
    Else
        Me.FilterOn = False
    End If

    If the row for a run has not yet been created it will not be listed so the user can select <New Run> and enter a new record.  Otherwise they can select a run from the list and the form will be filtered to that run, in which case they can add data to the existing record.  If they delete the current selection in the control the filter will be removed and all records displayed.
