It has never been possible to do this in Word. When started, Word always defaults to the Windows default printer and the settings selected in the printer Properties, and it is not possible to save specific settings with a given document (Publisher can do this). john korchok say:

In addition to Rich's technique, you can move a copy of Normal.dotm form the old template folder (Users/YourUserName/Library/Application Support/Microsoft/Office/User Templates) to the new template folder (Users/YourUserName/Library/Group Containers/UBF8T346G9.Office/User Content/Templates). Hold down the Alt key while clicking on the Go menu and choosing Library to show the Library folder.

There is a subtle difference in Word's operation when you do this. Since the Normal template is in 2011 format, all new documents will be in 2011 format as well and will display [Compatibility Mode] in the title bar. This means text boundaries still work as expected. Documents will remain in 2011 format until you choose File>Convert Document in Word, at which time they are updated to 2016.

ken sheridan say: The first thing you should do, regardless of your current issue, is split the database into separate front and back ends, if it is not already split, which the built in wizard can do for you.  Then all you'd need to do would be to import the tables from the back end into a new database file, selecting the 'definitions only' option.  You could then link a copy of your front end to the new empty back end file , using the built in linked table manager.

You could of course do the same without splitting the database, but any operational database should be split, and if used in a multi-user environment it's a sine qua non.  You can then five each user their own copy of the front end, each linked to the same back end.

However, as Scott points out, having separate databases for separate customers is almost certainly not necessary, and not a good idea.  You should have a Customers table in a single database and relate other tables to it.  You can generate separate sequences of numbers per customer at runtime.  You'll find an example as CustomNumber.zip in my public databases folder at:

john w. vinson say:

I do agree with Ken and Scott about the "right way" to do this, but - if you consider their good advice and decide that it doesn't apply to you - it's very easy to copy a database. Simply find the actual Access database containing the tables (this will be the "backend" if the database is split, which it certainly should be); it's just a file on your hard disk, with either a .mdb or .accdb extension.

Make a copy of it under a new name. Open it and go through all the tables deleting all the records [1] in each; if you have multiple tables in relationships, start with the "manyest" tables in the relationships tree and work up. [2]

[1] In most databases, there will be some utility lookup tables - e.g. a table of Titles like Mr., Ms., Dr. etc, or a table of States, or a table of Cities used to construct addresses. These will presumably be useful as they are in the new database and should therefore be left alone.

[2] You'll know you're deleting in the wrong order if you get error messages like "Unable to delete from Table X because there are related records in Table Y",.

scottgem say:

No, you do NOT use an ldb or mdw. These are companion files. The ldb file is a locking file that handles multi-user access. The mdw file is the file that manages user level security for the application. Note that having an mdw file does complicate making a copy. 

Access is NOT Excel. It is a VERY different product. 

You still haven't answered the main question here. Which is how the numbers are generated currently. That is a key question and we need the answer to advise. 

The simplest way to deal with this is to add a field to the table that indicates whether the data is for an internal or external customer. From there a number is assigned with either a B or G prefix. ken sheridan say: The .ldb file is a temporary file which only exists when the database is open.  It keeps track of current users, record locks etc.  The .mdw files are workgroup information files in which user and group permissions are stored.  The fact that you have a .ldb file rather than a .laccdb file, and .mdw files suggest that your databases are in .mdb format (or possibly .mde) rather than the current .accdb format.

The way to make it 'simple and easy' is to design the database correctly, on which we've advised you already.  If you are not comfortable with your ability to implement the advice then your organisation should be thinking in terms of hiring specialist assistance to do so.  It should be impressed on your management that good relational database design is something which is defined by formal principles on which there is a subtantial body of literature.  If these principles are not followed then you are at risk of producing a database which is not fit for purpose.  The learning curve is not a trivial one, and the development of an important business tool is not a task for the inexperienced.

Yes, he could very well have had that message before IE11, and, no, I don't know where the message originates.

You could use the Developer Tools (F12), Network capture to see if the message is arriving from the host.  Then it would be the server making a guess about something that it had detected.  Typically it is giving advice about a known cause.  It may be valid advice for the most common cause and therefore helpful in many incidents.  But if the same symptom has another less common cause which has a completely unrelated solution the other advice would be misleading and unhelpful.  Another possibility would be that the site's error handler doesn't even look at the symptom but just assumes what has happened.  So, if the advice doesn't work, always consider the possibility that all you know is that "something's wrong."
