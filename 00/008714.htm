graham mayor say:
Not so much impossible as tricky because when the row height is set to 'at least' a certain measurement, that measurement is not exposed to VBA and so when you read the measurement you get a string of 9s which bear no useful relationship to the current height of the row.

One possibility using a bit of lateral thinking is to set a range in the row and use the range information parameter to get the vertical position of the start and the end of the range (the cell content) and then scale that up a bit to give you an approximation of the height, which you can then fix.

Something along the lines of

Dim oTable As Table
Dim oRow As Row
Dim oRng As Range
Dim cellTop As Long
Dim cellBottom As Long
Dim rowHeight As Long
Dim TextSize As Long

For Each oTable In ActiveDocument.Tables
    For Each oRow In oTable.Rows
        Set oRng = oRow.Cells(1).Range
        oRng.End = oRng.Start
        TextSize = oRng.Font.Size * 2.2
        cellTop = oRng.Information(wdVerticalPositionRelativeToPage)
        oRng.Start = oRow.Cells(1).Range.End - 1
        cellBottom = oRng.Information(wdVerticalPositionRelativeToPage)
        rowHeight = cellBottom - cellTop + TextSize
        oRow.HeightRule = wdRowHeightExactly
        oRow.Height = rowHeight
    Next oRow
Next oTable


However, if your tables have merged cells this is not likely to work and the font sizes could affect the results, but it will otherwise be a lot quicker than manually going through the tables - even if you do have to make a few manual adjustments later.

macropod say:

The formatting of Table of Contents entries is controlled by TOC Styles.

When using Word's built-in Heading Styles, the TOC1 Style corresponds with the Heading1 Style, the TOC2 Style corresponds with the Heading2 Style, and so on. The TOC Style formatting is independent of the Heading Style formatting, so you could have an 18pt bold heading and a 14pt italic TOC entry, for example. If you want a particular TOC Style to have bold text, format that Style accordingly.

bob jones aka: cybertaz say:
The first place to check is Excel> Preferences> General. Make sure there is nothing in the field labeled At startup, open all files in:. That's what would cause the message if it's being generated by Excel.

OTOH, if you're using OS X 10.7/10.8 it could be OS X generating the message. If the above doesn't do away with it perhaps OS X is attempting to Restore... a nonexistent file. Check the settings in System Preferences> General.

It could still be a permissions issue -- note the 2nd bullet item in the message -- so I'd still do that. It can't hurt. Most importantly, though, restart your Mac.

s d morgan say: I tried checking the preferences both in Excel and under System Preferences.  I also repaired permissions.  No success.  When I try to launch Excel, I still get this same error.
