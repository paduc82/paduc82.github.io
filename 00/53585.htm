¡firedog say:

I then compacted.

 

I'm bound to ask why? and then how?

The only way I know of to force a 'compaction' - which in your case is not likely to compact anything much - is to set the 'compaction counter' on the Maintenance tab of the program options sheet to 1. This is basically asking the program to re-index a large database that may not even have finished its initial indexation after importing such a large number of messages.

The next time I closed Windows Mail Live, it started a process showing a box that says "Making a backup copy of the current database . . ."


That would suggest that the 'compaction counter' is still set at 1 instead of the default 100. I' can only guess at what's happening, but I suspect that the program has not finished indexing your 10GB of messages, so there is probably a large number of log files that have to be replayed to complete the process. In the midst of this, you ask the program please to create a backup of the index file. I'm not surprised it got confused.

Were you working offline while all this was happening, to avoid interference from sync operations? Was your AV disabled, so that it didn't need to scan your 2GB index file every time it changed, which it would for every 2MB of log files that got replayed?

After importing 10GB of messages, you should have left the program alone (e.g.overnight) to convert every one of those messages to an individual file and to write its headers, location and status to a log file ready for incorporation into the database index. This should preferably happen while not connected to the network and with security software disabled. Once that process is finished (you can see in Task Manager how busy wlmail.exe is), I would have closed down WLMail to allow it to do its housekeeping.

'Making a backup...' will normally only take a few seconds - it only involves copying the index file to the backup folder. You have asked it to copy an incomplete file, which the program is then trying to complete before copying.

The database may be able to recover from the state it's in if you give it time to do so. If you end up with a mess, it may be better to start afresh, this time importing under the most favourable conditions and allowing the program to adjust before starting to use Windows Live normally, send and receive messages, etc.

matthew_omega say:

Hello eqbalhussain,

 

It is never possible to determine how often a hardware issue will show up- or if it will show up on the diagnostics.  The intel architecture code of 0x9C means that a hardware error was reported to the processor.  It would be difficult to pinpoint the exact hardware even with a complete memory dump. 

 

If the diagnostics say that everything is fine, then that indicates that everything is working within the specifications that HP created in their hardware diagnostics. 

Since the server has been up for seventeen days at this point, this was probably just a one-time incident and everything is probably okay with it- just be sure to document that you received the stop code for future reference in case you have problems in the future.

 

Since you mention that this is a server, I would recommend that you contact the TechNet forums for further support on this as the Answers forum is not equipped for that level of support:
