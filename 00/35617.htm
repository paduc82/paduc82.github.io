tryx3 say:
  1. First check that Windows is activated "with a digital licence" at Settings, Update, Activation
  2. Make your own installation USB using http://windows.microsoft.com/en-us/windows-10/media-creation-tool-install
  3. The first stage of using that link is downloading the Media creation tool itself then running it.
  4. You then use the Media creation tool to create the installation USB - see Clean Install Windows 10 - Forum article
  5. During the clean install the whole drive is overwritten so make sure you have backed up anything you want to keep.
  6. During the clean install you will be asked to enter a product key but you must skip that [there is a bypass button labelled I am reinstalling or similar]. Entering anything at this stage would mess up your reactivation.
  7. You can then go online to [automatically] reactivate & to run Windows update.
  8. You should then also go to the computer maker's website to download & install their latest hardware drivers. Whilst Windows update tries to include hardware drivers it often gets this wrong.

Denis

As long as your Windows 10 was activated, you can boot straight from the installation USB I describe above to install Windows 10. 

There is no need to install Windows 8.1 & then upgrade again if that is what you are asking.  You just go straight to Windows 10.

Running a clean install is thought to give you a more stable system than you had after upgrading so your efforts will not be wasted. 

When you create your new user account, write down its password somewhere handy yet secure from loss & prying eyes.  The piece of paper you write it down on achieves everything that a Password reset disk achieves but at a fraction of the cost.  My user account passwords are in one of those little dog collar ID tags on my main keyring.

In addition to the user accounts that you want, create two additional [spare] password-protected local Admin accounts.

#1 is for use in troubleshooting so you do not end up adding excessive folder accesses to your normal accounts [in case a hacker ever infiltrates them while you are logged on].

#2 is to remain unused and therefore impervious to user profile corruption [which can afflict user accounts but only if those accounts are in use].  You could therefore always be confident that this Admin user account would operate correctly if you needed it in an emergency

bill_nh say:

System not recognizing Windows install on C:, using boot.ini from old install on alternate drive

Troubleshooting a computer hang issue, I flashed the BIOS.  Biostar has Global and USA sites and the two sites provide a different flash utility for the same model board.  I found the one on the global site first, used it, and it is apparently the wrong one.  I have restored the correct BIOS using a version from the USA site.

 

After this, WinXP wouldn't boot, but displayed a "missing or corrupt hal.dll" error. I replaced “hal.dll” from a working WinXP machine.  I used recovery console and "Bootcfg /rebuild" to attempt repair.  Recovery console doesn't find a Windows installation on C:, where it is.  It finds one on D:, typically the CD drive.  After using Bootcfg /rebuild", there are two operating system choices.  One works, the other still gives the "hal.dll" error.  Booting to the one that works, it appears it is using a Boot.ini file on the E: drive, a second physical drive in the machine, that had a windows install at one time, but not now.  The one that doesn't work seems to be trying to use the original boot.ini on the C: drive. The Advanced tab on My Computer/Properties shows the file from the E: drive.  The Run/Msconfig Boot.ini tab shows the contents of the Boot.ini on C:.   The Run/Msconfig Boot.ini tab shows the contents of the Boot.ini on C:.  The "Check All Boot Paths" button on Msconfig returns a message that "It appears that the following line in Boot.ini does not refer to a valid operating system".  Answering "Yes" to removing it, did not remove it.  Then used “Fixboot C:" in Recovery Console.  Symptoms remain same.  The "working" operating system choice appears to show "rdisk(1)" in boot.ini, while the bad choice uses "rdisk(0)".  Based on this I tried to use "Fixmbr \Device\HardDisk0", since my "Disk(0) seems to be an issue.  “Bootcfg /Scan” does not show the C: drive, where Windows is installed. 

 

Other things I’ve done:  Run chkdsk on Windows restart.  Run chkdsk /r from recovery console.  Ran both the Windows and DOS disk diagnostic utility from Western Digital.  None of these found disk errors.

 

I want to fix this, because the E: drive is a smaller old drive, used for backup.  I want to be running Windows fully from the larger, newer C: drive, as originally installed.

 

The “hang” issue was caused by a new version of antivirus, solved by going back to the prior version.

 

What else should I try?  Can I run a “repair” install from an original (no SP’s) install disk on a system updated to SP3?

 

Thanks for your help and advice.

 

Thanks, Bill

bill_nh say:

I eventually solved this.  Issue was that the BIOS default for "drive boot order" is IDE drive first, then SATA drive.  The C: drive was the larger, newer, SATA drive and needs to be first.  I apparently found this and set it right when I put the computer together a few years ago.  I didn't notice it after the BIOS flash.  Once I rixed this, I was able to go back to original boot.ini and hal.dll and all is well again.  The rebuilt hal.dll, from extracting from the Windows Install CD, also worked, but had a minor shut down issue, which the original hal.dll fixes. 

 

Duh! might be the right label for this solution.  Maybe this will save you some time if you have such a moment.
