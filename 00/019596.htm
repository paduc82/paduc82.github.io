albert d. kallal say:

The data may not be locked up, and you are talking about a local data file here. I would check if there is work group security here.

I mean even commercial products like Simply Accounting use Access database, but the file extensions were re-named. It took me about 5 minutes to figure out what was the mdb file, and what was the work group file. At that point in time could with ease open  up the Simply Accounting files directly from Access.

 

This is a local database and a local access mdb file here. This is YOUR data.

 

Since this is your data and your file it not clear that you have any special restrictions here on the use of the data. If this being the case, I see zero issues with you exporting the data out of this program.

 

The idea that special locking is occurring is ONLY your assumption here.

 

The fact that you cannot open + use this access file is not a given, but perhaps due to your lack of knowledge here?

 

Your setup sounds like the database is secured with ULS (User Level Security).

 

I mean as a general rule any Access database with ULS on it cannot be opened until you join the correct workgroup file. I suspect there no "special" locking up of this data, but only standard ULS at play here.

 

So ULS is there for providing a logon + some security in your work environment.

 

So I doubt some special "locking" is occurring here and the behavior you describe is standard fair to any experienced Access developer.

 

edit:

I think some people here reading this post have misunderstood your question since you provided a link to a web site. So I guess the first question here is, assuming that you don't have restrictions on the use of this data, have you checked If this database a secured using ULS, And if yes, have you tried joining the security group ?

In fact what does the command line startup look like for this program, I'm willing to bet that a path name to the security workgroups file is included.

In other words based on the information given so far, there may be no attempts of any Vendor intention to prevent you from using that data, just that of a standard security and password system of  which many access databases have .

Disabling the sound scheme is not enough for Outlook as there is a separate option in Outlook to play a sound when a new email arrives.
If Outlook is configured to play a sound when a new email arrives, then Windows will play the sound that is configured for “New Mail Notification” event in the Windows sound scheme. By default this is “Windows Notify.wav”.

Check for the sounds scheme tab in Control panel and check the box ‘no sound’ for new email.

Hi Roycenyc,

From the information you provide above, let me clarify the situation:

  1. Migrate the email data from Hotmail.com to Office 365 via IMAP migration.
  2. The migration was completed after six hours and close the migration connection.
  3. All the folders and emails data (2GB) appeared in Office 365 Exchange Online mailbox.
  4. All emails and folders disappeared suddenly in the next morning.

Generally, if the migration is completed, all the emails will be synced to Office 365 mailbox from the resource mailbox (Hotmail.com). When you close the migration by clicking “close connection”, the new emails that are received by the resource mailbox will not be synced to Office mailbox. But, it would not impact the emails that have been migrated to Office 365.

I have searched a lot of articles to try recovering those missing emails in Office 365, the possible way to achieve is using eDiscovery search. To use the tool, please refer to the following steps:

1. Assign permissions. Open Exchange Admin Center EAC, go to Permissions > Admin Roles > Discovery Management.

2. Add the admin account to the Discovery Management role group.

  

3. Go to Compliance management > In-place eDiscovery & Hold > click New +.

4. In the New In-Place eDiscovery & Hold window, on the Name and description page, type a name for the search, add an optional description, and then click Next.

5. Select Specify mailboxes to search, and then add the your mailbox. Click Next.

6. On the Search query page, select Filter based on the criteria and complete the following fields, like specific the start date and end date:

7. Click Finish to close the window.

8. Click Preview search results in the Search pane to preview the results after the search estimate is completed. Selecting this option opens the eDiscovery search preview window to check if the emails can be searched.

Note: if there is no email via eDiscovery search, it means those emails do not exist in the Office 365 Exchange Online server.

gary zhu msft support say:

Hi Roycenyc,

 

Thanks for your reply.

 

After doing a lot of research and consultation with our related team, I found the root cause of the issue.

 

When initial synchronization is complete for the migration batch, mailboxes in the IMAP messaging system and the corresponding Exchange Online mailboxes are synchronized every 24 hours (this is called incremental synchronization). During incremental synchronization, deletions made in an IMAP mailbox are synchronized to the corresponding Exchange Online mailbox, which means if a user deletes an item from an IMAP mailbox, it will be deleted from the Exchange Online mailbox.

For more information, please refer to the following article:

https://technet.microsoft.com/en-us/library/jj200751(v=exchg.150).aspx

 

I noticed the migration batch is not deleted, and you deleted all the email data in your Hotmail.com account. So this deletion is synchronized to your Exchange Online mailbox every 24 hours. That is why the emails which have already been migrated to the Exchange Online mailbox disappeared.

 

Given the situation, the deleted emails of your Hotmail.com account need to be recovered by Hotmail.com.

 

Thanks,

Gary
