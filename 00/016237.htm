hansv mvp say:

Remove the declaration

    Dim iResponse As Integer

from the cmd_del_rcrd_DblClick event procedure. Instead, declare it as a public variable at the top of a standard module:

Public iResponse As Integer

Change

    iResponse = MsgBox(strMsg, vbQuestion + vbYesNo, "Save Record?")

to

    iResponse = vbYes ' or vbNo if you want that to be the default response
    DoCmd.OpenForm FormName:="frmDialog", WindowMode:=acDialog

where frmDialog is the name of your popup form.

Set iResponse to vbYes or vbNo in the code behind the popup form, depending on the user's choice.

ken sheridan say: It's not good technique to get confirmation in a command button's code as a record might be deleted by other means.  Instead put code in the form's BeforeDelConfirm event procedure:

Private Sub Form_BeforeDelConfirm(Cancel As Integer, Response As Integer)

    DoCmd.OpenForm "frmConfirmDelete", WindowMode:=acDialog, OpenArgs:=Me.Name
    
    If Nz(Me.chkDelete, False) = False Then
        Cancel = True
    End If
    
    Me.chkDelete = False
    Response = acDataErrContinue

End Sub

Add a hidden check box named chkDelete to the form, e.g. in its header.

In the frmConfirmDelete form's module include a means of confirming the deletion by setting the hidden check box to True with:

    Forms(Me.OpenArgs).chkDelete = True
    DoCmd.Close acForm, Me.Name

Because the name of the calling form is passed to the frmConfirmDelete form via the OpenArgs mechanism, frmConfirmDelete can be opened in this way from any bound form.  The method of deletion in the calling form is for you to choose.  It could be via a command button, or any other means.  However it's done the code will be called and the dialogue form opened.
