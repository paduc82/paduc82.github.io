Assuming your W & L are in row 1 starting in column A (Adjust the ranges for your data)

These are array formulas so you need to enter them using CTRL-SHIFT-ENTER rather than just ENTER. If successful a pair of {} will be put around the formula

=MAX(FREQUENCY(IF(A1:P1="W",COLUMN(A1:P1)),IF(A1:P1<>"W",COLUMN(A1:P1))))

greginmich say:

Windows actually provides the infrastructure for running any program/task when it’s in a state of sleep or hibernation; and MSE is no exception. The ability to wake up a sleeping PC to run a program/task is an absolutely essential feature for centralized automation – and Task Scheduler is perfectly capable of doing this for any Windows application.

The problem with the MSE scheduled scan task (The Microsoft Antimalware Scheduled Scan task) is simply that the built-in options for waking the PC at the scheduled time, and for running the scan at the next startup when the scheduled time is missed, were both disabled. So if you want a reliable MSE scheduled scan, what you have to do is set up your own task for it and then enable these "missing" options. 

This is made possible by the MSE Command Line Utility (MpCmdRun.exe), which was specifically designed to interface with Windows Automation for the purpose of adding new features and capabilities to the Security Essentials app, and which thus allows users to go beyond the rather limited capabilities of the MSE GUI.

Here are a couple of old threads that explain how to use Task Scheduler, or a batch file, to wake up a PC; run an MSE scan; and then shut down the PC after the scan completes. I was previously updating some of these old threads in order to keep the information current – but the overseers have once again nicely managed to thwart my efforts by locking them.

ken sheridan say: A check box reflects Boolean TRUE and FALSE values, which in Access are implemented as -1 and 0.  So if you used a check box bound to your field you'd be inserting values of -1 or 0, not the 1 or 0 which you are using to represent TRUE and FALSE.

I'd suggest that you change to using -1 and 0 as you can then change the underlying field in the table to a Boolean (Yes/No) data type.  Before you do that, however, you'll need to update the 1 values in the table to -1 which you can do with a simple update query:

UPDATE [YouTable]
SET [YourField] = -1
WHERE [YourField] = 1;

The other point is whether you have any Nulls in the column.  A Boolean field cannot be Null, so I'd suggest another update query to make sure of this;

UPDATE [YouTable]
SET [YourField] = 0
WHERE [YourField] IS NULL;

You should then be able to change the data type of the field to Yes/No in table design view.

Next change the option group so that the values of the two option buttons are -1 and 0 for Yes and No.

In a report you can then use a check box bound to the field, or you can use a text box bound to the field and formatted to show Yes or No by setting its Format property to Yes/No  If for any reason you need to show 1 or 0 then use a text box with a ControlSource property of = Abs([YourFieldName]).

Ken Sheridan, Stafford, England

ken sheridan say: The point I was making was that, while values of 1 and 0 will be interpreted as TRUE and FALSE, if a bound checkbox is used to enter values -1 would be entered not 1, so there would be a discrepancy between rows.  It turns out of course that Veronica is actually using a column of text data type, but notwithstanding that  there is little sense in not using a Boolean data type in this context.  The only situation where it would not be the sensible choice would be if 3-way logic were required, in which case an integer column with a validation rule to allow values of 0 or -1, or to be Null would be appropriate.
