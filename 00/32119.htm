Ok, for anyone interested, I found the answer and resolution for my specific situation with respect to theError 0x80070005 (Access is Denied) after applying fix for Default User profilecorruption.

We recently had an Active Directory problem that turned into a disaster with 2 domain controllers failing and being force decommissioned. During the disaster recovery AD roles were Seized to a known stable domain controller, a new DC rebuilt to replace the 2 failed ones and everything appeared to be working from that point. However, there were residual effects with GPOs not properly applying, security permissions issues, etc.

This past weekend I had to work with MSFT Support to resolve as the reported problem above was a result of the continued problems mentioned. When I got MSFT Support on the phone and the Tech connected to my machine, we logged into the RID Master DC, instantly we got the same Windows Genuine License error. The problem was that the DC was no longer accepting sync requests, the security permissions on the entire disk were not working, registry permissions were also a problem and the logon profle list and GUID registry keys were also corrupted. This DC required a force dcpromo (demote) and repromote with the roles seized to yet another DC. All of the AD problems were resolved and the failing DC restored through the standard process.

Interestingly enough, as soon as we resolved all of the security permissions issues on the drive, we were able to then restore the DC to a working and stable state, all of the DCs began syncing properly again. We had to take an additional step for resolving the profile problem by copying the Profile List and Profile GUID Registry keys from a second DC that had no issues and the Windows Genuine License error and theError 0x80070005 (Access is Denied)disappeared on the failed DC.

The Windows Genuine License error and the Error 0x80070005 (Access is Denied) persisted on the Exchange server, however, we simply copied the same Profile keys to that server, resolved some security permissions issues on the drive and system folders (Windows, System32, Config and System Profile Folders specifically), rebooted and the problem no longer existed.

The specific problem causing this was security permissions on the hard drive, and the folders (Windows, System32, Config and System Profile Folders)

These were all either missing the necessary accounts or if they had the accounts listed in the security tab, the permissions were all unchecked. We had to start at the bottom level folder such as the System Profile folder in the following path - %windir%\System32\config\systemprofile

We had to take ownership, click ok to all of the open properties windows, re-open the properties on the folder, apply ownership to all subcontainers and contents, click ok to close the properties windows, then again re-opened the properties on the same folder and then applied the necessary permissions to the necessary accounts on the folder. The permissions applied were - Authenticated Users - Full Control, System - Full Control, Network Service - Full Control, Administrators - Full Control, Users - Read, Execute, List, Write; Trusted Installer - Full Control

We then worked back up the top through the folder hierarchy, one folder level at a time, to the drive level, applied all permissions at each level.

Once all permissions were confirmed as good, we rebooted and the problem was resolved.

Needless to say, the Windows Activation Center would not have been able to help with this and if you are experiencing this problem, I highly recommend you call into Microsoft Support as there are so many reasons for this problem, and none of the solutions I found on any of the forums worked due to all of the combined underlying issues with AD, Security, and profiles.
