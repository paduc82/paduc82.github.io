Outlook Search Workaround - Microsoft Community

jacques_van_tonder say:

Outlook Search Workaround

Good day,

An alarming amount of us have been struggling with Outlook 2016's search function.

It is just horrible and does not always return all the results it's supposed to and randomly displays error messages.

I have spoken to Microsoft in detail about this and they told me that they are aware of the issue and they are working on it, it should be fixed in the next update but we have all heard that before.

They provided me with a workaround which worked for me so I decided to share it with everyone.

The following solution worked for me, but is much slower than the default search but still better than it was.

So until Microsoft fixes the issue here is the workaround.

Please make a backup of the registry before doing the following

Detailed Instructions:

1.    Click Start, click Run, type regedit in the Open box, and then click OK.

2.    Locate and then click the following subkey in the registry:

 HKEY_LOCAL_MACHINE\SOFTWARE\Policies\Microsoft\Windows

3.    On the Edit menu, point to New, and then click New Key and name the key Windows Search.

4.    Click on the new key Windows Search.

5.    On the Edit menu, point to New, and then click DWORD Value.

6.    Type PreventIndexingOutlook for the name of the DWORD, and then press Enter.

7.    Right-click PreventIndexingOutlook, and then click Modify.

8.    In the Value data box, type 1 to enable the registry entry, and then click OK.

9.    Note to disable the PreventIndexingOutlook setting, type 0 (zero), and then click OK.

10. Exit Registry Editor, and then restart Outlook.

The search should now work correctly but at a slower rate, a message will be displayed at the top of the search telling the user that a group policy has been disabled so the search will be slower.

I hope this works for everyone as it did for me.

There isn't enough disk space or memory - Microsoft Community

ricardolgomez say:

There isn't enough disk space or memory

I'm trying to change the field format of 1 column in a table. But I keep getting a"There isn't enough disk space or memory" error...I've searched for solutions but I haven't found anything that applies to MS Access 2013

john w. vinson say:

If this is a large table, it may be that you're getting the error because Access brings the entire table into memory, changes the field format, and writes it all back out.

 

Try instead taking it in steps:

 

1. Remove (but note) all the relationships to the table

2. Rename the table to (say) tablename_Old

3. Copy and paste the table, DESIGN VIEW ONLY, to a new table with the original name

4. Change the fields as needed

5. Run an Append query to migrate the data from _old into the new table

6. Reestablish relationships

7. Check everything, if it's ok delete the _old table

ricardolgomez say: Not sure what you mean with DESIGN VIEW ONLY. Do you mean pasting the table as STRUCTURE ONLY?

ricardolgomez say: also, the field in the original table is a text field...doesn't this conflict with the append query since the field in the new table is not text but numeric?

john w. vinson say:

You're right, it is structure only.

 

The text vs. number problem would arise any way you do it - changing the field type will cause an error if the existing data is not numeric. I'd suggest first running a cleanup query, e.g.

 

SELECT * FROM yourtable WHERE [fieldname] IS NOT NULL IsNumeric([fieldname]) = False;

 

Manually correct any nonnumeric values (or use update queries if there are a lot and there's some pattern you can use), and then use

 

Val([fieldname])

 

as a calculated field in your append query to convert the text string to a number.

 

ricardolgomez say: Thanks, John

Isn't there a way to change the memory allotted to this operation? Having to manually update fields every time this problem arises doesn't seem sustainable in the long run

john w. vinson say:

You should not NEED to routinely change field definitions in your tables!!!!

 

If you're importing data from Excel or elsewhere, consider having your table set up with the appropriate datatypes; then link to - don't import! - the external data and run an Append query to populate your table. Use Val() or CLng() or CDbl() or CCur() as appropriate to convert external text fields to the appropriate numeric datatype.

sam mathai chacko say: Sure you can improvise with this. You started talking about hiding/unhiding, but then you mentioned protection in the code. I didn't know what you wanted in the end.....

Sub HideSheets()

    Dim strSheetName As Variant
    Dim strPassword As String
    Dim lngLoop As Long
   
    strPassword = InputBox("Enter the password for the worksheet")
    strSheetName = Array("Students", "grades", "courses", "levels", "professors")
    For lngLoop = LBound(strSheetName) To UBound(strSheetName)
        Worksheets(strSheetName(lngLoop)).Unprotect Password:=strPassword, Scenarios:=True
        'OR/AND
        'Worksheets(strSheetName(lngLoop)) = False
    Next lngLoop
   
    Erase strSheetName
    strPassword = vbNullString
    lngLoop = Empty
   
End Sub

Sub Hide_Me()
Dim ws As Worksheet

For Each ws In ThisWorkbook.Worksheets
  If Not IsError(Application.Match(ws.Name, v, 0)) Then

    End With
  End If
