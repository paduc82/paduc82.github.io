albert d. kallal say:

I think the first problem here is yours very much misunderstanding the difference between a popup form, a modal form, and that of a dialog form. They are VERY different kinds of forms in how they function.

 

So when you say that you're opening the form as dialog, that is VERY much different then setting a form as model. This also explains why you're stating that a modal form is behaving different than that of a dialog form. In fact, they are VERY different and are supposed behave differently.

 

If you set a form as modal, and open the form in code as normal then modal setting will be respected. Modal means that the current form open will keep the focus, and you can either have that form open another form, or close the current form and return to the previous one (a high portion of my forms are thus set model since I want the user to return back the way they came). However a modal form still permits you to move the focus to things like a popup form, or even to the menu bar or in your case the ribbon. Another significant difference of a modal form is that it does NOT halt the calling code as a dialog form does. So, the calling code continues to run.

 

A dialog form on the other hand again behaves VERY much different. First, dialog forms DO HALT the calling code UNLTIL YOU SET VISIBLE = FALSE! (this allows the calling to continue and explains your apparent bug). Also dialog forms do NOT permit the focus to move to the ribbon or even a popup form. So model forms allow both of these change in focus actions, dialog forms allow none. The difference between dialog and moral forms is thus rather large here and you want to choose the appropriate type of form for the appropriate type of the action and user interface.

 

> Setting the call to acWindowNormal on the parent, with both modal and popup set

 

Now you are mentioning a popup form and that's a third type of form. Again pop up form has different behaviors then the above two forms. For starters, a popup form can stay on top of model forms, but NOT HAVE to receive the focus. So, for help type screens or information boxes that a user needs to be on top, but allow the person to continue data entry in a form is a good example for popup forms. So, popup forms can be displayed on top of other forms, but NOT have to get the focus. And, you can move focus from a model form to either the ribbon or a popup form. They really are best suited for wizard or info type boxes. However, just out of the blue setting forms as popup can really mess up your ability to control what form is on top. So don't just start clicking on a whole bunch of options without giving them thought or you make a real mess of things.

 

I would not suggest using dialog forms for you goal here as they are rather severe in terms of restricting user actions. However, the issue here is calling code is halted UNTILL the form is closed OR you set visible = false. So, here what happens:

 

Form A code->form B code

 

Now, form A code waits until form b is closed OR VISIBLE = FALSE (note the visible = false is 2<sup>nd</sup> way for the form A code to run!)

 

Form B->form C

 

In the above call B form was JUST SET visible = false. This means that NOW form B code will return to form A code! (the visible = false). Now, things are still halted up by the fact that form C is in dialog mode. However, the instant you close C, then that code in A will run. Since form B was set visible = false.

 

So, just remember that setting a form visible = false permists the CALLLING code to run.

 

I explain the above detail in the following article of mine in which I use this "feature" to return code values from dialog forms to the calling form. So, setting visible = false kicks a dialog form OUT of dialog mode and permits the calling code to run.

 

http://www.members.shaw.ca/AlbertKallal/Dialog/Index.html

 

As a general rule, dialog forms are quite restricted user interface approach and design choice on your part (you can't even use the ribbon edit or print a report for example).

 

A dialog form is really only for yes/no type prompts and it DOES halt the calling code. In fact, it halts so very much of everything, that it can even halts the ability of access to often do house keeping and re-plot screens .

 

If I were writing this, I would advoid dialog forms. I would control the visible in the forms close event.

 

In each form I would declare

 

Dim frmPrevious      as form

 

Then, in the forms on-load event, I would go

 

Set frmPrevious = screen.ActiveForm

 

Now, in the forms close event, you can go:

 

frmPrevious.Visible = true

 

The above trick also allows you to also reference features and values of the previous form such as frmPrevous.Requery etc.

 

Note that you code to launch a form must be changed from

 

  DoCmd.OpenForm "form1"

  Me.Visible = False

 

The reason why we set visible = false AFTER the openform command is because in each forms on-load event we setting up form previous, and that fails if the form is not visible.
