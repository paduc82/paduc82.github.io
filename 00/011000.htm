rohn007 say:

A starting point for this sort of design process is identifying all of the "data stores" in your system.  That is places where any related data is stored: paper (order, inventory, use) forms, file folders, in/out boxes, filing cabinets etc.

That will give you an idea of all of the information you need in your system.

Paper forms also can give you an idea of the tables you need. Of course a single paper form can be using data from more than one table.

Do a google/bing search for "database normalization" .  Normalization is a process where you design the DB to store the minimum data required by removing redundant data storage.  Any time data is repeated you want it stored in a separate table.  The only "redundant" data will be the "keys" you use to retrieve the data.

http://timlin.net/Normalization.pps - this presentation is a good starting point.

Depending on which "authority" you look at there can be various different sets of rules.  As you get into the "high" level rules things get very esoteric.  Sticking to the first 3 rules is a good place to start, ie:

https://support.microsoft.com/en-ca/kb/283878

Start small, and build up. When you identify the need for redundant columns in a table, consider splitting it out into an new subordinate table.

Working your design on paper, or whiteboards is a good way to start.

ben herila say:

Recycling words from a recent e-mail exchange

First, what build is this? (i.e., what does Get-PackageProvider  return ? )  If you don't see Chocolatey, you need a newer version of OneGet (you can get the latest experimental build from http://oneget.org/oneget.zip )

Make sure you have the version of OneGet that supports Chocolatey -- the one from http://oneget.org/oneget.zip. The version of OneGet that is included in Windows 10 does NOT support Chocolatey! Download the .ZIP, unpack, and manually import the module. 

 

Otherwise, you'll have to wait until the next Preview build of Windows 10.

 

Then, you can make sure you get the chcoolatey provider:

 

> Get-PackageProvider -name Chocolatey -ForceBootstrap 

 

Then you can install a chocolatey package (the experimental build from oneget.org/oneget.zip should come with the chocolatey.org source pre-configured).

 

> Find-Package -Provider Chocolatey package_name

Thanks,

Ben

garrettserack say:

If you downloaded the one from http://oneget.org/oneget.zip you need to start a new powershell session and import that one before the system one gets imported.

> ipmo  c:\path\to\oneget.psd1

Next, get rid of the 'chocolatey' package source that is associated with the PSModule provider 

>unregister-packagesource -provider PSModule chocolatey 

The experimental version has a Chocolatey provider with it, so you should see it if you do a 

> get-packageprovider 

The existence of the the NuGet version 2.8.3.3 is a clear sign you're using an older OneGet.

G

edwardoendiz say:

I was having similar issues.

What fixed it was I created a C:\Temp\ folder, extracted oneget to that folder and then ran

> ipmo  c:\path\to\oneget.psd1

As I was looking at the error generated it wasn't giving the full path to 'C:\Program Files........'
It was reporting 'Files\WindowsPowershell\........'
Seems to me that it does not like the space in "Program Files"

But then I couldn't delete the 'C:\temp\' folder since OneGet was bound to it. So plan accordingly.

There is also a readme in the OneGet zip file that said to run a batch file called RunToUnBlock located in the oneget directory.
