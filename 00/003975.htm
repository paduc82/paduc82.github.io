albert d. kallal say:

You likely should not have to do much of anything.

However, what I would do is try re-linking the tables.

So launch the linked table manager. Do a select all, and then hit ok.

This should re-fresh the links.

If this refresh does not work, then re-launch the linked table manager. Select all, but ALSO click on the “Always prompt for new location”.

This will force/allow you to re-create the DSN. You can selecting your existing one, or simply create a brand new one.

Keep in mind I “strong” suggest when doing the above you choose a “file” DSN.

The reason for this is Access automatic converts all file dsn’s to DSN-less.

So this is a great default by Access, since if your tables link correctly, then you can deploy the application to any workstation on the network, and not have to setup or bother with creating a DSN on those other stations.

This also means you don’t have to use code or anything to re-link the tables, and you don’t have to do anything and the result is a DSN-less connection.

So the “default” for Access is what we call a DSN-less connection. No VBA code or anything is required for this preferred DSN-less connection. This also means that Access will ignore the DSN you have, and you could even delete it - Access will not care (Access only uses this during a re-link and only if you check the prompt for new location box).

And more important, if you modify the DSN - access will NOT see this change nor care. (so 2nd suggest is required here if you change the dsn). So if you modify the DSN, Access will not see or know about this change.

So first try the re-fresh. If that don’t work, then re-launch linked table manager again, and ensure you tick the “always prompt for a new location" – that will allow you to re-build the connection.

I suspect when you upgraded the server, perhaps the name or instance or sever name changed. Thus the 2nd  suggest above (prompt for new location) should fix this issue.

edit: I also don't think this is related to TLS 1.1 - as far as I know that is only related to https (web server), not your general network tc/ip that you using to connect to sql server.
