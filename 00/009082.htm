ken sheridan say:

A Subform, linked to the main form by the PK and FK of the two tables, containing the two fields I have mentioned with a record source of:

SELECT Donations.Currency, Donations.Totalfunds, Donations.Grantcode
FROM Donations;


A subform normally shows the rows in a referencing table which relate to the current row shown in the parent form, which is based on the referenced table.  It does this by means of a foreign key column in the subform's recordset being the LinkChildFields property of the subform control, whose LinkMasterFields property is there primary key of the parent form's table, which is the referenced table in the relationship.

You appear to have the reverse of this in that the subform is based on the referenced Donations table. That would imply that there can be only one donation per project, which seems an unlikely scenario, and would make the subform redundant.  If there can be multiple project-specific donations per project, which I would imagine to be a possible scenario, then there would be no need for a GrantCode foreign key  in Projects, but a ProjectID or similar foreign key in Donations, in which case ProjectID would be the LinkMasterFields and LinkChildFields properties of the subform control, and the combo box and unbound controls in the parent form would be unnecessary.

This would assume that each donation is project-specific, i.e. is made to one project only.  The other possible scenario is that a donation relates to one or more projects, i.e. a single donation can relate to multiple projects, either as stipulated by the donor, or, for instance, as part of an appeal by the receiving organization which covers one or more specified projects.  In this case the Donations table would have no foreign key referencing the primary key of projects, and subform would be based on a table which models the many-to-many relationship type between Projects and Donations by resolving it into two one-to-many relationship types, e.g.

ProjectDonations
....ProjectID  (FK)
....GrantCode  (FK)

along with any other non-key columns which represent other attributes of the relationship type between Projects and Donations.

If they have the full version of Access, then that will open the accdr file, but open as if it was the runtime. The problem here is that the runtime version is the same file name as the fully licensed version. So, if you set the association you need to make sure you associate it with the right MSACCESS.EXE.
