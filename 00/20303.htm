ken sheridan say:

I'm not sure why I wouldn't want to store this ID. It wouldn't be my primary key..........

Firstly the generated number would be a key, albeit not the primary key.  The reasons for not saving this structured key as a column in a base table are threefold:

1.  It violates the Guaranteed Access rule (Codd's rule #2), whose definition is:

Every datum (atomic value) in a relational database is guaranteed to be logically accessible by resorting to a combination of table name, primary key value and column name.

2.  It introduces redundancy, which leaves the table open to the risk of update anomalies.

3.  It's unnecessary as the structured value can easily be returned at any time by concatenating it's elements, each stored separately in its own column.

You might like to take a look at CustomNumber.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo file illustrates how to generate sequential numbers, catering for possible conflicts in a multi-user environment if necessary, and includes an option for doing so per group.  In my file the group is gender only, whereas in yours the group would be litterID and gender combined, so you would need to amend the code accordingly in the gender control's AfterUpdate event procedure, and the GetNameID function which it calls.

In my case the two columns NameID and Gender are defined as the composite primary key, but as you are using a surrogate key the three columns, LitterID, Gender and SeqNumber say, would be included in a single unique index, as Scott has pointed out, to define them as a candidate key.

A composite index is created by selecting Indexes on the Design ribbon.  In the dialogue which opens enter a suitable name for the index in the first empty row of the Index name column, MouseKey say.  In the Field Name column of the same row enter LitterID, and with this row still selected, in the Index Properties below, select 'Yes' as the Unique property.  Then enter Gender in the Field Name column in the row immediately beneath LitterID.  Don't enter anything in the Index name column for this row.  Do the same for SeqNumber in the FieldName below Gender. Close the dialogue and save the table design.

When returning the structured key, if you need to cater for litters of 10 or more, you will perhaps want to format the sequential number with a leading zero for numbers 1-9, so the expression in a query for instance would be:

SELECT LitterID & Gender & FORMAT(SeqNumber,"00") AS MouseKey

You might wish to format LitterID similarly with however many leading zeros considered appropriate..

In a computed control in a form or report you'd use a similar expression as the ControlSource property.
