Will Windows 10 finally address the issues with extremely lengthy - Microsoft Community

digitalzombie say:

Will Windows 10 finally address the issues with extremely lengthy software RAID array resynchronization?

Since Windows 7 I've tested Windows software RAID on and off as an initial level of data availability/redundancy. My experience has always been negative in the respect that if my PC didn't shut down properly the Windows-managed RAID array (I only used drive mirroring aka RAID 1) would always corrupt requiring an abnormally long period (days) to resynchronize the array. I have contacted Microsoft directly about this a number of times with no solution provided for Win 7 - 8.1. Eventually I gave up on drive mirroring entirely because of the lack of robustness of the product in Win 7- 8.1. If a proper solution was implemented I would use the technology again.

For the majority of users looking at RAID as a data availability/redundancy option having a working product native to the OS that was resilient enough to be useful is the best first-level option. Hard drives fail, power delivery is not 100% guaranteed and other hardware issues can all contribute to a Windows session from being disrupted so we need a real-time OE solution to maintain uptime and safeguard our data. In one week for example you can lose alot of data if you only back up on a weekly basis. Also running to the local computer store to buy a new hard drive and reload your Windows OS every time a hard drive fails is obviously not an efficient way to use a computer (not to say that hard drives fail that often but even a brand new hdd can die).

With the capacities of current hard drives running over 1 terabyte you are looking at days (my 3TB mirror took over 5 days to rebuild every time it corrupted) to rebuild a damaged array. This requires leaving your PC on until the resync process is complete which is 1) energy inefficient, 2) irritating if you need to restart for a windows or other software update, 3) only 100% possible if you have a battery back up in the case of brown/black outs, 4) please add to this list.

On the flipside, running hardware or hybrid RAID [motherboard RAID BIOS interface + third-party (usually the chipset manufacturer's) Windows software utility for monitoring/reconstruction] would only take a matter of hours to rebuild a damaged array.

It was honestly easier and less time consuming to delete the array, reformat the secondary RAID array member and build a new array with the source material on whatever drive was available after deleting the array. Note: In the event of a total drive failure adding a new drive to the array actually works properly but takes just as long to resynchronize.

davidf2 say:

My guess is that your new computer display has a much higher resolution than the one using Pub 2007 and Win7.

I see that you are using a page width of 760 pixels on the Publisher page. However it appears that your HTML page is much wider. If you go into your Display Settings and the DPI settings I would bet that it is set to 120 dpi or larger, which makes screen objects larger...and skews the rendering of your web page.

If you look at your old computer I would bet it is set at 96 dpi. If you reset your new computer display to 96 dpi (which is really PPI or pixels per inch) and then produce your web pages, they will probably look correct on a 96 dpi screen. However if you then go back to a 120 dpi setting and view the published page the layout may be skewed.

Publisher produces a fixed width page and one of the reasons MSFT began to deprecate the web building functionality as of Pub 2010 is this problem. A fixed width page produced at 96 dpi does not render correctly on a 120 or 140 dpi screen. They did not have an easy fix for this and it factored into their decision to drop the web building functionality of Publisher.

You can switch to 96 dpi setting on your new computer which will make everything smaller, and probably too small to read and view easily.  You can use you old computer with Pub 2007 to edit and produce your web pages. Your best choice is to find a different program to produce your web pages. There is no future in using Publisher for that purpose I am afraid. Sorry...
