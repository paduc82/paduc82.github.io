rob koch say:

Since anyone creating such a malicious script or program could simply create multiple layers of obfuscation to hide the original item that started the process, it wouldn't necessarily be accurate to display the process that launched the msiexec.exe or other Windows program either.

And in most cases the PC user knows far less than you, so providing that information would be of little true value anyway, though it's always possible this will be changed in future versions if enough people ask for it in Windows 10 feedback.

Personally, I think that Microsoft is initially providing CFA turned off by default in order to allow developers to fix their slightly misbehaving applications before they simply remove the ability to write to these folders by default, at which point they'll be forced to conform.  So anyone producing programs they still want to distribute widely will need to fix their nonconforming code, eventually leaving only the truly malicious programs attempting to write to these areas.

Since most of the folders that CFA protects aren't critical to installs and even the creation of a shortcut on the CFA protected Desktop folder isn't considered a best practice, I doubt you'd ever have any problems installing an otherwise well behaved major application.

In fact, program installation is one of the most potentially dangerous actions, so reducing protection of any sort at such a critical time is the last thing I'd personally do.  This is why I've always had a problem with software developers who tell you to turn off your AV during installation, since if they can't write an installation process that major AV consider safe, then how well written can their primary applications be?

I'd rather have to attempt a reinstall on the rare chance this might be necessary than allow some actual malware that somehow became attached to or replaced what I think I'm installing, whether by mistake or intentionally.
