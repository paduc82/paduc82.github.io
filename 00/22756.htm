graham mayor say:

Personally I would approach this from a different angle. For my playlists, I use PlayList Creator -http://www.oddgravity.de/app-opc.php
The m3u playlists produced are Media Player compatible and plain text so can easily be edited in Word. The following macro will format the m3u format playlist as a list that can be printed

Sub CleanM3U()
Dim iCount As Integer
Dim opara As Paragraph
Dim sFname As String
sFname = ActiveDocument.FullName
'remove the extension from the filename
sFname = Replace(LCase(sFname), "m3u", "")
'set a counter
iCount = 1
With ActiveDocument
    For Each opara In .Paragraphs
        If InStr(1, opara.Range, "#EXT") Then
            'remove unwanted paragraphs
            opara.Range.Delete
        Else ' or number the track
            opara.Range.InsertBefore Format(iCount, "00") & ". "
            iCount = iCount + 1
        End If
    Next opara
    'copy the title from the end to the start
    .Range.InsertBefore ActiveDocument.Paragraphs.Last.Range
    'delete the title from the end
    .Paragraphs.Last.Range.Delete
    'remove the added track number from the title
    .Paragraphs(1).Range.Words(1).Delete
    .Paragraphs(1).Range.Words(1).Delete
    'and apply a format to the title
    .Paragraphs(1).Range.Style = "Heading 1"
    'save the end result as a document (here DOC format)
    .SaveAs sFname & "doc", wdFormatDocument
End With
End Sub

Const OFFICEFOLDER = "DriveLetter:\SomeFolder\Databases\"
Const HOMEFOLDER = "C:\SomeOtherFolder\MyDatabases\"

Public Function LinkToHome()

    Dim dbs As DAO.Database, tdf As DAO.TableDef
   
    Set dbs = CurrentDb
   
    ' Loop through tabledefs and refresh link to each table
    For Each tdf In dbs.TableDefs
        If tdf.Connect <> "" Then
            tdf.Connect = Replace(tdf.Connect, OFFICEFOLDER, HOMEFOLDER)
            tdf.RefreshLink
        End If
    Next tdf

End Function

scottgem say:

Sorry Ron, but your approach is incorrect. The correct way to do this is to use relinking code. I have handled this a variety of ways, some of which have been mentioned here.

One app that is similar to your situation (a complex app supported remotely) was to use the user login. I had my own ID with a user level of Developer. Whenever anyone with that user level logged in a check box appeared to indicate whether to use test data. If the check box was checked, the code relinked to the back end in the current folder rather then the one on the server.  Later I added code to check whether the shared network folder existed and, if not, prompt the user to use a local copy. This permitted the client to copy the app and data to a laptop that he could take with him. Of course he couldn't edit the data then, but that wasn't an issue.

In my current position, I put all my development copies in a local folder named ITDev. When I need to have users test an app, I put it into a UserTesting folder. I then store the location of the production backend in a parameters table stored locally in the front end. On app load I test for the current folder of the front end, if its in either of those two folders it relinks to copies of the back end in the current folder. Otherwise it links to the production back end.

This is all transparent to the user. Unless the user looks at the status bar where it shows the relinking they don't know this is being done.

Hope this helps, Scott<> P.S. Please post a response to let us know whether our answer helped or not.
