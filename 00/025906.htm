give this a try:

  • Press Alt+F11
  • Menu Insert/Module
  • Copy the following code into the module
  • Press F5 while in code

Sub DeleteEmptyRowsInActiveSheet()
Dim lngI As Long

  For lngI = ActiveCell.SpecialCells(xlCellTypeLastCell).Row To 1 Step -1
    If Application.WorksheetFunction.CountA(ActiveSheet.Rows(lngI)) = 0 Then
      ActiveSheet.Rows(lngI).Delete
    End If
  Next
 
End Sub

 

In addition insert this macro in your personal macro workbook. If you are in Excel view press Alt+F8 > select the macro name and press Run.

john w. vinson say:

Hi and welcome to the wonderful world of Access.

First off... extensive Excel experience can help in some ways, but in others can be a positive hinderance to learning about relational databases! Excel is a spreadsheet, best of breed; Access is a relational database programming environment. They are different and require different mindsets, approaches, and techniques! There will be some things you need to "unlearn", that make perfect sense in a spreadsheet but are just plain wrong in Access. For example, you ask about "cells" - databases don't HAVE "cells"!

A properly indexed 300,000 row table is nothing special. I've used 3,000,000 row tables with good results, even without moving the data to SQL/Server; and SQL can handle terabytes of data if you need it.

Splitting the data into lots of little tables is a VERY bad idea - it's called "storing data in tablenames" and leads to nightmarishly complicated queries and reports. Just have one tall, thin table for each type of Entity: a table of Members, a table of Sessions, a table of LiftingSets (ONE table for all the members, it would have a MemberID as a link).

To look at smaller chunks of data, you'll use Queries to extract a subset of rows and fields. Unlike in Excel, you don't create a new table ("sheet") to do so; the Query is a virtual table, and can be used as the Recorsource for a form, report, export, or even another query. You'll want to use "Parameter Queries" which reference a control on an unbound Form to designate which person, or which session, or whatever other criteria you need.

Roger Carlson's tutorials, samples and tips:
http://www.rogersaccesslibrary.com/ 

A free tutorial written by Crystal:
http://allenbrowne.com/casu-22.html 

Crystal's intro to Access has a very good chapter on normalization, for example.

john w. vinson say:

A table should be big enough to hold the data you need. If you have performance problems, the first step would be to optimize your table indexes (an index will make searching and sorting faster on the indexed fields, but slower on updating); optimize your queries to retrieve no more data than needed, use indexes and Joins correctly, and avoid calling VBA or builtin functions as much as possible; review your table design; if need be move the backend to SQL/Server or another engine (not a trivial task but sometimes worth the effort).

But don't let worries about performance affect you too much at this stage. What you're describing is very comfortably within Access' capabilities for at least a couple of years, and improving performance on a properly normalized database is a matter of tweaking more than moving to a denormalized design. I can GUARANTEE that breaking the data into dozens of non-normalized tables is one of the very best ways to KILL performance - not to improve it!

ken sheridan say:

As I thought, there is a balance between having to large a table, and "storing data in tablenames" as you call it.

Not really.  Data should never be encoded as table names.  A fundamental principle of the database relational model is the Information Principle (Codd's Rule #1). This requires that all data be stored as values at column positions in rows in tables, and in no other way.

A couple of little files you might find useful as an introduction to the basic principles of relational database design can be found in my public databases folder at:

One is DatabaseBasics.zip.  As its name implies this file illustrates some of the basics of designing a simple relational database, including correct table and relationship design.

tom van stiphout say:

Large? That's really not a concept we worry about very much. Why? Because of the power of indexing. Let's look at a simple example: find the customer with ID=X among 1M records.

Or in SQL:

select * from Customers where CustomerID=InputBox("Customer ID?")

The index on CustomerID has the IDs sorted numerically and asks: is X below or above the middle value? Let's say it's below. Then it looks at the remaining half and asks the same question. And so repeatedly until the value is found.

At most this takes 20 loops.

If there was no index, it would on average take 500K loops.

That's why indexed lookups are so fast.

If you go to 10M records, it would take at most 24 loops, and for 100M 27 loops. That's why I can say that size does not matter much.

There are a couple of ways to solve this:
