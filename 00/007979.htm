gina whipp say:
Well, not sure only two tables are going to do it.  Where are you going to store the selection?  You would need a junction table that would *join* the two tables together and store the selections you made.  (The Form does not *join* the tables together it's just something for Users to use. Like a window allowing the view but it is not actually the view.) You're also going to need a Form so you could add some code or even a macro to push it out to Outlook for eMail.

ken sheridan say: As Gina points out, you need a table which models the many-to-many relationship type between the two tables by resolving it into two one-to-many relationship types.  The form would be bound to this table and include combo boxes, bound to the foreign keys which reference the primary keys of the two tables.

However, the question arises as to whether Points is an attribute functionally determined solely by the key of the referenced table, or whether it is also functionally determined by the key of the 'junction' table which models the relationship type?  The latter would be the case if the value of the Points column in the referenced table could change over time, but the value of the points as an attribute of the relationship should remain fixed as that current at the time when the relationship was created, i.e. when the row was inserted into the 'junction' table .  In this case the 'junction' table should also have a Points column, and the current value of the Points column in the referenced table should be assigned to the Points column in the 'junction' table when a row is inserted into the latter.  This is analogous to the very common situation in which the current unit price of a product is inserted into a column in an OrderDetails table or similar.

To output the relevant data via an email you can do so as a PDF file by calling the SendObject method of the DoCmd object.  The data would be output as a report, whose RecordSource should be restricted to the row or rows in question by one or more parameters, as the SendObject method, unlike the OpenReport method, does not have a WhereCondition argument.

Hi Gavin,

 

You can set the default location for the OneDrive folder via user configuration policies. First, you need to download the OneDrive for Business administrative template.

 

To use this policy, you must specify your tenant ID and the desired default path in Group Policy Editor. This policy sets the following registry key to a string that specifies the file path.

 

[HKCU\SOFTWARE\Microsoft\OneDrive\DefaultRootDir] "{Tenant ID}"="{User path}"

 

If you enable this setting, the local OneDrive – <tenant name> folder location will default to the path that you specify in the OneDrive ADMX file.

 

If you disable this setting, the local OneDrive – <tenant name> folder location will default to %userprofile%.

 

If you need to apply this setting to more than one tenant, enter the additional tenant IDs to the desired default path entries in Group Policy Editor.

 

Note: The %logonuser% environment variable won't work through Group Policy. We recommend you use %username% instead.

 

For more information, you can refer to this article: Use Group Policy to control OneDrive sync client settings.

You can choose up to three mobile or landline numbers worldwide, or you can choose to have calls forwarded to another Skype contact in your contact list. Just select add another number to add more phone numbers or another Skype account.
