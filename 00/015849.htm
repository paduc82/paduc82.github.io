To do it that way I'd suggest using a check box, chkAttended say, bound to a column in the form's query.  The column should return True if a row exists in the attendances table for the event/client/date in question, False otherwise.  Set the Enabled property of the check box to False (No).   Place a transparent command button over the check box and execute the relevant code in its Click event procedure. 

 

Without knowing the details of your Attendances table I can't be sure exactly what the code would be, but if we assume for this example that the table has columns EventID, ClientID and AttendanceDate, the value for each of which is represented in the currently selected row in the form,  the code would be like this:

 

Dim strSQL As As String

 

' if chkAttended is currently False then insert a row into

' Attendances table, otherwise delete relevant row

If Not Me.chkAttended  Then

    strSQL = INSERT INTO Attendances(EventID, ClientID, AttendanceDate " & _

        "VALUES(" & Me.EventID & "," & Me.ClientID & ",#" & _

        "Format(Me.AttendanceDate,"yyyy-mm-dd") & "#)"

Else

    strSQL = "DELETE * FROM Attendances " & _

        "WHERE EventID = " & Me.EventID & " AND " & _

        "ClientID = " & Me.ClientID & " AND " & _

        "AttendanceDate = #" & Format(Me.AttendanceDate,"yyyy-mm-dd") & "#"

End If

 

CurrentDb.Execute strSQL, dbFailOnError

 

 The form should refersh automatically, changing the value of the check box, but if not add a line:

Me.Refresh

ken sheridan say:

Firstly, it helps if you post the actual table name rather than something like 'main table'.  I assume this is the table of clients, so I'll call it Clients.  The query for your form would be:

SELECT Clients.*
(Attendance.PID IS NOT NULL) AS Attended
FROM Clients LEFT JOIN Attendance
On Clients.PID = Attendance.PID
ORDER BY BusNumber;

Include controls bound to whatever columns you wish to show from the table and the disabled chkAttended check box bound to the computed Attended column.  Add the transparent command button over it.  It's code would now be:

Dim strSQL As As String

' if chkAttended is currently False then insert a row into
' Attendances table, otherwise delete relevant row
If Not Me.chkAttended  Then
    strSQL = INSERT INTO Attendance(PID, [Date]) " & _
        "VALUES(" & Me.PID & ",#" & _
        "Format(Date(),"yyyy-mm-dd") & "#)"
Else
    strSQL = "DELETE * FROM Attendance " & _
        "WHERE PID = " & Me.PID
End If

CurrentDb.Execute strSQL, dbFailOnError

Me.Refresh

This will insert a row with the current date if there is currently no matching row in Attendance, or delete any matching row(s) if the check box is unchecked.  Is this what you want to do?

But I'd strongly advise against using Date as a column name.  As the name of a built in function it is a 'reserved word' and consequently should not be used as an object name.

ken sheridan say: Firstly, personal names should never be used as keys, even in combination with other columns.  I was once present at a clinic where two patients arrived, both female, both with the same first and last names and both with the same data of birth.  Always use a distinct numeric primary key, usually an autonumber, e.g. ArtistID

Secondly use separate FirstName and LastName columns.  You can then order a form's underlying query or rows in a combo box or list box by LastName, FirstName.

Use a combo box in the referencing table's form, bound to a foreign key ArtistID column, but set up to hide the bound column and show the name.  To insert a new artist into the referenced Artists table use the combo box's NotInList event procedure.  You'll find examples in the file NotInList.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

The combo box in the first (unbound) form in this little demo lists the names in last name order, concatenating the first and last names into a single column, so you can type in Jack Jones for instance.  Entering a new name invokes the controls' NotInList event procedure, whose code opens a form to which the names are passed as the OpenArgs property.  The named is parsed into separate first and last names in the second (Contacts) form's Open event procedure.  Other data can then be entered in the form.  Note that if no other data than the name is entered, however, the new record is not saved.  Try entering Jack Jones in the opening form's combo box and you'll see how it works.
