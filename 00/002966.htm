rohn007 say:

If the fixes for the 2 year old problem did not work, then it is NOT the same problem. It may have similar symptoms.

Printing is not "simple". It is a witches brew of stuff coming together:

  • application, ie Excel, sets up the content to be printed
  • the print command calls upon the printer driver to understand the "language" that has to be used to tell the printer how to put the content on the output device: Inkjet printer, pdf, laser printer, B&W or Color, plotter etc
  • the application then sends the printer language to the Windows Print System.
  • The Windows Print system plays traffic cop, getting the printer output from the application to the right device
  • FINALLY, the printer translates the printer commands into the mechanical actions required to create the final output (OK, no mechanics required for a PDF)

After all that long spiel, when you say the PDF's work OK, but printed outputs don't that strongly suggests to me that Word is OK, the problem is in the Printer Driver and / or the Window Print System.

MS is constantly (dis)"improving" Windows, in ways that often require new "drivers", like printer drivers, to work properly. 

You say the printer drivers are up to date and not corrupt because you can output PDFS.  You are making more of an assumption than is real.  The printer drivers used to make a PDF and a printed copy are totally separate, totally independent!  So, PDF can work fine, while the paper printer driver could be out of date or corrupt.

Please go to the printer maker site, check their support area to see if any one else is having the same problem.  Download and install the current version of the printer driver for your model.  Look around to see if there are alternatives.  For example, at one point my HP printer could work with AT LEAST 3 different printer drivers: HP PCL5, HP PCL6 and a Postscript driver.  As far as the application and Windows are concerned they are all the same. As far as the user is concerned there is very little to no difference between the outputs.  Check to see if you have similar options. You can actually install all of the drivers at the same time since they have different names. They show up as separate "printers" in the Printer dropdown, Windows knows where to send the output (correctly).

ken sheridan say:

This is how the Design view of the table looks, but the actual inputs are text input

This suggests that the column is a numeric foreign key referencing the primary key of a ProcessCapabilities table or similar.  You might even have used the 'lookup field' wizard when designing the table.  If so, that is unfortunate, as most experienced Access developers recommend that this feature be avoided like the plague.  For reasons why see http://theaccessweb.com/lookupfields.htm.  Don't be unduly alarmed at this, however, it's implications are not fatal, but take note for the future.

The use of a numeric foreign key to reference an autonumber primary key of another table is, per se, perfectly valid, and very common.  It does mean, however, that to select a text value as a criterion for a query, the actual value used by the query must be the number corresponding to the text value.  This is usually done by using a combo box in an unbound dialogue form as the parameter, and setting up the combo box so that the user selects from a drop down list of text values, or types in the text value, but the value of the combo box is the corresponding number contained in a hidden column in the control's RowSource property.  The column is hidden by setting the control's ColumnWidths property to zero, and its ColumnCount property to 2.

You'll find an example of this in DatabaseBasics.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

In this little demo file the section on 'Retrieving data from the database' includes a form in which the user can select a city and/or an employer in combo boxes to restrict a report to matching contacts.  This is done by the report's RecordSource query being the following query, which references the combo boxes as parameters:

SELECT [FirstName] & " " & [LastName] AS FullName, Address, City, Region,
Country, Employer, LastName, FirstName, Contacts.ContactID
FROM (Countries INNER JOIN Regions ON Countries.CountryID = Regions.CountryID)
INNER JOIN (Employers INNER JOIN ((Cities INNER JOIN Contacts
ON Cities.CityID = Contacts.CityID) INNER JOIN ContactEmployers
ON Contacts.ContactID = ContactEmployers.ContactID)
ON Employers.EmployerID = ContactEmployers.EmployerID)
ON Regions.RegionID = Cities.RegionID
WHERE (Cities.CityID = Forms!frmReportDialogue!cboCity
  OR Forms!frmReportDialogue!cboCity IS NULL)
AND (Employers.EmployerID = Forms!frmReportDialogue!cboEmployer
  OR Forms!frmReportDialogue!cboEmployer IS NULL);

Note how the criteria are in each case defined as  (Column = <parameter> OR <parameter> IS NULL).  This makes the parameter optional, so the user can leave the combo box empty if they don't wish to restrict the report on that particular attribute.  The parentheses around each OR operation are crucial to force them to evaluate independently of the AND operation.

Don't be put off by the SQL.  You can design the query without any criteria in query design view in the usual way.  You can then switch to SQL view and add the WHERE clause.  It can all be done in design view, but the logic of the WHERE clause is very simple, so it's an easy task to write it in SQL.  The above only references two controls as parameters, but it's easy to add more by writing additional parenthesized OR operations and tacking them together with the AND operator as above.  Be sure to then save the query in SQL view.  If you switch back to design view before saving it Access will move things around.  At best the logic will be obscured, but at worst the query could become too complex to open.  When writing queries in SQL I copy and paste the SQL statement into a text file, using Notepad for instance, as I go along.  That way I can always copy and paste the SQL statement back into the query in SQL view if something goes wrong.
