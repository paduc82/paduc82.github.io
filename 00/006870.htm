scottgem say:

First, your database should be split into a back end (tables) and a front end (everything else), you can have multiple back ends if it gets too large. You can also have multiple front ends if you want one application to have different functionality.

You are limited to a maximum of 2Gig in file size. Can you check the size of all your files and see what that totals to?

Second, yes you should have had a primary key from the beginning. A relational database requires that you use unique identifiers. Even if you did not have some natural key or combination of field to create a key, you should still have used an Autonumber.

Your main problem now, is that, even though you have added a Autonumber PK, because you didn't have a PK in the beginning, you have nothing to tell which child record goes with what parent. Before you can go any further, you need to make sure every table has a Primary key and that every table that relates to another table has a foreign key that identifies the related record.

ken sheridan say: Your database is not large in terms of the number of rows in each table, so should be well within the capabilities of Access, and with a good design should in itself perform adequately.  A sine qua non is, as Scott says, the splitting of the database into separate front and back ends.

On the basis of your description of the table structures, the only possible candidate key at present for AllBurials would appear to be a composite one of Name and Burial Location, being the only two columns in Pre-Needs.  Whether these two columns in reality constitute a candidate key on which the tables can be related only you can say, but if they do then it would be possible add an autonumber column, BurialID say, to AllBurials as its key, and update corresponding foreign key columns in Deeds and Pre-Needs, enabling you to delete the redundant Name and Burial Location columns from Deeds and Pre-Needs.

As regards the incorporation of photographs and other ancillary documents, you can do this by storing the path to the location of such documents as text in columns in a table or tables the database, and building in the means to open them from within the database's front end on demand.

Performance issues arising from your network infrastructure are a separate matter of course.

scottgem say:
First, having tables for each year is not good design. it violates the first rule of relational databases that data should be defined only by values in rows and columns. You are defining data using a table name.

You should have ONE table with a field that indicates the year. Either an actual date field if a date is part of the data or a fields for just the year.

You can then, if you want, have separate forms, that are filtered for the year. But a better way is to have an initial form where you choose the year, then open your main form filtered for the year. If you designed your database properly, then you proabaly would not have your issue.

ken sheridan say: You should not have separate tables or forms for each year; data for all years should be in a single table.  If the table in question contains a date column (field) you can easily filter the form on the year of the date, if not then you should have column with the year value.  Don't name the column Year, however, as that is the name of a built in function, so is a reserved keyword.  Name it something like TransactionYear or whatever is appropriate.

The single form should be bound to the table, and can either be opened filtered to a specific year, or you can include an unbound combo box in the form header to select the year, with code like this in its AfterUpdate event procedure:

    Me.Filter = "TransactionYear = " & Me.cboSelectYear
    Me.FilterOn = True
    Me.TransactionYear.DefaultValue = """" & Me.cboSelectYear & """"

The behavior you described occurs only if the entry being made in the Cell Range field isn't being made correctly... The best way to avoid the issue is to not have to putt anything in that field in the first place :-)

Ideally, the recipient list should be the only content on the worksheet. The field names (aka captions, headings, etc.) should be on Row 1 beginning in cell A1 with the list of records starting directly below them on row 2. If that layout is used you should need to do nothing in the Open Workbook dialog window other than select the correct sheet name... That isn't even necessary if the sheet is the foreground sheet in the workbook or if the list is the only content. Word will find the correct cell range in most cases

The Cell Range field needs to be used only if you want to indicate a specific range of cells where the records are located in the source workbook file.
