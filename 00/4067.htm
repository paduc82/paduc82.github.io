qa guy commelec say:

In Access, I have an unbound text box on a form that is populated from a another field on the form.

A text box must be bound to a field of a table - almost always via a query.

ken sheridan say: Can you explain in more detail what the value in the unbound text box represents.  If it is a value computed from other values in the same row (record) e.g. a control which shows the retail gross price of a quantity ordered of an item in an OrderDetails table or similar by means of an expression such as Quantity*UnitPrice*(1+TaxRate), then that price is always computable and should not be stored in the row in the table.  To do so would introduce redundancy and the risk of update anomalies.  The value should always be returned where necessary by means of a computed column in a query or a computed control in a form or report.

If, on the other hand, the value is derived from the value of another column in the row, e.g., staying with the above example, where the UnitPrice and TaxRate are both derived from the ProductID value by referencing the UnitPrice and TaxRate columns in a Products table, then, because, while both of those values in Products can change over time, the values in OrderDetails must remain fixed as those at the time of the order, so the derived values should be assigned to controls bound to UnitPrice and TaxRate columns in OrderDetails by means of code in the AfterUpdate event procedure of the control bound to the ProductID column.

You'll find an example in InvoicePDF.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

In this little demo file, the values are assigned to the controls as follows by referencing the Column property of the ProductID combo box in the invoice details subform:  

Private Sub ProductID_AfterUpdate()

    Dim ctrl As Control
    
    Set ctrl = Me.ActiveControl
    
    Me.UnitPrice = ctrl.Column(1)
    Me.TaxRate = ctrl.Column(2)
    
End Sub

No redundancy is introduced because the UnitPrice and TaxRate columns in the Products and OrderDetails tables are in each case functionally determined solely by the primary key of the table in question.

On the other hand, the NetAmount control in the same subform is bound, not to a column in a table, but to a computed column in the subform's query, for which the expression is:

[InvoiceDetails].[UnitPrice]*[Quantity] AS NetAmount

In this particular case the VAT (Value Added Tax) is not added in each row because the invoices are not for retail sales, so the VAT is computed and added at invoice level.
