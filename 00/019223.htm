In order to set up the hammock task you need to copy the start date of the swap task and paste link it to the start date of the support task AND also copy the finish date of the swap task and paste link it to the finish date of the support task. tom_bpc say:

John, please pardon the intrusion.  

I haven't trusted these paste-special links since they were first introduced, when they were extremely flaky.  Nevertheless, during recent testing I've managed to convince myself that the modern OLE hammocks are not so subject to corruption provided two cardinal rules are followed:

  1. You build them one at a time.  (i.e. never copy/paste tasks with links), and
  2. When the link becomes non-responsive or wrong, force an update using the Links window that you open with the Edit Links command (after adding the command to one of your ribbon groups through the "Customize the Ribbon" dialog.)  This one usually results in a correct result, but it can be frustrating to have to resort to it.

My primary objection to OLE hammocks remains the inability to easily examine, check, or modify them.  Even with trusted notes, you're never quite sure what is controlling their dates until you delete and re-build the links.  In addition, a minority of users might object to how they handle Fixed-work tasks (work is not fixed).

Coincidentally, my last blog entry was written to share an alternate hammock approach with a younger colleague.  It uses SS and FF predecessor links along with a "hammock" flag and a macro to tune the duration.  It's not as robust as a dedicated hammock task type might be, but its the best I've got.  An Alternate Approach to Hammock Tasks in Microsoft Project.
