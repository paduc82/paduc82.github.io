anannya podder say:

Hi 1mycoke55,

Thank you for writing to Microsoft Answers Community Forums.

Reportedly, this problem occurs as Internet Explorer in Windows Vista runs under protected mode (low integrity level) by default, and therefore will only be able to modify objects marked with a low integrity level in their system ACLs. When you relocate the Favorites to a different drive, you need to set the integrity level for the relocated folder, so that Internet Explorer (Protected mode) can write to it. This problem does not seem to occur if the Favorites folder is located within your user profile home directory.

Try the steps below

1. Click Start, type shell:favorites and press ENTER. The Favorites folder of your user account will open. Copy the full path shown in the Address Bar.

 

2. Open an elevated Command Prompt. To do so, click Start, click All Programs, click Accessories, right-click Command Prompt, and then click Run as administrator.

 

3. Type the following command:

 

icacls <favoritespath> /setintegritylevel (OI)(CI)low

 

Note: Replace the tag <favoritespath> with the actual path to your Favorites folder, enclosed with double-quotes)

 

Example:

 

icacls "D:\My Favorites" /setintegritylevel (OI)(CI)low

 

You should now see this message:

 

processed file: favorites

 

Successfully processed 1 files; Failed processing 0 files

 

4. Optionally, to verify if the Integrity Level is correctly set for the Favorites folder, run the command below:

 

icacls favorites

 

You should see the following line in the output:

 

Mandatory Label\Low Mandatory Level:(OI)(CI)(NW)

 

5. Type EXIT to close the Command Prompt window.

 

Hope this helps.

Let us know if this does not work or if you need further assistance. 

 

Regards,

Sharath

Microsoft Answers Support Engineer

john a.hart say:

Word Count

How do I get Word not to include certain text (such as a cover page) in a document word count? And, how can I make the word count field automatically update when I reopen a document? The date field does, but the word count field remains static.

macropod say:

You can't do either of those things without:

a) using a formula field such as {={NUMWORDS}-100}, where 100 is the number of words on the cover page; and

b) doing something that causes the field to update (e.g. putting the field into a page header so it updates when you do a print preview) or using a Document_Open macro that updates fields in the document.

The NUMWORDS field is not designed to update the same way a DATE field does.

greg maxey say:

In addition to Paul's methods and if you can use a macro enabled document consider this approach.

First NUMWORDS is horribly inaccurate and gets more so the larger the document becomes. 

1.  Create a document variable to store the calculated number of words  and use a DocVariable field instead of a NUMWORDS field in the document to display the value.

2.  I would wrap my coverpage text in a richtext content control titled "Cover" and use this code:

Sub Document_Open()
  ScratchMacro
End Sub

Sub ScratchMacro()
'A basic Word macro coded by Greg Maxey
Dim lngWords As Long
   With ActiveDocument
     lngWords = .Range.ComputeStatistics(wdStatisticWords) - _
               .SelectContentControlsByTitle("Cover").Item(1).Range.ComputeStatistics(wdStatisticWords)
     'Depending on where the DocVarialble Words field is located and if you want it in the count
     'lngWords = lngWords - 1
     .Variables("Words").Value = lngWords
     .Fields.Update
   End With
lbl_Exit:
  Exit Sub
End Sub

Process all files in a folder and, optionally, in sub-folders

A frequent request one runs into is for some way in which one can apply some change to all the files in a folder.  This case presents a modularized solution.  While the implementation may look complex -- it does use a recursive routine to process sub-folders -- the result is a "black box" that should be used without any modification.

An example of how one would use the code is below.  The ListAllFiles subroutine is the main code routine.  It calls the "black box" module named searchForFiles with the appropriate arguments, one of which is a 'callback' routine.  In this example, the callback routine is named processOneFile.

Sub ListAllFiles()
    searchForFiles "C:\tushar\temp\", "processOneFile", "*.*", True
    End Sub
Sub processOneFile(ByVal aFilename As String)
    Debug.Print aFilename
    End Sub

Copy the code below into a standard module in your Visual Basic project.  At the bottom is the example from above.  The solution uses one application specific method, the Run method.  Consequently, it can be used on any platform that supports the Run method.  These applications include -- but may not be limited to -- Excel, Word, and PowerPoint.

Option Explicit
'A modularized solution to process all files in a folder, and optionally all subfolders in the folder _
    Tushar Mehta
 
'There should be absolutely no reason to modify the searchForFiles subroutine.  Treat it as a blackbox _
routine.  Do *not* tweak it for each specific search.
'How to use the subroutine: _
Call it with the four arguments: _
    DirToSearch: The directory you want to search.  Note that it must *not* end in a path separator _
        ( "\" on a Windows OS) _
    ProcToCall: This is the callback procedure called with the full name of each file found _
    FileTypeToFind: This is a search pattern for the files sought.  For example, to find all Excel _
        files use "*.xls".  This argument is optional and defaults to "*.*" (or all files) _
    SearchSubDir: Boolean that specifies whether or not to search nested folders.  The default is False.
        
'The callback subroutine is where you process each file found.  The signature for the routine should be _
     Sub {subroutine-name}(ByVal aFilename As String)
'Use this callback subroutine to do whatever it is you want to do with each file found.  For an example, _
see the processOneFile subroutine below.
 
Sub searchForFiles(ByVal DirToSearch As String, ByVal ProcToCall As String, _
        Optional ByVal FileTypeToFind As String = "*.*", _
        Optional ByVal SearchSubDir As Boolean = False)
    'by Tushar Mehta
    'This subroutine recursively calls itself if SearchSubDir is true and at least one sub-directory exists. _
     There should be no need to make any changes to this routine for any specific search.
    On Error GoTo ErrXIT
    If Right(DirToSearch, 1) = Application.PathSeparator Then _
        DirToSearch = Left(DirToSearch, Len(DirToSearch) - 1)
    If SearchSubDir Then
        Dim aFolder As String, SubFolders() As String
        ReDim SubFolders(0)
        aFolder = Dir(DirToSearch, vbDirectory)
        Do While aFolder <> ""
            If aFolder <> "." And aFolder <> ".." Then
                SubFolders(UBound(SubFolders)) = aFolder
                ReDim Preserve SubFolders(UBound(SubFolders) + 1)
                End If
            aFolder = Dir()
            Loop
        If UBound(SubFolders) <> LBound(SubFolders) Then
            Dim I As Long
            For I = LBound(SubFolders) To UBound(SubFolders) - 1
                searchForFiles _
                    DirToSearch & Application.PathSeparator & SubFolders(I), _
                    ProcToCall, FileTypeToFind, SearchSubDir
                Next I
            End If
        End If
    Dim aFile As String
    aFile = Dir(DirToSearch & Application.PathSeparator & FileTypeToFind)
    Do While aFile <> ""
        aFile = DirToSearch & Application.PathSeparator & aFile
        If (GetAttr(aFile) And vbDirectory) = vbDirectory Then
            searchForFiles aFile, ProcToCall, FileTypeToFind, SearchSubDir
        Else
            Application.Run ProcToCall, aFile
            End If
        aFile = Dir()
        Loop
    Exit Sub
ErrXIT:
    MsgBox "Fatal error: " & Err.Description & " (Code=" & Err.Number & ")"
    Exit Sub
    End Sub

This is not a defect, but an anomaly of 64-bit binary floating-point, which Excel uses to represent numbers and perform arithmetic.

More generally, it is a well-known property of any arithmetic that has finite precision, notably all computer arithmetic, in contrast to mathematics, which has no limit on precision.  This is the subject of the field of study called Numerical Analysis.

To add to the confusion, if we implement all three formulas in VBA with type Double variables (not type Variant), we get the same result, namely:  0.00950084395720741.


I assume this is just a curiosity, and the difference does not really matter to you.

If it does, you should explicitly round at least the final result to the degree of precision that you truly require.

Moreover, instead of constants with 15 significant digits, you should enter the formulas that generate them, if you can.  For example, I presume the prices (B3 and B4) are derived from a NAV, which is usually accurate to 2 or 4 decimal places, divided by a number of fund shares, which is usually accurate to 2 to 8 decimal places.

The reason is:  there are probably 2 or more non-zero decimal places that we cannot see because Excel formats only up to 15 significant digits.

For example, what appears to be 108.497457761148 in B3 might be any of 71 more precise values from 108.497457761147,501287268823944032192230224609375 to 108.497457761148,49604709888808429241180419921875.  FYI, 108.497457761148 is represented internally by 108.497457761147,9986671838560141623020172119140625.

For now, I must use the constants that you provide.  But understand that the actual values in your Excel file might differ infinitesimally.  For a more precise explanation, upload an example Excel file that duplicates the problem to a file-sharing website, and post the public/share URL in a response here.

------

As you might know, Excel represents numbers using 64-bit binary floating-point, one of the standard representations supported by the CPU in your PC.

Consequently, most non-integers cannot be represented exactly.  Instead, they are approximated by the sum of 53 consecutive powers of 2 ("bits") times a binary exponential factor.

Moreover, most PCs have Intel-compatible CPUs.  They perform arithmetic using an 80-bit binary floating-point representation, which is the sum of 64 consecutive powers of 2 times a binary exponential factor.

VBA tries to retain intermediate arithmetic results in the 80-bit form.  But Excel rounds the result of each intermediate operation to the 64-bit form.  That is why there might be differences between VBA and Excel arithmetic.

In either case, the final arithmetic result depends on the order of operations and the relative magnitudes of the intermediate operands because the result of each intermediate operation is truncated or rounded to the finite number of bits in the internal representation (80-bit or 64-bit).

In your example, B2 is about 1.11363636363636 and B3 is about 108.497457761148.  Their magnitudes (binary exponent) differ by 6 bits.  B2-B3 can be represented in the 80-bit form (VBA) with no loss of precision.  But we might lose some precision when Excel rounds back to the 64-bit form.

(I say "might" as a generality, because precision is lost only if the rounded bits are non-zero.  In this example, they are.)

In contrast, B4 is about 108.414638813454, which is the same magnitude as B3.  So no precision is lost when we calculate -B3+B4 (B4-B3), even when Excel rounds the intermediate result to 64 bits.

Numerically speaking, it is better (more accurate) to group calculations with similar magnitude, if possible.  So B4-B3+B2 is more accurate than B2-B3+B4.

Finally, division might result in an unpredictable number of additional bits of precision, if not for the finite limitations.  So (a+b)/c can be very different from a/c + b/c in computer arithmetic, even though they are the same mathematically.

So numerically speaking, it is better (more accurate) to group addition and subtraction before division, as you did, instead of simplifying the calculation algebraically, as I did with my third formula.

On the other hand, do you really need to care (or worry) about such infinitesimal differences?  Usually not.
