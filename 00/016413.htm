suzanne s. barnhill say:
You are strongly discouraged from using a universal Normal.dotm for this purpose. That file is meant to be the individual scratch file for every user, to save personal settings. It is not unreasonable to want to establish custom corporate fonts for every document, and this is not easily achieved just by creating document templates (which would be the usual recommendation). Instead, your best bet would be to provide instructions to all employees on how to change the default fonts in their own copies of Normal.dotm. Alternatively, you could provide a corporate style set or custom font theme that could be added to their copies of Normal.dotm. This can be saved as a .thmx file that could be distributed to users.

charles kenyon say:

Sharing of the normal template on a network is strongly discouraged by Microsoft. You Cannot Share the Normal.dot[m] File Among Multiple Users This template is intended to be a repository for personal modifications. There are many ways to distribute customizations across a network. Base fonts for headings and text are at the heart of Themes in Word. Overwriting the normal template is perceived by many users as a virus-like activity.

The normal template is always located in the user templates folder. This should not be the same folder as the Workgroup Templates folder.

One of the best ways to have consistent formatting is to develop a consistent set of Workgroup templates that are easy for users to access and to use. Themes can help build this consistency.

All versions of Word since Word 2000 have made access to the custom templates on a computer more difficult than it was in Word 97-2000. That can easily be changed by adding the File New dialog to a keyboard shortcut or the QAT. This can be done through a macro-free Global Template distributed over the network. See File New Variations in the Versions of Word. You can download such an Add-In here or develop your own.

jay freedman say:

Martin,

One other comment that may or may not apply to your situation...

People who design corporate styles, especially those who are professional designers, have an unfortunate tendency to be creative by specifying uncommon fonts. In templates for Word and other Windows programs, that creates endless problems. On computers where the required fonts haven't been installed, documents may be displayed with font substitutions. Because different fonts don't have identical character widths, that causes layout problems, ranging from changed line breaks and page breaks to strange overlays. Other documents may refuse to open at all.

Please encourage the designers to use only Windows core fonts, especially for documents that will be sent outside your organization. If the corporate logo uses an uncommon font, have it supplied as a graphic file rather than as a text object.

charles kenyon say:

Remember, start with the theme, then create good, usable workgroup templates for letterhead, memos & reports that use that theme.

If these incorporate a logo, consider using AutoText and AutoText fields to incorporate the logo in your templates. The AutoText would be stored in one Global Template. That way, if you change the logo, it changes in all of your templates. The same can be done with addresses and phone numbers. The latter works especially well when devoloping templates to use in multiple locations. You simply have a different core global template for each location to hold the local info.

To me, a useful template includes either MacroButton Fields or Content Controls so that people find it easy to insert information. If the template has a date, it uses the CreateDate field rather than a Date field so that documents created based on the template maintain their creation dates rather that updating every time they are opened. Having these features like this makes templates easier to use than reusing old documents and encourages use. (So does putting the classic FileNew dialog on the QAT.)

Finally, with a Global Template, it is possible to distribute a Ribbon Tab that comes up with your custom templates and is the first thing people see when they open Word or create a document. Customize the Ribbon (It doesn't take rocket science)

In a corporate environment this would go much further to get people to use consistent styles than would all of the training and policies in the world. The screenshot below is from such a ribbon.

Such a ribbon tab can be stored in the same global template as the AutoText or a different one. Such a tab, if well set up, can drastically reduce training time and make it easy for people to use your templates.

Greg Maxey showed me how to make dynamic menus on the tab. Each of the buttons on the tab shown that have a little down triangle is such a menu. Everything except the refresh command in the menu below is simply a template in the folder for "Most Cases." Note the .dot and .dotx extensions.

Such customization can also remove commands from a ribbon. For instance, I do not want track changes turned on in my documents by accident and seldom use many of the reviewing features. The Review tab in my menus global template is abbreviated:

I put a button on it letting me restore that tab when I need the commands.

This level of customization requires some work and thought. Again, in a corporate environment, though, it is very much something that can be done in-house. (If you do it, in-house or with a consultant, make sure such customizations are well documented. I would recommend documenting what you do with themes and templates as well, so that future adjustments can be as painless as possible.)

bill manville say:

The problem is, as you say, a result of including the Sheet1! in the formula.

Sort sees the sheet reference and decides, wrongly, that it isn't a reference to a cell on the same row within the data that is being sorted, so it treats it like a reference to a cell outside the area being sorted and moves the formula unchanged along with the item to its left.

I agree that this is annoying and it annoys me too that the sheet name is automatically included when you have visited another sheet earlier in the formula and then return to the sheet on which you are entering the formula.

The only solution I know is to manually remove the Sheet1! from the first formula before copying it down to the rest of the table.
