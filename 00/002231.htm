booklet_3g say:

Mentioned in but this probably merits a thread of its own:

Does an SD just need some special format, or can the Booklet 3G's BIOS really not recognize it at all (though even the earliest EeePC could) ?

myroslav---01 say:

Unfortunately booting solely from SD card is not possible in Booklet 3G. SD card controller is not USB device but attached to PCI bus. BIOS is able to boot only from USB devices. Some Netbooks have their SD card readers attached as USB devices thus allowing to boot from SD card similarly to any other USB stick.

 

The only way is to load enough of the OS from HDD/USB-stick to have PCI driver for SD card reader and then continue booting from SD card. For Linux that would mean that bootloader and /boot partition with initrd (having PCI SD card support) should be on HDD/USB medium and root partition can be in SD card. For Widows things are trickier. You;d have to read a lot about Windows PE and have a lot of time to craft the solution.

 

Anyway for OS to be stored on SD card you need to have extra boot partition on HDD or USB medium, that BIOS of Booklet  is able to read.

0% Data calculating as 100% - Microsoft Community

brentlowrey say:

0% Data calculating as 100%

In using the formula =IF($S5>0,$G5/$S5,"100%") I end up getting a calculation of 100% even if the cell it's pulling from is 0 or blank. Any idea on how to rememdy this? Thanks in advance!

joeu2004 say: Brent wrote:
In using the formula =IF($S5>0,$G5/$S5,"100%") I end up getting a calculation of 100% even if the cell it's pulling from is 0 or blank. Any idea on how to rememdy this?

As your value-if-false expression demonstrations, you have the mistaken habit of entering numbers as text.  If S5 contains "0" instead of the actual number zero, S5>0 is indeed always true.

Also, if S5 appears blank because it contains the null string (""), S5>0 will be true as well.

If you correct any text "numbers" in column S, the IF expression should be written:

=IF(AND($S5<>"",$S5>0),$G5/$S5,100%)

formatted as Percentage.

But if you do not want to change the presumptive text in column S, you could write:

=IF($S5="",100%,IF(--$S5>0,$G5/$S5,100%))

 Actually, I said that your criterion was written as though the value were a
number, not text.  A text value has to be enclosed in quotes, *inside* the
string that provides the criterion.  You wrote this (correcting the field
name):
 
>    rs.FindFirst "[Expr1] = " & Str(Me![Combo78])
 
Now, there was no real need to use the Str() function, though you'll see it
in a lot of examples, but what you don't have there is any quotes
surrounding the value from the combo box, within the criteria string.
Suppose that the value chosen in the combo box was "Smith, John 1234".  Then
your statement would resolve to:
 
>    rs.FindFirst "[Expr1] = Smith, John 1234"
 
The database engine won't know how to parse that.
 
Instead, you should write your statement as this:
    rs.FindFirst "Expr1 = '" & Me!Combo78 & "'"
 or as this:
    rs.FindFirst "Expr1 = """ & Me!Combo78 & """"
 The first version wraps single-quotes (') around the value, so that it would
come out like this:
    rs.FindFirst "Expr1 = 'Smith, John 1234'"
 
That works fine in this example, because the value doesn't contain a
single-quote or apostrophe.  However, a name containing an apostrophe, like
O'Neill for example, wouldn't work -- it would have what looks like a
single-quote inside a single-quoted string, and that would again cause
parsing problems.
 
The second version I suggested puts double-quotes (") around the value,
which tends to be more reliable for names, since names don't generally
contain double-quotes.  However, because the whole thing is going inside a
double-quoted string, the internal double-quotes have to be doubled up in
order to be interpreted correctly.  For example, the literal """" is a
representation of a string containing just one double-quote character.
 
There are other ways to get double-quotes inside a quoted string.  For
example, you could write it like this:
    rs.FindFirst "Expr1 = " & Chr(34) & Me!Combo78 & Chr(34)
 That uses the Chr function to generate a quopte character, which is then
concatenated into the string.  I often do it that way, because the
doubled-up quotes of the other method are hard to read.
