jeeped say:
You need to compensate for W1 → X1 crossing midnight where X1 is the equivalent of 0 (00:00). Try,

=AND(F$1>=$C3,(G$1+(G$1<F$1))<=$E3)

jllatham say:

Just a side note - I see some #VALUE! errors on some rows.  If you want to keep the formulas in place, but get rid of the "uglies", wrap your formula in an error trap:

Instead of

=yourFormula()

try

=IFERROR( yourformula(), "")

that way it'll appear as an empty cell for you when the error would have come up, but will show the time properly when your formula actually does its job without breaking into tears.

jeeped say:
An improvement on that formula to compensate for shifts that cross midnight might be,

=AND(F$1+(F$1<OFFSET(F$1,0,-MIN($D3,COLUMNS($A:A)-1)))>=$C3,F$1+(F$1<OFFSET(F$1,0,-MIN($D3,COLUMNS($A:A)-1)))<$E3)

There is probably a more elegant way to proof that and some of the computation does depend on how you are arriving at E3; e.g. whether it is a typed 14:00 or =C3+TIME(D3,0,0).

How do I get my update query to work? - Microsoft Community

landon crews say:

How do I get my update query to work?

I have a table with many fields, but the ones in concern are -Quantity(kg), Input(kg), and Batch Size. I have a query with an expression: Expr1: [Input(kg)]*[Batch Size]/1000. I need this expression to update the Quantity(kg). The expression works just fine in select mode but when I use update mode and try to update the table I'm met with error: .......(the expr) is not a valid name. Make sure that it does not include invalid characters or punctuation and that it is not too long. 

What am I doing wrong and how do I get this query to update the table?
I'm thinking that I may have to make a table first and then use the update query to pull from that temp table. 

ken sheridan say: A computed value which is time-independently determined by values at other column positions in a row should not be saved in a column in the base table, but returned in the result table of a query, which you have apparently done successfully.  Otherwise redundancy is introduced into the table, and the table is not normalized to Third Normal Form (3NF) as the column is not solely determined by the whole of the table's primary key.  The table is consequently open to the risk of update anomalies.

Only where the computed value is not time-independently determined by values at other column positions should the value be saved, i.e. where the value can be legitimately edited to a value which is not a result of the expression.  I doubt that that's the case here, but if there is a cogent reason for doing so an UPDATE query should be straightforward:

UPDATE [YourTableNameGoesHere]
SET [Quantity(kg)] = [Input(kg)]*[Batch Size]/1000;
