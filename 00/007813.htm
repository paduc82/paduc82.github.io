What Features do I look for in a Backup Tool

1. Image copy backup - create a "snapshot" of your entire system at a given point in time. When you restore an image copy backup you have a working system from that point in time. These backups are relatively large and time consuming.  If there is an option to compress the backup file I select max compression. It adds a little time to backup, but saves a lot of space on the backup drive

2. Incremental Backup - identifies files that have been changed since the last image or incremental backup and only save them.  Much faster than image copy backup. During a restore is used in conjunction with prior image copy backup.

3. Backup to external storage - no longer much of an issue (was in the past). But it is important to be able to save your backup files to a drive other than the HD you are backing up.  If that HD dies, you don't want your backup files on it too.  A few years ago you could reasonably backup to DVDs (and before that to CD and "floppy" disks), but now given average HD size you can only reasonably backup to an external USB type drive.  This year you can buy between 4 to 8 TB external HDs for under US$300, even 1 TB drives are now under US$100. 1TB is now the minimum I'd suggest for image backup storage. It gives you enough room to save several backups.  The beauty of doing any backup to a large, 1TB or more, USB drive is that you can start and forget it until it is done.  In the past, using Floppies or CDs, you could be stuck at the machine for hours, feeding up to 40 disks to the machine to capture all of the backup. That was a real PITA, and a major reason why the average user never got into the habit of doing backups.

4. Bootable recovery disk/usb - In some situations Windows is still working so you can start the recovery from inside of Windows. But in the WORST CASE Windows won't run so you have to have a bootable DVD, or USB, with the recovery program on it.  From inside Windows, you can create a recovery disk and from most current backup programs you can do the same.

5. Selective file Recovery - while doing your recovery from image copies is the main feature you are looking for, it is more often useful to be able to look at your backup and extract individual files or folders that you want to recover. Say a file has been "lost", accidentally deleted or corrupted, then it is nice to be able to go into a backup and at least restore a slightly older version so that you don't have to start from scratch.

Backup Strategies

Backup strategies are defined by how much file change is happening on your system AND how important the data/files are to you.

A "full" image copy backup

requires more time to create, and to restore and more HD space. But it is also the easiest to restore.  Used for starting point of future recoveries, used for systems with much file change happening on them (ie after Windows Updates...).

Incremental Backup

require less time to create and restore, and less HD space.  Useful when you know there is not much file change happening on the system but you want/need to make sure the changes are not lost.

A General strategy I've used - Weekly backups

On a typical home system, where there isn't really much file change happening, I like to do an image copy once a month, followed by weekly incremental backups. The key question is when to do the full backup. Do you do it before the monthly Windows Updates, or after?  It depends on how optimistic/pessimistic you are.  If you believe MS is good at doing update, then you probably want to schedule the image copy after Windows updates are applied. That way you have a known good point to return to including the latest updates.  If you are are a realist/pessimist, you do the image copy BEFORE Windows updates are applied. On the assumption that MS is going to F'up and you will need to restore back to a working system without the updates.

Since these backups take time, I find it easiest to schedule them on the weekend at night, when I'm less likely to be using the machine.  So, on the first weekend of the month I do the full image copy (right before "Update Tuesday"), then the other 3 (or sometimes 4) weekends I do incremental backups.

Optional Strategy

Given the relatively large HD sizes we now have to work with I have also adopted a secondary strategy. I take the C: drive and shrink it, then create a separate D: partition for my data files.  C: contains Windows and installed applications.  It is relatively static, for the most part the only changes are due to Windows Updates. And occasionally installing applications.   With this approach, you could even get away with only doing monthly backups of the C: drive.   Now the D: data "partition" / drive contains all of your personal data files.  I also move the C:\Users folder to the data drive because it contains most of the file change happening on your system.  So now you have all of the file change happening in an easy to identify location, that is much smaller that your full drive.  It is now easier and faster to run more frequent backups, full or incremental, on just the D: drive

There is an additional benefit from creating a separate Data drive.  If you do have to restore Windows for any reason, you no longer have to worry about losing your personal data files. They are no longer on the C: drive!

"Ideal" Backup System

Back in 1980's a computing pioneer defined an Ideal backup system using the mnemonic "3-2-1", a play on the popular "1-2-3 Lotus Spreadsheet" program (that essentially drove the creation of PC computing in business).

  • ·    at least 3 copies of any file,
  • ·    at least 2 different storage mediums (say on your hard drive plus at least one of the following: thumb drive, DVD or the cloud),
  • ·    at least 1 copy offsite -- in case of fire etc!  (ie the “cloud” meets this requirement, so does using a thumb drive moved to a different building, or a server in a different building)

    Steve Gibson

PS: yes “correct” the plural of medium is media, but I don’t speak latin! <grin>

You can do your backups manually or make use of automated tools.  A strategy I have used is timed backups during the editing session (5 minute). Then the final end of day save is copied to second medium. I rotate the “second” medium with an Offsite copy.  Once a week I go in and clean up the 5 minute incremental backups to keep just the end of day copy (to free up space!)

I have used the “autohistory” tool at codeplex. It is easy to use, but like any automated process it uses up a lot of space.

Unfortunately, although Office apps have been around since the the mid 1980’s, MS has not figured out that we need a good “Autosave” backup feature. In the past there was a “versions” feature in some office apps, but it only kept incremental copies of files, not full files. Win7 has a file “versions” feature, but it is tied into “System Recovery Points”. Not very useful.  The following links tell how to setup macros that run automatically and use SAVEAS to save full copies of the file with the date/time included in the file name. You have to manually delete the “extra” backup files, but you control the process rather than Windows or Office. It is a bit more work, but I find it worth it for very important files that are changing frequently.

Exception: the Windows 8/10 "File History" feature now takes care responsibility for of making frequent file backups of data files away from the applications and moves it into Windows.

 

Note:

Windows 8 / 8.1 and 10 have done a great deal to improve the Windows backup system. Using native features in those versions of Windows, even 'normal' home users can setup an "ideal" backup system that meets the 3-2-1 rule.

Hope this helps.  If you have any further specific questions, feel free to ask.

Manual method:

Before you close up shop and start working on this reg fix, open the version of Word that you will want to be your default. In other words, say you have two versions of Word on your system. You want to use 2007, but you want 2003 to be the version that opens by default when you click on a DOC file. Open Word 2003 before you insert the fix below. This will ensure that Word 2003 is the last version opened and it will, if necessary, reset the registry to that version. (More on getting around this default in the next section.)

· Now make sure Word (office apps) is closed

· Start regedit  <Win  R> and type REGEDIT

· Note that if this doesn’t do anything, you’re using a system where the administrator has taken away your ability to edit the registry so you don’t mess it up!

· Once within the registry, click the + (plus sign) to expand the /HKEY_CURRENT_USER/ folder.

· Scroll down and locate the /Software/ folder and expand that one. We have a long journey ahead, so hang in there!

· Once within the Software folder, scroll down that branch of the folder tree until you find the folder for /Microsoft/. Click the + to expand that one.

· Now scroll down to locate the /Office/ folder.

· There you’ll see the numbered versions of the Office versions you have:

· 2013 is 15.0

· 2010 is 14.0

· 2007 is 12.0

· 2003 is 11.0

· 2002 is 10.0e.

· This fix can be used for all recent versions. Or at least, I know it works as far back as 2000. I can’t confirm anything earlier. But if you’re still using a version older than 2000, speed is obviously not an issue for you!<grin> Expand the folder for the version you want to speed up. (Note, this speed up fix will only help you if you are running multiple versions. So if you only have one version of Office on your system. Don’t bother with this setting, as it won’t do anything for you.)

· Move to the /Word/ folder and expand that. Almost there!

· Okay, locate the /Options/ folder and select that one. This is the folder where you’ll be adding a new registry key.

· With the Options folder selected in the route we took above, click Edit > New > DWORD Value

· This will insert a new registry key into the Options folder. It will be highlighted in edit mode so you can go right ahead and start typing the name of this key: NoReReg

· Hit the Enter key after typing the name to have the name accepted.

· Now double click this new key to open it. The Value data will read 0 (zero) by default. Change that setting to the number 1

· click OK to accept that value.

Then close the registry. You did it!

So what is it that you’ve done? Well, you went into the Word registry settings for the version of Word you selected and you set a new registry key. The key is an option that stops Word from reregistering itself if another version is the current default. The value of 1 means Yes. So the NoRereg (no reregistering) is set to Yes...i.e., don’t do it! This will cause Word to ignore the default version and just open, without taking several minutes to go through the reregistering hassle.

error 5174 Sorry, we couldn't find your file. Was it moved, - Microsoft Community

fr4nc01s3514577 say:

error 5174 Sorry, we couldn't find your file. Was it moved, renamed, or deleted?

This discussion is for those writing macros using VBA with Word, Excel, etc and have faced this error when executing your macro:

Documents.Open (..)

error  5174 Sorry, we couldn't find your file. Was it moved, renamed, or deleted?
 ("drive letter:\...\filename...")

I have seen a lot of posts around this error. There are 2 main scenarios, and the easiest one is the only one that is getting discussed.

Scenario 1

You are trying to open a "path\file" name that has actually moved between the time your program started and came up with the "path\file" name value, and the time it actually tried to open the file. That is easy to fix.

Scenario 2

THE MOST IMPORTANT ONE AND MOST DIFFICULT TO DETECT

The value you are using to designate the "path\file" name to open using your macro code such as:

            Documents.Open (....

...said value contains non printable characters (I am not talking about non alphanum like "." or "&, or characters not allowed, I am talking about ASCII 13 most likely being the last character at the end of the value for your filename. But when you use Debug.Print, that character is not printed.

Steps To Confirm:

When you are in debug mode, Debug.Print "the value of the path\file you are trying to open".

This will print as expect with no visible issues, you can then confirm the file exists by copy pasting the printed value in windows explorer and the file should open automatically. If it does open manually, then you are likely dealing with scenario #2

THEREFORE

while in debug mode, using the variable that contains the path\file vallue, write a FOR statement to iterate through each character of the string using Debug.Print Asc(<>), where <> is the variable in the FOR iteration that gets assigned 1 character from the path\file string at a time. Most likely you will see the last character will print as 13

The VBA function

            Trim(....

does not take care of this problem. you need to use the 

             Replace (...

function with "Chr(13)" as the value sought to be replaced with vbNullString

There are some hacks that consist in adding ASCII char 13 (carriage return, without a line fee) to file names and then appending the name, etc

There are utilities that clean those issues up, maybe not completely?... this might be the cause.

Another possible cause in my radar is the DOS prompt adding an extra char 13 when redirecting the output of a DIR command using ">>"  into a file to create a list of files from the DIR command. It does not happen for all files in the results of the DIR command, it could be for files having special characters that are allowed. I do not have the time to investigate that further.
