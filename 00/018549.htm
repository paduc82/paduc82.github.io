You have a Public declaration Public Player_Array().

Then you have Dim Player_Array in the Sub code.

Therefore the array that you were populating was the one in the sub in Module1 and not the Public array that was available to the Userform and the array was empty.

The code in the sub should be ReDim Player_Array(15, 2)

Another problem is in the example you will not see anything in the combo dropdown because you were populating the second dimension of the array in the following line of code. You have Option Base 0 and therefore all dimensions begin counting at zero.

Player_Array(iLoad - 5, 1) = Cells(iLoad, 1)

Should be Player_Array(iLoad - 5, 0) = Cells(iLoad, 1)     'Note the bold and underlined zero

I don't use Option Base. I prefer to dimension arrays as one based and I use the following method where Option Base is not required and the elements start counting from 1.

Dim Player_Array(1 to 14, 1 to 2)

albert d. kallal say:

I would suggest you remove all of the parameters from the query, and try using a where clause in the openReport command of your macro.

 

Your macro thus looks like:

 

openReport

 

ReportName: MyReprotName

View:  Print Preview

 

Where Condition: (id = forms!NameOfMyForm!NameOfTextBox)

 

In other words get the report running without any prompts and parameters at all FOR TESTING don't use your form to launch the report. Run + test the report from the reports tab area in access.

Once get the report working when you launch the report from access (not your custom form), then we move to the next step of making your form work. Of course in this scenario, the report will thus display and show all records in the table or query that the report is based on.

 

ONLY AFTER you have the above working, THEN build a little prompt form in which the user can type in some values into those text boxes. 

You then use the above type of where condition in the OpenReport command. This is a perferred approach since it means the report can be run without any prompts and from the Access interface AND also that of your custom form.

 

In the above example, you only see one condition, but you can add several more within and between them such as

 

Where Condition: 

(InvoiceId = forms!NameOfMyForm!NameOfTextBox and City = forms!NameOfMyForm!NameOfTextBox)

 

At the end of the day, you'll find the above approach better since there is now no conditions inside of a query. If you use conditions inside of that query, you'll find in that query will always prompt you for those parameters, and you might want to have a couple different parameters in different forms. The above approach allows you to choose + place the conditions in the form.

 

Albert D. Kallal (Access MVP)

Edmonton, Alberta Canada

todtown say: Thanx for info. I saw your post last night but I'd already ended for the day. Your detail gives me enough info to get us started.

Considering that you will be having someone else doing this each month, then we need a simple process that is as automated as possible. To reference your totals (instead of copying the data), considering that the sums could be in a different row in each sheet, it would be best to use formulas on your summary sheet that do the totals based on the data. 

So, this is just a general first idea. I would create a your summary sheet like this. Put in all of the possible plant names in the first column and months across the top in the firsts row. So now you have a grid for summarizing your data. The line chart is going to point to this grid. In the middle will contain your formulas that reference the plant name, the month and the data from the monthly sheet. Here's an example of the layout:

          Jan-13  Feb-13  Mar-13
PlantA
PlantB
PlantC
Sums  

For Cell B2 I will need to get the totals from the Jan-13 worksheet where the plant is PlantA. Let's say the monthly sheet is named Jan13Sheet, the plant names are in Column A of Jan13Sheet and the values I want to sum are in Column B. (You can adjust this example to fit your situation.) The formula would be:

=SUMIF(Jan13Sheet!A:A,A2,Jan13Sheet!B:B)

This looks for the plant name in Column A of Jan13Sheet and sums the values adjacent to that plant name. 

Do this formula for each Cell in the grid. You may need to go ahead and add sheets for each month of the year, even if you haven't gotten there yet, just so all the formulas will have someplace to point. 

Now your grid is full of data. You can create your chart based on this grid. As you type data into the monthly sheets, the summary grid and the line chart will update automagically.

But if I type "? thisdocument.Words(3).Start" I get the answer 19. So there's an inconsistency between the 2 mechanisms for looking at character count - "Words(x).Start" includes all characters including those marked for deletion, while "Characters" ignores them!

How can I evaluate a reliable start position for a word?

rich michaels - mvp (greatcirclelearning.com) say:

The difference in the numbers are because of the 2 changes that were made to the document text in your example. First "deleted" was deleted and then "added more" was added. This results in 2 hidden markers being inserted into the character string of the document.

Use the VBA routine below to validate what I am saying and hopefully it will also point out what you will need to do to get a correct start position via VBA for a specific position within a document, when changes are being tracked.

Sub CharCount()
    Dim doc As Word.Document, rng As Word.Range
    Set doc = ActiveDocument
    Dim cnt As Long
    If doc.TrackRevisions = True Then
        Debug.Print doc.Characters.Count
        Debug.Print doc.ComputeStatistics(wdStatisticCharactersWithSpaces)
        Set rng = Selection.Range
        rng.SetRange Start:=doc.Content.Start, End:=doc.Words(3).Start
        cnt = rng.Revisions.Count
        Debug.Print doc.Words(3).Start - cnt
    End If
End Sub
