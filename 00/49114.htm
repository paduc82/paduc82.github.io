To move numbered folders left over from a failed driver upgrade:

1. In the Windows\system32\spool\drivers\ folder, look in each of the IA64, W32X86, and x64 folders for the presence of a folder named "3" that contains a large number of numbered folders (\W32X86\3\1, \W32X86\3\2, \W32X86\3\3, and so on).

To start or restart the Print Spooler service:

 

  1. Press “Windows key” + “R”.
  2. Type Services.msc
  3. Locate the Print Spooler service in the list of services, and then confirm that its status is started.
  4. If its status is not started, select the service and then click Start on the Action menu.

Note: You can also do this from an elevated Command Prompt window by typing the following command: net stop spooler && net start spooler

 

In some instances, you might receive Error 1068 – "The dependency service or group failed to start" – when attempting to start the Print Spooler.

 

This can happen if the Print Spooler service relies on other services that are not running. To resolve this issue:

  1. Press “Windows Key” + “R”
  2. Type Services.msc
  3. Right-click Print Spooler and then click Properties.
  4. Click the Dependencies tab.
  5. The first box lists all of the system services that must be running for the print spooler to start. These are the dependencies.
  6. Make a note of each service, click OK, and then start each service that the print spooler depends on, if the service is not already started.
  7. If a particular service is not already started, right-click it, click Properties, and then confirm that the Startup type is Automatic.

"DRIVER_VERIFIER_DETECTED_VIOLATION" is the general bug check code for fatal errors found by Driver Verifier. Driver Verifier is used to detect and troubleshoot many driver issues that are known to cause system corruption, failures, or other unpredictable behavior. Driver Verifier can check many different aspects of a driver’s behavior. These capabilities are grouped into options or settings that are enabled by the use of flags. (The terms "options," "settings," and "flags" are typically interchangeable in Driver Verifier documentation. They represent similar concepts.)

Bluescreen errors can be caused by both hardware and software problems. If Driver Verifier detects a violation, it generates a bug check to stop the computer. This is to provide you with the most information possible for debugging the issue. When you have a kernel debugger connected to a test computer running Driver Verifier, if Driver Verifier detects a violation, Windows breaks into the debugger and displays a brief description of the error.

Driver Verifier tool is specifically developed to catch device driver bugs. If you have enabled Driver Verifier under normal circumstances, that might cause blue screen.

a. user say:

Hi,

The registry sub-key labeled “Wow6432Node” is typically used for 32 bit applications on 64 bit machines. If they are present on x86 machines, they do not cause any issues as they are not used. On 64 bit machines, Windows stores a separate entry for 32 bit applications to run. The “Wow6432Node” acts as a redirector to redirect the 64 bit and 32 bit applications to their respective registry entries. So unless you are using a 32 bit application in a 64 bit machine, this shouldn’t be a problem.

sjseattle say:

I AM using 32-bit applications on a 64-bit machine.

If I weren't running a 64-bit version of Windows, I don't think I would even have a Wow6432Node subkey in the registry. It would be difficult to run only 64-bit applications, given that most applications only have a 32-bit version.

The questions (in my mind) are:

* What things - particularly what bad things - happen if startup applications are in the non-Wow6432Node location, rather than the Wow6432Node location? Does it depend on the application? Before I edited the registry, some startup items were under one, some were under the other. Why?

* Why doesn't Windows recognize the Wow6432Node startup items if they're under HKEY_CURRENTUSER, when it DOES recognize them under the HKEY_LOCALMACHINE? Is this some sort of flaw? Can it be fixed?

Just to back up and clarify (in case it's not already):

There are four places similar startup applications can be in the registry:

"All Users 6432": HKLM\SOFTWARE\Wow6432Node\Microsoft\Windows\CurrentVersion\Run

"All Users plain": HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Run

"One User 6432": HKCU\Software\Wow6432Node\Microsoft\Windows\CurrentVersion\Run

"One User plain:" HKCU\Software\Microsoft\Windows\CurrentVersion\Run

Items in the "One User 6432" location don't seem to be recognized by Windows: they don't appear in the list at the startup tab in Task Manager, nor do they in the list produced by autorun. Items in all of the other three location do.

garrgodfrey say:

*Why* is a philosophical question.   While you do sometimes see a Wow6432Node in HKEY_CURRENTUSER, it is not used. It is only relevant on HKEY_LOCALMACHINE.  The answer is probably "because it isn't needed for HKEY_CURRENTUSER". Separately HKLM allows 32-bit programs to find 32-bit COM objects and 64-bit programs to find 64-bit COM objects. HKLM doesn't have much in the way of locations of installed libraries.

It shouldn't matter which location you put startup items in.  A 32-bit application generally puts them in the 32bit node, because that is the default and it would need to specifically redirect to the 64-bit version.
