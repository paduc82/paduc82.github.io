anne troy say:

You will have an area with data:

 

Let's suppose your right-most used cell is J5.

Let's suppose your bottom-most used cell is D200.

That means your "used" area will be A1 to J200. That is the range you need to encompass all of your used cells.

So, select, column K (one column beyond J) by clicking on the letter K. Hit Shift+Ctrl+Right arrow on your keyboard until it selects and takes you to the right-most column, XFD. Hit Delete.

Then select row 201 by clicking on the number 201 and hit Shift+Ctrl+Down arrow on your keyboard until it selects and takes you to the bottom-most row, 1,004,000 (something). Hit Delete.

ken sheridan say: I'm afraid I don't understand what you mean by "I want the chosen form to become a permanent part of the record".  A form is merely an interface with the rows (records) in a table, not a part of a record.

To retain the subform last used as the default for when the parent form is next opened you'll need to store the value of the option group in a data structure of some sort.  The most convenient is a table, named Subforms say, with a column SubformNumber say, of integer number data type.  First fill the table with one row with a value 1.

In the Load event procedure of the parent form put:

    Me.Frame225 = DLookup("SubformNumber", "Subforms")
    Ctl_Frame225_AfterUpdate

Check to make sure that the name of the option group's AfterUpdate event procedure is Ctl_Frame225_AfterUpdate and, if not amend the above code as necessary.  This will assign a value to the option group by looking it up from the SubForms table and call its AfterUpdate event procedure to load the relevant form into the subform control.

To store the last used value in the table put the following in the form's Close event procedure:

    Dim strSQL As String

    strSQL = "UPDATE Subforms " & _
        "SET SubformNumber " = & Me.Frame 225

    CurrentDb.Execute strSQL, dbFailOnError
