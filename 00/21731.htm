roady [mvp] say:

Sorry, but your reply is just a repeat of your original with no real additional information other than that it originates from an Exchange server which actually only raises more questions.

 

The main questions still remains; How was this backup pst-file created and how are you restoring the data?

 

Its size doesn't say anything about its content. A pst-file is a database which isn't directly being truncated upon removal of data from it.

 

A common mistake using the Archive feature and then only backup the original pst-file and not the archive. Another common mistake is to use the Export function without verifying the export. Yet another is backing up the pst-file while Outlook is still running. Regularly this is even mixed with storing the pst-file on a network share. All these scenarios could lead to a large but more or less empty pst-file.

 

A proper backup of an Exchange mailbox to a pst-file holds no relation to any data that is stored on the Exchange server; it's completely standalone. Note that the local cache file of the Exchange mailbox (ost-file) is tied to the Exchange mailbox and is only recoverable when you still have the Exchange account configured in Outlook or via 3rd party recovery tools.

  • Service: _sipfederationtls._tcp.<<website>>

argmarovski say:

That is the Execute Disable Bit error(aka No Execute Permission, No-Execute Page Protection Technology, NX, DX, etc). If there is no option in your Bios/UEFI to enable it, build 10103 will not install in your current hardware. It seems that for some reason build 10103 will not install if CPU virtualization is not enabled/supported.
