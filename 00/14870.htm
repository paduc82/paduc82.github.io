john mcghie say:

Hi Jennifer:

1)  You would normally "ignore" font substitution. The substitution will occur only when you display the document.  It will go back to normal when someone who has that font installed displays it.

2)  However, in your case, Word is using an old non-Unicode font to perform the substitution.  And hence, you have the wrong characters showing up.

The simplest way to resolve this would be to ask the originator for a copy of the fonts they used.

I would not worry about Cambria: that is a Unicode font and always has been.  I suspect the issue is an old version of Tahoma.  You could make a "permanent" substitution of Verdana Ref to simply "Verdana".

The copy of Verdana supplied with recent versions of Microsoft Office is a Unicode OpenType font that should resolve the issue. Go into your Font folder, select Verdana.ttf and do GetInfo.  If it says "TrueType OpenType" in the description, you are good to go.  Follow the instructions in the Font Substitution dialog to correct the substitution to Verdana and make it permanent.

If you don't have a modern copy of Verdana, ask the originator to send you the font, drag it into your Font folder, then re-start your computer.

Hope this helps

How to move your data to a new Identity


  • Do Not Rebuild Identities with Exchange and IMAP accounts. Delete the accounts (this deletes the cache files in Outlook). Do the rebuild then add back accounts. Better yet start over in a new Identity. If the rebuild does not work then you could sync unwanted changes or deletions to your server and lose data.
  • Make sure you have plenty of free space on your drive. Make sure that you have at least three times as much hard disk space available as the current size of your Outlook identity PLUS 10% free for virtual memory.
  • Rebuilding does not fix all issues. It can make some problems worse. Even though the Database Utility will make a backup, I advise making a copy in the Finder first.
  • You can delete the rebuild version after you are satisfied with the results. This copy will be named like this: Main Identity [Backed up 1-21-2012 13.35]  

Tip: If your Identity will open, before you rebuild, check "On My Computer"Inbox and Sent folders to be sure they are under the 2GB limit. Generally this is less than 10,000 messages but with large attachments could be less. Make subfolders e.g. Inbox 2011, Inbox 2010, Sent 2011, Sent 2010 etc to help reduce the number of messages in a folder. (Check for any large archive folders you might have in addition to Inbox and Sent).

dirk goldgar say:
I have been doing this for years and it has always worked fine up until a couple weeks ago. I have a user who received a new computer with Office 2010 32-bit and Windows 7 64-bit. The only difference between her computer and everyone else's is the 64-bit Windows 7. Everyone else has 32-bit Windows 7.
 
The Environ("username") command still works in a VBA module, but it no longer works in the Default Value property in a table or in the Control Source property on forms. It returns "#Name".


I would guess that this is caused by Access operating in sandbox mode on the new computer.  Sandbox mode is controlled by a registry key, and the registry key is different on 64-bit Windows, because 32-bit applications have to go through the Wow6432Node node in the registry.  If I'm right, either (a) the new computer simply doesn't have the sandbox mode key set to turn it off for Access at all, or (b) the 32-bit registry key was set instead of the 64-bit one.

 

For 32-bit Access 2010 running under 64-bit Windows, the key that must be set is:

 

[HKEY_LOCAL_MACHINE\SOFTWARE\Wow6432Node\Microsoft\Office\14.0\Access Connectivity Engine\Engines]
"SandBoxMode"=dword:00000002

ken sheridan say:
........and using the Environ command in the default value of a field is a very easy way to capture the user ID.

It's also a very unsafe one as it is extremely easy for a user to change the value of the environmental variable and masquerade as another user.  And don't believe anyone who says that this is not the case in later versions of Access.  It's been demonstrated that it is.

Use the Windows API call as Scott recommends.  It's very easily done.  You'll find an example as ChangedRecordDemo.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

ken sheridan say: PS:  You can call the GetUser() function or similar as the DefaultValue property of a bound control in a form, but not as the DefaultValue property of a column in a table.
