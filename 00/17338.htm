ken sheridan say:

Firstly you need to declare a module level variable in the declarations area of the parent form's module:

 

    Dim strUserName as string

 

In the parent form's Current event procedure you should assign the value of the UserNameCombo control to the variable:

 

    strUserName = Nz(Me.[UserNameCombo],"")

 

You should amend the code in the parent form's AfterInsert event procedure to assign the value of the UserNameCombo control to the variable as well as inserting a new row into the subform:

 

    Dim strSQL As String

 

    strSQL = "INSERT INTO [EmployeeEffort]([UserName],[DocID]) " & _

        "VALUES(""" & Me.[UserNameCombo] & """," & Me.[DocID] & ")"

 

    If Not IsNull(Me.[UserNameCombo]) Then

        CurrentDb.Execute strSQL, dbFailOnError

        Me.[EffortForm].Requery

    End If

 

    strUserName = Nz(Me.[UserNameCombo],"")

 

This caters for when a record is first inserted into the parent form.  To cater for a user then amending the selection in the UserNameCombo control you need to put code in the control's AfterUpdate event procedure to delete the row with the value of the variable in the UserName column from the subform's Recordset in the parent form's recordset, and then insert a new row for the newly selected value, along with assigning the value to the strUserName variable:

 

    Dim strSQL As String

 

    strSQL = "DELETE * FROM [EmployeeEffort] " & _

        "WHERE DocID = " & Me.DocID  & _

       " AND [UserName] = """ & strUserName & """"

 

    CurrentDb.Execute strSQL, dbFailOnError

 

    strSQL = "INSERT INTO [EmployeeEffort]([UserName],[DocID]) " & _

        "VALUES(""" & Me.[UserNameCombo] & """," & Me.[DocID] & ")"

 

    If Not IsNull(Me.[UserNameCombo]) Then

        CurrentDb.Execute strSQL

        Me.[EffortForm].Requery

    End If

 

    strUserName = Nz(Me.[UserNameCombo],"")

 

Note that the second call of the Execute method of the database object in the combo box's AfterUpdate event procedure does not use the dbFailOnError option.  This is because there might be a situation where the replacement UserName already exists in a row in the subform's recordset.  If the columns in EmployeeEffort have been correctly indexed  by including DocID and UserName in a single Unique index this would raise an index violation error, which you don't want to be trapped in this case.  Strictly speaking the specific error should be trapped and ignored by an error handler in the event procedure, but I don't want to over-complicate things for you at this stage.

 

You'll appreciate that I can't test this without access to your file, so I cannot guarantee that some debugging will not be needed.

 

Don't forget that you need to give any 'required' columns other than UserName and DocID in the  EmployeeEffort table a default value to prevent an error when the INSERT INTO statement is executed.
