scottgem say:

It depends on the database. In some cases it works, it others it doesn't.

Relationships are established by inserting a foreign key into a table. 

What you are referring to is called Referential Integrity, not relationships. RI can only be imposed on the table level between tables in the same file. However RI can also be handled within your user interface when it can't be established on the table level.

In my opinion Cascade Updates are unimportant since a Primary key should never change in the first place. Cascade Deletes are useful to ensure there are no orphans and RI itself can be useful to make sure records are not created where a parent doesn't exist. But that can be handled in the UI.

john mcghie say:
Yes, you are correct.  That fix was a temporary work around for 2001 and 2004 only.  It worked around an error that was fixed in a service pack.  All it does is clear the "undo stack", which does not suffer problems in 2008/2011.

Similarly, "Repair Permissions" should have no effect on this problem, because it does not touch "user files".  Repair permissions checks the permissions set on the components of Apple software against the install manifest for each piece of software.  It can be useful as an initial trouble-shooting step, but there is no "science" behind it: it "stirs things up" in the Application and System folders, and that sometimes forces applications to re-write their preferences if they had previously been unable to, after which they magically "work".  But as I said, it applies only to Apple software installed by the Apple installer.

The issue you are suffering has nothing to do with Unix file permissions: it is a problem in the user (document) file locking or releasing by the Operating System.  The Operating system is either not granting a lock to Word when it tries to create a file, or not releasing a lock when Word tries to delete a file.  Which may be because the user ID that "owns" the file is not the same as the one trying to write it.

Word doesn't create "unnecessary" temporary files.  They are all "necessary" for something :-)  When Word is working on a document, it can have more than 20 other files open, for each document.  A read/write/re-name error in any one of them will produce the error you are seeing.

The reason I suggested that you disable Autorecovery was to make Word lose interest in one set of files, to see if anything would change.  We got the expected change.  That tells me we are on the right track.

You should not save anything to the Microsoft User Data folder: that may be part of your problem.  The MUD is in the wrong place: it should have been in "Application Support", out of sight of users: but for reasons that seemed like a good idea at the time, they chose to place it in the Documents folder.  Don't be tempted to try to move it.  But leave it to the Microsoft Office programs.  Don't save anything in there yourself, otherwise you may create file name clashes that would produce the issues you are describing.  Despite its name, the MUD is system data, human beings shouldn't go in there :-)

There is supposed to be a MUD in the Document folder of every user who uses Microsoft Office on each computer.  If it's not there, Microsoft Office will create it at first run.  If you delete it, you will potentially lose all your contacts, email, notes, calendar and Normal template.  So don't be tempted :-)

There should not be any aliases to the MUD, anywhere.  If there are, that may be part of your problem.

"Always create backup" creates a single backup file for each document you edit.  When you have finished editing, you can simply delete it.  They will all be labelled "Backup of..." and sort to the top of the folder, so you can delete them all in one hit after closing Word.  Autorecover also creates files, usually ten or so per document.  You don't normally see them because they are saved in the MUD, and should be cleaned up if Word exits normally.

I am beginning to suspect that the User Permissions are wrong on your MUD folder, preventing Word from writing there.  Could that folder have been created on a different computer, or with a different user name?

To find out, right-click the MUD in Finder, and choose Get Info.  

You're on Lion:  I'm giving instructions for Snow Leopard, so you may need to interpret a little.

At the bottom of the dialog, look under Sharing and Permissions.  There should be three user groups listed: "Me", "Staff" and "Everyone".  "Me" should have "Read and Write" and everyone else should have "Read Only".

Now click the lock at the bottom right.  Supply the Administrator password when prompted.

Now highlight the "Me" line.  Then drop down the arrow next to the cog symbol.  If "Make Me the owner" is enabled, that indicates you didn't own the folder, and there's your problem.  Choose "Make Me the owner" if you can, and allow it to complete.

Then choose "Apply to enclosed items".  Even if you did own the folder, if you did not also own all the items in it, you would get these problems.

If it wasn't that, we'll have to keep looking.

BTW:  Changing operating systems is the most major change you can make to a computer.  Application software (such as Microsoft Office) is designed to work best on the operating system it was built for (OS 10.4 in the case of Microsoft Office 2008, OS 10.6 in the case of Office 2011).  The fact that Apple calls them both by the marketing name "OS X" means nothing: they are very different internally.  So neither Office 2008 nor 2011 will work perfectly on OS 10.7.

Microsoft is working busily to upgrade Office 2011 for OS 10.7.  But the reason I have not gone up to Lion on any of my machines is that I use Microsoft Office for my business, and I cannot afford to have it catching little "issues".  Or even big ones...
