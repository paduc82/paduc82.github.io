dhookom say:

It's a bit difficult to provide much direction without understanding your data tables and requirements.

Let's assume you have a main table for projects and you want to identify all of the tools required to accomplish the project. My suggestion would be to have the following tables:

Projects (Table with one record per project)

Tools (table with one record per tool)

ProjectTools (table with one record per project and tool combination)

This solution allows you to easily query all of the tools required for a project as well as all of the uses for a single tool.

ken sheridan say: What you have is a many-to-many relationship type between two entity types such as projects and tools.  A many-to-many relationship type is modelled by a third table which resolves the relationship type in two or more one-to-many relationship types.  In Duane's example the ProjectTools table does this.

You'll find an example of a many-to-many relationship type like this, and how to represent it in forms, in StudentCourses.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

In this little demo file think of students as being analogous to projects, and courses as analogous to tools.  A variety of interfaces are illustrated, but I'd strongly recommend the conventional one of a form/subform, which is easy to implement and code-free.  As you'll see there are other approaches, including the use of a multi-select list box, but for that you'd have to write code to read and write the date to/from the tables.

A major advantage of a conventional form/subform is that it allows for other attributes of the relationship type to be entered in the form, Status in my case.  You might want to record the dates when tools were assigned to and returned from projects for instance, for which you'd add two columns (fields) of date/time data type to the ProjectTools table, and include text box controls bound to the two column in the subform.

As you are new to Access another file you might like to take a look at in the same OneDrive folder is DatabaseBasics.zip.  As its name suggests this is a simple introduction to some of the common principles and methodologies of relational database design in Access.

***Post moved by the moderator to the appropriate forum category.***

jay freedman say:

Because each form is in a separate table, you can use the table that contains a particular picture control as the identifier in VBA for the rich text controls associated with it. With the code below, you can change all the picture content controls' titles to Pic, and all the rich text content controls' titles to PhotoBy and PhotoDate -- the titles are no longer used by VBA at all.

Private Sub Document_ContentControlOnEnter(ByVal CC As ContentControl)
    Dim tbl As Table
    Dim ccPhotoBy As ContentControl
    Dim ccPhotoDate As ContentControl
    
    If CC.Type = wdContentControlPicture Then
        Set tbl = CC.Range.Tables(1)  ' this is the table that contains the picture control
        Set ccPhotoBy = tbl.Range.ContentControls(2)      '  the second control in the table
        Set ccPhotoDate = tbl.Range.ContentControls(3)  '  the third control in the table
        
        ' user selects photo
        ' code gets metadata and puts photographer's name and photo date in string variables
        
        ccPhotoBy.Range.Text = "Ansel Adams"    ' put photographer's name variable here
        ccPhotoDate.Range.Text = "3 July 1940"   ' put photo date variable here
    End If
End Sub
