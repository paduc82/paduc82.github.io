discusgirl say:

Sharing database: Sharepoint lists vs. Network Drive

Good morning, I have a database that is ready to be used by mulitple usrs, about 10-20 people. I am wondering what would be the best way to share the database so that the data are centrailized as the users enter data. I have Access 2010 and Sharepoint 2010. So far, I have tried two options:

  1.  I have manually splitted the databases (tables on the backend) and saving the BE to the a shared network drive. When accessing the front end on my desktop, there is a significant lag for every button clicked. 30 seconds to open a form nd 60 seconds to run a report! My internet speed is considered decent 6.0mph and I know this is not the issue. I notice that opening any file on the network drive takes sometimes too. This lag  would be an issue for me and users if we go with this sharing option.

2. I have tried publishing the tables as Sharepoint lists, and this seems to work much faster than the network drive. However, since Sharepoint lists do not suport  referential integrity and cascade delete, that would cause a bit of a manual work if I need to delete a records and its pertaining data. Also I notice the front end of Access is bloated in size, even tough the tables are published and linked to Sharepoint? armen stein say:

Option 1 is basically using Access across a WAN, which doesn't work well and can lead to performance problems and corruption.  A variation of Option 1 would be to use SQL Server (or SQL Azure), but you still need to optimize your front-end application to use SQL Server efficiently.  See my slide deck Best of Both Worlds on our free J Street Downloads page.

Option 2 is the Access 2010 web services, which you should know did not move forward in 2013.  It's still supported, but the approach changed to Access Web Apps in 2013, which use SQL Azure to host the database instead of SharePoint lists.  I would not recommend this architecture for long term projects, since it isn't being enhanced at this point.

One reason that your database didn't get smaller is that in Access 2010, the tables are actually stored locally in Access as well as in SharePoint.  They're synchronized when you have a connection, but you can also work offline and sync them later.

Access web services actually do have a form of referential integrity through lookup tables, but it doesn't look the way you're used to.  And as far as I know, you'll need to do the cascade deleting yourself using data macros.

Again, I don't recommend building Access 2010 web services apps at this point - there's no future in them.
