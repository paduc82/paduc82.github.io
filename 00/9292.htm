Two possible issues:

As noted you cannot MIX x32 and x64 bit versions of office programs (for the SAME version).

You ALSO cannot mix and match CTR versions. So you cannot install Access 2016 x32 on a machine that has office 2016 x64.

It possible that the install ‚Äúerror‚Äù is wrong, but that install suggests ‚Äúsome‚Äù of office x64 is installed. However, if you ALREADY have office 2016 installed, then clearly you do NOT have office x64 (I going to assume you are sure of this!).

However, let‚Äôs assume the above is NOT your issue!

Even in the case where the BIT SIZE DOES MATCH, you are STILL unable to install. This is due to no mix/matching of CTR and NON CTR versions of the SAME given version of office (and same bit size). Read this paragraph carefully again!

So regardless if the ‚Äúerror‚Äù message is wrong due to bit size (you seem to suggest it ONLY x32 installed), once you pass that hurdle of ensuring the SAME bit size of office and the Access runtime, you ALSO then must ensure that you not attempting to mix and match CTR and non CTR versions of the office suite!

So I just saying let‚Äôs not try and solve the error x32/x64 bit issue, since if you do SPEND time solving this issue, you will STILL not be able to install!

And more worse is NO CTR version of the runtime is available.

To make a long story short?

About the only practical way around this issue is to install a DIFFERENT office version of the runtime. So I would suggest installing Access 2013 runtime if you have office 2016 and if you have office 2013, then install 2010 access runtime.

So you ‚Äúcan‚Äù mix and match CTR and non CTR installs of office programs WHEN they are NOT from the same version of office. I betting this quite much is your only reasonable option here.

Until we get a CTR version of the runtime, then we have to dodge the bullet by installing a previous (or different) version of the runtime - even when the bit size is matched. The "bad guy" here is CTR installs and no CTR runtime install is available. We are unlikely to get a CTR Access runtime, so above is your only solution at this point in time.

This little demo file includes an option to 'Restrict Report's Underlying Query on One Field' in which the query is as follows:

PARAMETERS [Forms]![frmEmployeeDlgParam]![txtEmployeeIDList] TEXT ( 255 );
SELECT DISTINCT Employees.*, FirstName & " " & Lastname AS FullName,
Projects.Project
FROM Projects RIGHT JOIN (Employees LEFT JOIN ProjectEmployees
ON Employees.EmployeeID=ProjectEmployees.EmployeeID)
ON Projects.ProjectID=ProjectEmployees.ProjectID
WHERE INLIST([Employees].[EmployeeID],[Forms]![frmEmployeeDlgParam]![txtEmployeeIDList])=TRUE;

The query calls the following function:

Public Function InList(varVal As Variant, ParamArray aList() As Variant) As Boolean

    ' this function is based on a method posted by Michal Dziubek
    ' in Microsoft's MSDN forum for Access

    ' pass the value for which the list is to be searched as the first argument
    ' and the list of values as a comma separated list of values as the parameter array
    ' e.g this will look for the value entered into
    ' an input box in a list of fruit:

    ' Dim strFruitToFind As String
    ' Dim strFruitList As String
    ' Dim strMessage As String

    ' strFruitList = "Apple,Banana,Orange,Pear"

    ' strFruitToFind = InputBox("Enter a fruit", "InList Function Test")

    ' If InList(strFruitToFind, strFruitList) Then
    '     strMessage = strFruitToFind & " was found in the list."
    ' Else
    '     strMessage = strFruitToFind & " was not found in the list."
    ' End If

    ' MsgBox strMessage, vbInformation, "InList Function Test"

    ' or it could be called in a query with a parameter as the value list, e.g.:

    ' SELECT *
    ' FROM Contacts
    ' WHERE Inlist([LastName],[Enter last names:] = TRUE;

    If Not IsNull(varVal) Then
        If UBound(Filter(aList, varVal)) > -1 Then
            InList = True
        End If
    End If

End Function

The fact that your list box returns three columns is irrelevant.  You need to restrict the query only on the key, ID, which is done by referencing a value list of the keys in a hidden control form.  This control is filled and the report opened (it could equally well open the query as a datasheet of course) by the following code in the form's module:

    Dim varItem As Variant
    Dim strEmployeeIDList As String
    Dim ctrl As Control
    
    Set ctrl = Me.lstEmployees
    
    If ctrl.ItemsSelected.Count > 0 Then
        For Each varItem In ctrl.ItemsSelected
            strEmployeeIDList = strEmployeeIDList & "," & ctrl.ItemData(varItem)
        Next varItem
        
        ' remove leading comma
        strEmployeeIDList = Mid(strEmployeeIDList, 2)
        ' assign value list to hidden text box control
        Me.txtEmployeeIDList = strEmployeeIDList
        
        DoCmd.OpenReport "rptEmployeesParam", View:=acViewPreview
    Else
        MsgBox "No employees selected", vbInformation, "Warning"
    End If

The other list box's selected items would also need to be written in the code to a hidden control, which would be referenced by a second InList fuction call in the query's WHERE clause using either an AND or an OR Boolean opeartion, depending on whether you wish the two list boxes to act restrictively or  in a complementary manner in relation to each other when determining the result set.   My demo includes examples of two list boxes interelating in both ways.
