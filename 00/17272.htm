albert d. kallal say:

For publishing of an access database, you need the correct server and correct systems and correct technologies installed on that server for this to be allowed to happen.

And keep in mind the above is really quite much like the desktop, if you develop an application FoxPro, then to run that FoxPro application on the desktop, you need a FoxPro Software installed.  And if you develop an application file maker on the desktop, then of course you need file maker installed.

And of course it to develop an application in access for the desktop, then you need access installed on my desktop.

And keep in mind exact same reasoning and logic occurs when you deploy an application to a web server.  If you build your application and use the database MySQL, then when you deploy to the web server, it will not work for nor operate unless that web server has the MySQL database system installed, and it even has to be the correct version of that database software.

And if you build a web application using the oracle database server, then again for your application to work and be installed and run on that web server, then the oracle database software in system has to be installed on that web server.  And again it even has to be the correct version of the database software.

 

So you always have to ensure that you have the exact correct database software installed on that server.

 

The next step would to be ensure that the coding and development system used such as asp.net, or in our case what is called Access web services will have to be installed and setup on the server.

 

In other words if you decide to develop your software using PHP, and such software is not installed or supported on the web server, then your applications not going to run on that web server.

 

So just keep in mind, that like a terminal or screen, just about any device or web browser can CONSUME and view and utilize web based software. However, the installing, the development, the setup, in fact ALL OF THE BITS and parts from the software coding system to database system and even the commands for the database system MUST ALL MATCH UP perfectly to the set of tools that you used to develop your software for that web server.

 

So no matter what set and web development tools you choose here, you'll have to ensure that all those tools match exactly the particular type of web server you're going to utilize, and there's no such thing as just publishing something to any old web server, and the development using the LAMP stack, or ASP.NET, or in this case using Access web services, and all of these cases you have to utilize a web server that matches your software development requirements exactly.

If just one part of the equation from the database system, to even the kind of web server you choose to use such as IIS or Apache does not match your coding system, then you cannot publish.

 

So how access works in this regards is exactly the same as the rest of the whole industry in terms of how web publishing and web development is approached, at the end of the day he simply have to use a server that matches and supports exactly the bits and part you use to develop that web a system.

 

So in the case of access web publishing, you need a server that is running what is called access web services.  And like every other system for web development, it simply development stack that you require on that web server.

 

In a nutshell this means you need either:

SharePoint 2010 (have to be enterprise edition)

Or

Office 365 (p1 plan supports Access web publishing ‚Äì starts at a very reasonable $6 per month)

Or, like every other system, you go and find a provider that supports the exact development stack you've chosen, a commercial provider such as

www.AccessHosting.com

Is another provider that supports access web services, and therefore would allow you to publish an Access web database.

 

Last but not least, keep in mind that if you're going to build a web system with access, you must specifically build web forms and choose a web database in access. You cannot convert or utilize EXISTING VBA and client forms ‚Äì there is NO type of automated conversion ability included inside of access.

 

So while access now allows you to build web forms, and allows you to do web publishing, you must specifically create forms designated as web based when doing so.

This means that existing forms cannot be published to the web, and they will have to be recreated as web based forms inside of access.

 

To see a web database in Action, take a view of this video of mine, and note how at the half way point I run the Access application 100% in a browser:

http://www.youtube.com/watch?v=AU4mH0jPntI

 

So to answer your question. No, you cannot publish "just" to IIS, you need Access web services running, and this means you need SharePoint or perahps office 365.

dale howard [mvp] say:

E_Malone --

 

There is no easy way to enforce the Start time of a task when you are using a Start-to-Start dependency.  If the Predecessor task starts at 8:00 AM, and you have a SS+1day dependency, the Successor will start at 8:00 AM one day later.  If the Predecessor starts 1:00 PM, and you have a SS+1day dependency, then the Successor will start at 1:00 PM one day later.  What you are seeing is the default behavior of the tool and there is no way to change it.  Hope this helps.
