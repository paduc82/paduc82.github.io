kathleen orland say:

Microsoft has never supported the use of a PST from a networked drive, and with good reason.

 

Outlook requires constant read/write access to the PST. The slightest interruption will cause corruption or issues with Outlook.

 

What you're doing is not supported.

 

What I've done for my users is written a very simple script that they can execute when they close Outlook that copies their PST file(s) to their personal drive on the network:

 

COPY %LOCALAPPDATA%\Microsoft\Outlook\*.pst u:data\outlook /y

 

Save the script as a batch file on the C drive. Make a shortcut. Change the icon to the MSN butterfly. Rename it to OUTLOOK BACKUP. My users love this.

 

If you don't have a policy against using the task scheduler, you can even trigger the backup with that.

VBA to clear clipboard - Microsoft Community

VBA to clear clipboard

Does anyone know any coed that will clear the clipboard after a copy and past operation?  I've tried everything I've found through searching the web, and nothing seems to work.

Mike

daniel pineault say:

This works fine for me.

Private Declare Function apiOpenClipboard Lib "User32" _
                                          Alias "OpenClipboard" _
                                          (ByVal hWnd As Long) _
                                          As Long

Private Declare Function apiEmptyClipboard Lib "User32" _
                                           Alias "EmptyClipboard" _
                                           () As Long

Private Declare Function apiCloseClipboard Lib "User32" _
                                           Alias "CloseClipboard" _
                                           () As Long


Function EmptyClipboard()
    If apiOpenClipboard(0&) <> 0 Then
        Call apiEmptyClipboard
        Call apiCloseClipboard
    End If
End Function

Regarding your code, the issue is you don't have an Exit Sub after your If statement so it continues to run the your error handler no matter what to get to the End Sub.  You need to do something more like:

Private Sub ServiceSummary_DblClick(Cancel As Integer)
    On Error GoTo Error_Handler

    If IsNull(Me.ServiceSummary) Then
        DoCmd.RunCommand acCmdPaste
    Else

Error_Handler_Exit:
    On Error Resume Next
    Exit Sub

Error_Handler:
    MsgBox "Nothing to paste", vbInformation, "No Data"
    Call EmptyClipboard
    Resume Error_Handler_Exit
End Sub

Thanks

Chris

dhookom say: I typically import into a temporary table and perform the updates in the second step. If this is a regular occurance I might create a temporary Access data file to hold the temporary imported table. This reduces bloat in the main data file.

hansv mvp say:

You have to use two separate queries:

  • An append query to add records that don't have a matching IP address
  • An update query to update records with a matching IP address.

marshall barton say:

Another way is to link to the CSV file as a linked table.  Then you can use two queries to perform the update, first a delete query to remove all records in your master table that have a matching IP address in the CSV file/table:

DELETE M.* FROM mastertable As M INNER JOIN csvtable As C ON M.IP =C.IP

then an append query to add all the records from the CSV file:

INSERT INTO mastertable SELECT * FROM csvtable

This is far simpler than trying to identify what has changed and what has not changed and if the number of records in the CSV file is only in the hundreds may be faster too. Bloat should not be an issue because the space from the deleted records should be reused by the appended records. 

Regardless of how you do the updates, to be on the safe side, you should make a backup copy of the table before doing the update.

dhookom say:

dhookom say:

I just did some testing and I couldn't get the delete to work with a join to the linked CSV table. I was able to get it to work with SQL of:

DELETE mastertable.*
FROM mastertable
WHERE IP In (SELECT IP FROM csvtable);

You can then create and run an append query.

john w. vinson say:

I thought I could artificially create a unique index on the linked CSV with a DDL query but that didn't work for me.

Doesn't work because you can't index a linked text file - there's no place in the text file to put the index, and Access can't assume that the text file will be stable. You can only index a native ACE table.

I've often linked to external text files, but the inability to index them puts some limitations - you can't update joins, as in this example, and if the file is at all big it can make searching the data (with or without a JOIN to an internal table) really slow.

It all depends on "how" you bought office, which license terms you are working under. "Normal" retail licenses allow you to install effectively unlimited times on the same hardware. Retail licenses allow you to have the product key active on at least 2 computers at the same time. And to transfer those installations to new computers when needed.

If you bought Office installed at the factory you had an "OEM" license. Or if you bought a price discounted "Product Key Card" you had a "PKC" license. They have the same terms. The license can only be activated on 1 computer, PERIOD. It cannot be transferred to another computer. But they don't give the message you are getting.

There is another class of licenses. They are generically known as "volume" licenses. They are not sold to individuals, they are sold to companies buying 5 or more licenses at a time. There are 2 types of volume  license. One has effectively unlimited number of activations associated with it, and the other has a limited number. These licenses give the sort of message you describe.

walt marston say:

Solution found. Since Data Macros for Access web apps does not include a Max function to find the last record, we came up with a workaround. We added a new field to the table, named EmailSent (Yes/No), set all the previous records values to "Yes". Then, when new record created, the value of EmailSent defaulted to "No" so we could pick that record. Then, we were able to grab the info needed from that record to include in the email notification. After the email was sent, we set the value of EmailSent to "Yes." Thanks everybody for your suggestions. 

albert d. kallal say:

The standard way and means to get the last “max” record in Access web applications is to create a query sorted by descending.

You then simply grab the first record via the lookup a record command. By sorting correctly you always get the first record, and with a sort descending on the desired max column value, then the first record will be the max record.

And you flag solution is certainly viable and thus gives you a means to track if emails were sent. (somewhat prefer your approach). However I wanted to share with you and the folks here that while no "max" command exists, you can achieve the same results by using lookup record based on a correctly sorted query.
