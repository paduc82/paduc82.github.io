john - project say:
Jules74,
I'm a little bit more pro on using cross-project links than Rod but I do share his concern for the possibility of corruption when using them.

Probably the easiest way to set up cross-project links is to first open a new blank file and then use Project/Insert group/Subproject to insert both files into the temporary master. You should also open each of the subproject files.

There are various ways to set up the links within the master but one easy way is to click on the start or finish of a Gantt bar and drag that to the task in the other subproject that you want to link. If the tasks to be linked do not reside within viewing distance in the view, you can use filters to isolate those tasks you want to link. You can even display the Predecessor and Successor fields and add lead or lag to links as necessary.

When all your links are set up use Save As to the same file name for each of the subproject files. That "sets" the link structure. Then close the master, you do not need to save it.

Now, very important, from that point on, do NOT move, rename or "save off" copies of the linked files. And as Rod said, linking files over a network is just asking for trouble so keep the linked files in a single directory, ideally in a single folder on your local PC.

Good luck.

John

alexander chizhov say:

Now it is sollved :)

The link above describes how to do manually what is made automatically by the IE installer. But you'll get the same error as well since the problem that causes the failure is still not solved.

The problem is that old update packages are still in the vault. When you try to install IE, installer discovers old ones in the vault and and since they differ from the new packages, that come with installer, it stops installation process, because it can not replace the packages. You have to remove old packs manually.

Here is what you need to do.

1. Start command prompt as administrator.

2. Enter: dism.exe /Online /Get-Packages > 1.txt. This will generate list of installed packages and store it into the file 1.txt right in the folder where you started dism.

3. Find in this file all the names of the packages that contain IE or Internet-Explorer. Even if they are language packs, you need to know them. In my case there were 4 names like this.

Microsoft-Windows-IE-Troubleshooters-Package~31bf3856ad364e35~amd64~en-US~6.1.7601.17514

Microsoft-Windows-IE-Troubleshooters-Package~31bf3856ad364e35~amd64~~6.1.7601.17514

Microsoft-Windows-InternetExplorer-Optional-Package~31bf3856ad364e35~amd64~en-US~8.0.7601.17514

Microsoft-Windows-InternetExplorer-Optional-Package~31bf3856ad364e35~amd64~~8.0.7601.17514

4. Remove these packages.

Enter: dism.exe /Online /Remove-Package /PackageName: Microsoft-Windows-IE-Troubleshooters-Package~31bf3856ad364e35~amd64~en-US~6.1.7601.17514 (or what name you will get first in your list) 

Don't reboot the system when it prompts to. Repeat it for all the names you find in the Get-Packages output, and rebooy after the last removed.

5. Download the offline IE installer. Start this .exe file with option /update-no.

INDIRECT Function is broken at least for Excel in German and French when using R1C1 Style

Hi there,

a user on the Office Network Forum posted that when using the R1C1 Style in the INDIRECT function of a French version of Excel 2016, the result is the error #REF! instead of the referenced and expected value. I tested this also for a German Excel Version and more language packs and experienced similar problems for the German version.

When using the INDIRECT function, you can pass as first argument a cell reference and as second argument the style type. Example for an English Excel version: INDIRECT("R1C1";FALSE) should give you the value in cell A1.

When using a localized version of Excel, you have use the localized R1C1 string, in German Z1S1, in French L1C1 and e.g. in Spanish F1C1. The problem is, that for German and French, this does not work anymore. I am using Excel 2016 Build 16.0.7167.2047 updated today. The error was also present in Build 16.0.7167.2040 but not in Build 16.0.6741.2063 (I asked some people to test).

Currently, when using this use case for INDIRECT, you get:

Excel Language Formula R1C1 localized Result Status
English =INDIRECT("R1C1";FALSE) No Value from A1 Correct
German =INDIREKT("Z1S1";FALSCH) Yes, as Z1S1 #BEZUG! meaning #REF! Not correct
German =INDIREKT("R1C1";FALSCH) - #BEZUG! meaning #REF! Correct
French =INDIRECT("L1C1";FAUX) Yes, as L1C1 #REF! Not correct
French =INDIRECT("R1C1";FAUX) - Value from A1 Not correct
Swedish =INDIREKT("R1C1";FALSKT) No Value from A1 Correct
Spanish =INDIRECTO("F1C1";FALSO) Yes, as F1C1 Value from A1 Correct

For testing, please install and switch to the corresponding Language Pack of Excel. For getting the status, I used the behaviour of Excel 2013 as reference and comparison.
