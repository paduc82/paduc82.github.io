hairyfool say:

It is not really the fault of HP but the way Windows reports disk space. HP Tools file set requires a set amount of disk space, just under 2GB. This would never increase so HP have only allocated sufficient disk space to take the files, it would be wasteful to give it more.

However Windows requires a certain amount of free space for maintenance so even though the disk is big enough for its purpose it flags up as an error. You can stop the error status through a registry mod but this will affect all disks.

hairyfool say:

Take a disk with partitions from one computer and fit it to another, it will be assigned partition letters in the 1st instance. They will not be the letters from the original PC, they are issued letters by the running O/S. If that disk is an operating system disk even the System Reserved which not have had a drive letter will get one, The running Windows assigns letters as required. There are rules as to how letters are issued by default, Windows just introduced the option to modify them.

The point is drives can exist with being flagged as requiring maintenance and therefore free space. With the advent of super large disks the arbitrary 15% leads to several hundreds of Gigabytes "unusable" and even this is unmodifiable.

that the kernel isn't allowed to have/catch (bound trap) or that
is always instant death (double fault).  The first number in the
bugcheck params is the number of the trap (8 = double fault, etc)
Consult an Intel x86 family manual to learn more about what these
traps are. Here is a *portion* of those codes:
If kv shows a taskGate
        use .tss on the part before the colon, then kv.
Else if kv shows a trapframe
        use .trap on that value
Else
        .trap on the appropriate frame will show where the trap was taken
        (on x86, this will be the ebp that goes with the procedure KiTrap)
Endif
kb will then show the corrected stack.
