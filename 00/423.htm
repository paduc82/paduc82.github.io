ken sheridan say:
I created a table in SQL (job_Categories), and I am importing it into an Access DB (ODBC)...   I created a form of checkboxes, the record source is the Job_Categories table, so each field has a checkbox.


Having a separate column in the table for each job category is a bad design as it is 'encoding data as column headings'.  A fundamental principle of the database relational model is the Information Principle.  This requires that all data be stored as values at column positions in rows in tables, and in no other way.

A correct design would be firstly to have a table with one row per job category.  Then you'd have a table which models the many-to-many relationship type between this table and your current table (sometimes called a junction table) by having two foreign key columns, one referencing the primary key of your current table, the other referencing the primary key of the new table of job categories.  So for each job category relevant to a row in your current table there would be a row in the junction table.

To return rows from your current table (now without the multiple job category columns) which have related rows in the junction table which include all of a set of selected categories you could use a multi-select list box to select as few or as many categories as desired.  Code in the form's module could then loop through the list box's ItemsSelected collection and build the SQL for a query to return the relevant rows. The type of query for this can be illustrated using the sample Northwind database:

SELECT *
FROM Orders
WHERE EXISTS
   (SELECT [Order ID]
    FROM [Order Details]
    WHERE [Order Details].[Order ID] = Orders.[Order ID]
    AND [Product ID] IN(7,51)
    GROUP BY [Order ID]
    HAVING COUNT(*) = 2);

This query returns all orders which include products 7 and 51 (Dried apples and dried pears) as rows in the related Order Details table.  It does this by the seeing if the subquery returns rows which have either of the two products, and the number of rows returned by the subquery is the same as the number of specified products, two in this case. With the version of Northwind shipped with Access 2007 this returns two orders, one of which includes only the two products, the other of which has three products, two of which are those specified.  Building the SQL for such a query in code on the basis of items selected in a multi-select list box is quite straightforward.  The RecordSource property of the form could then be set to the SQL statement, and the form requeried to show only the relevant rows.

trish_c say:

{ASK list "Insert List? (Y or N)" \d "N"}{REF List}{IF REF List} = "Y" "{INCLUDETEXT \\\\kk1\\share\\tw\\forms\\list.doc}

jay freedman say:
It might just be a transcription error, but the expression you showed is missing a couple of field markers. There should be a left brace (marker) between the IF and the REF, and another right brace (marker) at the end of the whole thing.

You can omit the {REF List} field between the ASK field and the IF field, unless you really want the user's answer to appear in the document.

Also, you should remove the unmatched double quote before the {INCLUDETEXT and add a pair of double quotes (with nothing between them) between the two closing braces at the end -- that's the 'else' part of the IF field.

Here's the final result:

{ASK List "Insert List? (Y or N)" \d "N"}{IF { REF List } = "Y" { INCLUDETEXT \\\\kk1\\share\\tw\\forms\\list.doc } ""}

VBA Refer to another sheet on same Workbook - Microsoft Community

ajxo say:

VBA Refer to another sheet on same Workbook

Hi

 

I want to get some data on Sheet 1 from Sheet2 in the same workbook.

Catch is the Sheet 2 may or may not have the same name everytime the VBA is run to fetch data (its a monthly sheet and tab might show diff names).

 

That sheet will always remain as second tab in the workbook.

Is there a way we can refer the to Sheet 2 by number rather than its name so VBA wont get confused when the Sheet2 do not have the same name.

 

jeeped say:
Given the nature of the formula, you could write the formula into Sheets(1).Range("C3:C7") then convert the formulas into their returned values in the next line.

Sub test1()
    Sheets(1).Range("C3:C7").Formula = "=COUNTIF(" & Sheets(2).Name & "!G:G, ROW(1:1)-1)"
    Sheets(1).Range("C3:C7") = Sheets(1).Range("C3:C7").Value
End Sub

This writes the formula into the C3:C7 range and then performs an operation that mimics the Copy, Paste Special, Values.

Alternately, you can use worksheet functions to return a value directly into the target cell within VBA code like this.

Sub test2()
    Dim r As Long
    For r = 0 To 4
        Sheets(1).Cells(3 + r, 3) = Application.CountIfs(Sheets(2).Columns(7), r)
    Next r
End Sub

jeeped say:
An integer can be used to refer to the ordinal position of the worksheets. This allows you to bypass worksheet names as the identifier.

   With ThisWorkbook
      .Sheets(1).Range("A1") = .Sheets(2).Range("A1").value
   End With

That will transfer (aka copy) the value of Sheet2!A1 to Sheet1!A1 regardless of the worksheet names.
