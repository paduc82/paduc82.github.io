Here is a function that can be called directly from the AfterUpdate event of an unbound combo box, where the name of the target field is either placed in the combo's Tag property, or else is part of the combo's name.

'----- start of code -----

Public Function ComboSearch()

    Dim frmForm             As Access.Form
    Dim cboCombo            As Access.ComboBox
    Dim strDelimiter        As String
    Dim strFindFieldName    As String
    Dim strFindCriteria     As String

    Set cboCombo = Screen.ActiveControl
    
    'If the combo box is Null, there's nothing to find
    If IsNull(cboCombo) Then Exit Function
    
    'Get a reference to the form object containing the combo box.
    Set frmForm = cboCombo.Parent
    
    With cboCombo
        If Len(.Tag) = 0 Then
            strFindFieldName = .Name
            Select Case True
                Case strFindFieldName Like "cbox*": strFindFieldName = Mid$(strFindFieldName, 5)
                Case strFindFieldName Like "cbo*": strFindFieldName = Mid$(strFindFieldName, 4)
                Case strFindFieldName Like "cb*": strFindFieldName = Mid$(strFindFieldName, 3)
                Case strFindFieldName Like "txt*": strFindFieldName = Mid$(strFindFieldName, 4)
                Case strFindFieldName Like "lst*": strFindFieldName = Mid$(strFindFieldName, 4)
            End Select
        Else
            strFindFieldName = .Tag
        End If
    End With

    strDelimiter = GetDelimiter(frmForm.Recordset(strFindFieldName).Type)
    strFindCriteria = strFindFieldName & "=" & strDelimiter & cboCombo.Value & strDelimiter
    
    ' If the form is dirty, ask the user what they want to do with their changes.
    If frmForm.Dirty Then
    
        Select Case MsgBox( _
                        "You have made changes to the current record.  Do you want to save them?" & _
                            vbCr & vbCr & "Choose Yes to save the changes, No to discard them, or " & _
                            "Cancel to stay on this record.", _
                        vbYesNoCancel + vbQuestion + vbDefaultButton1, _
                        "Save Changes?")
                        
            Case vbYes: frmForm.Dirty = False
            Case vbNo: frmForm.Undo
            Case Else: Exit Function
        
        End Select
    
    End If

    ' If we get here, we're going to move the form to a new record.

    With frmForm.RecordsetClone
        .FindFirst strFindCriteria
        If Not .NoMatch Then
            frmForm.Bookmark = .Bookmark
        Else
            MsgBox "No matching record was found.", vbInformation, "No Matchimg Record"
        End If
    End With

    Set cboCombo = Nothing
    Set frmForm = Nothing

End Function

'----- end of code -----

If you want to find the matching value in field "Foo", the combo box's name may be any of

  • Foo
  • cboFoo
  • cboxFoo
  • cbFoo
  • lstFoo
  • txtFoo

and you can add to that list of recognized prefixes in the code.  Or, you can name the combo anything you like and provide the name of the field to be searched in the combo's Tag property.

To use this, on your form, you would clear the ControlSource property of the two search combos, making them unbound, and you would set the Tag property of "cboName" to "Nama", since that is the actual field name being searched, not "Name".  You don't have to set the Tag property of cboEmpId, since "EmpID" is the field it will search.

Then, for each search combo, you would change the AfterUpdate property to:

    =ComboSearch()

That would replace the text "[Event Procedure]" on the AfterUpdate property line on the property sheet.  You don't need to create an event procedure to call this function directly.

One thing this code currently doesn't do is set the search combos to the field values from the current record.  The combos will continue to display the values you last entered in them.  If you want them to always show the field values of the current record (though not bound to those fields), you can use the form's Current event to set them.

Should I upgrade my existing version of Windows to Windows 10?

Since Microsoft made available the first pre-release build of Windows 10 in early October 2014, there has been some misunderstanding that this is not a product you should use to replace a production installation of Windows such as Windows 8 or 7 or Vista. Instead, if you must test it on a single machine, the best recommendation is to create a partition or have an a spare hard disk ready where you can install it. So if you must evaluate it, the best way is by installing it beside your commercial installation of Windows.
