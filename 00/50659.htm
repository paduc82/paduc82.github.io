When trying to add, install, uninstall or remove a program from Windows operating system, the setup will halt with the following error message “The Windows Installer Service could not be accessed. This can occur if you are running Windows in safe mode, or the Windows Installer is not correctly installed. Contact your support personnel for assistance”

This error symptom is most likely caused by the Windows Installer files are damaged or are missing, or Windows Installer Service is unregistered or damaged. Other reason can be incorrect permission issue on the related registry keys that Windows Installer trying to access. The error will only occur when the program’s setup installer uses the Windows Installer Microsoft Software Installation (MSI) package file.

This problem will also occur if the Startup Type property for the Windows Installer service is set to a value of Disable. To check whether the Windows Installer service is set to a value of Disable, follow these steps:

   1. Click Start, In the Start search, type services.msc in the Open text box, and then click OK.
   2. In the Services (Local) list, right-click Windows Installer, and then click Properties.
   3. If the Startup type drop-down list is set to a value of Disable, select the Manual option from the Startup type drop-down list, and then click OK.
   4. Click the File menu, and then click Exit.

I would also suggest you to follow the method of re-registering Windows Installer Service as given in the Method I in the link below

http://support.microsoft.com/kb/315346

If you’re still facing the problem, try to ensure that Windows Installer service is running, or force start it.

   1. Click Start, In the Start search, type in services.msc then press Enter. Locate Windows Installer in the all install services listing.
   2. Verify that the Status is Started. If the Status column is blank, right click on Windows Installer and select Start.
   3. If you get error trying to force start the Windows Installer service, try to change the Startup Type to Automatic (default is Manual), and then restart the computer.

You can download the latest version of Windows Installer from the link below:

These days I don't think we can get much that's useful from ping.  All it is really good for is checking DNS connectivity.  Since it uses ICMP instead of HTTP or HTTPS it doesn't even tell us whether there is something on the other side that is listening for those other protocols.  Probably your Connects measurement is more significant but depending on how that is implemented it might be interpreted by wary server operators as a possible Denial of Service attack and thus have its accuracy affected by whatever countermeasures the servers would be taking for that possibility.

Good to see that you got a clean test from Invoke-WebRequest.  I think that is a good way of checking how well your browser might do but again, depending on how servers would treat requests from unknown "browsers" even it may be inconclusive.  It works for me on Answers so I know that it is likely to help check for others too.  Something that using it seems to do is act as a kind of flush or priming of a connection for our browsers.  In any case it seems to me to be a more satisfying "Fix" than the unseen one that the Windows Network troubleshooter would otherwise hopefully be doing.
