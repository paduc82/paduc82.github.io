cindy meister say:

Hi Mac

The key with this is the fact that a graphic in Word that has "text flow" is anchored to a paragraph on the same page. And the graphic can be formatted to "Move with text" (up and down the page as the paragraph to which it is anchored moves) - or not. By default, this option is on.

Unfortunately, you don't provide any information about which version of Word you're using, which means it's difficult for us to give you the exact path to this option...

Try right-clicking, then choose Format AutoShape (or something similar - near the bottom of the context menu). Go to the Layout tab, click Advanced (or whatever the button is labelled). In the Picture Position tab, look for the checkbox - it will be near the bottom of the dialog box. Remove the check mark.

Note: I would probably insert a table with one column and two rows, then set the row heights to an "exact" measurement and apply a border only to the inner line between the two rows. This has a number of advantages, not least that you'll have to independent paragraph marks. With the approach you're trying, you're not going to be able to copy/paste easily, as Word will need to "type" paragraph marks until the text goes past the position of the line before you can paste. Using a table would also avoid the problem you're having currently.

suzanne s. barnhill say:

If I understand what you are trying to do, it would be much simpler to use the "2 pages per sheet" selection under "Multiple pages" on the Margins tab of Page Setup. With portrait orientation, you get two landscape halves printing on the same sheet. You don't actually have to duplicate the problems top and bottom, as you can just print pages 1,1 to print the same page twice.

karl timmermans [outlook mvp 2012-2018] say:

I want to move my current .pst files to a new Outlook setup, I have 4 pop accounts and 3 IMAP. 

"outlook files" folder is available in "My Documents" , it contains one .pst file per email address as well as one generic outlook file and 2 archive files (why?). 

I could use some help...


Just to clarify a few things

#1 - The data files that are of utmost importance are the ones currently in use in your Outlook profile (or the ones being used in any given profile if you use more than one profile). The definitive way to determine that is by going to 

File --> Account Settings --> Data Files tab

- that shows what data stores (.ost/.pst files are currently active along with their location

#2 - For simplicity - .ost files are of no value since these automatically get re-created any time the applicable email account gets configured.

#3 - In terms of <Imap> accounts in OL'2010 or earlier - the data store for the email folders related to a given Imap account also gets automatically re-created when the account gets configured.

#4 - POP accounts can all use the same PST file or each can have their own - that can easily be determined when looking at the accounts under the Account Settings --> Email tab and checking the destination file at the bottom of that window for each POP account highlighted.

#5 - What's of prime importance is the "default data file" which contains the default contact/calendar folders for the profile which can be either one of the pst files for a POP account or a stand-alone PST file. If it's a stand-alone file and you don't copy it over - you will be missing your contacts/calendar data.

#6 - You can have a 100 different files in your <Outlook files> folder - irrelevant since it doesn't mean that they are being used. All kinds of reasons for how those could have been created. As for your <archive> files - whether those are important or not depends on whether or not you are actively "archiving" your data. At a minimum, all that multiple <archive> files in the folder means is that at some point, the option was set and the file was created. Only you know your archiving history and the file(s) currently configured depends on your global and folder specific archive settings.

#7 - When moving to a new computer using OL'2010 - you should make sure that you use the existing PST file for that account instead of letting Outlook create a new one. If the default data file in the original Outlook configuration was stand-alone then add it to your profile manually and make sure it is set to be the "default" again in the new configuration.

#8 - Lastly, also important to remember how and what Outlook creates in terms of data files depends on the "sequence" in which the accounts are configured. Makes a difference if a POP account is configured before an IMAP account (or vice versa) in terms of what data file is automatically set as the 'default" for the profile. Also, what gets created automatically differs for OL'2013/'2016 versus OL'2010 or earlier.

Hum, I'm running a lot of windows 10 machines, and not seeing this issue (and we also using SQL server).

So, you turned off the computer from sleeping?

You as noted  turned off the power management for the network card:

eg this:

The other thing to check is if you have more then one work group or domain (if you have more then one, then win 10 can sometimes try to switch to the other network and thus you lose your connection. Same goes if you are using some type of VPN. 

And of course, we assume this is not wi-fi? (again, they can often go to sleep, or disconnect).

Because of "domain" and DNS resolution? Try linking to the SQL server using the IP address as opposed to the server name. 

So, in place of say:

SRV-PRODUCTION\SQLSERVER

192.168.0.50\SQLSERVER     (of course change the example IP here to the server ID).

So, if you are "super" confident the network card is not going to sleep, then the multiple network connections (such as VPN, cloud, your work-group, your work domain) etc. can cause the network stack to flip between such DSN resolutions with multiple networks, and thus you lose the name resolution to the server. By using the IP address to link the tables, you can thus often eliminate this issue from your troubleshooting efforts. 

Edit:

This also means turning off wi-fi if on the given computer - as once again, the flip from one network to the other can occur - again messing with "name" resolution.
