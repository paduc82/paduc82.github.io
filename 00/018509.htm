Don't make taskbars longer for nonworking days (weekends) - Microsoft Community

prj9029 say:

Don't make taskbars longer for nonworking days (weekends) in Gantt charts

One drawback of Project's Gantt charts is that nonworking days (i.e. weekends) are displayed in the bars, making tasks that begin before a weekend take up more time than tasks of the same work that don't span a weekend.  Wikipedia notes this deficiency of Project http://en.wikipedia.org/wiki/Gantt_chart .

Other software, such as OmniPlan, has the option to remove nonworking days from the Gantt chart completely, just displaying a line instead.  This helps you get accurate visualizations of your tasks.  It's especially helpful in the early stages, when you don't know the exact start date of a project.  There's no reason to elongate tasks if they happen to be currently expected to span a weekend.

How can I get Microsoft Project to do that - omit nonworking days from the Gantt chart completely?

dale howard [mvp] say:

prj9029 --

There is no way to hide weekends in any timephased view, such as the Gantt Chart view or the Task Usage view.  What you CAN do, however, is to display the weekends in FRONT OF the Gantt bars instead of behind them.  To do this, double-click anywhere in the Timescale bar at the top of the Gantt Chart pane.  In the Timescale dialog, click the Non-Working Time tab.  On the Non-Working Time page of the dialog, select the In Front of Task Bars option and then click the OK button.  I will warn you, however, that you may not like the results.  But keep in mind there is no way to remove nonworking time, such as weekends, from the Gantt Chart view or any other timephased view.  Hope this helps.

robert aldwinckle on forums say:

When I then refreshes the webpage, the browser has no problem showing it.

It is better to use the Fix connection problems button.  That will do an automatic Refresh for you.

The symptom is typically caused by a slow DNS lookup, so getting a faster DNS could change the frequency of it.  You will always see it to some degree but there are workarounds available if you get stuck in a repeat scenario which essentially involve caching enough pieces of the lookup to allow IE to proceed. 

Once IE has started a session it caches a lookup for a relatively long time, so ironically there can be problems in the other direction, e.g. where it may still be using an IP address that the host system might want to take down or switch over to some other use.

FYI
