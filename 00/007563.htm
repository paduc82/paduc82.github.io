The departments in my hospital submit reports to me each month, but because different staff members do the reports each month they do not always consistently use the same name for their department.  I have a list of all the variants I have found and have matched against the standard names (E.g., column FieldName1: variant spellings, FieldName2: normalized spelling). This table is then linked to another table so that when I run my reports all the names are normalized.

My question is on the variant list I have both FAMILY MEDICINE and Family medicine and in the Normalized list I have Family Medicine.  Is this correct: does Access see them as two different names; or is Access case-sensitive blind in this instance.  In which event, might Access double count?

 

Thanks.

dirk goldgar say:

The departments in my hospital submit reports to me each month, but because different staff members do the reports each month they do not always consistently use the same name for their department.  I have a list of all the variants I have found and have matched against the standard names (E.g., column FieldName1: variant spellings, FieldName2: normalized spelling). This table is then linked to another table so that when I run my reports all the names are normalized.

My question is on the variant list I have both FAMILY MEDICINE and Family medicine and in the Normalized list I have Family Medicine.  Is this correct: does Access see them as two different names; or is Access case-sensitive blind in this instance.  In which event, might Access double count?


Access string comparisons are not normally case sensitive, so "FAMILY MEDICINE" = "Family medicine" = Family Medicine".  If you have a query that joins the submitted reports to your normalizing table on something like DeptName = VariantName, you'll end up with duplicate records.

At the cost of execution efficiency, you can force a case-sensitive comparison with the StrComp function, which takes as arguments the two values to be compared and an optional third argument to force a binary comparison (which would be case-sensitive).  However, if you explicitly JOIN on such an expression, you'll have to build the query in SQL view, as the query designer won't be able to represent it in design view.

I would be inclined to fix up the departments' reports when I bring them into the database (which I guess is what you're doing), by running an update query that normalizes the department names immediately after import.  Then any later queries wouldn't need to worry about name variations.

Word 2013 Template Cascading - Microsoft Community

Word 2013 Template Cascading

Is it possible to create a Word 2013 template based on another template, such that any changes made to the base template are reflected in all templates that are derived from it.

Regards,

Andrew

jay freedman say:
Sorry, no -- Word isn't built on CSS principles. When each secondary template is created from the base template, the styles and other elements are copied from the base, and then their relationship ends.

If you open a secondary template and go to Developer > Document Template, you'll see that the "Document template" field (which, for a document, shows the attached template) is grayed out and the Attach button is disabled. There is no reference to the base template.

You could use the Organizer to copy changed styles and/or macros from the base to each secondary, but that would be a manual, one by one process. To do it in bulk, you would have to write macro code.

"This problem occurs when the file is on a server that runs Microsoft Internet Information Services and the server is using NT Challenge Response authentication.

 

To work around this problem, create a new registry key. To do this, follow these steps.

 

Warning Serious problems might occur if you modify the registry incorrectly by using Registry Editor or by using another method. These problems might require that you reinstall the operating system. Microsoft cannot guarantee that these problems can be solved. Modify the registry at your own risk. Make sure that you back up the registry before you modify it. Make sure that you know how to restore the registry if a problem occurs.

  1. Click Start, click Run, type regedit.exe, and then press ENTER.
  2. In Registry Editor, locate and click the following subkey:

Excel 2007:  HKEY_CURRENT_USER\Software\Microsoft\Office\12.0\Excel\Options

3.    On the Edit menu, click Add Value. Create the following registry value:

Value name: AlwaysSaveToCurrentPath
Data type: REG_DWORD
Radix: Decimal
Value data: 1

4.    Exit Registry Editor.

Excel will suppress the NT Challenge Response dialog box. Excel will open the Save As dialog box instead.
