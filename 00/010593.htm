ken sheridan say: Firstly. don't despair as you read this; there's a get out of jail card at the end.

You've made a very common mistake, I'm afraid, which is to do what is known as 'encoding data as column headings'.  A fundamental principle of the database relational model is the Information Principle.  This requires that all data be stored as values at column positions in rows in tables, and in no other way.   In your case each column represents a data value, the question.  The correct design is to have multiple rows per question in a related table, so the model would be made up of tables like this:

Employees
....EmployeeID (PK)
....FirstName
....LastName
....DepartmentID
....etc

Departments
....DepartmentID (PK)
....Department

Questions
....QuestionID (PK)
....Question
....QusesionIndex

The QuestionIndex column is simply to determine which order you want the questions presented in and would have values from 1 upwards.

ResponseSessions
....ResponseSessionID (PK)
....ResponseSessionDate
....EmployeeID (FK)

Responses
....ResponseSessionID (FK)
....QuestionID (FK)
....Response

The primary key of this table is a composite one made up of the ResponseSessionID and QuestionID columns.  The Response column would be of a Boolean (Yes/No) data type

For data entry you'd use a form in single form view based on the ResponseSessions
table with a text box control for the ResponseSessionDate and a combo box for the EmployeeID, set up in the usual way to show the employee's name's and department:

ControlSource:    EmployeeID

RowSource:     SELECT EmployeeID, FirstName & " " & LastName, Department  FROM Employees ORDER BY LastName, FirstName;

BoundColumn:   1
ColumnCount:   3
ColumnWidths:  0cm;3cm;3cm
ListWidth: 6cm

If your units of measurement are imperial rather than metric Access will automatically convert them.  The important thing is that the first dimension of the ColumnWidths property  is zero to hide the first column.  The ListWidth is the sum of the ColumnWidths dimensions.

Within the form would be a subform, in continuous forms view, linked to the parent form on ResponseSessionID and based on the following query:

SELECT Responses.*
FROM Responses INNER JOIN Questions
ON Responses.QuestionID = Questions.QuestionID
ORDER BY QuestionIndex;

In the subform would be a combo box bound to the QuestionID  column and a check box bound to the Response column.

As you want all questions answered you can automatically fill the subform as soon as a new row is inserted into the ResponseSessions table in the parent form by outing the following code in its AfterInsert event procedure;

Dim strSQL As String

strSQL = "INSERT INTO Responses(ResponseSessionID, QuestionID) " & _
    "SELECT " & Me.ResponseSessionID & ", QuestionID " & _
    "FROM Questions"

' insert rows into Responses for each question and current response session
CurrentDb.Execute strSQL, dbFailOnError

' requery subform to show new rows
Me.[YourSubformControl].Requery

Note that YourSubformControl is the name of the control in the parent form which houses the subform, not the name of its underlying form object, unless both have the same name of course.

You can see an example of this sort of set-up in action in the file Traetments.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

This little demo file includes a means of automatically inserting a programme of treatments for a patient. Go to the Patient Treatments form from the opening menu and go to a new record; then select a patient, the rabies immunisation treatment, and a starting date in the parent form; then click the Create Treatment Programme button.  You'll see that it inserts a set of rows into the subform ready for the date administered values to be entered for each.  It's a little more complex than yours as it dates the treatments according to a predefined schedule, but essentially it does much the same as yours would.

For transferring your current data to a new model as described above you'd first need to set up and fill all the tables bar the ResponseSessions and Responses tables.  Then it gets a little trickier as you need to firstly insert rows into ResponseSessions (one per row of your current table)  and then into Responses (one per row/column of your current table).  The first is easy to do, the latter requires a UNION ALL query which brings all the questions and answers into two columns only.  It's doable, but without knowing the full details of how your current table is structured I can't give you chapter and verse for the query to do this.

If your deadline is imminent I think you might find it difficult to get the database correctly restructured in the time available.  As a short term expedient you can create a UNION ALL query to work with:

SELECT [Name], [Department], "Q1" AS Question, [Q1]
FROM [YourCurrentTable]
UNION ALL
SELECT [Name], [Department], "Q2", [Q2]
FROM [YourCurrentTable]
UNION ALL
SELECT [Name], [Department], "Q3", [Q3]
FROM [YourCurrentTable]
UNION ALL
< and so on> to
SELECT [Name], [Department], "Q23", [Q23]
FROM [YourCurrentTable];

This will give you result table in which the data is correctly represented as values at column positions in rows, not encoded as column headings.  It's then a simple task to create queries based on the above query which aggregate the data for use as the RecordSource properties of reports.

While this should enable you to get the results you want at present, I'd strongly recommend that you bite the bullet and restructure the database as a set of correctly normalized table as a longer term solution.

ken sheridan say: You need to include another column in the result set, with the Q1, Q2 etc column name as a constant string expression (wrapped in quotes) in each part of union operation.  This will then identify which question the value in the Answer   column relates to:

SELECT PracticeGroupID, FeeEarnerID, "Q1" AS Question, Q1 AS Answer
FROM AuditData2012
UNION ALL
SELECT PracticeGroupID, FeeEarnerID,"Q2", Q2
FROM AuditData2012
UNION ALL

.

.

.

SELECT PracticeGroupID, FeeEarnerID, "Q23", Q23
FROM AuditData2012;

The aliases for the column headings only need go in the first part, and the whole query is terminated with a semi-colon.
