Parsing bug? - Microsoft Community

emmatosch say:

Parsing bug?

We have noticed an inconsistency in Excel formula parsing and/or evaluation for `SUMX2MY2`: the following expressions evaluate correctly:

`=SUMX2MY2(B1:B10, C1:C10)`

`=SUMX2MY2((B1:B10), (C1:C10))`

however, the following leads to a #VALUE! error:

`=SUMX2MY2((B1:B10, B12:B14), (C1:C10, C:12:C14))`

There are values in all of the cells referenced, which leads me to believe that the problem is in its handling of discontiguous (unioned?) ranges.

bernie deitrick say:

=SUMX2MY2(B1:B10, C1:C10) + SUMX2MY2(B12:B14, C12:C14)

or

=SUMX2MY2(B1:B14, C1:C14)

Which may work, depending on what is in B11 and C11.

emmatosch say:

Thanks for your reply.

I'm not actually looking for a workaround to use this formula; it's for a project that involves parsing Excel formulae and wanted to submit this as a bug somewhere. We believe the parens should coerce their contents into a range object/type, but for the discontinuity it's just causing an error. I'm trying to recover the source of that error and determine whether parens should behave as I expect them to.

bernie deitrick say:

You can believe what you want, but parens don't coerce anything - they don't consolidate ranges into one contiguous range or into a single contiguous array.  Your ranges each have two areas, which you can easily show using VBA, But the function does require two arrays or two contiguous ranges. Even if you broke a contiguous range into two areas that are actually contiguous (like (B2:B10,B11:B14) instead of B2:B14 ), Excel would still not treat them as one.

You could roll your own function using VBA code to do exactly what you want by looping through the areas of the ranges.  That's exactly why Excel has VBA function definition capability.

danbarowy say:

Bernie, just curious: how do you know that SUMX2MY2 takes two contiguous ranges as parameters (other than the fact that Excel fails when you try it)?  I have searched high and low for precise documentation (i.e., formal declarations) for Excel functions, and I am unable to find them.  The best I could come up with are:

  1. Microsoft's documentation (https://support.office.com/en-us/article/SUMX2MY2-function-9e599cc5-5399-48e9-a5e0-e37812dfa3e9)
  2. TechOnTheNet's docs (http://www.techonthenet.com/excel/formulas/sumx2my2.php)
  3. ExcelFunction.net's docs (http://www.excelfunctions.net/Excel-Sumx2my2-Function.html)

all of which are rather vague about what constitutes a "range".  Indeed, Microsoft's docs seem to imply that unioned range references are just range references (https://support.office.com/en-us/article/Calculation-operators-and-precedence-48be406d-4975-4d31-b2b8-7af9e0e2878a) but as this Stack Exchange post  points out (http://superuser.com/questions/447492/is-there-a-union-operator-in-excel), that appears not to be the case.

As Emma points out: we don't really need a workaround.  We're just trying to precisely model how Excel evaluates formulas, and this strikes both of us either a bug or a shortcoming in the documentation.  In fact, we found this because we have our own parser and it does something different.

static LPXLOPER12 TempStr12(const XCHAR* lpstr)
{
 LPXLOPER12 lpx;
 XCHAR* lps;
 int len;

 len = lstrlenW(lpstr);

 lpx = (LPXLOPER12) GetTempMemory(sizeof(XLOPER12) + (len+1)*2);

 if (!lpx)
 {
  return 0;
 }

 lps = (XCHAR*)((CHAR*)lpx + sizeof(XLOPER12));

 //lps[0] = (BYTE)len; <-- WRONG to cast to BYTE, this limits value to 255!
 lps[0] = len;

 //can't wcscpy_s because of removal of null-termination
 wmemcpy_s( lps+1, len+1, lpstr, len);
 lpx->xltype = xltypeStr;
 lpx->val.str = lps;

 return lpx;
}
