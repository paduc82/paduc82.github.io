Access Duplicate Records

I am using Access 2010 and I created a table "Database" and the table has the following fields 

ID, Name, Mobile no, Email

I did a form to easily see the records and easily add new records but the thing is I want to check if the Mobile no that I will insert is already there or not, if it is already in the table I want a warning that the record already exists, thats it !! 

vanderghast say:

You can either make Mobile an indexed field not allowing duplicated values (that will ensure you that no duplicated value will exist in the databe, whatever the mean used to add records is taken by the end user, and NOT JUST the actual form/ process that you have in mind), or, if you only want the check to be made in ONE specific case of appending record, under a FORM, then, in the before update even for the FORM, you can DCOUNT the number of record having this mobile value. Assuming Mobile is a string:

if 0 = DCount("*", "TableNameHere", "Mobile=""" &  Me.Mobile & """" ) then
    ' ok, continue
else
    MsgBox "Mobile already known: " & Me.Mobile, Title:= "RECORD NOT SAVED"
    Cancel = true
end if

 

But again, it will be possible to add a dup using other means, than that form, such as using a query (to append data), or by editing the table in data view, if you proceed this way. So, an index is probably the safer mean.

scottgem say:

If you set an index as Duplicates No, this will prevent any duplications. However, it will not inform you that a duplicate exists until you try to save the record. With a table as small as yours, this may not be a big deal. If you want to check for a dup as soon as you enter the Mobile No, then use code like Vanderghast gave you in the After Update event of the control where you enter the number.

As this appears to be your first foray into using Access, I hope you don't mind some tips. First, don't use spaces in object names. You can use MobileNo or Mobile_No as alternatives. Second, each field should be as discrete a piece of info as possible. Names should be broken out at least into First and last names. Generally I use 5 fields for people names (Salutation (Mr, Ms etc), First, Middle, Last, Suffix (Jr Sr, etc.)). Third, use meaningful names for your tables and objects. Instead of naming the table "Database", call it Contacts or tblContactInfo. Finally, name your Primary keys according to the table. So instead of ID call it ContactInfoID.

 

Hope this helps, Scott<> P.S. Please post a response to let us know whether our answer helped or not.
