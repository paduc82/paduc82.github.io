chatmaker say:

VLOOKUP doesn't care about how many headings or rows we have. We just need to adjust its column number parameter to match the column we want.

However, if we use the table approach AND the column headings in ENTRIES matches those in BUDGETS one formula can do it all:


=INDEX(INDIRECT("BUDGETS[" & ENTRIES[#Headers] & "]"),MATCH([@People],BUDGETS[People],0))

Here is how it works:

INDIRECT("BUDGETS[" & ENTRIES[#Headers] & "]")

This says get the ENTRIES header for this formula's column and put that name between the brackets of BUDGETS[]. Then get the address using INDIRECT which attempts to convert strings into ranges.

MATCH([@People],BUDGETS[People],0)

This says get the EXACT MATCH (the 0 at the end) of the value in the PEOPLE column that is in the same row (that @ sign) as this formula against the PEOPLE column in the BUDGETS table and return its row # within the BUDGETS table.

INDEX(..

This says get the value from the BUDGETS table column determined by our INDIRECT() function  that is in the row determined by our MATCH() function.

This type of referencing (BUDGTS[People]) is known as Structured Referencing. To learn more about structured referencing see:  https://support.office.com/en-us/article/Using-structured-references-with-Excel-tables-f5ed2452-2337-4f71-bed3-c8ae6d2b276e

Trust me. They don't take long to learn. And once you know them and start using tables, XL gets a lot easier.

There is no question the Adobe Acrobat and Internet Explorer integration is messed up (on both sides - Microsoft and Adobe)

For example, even if you disable the Acrobat plug-in it still loads.   This can cause major problems when say using Protected Mode to run IE and running IE in 64-bit.

If you are, the behavior is apparently very Acrobat version specific.   I have the latest patched version of X and here are some observations in case it helps you:

If I have the Display PDF in browser option  (in Acrobat Edit | Preferences  category Internet) unchecked, it runs anyway when trying to view a PDF file in Internet Explorer.   Blame this one on Adobe I believe for how they did the integration.    If anyone knows how to manually uninstall Acrobat's Internet Explorer plug-in that would be helpful!

Anyway, this results in the display of the following dialog:

"this webpage wants to run ' Adobe PDF Reader' which isn't compatible with Internet Explorer's enhanced security features.  If you trust this site, you can disable Enhanced Protected Mode for this site and allow the control to run".   (funny, I have this disabled ?!?!?!?!?)

You're given a yes | no option. 

If you don't select anything within a certain period of time, the prompt goes away and you'll get a blank window with a little black box with a white 'x' in the upper left corner.  

If you select 'yes",  IE goes into an infinite loop (really brilliant IE engineering... love how your security features in effect cause a hanging condition!).  During the loop, the PDF file will open but you'll be prompted over and over again until you select no.   For each time you select "yes" you'll get a opened window showing the PDF file.  

Now if you select 'no' the first time, you get the blank window with the black box / white 'x'.   HOWEVER, something interesting does happen when you select 'no' on first prompt. IE *REMEMBERS* the website so the next time you try to open a PDF file from that site, it won't run the Adobe plug-in but instead you'll get the "open | save | cancel" dialog which works as expected; this is if you have the Adobe Preferences set to not display PDF files in the browser.

If you do have the Adobe Preference to display in the browser set, selecting no simply causes the PDF to open out of window and selecting 'yes' causes it view in window.   IE will remember what you selected for this website and future attempts to view a PDF file at this site will directly result in whatever behavior you chose the first time.

Can it get any freak'in more complicated!!!
