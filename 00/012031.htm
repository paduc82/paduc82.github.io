scottgem say:

First, it is not good design to store a calculated value. Total hours and Week Total should be calculated in a query not stored in your table. 

Also, if your week always begins on a Monday, you don't need to store the Week Begin date as that can be calculated also. 

So I would use 2 queries. The first to calculate Daily Totals, the second Weekly Totals

SELECT EmployeeID, WorkDate, StartTime, EndTime, Lunch, EndTime-StartTime-Lunch As DailyHours

FROM table

WHERE WorkDate BETWEEN WeekBegin AND WeekBegin+7;

Then you can use A Group BY query based on that query

SELECT EmployeeID, Sum(DailyHours) As WeeklyHours

FROM qryDailyHours

GROUP BY EmployeeID;

ken sheridan say:

Why not use the date/time data type for the times?  This data type is implemented in Access as a 64 bit floating point number as a duration in days, so there is no reason whatsoever for not using it for time durations of less than 24 hours.  Data entry of the times is then more intuitive as times can be entered as hours:minutes rather than having to express the minutes as a decimal fraction of an hour.  Your expression would then be:

EndTime - StartTime - IIf(EndTime - StartTime > #5:30#,#00.30#,0)

You can see how it works by entering some literal values in the debug (aka 'immediate') window:

StartTime = #05:00#
EndTime = #12:30#
? Format(EndTime - StartTime - IIf(EndTime - StartTime > #5:30#,#00.30#,0),"hh:nn")
07:00
StartTime = #14:00#
EndTime = NULL
? IsNull(EndTime - StartTime - IIf(EndTime - StartTime > #5:30#,#00.30#,0))
True

To compute the cumulative weekly total hours per day sum the total daily hours for the week up to and including the current day.  A date/time value summed in this way cannot be formatted as "hh:nn" of course as for values of 24 hours or more the result would be the remaining hours less the number of complete days.  However it can be done by using the TimeElapsed function from TimeArithmetic.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo file illustrates the use of a number of functions for doing arithmetic on date/time values.  Again you can see how it works in the debug window by multiplying the result of the expression used above by 4 to give a value over 24 hours::

StartTime = #05:00#
EndTime = #12:30#

If we multiply the 7 hours returned above by 4 this returns 28 hours of course, but if we use normal time formatting on this we incorrectly get:

? Format((EndTime - StartTime - IIf(EndTime - StartTime > #5:30#,#00.30#,0))*4,"hh:nn")
04:00

Whereas my TimeElapsed function returns the correct value:

? TimeElapsed((EndTime - StartTime - IIf(EndTime - StartTime > #5:30#,#00.30#,0))*4,"nn")
28:00

A query to compute the total hours per day and the cumulative weekly total at each day would be along these lines, in which TimeLog is the name of the table:

SELECT T1.*,
Format(EndTime - StartTime - IIf(EndTime - StartTime > #5:30#,#00.30#,0),"hh:nn")
AS TotalHours,
   (SELECT TimeElapsed(SUM(EndTime - StartTime - IIf(EndTime - StartTime > #5:30#,#00.30#,0)),"nn")
    FROM TimeLog As T2
    WHERE T2.WeekBegin = T1.WeekBegin
    AND T2.Workdate <= T1.WorkDate)
AS WeekTotal
FROM TimeLog As T1
ORDER BY WorkDate;

If you continue with the use of a number data type for the times then the query would be the same of course, but the times in the query would be expressed as numbers in which the minutes are expressed as decimal fractions of an hour, and you would not need to call the TimeElapsed function to return the value as hh:nn.

If you want the query to return an updatable result table then you'd need to call the VBA DSum function rather than a subquery.  The Balances demo in my same OneDrive folder has examples of this, but using currency data rather than times.

Note that in the above I've removed the spaces from your column names.  Spaces and other special characters should be avoided in object names.  Use CamelCase or represent a space by an underscore character like_this.  If you do use spaces the table and column names must be wrapped in square brackets in the query [like this].

alan wu msft say:
Hi Jabbaaa,

Id like to answer your 2 questions one by one as follows:
1. May I know if the Oath token is shared between Skype and Office client?
Yes. To be more accurate, this is because Skype for Business clients connect to both Skype for Business Online and Exchange Online, tenant level modern authentication settings should match for Exchange Online and Skype for Business Online. For more information, you can refer to https://social.technet.microsoft.com/wiki/contents/articles/34339.skype-for-business-online-enable-your-tenant-for-modern-authentication.aspx (the fifth paragraph of Defaulst).
2. May I know if they all office rich client shares the same authentication token? We would like to understand which MS application will share the Authentication token (e.g. MS office) and which will not (e.g. Skype). Is there any document we can reference?
Yes, they all office rich client shares the same authentication token. daniel g. grau say:
Pages can be defined with arbitrary sizes/non-printing areas through File > Page Setup > Paper Size > Manage Custom Sizes. While some printers can print all the way to the edge of a page (borderless), others cannot (their print head cannot reach the edges of the paper, i.e., there is a non-printing area). For regular documents where text does not need to reach the edge of the page, this would not make any difference.

If your documents are being created as "US Letter (borderless)," you need to open and edit your Normal.dotm template to change its paper size (as indicated above) and perhaps its margins as well (Format > Document > Margins). To do this, see my August 29 post in this thread.
