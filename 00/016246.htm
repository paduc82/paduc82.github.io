paul domaskis say:

TransferSpreadsheet: Controlling field data type & source data range

I've been reading about these endemic challenges for years.  Users cannot control the data type of a field in an Access table created from TransferSpreadsheet.  Apparently, the number of Excel records that are read in to determine the data type is set in the registry.  In a corporate environment (where Access is likely to be used), it is not uncommon for user accounts to be unable to modify the registry.  Furthermore, changing this parameter doesn't necessarily solve the problem because the first records might not have data of the right type in all fields.

To provide indications of the right field data types, I ended up creating 30 dummy records at the top of my source data table in Excel.  I hope that 30 records will always be sufficient, but as I said, that seems to be determined in the registry, and who knows what forces may cause that value to change in the future.

This all seems barbaric.  I hope that TransferSpreadsheet is updated or superceded by something that (i) accepts an argument that specifies the number of records used to decide on the default field data types; (ii) allows the user to specify the data types for specific fields; and/or (iii) as the source data, accepts table names that have been formally defined in Excel.

john w. vinson say:

The basic problem is that Access has "strong" datatypes (each field must be defined to be Text, Date/Time, some flavor of Number, etc.), while Excel does not. There is (to my knowledge) nothing in an Excel workbook which constrains the datatype of a cell on a sheet, even if that cell is in a Table - you can only control the format, which is just the appearance. 

One thing that might help is to know the hierarchy by which Access "guesses" the datatype. If all the records in the trial area (which could be just one row!) are integer numbers, Access will use Long Integer; if a cell has what looks like a Date, it will use date/time; and so on. If it doesn't look like any defined datatype it will use the lowest common denominator, Text. What often happens is that a sheet contains mostly numbers but has some rows with (e.g.) "N/A" or "See Notes" or "Ask the boss"; if so a Text datatype is the least-bad alternative, and just putting "x" in the top cell of this column will force that outcome.

It would be nice if Excel offered the option of defining cell datatypes (separately from Formats!), but since that would risk breaking billions of spreadsheets across the world I don't see it happening!

paul domaskis say:

I get the strong vs. weak typing, but the import process currently chooses a strong type right now anyway.  And it does this based on factors that the user has no control over or visibility into. We have to do things like adulterate the source data (which might provided from elsewhere) and create dummy records to indicate the data types, and hope that we create enough of the dummy records.  Having the import function accept arguments (e.g., an array that serves as a lookup table to determine the type of each field) seems obviously more sensible.  Perhaps it might even be flexible enough to allow users to specify types for a subset of the fields, and have clearly laid out default rules for the rest of the fields (rationale rules).

Some of the current decisions on type seem to defy any kind of logic at all.  Why would text fields import into Access with right justification?  It took half a day to figure out how to reach into the field properties to make them all consistent.

TransferSpreadsheet may have been the cat's meow back in the day when we didn't have any idea as to a more reasonable design, but it's just a frustrating road block these days.  I'm baffled as to why a more civilized conduit between products of the same suite from a single vendor hasn't been updated.  Importing data properly is doubtlessly challenging for the reasons that you said, but TransferSpreadsheet makes it much more painful than it has to be.

paul domaskis say:

I think it's a total disservice to users to go through what I did, totally unnecessary, and even hurtful to Microsoft's user base.

In the interest of curtailing this sad situation, maybe I can start the ball rolling on the cheats that can save some pain due to uncontrolled data typing on import.  I fixed it by creating fake data of the unambiguously desired type at the top of the spreadsheet table.  That's OK for *one* spreadsheet, but if you have a dozen spreadsheets, it's an abortion.  Not only that, you're corrupting the source data.

In my second case, I was in fact faced with importing data from a dozen spreadsheets into a common table.  Each one imported into its own table in Access, then a series of SELECT INTO queries consolidated the info.  The problem was that, even though the field names were the same across the dozen input tables, the automatically determined data types were not.  They were determined by Access's best guess based on the data.  My real show stopper (temporary, thankfully) was that numerical fields were interpretted as text in some of the input tables.  Not even conversion functions like CInt worked, because some of the data *was* text (it only becomes purely numeric or null once I apply cleanup queries to excise rows that don't actually correspond to data).  So the key was to write a VBA function to be used in the SELECT INTO query, e.g., Function ReturnIntOrNull( InputField As Variant ) As Variant.  You can do whatever you want within the function, e.g., test if InputField is text (or non-numeric), return Null if so, return the original value otherwise.  One can experiment for different situations of input data.

Hopefully, this saves someone some time.
