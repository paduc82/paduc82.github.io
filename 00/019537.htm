Assuming you mean the number of fields in the data source, and not the number of Merge fields you have placed in your document (AFAIK the limit for that is either "available memory" or very large indeed), then

 a. as Doug says, it depends on the data source. But it can also depend on how Word connects to the data source. For example, if you connect to an Excel data source, there are at least 4 ways Word might do it:

 - converter (obsolete - you can still try it if you have it, but I wouldn't)

 - DDE (basically obsolete, but can sometimes still be used)

 - ODBC (obsolescent since Word 2002, roughly)

 - OLE DB (the default in Word 2003 and 2007)

For DDE, ODBC and OLE DB with Excel there is a limit of 255 (or 256, I forget) columns. With Excel 2007, which supports many more columns, the mailmerge limit is still 255/256 columns. Personally, I have sometimes experienced a more sever limit on DDE connections to Excel in 2007  - AFAICR in 2003 there was no problem with 255, but in 2007 I have sometimes only been able to use about 70-75 (that might be something to do with the version of Windows and/or performance factors).

AFAIK all the "desktop database drivers" (except possibly the SQL Server one) impose a 255 column limit. The desktop database drivers are used for ODBC and OLE DB connections to Excel, Access, and via Access, to some other types of data source such as dBASE).

When connecting to delimited text files, Word may connect using the Access driver/provider, in which case there's a 255 limit, or using its internal text converter, in which case you may find that you can use 1000 columns or more. However, that seems to be limited more by the line length and perhaps complexity of the file than by the column count itself, and the limitations may well be more severe in Word 2007 than Word 2003. I've never found a formula to predict this.

With SQL Server sources I believe that with the right SQL Server+provider you should be able to go well beyond the 255 limit, and perhaps up to the maximum number of columns in a SQL Server table. (The same may well be true of other Server databases such as Oracle).

One other limitation is the number of field names that Word will display in its dropdowns (e.g. in field picklists) and the number of columns it will display in Edit Recipients. Those may be limited to a fixed number, or in the case of Edit Recipients, AFAICR there is a fixed maximum, but record size may reduce that. (There's also a maximum row count in Edit recipients.

ken sheridan say: Firstly, what you are asking for, i.e. that a row in a referenced table only be committed to the table if all constraints on rows in a referencing table are satisfied, is impossible in a correctly designed relational database.  In a relational database a row in the referencing table must exist before a matching row can be inserted into a referencing table.  Consequently you cannot enforce any constraints in the referencing table before the row in the parent table is saved.

In a form/subform, where the relationship type between the form's and subform's tables is one-to-many, the above is reflected by the fact that, as soon as focus is moved from the parent form to the subform, the data in the parent form is committed to a row in the referenced table.

What you have to do to replicate a constraint that no row can exist in a referenced table if any row in a referencing table fails to meet one or more constraints, is prevent the parent form moving off the current record, including by closing the form, until the constraints in all rows in the subform's table which reference the parent form's current record have been met. 

This cannot actually be done per se, so has to be replicated by doing the enforcement when the parent form moves to another record, i.e. in the form's Current event procedure.  This will execute when the form moves to the next record, including an empty new record, so if the constraints are not met the parent form is immediately moved back to the record whose referencing records do not meet the constraints.

An example of this is illustrated in Families.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.

In my case the only constraint is that at least one matching row should exist in the referencing table, so the code for the parent form's Current event is relatively simple:

Private Sub Form_Current()

    Const conCRITERIA = _
        "NOT EXISTS(" & _
            "SELECT * " & _
             "FROM FamilyMembers " & _
             "WHERE FamilyMembers.FamilyName = Families.FamilyName)"
   
    Dim varFamilyName As Variant
    Dim strMessage As String
   
    varFamilyName = DLookup("FamilyName", "Families", conCRITERIA)
   
    If Not IsNull(varFamilyName) And varFamilyName <> Nz(Me.FamilyName, "") Then
        strMessage = "A row for family " & varFamilyName & _
            " must be entered in the Family Members table."
   
        With Me.RecordsetClone
            .FindFirst "FamilyName = """ & varFamilyName & """"
            If Not .NoMatch Then
                 Me.Bookmark = .Bookmark
            End If
        End With
       
        MsgBox strMessage, vbExclamation, "Incomplete Data"

    End If

End Sub

You could adapt this to establish and loop through a recordset of all rows in the referencing table, and, where any violation of a constraint in a row in the referencing table is encountered in a row, immediately exit the loop and move the form to the record in the referenced table which that row references.  However, this would raise the question why such a row exists in the first place, rather than the insertion of a row into the referencing table via the subform being prohibited until all constraints have been satisfied.  This would then mean that the only uncovered constraint would be that at least one referencing row exists per parent record, which is what my demo enforces.

The similar code to enforce this if the user attempts to close the from without a row existing in the referencing table goes in the parent form's Unload event procedure as below:

Private Sub Form_Unload(Cancel As Integer)

    Const conCRITERIA = _
        "NOT EXISTS(" & _
            "SELECT * " & _
             "FROM FamilyMembers " & _
             "WHERE FamilyMembers.FamilyName = Families.FamilyName)"
   
    Dim varFamilyName As Variant
    Dim strMessage As String
   
    varFamilyName = DLookup("FamilyName", "Families", conCRITERIA)
   
    Me.Dirty = False
   
    If Not IsNull(varFamilyName) Then
        Cancel = True
        strMessage = "A row for family " & varFamilyName & _
            " must be entered in the Family Members table."
        MsgBox strMessage, vbExclamation, "Incomplete Data"
   
        With Me.RecordsetClone
            .FindFirst "FamilyName = """ & varFamilyName & """"
            If Not .NoMatch Then
                 Me.Bookmark = .Bookmark
            End If
        End With
    End If

End Sub

The other approach illustrated in my demo is to use an unbound form and insert rows into both the referenced and referencing tables simultaneously in a Transaction.  This makes for rather clumsy interface, however.  The above approach, using a bound form/subform is more intuitive for users.

Unfortunately, due to the nature of bound forms, AND that of relational databases, you have to ALWAYS add the parent record first, and then start adding child records. This is a basic requirement of ALL relational databases.

 

So, you really can‚Äôt skip above.

 

There are two ways over the many years I used to solve this problem.

 

The first approach is a real mess. That approach is to either attempt some un-bound (disconnected recordsets). A real pain.

 

The next possible approach is to pull the data from the back end, and fill some local tables, and edit that way (and thus either commit/update the data, or bail out). But this can be a pain because you have to deal with rows added, deleted, and edited.

 

A lot of systems allow you to ‚Äúedit as a whole‚Äù the master record, and sub recodes, and then have one submit button.

 

However, because access uses a bound forms model, then ANY TIME you break free of this model, then things become rather nasty.

 

So, over the years, the best and most simple solution was a post flag column added to the main record.

 

What I then do is allow the user to freely edit. Perhaps even close the form, go for coffee or do other tasks.

 

However, then what I do is have a post button. And I say the user is entering a batch of dentations, then they enter the user, time of day, the event, and then start entering their work.

 

At the end of the work flow, I then provide a post button. This routine checks all the data, and if criteria is missing, then they are prevented from posting.

 

So, my screen looks like this:

 

 

 

Note the post batch button. It simply does not allow the posting status to become true until such time that the data meets the criteria I want.

 

In above, we have a classic QuickBooks type of distribution for a given check (donation). The amount is entered, and then they can enter several accounts to distribute the amount(s) over several accounts. And of course, the total has to MATCH the total donation amount. If the totals don‚Äôt add up, then the posting routine will inform the user of this issue, and they are not allowed to ‚Äúclose‚Äù or post their run of work.

 

Of course, the post button does not really post anything but simply checks the data, and if good, then the post status is set = True.

 

The OTHER big advantage is once posted, then users are prevented from editing or modifying the data.

 

And note how I even have conditional formatting go ‚Äúgreen‚Äù when your balance matches.

 

So, users are free to enter nice and fast, free to sip on coffee, and this speed up data entry rather fast, and if the phone rings, then they can exit, go enter or look up data, and then go back to their posting event.

 

In effect, what I am suggesting is you allow the data to be entered, and either like above, allow them to post their whole session of work, or perhaps you could provide a post button on the main part of the form. If they hit the post button, then you check the data to be correct, and then set the posting status = true, and you can also then thus prevent further editing based on this status. I even have a special admin password to by-pass this posting flag and allow the accountant to un-post if further editing is required, but only the special user has this ability.

 

Because we can‚Äôt wrap the main one record, and the many child records in a transaction, then the above ‚Äúfake‚Äù of a posting concept works rather well. You then as a developer are free to build up a nice user interface and experience, and the additional step of posting is a final check of their work.

 

Now, your form might not be like the above in which you have a "work run". However, adding some kind of post button often will get around the Access limitation of not being able to edit a master + child form as a "single" object or thing which of course you, and your users are thinking of when they hit "save" or now a "post" button.

Moving the focus to records and updating data in records

When you move the focus to an existing record on a form, enter or change data in the record, and then move the focus to another record, the following sequence of events occurs for the form:

Current (form) BeforeUpdate (form) AfterUpdate (form) Current (form)

When you leave the record whose data has changed, but before you enter the next record, the Exit and LostFocus events occur for the control that has the focus. These events occur after the BeforeUpdate and AfterUpdate events for the form, as follows:

BeforeUpdate (form) AfterUpdate (form) Exit (control) LostFocus (control) RecordExit (form) Current (form)

As you move the focus among the controls on a form, events occur for each control. For example, the following sequences of events occur when you do the following:

  • Open a form and change data in a control:

    Current (form) Enter (control) GotFocus (control) BeforeUpdate (control) AfterUpdate (control)

  • Move the focus to another control:

    Exit (control1) LostFocus (control1) Enter (control2) GotFocus (control2)

  • Move the focus to another record:

    BeforeUpdate (form) AfterUpdate (form) Exit (control2) LostFocus (control2) RecordExit (form) Current (form)

As far as navigating back to the previous record is concerned, the term 'previous' is only accurate here in terms of the user's perception of what happens.  Because the routines in my demo force the deletion of any row in the referenced table which does not have at least one matching row in the refencing table before the form can, from the user's perspective, 'leave' that record, there can only ever be one record to navigate back to.  This is done by applying the following subquery as the criterion  for a DLookup function call:

    Const conCRITERIA = _
        "NOT EXISTS(" & _
            "SELECT * " & _
             "FROM FamilyMembers " & _
             "WHERE FamilyMembers.FamilyName = Families.FamilyName)"

BTW, I should perhaps point out that I've used family names as a key in the demo for simplicity.  In an operational database this would not be done of course, as family names can legitimately be duplicated, so are unsuitable as keys.

If the DLookup function call returns a Null the form is moved to the relevant record with:

        With Me.RecordsetClone
            .FindFirst "FamilyName = """ & varFamilyName & """"
            If Not .NoMatch Then
                 Me.Bookmark = .Bookmark
            End If
        End With

Enforcing a constraint through the interface is not bulletproof of course.  Rows could be inserted into the referenced table by some other means, thus subverting the constraint.  I've not tried it, but I think it would be possible to enforce the constraint completely if (a) the rows are inserted into both tables in a Transaction via an unbound form, and (b) a CHECK CONSTRAINT is applied to the referenced table, Families in my demo.  Conventional wisdom has it that a Transaction must be 'atomic', which means that constraints are not implemented immediately.  Chris Date has lot to say about this, and argues, against the conventional wisdom, maintaining that all constraints should be applied immediately, which would of course rule out the constraint you are trying to apply.  I suspect that Access, and probably all relational database products, adopt the principle of atomic transactions, however.  When I have the time I'll try it with my demo.  I'd have to add a second set of tables for this of course; the CHECK CONSTRAINT would undermine the bound form solution in my demo otherwise.
