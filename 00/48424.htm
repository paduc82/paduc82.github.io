Windows 10 JScript Text File Open/Close -- File not really closed!

This is a flaw in Win-10. Explanation follows.

I write a lot of programs in JScript. (This has nothing to do with web development, I'm using JScript as a scripting language.)

I have a certain function I use a lot, called "log( )". You give it a string, and it copies the string to the command window, and to a log file. It's simple. In outline, it looks like this:

function log( message )
{      WScript.Echo( message );
LL = fso.OpenTextFile( LOGFILE, 8, true); //8=appending, true=create if needed
LL.WriteLine( message );
LL.Close();
}

I left out a few details for clarity, like some error checking, and fso = new ActiveXObject("Scripting.FileSystemObject"); To avoid any chance of collision, this function creates it's own fso, every time you call it.

When I started to use this function on Win-10, it FAILED ALL THE TIME! How can it fail, you ask? It's been working fine on Win-EverythingElse for the last 12+ years.

Well, for starters, I very often use it like this:

log("ANZIncremental  = " + ANZIncremental  );

log("ANZInvariant      = "  + ANZInvariant       );

log("ANZCycleBased   = " + ANZCycleBased   );

log("ANZTraditional    = " + ANZTraditional    );

log("DDGIncremental = " + DDGIncremental  );

log("DDGInvariant      = " + DDGInvariant       );

log("DDGCycleBased  = " + DDGCycleBased   );

log("DDGTraditional   = " + DDGTraditional    );

That is to say, one rapid-fire log( ) statement after another. So it's trying to open, close, and then re-open the same file in rapid succession.

It was failing on LL = fso.OpenTextFile( ). So my hypothesis was, "What if the LL.Close( ) method doesn't really close the file? What if it's still open when you call OpenTextFile( ) the next time?"

Sure enough, I fixed it like this. Again, I've left out some details for clarity.

function log(message)
{    WScript.Echo( message );

     success = false;

     while( !success ){
         try {
     LL = fso.OpenTextFile( LOGFILE, 8, true); //8=appending, true=create if needed

             success = true;
         } //try
         catch(e){
             success = false;   //and do something sensible
             WScript.Sleep(1);
         } //catch(e)

         LL.WriteLine( message );

         LL.Close();
     } //while( !success )
} //function log( )

Via this and various other tests, I was able to convince myself that my hypothesis was correct. LL.Close( ) doesn't really close the file.

I think this is a flaw in the operating system. When you call a close( ) function, the file should be well and truly closed before it gives control back to the caller.

The message sent is not initially being classified as spam. It is only being flagged at the lowest possible level of detection for bulk email (level 1). Since it is a bulk/mass email sent to multiple Skype users in a short period of time, that classification would be rather unavoidable. In order for any email system to perceive bulk email as junk mail, the policies of the client, server or mail detection software would have to be set to automatically reclassify bulk email at a certain level or all bulk email as spam. Once reclassified, it would be placed in the junk mail folder. It is apparent from viewing my own Outlook webmail that any detection of bulk email is what is directing the mail to the junk folder. This might not occur in the actual Outlook desktop client as its default configuration for bulk email to be classified as spam, is not that strict (level 5). 

If a strict spam policy like that is enabled with no option for the end user to change it, the solution would be for more end user control in the mail client. As it stands now, you cannot weaken those default settings in the Outlook web client. You can only whitelist domains you trust. Any other workaround would be external to Skype (even if under the same parent company) and such a request or implementation, may still prove ineffective, at this time. It may prove more effective to click "It's not spam", so that Outlook's email system might learn that the email received was/is legitimate.

tryx3 say:

Steve,

cd\ works in the Powershell console.

If the command Install-WindowsFeature WindowsPowerShellWebAccess works then you can issue it straightaway.  There is no need to fiddle about with choosing a folder.  If the command requires Admin permission then you will have to open an elevated Powershell window as I wrote above.

Running a computer in the Built-In Admin account, as that Instant Windows PowerShell author is doing, would be reckless.  If a hacker managed to access your computer while you were logged on with that account they would have complete access to your system without any Admin prompts being generated for any actions they took. 

Do please note that the Instant Windows PowerShell author is describing procedures for Windows Server not for Windows 10.  The procedure may not work for a Windows 10 computer i.e. a computer that is not a server.
