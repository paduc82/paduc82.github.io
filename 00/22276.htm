TimeElapsed(SUM((TimeEnd-TimeStart)*IIf(Weekday(WorkDate,1)<6,1,0)),"nn") AS TimeWorkedToThursday,

            (SELECT MIN(TimeStart)
             FROM TimeLog AS TL2
             WHERE TL2.EmployeeID = TL1.EmployeeID
             AND WeekStart(1,TL2.WorkDate) = WeekStart(1,TL2.WorkDate)
             AND Weekday(TL2.WorkDate) = 6)
,#08:00#) + (40/24) - SUM((TimeEnd-TimeStart)*IIf(Weekday(WorkDate,1)<6,1,0)),"nn") AS TargetFridayEndTime
FROM TimeLog AS TL1
GROUP BY EmployeeID, WeekStart(1,WorkDate);

ken sheridan say: Most timesheet systems have a particular 'granularity'.  In my organisation it was 5 minutes, yours is 6.  In the demo I used 15 minute slots for simplicity, so as to keep the list relatively short.  Creating a TimeSlots table as the basis for a combo box means you don't then have to do any rounding.

An easy way to fill a TimeSlots table with times at 6 minute intervals would be to serially fill down a column in Excel and import it into an Access table.  Or you could execute a little VBA procedure like this:

Public Sub FillTimeSlots()

    Dim dtmTime As Date
    Dim strSQL As String
    
    For dtmTime = #12:00:00 AM# To #11:54:00 PM# Step #12:06:00 AM#
        strSQL = "INSERT INTO TimeSlots(TimeSlot) " & _
            "VALUES(#" & Format(dtmTime, "hh:nn") & "#)"
            
        CurrentDb.Execute strSQL, dbFailOnError
        
    Next dtmTime
    
End Sub

Don't worry about the rather strange Step value; it's just how Access automatically formats the value #00:06# (6 minutes) in the VBA window.

If you copied my TimeSlots table you might have fallen foul of a ValidationRule on the TimeSlot column in the table design, which limits the times to 15 minute intervals.  You'd need to delete or amend this rule first, or just create a new table to replace mine.

Note that because my demo does not include durations for lunch breaks, but has morning and afternoon start and end times entered separately as two ranges, the query I posted would need amending so that the simple expressions which subtract the start from the end time, would in your case also need to subtract the duration of the lunch break.

ken sheridan say: A subform is not a member of the Forms collection, which only includes open parent forms.  It has to be referenced via the Form property of the subform control in its parent form:

Forms.NameOfParentForm.NameOfSubformControl.Form.NameOfControl

The subform control is the control in the parent form which houses the subform.

Consequently, to reference a control in a subform which is itself a subform of a parent form, the first subform has to be referenced in the above way, and the second subform by way of its Form property:

Forms.NameOfParentForm.NameOfSubformControl.Form.NameOfSubformControl.Form.NameOfControl

While this can be done in a single long expression, I'd be inclined to break it down in a little function in which a series of object variables progressively return a reference to each object in turn, with the return value of the function being the value of the final control.  The following is an example using objects in the nested subforms option in my CorrelatedSubs demo:

Public Function GetProduct() As String

    Dim frm1 As Form
    Dim frm2 As Form
    Dim ctrl As Control
    
    Set frm1 = Forms.frmCompanies_2.frmsubOrders_2.Form
    Set frm2 = frm1.frmsubOrderDetails_2.Form
    Set ctrl = frm2.ProductID
    
    GetProduct = ctrl.Column(2)

End Function

Called in the debug window it returns a value as follows when the form is open at its first record:

? GetProduct()
Northwind Traders Chai

The demo uses Northwind data, and the value is in the third column of the ProductID combo box, referenced as Column(2) because the Column property is zero-based.

john - project say:
Maria,
The most "efficient" way to deal with resource overallocation is to use Project's resource leveling feature. Personally I like the manual approach you are currently using because it give more control, but it can be slow and tedious.

Resource leveling is based on the Priority field. The default priority is 500. Lower priority tells the leveling algorithm that the task can be moved to avoid the overallocation and higher priority tells the algorithm to either not move the task at all (priority 1000) or only move it after the lower priority tasks have been move and overallocation still exists. Leveling can be applied to a select resources or all resources and can be applied to the whole project or limited to a time period. Also be sure to check out the leveling options found under Resource/Level group/Leveling options.

Hope this helps.
