Why can't Excel add up!? - Microsoft Community

peters81 say:

joeu2004 say: Peter wrote:

In a worksheet, if I type 760.0 in a cell, -667.2 in the next, then =a1+b1 in the next, the result -92.8 is correctly displayed.   If I change the formatting to show 30 decimal places, all the additional digits shown are 0.   However, if my first two cells are sums of data in other columns within the workbook, the first coming to 760.0, the second to -667.2, the result of adding the two is 92.800000000000100000000000000000 !

[....]

So any ideas where the extra 0.0000000000001 is coming from?

Probably 760.0 or -667.2 is not "exactly" that when they are the sums of other data.

And you might not have any clue to that fact because Excel formats only up to the first 15 significant digits.

In general, the remedy is to explicitly round arithmetic with non-integers.  For example, =ROUND(SUM(A1:A1000),2) for most dollar-and-cents calculations. [1]

Generally, non-integers are not represented exactly in Excel because Excel relies on the native computer's 64-bit binary floating-point to represent numbers and perform arithmetic.

Typically, that introduces infinitestimal anomalies ("errors") in the result.  These anomalies can add up and become signficant.  Or they can cancel each other out.  It is unpredicable.

For example, the constant 667.2 is really represented internally as 667.200000000000,045474735088646411895751953125.

The constant 760 can be represented exactly.  But =760-667.2 is 92.7999999999999,54525264911353588104248046875.

Note that that is not exactly the same as the constant 92.8, which is represented internally as 92.7999999999999,971578290569595992565155029296875.

Note that in this case, both 92.8 and =760-667.2 would display as 92.8000000000000 when formatted to 13 or more decimal places.  The fact that you see 92.8000000000001 suggests that the internal representation of the sums is not exactly the same as the internal representation of the constants 760.000000000000 and -667.200000000000.

and =MATCH(B1,B2,0) returns #N/A, indicating that they are not truly equal.  Other "inexplicable" inconsistencies also arise.

Excel's "correction" is due to the dubious heuristic poorly described under the misleading title "Example When a Value Reaches Zero" at http://support.microsoft.com/kb/78113.  But as you can see, it is applied inconsistently.  So it does more harm than good, IMHO.


-----
[1] Instead of using ROUND everywhere, you might set the "Precision as displayed" (PAD) calculation option.  I do not recommend that, however, for a number of reasons.
