Private Sub Worksheet_Change(ByVal Target As Range)

Hi, I'm an independent advisor and will try to help.

Your question is not clear. Are you trying to import data stored in Outlook into an Access database or are you trying to send Access data through e-mail and then import it. The two are different processes. If you are trying to send Access data attached to an e-mail, then you have to either rename the extension or Zip the Access file. Outlook will not let you download an accdb file because it can contain malicious code. So you can make a copy and rename the copy as accdx (or whatever) and attach that. Or you can ZIP the file (using a compression utility or Windows built in compression folders) then attach that file.

Once you download the file from the e-mail, you will then need to rename the file back to accdb or unzip the compressed file. Once that is done, you can then Import the data using the External Data ribbon in Access.

Juan:

Those are not "line breaks".  You are replacing with the "New Paragraph" command (^p).  You need ^l to give you a line-break.

The difference is that a line-break will continue the paragraph with the same formatting as the previous line, which the new paragraph will have different formatting.

The paragraph mark is a container that contains all of the formatting for the preceding text: about 1,200 separate properties.

simon__simon say:

Excel for Mac crash on file closure/High Sierra

jim gordon mvp say:

I assume you took care of updating. Go to the Excel menu and choose About Excel. Make sure you have version 14.7.7 (the final update) before doing anything else.

The next step in troubleshooting is to start your Mac in Safe Mode.

Perform a Safe Boot. It will take a long time for your Mac to start up in Safe Mode. When you do a Safe Boot Mac OS fixes troubles with your Mac's directory and file system. The font cache is emptied. Only a limited number of fonts are enabled. Applications that run at startup don't run automatically. This allows you to test your app(s) by running them with a minimal set of fonts, an empty font cache, and without potentially conflicting apps running.

 

Once you reach the desktop in Safe Mode try running the app having a problem. Is the problem still there?

If the crash is gone: Performing a Safe Boot may have fixed the ailment. Or perhaps one of the fonts that wasn't loaded is bad. Or perhaps some other software utility is interfering. After testing in Safe Mode, do regular restart.

 

If the crash is still evident in Safe mode, you may still have a bad font. Use Font Book to restore system fonts, then test again in Safe Mode.

 

If the crash was gone in Safe Mode but returns during a regular boot: I think the first thing to do would be to use a font utility to validate ALL your fonts. Mac OS comes with a free font utility called Font Book, which you can use for this purpose. Get rid of any font that doesn't pass as clean and green. Then, restart your Mac and immediately empty the trash. If you've gotten this far and still have a crash, then there are more things to explore. But if you get this far you've probably gotten rid of the crash. Either way, please reply with your results.
