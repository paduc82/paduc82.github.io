This happens for other numbers but not all (eg. std dev is non-zero if all numbers are 0.2 or all are 0.8 but the standard deviation calculates correctly, i.e, = 0, if all numbers are 0.5 or 0.6). Is there a fix for this? I use this spreadsheet for subsequent calculations and need the std dev to be correct.
mike h.. say:

Hi,

 

This is a to do with the way excel handles floating point numbers. Excel works in binary and often there is no precise binary equivalent of a decimal number so these rounding errors creep in. This isn't exclusive to Excel which in common with most other packages uses the IEEE754 (See link below) standard for handling floating point numbers.

 

Have a look at the link and in particular the section on rounding and for your STDEV you can use

 

=round(stdev(a1:a3),n)

 

where n is the number of decimal places to round to

 

http://support.microsoft.com/kb/78113

Sub OneLine()
    Dim vSrc As Variant
    Dim vRes() As Variant
    Dim rDest As Range
    Dim i As Long, j As Long, k As Long, L As Long
    Dim m As Long, N As Long
    Dim v As Variant
    Dim s As String

Set rDest = Worksheets("Sheet2").Range("A1")

'Clean up data by removing any leading or trailing vbLF's & spaces

        Do While Right(vSrc(i, j), 1) = vbLf
            vSrc(i, j) = Left(vSrc(i, j), Len(vSrc(i, j)) - 1)

                Case Else
                    vRes(L, m) = v(N)
            End Select
        Next m
        N = N + 1
    Next L
    k = L
    N = 0
Next i

john w. vinson say:

That sounds quite a bit more hopeful. I'd suggest creating an empty table with JUST the fields that you need for the analysis (including a Date/Time field). Then use External Data... Link to connect to one of the csv files.

Construct an Append query selecting only the fields that you need. For the existing text date/time field, you should be able to use an expression like CDate(Left([textdate], 12)) or however long the actual date/time value is; or CDate(Replace(Replace([textdate], "EDT", ""),"EST", "") to expunge the timezone. Get this query working with one of the csv files, then empty the target table, compact the database, and run the append query on each of the csv files, monitoring the size of the database as you go. This should give you a much smaller footprint, and by massaging the data as it's imported you should get much less bloating. All of the analytics you describe can be very readily generated by Totals or Crosstab queries. Dunno about Pivots, I believe that they rely upon Excel and the size of the dataset may interfere, but once you have the data in an Access table it would certainly be worth a try.

Ideally you would probably want to do a normalized database with a ProductType table, People table (or tables) for the by who and buyers, and numeric keys in the master data table, and migrate the data into it; but if this is a one-off task it might or might not be worth the effort. On the other hand, 5 million records with a 4-byte foreign key in three fields is going to be a whale of a lot smaller than 5 million records with 20 or 30 characters in the "same" three fields!
