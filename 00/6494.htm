john w. vinson say:

A Primary Key does not need to be an autonumber (though Microsoft may give you that impression). Is there a field in the Excel data which is a "Candidate Key" - i.e. unique, no repeats; never NULL; and (preferably) reasonably short and stable? Is there a field which can be used to reliably and uniquely link that row in the spreadsheet to a single record in the existing Access table?

Do note that if (as may well be the case!) you want a new Autonumber primary key for the data currently in Excel, you can create the table, empty, with an Autonumber as the primary key, and then link to (not import!) the existing spreadsheet and run an Append query to populate all the fields except the Autonumber. The Autonumber will increment and provide a unique identifier for the data.

I noticed your criteria in the query,a nd I think more accurately it should be:
forms!orderentry!orderdraperylineitems.Form!UOM
I think of .Form as stepping into the subform.
That's probably not the root cause of your issue.

With regards to the Draw value disappearing in line 1: the dropdown should have all possible values in it, or this may happen.
The classic problem is with cascading dropdowns, where one may innocently set the RowSource to "select [fields] from [table] where [FK] = Forms!myForm!myPreviousField
There are several ways to solve this - I have to dig up all the details, but one way (not necessarily the best) is to set the RowSource to a 3-part union query. In pseudo-code:
select [fields] from [table] where [FK] = Forms!myForm!myPreviousField
union all
[separator record]
union all
select [fields] from [table] where [FK] <> Forms!myForm!myPreviousField

This combined with code to ensure users don't select from the second and third set of rows.

Another way is to have a textbox on top of the dropdown. Again, if more details are needed I can dig them up. As Gina is pointing out, there are important differences between datasheet (more limited) and continuous forms (more options available).

A(nother) Method for Running Beta OSs - Microsoft Community

drew1903 say:

A(nother) Method for Running Beta OSs

It has been repeatedly stated to run a Beta OS, such as Windows 10 currently in the following ways:
VM
VHD
Spare Box
Dual-boot

It requires a physically separate drive be available but, this is, also, a method...

 Example PC:
 Running Windows 8.1
 Unplug, disconnect the drive holding Windows 8.1
 Install Windows 10 to the other drive.
 Once done reconnect all drives.

 You shut down & F12 to pick the drive you will boot, either 8.1 or 10.

This is not the same as dual-boot. They are distinct.  They are each booting themselves separately; not sharing a boot ini.

Easy, safe and creates a very 'normal' set-up.  Much better than a VM & even better than dual-boot.  You can 'see' all drives from both OSs this way and all USBs or peripheral gear from both.

Cheers,
Drew

eric ible say:

I also prefer "distint" installs that don't share boot or folders. Never been a fan of dual boot so never got good with it.

If running MBR partition system and not secure EFI boot you can use a tool/utility called PTEdit32 to change partition IDs and boot status without affecting content.

So, if you had two partitions:

08 07 (Boot and fs) Windows 8.1 or what have you

00 07 (empty partition of NTFS fs)

Use any partition manager or PTEdit32 to alter:

00 12 (No longer Active)

08 07 (a place to install 10 without affecting the installed OS above)

This method doen't use BIOS boot options only boot active status of partition.

You could use this on dual drive systems too.
