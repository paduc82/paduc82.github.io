scottgem say:
First, you are using a denormalized structure. Your table is like a spreadsheet not a database. You should NOT have fields for each question.

A survey database requires a specific structure. Something like this:

tblQuestion
QuestionID (Primary Key Autonumber)
Question

tblRespondent
RespondentID (PK Autonumber)
FirstName
Lastname
Other respondent info

tblResponse
ResponseID (PK Autonumber)
RespondentID (Foreign Key)
ResponseDate

tblAnswer
AnswerID (PK Autonumber)
ResponseID (FK)
QuestionID (FK)
Answer

You enter the data by first entering your 30 questions into tblQuestion. Next you enter your 8000 respondents into tblRespondent.

Finally you have a main/sub form combo to enter the answers. The main form is bound to tblResponse and the subform to tblAnswer and linked on ResponseID. In the main form, you select the Respondent from a combobox and enter the date. In the subform you select the questions they answered from a combo and enter the Answer. 

If you already have a table that lists what questions the respondent saw, then you can use that to constrain the Question combo on the subform.

scottgem say:
Whenever designing a relational database, you should think vertically, not horizontally. So yes, tblQuestion is just a table of all your questions and an unique identifier. In tblAnswer each survey response is identified by the respondent and the question. So if John Doe was asked how much on a scale of 1-10 he likes chocolate, and he answers 8, then you have a record that identifies the respondent (John Doe), the question (How much do you like chocolate) and the answer (8).

While this doesn't actually make data entry that much easier (though in your case, since each respondent may have a different set of questions, it may), it WILL make tabulation and analysis of your data, way easier.

I have a spreadsheet with many rows and columns of data in it.  Let's say the data under column D contains 5 different animal types: "Cat", "Dog", "Goat", "Cow" or "Horse".  I want to create a macro that looks at the 5 different types and then creates 2 new tabs: "House" and "Barn", and then copies all rows that contain "Cat" and "Dog" in column D into the newly created "House" tab, and copies all the rows that contain "Goat", "Cow" and "Horse" data into the newly created "Barn" tab.

I found the macro below searching the web, and it's the closest thing I've found to what I want, but it creates one new tab for each type of animal and populates that tab with the rows that have that animal in column D.  Can anyone modify this to separate them as described above (or if that's not practical, give me a different type of macro--I'm not married to this one...)

Sub CreateSheetonColumn()

Dim lastrow As Long, LastCol As Integer, i As Long, iStart As Long, iEnd As Long

Dim ws As Worksheet

Application.ScreenUpdating = False

With ActiveSheet

    lastrow = .Cells(Rows.Count, "A").End(xlUp).Row

    LastCol = .Cells(1, Columns.Count).End(xlToLeft).

    .Range(.Cells(2, 1), Cells(lastrow, LastCol)).Sort Key1:=Range("D2"), Order1:=xlAscending, _

        Header:=xlGuess, OrderCustom:=1, MatchCase:=False, Orientation:=xlTopToBottom

    iStart = 2

    For i = 2 To lastrow

        If .Range("D" & i).Value <> .Range("D" & i + 1).Value Then

            iEnd = i

            Sheets.Add after:=Sheets(Sheets.Count)

            Set ws = ActiveSheet

            On Error Resume Next

            ws.Name = .Range("D" & iStart).Value

            On Error GoTo 0

            ws.Range(Cells(1, 1), Cells(1, LastCol)).Value = .Range(.Cells(1, 1), .Cells(1, LastCol)).Value

            .Range(.Cells(iStart, 1), .Cells(iEnd, LastCol)).Copy Destination:=ws.Range("A2")

            iStart = iEnd + 1

        End If

    Next i

End With

Application.CutCopyMode = False

Application.ScreenUpdating = True

End Sub

john w. vinson say:

No VB code is needed at all (though that's certainly one option). You can instead use a table driven solution. Create a little table Grades with a number field Score and a text field Grade with exactly the values you show above (except with 0 for U):

Join this to the table containing Mark 1 with a "non equi join":

SELECT <whatever fields from your table>, Grades.Grade
FROM yourtable INNER JOIN Grades
ON yourtable.[Mark 1] >= Grades.Score
WHERE Grades.Score < (SELECT Max(X.Score) FROM Grades AS X WHERE X.Score > yourtable.[Mark 1]);

This won't be updateable; if that's needed, this is less efficient but will allow updating:

SELECT <whatever fields from your table>, DLookUp("[Grade]", "[Grades]", "[Score] = DMin("[Score[", "[Grades]", "[Score] > " & [Mark 1])) AS Grade FROM yourtable;

scottgem say:
First, I agree with John and Van to use a table based solution. But my reason for responding is I'm bothered by the names of your fields. Having fields like Subject1 and Mark1 imply you have other fields like Subject2 and Mark2. This is denormalized design. You should have a separate table something like this:

tblMarks
MarkID (Primary Key Autonumber)
StudentID (Foreign Key)
SubjectID (FK)
SessionID (FK)
Mark

The StudentID identifies which student the Mark is for. The SubjectID identifies what class (subject). The SessionID identifies when the class was taken (either date or semester etc.) And the Mark is the numercial mark. You then use the join as John suggests to supply the Letter grade.
