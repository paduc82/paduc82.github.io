Transparency and background color in VBA and ActiveX controls in Word - Microsoft Community

johnd_100 say:

Transparency and background color in VBA and ActiveX controls in Word forms

There have been several discussions over the years on this forum (and the Office support forum) about problems with setting the transparency property of ActiveX controls.  Controls such as radio buttons can be set to be 'transparent' using the fmBackStyle property, which allows a background color to show through.  A common issue is that this seems to work ok while in Design mode, but when this mode is turned off and the form is protected, the transparency is lost and the controls revert to their native (usually white) color, thus messing up the appearance of the form.

I have encountered this problem when creating user forms in Word 2010, but the same issues have been reported with later versions of Word, Excel and other Office applications. This problem was most recently reported with Office 2016 and 365, but was apparently fixed (see https://support.office.com/en-us/article/VBA-forms-are-not-working-as-expected-e14ada52-b1de-4345-80ca-f8601ccde3ce ).  However, I've so far not found any fixes for this problem for earlier versions of Word.

Instead the workarounds that have been suggested require the control's fmBackStyle property to be set to Opaque and it's BackColor property set to the same color as the form background.  Unfortunately the BackColor property palette is very limited, which then restricts the range of colors that can be used for the form background.  I experimented with trying to change the Hex Code of the control's BackColor to see if this would display the control in other colors apart from the limited palette offered in the Properties panel.  This worked up to a point - I could generate a wide range of different colors, but they didn't correspond to the RGB equivalent values!

After much head-scratching and trial-and-error I was on the point of giving up this approach when I came across a posting on this forum which suggested that Word used Hex Codes corresponding to BGR values rather than RGB.  A reply (from Doug Robins, an MVP) disputed this (see https://answers.microsoft.com/en-us/msoffice/forum/msoffice_word-mso_other-mso_2007/background-color-with-radio-buttons/55b706ee-b18c-49ed-87b7-9f3b5d7a4313?messageId=2cf2ff62-e0a5-4056-8c77-5cf68d869bf0 ).  He posted a screenshot showing the color-picker control (for a text font color) displaying RGB values as expected.

But I thought I'd give it a try anyway.  I was trying to match a form background color with values R=214, G=227, B=188.  The Hex equivalent for this is D6E3BC.  But putting this Hex value into the control BackColor property (in the format &H00D6E3BC&) produced a very different color from the one shown in the form background.  So I switched round the R & B values, and entered &H00BCE3D6& as the BackColor value.  It worked!  The control color now matched the background color exactly.  So it seems that whoever in Microsoft coded the ActiveX form controls decided for some obscure reason to use BGR values, while the main Word application does indeed use RGB values.  Thus confusion reigns!

Apologies for the length of this post but I hope it helps anyone who is still struggling to work around the problems of 'disappearing' transparency of ActiveX controls in earlier versions of Office, as I guess Microsoft is unlikely to fix it.  I would have posted this message on the above thread but it's now locked out.

  • If the user is not online in Yammer, the notification is sent immediately. By not online we mean he is not logged in to Yammer from any device.

  

Microsoft is making some changes to how notifications work in Yammer, here is a summary.

 

  • Logic has been added that holds email (15 minutes) and push notifications (2 minutes) while you are active on the Yammer web site or in the mobile app
  • After that delay Yammer will check that a message thread is still unread before sending an email or push notification to avoid notifying you about things you've already seen on the web or in the mobile app
  • They are trying to make it easier to triage your Yammer emails by providing very explicit explanations for why you're getting the emails
  • They are also making some changes to the format of email notifications, Tuck has provided Microsoft with feedback on what we did not like about examples we have seen

 

They have developed new business logic for what triggers email notifications and will send notification emails for the following scenarios:

 

  • New announcements
  • When someone mentions you on Yammer
  • When someone is added to a direct message
  • When a non-group participant is added to a private group conversation
  • When a message thread conversation in your Yammer Inbox goes from read to unread
  • When someone replies directly to your message
  • When someone sends you a new direct message

 

With the exception of the above scenarios, Yammer will stop sending email notifications if you haven't read a conversation thread since receiving a previous notification for the same conversation. In other words, lack of engagement will be seen as lack of interest.

john korchok say:

Your old file has been over-written by the new one.

To prevent this in the future, please enable the macOS Time Machine utility. This will save multiple versions of your files automatically as you work.

Personally, I don't use Time Machine, but I frequently Save As and add an incrementing number to the file name, so I can easily revert to previous versions. A good habit I learned in the dark ages of computing.
