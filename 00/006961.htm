Greek letters with breathings (Greek Extended Unicode region - Microsoft Community

hermolaou say:

Greek letters with breathings (Greek Extended Unicode region 1F00-1FFF) don't get collated

Hello!

I have a table with Greek words that contain letters with breathing and accents, so the words contain characters that belong to Unicode 'Greek and Coptic' (0300-0370) and 'Greek Extended' (1F00-1FFF) ranges.

The problem is that in all comparisons the letters of that higher range (Greek Extended) are not counted at all, as if they are not present. Letters with breathing marks (ἀ, ἐ, ὑ, ὦ etc.) fall into that region.

For instance, quering for ἄρχομαι (select * from grc where lemma='ἄρχομαι') returns both ἄρχομαι and ἔρχομαι, as well as ρχομαι.

Suppose the table has αὐγής but doesn't have ἀαγής (notice the breathings). Quering for ἀαγής returns the αὐγής as the result, because the letters ἀαγής returns the αὐγής as the result, because the letters ὐ and ἀ are not collated, it's as if the engine was looking for 'αγής'.

But this can be avoided with strcomp function, which makes the comparison exact. The problem is that when there's a unique index on that column you can't insert many words that are counted as duplicate because Greek Extended characters are ignored. There are other cases as well where correct collation is desired.

To reproduce the problem, have a unique index on a Unicode column and see if you can insert both ἄρχομαι and ἔρχομαι, or both αὐγής and ἀαγής into it.

I'm using not Access but Access Runtime 2016 ODBC driver and operate it with VBScript, the database source is a simple mdb file.

I wonder what's the reason for this behavior (ignoring some Unicode characters on collation)? Was it meant to work that way? Is there some way to get control over how Access Runtime over ODBC collates Unicode?

People are saying (http://www.utteraccess.com/forum/Collate-Access-t1940463.html) that it is possible to set collation per database. But how can it be done?

Thank you!

tom van stiphout say:
I am not an expert on this, but I do know that if you had a binary collation, those problems would go away. Each character would stand on its own as its Unicode value. Maybe that would work better for you, or maybe it would introduce other problems. That would be for you to decide. SQL Server supports binary collation; Access (better: ACE) does not.

ken sheridan say: I'm not quite clear what you are attempting here.  On the one hand you have a button to insert a new row, presumably into the table on which the form is based.  On the other hand you are entering a new value into the combo box, which would be done to insert a new row into the referenced table from which the combo box draws its list.  The two can be one and the same table, however, and the example from my FindRecord demo which I give below is a case in point.  Alternatively they can be separate tables, in which case the examples in my NotInList demo are more appropriate.

Firstly, to insert a new row into the table on which the form is based you can either use a button, in which case the code in its Click event procedure would usually just navigate to a new empty record in the form with:

    DoCmd.GoToRecord acForm, Me.Name, acNewRec

Another approach would be to dispense with the button and include a <New Record> item in the combo box's list.  You'll find an example of this in FindRecord.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

In this little demo file the 'Open form to go to a record' button opens a form which includes an unbound combo box to either got to a selected contact or add a new contact record.  It does this by setting up the combo box's RowSource property as follows:

SELECT ContactID, FirstName & " " & LastName, 1 As SortColumn,
LastName,Firstname  
FROM Contacts  
UNION
SELECT 0, "<New Contact>", 0,"",""  
FROM Contacts
ORDER BY SortColumn, LastName, FirstName;

You'll see that the hidden first column has a value of zero for the <New Contact> row.  In the control's AfterUpdate event procedure the code moves the form to an empty new record if the value is zero, otherwise it navigates to the selected contact.  The code is:

Private Sub cboGotoContact_AfterUpdate()

    Const MESSAGETEXT = "No matching record"
    Dim ctrl As Control
    
    Set ctrl = Me.ActiveControl
    
    If Not IsNull(ctrl) Then
        If ctrl = 0 Then
            ' go to new record and move focus to FirstName control
            DoCmd.GoToRecord acForm, Me.Name, acNewRec
            Me.FirstName.SetFocus
        Else
            With Me.RecordsetClone
                .FindFirst "ContactID = " & ctrl
                If Not .NoMatch Then
                    ' go to record by synchronizing bookmarks
                    Me.Bookmark = .Bookmark
                Else
                    MsgBox MESSAGETEXT, vbInformation, "Warning"
                End If
            End With
        End If
    End If
    
End Sub

The combo box also includes code in its NotInList event procedure which allows the user to type in a new contact name to insert a new record.  The code parses the name into FirstName and LastName  string expressions and then inserts these into the relevant controls after moving the form to a new record:

Private Sub cboGotoContact_NotInList(NewData As String, Response As Integer)

    Dim ctrl As Control
    Dim strMessage As String
    Dim strFirstName As String, strLastName As String
    
    Set ctrl = Me.ActiveControl
    strMessage = "Add " & NewData & " to list?"
    
    strFirstName = Left(NewData, InStr(NewData, " ") - 1)
    strLastName = Mid(NewData, InStr(NewData, " ") + 1)
    
    'if user confirms insert move form to new record
    ' and insert values into first and last name controls
    If MsgBox(strMessage, vbYesNo + vbQuestion) = vbYes Then
        Response = acDataErrContinue
        ctrl.Undo
        DoCmd.GoToRecord acForm, Me.Name, acNewRec
        Me.FirstName = strFirstName
        Me.LastName = strLastName
    Else
        Response = acDataErrContinue
        ctrl.Undo
    End If
    
End Sub

This last code is only for demonstration purposes, however, as in real life it would be inappropriate, as personal names can legitimately be duplicated.  In cases like this a button is the true solution, and you'll find this used in another demo file in the same OneDrive folder, NotInList.zip.  The code in this case is:

Private Sub cmdNewContact_Click()

    DoCmd.OpenForm "frmContacts", _
        DataMode:=acFormAdd, _
        Windowmode:=acDialog, _
        OpenArgs:="New Contact"
    
End Sub

This form is an unbound dialogue form, so the code opens a separate bound form in dialogue mode.  Where the combo box is in a bound form, as in my FindRecord demo, the button would just need the one line of code as I said earlier:

    DoCmd.GoToRecord acForm, Me.Name, acNewRec

You'll find other examples of the use of the NotInList property in the same demo, both to insert a new row into the referenced table transparently by the code in the NotInList event procedure, and where the code in the NotInList event procedure opens a separate form and passes the new value to it via the OpenArgs mechanism, allowing the user to then enter further data in the form.  Note that in each case the visible column of the combo box is not the BoundColumn, which is a hidden column of distinct numeric keys values.  The first column is hidden by setting the ColumnWidths property of the control to zero.
