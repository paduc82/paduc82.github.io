albert d. kallal say:

OnStart must be a standard VBA routine in a standard code module.

So you not running a macro anymore, but are free to call standard VBA code.

eg:

accessApp.Run "MyUpdateData"

accessApp.Run "ExportReportData"

accessApp.Quit

I should point out that if any startup prompts exist, or any kind of access prompts appear, then you in big trouble since such forms and code can interfere with the above script code. So make sure no startup forms, or autoexec macros run that could/can cause problems.

And keep in mind that you calling VBA Subs (not functions) that must be declared as public Subs, and must be placed in standard VBA code modules (not forms modules, and not class modules).

So you are free to call multiple VBA routines that do what you please. However, you also want to be sure that the startup code, startup macros, and startup forms don't cause any "user" prompts, else the above will fail.

I often create a small database with linked tables and JUST the code routines I need, and specify that in above in place of some complex access application with tons of startup code etc. that for "batch" processing as above is not recommend, nor required.

So declare you VBA routines as a public Sub - not a function.
