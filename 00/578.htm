It is indeed not the best formulation that is used.

Changing any cell will trigger the Worksheet_Change event. It is the code within the event that determines if something should be done or not.

Set the cursor before Set Keycells and hit F9, that line has no a red background (breakpoint in the code).

Change a cell outside the KeyCells range and you will see that the code stops as the breakpoint. Hit F5 to continue.

Do the same within the KeyCells range. The msgbox will now pop-up because the condition is true.

Note, the Worksheet_Change event is bound to the sheet were the code is placed. Changing a cell in another sheet will not trigger this event.

A general change event that works on all sheets is Workbook_SheetChange in ThisWorkbook.

ken sheridan say: To compute the difference in days between a date in a row and in the same column in the previous row in date order you'd use a query which joins two instances of the table.  The following is an example using a Transactions table:

SELECT T1.TransactionID, T1.TransactionDate,
T1.TransactionDate - MAX(T2.TransactionDate) AS DateDifference
FROM Transactions AS T1 LEFT JOIN Transactions AS T2
ON (T2.TransactionID<T1.TransactionID OR T2.TransactionDate<>T1.TransactionDate)
 AND (T2.TransactionDate<=T1.TransactionDate)
GROUP BY T1.TransactionDate, T1.TransactionID;

Note how the primary key Transaction is brought into play as the tie breaker in the event of two or more transactions taking place on the same day.  Where this is the case the date difference for the second and subsequent rows of the same date would be zero.

BTW do not use Date as column name.  As the name of a built in function it is a reserved keyword and should not be used as an object name.

ken sheridan say: There is only one Transactions table.  What the query does is join two instances of that single table, giving them aliases of T1 and T2 respectively.  There is no way you can represent my query in design view however, because the join expression is too complex to be represented in design view. Let's look at it in more detail:

(T2.TransactionID<T1.TransactionID OR T2.TransactionDate<>T1.TransactionDate)
 AND (T2.TransactionDate<=T1.TransactionDate)

A join expression at the end of the day is no more than a Boolean expression which returns TRUE or FALSE, returning a row where the expression returns TRUE for that row.  

The first line of the above requires that the TransactionID value in the second instance of the table is less than the TransactionID value in the first instance OR that the TransactionDate values in both instances are not the same, i.e. where there are two or more rows with the same date they are only joined where the TransactionID value is lower in the first instance.  This is how the TransactionID value is used as the tie breaker when two or more transactions occur on the same day.

The second line requires that the TransactionDate value in the second instance is before or the same as that in the  first instance.

The two rows are tacked together with the Boolean AND operator.  This requires that both lines must evaluate to TRUE for the rows to be joined.

The number of days between transactions is simply the TransactionDate in the second instance subtracted for the latest (MAX) date in the first instance which satisfies the above join criteria.

With my table the following is a subset of the rows returned by the query:

TransactionID    TransactionDate    DateDifference
26                       01/01/2009    
2                         05/01/2009             4
1                         06/01/2009             1
27                       01/02/2009           26
3                         05/02/2009             4
4                         05/03/2009           28
6                         05/04/2009           31
19                       05/04/2009             0
8                        10/04/2009              5

The dates are in UK short date format of dd/mm/yyyy.  Note how for the second transaction on 5 April 2009 the difference is zero, because the dates in the two rows are the same.  The primary key has broken the tie, and arbitrarily put TransactionID 6 first because it is smaller that TransactionID 19.

Provided that your Transactions table has an autonumber TransactionID as the primary key, and a TransactionDate column of DateTime data type, you should be able to paste my query's SQL statement into the query designer in SQL view, and open it without any further amendment.  If you do want to show other columns in the result table, e.g. a TransactionAmount column, you can simply add these to the SELECT clause, qualifying each with T1.
