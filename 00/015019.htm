    suzanne s. barnhill say:

    Although I confess I don't really understand how it works in practical terms, Word 2010 does allow you to define a "Body" and a "Headings" font for each document/template, and these selections can be saved as a font "theme," but you may be wanting to apply the fonts differently (for example, the "body" font for some headings"), and adding a third one does complicate matters.

    I agree with Shauna that modifying built-in fonts is generally the best idea, especially in the case of the built-in heading fonts, but I also like your idea of creating several "base" styles with the requisite font. To combine those ideas, I'd say use all the built-in heading styles with these changes. By default, Heading 1 is based on Normal, with the Headings font applied. You could leave this as is or base Heading 1 on one of your custom styles with the desired font or base it on "(no style)" and define the font explicitly. If Heading 2 has a different font from Heading 1, then, instead of using Heading 1 as the base font (which is the default), you could base it on your second custom style or, again, on "(no style)" with the font specified. The remaining headings could be based on Heading 1 or Heading 2 or (if you're using your third font at some level) some other heading.

    You could follow the same principles with body text styles. One caveat: It is a general rule/convention of typesetting that the best design uses only one serif and one sans serif font (in various sizes, varying them with bold/italic, etc.). Obviously, there are exceptions: you may want to add a script font to represent handwritten text or a monospaced font such as Courier New for code samples or machine commands. But if your font usage is similar to those examples, it might be better to use just the Body/Headings dichotomy and then define specific styles using the third font as exceptions. That's not to say that you couldn't use them as base styles: for example, if you had a Code Sample style, you could also have a Code Sample Indent style based on it, and so on (by the way, there is actually a Plain Text style that starts out in a monospaced font, not to mention a slew of HTML styles).

    suzanne s. barnhill say:

    This is an interesting experiment. I would expect that a character style could be used as the base style only for another character style and the same for paragraph styles. I avoid using linked styles entirely.

    It is a documented fact that changing the "based on" style affects only those properties that have not been specified in the given style. If the font of a style has been changed from that of the "based on" style, then changing the "based on" style will not affect the font formatting.

    As for the Body and Headings fonts, those are set in the font theme. On the Home tab, in the Styles group, if you click Change Styles, you will get a menu that includes Fonts, which opens a gallery showing the available font themes, each with its Headings and Body font (Cambria and Calibri for the default Office theme, for example). You can use Create New Theme Fonts... to create a new font theme that will be available to your documents (I have one called Prairie View that was created for a client); it will be at the top of the gallery.

    This whole scheme is pretty complicated, but I know that if I select the Word 2003 style set (Home | Styles | Change Styles | Style Set), the paragraph formatting changes, but the fonts don't change unless I also choose the Office Classic font theme.

    Your analysis is correct.

    When an attribute link is broken between a child style and the parent style upon which it is based, as in different fonts, changing the attribute in either (i.e. changing them both to having the same font) the link for that attribute is restored. Changing the attribute in the base syle after the link is restored will carry into the child style.
