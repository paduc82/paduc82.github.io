alkal---01 say: Ok, I just managed to correct it. Apparently in my case it was a problem of the Native Address Translation, a protocol that allows
many local clients to connect behind a single access point, using only a single IP address, which can create problems for a number of applications among which voip (more on NAT at: http://www.microsoft.com/technet/prodtechnol/winxppro/support/upnp01.mspx)

Anyway what I did to solve the problem was to setup my wifi router to directly detect and manage UDP connections over the 5060 port (this is what my voip provider uses, and I think most of the voip providers) through the port triggering interface of the device. The settings I put were
Local Protocol : UDP
Local Port : 5060
Public Protocol : UDP
Public Port : 5060

Quoting the manual on port trigerring:

"Port Triggering

Special applications on clients in your local network can also use the Internet connection on your device with the assistance of Port Triggering. These applications would normally otherwise require direct access to the Internet. Triggered by data that a client sends to the Internet, certain incoming traffic from the Internet can also be permitted over the Internet connection on your device and forwarded automatically to the correct client. Information on which protocols and ports can be used by your application for transferring data can be found in the appropriate documentation."
