We have added the following tags to the Default MRM Policy:
- Archive after 3 months (Type: Default, applies to entire mailbox)
- Calendar never archive (Type: Calendar)
- Tasks never archive (Type: Tasks)
- Notes never archive (Type: Notes)

Our result is that the Archive after 3 months tag appears to apply but the specific "never archive" do not apply. Is this because the "Archive after 3 months" tag is taking precedence over all other types of retention tags in the policy?

alan wu msft say:

Hi Jono_365,

If you want the entire mailbox to be "3 months to Archive", then the retention policy tag should be a "Defaul Retention Policy Tag" (DPT) but not a "RPT". Here, please be noted that a "RPT" is used for the mailbox's default folder, ie, the "Inbox", "Sent Items", "Calendar", etc. 

On another hand, to isolate it from the original MRM (default MRM) retention policy, I suggest you create a new rentention policy which only includes 1 DPT (3 months move to archive) and 3 RPTs (never move to Archive) and then replace the default MRM of the mailbox with the newly created retention policy (1 DPT and 3 RPTs) to see if it works. 

You may also refer to https://technet.microsoft.com/en-us/library/aa998864%28v=exchg.160%29.aspx?f=255&MSPPError=-2147217396 to start the MRM process.

Alan

scottgem say:

Can you explain more about your app and the form?

It is not a good idea to use a multi-table query as the Recordsource for a form. If you want to enter/edit multiple tables on one form, use a mainform/subform. 

So your Recipt table would be a subform on your Customer form. Also using text fields is not a good idea for Primary/Foreign Keys. I would have an Autonumber as the SubdivisionID. Another issue I see is can't their be more than 1 customer in a Subdivision? If so SubdivisionID should be a FK in the Customer table (unless customers can have multiple subdivisions). 

Finally, your Receipts table should be further broken down. The Fees should be entered in a child table like this:

ReceiptFees

ReceiptFeeID (PK Autonumber)

ReceiptID (FK)

FeeTypeID (FK)

FeeAmount

FeeTypeID would come from a lookup table. 

dirk goldgar say:

It is not a good idea to use a multi-table query as the Recordsource for a form. If you want to enter/edit multiple tables on one form, use a mainform/subform.  

I differ with Scott on this issue -- many times it's perfectly fine to use a multi-table query as a recordsource.  However, you do have to do it right, because it can introduce presnickety complications.
