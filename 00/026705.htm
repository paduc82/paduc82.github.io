Try this:

 

Sub Alert()
    Dim rng1 As Range
    Dim rng2 As Range
    Dim rng3 As Range
    Set rng1 = Range("D11:M151")
    On Error Resume Next
    Set rng2 = rng1.SpecialCells(xlCellTypeFormulas, xlErrors)
    Set rng3 = rng1.SpecialCells(xlCellTypeConstants, xlErrors)
    If Not rng2 Is Nothing Then
        If Not rng3 Is Nothing Then
            Set rng2 = Union(rng2, rng3)
        End If
    ElseIf Not rng3 Is Nothing Then
        Set rng2 = rng3
    Else
        Exit Sub
    End If
    If rng1.Count = rng2.Count Then
        MsgBox "Output not handled!", vbInformation
    End If
End Sub

HI,

First thing you really need to do is store the path as a text field rather than use the 'attachment' data type.

If you have a copy of the photo's in a folder, and you do have the file name in the table (as shown), you should be able to append the path to the file name and remove the attachment i.e. store the path/filename instead.

If you only have the photo's as the attachment then you will need to save the photo's to a folder first and, as above, append the path to the file name.

Saving each attachment one at a time would be time consuming considering you have 'hundreds'. the following code will get you started to export all photo's to a folder.

To display the photo/image in the report you can use an image control with the control source set to the appropriate field name.

Hope this helps?

Public Function SaveAttachments(Optional strPattern As String = "*.*") As Long

        Dim dbs As DAO.database

        Dim rst As DAO.Recordset2

        Dim rsA As DAO.Recordset2

        Dim fld As DAO.Field2

        Dim strFullPath As String

        Dim strpath As String

        strpath = "C:\My Data files\MyFolder\"

        'Get the database, recordset, and attachment field

        Set dbs = CurrentDb

        Set rst = dbs.OpenRecordset("Mytable")

        Set fld = rst("MyPhoto")

        

        'Navigate through the table

        Do While Not rst.EOF

        

            'Get the recordset for the Attachments field

            Set rsA = fld.Value

            

            'Save all attachments in the field

            Do While Not rsA.EOF

                If rsA("FileName") Like strPattern Then

                    strFullPath = strpath & "\" & rsA("FileName")

                    

                    'Make sure the file does not exist and save

                    If Dir(strFullPath) = "" Then

                        rsA("FileData").SaveToFile strFullPath

                    End If

                    

                    'Increment the number of files saved

                    SaveAttachments = SaveAttachments + 1

                End If

                

                'Next attachment

                rsA.MoveNext

            Loop

            rsA.Close

            

            'Next record

            rst.MoveNext

        Loop

        

        rst.Close

        dbs.Close

        

        Set fld = Nothing

        Set rsA = Nothing

        Set rst = Nothing

        Set dbs = Nothing

    End Function

ken sheridan say: The GetFilePath function in the basGetFilePath module is a simple little function which uses the FileDialog method of the Application object to browse to an image file.  In the relevant form it is called to assign the path to the ImagePath column in the table.  You can see more easily how it is used in the Images_Simple demo:

Private Sub cmdAddImage_Click()

    On Error GoTo Err_Handler
    
    Dim strPath As String
    
    ' open 'file open' dialogue and get path to selected file
    strPath = GetFilePath()
    
    If Len(strPath) > 0 Then
        Me.ImagePath = strPath
        Me.ctrlImage.Visible = True
        Me.lblNoImage.Visible = False
    End If
    
Exit_here:
    Exit Sub
    
Err_Handler:
    MsgBox Err.Description, vbExclamation, "Error"
    Resume Exit_here
    
End Sub

You should copy the module into our own database.  You'll then be able to use code like the above when associating a record with an image.  In the  Images_Simple file the code assigns the path to a column in the Addresses table.  In the other files the model is different, to allow multiple images to be associated with each record.  In this the paths are stored in an Images table and the many-to-many relationship type between the Addresses table and the Images table is modelled by an Addresses_Images table which resolves the relationship type into two one-to-many relationship types in the usual way.  When you browse to an image file the code first checks to see if the path already exists in the Images table before inserting a row into the Addresses_Images table.

If the relationship type between the Addresses table and the Images table were one-to-many, i.e. each image can only be associated with one record, then the Addresses_Images table would be unnecessary and the Images table would have an AddressID foreign key column referencing the primary key of Addresses.
