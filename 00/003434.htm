xor lx say:

You're welcome.

1) =SUM(IFERROR((MMULT({1,1},0+(ISNUMBER(FIND({"spaar";"deposito"},T1139:CK1139))))=0)*T1141:CK1145,0))

2) If you want, yes. MMULT simply performs the equivalent of adding the two OR clauses, and so also abbreviates the formula. Unfortunately, it will still need to be an array version with SUM, e.g.:

=SUM(IFERROR((ISNUMBER(FIND("spaar",T1139:CK1139))+ISNUMBER(FIND("deposito",T1139:CK1139)))*T1141:CK1145,0))

The reason that SUMPRODUCT is not an option is for the following two reasons:

a) You have text values in your sum range (at least, this is what I presume from your statement that it contains "formulas resulting in numbers or blanks" - since the "" as a result of formulas in those cells are technically text).

b) Your ranges are not of equal dimensions: T1141:CK1145 is a 5-row-by-70-column array whereas T1139:CK1139 is a 1-row-by-70-column array.

Interestingly, if only one of the above were the case, we would be able to use SUMPRODUCT, though the form would be different in each case.

For example, if the ranges were of precisely the same dimensions, we could use the so-called "native" (comma-separated) form of SUMPRODUCT, which ignores text values in the ranges passed to it (many people aren't aware of this crucial difference between this and the "product" form of SUMPRODUCT, the latter of which errors when passed ranges containing text values). For example, if you just wanted to sum the range T1141:CK1141: 

=SUMPRODUCT(((ISNUMBER(FIND("spaar",T1139:CK1139)))+ISNUMBER(FIND("deposito",T1139:CK1139))),T1141:CK1141)

would be correct, even with text values (the "") in the range. However, these very text entries would cause the near-identical "product" form:

=SUMPRODUCT(((ISNUMBER(FIND("spaar",T1139:CK1139)))+ISNUMBER(FIND("deposito",T1139:CK1139)))*T1141:CK1141)

to error.

Conversely, if there were no text values in the range, then the "product" version would be the correct choice, since, unlike the "native" version, it does not require that the dimensions of the ranges be identical (though of course they should contain either an equal number of rows or columns). Hence, if you were to delete all occurrences of "" in the range, then you could use:

=SUMPRODUCT((ISNUMBER(FIND("spaar",T1139:CK1139))+ISNUMBER(FIND("deposito",T1139:CK1139)))*T1141:CK1145)

It's just unfortunate that your circumstances prevent you from taking advantage of either of these forms.

graham mayor say:

Hmmm. Maybe not so bleak after all. The following Outlook macro may serve your needs. It looks in the name strings for a matching entry to the input box (case insensitive) and displays the appropriate address list. It appears to work OK in brief tests.

Sub FindInAddressBook()
Dim oAL As AddressList
Dim oAE As AddressEntry
Dim oDialog As SelectNamesDialog
Dim strName As String
    strName = LCase(InputBox("Find name ... "))
    For Each oAL In Application.Session.AddressLists
        For Each oAE In oAL.AddressEntries
            If InStr(1, LCase(oAE.Name), strName) > 0 Then
                Set oDialog = Application.Session.GetSelectNamesDialog
                With oDialog
                    .InitialAddressList = oAL
                    .ShowOnlyInitialAddressList = True
                    .Display
                End With
                Exit For
            End If
        Next oAE
    Next oAL
End Sub

alexsun_2012 say:

Hi Tyler,

The delegations and other configurations and rules on the mailboxes are not migrated because during the migration only the user account and mailbox data are copied. The resource mailboxes should be migrated. However, they could be migrated as user mailboxes. So you may need to convert them into resource mailboxes using PowerShell cmdlet.
