john sr wilson say:

This should send slide 3 - change as needed.

It needs the full version of Outlook to work.

Sub mailme()
Dim outlookApp As Object
Dim outlookMail As Object
Dim currPres As Presentation
Dim singleSlide As Presentation
Dim L As Long
Set outlookApp = CreateObject("Outlook.Application")
Set outlookMail = outlookApp.CreateItem(0)
Set currPres = ActivePresentation
currPres.SaveCopyAs Environ("TEMP") & "\SingleSlide.pptx"
Set singleSlide = Presentations.Open(Environ("TEMP") & "\SingleSlide.pptx")
For L = singleSlide.Slides.Count To 1 Step -1
    If L <> 3 Then  ' three would be the slide to keep alter as needed
        singleSlide.Slides(L).Delete
    End If
Next L
singleSlide.Save
On Error Resume Next
With outlookMail
    .To = "*** Email address is removed for privacy ***" ' change of course
    .Subject = "Test Mail"
    .body = "This is the message" & vbCrLf & "Line two here"
    .Attachments.Add singleSlide.FullName
    .Send
End With
singleSlide.Close

Kill (Environ("TEMP") & "\SingleSlide.pptx")

End Sub

Your mention of the development tools is likely key, since developers often have problems when using the Microsoft antimalware engine based security products.  This is because the behavioral monitoring portions of these programs will both try to monitor all potentially suspicious process activity as well as run emulation for any unsigned, unrecognized files found stored on a system.

For the typical consumer PC user, such files and activity are usually rare, since the only software most of these people have are well known applications from major vendors.  A somewhat smaller group may have less well known applications or even questionable PUPs (Possibly Unwanted Programs installed which can cause the sort of heavy engine activity that's often discussed by those coming here.

For a developer though, the probability is very high that there may be many files and especially processes operating during development that are quite uncommon.  Since the debuggers and unidentified dll and other executables being used have no real identity yet, the antimalware engines can go into overdrive trying to determine what they are and what they might be attempting to do.

So though you see the common answer of use something else as a cop out, that's the most simple way for those not wishing to explain or possibly not truly understanding the complexities of these issues to answer.  The reality is that the Microsoft free consumer security apps such as MSE and Defender are designed precisely for those consumer users and so may not perform as well for some with specialty needs like developers.

You can possibly lessen this issue by signing your executable and other files if not already doing this, but even then any new unsigned portions or especially the processes in memory will likely still generate lots of monitoring overhead.  So talking to other developers to determine which security products work best for them might be worth your time.

It's also possible that many developers simply disable the real-time protection of whatever security application they're using while performing development, though of course most like Microsoft would say you shouldn't do this.  However, those with a deep understanding of the risks involved often realize that having a strict process which includes no web browsing or other Internet access while developing might find this a workable solution.  They also may simply use specially isolated development machines which have no Internet access at all, since that's the surest way to avoid malware and a probable method used in larger development firms with the resources to do this.

I know that we provided such a separate development network and machines for the day to day operation vs. the build network in a PC manufacturing concern were I was the network administrator.  The machines under development and test rarely had antivirus unless that was the application being tested, so the entire engineering network was often unprotected on a per machine basis, but also had no direct Internet access at all.
