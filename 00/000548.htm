joeu2004 say:

I suspect that Excel GEOMEAN has an arbitrary limit of 2048 products, at least in Excel 2010 (my version).

It does not matter whether that is one range or multiple parameters.

Excel PRODUCT does not seem to have the same limitation.  So, as a work-around, use the following:

PRODUCT(GA3:GA2638)^(1/COUNT(GA3:GA2638))^254.24 .

To demonstrate, download "geomean error.xlsx".  Ignore any preview error or request to log in, and just download the file.

The following image shows part of it.

Formulas (copy down the columns):

B2:  =PRODUCT($A$2:A2)

C2:  =PRODUCT($A$2:A2)^(1/COUNT($A$2:A2))

D2:  =GEOMEAN($A$2:A2)

E2:  =C2-D2

F2:  =COUNTIF(E2:E2637,0)

G2:  { =10^AVERAGE(LOG($A$2:A2)) }

H2:  =C2-G2

I2:   =COUNTIF(H2:H2637,0)

Note that the formulas in column G are array-entered:  do not type the curly braces; and press ctrl+shift+Enter instead of just Enter.

I constructed random data in A2:A2637 so that PRODUCT(A2:A2637) is 90.4%.

(For simplicity, PRODUCT of every even number of cells is 90.4%.)

Note that GEOMEAN in column D returns #NUM starting with D2050, a range of 2049 cells.

But the equivalent PRODUCT calculation in column C has no problem.

Also, the equivalent AVERAGE(LOG) calculation in column G has no problem.

(There might be infinitesimal differences between columns C, D and G.  For my example, the differences in columns E and H demonstrate that columns C, D and G are equal to 15 significant digits.  Well, sort of; it's complicated.)

joeu2004 say: In reply to deleted message

Dana wrote:

If we limit your data to around 0.99   (   =0.98+0.02*RAND()   )

Then the size that GEOMEAN can handle varied.

This might depend on the Excel version.  When I enter that formula into A2:A2637 in "geomean error.xlsx", GEOMEAN handles ranges of only 1038 or 1039 values, for repeated recalculations.

But you're right: the GEOMEAN limitation seems to depend on the values themselves, not just the number of values.

(That particular value sensitivity has nothing "obvious" to do with the binary arithmetic limits, since PRODUCT(A2:A2637) works fine.)

BTW, there seems to be something wrong with your examples, supposedly populating the formula =0.98+0.02*RAND().

The values should be less than 1 (or infinitesimally larger); and that is true when I do it.

But the image that you posted shows some values between 1.005 and 1.009, much too large to be explained by binary arithmetic anomalies.

So on a whim, I populated A2:A2637 with the formula =1+0.009*RAND().  In that case, GEOMEAN handled the full range; no #NUM errors(!).

Well, I think we've beat this horse to death.  My conclusion:  GEOMEAN is unreliable to use, and my previous attempts to explain it were misguided.

IMHO, in general, the best alternative to use is the array-entered expression of the form 10^AVERAGE(LOG(range)), since that allows for ranges of values that even exceed the 64-bit binary floating-point limitation of PRODUCT(range).

DoCmd.RunSQL "UPDATE Table1 SET Table1.Field2 = Table2.Field2 WHERE Table1.Field1 = Table2.Field1;"

If this is a correct transcription except for the table and field names, then the SQL statement is not valid, because Table2 is not participating in the query at all.  You would have to join Table2 to Table1:

DoCmd.RunSQL "UPDATE Table1 INNER JOIN Table2 ON Table1.Field1 = Table2.Field1 SET Table1.Field2 = Table2.Field2;"

That may not be the only problem, but it's one problem.  I'd expect you to get a parameter prompt, though, rather than a syntax error, when you try to run it.

If your query actually does include Table2, then we need a better transcription of the SQL statement to see where the error is.
