ken sheridan say: If the difference is always a negative value then try this query:

SELECT PP1.Date,
PP1.Product AS Item1,
PP2.Product AS Item2,
PP1.Price - PP2.Price As Difference
FROM ProductPrices AS PP1, ProductPrices AS PP2
WHERE PP1.Date = PP2.Date
AND PP1Price > PP2.Price
ORDER BY PP1.Date, PP1.Product;

BTW, do not use 'Date' as a column name .  It is the name of a built in function in MS Access, so as a 'reserved word' should not be used as an object name.  Use a more specifically descriptive name like TransactionDate or similar.

ken sheridan say: The one-to-one relationship types worry me.  A one-to-one relationship type is generally only used to model a type/sub-type.  Chris Date, in his classic book Introduction to Database Systems, gives the example of type Employees with sub-type (amongst others) Programmers, and as sub-types of programmers, System Programmers and Application Programmers.  A sub-type is caharcterized by sharing all attributes of its (super) type, but not those of other sub-types.  So while the Employees table will have columns representing attributes common to all employees, such as FirstName and LastName, the Programmers table will have column representing those attributes only applicable to programmers.  Similarly the SystemProgrammers and ApplicationProgrammers ables will have columns representing attributes specific to the type of programmer in question.  An employee could of course be represented in each of these tables, all of which will have an EmployeeID primary key, which in the case of the tables modelling sub-types, is also a foreign key.

You say that you have a table Company Contacts.  If the relationship type is one-to-one then this means there can be only one contact per company, which may be the case in the context of your business model, but sounds unusual.  If this is the case, however, then you do not need a separate table at all, you can simply include the columns representing attributes of the single contact in the Company Details table.  If, on the other hand, there can be one or more contacts per company, then you do need a separate Company Contacts, in which Company Name is a foreign key, but the relationship type between Company Details and Company Contacts is one-to-many.  In this context a Company Contacts subform within a Company Details parent form is the normal user interface.  The users are still opening a single form so your aim is achieved, but without the possibility of duplication.

I think you need to examine the logical model of your database before considering  the method of data entry.   At present, to judge from your description, it does not appear to represent an accurate model of the underlying reality.  Getting the model right at the outset is the key to a successful and robust database.  Get it right and the interface falls into place relatively easily; get it wrong and you'll end up spending a lot of time and effort trying to work around the flawed design, and often not succeeding.

daniel g. grau say:
It is possible that your Note Level styles have incorporated the unwanted text attributes to their style definitions. This could be done manually or automatically through automatic updating of styles. Try this:
  1. Change to Draft View or Print Layout View
  2. In the main menu bar, select Format > Style
  3. In the list to the left, click on style "Note Level 1"
  4. Click on the Modify button
  5. If there is a checkmark next to "Automatically Update," take it off
  6. Check that the unwanted text attributes are not part of the style
  7. Check the "Based on" attribute (this defines how changes to other styles propagate to the style being modified)
  8. Repeat the process for "Note Level 2," etc.
  9. Go back to Notebook Layout and test

If that fails to work, try disabling the AutoCorrect function altogether (checkmark off in Word > Preferences > AutoCorrect, next to "Automatically correct…") just to see if that feature is malfunctioning.
 
-----------------------------------------------------------------------------------
The "Format beginning of list item..." only works when:
  1. the previous bullet is at the same level,
  2. it has punctuation (commas don't count), and
  3. the style before such punctuation has some text attribute that is absent in the rest of the bullet text.

This option will make Word follow the same "dual text style" for the next bullet, auto-switching formats at the punctuation symbol. For instance:

  • This first part is bold. Then I take bold off. When I press Return…
  • This automatically becomes bold until the punctuation mark. Then it switches to not-bold by itself.

The way Word usually works, after hitting Return you should still be typing in plain text (unless the Note Level paragraph style imposes otherwise).

It doesn't work the other way around (plain-bold does not trigger plain-bold) and it works with some other formatting attributes as well (color, underline, italics…).
