jose est say:

Hi,

Error code 0x807800C5 occurs when trying to back up a second system image of a Windows 10 computer to NAS.

The cause of this issue is the fact that Windows only allows one system image to be backed up to NAS. Windows 10 is not capable of overwriting or deleting the first system image, ultimately causing the second image backup to fail.

We can resolve this issue by renaming the original backed up system image so that Windows will be able to successfully create the new system image. To do this, we need to:

  1. Gain access to the NAS where we are trying to back the system image. We will not be able to apply this solution unless we have access to the NAS or the NAS shell as we cannot rename the original system image if we don‚Äôt have access to where the original system image is stored.
  2. Locate the original system image.
  3. Rename the original system image from __nas_backup_WindowsImageBackup_host-name to __nas_backup_WindowsImageBackup_host-name.bak.
  4. Return to your computer.
  5. Run Windows 10 Image Backup and try backing up a system image to the NAS, we should be able to successfully back up without running into an error.

Let us know if you need further assistance.

tryx3 say:

You are looking in the wrong place.

 

 

Look in:-

Applications and services logs,

Microsoft,

Windows,

NetworkProfile,

Operational

 

 

Event ID 10000 includes network connections and also some of the interim events that occur as a connection is being made.  So the event ID itself is not enough to run a task as it gets generated by those interim steps as well.

 

 

If you are going to go on to run a task using this, you will have to get to grips with the Windows 7 wevtutil utility.  You can get its help page by entering wevtutil /? in a command prompt.

 

 

You will need to use event ID 10000 to trigger a batch file that runs this line

 

 

wevtutil qe Microsoft-Windows-NetworkProfile/Operational /c:3 /rd:true /f:text >D:\Desktop\NetworkEvent.txt

{substituting your own chosen folder & filename of course}

 

 

I chose the switch /c:3 to capture the actual network connection as well as the interim ones.  By capturing the last 3 events, I am confident of capturing the connection event [and I don't mind that it also captures the interim steps].  There is no method for capturing the connection event without including some interim events.

 

 

Then you will need to use commands to find the text Network Connected within the output file and then read the next line after the word Name: as the rest of that line contains the name of the network that has just been connected to.

 

 

The output from the wevtutil  command above will contain several events during network connection which is why you need to seach for the Network connected entry.  A sample of that output file [a third of it] is

Windows Vista Business > Windows 7 Professional, Enterprise or Ultimate
