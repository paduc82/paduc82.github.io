Dynamic format changes do not work correctly on Windows 10 technical preview 9926

We have a 1394 audio device driver which creates a WDM audio device. The device driver for this WDM device is based on wavecyclic miniport driver model. Driver publishes one playback and one recording interface to Windows which appears in Playback and Recording devices respectively in Sound and Audio control panel. Device supports sample rates in range of 32 kHz - 192 kHz. 1394 device has its control panel and sample rate is controlled by the device control panel. This means that playback and recording devices show formats (in Sound and Audio CPL) for device CPL selected sample rate. Wavecyclic miniport driver supports dynamic format changes and hence it implements KSEVENT_PINCAPS_FORMATCHANGE. This implementation works accurately under Windows 7 operating system (both 32 and 64 bit).
However we are seeing that under Windows 10, Sound and Audio control panel is not updating "Default Format" combo (on Advance tab) for our playback and recording devices correctly after a sample rate change. Default Format control on Advance tab shows entries for newly selected format as well one entry for last selected format for both out and in WDM devices.

Here is more detail on the problem:
- When there is a format change request (sample rate, bit depth etc) on device control panel, driver calls GenerateEventList with KSEVENT_PINCAPS_FORMATCHANGE on both playback and recording pin.

- Windows sends KSPROPERTY_PIN_PROPOSEDATAFORMAT with SET request with old/last selected default format on both the pins. Driver returns STATUS_NO_MATCH to this call. After this nothing happens.

Following calls come to driver when I open device properties in Sound and Audio CPL for playback device:

- Windows sends KSPROPERTY_PIN_PROPOSEDATAFORMAT with SET request on playback pin. The buffer sthis call is lower than what driver expects. Precisely ValueSize field of PCPROPERTY_REQUEST comes with a value 40 whereas driver expects it to be sizeof(KSDATAFORMAT) + sizeof(WAVEFORMATEXTENSIBLE) which is 68. Windows sends this request with same size 40 thrice. Driver fails this request with STATUS_BUFFER_TOO_SMALL.

- Windows sends SET request for KSPROPERTY_PIN_PROPOSEDATAFORMAT once again with a small buffer size and driver returns StATUS_BUFFER_TOO_SMALL.

- Now Windows sends GET request for KSPROPERTY_PIN_PROPOSEDATAFORMAT. This time ValueSize field of PCPROPERTY_REQUEST is set to zero. Driver sets ValueSize to sizeof(KSDATAFORMAT) + sizeof(WAVEFORMATEXTENSIBLE) and retruns with STATUS_BUFFER_OVERFLOW.

- Windows sends GET request again with a buffer size that driver expects. Driver fills preferred format in Value field of PCPROPERTY_REQUEST and returns STATUS_SUCCESS

- Windows calls NewStream followed by Init on the filter.

- Windows sends KSPROPERTY_PIN_PROPOSEDATAFORMAT several times to query formats supported by the driver. Driver sends STATUS_NO_MATCH on all formats that are not supported and STATUS_SUCCESS for the format supported.

Result is Sound and Audio control panel shows old/last selected format as default entry. If I open default format control it lists entries for newly selected format as well as one entry for last selected format. Windows should query device's preferred format when driver fails KSPROPERTY_PIN_PROPOSEDATAFORMAT SET request first time after GenerateEventList.


Here is more detail on how driver is implementing KSEVENT_PINCAPS_FORMATCHANGE:

- Driver declares an event item for format change

static PCEVENT_ITEM PinFormatChangeEventTable[] =
{
  {
    &KSEVENTSETID_PinCapsChange,   // Something on the pin changed!
    KSEVENT_PINCAPS_FORMATCHANGE,  // Format changes
    KSEVENT_TYPE_ENABLE | KSEVENT_TYPE_BASICSUPPORT,
    WaveEventHandler
  }
};

- Driver creates automation table and uses the automation table on the wave filter.

- Driver handles KSEVENT_PINCAPS_FORMATCHANGE event in the WaveEventHandler. It returns STATUS_SUCCESS  in PCEVENT_VERB_SUPPORT and calls AddEventToEventList in PCEVENT_VERB_ADD.

- Whenever there is a sample rate change on 1394 device control panel, WDM miniport driver calls GenerateEventList on playback and recording pin.

- Driver also declares KSPROPERTY_PIN_PROPOSEDATAFORMAT as follows:3

{
        &KSPROPSETID_Pin,
        KSPROPERTY_PIN_PROPOSEDATAFORMAT,
        KSPROPERTY_TYPE_GET | KSPROPERTY_TYPE_SET | KSPROPERTY_TYPE_BASICSUPPORT,
        PropertyHandler_WaveFilter
}

- When KSPROPERTY_PIN_PROPOSEDATAFORMAT sends a Get/Set request, driver validates buffer size. In case buffer size is zero, driver sends required buffer size in ValueSize field of PCPROPERTY_REQUEST with a  STATUS_BUFFER_OVERFLOW. In case buffer size is less than what driver expects (which is also returned with STATUS_BUFFER_OVERFLOW) driver returns STATUS_BUFFER_TOO_SMALL.

- If driver receives appropriate buffer size then driver sends preferred format in GET request.

- When KSPROPERTY_PIN_PROPOSEDATAFORMAT sends a Set request driver validates format. In case driver does not support that format, it returns STATUS_NO_MATCH otherwise STATUS_SUCCESS is returned.

Behavior on Windows 7:
- When there is a format change request (sample rate, bit depth etc) on device control panel, driver calls GenerateEventList with KSEVENT_PINCAPS_FORMATCHANGE. Windows send SET request for KSPROPERTY_PIN_PROPOSEDFORMAT with last selected format.Driver returns STATUS_NO_MATCH to this.

- Windows send GET request with appropriate buffer size and driver sends preferred format which is newly selected format.
- Windows send SET request with new format followed by NewStream and Init.

By default, Word saves templates in

C:\Users\<yourusername>\AppData\Roaming\Microsoft\Templates

if you're on Windows Vista or Windows 7, or in

C:\Documents and Settings\<yourusername>\Application Data\Microsoft\Templates

if you're on Windows XP.

In Windows XP: start Windows Explorer, then select Tools | Folder Options...
