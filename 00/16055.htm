The question you asked is an FAQ

 

The phone book syncs do not work correctly on SIM Access devices.

/discussions/board/message?board.id=topfaq&thread.id=97

 

As for Remote SIM Paring, this normally has to be initiated from the Car itself, you normally have to search for handsets with the car, and then when its found provided you have the SIM Access switched on the car will offer a 16 digit passcode for you to pair your handset to the car.

Remote SIM only works with a 16 Digit code. If you enter 4 digits then it would appear that Hands free mode is being used not Remote SIM.

 

The two are different systems, remote SIM allows the car to handle the cellular connection not the handset.

Yes, it it is. In fact, it quite much the norm.
 
The reason for this of course is that if you take an existing application with years of code (which in most cases is likely to be DAO), then if you migrate the tables to SQL server you can well keep all of your forms, reports and existing code. In fact, the vast majority of the code and forms and reports will not have to be rewritten when you do this. Thus, it makes little sense to rewrite lots of code that is running fine, and you've already paid to create and maintain that code over many years.
 
So in access land, in addition to the above the issue of taking existing applications and being able to continue to use that DAO code when you link the tables to SQL server, most access developers by nature are more familiar with DAO code. And, likely the vast majority of the library code one has will also be DAO.
 
If you are familiar with ADO, or are you starting a project from scratch, and you plan to use SQL server, then I think ado is a great choice.  However if you're going to build an application that's going to run split with an jet back end, then perhaps down the road move that backend data to SQL server, then it's a bit more of a tossup as to which is a better choice.
 
About half of my applications I write in access are designed using DAO and in fact run with an SQL server back end.
 
From a performance scalability point of view, using DAO or ADO not really going to give you any significant gains in performance. It's really going to come down to other issues such as do you have existing DAO code. While ADO is much being superseded by ADO.NET, we see that the DAO and JET (now called ACE data engine) is receiving new features. (such as off line tables from SharePoint).
