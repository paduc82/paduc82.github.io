ken sheridan say: Although it operates in a different way to your application you might like to take a look at the SudentLog.zip file in my databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

You might have to copy and paste the text of the above (not copy the link location) into your browser; for some reason it sometimes doesn't seem to work as a hyperlink.

The file is principally designed to illustrate how to build correlated subforms, but it so happens that it uses a simple student registration and attendance database as the basis for this.  I have updated it slightly so that the nearest equivalent to your interface is the Courses form, which includes a registrations subform, and correlated course sessions and attendances subforms.  Registration is per course, not per session.  Even if the interface does not suit your requirements, you might find it helpful to take a look at the relationships window, as this shows the underlying model appropriate to this kind of thing.

When modelling attendances and the like it is normal to do so by recording attendances per event rather than attendance or non-attendance in Boolean (Yes/No) columns.  The latter really amounts to what is known as 'encoding data as column headings', which is not good database design as all data should be stored only as explicit values at column positions in rows in tables.  This is known as the 'Information Principle' and is one of the basic principles of the database relational model. To use check boxes rather than conventional subforms the usual solution is to use unbound controls and read/write the data to the tables in code.  There is a file VolunteerRota.zip in my SkyDrive folder at the above link which shows both the conventional approach, using bound forms and subforms and the use of unbound forms/subforms with check boxes, each using the same tables.  In the case of the subform with check boxes the data is written by means of a transparent button overlying the check box, whose ControlSource is an expression which determines whether it is True or False.

While, when first using a relational database system like Access it seems logical to have Boolean columns to represent a True or False value for some attribute, one thing you'll find is that this makes it difficult to query the database, especially when aggregating entity types being modelled by tables, whose columns model the attribute types of the entity type, with the values the attribute type, i.e. the data, being stored at the column positions in each row.
