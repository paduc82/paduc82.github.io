Stop processing only applies to messages touched by the that rule - this is to prevent duplicate messages or other weird behavior if you have more than one rules that applies to the message.

Rule 1: delete messages from Jim
Rules 2: move messages with important in the subject

With stop processing on rule 1, a message from jim where the subject is "Important! Read me!" is only processed by Rule 1. Without stop processing, it would also be processed by Rule 2.

A message from Bob with Important in the subject would be processed by rule 2 - as Rule 1 does not apply.

If your other sheet names are '1', '2', '3', etc. Then this will work for you.  The result of ROW() or of the calculation of ROW()-2 would be used as a sheet name.

If you put the formula =INDIRECT(ROW() & "!$F$8")  into a cell on row 1 in a sheet then it will show the value from sheet '1', cell F8.  It would be the same as writing ='1'!$F$8

Encrypted connections stop working on one user's Outlook - Microsoft Community

ianjustman say:

Encrypted connections stop working on one user's Outlook 2013(?) yet still work on others'

Hi, all.

I am writing for someone who may be using Outlook 2013 (I will try to follow up to see if I can get an exact version) to access her e-mail.  She said it worked as recently as Tuesday or Wednesday of this week.  Today, she tries to get on and gets an error 0x800ccc1a.  Everyone else at her location isn't getting the problem.  She's connecting via POP3 and SMTP secured by SSL and TLS respectively.

The server they use is one I own and operate myself.

To REALLY give testing some rigor, I used my own copy of Outlook 2013 and connected to an account I have for myself on my server.  I configured Outlook in exactly the same way.  Everything works.  I tried both POP3 (which apparently can only be secured via SSL) and IMAP (my preferred mailbox access method, which can be secured via TLS) with SMTP secured with TLS.

It bears further noting that recently, there have been significant design flaws found in TLS 1.0 and any manner of fallback.  As a result, I have changed my server's configuration so that the only way you can connect securely to it is via TLS1.1 or 1.2.  I have also disabled several insecure ciphers, like RC5 and DES along with several insecure hashing functions like MD5 and SHA-1.  That along with keeping the software that implements this security suite up to date, OpenSSL in this case.

I have had the user disable encryption for the time being until we have a possible solution.  One thing I have recommended she do when she has some downtime is perform a Windows Update in the hopes that some updates for Outlook 2013 might have been pushed out to address this issue.

Thus far, she's the only person I know of who has this issue, and since I own an operate the server AND have tested it myself with my own copy of Outlook 2013, I can say with complete confidence that my server is not to blame.

Any ideas?

Thanks!

ianjustman say:

I found the real problem a different way, and it was by way of my server's web-based spam quarantine management system.   I normally assume it uses an unencrypted connection to localhost since I don't explicitly specify it as being encrypted.  Imagine that, an unencrypted connection pointing out a cryptosystem configuration flaw!  To be fair, it probably attempted to negotiate an encrypted connection first, which made my looking for a solution all the more urgent since people use that interface.

As it turns out, I improperly set up disabling of parts of my IMAP/POP3 server's cryptosystem, served by Dovecot and secured by OpenSSL.  I accidentally told it to disable SSLv3 CIPHERS, which is incorrect.  What I meant to do was disable the SSLv3 PROTOCOL.

Further technical details: some ciphers in use with SSLv3 are still in use with TLSv1.0, TLSv1.1 and possibly TLSv1.2 are still viable.  Using SSLv3 and TLSv1.0 as PROTOCOLS, on the other hand, is a mistake given fatal flaws discovered in both protocols, not in any of the software which implements it.

Given the fact I discovered that the issue was entirely server-bound via a server-based client and was misconfiguration on my part, I have to call mea culpa on this one.

--Ian.

P.S.  I don't  have SPA configured on the server side, so I always tell my users who use Outlook to not use it anyway.
