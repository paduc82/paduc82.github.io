    Michael wrote:

    Thanks a lot! I understand the technical background now.

    You're welcome.  I meant to add the following, which shows the exact decimal approximation of the other values with decimal fractions.

    Note, for example, that 100.2 rounds to 100.20000000000000, which is why it is simply 100.2 in the XML file.

    The surprises are 100.4 and 100.9, which we might expect to round to 100.40000000000001 and 100.90000000000001 in the XML file.

    I suspect that the reason they do not has to do with "banker's rounding" (round half to even) v. "normal" rounding (round half away from zero).  But that is only an educated guess at this point.  And probably TMI.

    joeu2004 say:

    -----

    Actually, the number is more, not less, precise.

    And there is nothing wrong.  That is actually the decimal approximation of the binary value that is stored internally in Excel.

    The extra decimal places are unexpected, albeit correct, for several reasons.

    First, Excel format (displays) only up to the first 15 significant digits (rounded).  So the constant 68.4 appears to be 68.400000000000000 even when formatted to display 15 decimal places (17 significant digits).

    But numerical values are stored internally using 64-bit binary floating-point, and it requires

    So Excel formats up to the first 17 significant digits (rounded) in the XML file, in part because (compressed) XML is used to repesent saved "xlsx" and "xlsm" Excel files.

    The 64-bit binary floating-point representation of numerical values is the sum of 53 powers of 2 ("bits") times an exponential factor. Consequently, most decimal fractions cannot be represented exactly.  Moveover (although not relevant for this question), the approximation of the same decimal fraction varies, depending on the magnitude of the integer part.

    For example, the decimal approximation of the constant 68.4 is exactly 68.4000000000000,05684341886080801486968994140625, which is 68.400000000000006 when rounded to 17 significant digits.

    (I use period for the decimal point and comma to demarcate the first 15 signficant digits.)

    Help with formula, trying to do VLOOKUP in a QUERY in importHTML

    I have a formula to pull in some data from a table on a website.  The formula is:

    =QUERY(importHTML("http://www.pro-football-reference.com/players/G/GurlTo01/gamelog/2015/","table",1),"select Col10, Col12, Col14, Col15, Col17 Limit 1 OFFSET 12")

    I would like to search in that table for one value to define the row to pull data from rather than using OFFSET so that I can apply this formula to tables of different sizes with similar data. Is it possible to add a VLOOKUP function to this formula to search for that one field?

    Also, is it possible to have the data pull into my spreadsheet without the column headings pulling in as well?

    chatmaker say:

    Where do you pull your data from?

    Sports.Yahoo.Com - Like pro-football-reference.com, Yahoo doesn't provide "Week" number in each player's stat table.  It provides date.  Unlike pro-football-reference.com, Yahoo lists the most recent game in the table's first row.  So when we select Load/Refresh Team from the BXL ribbon tab, it opens Internet Explorer, navigates to each player's stat table in Yahoo Sports, and loads the first row.  That may or may not be the most recent week (week 16). 

    Determining Week

    Click Display Score. Any players on my team that did not play in the most recent week are highlighted in red using conditional formatting. The conditional formatting relies on a named formula Ineligible. If we open the Name Manager and examine Ineligible we find this formula:

    =INDIRECT("tblScore[@[Game Date]]") <=MAX(INDIRECT("tblScore[[Game Date]]"))-6

    In English that is saying: If the value in the Game Date column of the current row is less than the maximum value in the entire Game Date column minus 6 days then the player's stats came from a prior week and is thus, ineligible for this week's scoring.

    Impact on Scoring:

    If we click on any of the scores in the Score table, we see this formula:

    =IF(Ineligible,0,Stats*Points)

    In English that says, if the player is Ineligible, give them 0 points, otherwise, give then a score equal to the value in the players Stat table multiplied by the Points awarded for that stat.  So not only are we highlighting the player's stats in red, but we are also excluding him from any scoring.  BTW Stats and Points are also named formulas so to see their actual formula we have to open Name Manager and examine them there.

    pro-football-reference.com

    Your source puts the most recent game at the bottom of the table.  If I were using your source, I would scrape the last row instead of the first row which is more difficult and one reason I use Yahoo Sports. But after scraping the last row, we would still have to apply similar logic which is to compare the value in the date column to the maximum value for all other players on our team.  If it is within a week, they are all in the same week.  Of course, this means I must download my team on the day after the last day of the week.

    Finding the last row:

    Assuming we downloaded the entire table for Todd Gurley, to find the most recent game we need to determine the maximum value for the RK column then we could use VLOOKUP to get stats based on that value.  We would still need to test the date to make sure the date value is within the week we want.  

    Example:

    If we used POWER QUERY, it would download this data into a QueryTable and given that table the default name Table_0.  So our formula to find the most recent game's rushing yards would be

    =VLOOKUP(MAX(Table_0[Rk]),Table_0,10,FALSE)

    Determining the Week date range
    It would probably make sense to create a table with three columns: Week #, Start Date, and End Date.  With this table we could use VLOOKUP to find week 16's start and end dates and then test the Date value Todd Gurley's last stat row to see if it is within that date range.

    Finding a Better Source
    A large part of web scraping is finding the best source.  For what you have described, I would recommend something like: 

    http://sports.yahoo.com/nfl/stats/byposition?pos=RB&conference=NFL&year=season_2015&timeframe=Week16&sort=49

    This web page allows us to set the week number and position in the URL (timeframe=Week16, pos=RB)  so we ONLY get the week we want.  From there, I'm assuming you would use VLOOKUP to find Todd Gurley and retrieve his stats.

    BTW - The POWER QUERY source is: 

    let
        Source = Web.Page(Web.Contents("http://sports.yahoo.com/nfl/stats/byposition?pos=RB&conference=NFL&year=season_2015&timeframe=Week16")),
        Data0 = Source{0}[Data],
        #"Changed Type" = Table.TransformColumnTypes(Data0,{{"", type text}, {"2", type text}, {"3", type text}, {"4", type text}, {"5", type text}, {"Rushing", type text}, {"Rushing2", type text}, {"Rushing3", type text}, {"Rushing4", type text}, {"Rushing5", type text}, {"Rushing6", type text}, {"6", type text}, {"Receiving", type text}, {"Receiving2", type text}, {"Receiving3", type text}, {"Receiving4", type text}, {"Receiving5", type text}, {"Receiving6", type text}, {"Receiving7", type text}, {"7", type text}, {"Fumbles", type text}, {"Fumbles2", type text}, {"Fumbles3", type text}})
    in
        #"Changed Type"

    albert d. kallal say:

    Sure. And you find in this answer why I consider this a different task then validation.

     

    To convert some text you enter into a column as lower case, you open up the table, click on table on the ribbon tab. You then click on the before change event

     

     SetField
      Name:     LastName
    Value     LCase([LastName])

     

    And I try this one in which the first character is forced to upper case, and the rest lower case.

    SetField
    Name:    LastName
    Value    UCase(Left([LastName],1)) & LCase(Mid([LastName],2,99))

     

    So above would set first letter to upper case, and the rest to lower case.

     

    So, such type of code would not be placed in validation, but in the before change event. The validation setting is to give an error message when input is wrong, but the above is code that runs to change things automatic.
