Access 2010 form combobox with composite primary key - Microsoft Community

jameschancey say:

Access 2010 form combobox with composite primary key

I am trying to create a combobox to populate data into an Access table. The table the combobox is looking up the info from has a composite key composed of two integers, ServiceID and GroupID. The Group table has 3 fields, ServiceID, GroupID, and Group Name. ServiceID and GroupID are the composite Primary Key.

The way it works is that there are 10 services, each of which has between 4 and 20 groups, so a group's PK is composed of its service number and its group number. Right now the combobox is only looking at the group number, so for example for Service 1, Group 1 it gives the correct group name, but for Services 2 - 10 it repeats the name for the group that corresponds to that group number from service 1.

Current row source is: SELECT [Group].[GroupName], [Group].[GroupID] FROM [Group] ORDERBY [GroupID]

I know I need to get [Group].[ServiceID] in there somewhere, but I don't know the exact syntax to use.

What I want is for the combobox to display the proper group name and place the value of GroupID in another table. It actually works now as far as putting the correct value in the other table (since it is displaying a group name with the same ID number), but when I look at the form after data has been entered it is displaying the wrong names, and this will confuse anyone else who looks at the data.

ken sheridan say: Your logical model presupposes there is a many-to-many relationship type between Services and Groups.  I'll reply below on that basis, but is this in fact the case or does Group functionally determine Service?  If the latter then the referencing table into which you are inserting a row requires only a GroupID foreign key, not a composite foreign key referencing a composite primary key of the Group table.  BTW Group is not a good object name as it is an SQL keyword; something like ServiceGroups would be better.

When inserting a row into a table with a composite key like this you should use two combo boxes.  The first should be bound to the ServiceID column and list all available services.  The second should be bound to the GroupID column.  The GroupID combo box should be correlated with the ServiceID combo box so that it lists only those groups which relate to the selected service.  So its RowSource property would be:

SELECT GroupID, GroupName
FROM [Group]
WHERE ServiceID = Form!ServiceID
ORDER BY GroupName;

In the ServiceID control's AfterUpdate event procedure set the GroupID control to Null and requery it with:

    Me.GroupID = Null
    Me.GroupID.Requery

In the form's Current event procedure requery the GroupID control with:

    Me.GroupID.Requery

This should work fine if the form is in single form view, but if it is in continuous forms view, because you are using surrogate keys, you'll find that the GroupID control will appear empty in those rows where the selected group differs from that in the current row.  No data will be lost, merely hidden.  This is a situation where natural keys have the edge because this problem does not occur, but where surrogate keys are used, and sometimes they can't be avoided die to the lack of any distinct natural key, the only option in a continuous form is to resort to a hybrid control in which a read-only text box is superimposed over a combo box to give the appearance of a single combo box control.  You'll find examples in ComboDemo.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

The structure of  this little demo file does not match yours, however, so the examples do not provide an exact paradigm.  They in fact more reflect the alternative linear model which I considered as a possibility in my opening paragraph.
