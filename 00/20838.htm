albert d. kallal say:

There are several approaches that can work.

Microsoft Sync Services can be used. You have to add some .net code and be “comfortable” building and working with .net. The sync framework behind the scenes indeed does use compact edition of SQL.

It would be really (but really really) cool if we could connect to SQL compact via ODBC. This would allow one to “sync” the compact edition SQL running on your laptop computer to SQL server at “home” base or even SQL Azure in the cloud. And this would work with the LEAST hassle!

However, because ODBC to SQL compact edition IS NOT ALLOWED, then you have to roll your .net sync from Access to compact. You thus cannot use compact edition of SQL with Access (oh, how I wish we could!!!).

As noted, you can still do the above, but you have to adopt the .net sync framework, and as a result you need developer experience with .net. So now you be using SQL compact, .net, and the .net sync framework. Certainly possible, but you now having to adopt 3 systems - each of which you need experience with.

Next up, and most easy, and my BEST recommend?

I would consider using SharePoint tables.

This is least hassle, and least cost and least code changes ****IF**** you application fits within SharePoint (SP) limitations.

And you can use low cost office 365 for this.

Access 2010 (or 2013) + office 365 allows you to send your back end tables up to office 365 as SharePoint lists.  Once done the your existing application (with little if any changes) will work in both off line mode, and “sync” the data tables to the back end SP tables. And really nice is this setup means you don’t have to write sync code, or even modify your application.

The above is my FIRST best choice. There are however some SIGNIFICANT issues before this suggestion will work or make sense for you. (more on this in a bit).

The simple Small Business Essentials office 365 plan at $5 per month will work for above. And that ONE ACCOUNT allows “many” users. So for a grand total of $5 per month, you can have 15 laptops out in the field in “off line” mode and with a “sync” mode in less time then it takes for me to write this post!!!

There are downsides to this setup. So NOT ALL applications will work well for this setup.

First, your relationships have to conform to SP lists. That means a standard simple auto number Primary key, and all child tables must use the traditional simple long number foreign key to relate back. (So PK and FK). This PK + FK setup has bene the “default” setup in Access for OVER 20 YEARS!

So there is a high possibility that your existing related tables will work as SP tables on 365.

However, if your relationships DO NOT conform to above PK setup such as using “natural” PK values, custom incrementing PK etc? Then such applications will require modification to work via the standard auto number PK and child FK. Thus how much work this will be depends on each application and how far it "strayed" from the default Access setup of 20+ years.

Next up is table size. During an up-size to 365, your data tables each should be less then 5000 records. And correct indexing for any columns used in searching etc. needs to be set BEFORE you upload. Tables can grow beyond 5000 rows, but you cannot “change” or “add” indexing to tables once they reach over 5000 rows on 365 tables (unless you purchasing VM hosted SharePoint or  you have your own SP server).

So while restrictions exist, the above is a GREAT choice. Here is a video of mine explaining how to up-size tables to 365.

http://www.youtube.com/playlist?list=PL27E956A1537FE1C5&feature=plcp


The NEXT best choice, and one with the LEAST restrictions is to run SQL server express local on each laptop. You thus migrate your tables from Access to SQL server, and run a local copy of SQL server on each laptop in the field.

These local editions of SQL server thus become “replication” clients to your copy of SQL server running at your work place, or they can be SQL Azure running in the cloud. This setup is essentially the same as setting up a sync framework with .net, but would not require .net skills, would not require you adopt .net sync. If you have existing SQL server skills and you migrated Access back ends to SQL server, then this choice ranks high on my list if the SharePoint tables option cannot be used.

The other possible is to “roll” your own. But this suggestion makes sense for small application with say 2-5 tables, and NOT a large or complex related schema like you suggest you have now.

The “more” complex the relationships and application? Then the MORE you want to adopt a technology stack that automates the sync and replication of data from the “clients” to the server.

I have never used John Walkenbach's add in, but as I recall, it provides functions to work with "strings" that look like dates before 1900.  However, it does not change the built in behavior of Excel. 

Dates in excel are stored as the number of days from a base date.  The base date for windows is 1/1/1900.  So excel is written not to support dates before that base dates - it doesn't support negative date serial numbers.  However, the addin might have a function that reads a cell with a string that can be translated into a negative date serial and put that value in a cell.  Sort would then be able to be done using these pseudo date serial numbers.   Again, I don't know anything specific about this addin but that seems like something logical it might order or it might provide a its own sort code that will sort these string dates.   I would assume it comes with some documentation or a help file.
