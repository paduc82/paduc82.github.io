I found a solution to the problem of Visio viewing lines or segments of arc as separate entities, or filling joined line/arc combinations inappropriately, when in fact we intended for them to be filled as closed figures. 

As I mentioned above, working with the Shape Sheet's Geometry section to reconcile the end points to form a closed shape is part of the solution.  The part I learned later is that use of line segments from 'Drawing Tool Shapes' does not result in a line that is readily integrated with a series of arcs or other lines to become a closed shape. 

I found that I needed to replace the line segments I originally created with "Drawing Tool Shape' with lines draws using 'Drawing Tools' in the Toolbar options.

Once I replaced the errant line segments, reconciling the end points in the Geometry section of the Shape Sheet for the structure was almost 'automatic'. It became a case of ensuring the beginning and ending X,Y coordinates matched.  Clicking on a cell in the Geometry section causes a box to appear on the drawing, where the X,Y coordinate exists. Directly edit the contents of the X and Y cells to change the values to match the end point you want to join to.

Ensure the value for the Geometry1.NoFill field in the Shape Sheet is "FALSE" then select the structure, select Format, then 'Fill' to cause the closed structure to be filled.

This still doesn't solve the problem of structures in DFX files being imported as 'open' structures, but I wonder if the 'lines' in the converted DFX files are coming into Visio with characteristics similar to the lines I found are created through the 'Drawing Tool Shapes' selection. Once these line segments were replaced with lines created with the snap to endpoint capability used by the Drawing Tools line segment in Toolbar options, the end points of those replacement lines were able to be readily joined with arc segments' end points in the Shape Sheet.

a. user say:

FWIW I agree with Doug, because although there has long been a problem in Word there has never been any sign that Microsoft has any intention of fixing it.

There are at least two problems that can affect what Word retrieves:

 1. columns with "mixed data types"

 2. a long-term problem in Word to do with the way it generates the SQL query from your criteria.

The criteria in the dialog box are translated internally by Word into a piece of SQL that (should be) appropriate for the "database" you are using. For Excel, by default Word uses the ACE OLE DB driver, which basically uses the same SQL dialect as Microsoft Access.

So we might hope that the SQL generated would be something like

SELECT * FROM `Sheet1$` WHERE `ZIP DIST` <= 50 AND `RSVP` <> 'Yes' AND `RSVP` <> 'No'

However, Word generates different SQL depending on the type of data it thinks is in each column (for an Excel data source, it will be able to find out the data types (NUMBER< TEXT, DATE etc.) from the OLE DB Provider.

So in this case, it will probably decide that RSVP is TEXT, and in that case, it will instead generate something more like

SELECT * FROM `Sheet1$` WHERE `ZIP DIST` <= 50 AND (`RSVP` IS NULL OR `RSVP` <> 'Yes') AND (`RSVP` IS NULL OR `RSVP` <> 'No')

This is to do with the fact that as far as SQL is concerned, NULL values can exist (and may be different from a zero-length string such as '') and you can't "=" or "<>" NULL. In other words, if there is a record where RSVP is NULL, it will not be retrieved by a query that says SELECT * WHERE `RSVP` <> 'Yes'.

Now that Word has decided to expand each of those RSVP criteria into 2 criteria, it cannot stop itself. If you go back into the Filter dialog, you will probably now see 5 lines of criteria. Next time, possibly 9. Word only really a smaller number, and sooner or later it will generate a piece of SQL where you get an OR where there should be an AND. At that point, you will see ZIP DIST values greater than 50.

If you can use Word VBA, you can inspect the generated query string by issuing the following command in the Immediate window when your mail merge main document is the active document:

?ActiveDocument.MailMerge.DataSource.QueryString

I suspect in your case the ZIP DIST is not the problem, but if it is actually a text value, or if the OLE DB provider has decided that it is a text value, then you will get the same problem for that column.

If for some reason you have text values in the first 8 rows (say) of your table, the provider will decide that the column has data type TEXT and will convert susequent numbers to text. Because it's text, you will get that extra "IS NULL" test for ZIP DIST as well.

If the provider thinks ZIP DIST is numeric, Word just does the obvious test and does not test for NULL. But that doesn't matter because the problem is caused by those other tests for NULL.

IMO Microsoft could probably fix the problem by saving your criteria as you entered them and not trying to reconstruct them from the SQL it generated. I don't see any sign that it does so. I believe it could also do better on the "mixed data types" issue, too, but that is probably not the problem in this case.

If you need to fix it outside Excel, there are various possibilities, but they all involve "extra steps" or code and as with many workarounds, each approach tends to solve one set of problems and not another.

Hi,

 

Switching back to Hotmail will chaged back your interface from Outlook to Hotmail but it won't automatically change your email address back to your original address @hotmail.com. You'll need to rename the account back your Hotmail address manually. To do so, please follow the steps provided below:

 

Note: The following steps will currently only work if your previous account was @Hotmail, @Live or @Outlook ending in .com, .jp, .kr, es, .au, .ca, etc.  This rename process will not work if you previously renamed an @MSN.com account.  There is no way to rename an @Outlook, @Hotmail or @Live account back to an @MSN.com account at this time.

 

1.             In your browser, navigate to https://account.live.com/ManageAssocIds

2.             Sign in with your new @outlook, @hotmail or @live account.  (if you are already signed in with your outlook.com account this step will not apply)

3.             Under Other email addresses click the Remove link below your previous address.

4.             Click the Remove button to confirm the message.

5.             You should now see a message on the page stating, You've removed an email address from your Microsoft account.

6.             Navigate to https://account.live.com/ChangeId.aspx in your browser.

7.             Verify that Create a new email address is selected.

8.             In the New Microsoft account box, enter your previous address and select your previous domain (hotmail.com or live.com).  Note:  You must enter your previous address or the process will fail.

9.             Select the Save button.

 

Now you can use your original @Hotmail, @Live.com or @Outlook account to sign into your Microsoft services as you did before renaming your account.  The @Outlook, @Hotmail or @Live address will now be an alias on your account from which you can still send and receive messages within Outlook or Hotmail.
