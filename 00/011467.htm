Removing duplicates from a verily large amount of data in Excel - Microsoft Community

kzen9 say:

Removing duplicates from a verily large amount of data in Excel

I have a huge excel file that needs cleaned up and tons of duplicate data removed. The file is a blend of several files and apx 60,000 rows + 180 columns.  Entries over 15 years have created a cumbersome, slow, inefficient file with a lot of redundant data. I would like to find the most efficient way to remove duplicates.  Any ideas on the best approach on a file thus large?
jllatham say:

I'm not sure that CF would be of a great help to you - the formulas used in Conditional Formatting are "volatile" and have to be recalculated every time that anything on the sheet changes (formula results or cell value).  That's a lot of potential recalculation as you delete rows.

 

If you can decide which columns being identical on a row causes them to be considered duplicates of other entries in other rows, we can probably come up with a macro (VBA) solution that would deal with the issue - it might take some hours to run, but it would be a way of dealing with it.

 

The only 'prerequisite' I can think of at this point would be that all data in all columns be of the same type and formatting, row by row.  That is, each column should only contain one type of data entry, such as text, or numbers or dates or currency amounts.  Column A, for example, should not have people's names in some rows and their birthday in other rows in column A.

 

If the number of columns to be compared is large, the code won't look pretty - but still should work.  I believe that instead of a test like this for just 2 columns:

If Range("A" & testRow) =Range("A" & loopRow) AND Range("B" & testRow) = Range("B" & loopRow) Then

would be best split as

If Range("A" & testRow) = Range("A" & loopRow) Then

   If Range("B" & testRow) = Range("B" & loopRow) Then

'at some point all tests match up and one of the rows would be deleted (most likely the loopRow)

  End If

End IF

So with 100 or more columns, imagine how deep the IF statements are going to get.  Ugly, but working.  The premise being that in the first example, all testing has to be done even if the very first test fails, where as with the individual IF tests, the testing stops at the very first failure to match.  Quicker overall.

 

Also, with that in mind, if you know (based on experience with the data) which columns are most likely to be different, then those get tested earlier to rule out mismatches soonest.  It's not necessary to test in any particular sequence of columns: e.g., column AZ could be tested first, the column B, then Column D, then column A, back out to BA, etc. etc. - whatever sequence appears to have the best chance of ruling out duplicates earliest.

    That's a lot of questions for one post. I'll try to hit the
    highlights.
    As you already said, Access is designed to do control-level validation
    in the myControl_BeforeUpdate event. Set Cancel=True if the validation
    fails.
    In most cases LostFocus should not be used for validation. It does not
    have the benefits of BeforeUpdate. I write entire applications without
    ever using the Focus events.
    You should typically allow people to tab through your control without
    entering anything. Later on, when they try to save the record, it will
    fail because the control is bound to a Required field. Then they have
    to come back to the control, enter some data, and
    myControl_BeforeUpdate will fire.
    To test for spaces in the value write this one-liner in
    control_BeforeUpdate:
    Cancel = (Instr(Me.myControl.Value, " ") > 0)
    Or you could write a bit more verbosely:
    dim blnHasSpaces as boolean
    blnHasSpaces = (Instr(Me.myControl.Value, " ") > 0)
    if blnHasSpaces then
    Msgbox "Yo! No spaces.", vbCritical
    Cancel = True
    end if
     
    If you wanted to intercept space characters and "eat" them, you can
    write code in the myControl_KeyDown event:
    If KeyCode = vbKeySpace And Shift = 0 Then
    KeyCode = 0
    End If
