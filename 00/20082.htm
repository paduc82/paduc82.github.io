Conflicting Objects in an Active Directory infrastgructure - Microsoft Community

alex rizeanu msft say:

Conflicting Objects in an Active Directory infrastgructure

Technical Level : Intermediate

Summary

Two objects are in conflict when they share one or more of the same unique identifier attribute values. For example, two accounts in the same Active Directory database cannot share the same User Principal Name (UPN). If they did, the system could not choose an account to use to authenticate a user who is passing that specific UPN.

Conflicting accounts are rarely generated from within the same Active Directory. When it does happen, its usually due to a misbehaving provisioning system that contacts two separate domain controllers at or near the same time and generates two copies of the same account information. When these separate copies eventually replicate to the same domain controller, one is deemed a conflict object (CNF) and is marked as such, while the other is considered the "legitimate" copy. The presence of a CNF object in an Exchange directory can cause issues with account management, mailbox provisioning, and login.

Much more common, and far easier to reproduce, is the attempt to replicate, or synchronise, conflicting account information from one separate Active Directory to another. For example, attempting to export a user account from on-premises with UPN X to AzureAD when a cloud-only account with that same UPN already exists. You can have a mail-enabled Public Folder in Exchange Online (EXO) with smtp address X, and then you can create a user account in AzureAD with the same address. Within minutes, you'll receive an error in AzureAD for that account after AzureAD attempts to copy that information to EXO.


Details

IMPORTANT: As long as an account in AzureAD has a validation error, it will not sync to EXO until the conflict is resolved. Any changes to the account, including licensing it to create a mailbox, will not flow and will not complete.

It's also important to realize that any object, no matter what it's type, can conflict with another in this way. A user account can conflict with a group if they share the same email address. A contact can conflict with a public folder if they share the same address, and so on.

Unique Identifier Attributes

These attribute values must be unique across all instances of an account across all directories it replicates to.

  • EmailAddresses
  • UserPrincipalName
  • MicrosoftOnlineServicesID
  • WindowsLiveID
  • NetID
  • SamAccountName
  • ObjectID

  • ExternalDirectoryObjectID
  • ExchangeGUID
  • GUID
  • ArchiveGUID
  • DisabledArchiveGUID
  • DistinguishedName

All other attribute values, can and in some cases, will be the same between different objects. Display Name for example does not have to be unique. Many mailboxes may share the same servername if they are hosted on the same database, and so on.

Typical Errors

Conflicts can occur on accounts between AzureAD and EXO and between on-premises AD and AzureAD. There are two places where you will see errors recorded for detected conflicts:

  1. The account in AzureAD that attempted to sync to EXO.
  2. The on-premises directory sync tool (ex. AzureAD Connect) installed in the on-premises organisation, typically in the form of an Export error to AzureAD.

Common errors seen on AzureAD accounts

  • The proxy address "" is already being used by "". Please choose another proxy address.
  • The value "" of property "UserPrincipalName" is used by another recipient object "". Please specify a unique value.
  • The value " GUID " of property "ExchangeGuid" is used by another recipient object. Please specify a unique value.

The value " GUID " of property "ArchiveGUID" is used by another recipient object. Please specify a unique value

Common export errors seen in directory sync

  • Unable to update this object because the following attributes associated with this object have values that may already be associated with another object in your local directory services: [ProxyAddresses <smtp:conflict address;]. Correct or remove the duplicate values in your local directory.

Remediation

Plan

Well to be able to solve an y conflicting issues between an on-premises Active Directory organisation and Office 365 (Azure AD + EXO) we need first to answer a couple of questions and depending on the answers we can implement an Action Plan:

  1. You must identify the account you want to fix and the account that is conflicting with it

In the AzureAD admin centre we have a filter view that can show only the active users that have an error and this should be the starting point of the investigation. If this filter doesn't help or it doesn't display the whole error then we would need to use Powershell to see the whole error:

(Get-MSOLUser -UserPrincipalName user@domain).errors.errorDetail.objectErrors.errorRecord.errorDescriptionWe should know now what attribute is actually the root cause of our conflict.

  1. Second step would be to delete or modify the identity that is causing the conflict and this action would raise additional questions like:

         a. Do I need to modify the "old" user or the new one...? 

          b. If the identities cannot be modified and we need to delete the "duplicate", can we backup the existing data associated with this account?

Well this article cannot answer these questions, but I have posted these here to raise awareness on we need to take into consideration before taking any actions in the attempt to solve the the issue.

You need to know that by default in O365 there is no backup, if we need to delete an account (to solve the conflicting issue) then all the data associated to it will be lost so we need to find a way to save this data.

Depending on the license model, in Exchange Online we can use Litigation Hold/ In-Place Hold to retain the content of a mailbox for as long as we need without keeping the user in the service. In other words we can put the mailbox on hold and then just dispose the user associated with it.

Another approach is to user the eDiscovery process that will enable an administrator to download the mailbox content into a PST file, available to download locally or to copy the content of the mailbox to a Discovery Mailbox on the tenant. This will ensure that we have a copy of the data in that mailbox and we can now safely delete the user and mailbox from the service.

I got this working, but there were a couple of gotchas...

1) Regardless of the different connector you choose to link the Access 2013 query to the pivot table in Excel (Jet OLE DB, Access 15.0 connector, ODBC, MS Query), your Access query can no longer include the 'Nz' function (nor I suspect any other customer Access functions). In my case, I rewrote my queries to use the Iif(isNull(x), <null value>, x) pattern. See http://dailydoseofexcel.com/archives/2008/01/10/cant-access-table-in-ms-query/

2) Excel 2013 has a new feature with Pivot Tables called Data Models - whereby you can link use more than one data source for a single pivot table. Pretty cool - except in my case (where I only wanted one source anyway), I discovered that checking the "Add to data model" (or however it is actually worded) in the create connector phase meant that grouping on date fields (if you want to show your data points grouped by month, quarter, year etc...) was no longer available. 3) Not really a gotcha, but I think I wound up using the Jet OLE DB connector (called "Office Data Connection" when you check the Connection settings in Excel 2013) to actually connect to my query in Access 2013.

4) Also, not a big deal, but I added the location of my Access 2013 database to the list of trusted locations within Excel 2013 to save having to "Enable" the Excel spreadsheet every time I opened it. You may not be so trusting of data on your own computer...
