jay freedman say:
XML. The custom XML approach is also good for the same kind of application. Each approach has strengths and weaknesses:

  • The DOCVARIABLE approach requires that you can display the userform more than once in the same document, to allow users to correct mistakes or update changed data. That's because there is no access from the user interface of Word itself to edit the values of document variables; only VBA code can do that. Your UserForm_Initialize code has to look for existing document variables and put their values into the appropriate controls on the userform, so the user only needs to edit the ones that have to change. The positive side of this is that the userform code has absolute control of what values can be put into the document. You can write verification code to ensure that all values are valid and that all necessary data has been entered.
  • The mapped content control approach allows the user to edit the data values directly in the content controls in the document, and the changes propagate automatically to the other controls that are mapped to the same XML node. On the positive side this seems more natural to users, but the negative side is that it's harder to write verification code and be sure that it runs. For example, you can write a class module that includes a Document_ContentControlOnExit event handler, and that code will check the content of a content control when the user moves the cursor out of the control. However, if the user never enters one of the controls, the code will never check it. You would also need a BeforeSave event handler, and maybe some others, to catch all the possible paths.

If you decide to use content controls, the way to have VBA code populate a control is something like this: 

        

Private Sub cmdOK_Click()
  Dim CClist As ContentControls
  Dim CC As ContentControl

  ' Validate free-entry Supplier box
  If Me.tbSupplier.Text = "" Then
    MsgBox "Supplier name is required", vbExclamation, "Blank entry"
    Me.tbSupplier.SetFocus
    Exit Sub
  End If

  Set CClist = ActiveDocument.SelectContentControlsByTitle("Supplier")
  For Each CC In CClist
    CC.Range.Text = Me.tbSupplier.Text
  Next CC

  Set CClist = ActiveDocument.SelectContentControlsByTitle("Entity")
  For Each CC In CClist
    CC.Range.Text = Me.lbEntity.Value
  Next CC

  Me.Hide
End Sub

This requires that each content control in the document has a Title value assigned to it in its Properties dialog. Alternatively, you could assign each one a Tag value, and use the ActiveDocument.SelectContentControlsByTag function. For both functions, the argument is case-sensitive.

If the content controls are mapped to custom XML, then you need only assign a value from the userform to the first control in the CClist, as in

   ActiveDocument.SelectContentControlsByTitle("Supplier")(1).Range.Text = Me.tbSupplier.Text

If you're going to use mapped content controls, get Greg Maxey's add-in from his page at Content Control Tools. It makes dealing with the mapping much easier.

john - project say:

Danbet99,

Julie provided an excellent answer to your basic questions but allow me to add a bit of caution. Linked structures, (e.g. master/subproject, inter-project links, resource pool/sharer files), are fragile and prone to corruption. It sounds like you can do a static master to integrate your upper level milestone file with each schedule plan, and as Julie noted, you will have to create a new static master each update period but that can be easily automated with a recorded macro. The default for inserted projects is to create a dynamic master. To create a static master, uncheck the "link to project" option in the lower right corner of the Insert Project window.

If you want/need links between tasks in one project with tasks in another project you will need to use external predecessors/successors. There is no "static" equivalent to those so you will have to enter the arena of a linked structure. That can be done successfully if you follow some rules: Never move, rename, overwrite, or save off any of the files in the structure. All files should reside in a single folder on a local drive. Do not operate the structure over a network.

Hope this helps.
