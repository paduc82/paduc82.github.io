Bad interaction between Outlook PST files and Windows 8/8.1 File - Microsoft Community

jedisql say:

Bad interaction between Outlook PST files and Windows 8/8.1 File History

I use auto archiving in Microsoft Outlook.  Every 3 to 4 years I rename the archive PST file with a date range and start with a fresh, empty PST file for auto archive.  With 3 to 4 years of email in them, my old email PST files have sizes measured in gigabytes.  My problem is that the way Outlook uses PST files makes it impossible to have a reasonable backup strategy.


It is my intent that my old email PST files never change.  If they are allowed to change, I could accidentally delete items from them or accidentally move items into them that don't belong.  I would really prefer to mark them read-only and burn an optical disc backup so that the backup process is permanently over and done with (except for maybe reburning the discs every 5 years).  However, Outlook requires that PST files be mounted in a writable mode so that Outlook can use the file as a scratch pad for sorting, searching, etc.


The requirement that PST files be mounted in Read-Write mode presents two problems.
(1) My archives are always subject change when I really, really want them to be protected absolutely from change.
(2) They become a backup nightmare.


Almost all backup solutions, including Windows File History in Windows 8 and Windows 8.1, just look at files and check if one bit in the file has changed.  If so, then the entire file is backed up.  Well, with writable PTS files in Outlook, I have several multi-gigabyte files that change by a few bytes each day.  The result is that Windows File History quickly fills up my backup media with almost-entirely redundant backups of these huge PST files.  I have to back them up continuously because Outlook forces me to keep them in danger of modification.

I see two options for this.

(1) Outlook allows PST files to be mounted read-only.  In this case it creates a partner file that is used for the scratch pad area.  The scratch pad files should NOT be required for later re-mount so that users don't have to keep track of them.  This way one can make a permanent backup copy of the file, and Windows File History will see the huge PST file never change, and will make only one backup copy of the file on its backup media.
(2) Windows File History implements a differential/incremental mode for huge files.  That way Windows File History will write only the constantly occurring small changes in archive PST files to its backup media.  This will save a lot of disk space.  Currently I have a weekly reminder to go to my backup media external hard drive to manually delete the older copies of my PST archive files.  That disk has 4.2 GB of non-PST backup files, and with all but the one most recent copy of each PST file deleted, those PST files take up 8.1 GB of disk space.  I have a weekly reminder to delete all but the one most recent copy of each PST file.  I calculate that with Windows File History set to 6 month retention, and not manually delete old PST copies, I would need a 3 terabyte external hard drive to backup my computer's hard drive that has only 83 GB of data on it (including the Windows and Program Files folders that are not backed up).  Doesn't that seem a bit impractical?

I have also posted this under Category "Windows" as "Bad interaction between Windows 8/8.1 File History and Outlook PST files".

tom van stiphout say:

You could add a field to tblEmployees named AreaCode, and at startup time after Login lookup the area code and save in a TempVar. Then you can use that TempVar in any Customer-related query.

One way to do that, is first to create qryCustomers, and have it filter for the user's area code. Then *any* customer related queries you create are based on qryCustomers, *not* on the underlying table.

niranjan i k say:

Turn off the “Auto Arrange” option for icons on the desktop, and then place icons on the desktop where you want them.

To turnoff the “Auto Arrange” feature,

For windows 7:

Right click a blank area on the desktop and select View and uncheck Auto arrange icons.

For Windows XP:

Right-click a blank area on the desktop, and then click Arrange Icons. Click the command that indicates how you want to arrange the icons (by Name, by Type, and so on), uncheck Auto Arrange.
