john w. vinson say: Finite computers cannot handle infinite numbers, so the answer is You Can't. Adding one new record a second, day in, day out, no breaks, no holidays; and using the fact that an autonumber will "wrap" from 2^31-1 to -2^31 and then start counting up toward zero, you get 4294967296 entries. Adding one record per second, starting now, running 24/7, no weekends or holidays, you will run out sometime during the afternoon of February 24, 2147. If that's not good enough, you can use a Currency field and use VBA code to increment it by 0.0001 (its limit of resolution). That will let you handle some quadrillions of records.

john w. vinson say:

Well, there is a possible scenario whereby one could run out of autonumbers.

Consider a database with a local table with an autonumber ID, linked to some other database with a large table. Periodically - frequently let's say - you have to run an Append query appending a subset of records from the large table into the local table. In my experience, Access will SOMETIMES - not always - grab enough autonumbers to accommodate all the records in the source table, ignoring the criteria. This is one of the ways you can get big gaps in autonumber sequences.

Do this repeatedly, appending (and perhaps later deleting) ten or twelve records from a 10,000,000 row SQL/Server source table, and you could end up with a very sparse set of autonumbers with the "next number" exceeding the 4 billion limit, without ever having more than a dozen records in the table at any one time.

John W. Vinson/MVP

Sub CustomFormatCells2()
Const FirstR As Long = 1   '<<< data starts from row 1, change
Const ISOcol As String = "A"  '<<< ISO in column A, change
Dim r As Long
r = Cells(Rows.Count, ISOcol).End(xlUp).Row
For Each rr In Range(ISOcol & FirstR & ":" & ISOcol & r)
Select Case Trim(rr.Value)

Case Is = "USD"
rr.Offset(, 2).Value = rr.Offset(, 1).Value
rr.Offset(, 2).NumberFormat = """USD"" #,##0.00"
Case Is = "BDT"
rr.Offset(, 2).Value = rr.Offset(, 1).Value

dianegaleone8469 say:

There is a way to find out what folder it delivered to.

In powershell, first get the messagetraceID.
Then get the messagetracedetails on that messagetraceID

Example:

Get-MessageTrace -RecipientAddress *** Email address is removed for privacy *** | select senderaddress, subject, messagetraceid

Gives me the relevant messages. I find the one I am looking for in the list:  (I picked this one because I knew it would be subject to an inbox rule)

Then get the messagetracedetails on that messagetraceID

get-messagetracedetail -MessageTraceId d4f25e3b-f614-4eb2-f6d0-98d2cd8d6b28 -RecipientAddress *** Email address is removed for privacy *** | select messagetraceid, date, event, detail

The output is this:

MessageTraceId   : d4f25e3b-f614-4eb2-f6d0-08d2cd8d6b28
DefaultFolderType:DeletedItems


If that's happening to your end user, have a look at his inbox rules. That's the most likely culprit.
