Is the date column a candidate key of the parent form's table?  A linked subform will normally reflect a one-to-many relationship type between the parent form's table and the subform's table.  In each case the relevant table can be a referencing table if the form or subform's RecordSource is a query which joins two or more tables.  The LinkMasterFields property will normally reference the primary key of the parent form's table and the LinkChildFields property will reference the corresponding foreign key of the subform's table.

 

In cases where the primary and foreign keys are multi-column. Both columns can be referenced in the properties, separated by a semi colon.

ken sheridan say: I see no reason why the form/subform should not work as expected, but as PointSystemMain has only the one column you can achieve the same result without a subform, by means of a form, named frmPointSystemSub, with the following as its RecordSource property:

PARAMETERS Forms!frmPointSystemSub!cboProductionDate DATETIME;
SELECT *
FROM PointSystemSub
WHERE ProductionDate = Forms!frmPointSystemSub!cboProductionDate
ORDER BY TimeStamp;

In the form header add an unbound combo box named cboProductionDate, with the following as its RowSource property:

SELECT ProductionDate
FROM PointSystemMain
ORDER BY ProductionDate;

In the combo box's AfterUpdate event procedure put:

   Me.Requery

This reloads the form's recordset which is then restricted to those rows which match the date selected in the combo box.

ken sheridan say: Reading between the lines of your last post I suspect that the problem lies with the RecordSource properties of the parent and subform respectively.  My guess is that the latter includes the PointSystemMain table in a query.  The correct setup, using a subform, is for PointSystemMain to be the RecordSource of the Parent form and PointSystemSub to be the RecordSource  of the subform.  The LinkMasterFileds and LinkChildField properties of the subform control would then both be ProductionDate.

To navigate to a production date the combo box, cboProductionDate, in the parent form should be unbound, and have the following code in its AfterUpdate event procedure:

Dim strCriteria As String

strCriteria = "ProductionDate = #" & Format(Me.cboProductionDate,"yyyy-mm-dd") & "#"

With Me.RecordsetClone
    .FindFirst strCriteria
    If Not .NoMatch Then
         Me.BookMark = .Bookmark
    End If
End With

If you also wish to enter a new production date in the parent form into a new empty record in PointSystemMain this should be done by means of a bound text box.  

However, as I pointed out in my last post, it is unnecessary to have a subform at all.  You can simply base a form on a query on PointSystemSub  as its RecordSource which references an unbound control as a parameter.  I gave you detailed directions of how to do this in my last post.  A new row can be inserted into the PointSystemMain table where necessary by means of code in the unbound combo box's NotInList event procedure.

Another alternative to a subform would be to have a form with the PointSystemSub table as its RecordSource property, and to filter the form in the AfterUpdate event procedure of an unbound combo box, set up as described in my last post, with the following code:

Dim strFilter As String

If Not IsNull(Me.cboProductionDate) Then
    strFilter = "ProductionDate = #" & Format(Me.cboProductionDate,"yyyy-mm-dd") & "#"
    Me.Filter = strFilter
    Me.FilterOn = True
Else
    Me.FilterOn = False
End If

Note how the ISO standard for date notation of YYYY-MM-DD is used above to ensure international unambiguity.  Without this the code would fail on systems, like mine, where the system date format differs from the US date format of mm/dd/yyyy.

In the template from which you create your documents, check the language set in the Footnote Text style. If it is set to US English, your footnotes will be US English regardless of your system settings. Here are the steps:

  1. Open the template for your document. If you're scratching your head saying "Template?", then you're probably using the default Normal.dotm. To open it, choose File>Open, then navigate to Users/YourUserName/Library/Application Support/Microsot/Office/User Templates and open Normal.dotm.

  3. In the Style dialog, change the List: dropdown to All Styles.
  4. Find Footnote Text and select it.
  5. Click on the Modify button.
  6. In the Modify Style dialog, click on the Format button and choose Language. If it's set to English (US), change it to English (AUS). OK out, save and close the template.
