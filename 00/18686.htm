By default, placeholders are set to Shrink text on overflow, which creates the effect you're seeing. To change this setting for all placeholders in a presentation:

  1. Choose View>Slide Master.
  2. In the left-hand thumbnail pane, scroll up to select the larger slide at the top (this is the actual master slide).
  3. Select the Title and Text placeholders, then right-click on the border of one and choose Format Object. The Format Shape pane opens.
  4. Click on Text Options, then on the Textbox icon.
  5. Change Shrink text on overflow to Do Not Autofit or Resize shape to fit text.
  6. Close the slide master and check the result in your slides.

Limit Choices in Drop Down List in Access 2007 - Microsoft Community

trentrondot say:

Limit Choices in Drop Down List in Access 2007

I need to create a simple form that has two drop down lists...one for states and one for cities.  For instance, if I select "Texas" in the State field, I want only the cities available in the City drop down list to be limited to those within Texas.  

scottgem say:
This is a standard technique called Cascading or Synchronized comoboxes. A search on either term will come up with many articles on how to do them.

Basically, you need a table of States, and a table of cities that include the state they are in. You then set the RowSource of the City combo to reference the state combo Form example:

SELECT City FROM tblCityState
WHERE State = Forms!formname!cboState
ORDER BY City;



trentrondot say: I have never worked with code before...does this happen in the Table Design Row Source Line, or within a Form Design?  I was setting up all the tables first and had not created forms yet.

Thanks

scottgem say:
Who said anything about code? 

I do not recommend setting up lookup fields on the table level. Nor can you do Cascading combos on the table level. They HAVE to be done using forms. They are done by building a SQL statement in the RowSource of your combos, which can be done using Query Design mode.

The only code that might be needed is to requery the City combo after the State is selected. And that's one line:

Me.cboCity.Requery

which would be put in the After Update event of the State combo.

ken sheridan say: A couple of important points to note:

1.  City names should not be used as keys as they can legitimately be duplicated.  There are 28 Springfields in the USA for instance, and to make matters worse five of them are in Wisconsin, ruling out a two-column key.  As with personal names, towns or cities should be distinctly identified by a numeric primary key, CityID say.

2.  A table of locations should include a foreign key CityID to identify the city, but no foreign key column to identify the state.  State is functionally determined by city so to include a column for the state would introduce redundancy, and the table would not be normalized to Third Normal Form (3NF), leaving it open to update anomalies resulting in inconsistent data.

You can still do what you are aiming at in the context of a correctly normalized set of tables by using an unbound combo box for the state, which restricts the RowSource of the bound CityID combo box.  In a form in continuous forms view, however.  The solution is to use a hybrid control for the state, composed of a bound text box superimposed on an unbound combo box so that it appears to the user as a single combo box control, basing the form on a query which includes the Cities and States tables in addition to the referencing table into which data is being entered.  You'll find examples of the use of correlated combo boxes like this in ComboDemo.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This demo file uses the local administrative areas of county, district and civil parish in my neck of the woods, but the principle is exactly the same whatever the scale.  As you are dealing with a two-level hierarchy rather than three-level, the form in the demo illustrating a simple two-level hierarchy would be an appropriate model in your case.
