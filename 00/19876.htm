    First use the Windows App uninstaller or a third party uninstaller app (i.e. ccleaner) to search for and uninstall HP Mobil Broadband Drivers

    Then reboot your PC and go into Device Manager and uninstall any drivers that still may be left in there for the hs2340

    When uninstalling drivers from Device Manager, make sure you also tick off the box: Remove driver as well.

b) Reboot your PC after every attempt to clean out old rubbish. 
    Then uninstall the HP Hotkey Support App. This is incompatible with the hs2340. You need to use the old Quick Launch button instead (or none at all if you don't care to activate quick launch buttons)

d) Uninstall the HP Connection Manager Software 

2. Download (but do not install) the correct driver from the HP Softpaq library: sp52715

a) Follow the description found here on how to disable the compatibility assistant completely. (remember to reverse this procedure after you are finished.

HI,

That sounds like a grounding issue.  The touch of your finger changes the capacitance of the location of the screen.  If the phone is not properly grounded, that will interfere with how the screen can sense your finger.

The first thing to check would be the USB cable.  If it has a kink or has an internal fault, that code cause the problem.  If the USB port or the computer itself is not properly grounded, that may cause an issue.  If you have the computer plugged into a faulty power strip, that could cause the problem.

You can verify this by plugging the cable into into the phone and into the charger that comes with the phone.  if the problem still exists, then you know it's not the PC and it's either the cable or the outlet.  Try replacing the USB cable and see it goes away.

If none of that works, you may have a fault in the USB port on the phone.  Either it's damaged or you may have some material stuck in the port causing a short.  In the case , get a can of compressed air and trying blowing out the port.  Otherwise take the phone to be serviced.

petrib say:

The SIM (Subscriber Identity Module) card usually has two PIN (Personal Identification Number) codes; a primary PIN code and a secondary PIN2 code.

 

The PIN is used to unlock/use the SIM card. The PIN2 can be used for additional restrictions (e.g., fixed dialing limits your calling to a specific set of numbers, but with the PIN2 you can bypass that, if you know it).

 

The SIM card has also, so called, PUK codes (Personal Unblocking Key), which can be used to reset a forgotten PIN code. For PIN2 there's a PUK2, too. You get the PIN/PIN2 from the network operator (carrier, service provider) from whom you get the SIM card from. If you need the PUK/PUK2 codes, you get also those from your operator. By default, the operator usually set the SIM card's PIN/PIN2 codes always to the same default values (usually 0000 or 1234, but it could really be anything they decide to use). The PUK/PUK2 codes are always unique and SIM dependent (i.e., one SIM card's PUK will not work with another SIM card from the same or any other operator).

 

The phone also has a security/lock code, which is irrespective of the PIN and PUK codes, as it is phone-specific, and not related to the SIM card at all. So far, to my experience, all Nokia phones have had 12345 as the default security/lock code.

 

The security/lock code you can use to lock the screen, or prevent changing of SIM cards (e.g., if someone steals or finds your lost phone, when they try to replace the SIM card, they need to know and enter the security/lock code, if you have made the phone to require it).

 

Installabe memory cards have also a password that you can set. If you do, then the memory card cannot be used in another device, unless the password is known.

 

And if you use Bluetooth radio to connect ("pair") two Bluetooth devices, then a Bluetooth security/PIN code is also required to set up the connection (it is used to generate encryption/decryption keys that the devices use to "scramble" the data between themselves).

ken sheridan say: Firstly, the use of the lookup field wizard is generally deprecated by experienced developers.  For reasons why see: http://www.mvps.org/access/lookupfields.htm

Secondly, a value list should only be used as the RowSource for a combo box (which is what a lookup field uses as its display control) for a set of values which is immutably fixed in the external world, e.g. days of the week or weeks of the year.  Otherwise the values are data and should be stored in a referenced table.  A fundamental principle of the database relational model is the Information Principle (Codd's Rule #1).  This requires that all data be stored as values at column positions in rows in tables, and in no other way.

You can do what you wish, however by concatenating the value of the text column from the referenced table.  Whether this can be done in a calculated field in the table, I can't say as I'm using Access 2007.  The usual way to do this would be in a computed column in a query which joins the two tables, e.g.

SELECT <field list from referencing table>,
Category & " " & [Customer Name] AS Category_Customer
FROM [Your Referencing Table] INNER JOIN Company
ON [Your Referencing Table].Customer = Company.CompanyID;

where CompanyID is the primary key of Company referenced by the foreign key Customer column in your referencing table.
