converting MBR2GPT to UEFI - Microsoft Community

smitsagm alex say:

converting MBR2GPT to UEFI

Mijn BIOS=outdated&I wanted to replace-it by the new UEFI,runned all the scriptcmdlets without succes ,can any one of you explain the proces?

don cuthbert say:

There is a new utility (MBR2GPT.EXE) in Windows 10 version 1703 that will do this for you, assuming of course that you can set your computer BIOS mode to UEFI. If you do not have UEFI mode in BIOS, you will not be able to boot from a GPT-initialized disk (which is required for UEFI).

Here is a link to info on how to use the new utility:

https://technet.microsoft.com/en-us/itpro/windows/deploy/mbr-to-gpt

So the bottom line is that if your BIOS cannot switch to UEFI mode, running this process will only make your disk not bootable. If you have done this, you would have to switch the disk back to MBR-initialized. I don't think Microsoft provides any means to switch from GPT to MBR without first wiping all partitions from the hard disk. I believe that it can be done by booting with a Ubuntu Live USB or DVD and running Gparted or there may be other utilities available online to do this.

I have some older computers that also cannot switch to UEFI BIOS mode, but are still able to run the latest Windows 10 version 1703.

shane devenshire say:

Lets suppose the balances are in column C between C2:C100

Private Sub Worksheet_Change(ByVal Target As Range)
    Dim cell As Range
        For Each cell In Range("C2:C100")
            If cell <> "" Then
                If cell = 0 Then cell.EntireRow.Hidden = True
            End If
        Next cell
End Sub

You can unhide the rows manually.  Or a similar macro with Hidden = False

This will sum column B for a batsman in column A even if there is a * at the end of the number. The batsman to sum is in D1 and note it will not handle blank cells in column B

albert d. kallal say:

In the case of Access, since it is NOT a document like Excel, or word, the data part can be opened by SEVERAL people at the same time. In fact this is how (and why) Access allows say 5 people in your office to SHARE a multi-user database system.

So if you open Access two times, the tables remain in the file on the disk drive, and are not pulled into memory. This design is on purpose so as to allow multiple users to share and edit data and thus use the ‚Äúsame‚Äù information system.

So if you place the database in a shared folder, then multiple users can open the application and edit the same data. And if you open two copies on your desktop, then again you get the same effect.

So the data part of the application is NOT limited to each user that opens the application ‚Äì but is shared! And in your case if you open the same file two times then you have the same effect. If you want to separate the tables, then you need to make a copy.

So the current design allows you launch the application, and start editing a customer invoice. And then you can launch another copy and edit a DIFFERENT invoice. However, the data and tables is common to both (since you are directly editing the data).

Without this shared ability, you could never build a multi-user system that allows several people in your company to share and edit a copy database. So this is very much the ‚Äúintention‚Äù of a database system ‚Äì it allows many people to edit and use the application ‚Äì they are document based).

What you can do however is split the database. So tables that are (or should apply for each user) thus can be deleted etc. So you can isolate tables to each user (or each copy you launch). And tables in which you wish to SHARE data or have common to both copies open (or many users who run the application) will thus edit and share ONE set of common tables.

If several users (or you open two copies ‚Äì the same effect), then any changes to the tables are thus shared and seen by all users. As noted, this is by design to allow a common depository of data and information. So Access is NOT like word or Excel in that each user gets a copy of the data. In access, each user opening the same application thus SHARES the data common.

So if you delete a table, then all users will see the table as deleted and gone.

You can however get around the above by splitting your application. You then place the tables that are NOT supposed to be common and shared in that front end.

I explain the how, the why, and how the splitting concept works in this article:

http://www.kallal.ca/Articles/split/index.htm

Once you read the above, and learn this split concept, then you also have learned how to create a multi-user application for your company that allows everyone to edit the same data. It is how one can build a information system, and allow sharing of data at the same time.
