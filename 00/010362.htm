jcanup say:

I want to create a word document with a Recipe Layout that pulls in one recipe per page and populates the layout with that recipe's XML data. These are new recipes that are being created, tested and modified several times by many different people, so it is constantly in flux.  I just want to point the Word document to the XML file and have MS Word do the work. 

a. user say:

There are various ways you could consider getting your data from your XML file into a Word document, but only one or two of them can be done without programming using either XSL, or using (say) VBA, VB.NET or C# to interpret the data in your XML file and generate the document that you want.

In fact the only scenario where you may be able to get away without coding of some sort is the one where your XML file is already structured rather like a Word Mail Merge Data Source - in other words, it is effectively composed of a number of "records" (perhaps one for each Recipe), each of which has a fixed number of "columns". For example, suppose your XML just consists of one XML Recipe element for each recipe, and within each Recipe element you might have XML Elements or Attributes for (say) Title, Authors, Description, Ingredients, Method and a Photo element that contains a link to a .jpg. In that case the main remaining problem is that Word can't use a .XML file as a data source. Personally, I would try to open it in Excel, and see if I could use Excel to convert it into a standard data source with one Worksheet row per Recipe and columns for Title, Authors etc. If that does not work, then next thing I would (probably) try would be to write a piece of VBA or XSL that converted the XML into an HTML or Word table which could then be used as a data source by Word. Then, you could create a Mail Merge main document and merge the data in the usual way.

However, if your .xml is a little more complex - for example, you have a list of ingredients but it's structured like this: <Ingredients><Ingredient><Description/><Quantity/></Ingredient></Ingredients>, and you have a list of steps with one Element per step, trying to turn it into a Word Mail Merge Data Source might be much more difficult. What to do in that case would depend on what programming resources are available to you. You could for example
 a. Write VBA code that stepped through the elements of the XML file, using the Word object model to create the document you want from scratch
 b. As (a), but use C# or VB.NET to do the same thing
 c. As (b), but use Microsoft's Office Open XML SDK to generate the document
 d. use XSL to generate either a complete Word 2003 XML format document, or a complete Word 2007-and-later OOXML format document (in this case you would generate the Flat OPC format, a single-file representation of a .docx file). FWIW there is a feature in Word (not sure it is still there in Word 2016) that lets you specify a .xsl transform when opening a .xml document. Word then applies the transform, so you actually end up with a complete Word document.

(a) and (b) require understanding of the Word Object model and how to use it to generate a complete document. (c) is similar, but IMO if you were starting from scratch, is a significantly tougher proposition. (d) wouldn't necessarily involve "difficult" XSL, but you would have to understand how Word uses XML to represent every structure you need to use. For example, if you want your document to contain tables, you have to know how a Word table is represented in WordML. In my view, that is a *lot* harder to get to grips with than using the Word object model to generate a document, partly because debugging XSL is very difficult IMO. But maybe you have someone with exactly the right skills?

In my view, if you have some XSL and WordML resource available, I think there is something to be said for combining the Mail Merge and the XSL approaches as follows. This approach is workable as long as
 a. your XML file is structured as a set of top-level elements (say "<Recipe/>") that basically correspoond to the chunks you want to insert in your Word document
 b. Everything related to a Recipe element is contained within the element (not all XML documents are organised in that way
 c. Each <Recipe/> element has a unique ID.

In that case, what you can do on Windows Word is
 d. write one XSL transform that generates a Word Data Source with one column, containing a list of the Recipe IDs.

 c. Create a Word Mail Merge Main Document that defines the general layout you want, and contains one or more INCLUDETEXT fields to bring in the data from the XML file. For XML files, the INCLUDETEXT field lets you specify
    1. An XML source file
    2. A piece of XPath code that specifies a single Element in the XML file (such as a <Recipe/> Element)
    3. An XSL file that can transform all, or selected, data in that Element into WordML

(For any mailmerge experts reading this, this is a very similar approach to the one where you use a DATABASE field to insert a table of data related to the current record in the Mail Merge Data source, except that the INCLUDETEXT field and XSL lets you insert anything you want into the body of the document - as long as you can work out the WordML to do it).
