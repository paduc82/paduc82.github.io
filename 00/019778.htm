Mailboxes - Mailboxes - Sync Anchor Values - Microsoft Community

alex rizeanu msft say:

Mailboxes - Mailboxes - Sync Anchor Values

Technical Level : Expert

Summary

All account provisioning in Exchange Online is built off on synchronization between AzureAD and Exchange Online. At a minimum, there are two completely separate directory services involved with account creation and maintenance. For example, all user, group, and contact account objects in Exchange Online are synchronized with a copy of that object in AzureAD. When admins choose to provision their accounts by using their own private On-Premises Active Directory, a third directory service is introduced. Each copy of the account in its respective directory must be in sync with the other copies across all directories. When that does not occur, you run into a plethora of issues and unexpected behavior.

In order to achieve a consistent sync state, each object must be tied to the other through a consistent linkage that doesn't change. This is referred to as the anchor attribute. If the anchor attributes don't match between accounts, then the accounts are not synchronizing and are considered to be completely separate from each other, no matter if they share other values like email address or UPN.

Note: Accounts that do not share matching anchors but do share other identifier attributes like email address, UPN or ExchangeGUID are considered to be in conflict. When conflicts are detected during synchronization, all further processing is blocked for the offending account until the situation is resolved - almost always with manual intervention.

Details

Applicable Scenarios

Each of the following scenarios depends on functioning synchronization of the account in question.

  • Changing a synchronized attribute value
    • Example: Add a new email address to a user
  • Creating a new account
  • Removing an existing account
  • Licensing an account to provision a mailbox
  • Removing a license to de-provision a mailbox

Recovering deleted accounts

Comparison

The anchor value pair between On-Premises and AzureAD is objectGUID and immutableID.

  • The On-Premises objectGUID is represented as a GUID in most tools, including AzureAD Connect.
  • The AzureAD immutableID is the On-Premises ObjectGuid encoded in base64. The two values are actually the same.
  • You can use powershell to convert the values back and forth to verify a matching pair.
    • Convert a GUID to an ImmutableID 

    • [system.convert]::ToBase64String((new-object system.guid("bdca97da-65da-47d4-bec0-06abd4d0a057")).ToByteArray())

    • Convert an ImmutableID to a GUID 

    • new-object -TypeName system.guid -ArgumentList (,[system.convert]::FromBase64String("2pfKvdpl1Ee+wAar1NCgVw=="))

The anchor value pair between AzureAD and ExchangeOnline is objectID and ExternalDirectoryObjectID.

  • These two values are always represented as a GUID, they are exactly the same as well.

The table below gives an example of how a directory synchronized object is linked through the directories using these attributes and how a cloud-only object is linked.

Note: The following are example values, the actual values for objects you will be validating will be different.

Wrong font for return address - Microsoft Community

ritaoliver say:

Wrong font for return address

When I create addresses for envelopes in Word 2016, I can use the font I wish in the delivery address.  When I chose the same font for the return address, it does not print that font, it reverts to the default font.  How can I get the return address to print the font I have chosen for it?

[Original title: Rita]

doug robbins - mvp office apps & services (word) say:

There has been no change since at least Word 2007.   The procedure that Suzanne gave you was the way to make the change permanent so that all Envelopes would be formatted in that way.

If you only want to make the change on a one-off basis, you can either select the text in the Return Address area of the Envelopes tab of the Envelopes and Labels dialog and the press the right mouse button and select Font, which will cause the Font dialog to appear,

or,

you can click on the Add to Document button and then select the text and use the facilities on the Home tab of the ribbon to change the font appearance.

suzanne s. barnhill say:
You need to modify the Envelope Return style, which is used for the return address. Press Ctrl+Shift+S to open the Apply Styles dialog. Type "Envelope Return" and then click Modify. In the Modify Style dialog, change the font as desired, select the radio button for "New documents based on this template" (assuming you want this as the default for all envelopes), and click OK.
