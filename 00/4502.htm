ken sheridan say:
In addition, should I be including the FilterName parameter (which would contain the same query as the report's Record Source), or is that redundant?

If you are filtering a report by means of the WhereCondition argument of the OpenReport method you should omit the FilterName argument.  The latter is an alternative method of filtering a report on the basis of the restricted result set of a query specified as this argument, not the same query as the report's RecordSource.

An alternative is to change the value of the report's RecordSource property in its Open event procedure by passing the SQL statement as a string expression into the report by means of the OpenArgs argument of the OpenReport method.  This is one of the approaches illustrated in MultiSelect.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo file illustrates various means of restricting a report's output on the basis of values entered/selected in a dialogue form.
