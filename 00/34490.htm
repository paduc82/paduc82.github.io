Problem with access to remote Samba share from Windows 10 machine - Microsoft Community

grzegorzmwr say:

Problem with access to remote Samba share from Windows 10 machine

Hi all Windows gurus,

This subject appeared from time to time on this forum but my case is a bit different.

I am using Windows 10 Home edition and since installing the KB3193494 update I cannot access an SMB share on remote Unix server. Immediately after typing the share name into File Explorer a message appears "...A specified logon session does not exist. It may already have been terminated".

For sure it is the matter of the update, because after uninstalling it I do can access the share again. Sadly, Windows 10 installs updates automatically and after some time when the KB3193494 is installed back, I can't access the share.

Interestingly, this symptom applies to Windows 10 Home edition only. On Pro and Enterprise editions I do can access the share without any problems. I verified that on several computers running above Windows editions.

I revealed that the culprit is the Security Signatures Required bit in the SMB request header that Windows sends in the first packet when connecting to the remote server. On Windows 10 Home (where the problem occurs) this bit is set to 1, but on Pro and Enterprise (where there are no problems) is set to 0. This is the only difference in communication between the remote server and different Windows editions. See the following Wireshark decode:

On Microsoft support page I learned that this bit depends on the registry key value HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Services\LanmanWorkstation\Parameters\RequireSecuritySignature. When the value of this key is 0 then the above mentioned bit in the SMB request header is set to 0. Similarly if the key value is 1 then the bit is 1.

And here is the point: this rule works on Pro and Enterprise, but on Home is DOESN'T! Regardless of the RequireSecuritySignature value, Windows Home edition ALWAYS sets the Security Signatures Required bit to 1. I played with the above registry setting on Pro and Enterprise and these editions set request header bit to the proper value as expected. Moreover, when on Pro or Enterprise this registry value is 1 (by default it is 0), then I cannot access the share and getting the same message as on Windows Home. After reverting the registry value back to 0 (on Pro and Enterprise of course) the share is accessible again.

Summarizing, my questions are:

1. Why Windows Pro and Enterprise allow to control the Security Signatures Required bit in the SMB request header with the RequireSecuritySignature registry value, but Windows Home does not?

2. How to force Windows 10 Home to set the Security Signatures Required bit to 0 to allow access to the share?

Any hints will be greatly appreciated.

Grzegorz
