    travelling gal say:

    Problem with filter in Word 2010 Mail Merge

    I have a main document with associated Excel database.

    I click on "Edit Recipient List," and create this filter:

    and Field2 is blank

    I click on OK.

    and Field2 is blank

    This effectively eliminates the restriction in Field1. I clear the filter and start over. Same result every time.  I try reversing the order of the two filters, but it still adds extra "or" conditions that mess up the result. The filter only works properly if you are filtering just one field at a time.

    How can this problem be fixed?






    travelling gal say:
    this is a long-standing bug in the Word mailmerge filters when filtering for blanks. See, for example: http://support.microsoft.com/kb/286880. The bug raises its head as soon as you have an OLE DB connection to a data source and specify "is blank" or "is not blank" as one of the "Advanced criteria". A possible workaround in this case would be to use a different data connection method (e.g. DDE).

    Aha! Then I have an extremely simple workaround:

    I can edit my .xlsx database, entering a hyphen (or any other letter or symbol) in each of the blank cells in my "Field2" column. I will avoid blank cells in all new databases in the same manner. Now my search criteria is as follows:

            Field1 = Y
    AND Field2 = -

    Works every time! Yaaaaaaaay!!


    P.S. Add to the list of specifications that do not work: "Not equal to." Surprisingly, "Does not contain" does work.

    I was wrong when I said filtering on more than one field created the problem. Filtering with any of these three specs (with one or more filters) will create the problem.


    P.P.S. I'm still fooling around with these files, and I have now found an even simpler workaround:

    Without changing anything in either the .xlsx database or the Word main document, I have discovered that if I open the Mail Merge Helper and use its Query Options for my two-field filter, it opens a window that is identical to the one that opens when I click Filter in the Mail Merge Recipients window.     macropod say:

    Do the lines end with a "soft return", ie <SHF><ENTER>.  In that case go in to File menu > Options command > Advanced option > (scroll down to) Layout options for: section. In there turn on "Don't Expand character spaces on a line that ends with Shift + Return".

    There is another possiblity. There is a 5th form of justification that is not available through the user interface. It is only applied using a keyboard shortcut. There is a slight chance you may have accidentally turned it on using the <CTL><SHF><J> shortcut. 

    Lines ending with manual lines breaks (what you call a "soft return") cannot be justified; neither do they work in bibliography fields since, as with any other field, they'd have to be typed manually and would be deleted immediately the field is refreshed, so clearly that's not the issue. Furthermore, it's perfectly clear the lines are being justified.

    Few would regard left-aligned, right-aligned or centred text as 'justified' Certainly Microsoft doesn't call it that!

    As for the Ctrl-Shift-J shortcut, Microsoft calls that the distributed text function (i.e. not justification) and pads the space between all characters, not just between words. Unlike justification, it does not decrease the space between words (where appropriate).
