ken sheridan say: Firstly, ignore the basConcat module; it is not relevant to your situation, and you don't need it unless you want to concatenate the address into a single string in a report as in my demo.

My ParentActivities demo is just a simple illustration of a basic binary (2-way) many-to-many relationship type and how to represent it in a form, subform.  A many-to-many relationship type is modelled by a table which resolves it into two one-to-many relationship types.  So in my demo the model is:

Prarents----<ParentActivities>----Activities

In your case the CustomerAddresses table models a many-to-many relationship type between Customers and AddressTypes, so the model is:

Customers----<CustomerAddresses>----AddressTypes

The columns CustomerID and AddressTypeID in CustomerAddresses are each a foreign key, referencing the primary keys of Customers and AddressTypes respectively, i.e. the table has resolved the many-to-many relationship type between Customers and AddressTypes into two one-to-many relationship types.

The Customers table would have columns for the basic customer data such as the customer name etc.  The AddressTypes table would have, in addition to the AddressTypeID primary key column, a column of text data type for the address type, e.g. billing, warehouse etc.  The CustomerAddresses table, in addition to the AddressTypeID and CustomerID foreign key columns would have columns for the various address lines.

For data entry you would have a form in single form view based on Customers, and within this a subform based on CustomerAddresses, as in my demo.  The form and subform would be linked on CustomerID.  Whereas my demo has only the one combo box in the subform in which to enter data, yours would have text boxes for all the address lines, along with a combo box in which to select the address type.  The subform can be in continuous forms view (as in my demo) or in single form view if preferred.  The address type combo box would be set up as follows:

RowSource:     SELECT AddressTypeID, AddressType FROM AddressType ORDER BY AddressType;

BoundColumn:   1
ColumnCount:    2
ColumnWidths:  0cm

If your units of measurement are imperial rather than metric Access will automatically convert the unit of the last one to inches.  The important thing is that the dimension is zero to hide the first column.

To enter a new address type not currently listed via the combo box you can put code in its NotInList event procedure, as with the activities combo box in the subform in my demo.  In your case the code would be:

    Dim ctrl As Control
    Dim strSQL As String, strMessage As String
    
    Set ctrl = Me.ActiveControl
    strMessage = "Add " & NewData & " to list?"
    
    strSQL = "INSERT INTO AddressTypes(AddressType) VALUES(""" & _
            NewData & """)"
   
    If MsgBox(strMessage, vbYesNo + vbQuestion) = vbYes Then
        CurrentDB.Execute strSQL, dbFailOnError
        Response = acDataErrAdded
    Else
        Response = acDataErrContinue
        ctrl.Undo
    End If

You can now enter as many or as few addresses per customer in the subform as desired.  Selecting the address type for each in the combo box.

A few general points:

1.  Do not include spaces or other special characters in table or column names.  Either use CamelCase, e.g. AddressTypes or represent a space by an underscore character Address_Types.

2.  Use plural or collective nouns as table names, e.g. Customers rather than Customer.  This reflects the fact that a table is a set.

3.  Use singular nouns as column names, e.g.  AddressType.  This reflects the fact that a column represents a single attribute type.

scottgem say:

First, most of the people who answer questions in the Answers community are volunteers. You can "reward" helpful answers by voting them as helpful and/or marking them as an answer.


Second. The table structure that Ken is (correctly) advising is based on a one to many relationship. One customer can have many addresses. The problem occurs because an address can also belong to multiple customers. So you cannot directly identify the addresses a customer has in the customer table, nor can you directly identify the customer an address belongs to in the address table. You need to use what's called a join or junction table to model the relationship. So your customer form should have a subform bound to this junction table that can list all the addresses assigned to a customer. You would then use another form (either a subform or popup) to enter new addresses.

Cities

States
....StateID
....State

The CityID combo box will be set up as follows:

Name:    cboCity

ControlSource:    CityID

RowSource:  SELECT Cities.CityID, City, State FROM Cities INNER JOIN States ON Cities.StateID = States.StateID ORDER BY City, State;

BoundColumn:   1
ColumnCount:    3
ColumnWidths:  0cm; 3cm;2cm
ListWidth:          5cm

If your units of measurement are imperial rather than metric Access will automatically convert the dimensions to inches.  The important thing is that the first dimension of the ColumnWidths property is zero hide the first column.  Experiment with the other two dimensions to get the best fit.  The ListWidth should be the sum of the ColumnWidths dimensions.

In the subform the state will be shown in an unbound text box with a ControlSource property which references the third column of the combo box:

=cboCity.Column(2)

2.  You won't of course as my demo uses completely different types of data.  All it is intended to do is show the structure of a binary many-to-many relationship type and how to represent it in a form/subform.  In this it is a basic paradigm for any binary relationship type.

The Column property is zero-based, so Column(2) is the third column, State.

3.  I know it well, I lived there from the age of 5 onwards.  If you are including areas of a city like this in the address, strictly speaking you should have a column AreaID in the CustomerAddresses table and no CityID column.  An Areas table would reference the Cities table via an Area ID foreign key.  Unbound text boxes in the form would then show the city and state for the selected area.  However, you might not want to go this far as, if this argument is taken further you would only store the SteeetID or similar, and this would determine all other elements of the address.  In fact this is regularly done in large databases in the UK by storing the post-code only, as a UK post-code determines a very limited location; mine for instance determines one side of my street, so all that is needed in a table to define my address is my house number and post-code.  Most databases of the type for which Access would be used would not do this, however, so you might want to have both Area (as a text column) and CityID (as a long integer number column) in CustomerAddresses.

talking moose say:
Rules in Outlook are triggered two ways:
  1. As messages arrive.
  2. Manually, by you selecting messages and applying rules.

I suggest rather than trying to move messages to a folder that you simply create a rule for each folder and then disable each rule. Disabling prevents the rule from running when you receive messages but it still allows you to run the rule manually. When you've read a message you can manually run your rule and move the message according to whatever criteria you choose.

Connect Entourage 2004 to Gmail’s IMAP service

http://www.officeformachelp.com/2007/11/connect_entourage_2004_to_gmails_imap_service/

  • Use root folder [Gmail]/
  • Do not select to sync All Mail folder

GMAIL'S RECOMMENDED IMAP CLIENT SETTINGS


Do NOT save sent messages on the server. If your client is sending mail
through Gmail's SMTP2 server, your sent messages will be automatically
copied to the [Gmail]/Sent Mail folder.

DO save draft messages on the server. If you want your drafts in your mail
client to sync correctly with Gmail's web interface, set your client to save
drafts to the [Gmail]/Drafts folder.


Do NOT save deleted messages on the server. Messages that are deleted from
an IMAP folder (except for those in [Gmail]/Spam or [Gmail]/Trash) only have
that label removed and still exist in All Mail. Hence, your client doesn't
need to store an extra copy of a deleted message.

Do NOT save deleted messages to your [Gmail]/Trash folder because this will
delete a message in all folders.

Do NOT save deleted messages to your [Gmail]/All Mail folder as some clients
will try to empty this folder and ultimately fail. This can lead to delayed
mail access or excessive battery consumption on a mobile device.


Do NOT enable your client's junk mail filters. Gmail's spam filters also
work in your IMAP client, and we recommend turning off any additional
anti-spam or junk mail filters within your client. Your client's filter will
attempt to download and classify all of your existing messages, which may
slow down your client until the process is complete.
