ken sheridan say:

Then... why the "AutoHeight" function exist after all !?

If the AutoHeight property has a value of True the section will automatically be resized at design-time if a control's height property is changed, so that it can no longer be accommodated within the height of the section.  If the value of the property is False on the other hand, a control cannot be resized to a height which cannot be accommodated within the height of the section.

My understanding is that the property is intended for use when controls are resized programmatically, and does not apply to the resizing of a control by allowing it to grow on the basis of the data, i.e. it is for use in advanced report programming situations.

diane poremsky mvp (slipstick.com) say:

First, did outlook tell you the pst was corrupt and needed to be repaired? You shouldn't run scanpst on it as a precaution.  There will always be a little corruption reported by scanpst - this is normal (its a database with constant read/write) - you could run scanpst hourly and it will find errors.

The best way to avoid corruption is by not using the pst in a folder that syncs to a cloud service or on a network drive. Keeping it reasonably small will also help (reasonably small is around 8 - 10 GB).  You can store a backup copy in the cloud or on a network  drive - the problem is with opening the pst from cloud or network drives. 

On to the error - are you using 32- or 64-bit office? How large is the pst file? Did you export from Live mail to Outlook to create it or use a utility to convert?  AFAIK, the error is normal. If you are worried, export the pst to a new pst and use the new pst.
