I am wondering if somebody could help me with the limitations of Name’s RefersTo property when referring to ranges. Based on testing I came to the below assumptions:

  1. If RefersTo character length exceeds 225 then RefersToRange property fails, though the name can handle longer length (ranges are selected when I select the name refersto in the Name Manager)
  2. In name manager it seems that the max character length is 2084, anything longer is just not added (it is cut off), but in VBA it is possible to put longer string and it is possible to select the name’s range via VBA.
  3. Referring to the 2nd point until the RefersTo’s characters length does not exceed 8198 characters it is possible to select via VBA, once it is exceeded workbook error and workbook can not be saved.

Below is the code I used for testing (empty workbook with one sheet, sheet name ’1’):

Sub test()

    Dim i As Long, rng As Range, nm As Name

    For i = 1 To 2000 Step 2
        If Not rng Is Nothing Then
            Set rng = Union(rng, ActiveSheet.Cells(i, 1))
        Else
            Set rng = ActiveSheet.Cells(i, 1)
        End If
    
        Names.Add Name:="Testing", RefersTo:=rng
    
        Set nm = ThisWorkbook.Names("Testing")
    
        On Error Resume Next
        Set rng = Range(nm.RefersToRange.Address)
        
        If i = 47 Then Stop     'nm.refersToRange.address fails at 47, char length 235
        
        If Err.Number = 0 Then
            Debug.Print i & vbTab & Len(nm.RefersTo) & vbTab & nm.RefersToRange.Address
        Else
            Debug.Print i & vbTab & Len(nm.RefersTo)
            
            If Len(nm.RefersTo) > 2084 And Len(nm.RefersTo) < 2095 Then
                Stop        'check in Name Manager the RefersTo field doesn't hold the full range string
                Range(nm.Name).Select
            End If
            
            If Len(nm.RefersTo) > 8150 And Len(nm.RefersTo) < 8197 Then
                Stop
                Range(nm.Name).Select
                Exit Sub       'if len exceeds 8198 error, workbook can not be saved
            End If
            
            On Error GoTo 0
        End If
    
        Set nm = Nothing
    Next i

End Sub

ken sheridan say: The only context in which it is legitimate to assign the values of non-key columns from one table to a referencing table is where the columns in each table are functionally determined solely by the whole of the table's key.  This is the basis of Third Normal Form(3NF) whose formal definition is:

Third Normal Form: A relvar is in 3NF if and only if it is in 2NF and every non-key attribute is nontransitively dependent on the primary key.

In your case this would firstly mean that the funeral companies tables and funeral directors tables should be coalesced into a single table.  At present the tables are related one-to-one.  A one-to-one relationship type is used to model a type/sub-type which is not the case here.

The implication of this is that only the name and phone number of the current funeral director of each company is stored.  When, in the fullness of time, the director changes the director's name and phone number values are updated to the new one.

The legitimacy of assigning the values to columns in the main table is dependent on a premise of the database being that the main table should store the two values  which were current at the time each row is inserted into the table, but as you've not told us what the main table is modelling we cannot say whether that is the case or not.  If not then only the primary key of the funeral companies table, which now contains the director data as described above, should be referenced by a foreign key in the main table.

If, on the other hand, the two values are functionally determined independently by the keys of the two tables, then the values should be assigned when a row is inserted into the main table.  This is analogous to the common scenario of the assignment of a product's current UnitPrice value from a Products table into an OrderDetails table, and is illustrated by the order details subform in the orders form in my DatabaseBasics demo to which I gave you a link earlier.  The methodology for this is that the two columns are included in the product combo box's RowSource and assigned to bound controls in the subform by code in the AfterUpdate event procedure of the combo box.  You would do the same in your data input form.

ken sheridan say: You have two options here:

1.  Coalesce the funeral companies and directors tables into a single table and assign the non-key values which will change over time this to columns in the main table when a row is inserted into the latter.  This was what I described in my last post.

2.  Have separate tables, as I described in my first reply, and reference the primary key of the directors table in the main table.  This table would maintain all directors' details over time, i.e. Its relationship to the companies table would be many-to-one , and when the director of a company changes a new row is inserted into the directors table.  The new row would have a value of TRUE in a Boolean (Yes/No) column named Active or similar to indicate that this is the current director.  The main table would then have only a foreign key column referencing the primary key of the directors table.  As each row in the directors table maps to a single row in the companies table, it is then simply a matter of joining the tables in a query to return all the relevant data.  No redundancy is introduced, and consequently the integrity of the data is protected.

To select the current director when inserting row into the main table  a combo box bound to the DirectorID foreign key column would be an SQL statement which restricts the rows to the 'active' directors.  However, this would mean that rows relating to an earlier director of that company would not show that director in the combo box as they are now excluded from the list.  This is overcome by making the RowSource an SQL statement which returns not only the 'active' directors, but also the director selected when the row was originally inserted.  An example of how to do this can be found in the ActiveEmployees demo in the OneDrive folder to which I gave you a link earlier.  In this little demo the RowSource property of the combo box bound to the EmployeeID column is:

SELECT EmployeeID, LastName & ", " & Firstname
FROM Employees WHERE Active OR EmployeeID = Form!EmployeeID
ORDER BY LastName, FirstName;

Code in the form's (actually a subform in the demo) Current event procedure requeries the combo box with:

    ' requery combo box to include current employee
    ' in list if not active
    Me.cboEmployee.Requery

In a form in single form view, that is all that's required, but this would not work in a form in continuous forms view or datasheet view, as the control would appear empty for any 'inactive' employee when the focus is on row with an 'active' employee.  This is overcome in the demo by using a hybrid control in which a text box is carefully superimposed on the combo box to give the appearance of a single combo box control.  The text box is bound to the text Employee (name) column returned in the form's RecordSource query.

Which of these options to use is for you to decide.  The first is simpler, but it means that there is no record of 'inactive' directors in the database if they are not represented in the main table.  My guess would be that this would not be a problem in the context of your application.
