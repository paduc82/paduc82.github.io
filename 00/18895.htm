jay freedman say:

Referring specifically to the question in the subject line of the thread, the "style for following paragraph" as set in each style's definition applies only to the situation where you're typing in the defined style and then press Enter at the end of the paragraph. It doesn't affect what happens when you apply the style to an existing paragraph -- the following paragraph will still  have whatever style it had before.

The only way you can re-style an existing paragraph that follows the selected paragraph is to create a macro to carry out the necessary steps, and assign a keyboard shortcut or Quick Access Toolbar button to run the macro.

For example, assuming your numbered lists use the List Paragraph style, and the Body Text Indent style is set to align on the left with the numbers, then this macro will apply the indent to every non-numbered paragraph that immediately follows a numbered paragraph. (This may not be exactly what you intended, since it may indent some paragraphs that you didn't want indented...)

Sub x()

    Dim objPara As Paragraph

    For Each objPara In ActiveDocument.Paragraphs

        If Not objPara.Range.End = ActiveDocument.Range.End Then

            If objPara.Style = ActiveDocument.Styles("List Paragraph") _

                And objPara.Next.Style <> ActiveDocument.Styles("List Paragraph") _

                Then

                    objPara.Next.Style = ActiveDocument.Styles("Body Text Indent")

            End If

        End If

    Next objPara

End Sub

Word left-to-right/right-to-left handling (LTR/RTL) works on two levels: the paragraph level, which can be controlled by right-clicking on the paragraph and choosing the appropriate direction (right-to-left, or left-to-right); and the run level.

Consider the following sequence of letters:

ABCD

This is an example of an LTR run. If we are using an RTL language, even though the sequence of letters is the same order, the text should be displayed in reverse, as a RTL run.

Actual order: ABCD
Visual order: DCBA

When we add netural characters -- characters that don't have rules about whether they are displayed as LTR or RTL -- we have to explicitly define the direction of the run:

(assumes LTR paragraph direction)
Actual order:    ABCD0123
LTR run:         ABCD0123
partial RTL run: DCBA0123
full RTL run:    3210DCBA

When typing, Word manages setting the run direction automatically based on the direction of the current language you are typing in.

  2. Press ALT+F11
  3. Click on the Immediate pane
  4. Type Selection.LtrRun and press ENTER

What, if anything, you need to transfer first depends on the type(s) of email account(s) you have configured on the old computer so the starting point on the old computer would be

Control Panel --> Mail -email tab

Lists the email accounts configured along with the applicable protocol for each

There is no singular one size fits all answer to your question - all depends on what you have. Moving data go from "nothing" to "really simple" having only 1 POP3 account to a "tad more involved" if you use a myriad of different account types - especially ones involving IMAP. 

If you are using stand-alone PST file (or files) AND/OR have a PST file designated as the default for the profile - you need to take inventory of those which are listed under the data files tab.

<IF> you're using multiple email accounts - the sequence in which you add your accounts on the new computer can equally have an impact on how things get set up as compared to your old computer.

I mean it does not make sense for me to learn all about Outlook migration to do it just the one time (with the possibility of making beginner mistakes along the way), while there might be an expert out there who does it day in and day out, and can thus understand each situation and perform the migration quickly and efficiently.

Doubt that migrating Outlook from one machine to another, day in and day out, is a viable activity for anyone.

Reality is that "migration" is <virtually> the same as "configuring Outlook from scratch" with the exception of knowing the original account types (POP, IMAP, Outlook.com etc) involved along with which data file(s) needs to be copied (if any). 

The "migration" can be as simple as just configuring the same email account on the new machine with nothing needing to be copied (i.e. if the email account is Outlook.com or Exchange).  It "may/can" get slightly more involved depending on Outlook versions, number and type of email accounts but reality is that you should know that information.

The other problem with a complete "hands-off" approach is that you are probably also not backing up your relevant local Outlook data (if applicable) on a regular basis which can end up having devastating consequences should a catastrophic problem/issue ever occur.

rohn007 say:

MS has moved that feature.

Go to Page Layout tab > Themes group (first one).

In there you can change colors 2 ways.  One is by picking a new color "theme" (but I don't see a black and white, or greyscale option).  The other is by picking a new option from the "colors" drop down. There is a greyscale one there.  You can hover your mouse pointer over the options in both of those drop downs to preview the effect on your document.

This themes feature is key to MS's "view" of all of Office applications as part of a unified "Office Suite of applications".   The theme is shared by all office applications.  The idea is you pick a theme (or custom build one) then use that same theme in all of your corporate Office documents.  The theme consistently applies preselected colors to various elements in your various document types.  If you use the themes attributes then you can pick another theme and all associated colors will automatically change to the new theme colors.

When you go into any color picker, the drop down now shows a scale of "Theme colors" with various shades. Below that are the old "standard" colors blocks.

I can provide more links to detailed articles about how MS wants us to use themes in our documents.
