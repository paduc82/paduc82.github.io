Combo Box in Forms - Microsoft Community

liamk100 say:

Combo Box in Forms

Hello,

I would like to add a combo box to my form to make using the form easier. How do I go about linking that to the original table that the form was made from?

ken sheridan say: If the combo box is for data entry, you would set its ControlSource property to the name of the relevant column (field) in the form's table.  The combo box's RowSource would usually be an SQL statement which returns a set of values from a column or columns in the referencing table.

If the combo box is to be bound to a numeric foreign key column which references the autonumber primary key of a referenced table, then you would hide the numeric key column and show the relevant text column or columns from the referencing table.  For instance this is how a combo box bound to an EmployeeID foreign key column in a table might be set up:

ControlSource:   EmployeeID

RowSource:     SELECT EmployeeID, FirstName & " " & LastName FROM Employees ORDER BY LastName, FirstName;

BoundColumn:   1
ColumnCount:    2
ColumnWidths:  0cm

If your units of measurement are imperial rather than metric Access will automatically convert the unit of the last one to inches.  The important thing is that the dimension is zero to hide the first column.

Alternatively, you can concatenate the names so that the last name is first, which would be better with a large list of names as it allows the user to enter the initial characters of the last name and progressively go to the first match as each character is entered:

RowSource:     SELECT EmployeeID, LastName & ", " & FirstName FROM Employees ORDER BY LastName, FirstName;

scottgem say:

There is a combobox wizard on the Controls section in Design view. That will lead you through the process of creating a combobox.

Have you designed your tables properly yet? Most of the time, a combobox looks u against a lookup table for standardizing or easing input. What is your form bound to? What is the combobox supposed to populate.

If you don't plan this all properly, then you are wasting your time as you will probably need to rip it out and start all over once you understand things better.

liamk100 say:

I have a few tables for the information in the combobox and I am able to use it properly in one of the forms that I have created. I would like to have the combo box show values based on an above box that a user could type in. I am looking to have the user type in their section and then have the combo box display only the supervisors in their section.

scottgem say:

Ok, This is a standard concept called cascading or synchronized comboboxes. It is well documented if you search on either of those terms. However it requires a proper table design.

So you would need a Section table like this:

tblSections

SectionID (PK)

SectionName

you would then need a Supervisors table or a personnel table that identified who the supervisors are. Finally, since it sounds like multiple supervisors can be assigned to a section, you need a section supervisors table;

tblSectionSupv

SectionSupvID (PK Autonumber)

SectionID (FK)

SupervisorID (FK)

This is why I have been stressing that you need to get your table design down first.

So the user then selects their section from a combo with a Rowsource liek:

SELECT SectionID, Sectionname FROM tblSections;

In the After Update event you will have to set the Rowsource of the Supervisors combo to something like:

SELECT tblSectionSupv.SupervisorID, tblSupervisors.SupvName INNER JOIN on tblSupervisors.SupervisorID = tblSectionSupv.SupervisorID

FROM tblSupervisors, tblSectionSupv

WHERE SectionID = Forms!formname!cboSection

ORDER BY SupvName;

then the Supervisor combo will only show supervisors for that Section.

Disclaimer: This is aircode off the top of my head and may not be syntactically correct, but it should give you the idea of what needs to be done. 

And again, you have to get your table structure in place before you start designing forms.

ken sheridan say: You might like to take a look at DatabaseBasics.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

This little demo file includes examples of correlated combo boxes for data entry, or for 'drilling down' to rows in a form.

The contacts form in the demo has correlated combo boxes for selecting a country, region, then city.  An important point to note is that only the last is bound to a column in the Contacts table.  To include columns for the region and country in which the city is located would introduce redundancy and the risk of update anomalies, as city determines region, and region determines city.  The two unbound combo boxes are assigned values when navigating to an existing record by calling GetRegion and GetCountry functions in the form's Current event procedure.

The region and city combo boxes are requeried in the AfterUpdate event procedures of the country and region combo boxes respectively.

A new country, region or city, not currently represented in the database, can be entered by tying the name into the relevant combo box, which executes the control's NotInList event procedure to insert a row into the relevant referenced table.  Note that this assumes that no country can have two regions of the same name, and no region can have two cities of the same name.  There can be cities of the same name in different regions, and regions of the same name in different countries, however.

If your situation differs from this in that a member of staff can be a supervisor in more than one section, you will need a table such as Scott's tblSectionSupv to model the many-to-many relationship type between sections and supervisors.  As SectionID and SupervisorID constitute a composite candidate key of that table it would be legitimate for an Employees table to include both SectionID and SupervisorID foreign key columns, and consequently two bound combo boxes in the form.  If, on the other hand, the model is a tree-structured hierarchy in which a member of staff is a supervisor in one section only, then supervisor determines section, and consequently there should be no SectionID foreign key in an Employees table.  A table such as Scott's tblSectionSupv would still be necessary, but in this scenario would be modelling a sub-type of Employees.  The relationship type between Employees and tblSectionSupv would thus be one-to-one, i.e. the SupervisorID column would be its primary key as well as being a foreign key, and there would be no surrogate autonumber key.  In the form the supervisor combo box would be bound, but the section combo box would be unbound, as with the region and country combo boxes in my demo.

In the same demo another form has three unbound correlated combo boxes in the form header.  Again these are for selecting a country, region, then city, but in this case the selection of a value in each 'drills down' through the hierarchy, progressively restricting the rows returned in the form.  This is achieved by referencing the three controls as parameters in the form's RecordSource query.

jaymri say:

Updating hyperlinks - is there a way to update all links at once?

I have a spreadsheet with numerous hyperlinks. They are all linked to the same server. I got notification today that we will be moving everything to a different server. Is there a way to do a mass update to my hyperlinks? Can VBA code handle this? (I do not have much experience with VBA).

Thank you,

Jay

gary's student say:

Easy for a macro.  Say a typical hyperlink is like:

\\serv1\jan\actual\income\first.xls

and we want to make all the serv1 into  serv2:

 

Sub HyperFixer()
Dim h As Hyperlink
For Each h In ActiveSheet.Hyperlinks
    h.Address = Replace(h.Address, "serv1", "serv2")
Next
End Sub

I suggest another approach:  Assign a mapped drive designator to the server (like Z:) and have the hyperlinks reference the drive.  The next time you change servers, you only need to change the mapping and can leave the hyperlinks alone.
