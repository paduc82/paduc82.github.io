Files are editable in OneNote 2016 but are read-only in Onenote 17 - Microsoft Community

guypennington say:

Files are editable in OneNote 2016 but are read-only in Onenote 17 ?

I have the full OneNote 2016 and also OneNote Version 17.9126.20561.0 (from the store) installed on my Windows 10 Laptop.

In OneNote 2016 I can open inserted files (excel etc), edit and save, no problem. If I try to do the same from OneNote 17 I cannot edit the files as they are marked read only. I don't understand why, it cant be the file permissions as works fine in ON2016. 

Any suggestions ?

Thanks

swischner say:
Unfortunately, your observation is correct and it's not something you can change.
I already talked to a OneNote engineer at Microsoft about this. The reason for this restriction is the architecture of UWP apps (which in this matter is similar to MacOS or iOS where the same restrictions to OneNote apply).

When you embed a file onto a OneNote page, a copy of the file is created and put into the OneNote section file. An additional copy of the note content -- including the embedded file -- is created in a temporary storage area, the OneNote cache.

Now if you open an embedded file from that cache (that's what happens with all edits in OneNote notebooks) with another program (like Word or Excel), that program must have write permissions to those cache files. This is no problem with Windows and W32 programs (the standard architecture of Windows programs in opposite of UWP/Windows 10 apps).

That's why you can open the file from OneNote's cache area in Word or Excel and after editing it write it back to that cache.

On other operating systems (like MacOS) and with the new Windows UWP apps (as the OneNote app with the 17x version number) this is different: Each application has its own "sandboxed" area and no other program may write to that area. OneNote for Windows 10 (the UWP app) uses another cache area than OneNote 2016 and does not allow other programs like Word or Excel write to it.

That's why your files are opened read-only. The access rights to or file attributes of your original file have nothing to do with it; that file isn't even touched because the embedding created a separate copy whatsoever. Just in case you did not notice: If you embed a file in OneNote, open it from OneNote 2016 in Word or Ecxel, edit it and save it back, the original file that you had embedded does not get changed at all. It's only the copy in the cache (and after synchronization in the OneNote files) that you edit.

There is no way around that restriction of the UWP apps but I was told that Microsoft is looking into it as they are well aware of the confusion this may create.

Your only options are:
- Just edit the embedded files from OneNote 2016 or
- Don't embed files at all but just insert a link on the page pointing to the original file.

These restrictions also make it clear that OneNote has been designed to be a note taking and -- to some extent -- information archiving system but not as a dockument management. It's rather weak in that area.
