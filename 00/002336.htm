Your basic problem is that you're trying to treat the table cells as shapes, which they're not. Try something based on:

Sub CellBevel()
If Selection.Information(wdWithInTable) = True Then

Thus, please, is it possible to call several datasources from a single Word document?

a. user say:

I don't know which methods you are using at the moment, but there are 5 main ways you can get data in an external Excel sheet into a Word 2011 document:
 a. using LINKs (e.g., Edit->Copy a range of cells in Excel and Edit->Paste Special into Word)
 b. using Mailmerge - set up the Excel sheet as the data source for a Mailmerge and use { MERGEFIELD } fields to insert values from the sheet
 c. using a { DATABASE } field to insert a table of data from a Sheet
 d. Using VBA (or Applescript) to retrieve data from Excel and insert it into your Word document.
 e. Do as much data manipulation as possible in Excel first, creating "the data source that you need"

I would guess from the fact that you are talking about Data Sources that you are probably using (b). If so, and if what you need is a multi-part report where each part of the report uses a different data source, then you can consider
 a. setting up each part as a separate document
 b. perform a separate merge for each part
 c. combine the documents. You could, for example, set up a "main" document that included the results of all the merges using { INCLUDETEXT } fields.

If you are going to do that, a good way to proceed would be to create the entire Mail Merge Main document as a single document in the first instance to avoid introducing formatting and style clashes. Then split it when you're ready.

However, from your description I would guess that you may want to do some "one-many" reporting, e.g. perhaps to list a variable number of members per group.

Word "out of the box" is not very good at that kind of thing. You may be far better off using tools that are designed for reporting from databases (FileMaker is a possibility, although I do not know how good it is for that purpose.)

But even in Word you can
 a. use one Excel table as the data source for a merge, and use { DATABASE } fields to insert related data from other data sources. The main problems with that approach are that
   - the data sources all need to be tabular
   - Word 2011 may ask a "security" question every time it needs to get data from a sheet (not sure)
   - the layout options are limited - you may need to reformat the resulting tables manually or using VBA

 b. use the "one-many" approach described in 
http://support.microsoft.com/?kbid=294686 .

Word MVP "macropod" has produced a much more complete description of this approach - there's a version on Graham Mayor's site at
http://www.gmayor.com/Zips/Catalogue%20Mailmerge.zip

However, both those approaches depend on a single MailMerge data source, i.e. one that has already combined all the data you need in the sequence required by the document.

But if you can spell out esxactly what you need to achieve, it may be a little easier to make useful suggestions.

a. user say:


About the second part of your message on "one-many" reporting, I do need to read the proposed material again and think about it. Our take now is to do this kind of analysis out of Word, by using a number o different tools, get the tabulation done and read it with the same { INCLUDETEXT } approach you proposed.

I'm still not sure if could it be worth to rely on Word to accomplish this kind of reporting work. We started to approach to LyX, LaTeX and literate programming to deal with these problems, but it is by no means easy to introduce new tools in a big organization having Word as a kind of SwissKnife!


To me, the key question when discussing Word-based reporting is "who's doing it?" Is it end users, who can do all the layouts, may struggle a bit with fields but can probably work through that, but may not have the time to plunge into VBA, and may not be allowed to do so. Or is it "IT people" who may have access to a much more flexible toolkit. Or is it end users who have also been provided with a 3rd-party reporting capability?

As far as report production using Word is concerned, the fact is that there's considerably more flexibility on the Windows platform. The main reasons are:
 a. out-of-the-box, data connectivity is significantly more flexible. On Windows, you can connect to Excel and even plain text files using OLEDB and most of the features of Jet/ACE SQL. Unicode support is better (unless Mac Word's Excel converter has improved). 
 b. it is possible to use MailMerge Events (even though they seem to be rather shaky at times).
 c. beyond that, if you also have VS.NET etc., it's possible to do a lot more using VSTO and/or the Open XML SDK (nice if it worked on Mac with Mono, but I have never managed to achieve that). 

If (c) is an option for you, it is probably worth having a look at Eric White's blog at
http://ericwhite.com/blog/category/open-xml/document-generation-series/

teylyn say:
Hello,

if you use INDIRECT() because the value in cell A1 can change, or because you want a formula that you can easily copy down, so it refers to A2, A3, A4, etc,  then the INDEX function suggested by Tom-S will not work, because the workbook reference is hard-coded.

In this case, INDIRECT() goes in the right direction, but it will not work with closed workbooks. 

You can download and install a free add-in called morefunc.xll, which has a custom function called INDIRECT.EXT that will work with closed workbooks, too. The add-in can be downloaded here:

I did want to let you know of one other (fairly) simple approach. You can update PowerPoint from SharePoint columns by way of Excel.
