suzanne s. barnhill say:

The keyboard shortcuts Ctrl+PageUp and Ctrl+PageDown are assigned to the BrowsePrev and BrowseNext commands. In previous versions of Word, the double arrows at the bottom of the vertical scroll bar also execute these commands, and there is a Select Browse Object button between the arrows that allows you to select which object you want to browse by. The default actions at Word startup are Previous Page and Next Page, but after the Find (or Replace or Go To) dialog has been used, they become Previous Find/GoTo and Next Find/GoTo. To restore the original action, you have to use Select Browse Object to select Browse by Page.

This works great in Word 2010 and earlier (it's particularly helpful, in versions that don't have the Navigation pane, to be able to use them to navigate quickly through search results). In Word 2013, an unfortunate design decision was made to eliminate the browse arrows and the Select Browse Object gallery. In the Navigation pane (Results tab), you can click the arrow at the end of the search box and find some of the "browse objects" to search by, but Page is not one of them.

There are several workarounds for this:

  1. The article at http://wordribbon.tips.net/T011206_Setting_a_Default_for_the_Object_Browser.html provides some macros that you can use to set the browse object back to Page.
  2. You could add Next Page and Previous Page buttons to the Quick Access Toolbar (File | Options | Customize Quick Access Toolbar: All Commands).
  3. You could assign the Ctrl+PageUp and Ctrl+PageDown shortcuts to the NextPage and PrevPage commands (File | Options | Customize Ribbon: Keyboard shortcuts). (If these commands don't work as expected, there are also commands called GoToNextPage and GoToPreviousPage; these would surely work.)

Indic typography - serious Uniscribe deficiency - Microsoft Community

graphityau say:

Indic typography - serious Uniscribe deficiency

There is a serious long-standing deficiency in the Indic shaping engine (Uniscribe?) used by Microsoft Word (both Windows and Macintosh versions).

The problem is the lack of support for Type 6 GSUB lookups ("Chaining Context Substitutions"). It is known as the cross-cluster contextual substitution problem.

This deficiency has been known and discussed since at least 2013 and it has since been resolved by both Apple's CoreText engine and also by Harfbuzz. In the case of Harfbuzz the "workaround" recommended by a 2014 working group was implemented, allowing critical Type 6 substitutions to be relocated to a separate feature <rclt> ("Required Contextual Alternates"). In the case of Apple's CoreText they implemented the required fundamental change to the basic engine allowing post-reordering features to apply to whole glyph runs rather than to discrete clusters, thereby enabling existing presentation form layout features such as <pres> <abvs> <blws> etc to include cross-cluster input sequences and context statements. Apple also implemented support for the <rclt> workaround.

The lack of support for cross-cluster contextual substitutions in Uniscribe (either by inclusion in the basic engine or by support for the recommended <rclt> workaround) means that in Microsoft applications features that rely on cross-cluster glyph sequences or contexts cannot be used to perform substitutions, effectively disabling crucial typographic mechanisms at the font level to resolve relationships of adjacency at cluster boundaries. This results in collisions between adjacent shapes/marks, crowding and other disruptions of readable text. Such situations are extremely common in Indic typography.

We are in the final stages of development in a global Sanskrit font project (dev2 script - Devanagari) and the problems caused by this deficiency in Uniscribe on both Mac and Windows platforms is causing considerable concern for us, for our client and for the growing number of users of the font worldwide. We are in the unfortunate position of having to warn potential users of the font that if they use Word they will not have a fully-functioning font. Regrettably this means we find ourselves having to recommend that they should use other applications. This cannot be welcome news for you and your colleagues.

We are prepared to assist Microsoft in any way we can with the implementation of at least the <rclt> workaround in Uniscribe for this problem. Ideally, in the long term, the change needs to be made at a more fundamental level (as with Apple's CoreText) but the workaround would solve the immediate problem for us and many, many other Indic font developers.

We will happily supply test font and text strings to expedite your analysis and testing. We will provide prompt and comprehensive feedback on any betas you make available to us.

You dont mention if the tables in question are related or not.

The basic approach here is to send up the tables to SharePoint.

Once done, the result is a desktop application (that you run in regular Access), but the data (back end) resides on SharePoint.


>Do they need Microsoft Access as well or can SharePoint handle the front end as well?

Not for existing forms and code. For an existing application with code (VBA) and forms, then you will continue to install + run Access on each desktop (you can use the free runtime edition for Access and the runtime is essentially the same install foot print and trouble as installing the full edition of Access.
