Sort columns without changing the cell color in EXCEL 2010 - Microsoft Community

jimyvr say:

Sort columns without changing the cell color in EXCEL 2010

In order to make it easier to read long multiple columns of data, I shaded (fill color) every second row a light grey color.  When sorting the columns though it also changes the fill color so the rows no longer have alternating shading.  How can I sort just the data and not the fill color.  In other words, whenever I sort columns, I'd like every second row to still be shaded a light grey regardless of what the data is.

 

e.j. gun say:

You can use a clever trick with conditional formatting!

  • Select your data (all rows, columns)
  • Clear out all existing fill colors
  • Select Conditional Formatting/New Rule...
  • Select "Use a formula to determine which cells to format"
  • In the empty box under "Format values where...", enter this formula:

=MOD(ROW(),2)=0

  • Click on the Format... button and select the fill color you want for all the even rows.
  • Click OK
  • Click OK

Now you have alternately shaded rows!

If you want to be even more clever, select the same data and repeat the steps above, but for the odd rows, use a slightly different formula:

=MOD(ROW(),2)<>0

Now you can choose the fill color for the odd rows too.

When you sort your data, the conditional formatting (which is tied to the cells themselves) does not change.

HTH,

Eric

Access 2010 Query Returns Duplicate $ values - Microsoft Community

almc_749 say:

Access 2010 Query Returns Duplicate $ values

Hi

I have been trying to compare a 1996 industry-wide table of department names and total expenses for each named department with a 2013 industry-wide table of department names and total expenses for each named department.  In other words, lets say there are 40 Personnel Department totaling $100 million across the industry for 1996 and 30 Personnel Departments totaling $200 million in 2013.    I used the unmatched query wizard generated SQL changing the "is null" to "not is null." 

SELECT [1996 DEPARTMENT].NAME, [1996 DEPARTMENT].SumOfExpenses, [2013 DEPARTMENT].NAME, [2013 DEPARTMENT].SumOfExpenses

FROM [1996 DEPARTMENT] INNER JOIN [2013 DEPARTMENT] ON [1996 DEPARTMENT].NAME = [2013 DEPARTMENT].NAME

WHERE ((Not ([2013 DEPARTMENT].NAME) Is Null));

The problem is that the resulting query yields a comparison of 1996 and 2013 departments and total expenses that repeat the same number.  I'll get something like:

Personnel Department    $100

Personnel Department    $100

I would like to know if there is a way to insert something into the SQL that says don't duplicate categories?  I tried playing around with DISTINCT but was unable to make it work.  Is there an alternative SQL that might work better?

Thank you for taking the time to read and respond to this question.

ken sheridan say: You don't need the WHERE clause.  AN INNER JOIN only returns rows where there is a match in each table.  However, that's not the cause of the problem, which, as Duane has pointed out, apparently results from there being multiple rows per department in some cases in one or both of the tables.  Once you've eliminated the duplicates, the NAME column should be uniquely indexed to prevent this.

Bear in mind, however, that any departments which are not represented in one of the tables will not be returned at all, so if a department has been created, deleted or renamed since 1996 it will not be represented in the query's result table.  A JOIN type to cater for this is not supported by Access and would have to be simulated by a UNION of two OUTER JOINs.  

You should also be aware that there are more fundamental flaws:

1.  You are encoding data in the table definitions by having separate tables per year.  A fundamental principle of the database relational model is the Information Principle (Codd's Rule #1).  This requires that all data be stored as values at column positions in rows in tables, and in no other way.  The data for all years should be amalgamated into a single table with an additional column for the accounting year, which can easily be done either by means of a UNION ALL operation, or by means  a set of separate append queries, one per year.  The key of the single table would be a composite one of the department name and accounting year columns.  A further table, Departments, should be created with DepartmentName as the key (though the use of 'surrogate' numeric keys in each table would be preferable), and an enforced relationship created between this and the single DepartmentalExpenses table.  This will protect the integrity of the database by preventing invalid department names being entered into the DepartmentalExpenses table.

2.  NAME should not be used as a column name.  It is the name of a built in property in Access, and as a 'reserved word' should not be used as an object name.

3.  While spaces and special characters are permitted in object names in Access, the overwhelming opinion  amongst experienced developers is that they should not be.  Instead use CamelCase or represent a space by an underscore character like_this.
