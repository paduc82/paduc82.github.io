donwalt say:

All queries do not show up when mail merging.

I am trying to mail merge using an access database.

When I locate the database to use it will not reveal all the queries available in the Access database.

This worked fine in Access 2000 and Word 2000 until Office 9007.

I have another very similar database which merges fine.

a. user say:

This is to do with the way that Word connects to the data source.

In Word 2000, by default Word connected using DDE, which communicated with Access and could "see" all the queries that Access could see.

In Word 2002 (XP) and later, by default Word connects using OLE DB, which works directly with the underlying database engine, Je (or ACE, in Access 2007 or later). There are a number of types of query that OLE DB does not "see", or does not present to you, including

 a. parameter queries

 b. queries that use certain kinds of finiancial functions

On balance it is better to use OLE DB where you can because DDE is "deprecated", but in Word 2007, you can still try DDE by going to Word Office button->Word options->Advanced-General and checking "Confirm file format conversion on open". Then go through the process of connecting to your data source again. After the step where you select your database, you should see an additional dialog box. If your database has a .mdb extension, select the option "MS Access databases via DDE". If your database has a .accdb extension, you will need to check "Show all" to reveal that option.

NB, OLE DB may "see" the query but return no results when the query has "wildcards", e.g. as in LIKE 'A*', because Access usually expects the wildcards * and ? whereas OLE DB expecs to use % and _. In that case, if possible, the best approach is usually to duplicate the query in Access but using the OLE DB wildcards.

No, you can't. Several types of query do not appear when the connection type is OLE DB or ODBC, as I have indicated. There are (probably) several reasons:

 a. with parameter queries, Word used to rely on Access to prompt for the parameter values, which it could do because it was communicating with Access. From Word XP onwards, Word uses OLE DB by default and since neither Word nor OLE DB has a built-in mechanism for prompting for parameter values, you can't use this connection method with a parameter query, at least not without using VBA. It's the same with ODBC.

 b. Access queries can use Jet VBA functions, but some of these (such as the replace() function ) are missing from an internal table (that's the best description I can give because I don't know exactly what is going on internally) and queries using those functions cannot be used by Word

 c. Word probably won't "see" Union queries (although it may, if you wrap the Union query up in a further query and use that). This may be because there can actually be two types of "query" in a Jet database - "queries" and "procedures", Word/OLE DB can see some but not others.

As for the other data source types,

 d. AFAIK, Word never actually uses MAPI to access any type of data source. for example, when it accesses Outlook Contacts, it actually does so by using the Jet/ACE Outlook IISAM to look at the Outlook Address Book. There are a couple of legacy address book formats that Word can read (cf. the Schedule+ list you mention) but I doubt if you would be able to read any other address book unless it know how to expose itself to the Outlook Address Book

 e. You should be able to use any OLE DB/ODBC data source that implements the necessary interfaces, not merely "file-based" ones. i.e. you can use SQL Server (which I wouldn't normally describe as "File-Based", Oracle,and so on. However, not all OLE DB source will do - for example, I don't think you can get anything based on the "OLE Db simple Provider" to work, probably because it doesn't impleent a Command interface.

 f. with some limitations, you can use any "tabular" data source for which Word has an internal or old-style external  converter. I'm not so sure about files that are read by the new COM-based converter in Word 2007 SP2.

graham mayor say:

You could do this easily enough with a macro. The following will reset the contents of a selected dropdown form field to insert the list of names in the macro
http://www.gmayor.com/installing_macro.htm

Sub ResetDD()
Dim oDD As DropDown
Set oDD = Selection.FormFields(1).DropDown
With oDD.ListEntries
    .Clear
    .Add "Fred"
    .Add "Bill"
    .Add "John"
    .Add "Eric"
    .Add "Will"
    .Add "Tom"
End With
End Sub

If all the dropdown fields in all the documents have the same bookmark name you could use a batch process to update them all as follows. In this example, the list of names is stored in a Word document its name and path defined at the line
strNameDoc = "D:\My Documents\Test\Versions\Odd\NameList.docx"
Change the name and path as required.
In this example, the macro opens every document in a selected folder and changes the dropdown1 named form field to hold the list of names in your name document. Maintain the name document as the list of names to be used in the dropdown and run the macro on the folder containing all the documents to make the changes.

Sub ResetDD()
Dim oDD As DropDown
Dim strFilename As String
Dim strPath As String
Dim strNameDoc As String
Dim oDoc As Document
Dim oListDoc As Document
Dim oRng As Range
Dim fDialog As FileDialog
Dim i As Long, iName As Long

'Change the path to match the name of your namelist document
'which must comprise of one name per line with no blanks
'Maximum number of lines is 25!
strNameDoc = "D:\My Documents\Test\Versions\Odd\NameList.docx"

Set fDialog = Application.FileDialog(msoFileDialogFolderPicker)
With fDialog
    .Title = "Select folder and click OK"
    .AllowMultiSelect = False
    .InitialView = msoFileDialogViewList
    If .Show <> -1 Then
        MsgBox "Cancelled By User", , _
        "List Folder Contents"
        Exit Sub
    End If
    strPath = fDialog.SelectedItems.Item(1)
    If Right(strPath, 1) <> "\" _
    Then strPath = strPath + "\"
End With
If Documents.Count > 0 Then
    Documents.Close SaveChanges:=wdPromptToSaveChanges
End If
Set oListDoc = Documents.Open(FileName:=strNameDoc, _
    Visible:=False)
For i = oListDoc.Paragraphs.Count To 1 Step -1
    If Len(oListDoc.Paragraphs(i).Range) <= 2 Then
        oListDoc.Paragraphs(i).Range.Delete
    End If
Next i
If Left(strPath, 1) = Chr(34) Then
    strPath = Mid(strPath, 2, Len(strPath) - 2)
End If
strFilename = Dir$(strPath & "*.doc")

While Len(strFilename) <> 0
    Set oDoc = Documents.Open(strPath & strFilename)
    Set oDD = ActiveDocument.FormFields("Dropdown1").DropDown
    With oDD.ListEntries
        .Clear
        If oListDoc.Paragraphs.Count > 25 Then
            iName = 25
        Else
            iName = oListDoc.Paragraphs.Count
        End If
        For i = 1 To iName
            Set oRng = oListDoc.Paragraphs(i).Range
            oRng.End = oRng.End - 1
            .Add oRng.Text
        Next i
    End With
    oDoc.Close SaveChanges:=wdSaveChanges
    strFilename = Dir$()
Wend
End Sub

doug robbins - mvp office apps & services (word) say:

If you had the items that you want to appear in the dropdown stored one to a paragraph in a document, you could use the following code in a macro in your form to populate the dropdown with the items in the document.

Dim Source As Document
Dim Target As Document
Dim item as Range
Set Target = ActiveDocument
Set Source = Documents.Open("C:\Users\Doug\Documents\ListEntries.doc") 'modify to reflect the name and path of the file containing the items
For i = 1 To Source.Paragraphs.Count
    Set item = Source.Paragraphs(i).Range
    item.End = item.End - 1
    Target.FormFields("DropDown1").DropDown.ListEntries.Add Item.Text
Next i
Source.Close wdDoNotSaveChanges

If you form is a template, then you could name the macro autonew and it would then load the dropdown formfield with the list items when a new document is created from the template.
