tom van stiphout say:
That's right, Access does not have a designer for union queries, nor do many other tools. Rather you are supposed to know a little bit about query syntax and type in your sql statement. I gave you an example that's pretty close. The Help file also has info on this type of query.

I understand the desire to keep data separate, but that's an Excel-way of thinking and not at all a Relational Database way of thinking. As an example think of teh Northwind sample application which is about customers and their orders. Some business owner might say "I want an order table for each customer so we can keep them apart". Database designers would say LUDICROUS, you just add a CustomerID field to the Orders table, and if you want to see the orders for customer 5 you query: select * from Orders where CustomerID = 5. Some might say: but I'm going to have SO many rows, that must be slow. Again the database designer would (with a sense for drama :-)) say LUDICROUS: the power of indexing GUARANTEES very fast queries.

Your situation is a bit different if I understand you correctly, because you would say: the fields in each Order table are different. In your case in each District. So you have 800 tables, each with different fields. Well, perhaps not THAT bad, I'm sure SOME fields occur in most or all, like DistrictID (autonumber, PK) and ZipCode (text5, required, duplicate index).

There are two ways to solve this:
1. (if there are certain patterns that arise, with groups of districts having more or less the same fields) Have a Districts table with the main fields that everyone has, and one table per group with the additional fields. They would be linked 1:1 (in a one-to-one relation) to the main table.

2. (if there are no discernible patterns) the classic solution is to create a M:M (many-to-many) relation with 3 tables: Districts as above with the common fields. Attributes (AttributeID autonumber PK; AttributeName text50 required uniqueindex), and AttributesForDistricts (DistrictID long int required PK; AttributeID long int required PK, TheValue long int required). The Attributes table would have one row for each field you have today, probably things like 1=PopulationCount, 2=RestaurantCount, 3=ParticipantsInProgramXCount, etc. Then in what's often called the "junction" table AttributesForDistrict you would put things like:
District=1, Attribute=1, TheValue=5000
District=1, Attribute=3, TheValue=20
District=2, Attribute=1, TheValue=4000
etc.

Believe me, this is the relational way, and it has proven over 50 years to be superior to any other organization of data. It will pay dividends every time, whereas your solution is going to be one big struggle.

One more example: how do you enter your data, or report on it? Surely you don't want to create 800 forms with 800 different sets of fields. In my example there is one form and one subform.

1. VSD files saved to your hard-drive (e.g., your desktop) should open in the standalone version of Visio Viewer, not in Visio Viewer loaded in an instance of Internet Explorer. [1]
