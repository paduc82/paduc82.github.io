marshall barton say:

You should not care how the option value looks in its table.  Actually, you and your users should never see a table's datasheet when entering, editing or viewing the data.  Instead you should always use forms and reports to interact with the data and your option group is as good a way to do that as any.  And, if you did change the values in the table, the option group could not be used to edit or view the text that you wanted to store.

 

The preferred way to handle this situation is to add a text box to your form to display the converted text string.  For the example you used, the text box could use this expression to display (not store) the letter:

   =ASC(64 + [name of option frame])

 

For more complex text strings, it is highly recommended that you create a table (named Options) with one field (primary key) that has the option value and another field for the text string.  Then, in any report or if the option group is not use in a form to modify the value, the form/report's record source query can join the two tables and simply display the text string.  In a form where the option group is used to enter/edit the value in the table field, the text box next to the option group can use an expression like:

   =DLookup("[option text field], "Options", "[option number field] = " & [name of option frame])

 

Note that the word Field refers to a column in a table or query.  It does not refer to any control on a form or report.

mzzfmly5 say: This works but the option buttons in the group don't stay selected in the form?   

scottgem say:
This works but the option buttons in the group don't stay selected in the form?   
If you are storing something different that an integer in your table, then the Option Group needs to be unbound and you need to populate it from the table. 

The better way is to create a lookup table with an integer key and the a corresponding description. Store the integer as a foreign key. Then join to your lookup table to display the description on your reports.
