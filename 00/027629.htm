ken sheridan say: Firstly avoid the 'lookup wizard' like the plague when defining a column's data type.  For reasons why see:

http://www.mvps.org/access/lookupfields.htm

As it's quite rightly in the context of a form where you want to enter data and see the 'looked up' values, using the lookup wizard in table design serves no useful purpose which you can't achieve simply by other means, i.e. by using a combo box as the bound control.  Let's assume that it's an Admissions form and you want to select a patient and fill a PatientID numeric column in the Admissions table with the ID for a patient selected by name.  For this you'd set up a combo box along these lines:

ControlSource:    PatientID

RowSource:     SELECT PatientID, LastName & ", " & FirstName FROM Patients ORDER BY LastName, FirstName;

BoundColumn:   1
ColumnCount:   2
ColumnWidths:  0cm;8cm

If your units of measurement are imperial rather than metric Access will automatically convert the last one.  The important thing is that the first dimension is zero to hide the first column and that the second is at least as wide as the combo box.

You'd then be able to select a patient from the combo box's drop down list by name, in the format 'Doe, John' and their numeric PatientID value would be inserted into the PatientID column in the Admissions table by virtue of this being the hidden bound column of the combo box.

One caveat with this, however.  Remember my account of the two women with exactly the same names and dates of birth who turned up at a clinic at which I was present.  Selecting a patient solely by name is very risky.  In the case I described patients were identified by a combination of name, gender and date of birth, but as you see even this wasn't good enough.  Fortunately the nursing staff spotted the problem before any unfortunate consequences could follow, but it doesn't take a lot of imagination to see what might have been the result otherwise.  So with the above example you'd really need to consider what other columns you might need to include in the combo box's RowSource to guarantee the identification of the correct individual.  A combination of names, address and date of birth is probably enough, as the possibility of two identically named patients with the same dates of birth, living at the same address, is pretty remote.

If, with the above example, you wanted to show other data from the Patients table in the Admissions form, e.g. address data etc, you can base the form on a query which joins Patients to Admissions on PatientID, and return all the columns from Admissions in the query, plus those from patients which you want to see in the form.  Don't return PatientID from Patients, however, only from Admissions and don't return FirstName or LastName as these are already shown in the combo box.  Bind text box controls to the columns from Patients returned by the query and set the Locked property of the text boxes to True (Yes) and their Enabled property to False (No) to make them read-only.  Once you select a patient in the combo box the other data from Patients will show in the relevant text box controls.
