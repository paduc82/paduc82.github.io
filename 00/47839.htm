¡firedog say:
@Jonathan

I've had a look at the message source you posted. The message has been composed in Word, but I couldn't immediately detect anything in the markup that would by its absence or presence cause WLMail/IE to fail to display it.

When I saved the message as a text file with the extension .eml and tried to open it with WLMail, I saw what I suppose you also see - the header bar containing the subject, sender, date and addressee, along with an .htm attachment, and a blank body. The attachment opens normally in IE.

The one thing that looks odd is the primary Content-Type (in the message header): multipart/htmlmessage is not as far as I can see a standard one (i.e. registered with IANA). A multipart message doesn't have to contain more than one part, but the type is usually used in the message header as multipart/alternative, comprising, say, a text/plain part for display by clients that can't handle HTML and a text/html part for those that can. To find out whether this could be the reason, I changed the Content-Type in the message header to multipart/alternative. The message opened normally.

My conclusion is, then, that messages with this non-standard MIME type will not be displayed in the normal manner in WLMail. This should be configurable in the composing application. I would expect it to be possible to generate a multipart/alternative message with text/plain and text/html parts, or at least a text/html message in a single part. In any case, the 'fault' - if fault it is - lies with the sender. 

Noel
