Creating a company directory - Microsoft Community

brooketesterman say:

Creating a company directory

I am trying to create a company directory that will include a photo, name, job title, and the floor they are on.  I would like to group these by company (we are an umbrella company with a couple of different companies underneath us) and then department.  I have created a table that lists the information and then a query that includes all the information (including pic) and am basing my report off of that.  I would like it to be a multi-column report (4 or 5 columns) and have the picture, with the name, job title, and floor directly underneath the picture.  I would like the company and department heading to extend across all the columns, and also to have the company name and department repeat when they are on more than one page.  I have figured out how to group them all together and such, but am having problems extending the two groupings across multiple columns and repeating them on multiple pages.  Someone once suggested doing a subreport, but I can't seem to figure it out properly.  Could someone point me in the right direction?  I am using Access 2010.

Thank you!

marshall barton say:

Create a report with the columns and the details (picture, name, dept, etc) the way you want them, but without any headings.

 

 

Then create another report with only a single column and a company group header.  Make sure the record source query only returns each companies info once:

 

   SELECT DISTINCT company FROM yourtable

 

Set the company group header sections RepeatSection property to Yes

 

 

Now drag/drop the details report into the companies report's detail section and position/size it to fill the main report's detail section.  Make sure the LinkMaster/Child properties are set to the company field.

brooketesterman say: Question, I have a company table that lists each company and then I have an employees table where I have the look-up set to look-up the name of the company from the company table.  So, the Company field in the employee table is "long integer" and the Company field in the Company table is "text".  When I try to link them it doesn't work.  What should I do here?

ken sheridan say: What you see in the Company column in the Employees table is not its value but the value in the non-key column in the Companies table in the row to which the row in Employees maps via the relationship.  The actual value in the Company column in Employees is a long integer number.  The Companies table probably has a numeric primary key column such as CompanyID, most likely an autonumber.  It is this column which the Company column in Employees references, not the non-key Company column.  When joining the tables in a query the join should be to the primary key CompanyID column.  The non-key Company column from Companies can be returned in the query.

If you used the 'lookup field' wizard when designing the employees you should note that this feature is generally deprecated by experienced MS Access developers.  For reasons why see:

http://www.mvps.org/access/lookupfields.htm

In addition to the many other problems outlined at the above link, the naming of the foreign key column in the referencing table with a name which differs from that of the primary key column of the referenced table only serves to compound the confusion.  Don't worry unduly if you have used this feature; just be aware of the consequences and make a mental note not to use it in future.  Exactly the same functionality can be achieved by using a combo box as the control bound to the foreign key column, in a form based on the referencing table.  Data should always be entered via a form, never in a table's raw datasheet view, so the 'lookup field' wizard serves no useful purpose whatsoever.

marshall barton say:

As Ken said, the LinkMaster fields property needs to be whatever you named the company ID field.  Double check that this field is included in the main report's record source query.

 

The LinkChildFields property must be the name of the company lookup field and the lookup field's BoundColumn must use the number of the company ID field in the employees table.

 

I want to strongly endorse what Ken said about not using lookup fields in tables.

 

Your current problem is entirely caused by your confusion over what the lookup field is actually doing.  If you only use a combo box on the employees form to select the company, then you can and probably should change the lookup field to text box so you can see what is really going on in the table and avoid all this confusion.
