ken sheridan say: The following are the first two queries from my demo:

INSERT INTO Countries ( Country )
SELECT DISTINCT Country
FROM MasterTable;

INSERT INTO Regions ( Region, CountryID )
SELECT DISTINCT Region, CountryID
FROM MasterTable INNER JOIN Countries
ON MasterTable.Country=Countries.Country;

The first simply uses the DISTINCT predicate to insert one row per country into the Countries table, which has a CountryID autonumber primary key column and a Country text column.  County is analogous to Company in your case.

The second joins the newly populated Countries table to MasterTable on the text Country column, and inserts distinct values of the Region text column from MasterTable and the CountryID from Countries into The Regions table.  The primary key of Regions is an autonumber RegionID column.  Region is analogous to the contact in your case, so you'd insert the contact's names and other attributes.  It's important that the join is on a candidate key of Countries, which Country is of course, as all country names are distinct.  If you have some distinct attribute of Contacts, such as their phone number, you can join on that.  Otherwise you will need to identify a composite candidate key.  Using personal names is unsafe as they can legitimately be duplicated, so you would need to identify some other combination of attributes, possibly including the personal names, which make up a candidate key and join the tables on those columns.

ken sheridan say: I assume that ImportID is the primary key of the non-normalized Sheet1 table.  You can forget about that column.  Give both tblCompany and tblContacts an autonumber primary key, named CompanyID and ContactID respectively, and include a CompanyID foreign key column of long integer data type in tblContacts.

The first query to be executed is a very simple one:

INSERT INTO tblCompany([Company Name], DataTag, Address1)
SELECT DISTINCT [Company Name], [Data Tag], Address1
FROM Sheet1;

This should insert one row per company, with the CompanyID column's values being inserted automatically.  This assumes that the values of Company Name, DataTag and Address1 in Sheet1 are the same per company, with no slight inconsistencies as a result of a typo for instance.  If any company is duplicated look for small inconsistencies in one or more of these columns.

The second query needs to get the CompanyID value from tblCompany, and the other values from Sheet1, so if we assume that no two companies have the same Company Name value we can join the tables on Company Name:

INSERT INTO tblContacts([First Name], Surname, Position, CompanyID)
SELECT [First Name], Surname, Position, CompanyID
FROM tblCompany INNER JOIN Sheet1
ON tblCompany.[Company Name] = Sheet1.[Company Name];

Note that I have not included the DataTag column in this query, as I've assumed that its an attribute of companies rather than contacts, and have consequently included it in the first query.  However, it's not possible to know this on the basis of the column name, so if I've guessed wrongly, and it is in fact an attribute of contacts, not companies, it should be removed from the first query and included in the second.

Also note that, unlike the example from my demo, in your case the second query does not need the DISTINCT predicate, as each contact is represented by one row only in Sheet1.

And yes, you can create these queries in design view. The only thing out of the ordinary is that you would need to set the first query's Unique Values property to 'Yes' in its property sheet.  This is the equivalent of the DISTINCT predicate in SQL view.

doug robbins - mvp office apps & services (word) say:

You could use a macro containing the following code

Dim i As Long
With ActiveDocument
    .Unprotect
    For i = .Fields.count To 1 Step -1
        If .Fields(i).Type = wdFieldFormTextInput And .Fields(i).Result = "" Then
            .Fields(i).Delete
        End If
    Next i
    .Protect wdAllowOnlyFormFields, NoReset
End With
