davidf2 say:

There is a compatibility issue with IE8 and Publisher 2003 and 2007. Any
design elements that are 'grouped' together, which includes the Publisher
wizard built navbars, do not render when you view the web page in IE8 . The
fix in general is to ungroup the elements. There is both a manual fix to
these issues and a Service Patch that has been issued to fix it for Pub
2007.

Reference: Navigation bars and other content is missing from Publisher HTML
output in Internet Explorer 8: http://support.microsoft.com/kb/969705

A manual method of fixing this grouping issue:

After making and saving any changes to your Pub file, and prior to uploading
new web files, make a copy of your publication by doing a 'File > Save As'
and in this copy go to each page > Edit > Select All > Arrange > Ungroup.
This will ungroup the Publisher built navbar and disconnect it from the
wizard, and the navbars will render correctly in IE8. 'Publish to the Web'
from this copy of your publication. When you want to make further changes in
your web, go back to the original Publisher file, make the corrections
there, save your changes, and again make a copy, ungroup the
navbars and produce new web files for uploading. The advantage of this
workflow is that you will not have to rebuild the navbar if you choose to
add a page to the navbar. If you do not need to add a page or section to
your site, you can leave the navbar ungrouped and skip the step of saving a
copy.


Publisher 2007 can be fixed manually or with the Office 2007 SP2:

Reference: Description of 2007 Microsoft Office Suite Service Pack 2 (SP2)
and of Microsoft Office Language Pack 2007 SP2:
http://support.microsoft.com/?kbid=953195

Some users have found SP2 breaks Publisher, and makes it impossible to open
existing Pub files...*Publisher cannot open this file Error 1100009.*...
you know, break one thing to fix another, so there is a hotfix for that:

Description of the Publisher 2007 hotfix package (Publisher-x-none.msp): October
27, 2009
http://support.microsoft.com/default.aspx/kb/976177/en-us

3) Changing the numbering system that is used with a style: yes, that's customisations. And it that can cause the problems you are having. That said, it's an essential function for using Word, it can't be avoided. Where it is likely to cause problems is if you assign a single-level bullet or numbered format to a style that is a member of an outline numbering style. The only built-in styles this affects are the "Heading 1" through "Heading 9" series. Provided you only assign Outline numbering schemes to them, and do it only by using Format.StyleModify... you will not cause trouble.

4) On the Standard toolbar, about three cons in from the right-hand end there is a "Show/Hide Toolbox" button. Click to show, click again to hide. Word with it hidden: reveal it only when you need it, and close it immediately.

5) I will allow you Command + B for Bold instead of the Strong style, and Command + i for Italic instead of the Emphasis style if you must. Personally, I assigned those built-in character styles to the keystrokes, so I apply the styles without thinking about it.

So no: It's not cumbersome, it is dramatically faster than any other method, but it is something you need to make a conscious decision to do in the first day or two, until it comes naturally.

The Format Painter works perfectly. But the design of it leads to document corruption.

When it copies, it can copy only the formatting that is available in the text selected. If you select some words in a paragraph and click the format painter, it can't see the paragraph formatting unless the paragraph mark is selected. When you apply the Format Painter, it cannot apply paragraph formatting unless the paragraph as a whole is selected. It can't copy a Table Style, but it can copy parts of the formatting properties of a table. It can't copy all of a list style...

My rule with the format painter is that I might use it once or twice only, in a document that is not going to be saved. I would never use it in a commercial document. Remember Word gives you the ability to perform hack-and-chop editing in the middle of a file of computer code. It shouldn't, but it does. If you want your documents to last, you need to edit with care and precision.

For me: All of the styles I use regularly (there's actually only about 20 that I use regularly, only 10 that are common enough to put on the toolbar...) are either keystrokes or on the toolbar, so applying the style is at least three times faster than using the Format Painter.

"Click in the paragraph, click the style, done!" I never have to look at or think about the formatting: if the name of the style is correct, the formatting is correct, I don't have to check it. Remember, I do this for a living, and time is money.

Yes, it might take you three days to learn to format only with styles. But for the rest of your life, that means document production will happen in one eighth to one tenth of the time. For every document.

Assuming you are getting a degree because you want to become a knowledgeworker/leader/manager in the future, I am sad to say you are going to handle a LOT of "documents" before you get your gold watch! I know this because my gold watch is due next year!

7) Understand the difference between a carriage return and a paragraph.

A carriage return (Shift + Enter) is just that: an instruction to begin a new line. It's a character in the text.

A Paragraph is a container structure in the code; that holds as many as 1,200 formatting properties applying to the preceding text.

The paragraph mark holds the formatting for the text BEFORE it. When you delete one, Word would normally simply apply whatever formatting was in the following paragraph mark. Users didn't like that, so Microsoft changed Word to play the formatting of the deleted paragraph mark forward. This immediately gets the programmer who wrote the code into the Hurt Locker. The possibilities and exceptions are endless. One of the most popular times for Word to crash is when you delete a paragraph mark, especially one adjacent to a table, graphic, equation or other complex structure. Add in change tracking so the paragraph mark is only "kinda deleted" and all  h.e.l.l. breaks lose.

I too will sometimes whack a few "returns" in there to make room to do something. Theoretically, that should not matter because all of the paragraph marks will have the same formatting, so if you delete one, there should be no differences to compute.

I still do it very carefully, because I am a little gun-shy from years gone by :-) It's been a few years since I have had a crash (about three or four, I think...) but I am good at carrying a grudge! While my opinion and your opinion coincide quite closely, we really should not bore the other forum members rigid with our mutual appreciation :-)

Use HelpSend Feedback in Word to send your opinion in to Microsoft, where it will actually do some good. That automatically adds it to the Bug Database. The more people who add the same complaint, the higher up the fix list it travels. With enough votes, it will get into the bucket where a developer will be assigned to fix it. Of course, if you say what you really think, the Naughty Word Filter will drop it sight-unseen into the bit bucket. So express yourself freely, just keep it polite :-)

8) Yes, of course Endnote is the best citation manager. However, you need to know that it does stress the document and problems are common unless you are very careful. Microsoft has fallen out with Endnote, and now include their own tool in Word. As a result, I am afraid we can't expect Endnote to get much more reliable.

9) Templates that start off in .doc start off broken. You should probably make this point to your University. I don't believe there are any versions of Word still in use that can't open a .docx, although some of them may require the free converter Microsoft supplies (which is better than a converter: it is actually the File I/O module from Word 2011, so it allows output also).

There is a myth out there that .doc and .docx formats are equivalent. They are not, and never were. There are many things a .docx can store that have to be downgraded and stored as something else in a .doc because the .doc can't store the original data type. Whenever a document is cross-saved between .doc and .docx, a few more errors creep in. Until eventually the document corrupts and begins to misbehave.

Ideally you should start off in the format you intend to use. If you can't, save it into .docx then make sure it stays that way.

10) Since I have not actually seen the Word 2011 source code, I can't tell you WHY Word blows up in a feature you are not using, every time you insert a caption. But the way a modern application is constructed, with code shared and re-used all over the place, it doesn't surprise me.

You say you never use floating graphics? Great! Instead of selecting the graphic and then using Insert Caption, try this:

Click AFTER the graphic, add a paragraph mark, THEN use InsertCaption.

I think you will find you will get your caption, with the correct number, and no crashes.

11) The cleanup method Microsoft always recommends is to save the document as RTF, then re-start Word and save the RTF version back to .docx. That should always be tried first, because it does less damage.

The one I recommend is the Maggie, which will hammer the tracked changes, but should keep everything else:

The Maggie:
1. Create a new blank document in .docx format
2. Carefully select all of the text in the bad document EXCEPT the last paragraph mark
3. Copy it.
4. Paste in the new document.
5. Save under a new file name and close all, then re-open.

This technique for de-corrupting is known as "Doing a 'Maggie'", after Margaret Secara from the TECHWR-L mailing list, who first publicised the technique.

Conditional format if #value! in another cell? - Microsoft Community

gebobs say:

Conditional format if #value! in another cell?

Operators enter a part number into a column in a table called SKU.

e.g, BORT14-200X50

The nominal dimensions are calculated from this using some simple MID functions. In this case, thickness is 0.200" and width is 50". 

e..g. =MID([SKU],8,3) 

Operators then enter measured dimensions. These are compared to the calculated nominal and are highlighted if they fall outside of specification (i.e. nominal  specification).

One potential problem is if the operators enter the part number incorrectly. Simply adding an errant character results in a #VALUE! in the calculated fields and the the measured dimensions will not be conditionally formatted if they are out of spec.

So my first thought was to conditionally format the SKU field if the corresponding nominal fields have #VALUE! errors , but I couldn't wrap my puny little brain around just how to do that. I also tried to do it if the nominal value was not greater than zero, but that didn't seem to work.

My next idea was to do some data validation though it doesn't seem like there are any text options other than text length and this won't work because part numbers aren't always identical in that respect i.e. sometimes there are additional characters at the end.

TIA

hansv mvp say:

Select the range where part numbers are entered. I'll assume in the following that this range starts in row 2.

 

On the Home tab of the ribbon, click Conditional Formatting > New Rule...

Select 'Use a formula to determine which cells to format',

 

Let's say the calculated fields are in columns K and M.

Enter the formula =OR(ISERROR(K2), ISERROR(M2))

(The 2 here is the start row; if the data start in row 5, for example, change 2 to 5 in the formula)

Click Format...

Specify the desired formatting.

Click OK twice.

hansv mvp say:
Word doesn't display font, font size, font styles (bold, italic etc.), text alignment, left and right indents, line spacing, paragraph style etc. if more than 50 paragraphs have been selected, even if all selected text has the same formatting. If the selection contains 50 paragraphs or less, characteristics that are the same for all selected text are displayed in the ribbon and in dialogs.

hansv mvp say:

Apparently you have track changes turned on.

Activate the Review tab of the ribbon.

If the Track Changes button is highlighted, click it to turn it off.

Click the lower half of the Accept button and select "Accept all changes in document".

That should get rid of the text in the margin.
