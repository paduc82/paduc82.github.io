I3E754 describes how to convert decimal values to binary. Where this is possible for all whole numbers, it isn't for all decimal values.

The internal representations for:

0.07

is 3FB1EB851EB851EC

converting back to decimal gives:

7.0000000000000006661338147750939242541790008544921875E-2

0.05 becomes

5.000000000000000277555756156289135105907917022705078125E-2

Although Excel displays only 15 significant figures, the I3E754 value is used in all calculations.

Excel has an heuristic issue with rounding values to "close enough".

But I was referring specifically to the anomaly that A1-A2-A3 might be exactly zero, but A1-A2-A3-0 is not, which was

The result of the first formula (A1-A2-A3) is due to an Excel heuristic.  It is poorly and incompletely described in KB 78113 (http://support.microsoft.com/kb/78113) under the misleading title "Example when a value reaches zero", to wit:

``Excel 97, however, introduced an optimization that attempts to correct for this problem. The example above when performed in Excel 97 and later correctly displays 0 or 0.000000000000000E+00 in scientific notation.``

That heuristic is not specified in IEEE 754.  And I don't know of any other product that performs such "corrections".

Moreover, Excel applies the heuristic inconsistently, which is demonstrated by the second formula (A1-A2-A3-0).

Also, using your example where A6 is =A1-A2-A3, =A6=0 returns TRUE, but =A1-A2-A3=0 returns FALSE.  That is because Excel applies the "last subtraction correction" only to formulas.

The SUM function is also inconsistent.  SUM(A1,-A2,-A3) returns an infinitesimal difference. But if B1 is =A1, B2 is =-A2 and B3 is =-A3, SUM(B1:B3) is exactly zero.

-----

As for normal arithmetic, both Excel and VBA follow the same "rules" insofar as they both rely on the CPU to perform the arithmetic.  Most modern CPUs follow IEEE 754 for 32-bit and 64-bit binary floating-point arithmetic.

However, there is a subtle difference between Excel and VBA arithmetic on Intel-compatible CPUs (almost all computers, nowadays).  These CPUs actually perform the arithmetic using an 80-bit binary floating-point representation and 80-bit FP registers.  Excel rounds each pairwise operation to 64-bit binary.  But for explicitly type Double operands, VBA tries to use the 80-bit result in FP registers.  (VBA does not do that for type Variant operands, even if the subtype is Double.)

Consequently, Excel and VBA can yield different results.  For example, with 1.5 in A1 and 0.05 in A2:

=A1-A2*INT(A1/A2) returns exactly zero (0.00E+00)

=vbmod(A1,A2) displays 0.0499999999999999

=vbmodVariant(A1,A2) returns exactly zero

Function vbMod(x As Double, y As Double) As Double
    vbMod = x - y * Int(x / y)
End Function

Function vbModVariant(x As Variant, y As Variant) As Variant
    vbModVariant = x - y * Int(x / y)
End Function
