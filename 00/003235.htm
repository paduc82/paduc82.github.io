scottgem say:

This would be better solved with properly normalizing your data. What you should have is a child table:

tblProjectRoles: ProjectRoleID (PK Autonumber), ProjectID (FK), RoleID (FK), EmployeeID (FK)

You would also need a Roles table:

tluRoles: RoleID (PK Autonumber), Role

With this structure, you simply query this table for EmployeeID and you get all projects the employee had a role in and what their role was.

I would recommend redesigning your database. It would be fairly easy to create these tables and populate them using Append queries. If you need more help with that feel free to ask.

Another option is to use a UNION query to get the information.

SELECT [Partner in Charge] as Person, Project

ken sheridan say: Having separate columns for each role in the project is not a good design in a relational database.  What you really have here is a many-to-many relationship type between Employees and Projects, with the role in the project as an attribute of the relationship type.  This type of relationship is modelled by a third table, sometimes called a 'junction' table, which has two or more foreign keys, respectively referencing the primary keys of the related tables, so the table would be like so:

ProjectEmployees
....ProjectID
....EmployeeID
....RoleID

The primary key of this table is a composite one made up of all three columns.  If you wish to constrain the table to allowing only one employee of each role per project then a unique index should also be created on the ProjectID and RoleID columns in combination.

In fact the relationship type is a ternary (3-way) one as Roles is also a party to the relationship, so you'd also have a table:

Roles
....RoleID
....Role

With this structure you are now correctly representing all data as values at column positions in tables, which is the only way in which data should be represented, not encoded as column headings as you are currently doing.  This accords with one of the basic principles of the database relational model, the Information Rule, which was Codd's Rule #1 when he introduced the relational model for databases in 1970.  Form a more practical point of view, you can of course make provision for other rolls in a project simply by adding new rows to the Roles table.

To find out who participates in each project as what join the four tables in a query, with ProjectEmployees joined to each of the other three.  To restrict this to a specific employee include a parameter on the EmployeeID column.

Ken Sheridan, Stafford, England

aussielaurie say: Good news - I was having the same probs using the same things as you were, BUT on a Google search came across a DIFFERENT link to the Nokia Europe site which fixed all my probs.

http://europe.nokia.com/A4423134


Once I loaded the Multimedia Transfer, installed it, followed all the instructions in this link http://europe.nokia.com/A4372155

by some miracle I connected.

Don't ask how. I have been trying the various Ross Barkman scripts for HSDPA Nokias for 24 hours, but to no avail.

I am pretty pleased. But you really must follow the instructions to the letter. I am using Leopard, btw. In the System Prefs > Network Prefs, the Location I left at Automatic.
Then I made a new Configuration which I called Nokia USB. (Something about having installed the Multimedia software made the Mac recognise that A Nokia 6120 Classic was "there".)

Note, I am on the (Australian) Telstra NextG network, so I left all the fields - Phone number, Account name and Password - blank.

TICK - show modem status in Menu Bar.

Press ADVANCED.
The Modem tab - select Nokia as the Vendor.
The only option under Model says GPRS (GSM/3G).
For a casual telstra datapack, the idea is to put telstra.datapack in the APN field.
Underneath is CID and the only option is 1.

All the other Tabs are ok as is.

I pressed OK. Then the Connect button and I was in.

I will test the same idea using the Barkman scripts later and check speeds. I should be getting upwards of 1.8 Mbps for downloads where I am right now.
