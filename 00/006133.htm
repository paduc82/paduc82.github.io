john w. vinson say:

There's nothing evil about Null. You just have to understand what it means, and how Access handles it!

 

A NULL value should be interpreted as "this value is unknown, unspecified, not defined". It's not zero; it's not a blank text string (though that is how it will usually be displayed). As such, any arithmetic operation involving a NULL returns NULL.

 

That is, if you have FieldA containing 13, and FieldB which is NULL, then the expression

 

FieldA + FieldB

 

will be NULL. It's 13 plus some value that you don't know - so you don't know the sum either!

 

You can use the builtin NZ() function to give you a zero (or actually any other value) for a null:

 

FieldA + NZ(FieldB)

 

will return 13 if FieldB is NULL, or 19 if FieldB is equal to 6.

 

FieldA * NZ(FieldB, 1)

 

will multiply by 1 if FieldB is null.

 

One handy thing is that if you are using SUM in a query to sum multiple rows, Access silently converts all the NULL rows to 0 (or maybe just excludes them from the sum, the result is the same): i.e. if you have rows with 1, 2, 3, NULL, 4, NULL and use Sum or DSum to add them up you just get 10, ignoring the nulls.

 

So - you can allow NULLS in your table fields or you can forbid them (by making the field required); you can use a default value of 0 if that makes sense in your business model, or not use any default and deal with the NULLs, or make the field required and don't use a default if the value must have something filled in. I've done all three depending on the business need.

 

hansv mvp say:

There is nothing wrong with Null values. Null means 'not filled in', and 0 means 'the value is really zero'.

 

In Access 2003 and before, Access would set the default value of newly created number and currency fields to 0, but in Access 2007 and later, no default value is set automatically, and rightly so in my opinion. I prefer to leave the default value blank unless there is a very good reason to have a specific default (usually not 0).

 

If you sum a field that contains a mixture of 'real' values and Nulls, Access will ignore the Nulls and add the 'real' values together. Only if ALL values are Null, the sum will be Null too. In some situations, you'd rather have a sum of 0 in such a situation. You can do that by using the Nz function, e.g. Val(Nz(Sum([SomeField],0)). This will return 0 even if SomeField contains only Nulls.
