SELECT MIKE.[Cust#], MIKE.AutoKey, MIKE.PCd, Sum(IIf([XX].[PCd]<=[MIKE].[PCd],1,0)) AS RecNUM

SELECT MIKE.AutoKey, MIKE.[Cust#],  MIKE.PCd, "PCd" AS MyField, MIKE_RecordNUM.RecNUM

TRANSFORM First(MIKE_RecordNUM_Split.[PCd]) AS FirstOfPCd

GROUP BY MIKE_RecordNUM_Split.[Cust#], MIKE.[Sale#], MIKE.Age, MIKE.Gender
PIVOT [RecNUM] & " - " & [MyField];

ken sheridan say: Why do you want to convert a table which is not normalized to Third Normal Form to one which is not even normalized to First Normal Form?  By the look of it my guess would that this is data imported from Excel, but whatever its origin the correct model would require the decomposition of the first table into three tables, e.g.

Customers
....Cust# (PK)
....Age
....Gender

Sales
....Sale# (PK)
....Cust#

SaleDetails
....Sale#
....In#
....PCd
....SlsD

From the data you've posted there is no clear candidate key for SaleDetails; it could be a combination of Sale# and any or all of the other three column.

To decompose the first table into these thee tables is a simple task:

INSERT INTO Customers([Cust#], Age, Gender)
SELECT DISTINCT [Cust#], Age, Gender
FROM [The FirstTable]
WHERE [Cust#] IS NOT NULL;

INSERT INTO Sales([Sale#],[Cust#])
SELECT DISTINCT [Sale#], [Cust#]
FROM [The FirstTable]
WHERE [Cust#] IS NOT NULL;

INSERT INTO SaleDetails([Sale#], [In#], [PCd],SlsD)
SELECT [Sale#], [In#], [PCd],SlsD
FROM [The FirstTable]
WHERE [Cust#] IS NOT NULL;

BTW a customer's age stored in a table will soon be out of date.  Normally a person's date of birth would be stored and the current age computed with an expression or a function such as:

Public Function GetAge(varDoB As Variant, Optional varAgeAt As Variant) As Variant

    If IsMissing(varAgeAt) Then varAgeAt = VBA.Date
    
    GetAge = DateDiff("yyyy", varDoB, varAgeAt) - _
        IIf(Format(varAgeAt, "mmdd") < Format(varDoB, "mmdd"), 1, 0)

End Function

With a correctly normalized model the data can be returned in the sort of format you indicate in a number of ways.  One would be to concatenate the values into a single string expression per customer, using one of the various concatenation functions available online.  My preferred solution for presenting data in this way is, wherever possible, to use a multi column subreport, with across-the-down column layout, in a parent report which returns one row per customer.  You'll find an example as ClubMembership.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

This little demo file lists all residents at each address horizontally in the rptAddress_Sub report.  The other report, rptAddress_Code, does the same in a single report by manipulating the layout with code in the report's module at runtime.  The subreport solution is far easier, though.  In the same folder is Concat.zip, which includes a means of concatenating data from a related table by means of the highly efficient GetString method of the ADO recordset object.

You'd need to create event handlers for the NewWorkbook and WorkbookOpen event of the Application event in an add-in. This requires a special syntax - see http://support.microsoft.com/kb/213566.

Dim excelObject as New Class1

Sub runHandlers()
Set excelObject.appEvent = Application

Public WithEvents appEvent as Application

Private Sub appEvent_NewWorkbook(ByVal Wb as Workbook)
Msgbox "New Workbook"

Call runHandlers from the add-in's Workbook_Open event procedure.
