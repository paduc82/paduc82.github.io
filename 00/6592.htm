scottgem say:

"Why should data not be in more then one table?  "

Because this is a fundamental principle of relational database design. The principles of designing relational databases were first put forth in the early 1970s. They have stood, largely unchanged since then. So they have stood the test of time. One of the reasons for this is to prevent update anomalies. Lets say a person changes their name. if you are storing the name in multiple tables you have to change it every place.

"As for the LookUp function, I am not sure what "Form level using list controls"

Users should never interface with tables directly. All interaction with data should be done on forms. Therefore, using lookup fields in tables can cause problems. If you want to do a lookup (and lookups are an important tool) then should be done on forms using list controls like comboboxes, listboxes and option groups. 

"Producing reports from Queries is a great idea"

Its not just a great idea, its the only idea. In a relational database, reporting often requires the use of multiple tables (because data is stored in one table only). So queries are necessary to reporting. 

" how can I create a master table to enter the data I need, when only certain items apply to certain sections?  Meaning that the columns for Transfers are completely different from the Receipts?"

This is why we need to see your complete table structure and understand your application. Two possibilities come to mind. If you need to capture different data for Transfers than Receipts, then you may need to use at table in a one:one relationship. With a separate table for Transfer and a separate table for Receipts. 

But looking at your Master Table that would need to be changed:

Master Tracker Entry, Table

Column 1: MasterID (Unique to Master Tracker Entry)

Column 2: ReportedByID  Select from a table of people who will report issues

Column 3: SectionID This would pull the ID field from a Lookup table of Sections

Column 4: ClerkID Select from a table of clerks.
Column 5: Issue  (Fully editable to type in exact issue notes)

Column 6: Assigned (DateTime)

Column 7: Due Date (DateTime)

Column 8: 10 Day Warning (DateTime)

Column 9: Overdue (DateTime)

You probably don't need 8 and 9 as they should be calculated according to your business rules. Fields 2-4 are Foreign Keys that relate back to other tables. " If the issue is for a Transfer, can I make it so I can click Transfer and only have to enter in the data required for Transfers?"

Yes, definitely. If you select Transfer as the Section, you can make visible a subform for entering Transfer data, If a different Section value is selected then you make visible the appropriate subform.

jcrobertson say:

SOLVED!!  Here's what I did to correct the problem in my original post.

1. Back up any important documents such as certificate keys and other important stuff as THE FOLLOWING STEPS WILL DELETE ANY DATA NOT BACKED UP!

2. Create second computer account and assign it administrator access. Reboot.

3. Sign into this just created administrator account. Go to File Explorer.

4. In the left pane, right click on "This PC" and select "Properties"

5. In the left pane of the window that appears, select "Advanced system settings."

6. In the System Properties window that appears, click "Settings..." within the "User Profiles" frame.

7. Select the profile whose Groove Music app is not working and select delete.

8. Return to the File Explorer and in the left pane, right click on "This PC" and select "Manage"

9. Expand "Local Users and Groups" and click on "Users."

10. Right-click on the account whose Groove Music app is not working and select delete.

11. Click Start, then "Settings" and then select "Accounts"

12. In the left pane, select "Family & other users".

13. There you can add back the recently deleted account.

14. Reboot and sign into the re-created account..  Groove Music should now work.
