john w. vinson say:

If you are using just one copy of the single database with multiple users connecting to it by shortcuts... you MUST Split the database. The "server" (and a shared-drive PC running desktop Access is fine) should have a "backend" database (which can be a 2013 .accdb or a 2003 format .mdb, 2013 can handle either) containing only the tables; each user should have their own private copy of the frontend, a database containing the forms, reports, queries and code linked to the shared backend. BinGoogle for "split Access

This was a good idea in 2003; it's a lot more vital with newer versions, and could explain your problems.

scottgem say:

In addition to what John said (which I agree with 110%.), When you split the database you need to link the tables using UNC coding (not a mapped drive letter). In a workgroup setting, the "server" PC will have a different mapping so the links have to be by computer name.

P.S. I can understand you wanted to get them on a domain, but for their situation (based on what you have said) I don't really see the need for it. While there are advantages to a domain based network, those advantages are more in larger organizations. a 5 user setup can get along fine with a workgroup. What I WOULD recommend to them is a NAS. Let them put their Access back end and shared files on the NAS. Make for easier backups and everyone can then use the same mapped drive letter. The cost would be small (You can get a 3 Terabyte My Cloud from Western Digital for around $200 as an example) and the benefits large.

scottgem say:

Access includes a splitter wizard. I forget where this is for 2013 (I think its on the Database Tools ribbon under Move Data (Access)). When you run the wizard (have a backup copy first).it will prompt you to select the location to store the back end. Instead of picking a drive letter, scroll down to Network section in the Left pane and select the computer that will be serving as your server (If you do use a NAS, it should still be listed there). Then select the name of the shared folder (that you create beforehand) and navigate to the folder where you want to store the back end. This will create the link as:

\\computername\sharename\foldername\filename

That is called UNC coding. So no matter what drive letter the user has mapped to that share, Access will find it.

Since the "server" is also a used PC (unless you go NAS), it should have the version of Access that the user will use. So no you shouldn't install 2003 on it. jay freedman say:

macropod is the undisputed genius of complex fields, and I'm in awe. The advantage of doing the job that way is that once the main merge document is set up with those fields, you never have to do anything after the merge.

There is another way, simpler to explain but requiring you to run a Replace (manually or with a macro) on the merge result document.

Manually, you turn on the Use Wildcards option in the expanded ("More") Replace dialog. In the Find What box, enter the expression (I recommend using copy/paste from this post)

\(([0-9]{3})\) ([0-9]{3})-([0-9]{4})

This will match any phone number in the format (###) ###-#### . The three pairs of parentheses that are not preceded by a backslash will mark the three sets of numbers as matched number groups.

In the Replace With box, enter the expression

\1\2\3

This says to use the three groups stuck together. Notice that the backslashed parentheses, space, and dash in the Find What expression are not included in any group, so they're dropped.

Click the Replace All button.

As a macro, this process is coded as

Sub FormatPhoneNumbers()
    Dim rg As Range
    Set rg = ActiveDocument.Range
    With rg.Find
        .MatchWildcards = True
        .Text = "\(([0-9]{3})\) ([0-9]{3})-([0-9]{4})"
        .Replacement.Text = "\1\2\3"
        .Execute Replace:=wdReplaceAll
    End With
End Sub

macropod say:

I should have included that this is not a one-time mail merge. It would be an ongoing process for multiple users of the CRM.

Setting up the field code I described is a once-off process; once added to your mailmerge main document, it will automatically strip off the formatting from every merge you run without the need for a macro. The macro I referred to in my previous post was only for the once-off conversion of the text representation of the field code I posted to a working field; you can create the field manually if you'd prefer a character-building exercise.

Conversely, for a process that would would be seamless for the users and doesn't require the complex field code I posted, add the following macro to your mailmerge main document. From then on, clicking on the 'Edit Individual Documents' button will intercept the merge, finishing it and reformatting the output.

Sub MailMergeToDoc()
Application.ScreenUpdating = False
ActiveDocument.MailMerge.Execute
With ActiveDocument.Range.Find
  .MatchWildcards = True
  .Text = "\(([0-9]{3})\) ([0-9]{3})-([0-9]{4})"
  .Replacement.Text = "\1\2\3"
  .Execute Replace:=wdReplaceAll
End With
Application.ScreenUpdating = True
End Sub

If you're using Word 2007 or later, your mailmerge main document will need to be saved in the .doc or .docm format, as documents using the .docx format cannot contain macros. The only potential disadvantage of intercepting the 'Edit Individual Documents' process this way is that you no longer get to choose which records to merge at that stage. However, you can still achieve the same outcome - and with greater control - via the 'Edit Recipient List' tools.
