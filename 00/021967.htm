ACCESS 2010 How to autopopulate a field? There is one-to-one - Microsoft Community

margar_inthecity say:

ACCESS 2010 How to autopopulate a field? There is one-to-one relationship

I didn't see an answer already, so here goes:

I am working with Access 2010, and I am trying to enter two different types of data about the same customer base into two separate tables.  One table is called "CustomerStats" and contains information about current internet provider, signal source, customer referral, etc. for marketing purposes, and the second table is called "CustomerContact" and contains street address, cell number, etc.  I created (so I thought) a one-to-one relationship between these two tables where the Primary Key in both is called "CustomerID".  I am hoping to make it so the "CustomerContact" table AND the "CustomerStats" table can both be autopopulated.  I don't want it to be that one table is considered the MainForm and the other the SubForm.  I would like to be able to add a new customer either by contact info or signal info, and know that the CustomerID will autopopulate the customer's name into the other table.  What do I need to do, and is this even possible?  I am not a professional programmer, but I have been hired to create a database for a client who has given me no particular timeframe.

Thanks in advance.

ken sheridan say: Margaret:

A one-to-many and many-to-one relationship type is just the same thing looked at from different  directions.  Where a customer can have more than one address the relationship is one-to-many form customers to addresses, and many-to-one from addresses to customers.

With things like addresses, however, you should first ask yourself whether it could be a many-to-many relationship type, which would be the case if one customer could have more than one address, and one address can be shared by more than one customer.  I'll come back to this later, but first let's deal with a one-to-many relationship, where a customer can have more than one address, but each address applies to one customer only:

In a case like this you'd have two tables, Customers and Addresses in a one-to-many relationship.  The Customers table would have a primary key, CustomerID say, and other columns (fields) each of which represents an attribute of the customer you store once only, e.g FirstName, LastName etc.  The Addresses table would have the usual address columns plus a foreign key CustomerID column.  While the primary key CustomerID column in Customers can for convenience be an autonumber, that in Addresses would be a straightforward number (long integer) data type.  You can add an AddressID primary key autonumber column to the table, however.  The relationship diagrammatically would be like this:

Customers----<Addresses

In the second scenario of a many-to-many relationship type you'd have the same two tables, but the Addresses table would not have a CustomerID foreign key column.  This is because a many-to-many relationship type is modelled by a third table, CustomerAddresses say, sometimes called a 'junction' table.  This would have two foreign key columns, CustomerID and AddressID (not autonumbers).  This table forms the bridge between the two main tables , so the relationship diagrammatically would be like this:

Customers----<CustomerAddresses>----Addresses

As you see the many-to-many relationship type has been resolved into two one-to-many relationship types, which is how a many-to-many relationship type is always modelled, never directly between two tables.  In this case the relationship type is binary (2-way), but it could be ternary (3-way) or more, modelling a relationship type between three or more tables.

I said above the table which models a many-to-many relationship type contains two or more foreign key columns, but it can also contain other non-key columns.  These represent attributes not of the two or more main entity types which are related, but attributes of the relationship itself.  So you might have an AddressType column for instance, home address or work address say, or DateFrom and DateTo columns to record the date range when the address lies to a customer.  The fact that a relationship type can have attributes of its own illustrates that a relationship type is really just a special kind of entity type.

When it comes to data entry the most common way is to have a form, in single form view, based on the referenced table (Customers) in this case and a subform based on the referencing table (Addresses or CustomerAddresses depending on the relationship type).  The following link explains how this is done:

http://office.microsoft.com/en-us/access-help/create-a-form-that-contains-a-subform-a-one-to-many-form-HA010098674.aspx

As regards migrating existing data into a new model, this is almost always possible provided the information is there in the existing data, even if in a not very good form.  How complex this is depends on the complexity of the new model, but in your case it shouldn't present any insurmountable problems.  The process is essentially one of executing append and/or update tables based on the existing tables to populate new tables.  The key is to do this in the right order, which in essence means appending data to the referenced table(s) first; in your case Customers, so each row represents one customer.  This is followed by appending data to the referenced table(s), Addresses in your case, so that it contains one address and the correct CustomerID for each if the relationship type is on-to-many.  If its many-to-many you'd follow this by appending data to the CustomerAddresses table.  I did create a little demo file showing how this sort of thing is done some years ago, starting with an Excel worksheet.  It's not available on-line, however, but if you'd like a copy mail me at:

kenwsheridan<at>yahoo<dot>co<dot>uk

If it's simply a case of combining two tables into one then that is generally very easily achieved by joining them on the key (CustomerID) of each and appending the data to a new table, but as you do seem to need to store multiple addresses per customer then 'decomposition' (that really is the correct technical term!) as outlined above and illustrated by demo file, would appear to be the order of the day.
Ken Sheridan, Stafford, England
