petrib say: The device, in order to display Kanji, would have to have a font installed that has the Kanji characters/symboals/glyphs.

Alternatively you would have to install a new default/system font file that has the characters.

Or if you do not install it as a default/system font, you'd also have to rewrite and install a messaging app that uses your font.

By default, I don't think there'll be a Japanese/Kanji font in any Nokia device, except if it is intended for the Japanese market.

That (suitable fonts) is only enough for displaying text (and assuming it is already encoded using a suitable character set, such as Unicode).

And going further...

If you'd want to enter Kanji text, then there'd have to be a suitable input method editor (Front-End Processor) for doing that.

And then having applications also be in Japanese, for example, translated resource files would be required, too.

In addition, specific/separate versions of some apps like a Calendar handling non-Western ways of counting days may come into play, etc.

petrib say: S60/Symbian is built around the Unicode character set support, and Unicode encodings handle both Kanji, Katakana, Hiragana and every other modern and many dead languages and their characters too.

When an Unicode string has a character for which there is no symbol in the font, a blank square is shown.

I just finished doing something similar on a bunch of our apps. We have several YTD reports, but, of course in January, then want to show the current year rather than the previous year. So I added a column to my queries: YR: Year([datefield]). I then set the criteria for that column using an IIF that returns the previous year if we are in Jan.

Another good way to get year-to-date results at any time of the year is to use a criterion

>= DateSerial(Year(Date()), 1, 1)

Month-to-date would be similar -

>= DateSerial(Year(Date()), Month(Date()), 1)

DateSerial is a clever and useful function; you can even do things like

>= DateSerial(Year(Date()), Month(Date()) -1, 1) AND < DateSerial(Year(Date()), Month(Date()), 0)

to get the past month's data (even in January).

That would be an incorrect assumption about what was meant by "working as designed".   E.g. the key's default function may be tied specifically to the program that it is working with and not designed to have any more general functionality, such as the one that you are imagining it might have.  To check that idea you could run ProcMon to find out exactly what pressing that key does.   If you traced both cases, working and not working (e.g. pressing it in a different context) then it would probably be pretty obvious why the "not working" case was "not working" (i.e. not working by design.)

kathydake say:

After much researching and trial and error, we've found a fix.

In some cases, the scanpst option (from: support.microsoft.com/.../2652320) worked, but in most it did not. This worked in all situations:

1.Apply registry fix: www.reddit.com/.../cw2av3j

2.Rename the C:\Users\<User>\AppData\Local\Microsoft\Outlook folder to C:\Users\<User>\AppData\Local\Microsoft\Outlook.old

3.In Control Panel > Mail, delete ALL profiles

4.In Control Panel > Programs and Features > Office 2016 > Change > Online Repair

5.After repair is complete, open Outlook again and step through profile setup
