Good question... I can only guess...

If we call FIND from VBA we start the same code as we initiate a search from Excel.

The first ESC terminates the FIND but the macro continues, the second ESC is recognized by VBA and interrupts the code.

We can prevent this if we use CTRL-Break instead of ESC, this interrupts FIND and VBA at the same time.

EDIT:

I have spoken too quickly. Sometimes we see a difference here too. The only logical explanation is that the interrupt request permanently stops FIND to retrun a result, but VBA interrupts the code some times later.

Seems this is the time difference between the running code and the execution of message stack in Windows.

If we press Ctrl-break, then we send a message from the keyboard to the active window.

We press Ctrl-break while the code is inside Excel-FIND, FIND terminates an returns NOTHING, the macro continiues and calls FIND again. FIND is asked to stop and returns NOTHING again. That's the reason why we see the counter running, but no cells are changed from this moment.

Within the next timeslice Excel returns the control to windows and windows recognize that there is a message waiting for VBA and gives the control to VBA, and finally VBA interrupts the macro.

I guess something like this must happens... Multitasking processes...

Sub mkThemeCols()
