In A1, the first MATCH function returns 2 (location of "b2" in B1:B20), and the second MATCH function returns 5 (location of "c5" in C1:C20).  Their product is 10, so the INDEX function returns "a10", the 10th value in A1:A20.

The example also demonstrates a potential flaw in Mostofasohel's formula.

For example, my formula will fail for MATCH("b2",...)*MATCH("c11",...) because the product is 22, longer than the index range A1:A20.

Generally, the size of the index range must be the product of the two match ranges, unless circumstances ensure that the product of the matches is not larger than the index range.

In Mostofasohel's case, INDEX(...P$11:P$581,...) should be INDEX(...P$11:P$326051,...) because each match can return as much as 571, in theory.

And of course, that assumes both matches always succeed; that is, neither returns #N/A for "no match".

I suspect the product of MATCH's is not correct for Mostofasohel's design.  I wonder if Mostofasohel should use LOOKUP instead of MATCH.  And even then, perhaps not the product of LOOKUPs.
