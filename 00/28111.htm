Well first, you have design issues. Its hard to tell, but you need at least three tables here.

You have your contacts table, a plans table and a commissions table. Your commissions table should NOT have fields like Y1 Commission. That is known as a repeated group and violates normalization rules. The commission table should look like this:

tblCommission
CommissionID (PK Autonumber)
PlanID (FK)
CommissionType
CommissionPercent

So instead of fields for Y1 Commission, Y2 Commission, you would have records for Y1 Commission and Y2 Commission etc. 

Now, to do a report on the commission amounts, you would create a query that joins the tblCommision to tblPlan and calculates the commission amounts.

PK means primary key. PlanID is a Foreign key, So the data type for PlanID depends on the datatype of the Primary key of your Plans table. 

No, whenever you have fields that are numbered like CommissionPercent1, CommissionPercent2, etc. that is a repeating group. I suspect you are more used to working with spreadsheets then relational databases. CommissionType would identify the year or you could add a Year field if there are different types of commissions within a year. You may need a lookup table that just lists the commission types. In that case, you would select the commission type from a combo that uses the lookup table. 

You want a vertical structure where each record represents a type of commission rather than a horizontal structure where fields identify commissions.

As for the Plans table, it depends on your data. If each client has one plan and one plan only, then you would not need a separate plans table. and instead of using PlanID as a foreign key you could use ClientID. If a client can have multiple plans, then you need a separate Plans table with whatever info you need about each plan,
