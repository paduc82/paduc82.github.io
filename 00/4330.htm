adam vero say:

The answer to the first problem is, as others have stated to copy rather than move the cells.

 

The second problem of how to get someone else to do this (you have some empathy from me here!) is more tricky. I have had similar problems using spreadsheets to get people signed up to training courses (when I run in-house programmes for clients) and people want to change sessions so they drag and drop their name around. Major headaches for formulas and things like conditional formatting.

In your particular case you could change the formulas to use named ranges or positional functions rather than specific cells. Particularly the OFFSET function might help in the example you gave, or possiby INDEX combined with some named ranges.

By using OFFSET you specify a starting cell and then "walk" to the cell you really want - if you make sure to specify a starting cell which has been locked and therefore cannot ever be moved, your formula is safe. This will make some of your functions more complex though, unless you put the nasty OFFSET bit in a defined name and just use the name in your formula. If cell A1 is locked and therefore safe as a starting point, cell G2 would be OFFSET($A$1,1,7) - one row down and seven columns across.

You can return more than a single cell, so instead of =IF(AND(H2="",G2=""),""... you could use =IF(COUNTA(OFFSET($A$1,1,7,1,2)=0,""... this returns 1 row of two cells (G2 and H2), counts if they have anything in them and if there is nothing there (both blank) that's the same as your original function.

Also bear in mind that if you have a named range of a single cell, the name is redefined if the cell gets moved, but if you have a named range of two or more cells, it does not. So if you name cells G2:H2 as "InputValues" (or whatever) you could use this name (maybe in a COUNTA function as above). If either cell is moved, the name remains the same.

If both are moved together, the name would be redefined, but one way to prevent this is to include in the name another neighbouring cell which is locked (possibly a label cell). Now it is impossible for all three to be moved since one is locked, and the name is reliably fixed. (you might add a narrow spacer column or row in convenient places to help you do this.

If you want to get at only one value from your range use INDEX eg INDEX(ThreeInputValuesInARow,,3) would give the third column of the range (note the ,, denotes where the optional row parameter would go - for a single row range this is not needed or you put in a 1 for clarity).

If you have long columns of input values and calculations from values on the same row then you could define a name for a whole column ("price", "quantity") and then let Excel work out the implied intersection when you work out (cost) =Price*Quantity

Hope this gives you some ideas as a starting point to help lock down this sheet from accidental changes.

ken sheridan say: I'd suggest an alternative means of validation.  Firstly disallow all but numeric digits in txtOrderNumber by putting the following in its KeyPress event procedure:

    Select Case KeyAscii
        Case 48 To 57
        ' number entered, allow
        Case 8
        ' backspace entered, allow
        Case Else
        ' disallow
        KeyAscii = 0
    End Select

It's theoretically possible a value could be pasted in of course, so you can disallow this in its BeforeUpdate event procedure with:

     Const MESSAGETEXT = "Value must be a number"
     Dim ctrl As Control
     
     Set ctrl = Me.ActiveControl
     
     If Not IsNull(ctrl) Then
        If Not IsNumeric(ctrl) Then
            MsgBox MESSAGETEXT, vbExclamation, "Invalid Operation"
            Cancel = True
        End If
    End If

Change the code for the Command416 command button's Click event procedure to:

    Dim strFilter AsString

    strFilter = "Order_ID = " & Me.txtOrderNumber

    If Not IsNull(Me.txtOrderNumber) Then
        If IsNull(DLookup("Order_ID", "Orders", strFilter)) Then
            strMessage = "Quote not found"
                MsgBox strMessage, vbInformation, "Warning"
           Else
               Me.Filter = strFilter
           Me.FilterOn = True
         End If
    Else
        strMessage = "Please enter a valid quote number"
        MsgBox strMessage, vbInformation, "Invalid Operation"
     End If

Your applyFilter procedure now does nothing, so can be deleted.  As  side issue, however, note that local procedures or functions in a form's class module should be declared as Private unless you wish to expose them as a method or function of the class outside of the form.

Also, I'd recommend that you do not accept the default names like Command416 which Access gives to controls, but change the Name property of the control to something meaningful immediately after adding the control to the form.  If you do so after entering code in an event procedure the link between the control and the event procedure will be broken, and you'll have to delete the procedure and create it again.  Naming controls meaningfully provides a degree of self-documentation to the application.

The above assumes Orders is the name of the table on which the form is based, so you might need to change the name when calling the DLookup function.  It also assumes that Order_ID is a number data type.  If it's a text data type amend the code to:

    strFilter = "Order_ID = """ & Me.txtOrderNumber & """"

Of course, if, rather than a text box, you used a combo box with a RowSource property of:

SELELECT Order_ID FROM Orders ORDER BY Order_ID;

and set its NotInList property to True (Yes) you would not need to validate the value entered.

Use a form to modify the data.  Then you can use the form's BeforeUpdate event to set the last modified date field:
   Me.[last modified date field] = Now

If you also want a date created field in the record, set the table field's DefaultValue property to Now()
