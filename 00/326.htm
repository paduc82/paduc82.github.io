marshall barton say:

That's because the "other methods" are different in different situations.


It's not that the domain functions are any slower than running a query some other way.  The issue occurs when they are used in ways that require them to be used more often than might be needed if a different kind of query (probably a Totals type) were used.


So, I recommend you think about the algorithm you are using and try to find a way to set things up so a single query can calculate all the values instead of using lots of domain functions.  If you have trouble doing that, post back with a Copy/Paste of your VBA code and we'll try to suggest some alternative approaches.

marshall barton say:

That is a pretty complex compilation of data.  I don't have the time to totally unravel all that, so I'll make a few suggestions that may help a little or a lot.


Potentially huge, if you haven't done it already, is to make sure you have referential integrity enforced on the relationships between all those tables.  Also make sure you have an index on every field used in a WHERE clause and in the domain aggregate functions' where condition.


It may be significant to clean up the extraneous stuff in the queries.  Delete the ORDER BY clause in all but the last query.  Also, change the HAVING clause in Avail A to a WHERE clause (this can be done in the query designer by changing the strEmploymentStatus, strGeneralTitle and Leave_or_Training fields' Total row entry from GROUP BY to WHERE).  Those same fields and any other fields that are not used in any subsequent query should have their Show box unchecked.


Probably most important is the DSum in Avail E.  This is the kind of use of a subquery or domain aggregate function that really slows things down.   Remember that something like the sum of a sum over partial datasets is the same as the sum over the entire other fields needed in the final query (only the Sup Name field?) and join it to the first query to pick up all the aggregated values.

say:

Although an old question, just to add an answer:

Windows Phone 7 did not support anything else than SSLv3 for POP3, IMAP, and SMTP. Back on 15th October 2014, many providers like MacHighway turned-off SSLv3 and changed to TLSv1 because of SSL-POODLE. Windows Phone 7 supports TLSv1 everywhere like the Internet Explorer and for Exchange ActiveSync (EAS) but not for POP3/IMAP/SMTP.

You can check whether your E-mail provider still supports SSLv3 via a UNIX computer and the command

openssl s_client -connect jimbo.machighway.com:993 -ssl3
