rohn007 say:

OK, you have considered the issue (at length) and concluded that you need to use the SSN to uniquely identify individuals with the same/similar names.  I understand that, and it is fine (reasonable).  That is an issue on the user side, that only requires the SSN DATA to be stored in one place, in the "customer" (or whatever you call it) table/record. You do NOT need to carry the SSN over to other subsidiary tables for use to join them. Actually it is one of the "Normalization" rules that you do not duplicate it because it is not DATA. This is a database design issue that you do not recognize, but that someone trained in database design and normalization would. One of the advantages of hiring a professional  company to help you with the design and building of your application.

And although I didn't say it before, once the application has been professionally designed and built, there is no reason you couldn't take over the maintenance of it. You could use the project to build it as a learning opportunity. A chance to be a PITA, asking "why this ... "and "how come you did that ... " and "why not do it this way ..."

Things that a professional design could/would give you include a consistent naming standard for data fields and tables.  The first (and I have to say best managed from an IT development POV) company I worked for used a naming standard called the "IBM OF" language. It was based on a standard dictionary of 3 letter abbreviations (with a few 2 letter exceptions) of standard words used to name data fields and a short (less than 1 page) list of 1 character data type codes. As well it used a standard syntax to piece together the abbreviations.  After about a year any programmer could read any field code name and translate it on the fly, and get it right most of the time.  The advantage of this approach is that every time you saw CUSN, customer number, you knew exactly what to expect and how it was used to join the various customer related tables. The same with SO (one of those local exceptions). It was the Service Order Number given to the customer and service rep to uniquely identify each order.  It takes discipline to implement in the first place, and discipline to continue use consistently, but the benefits are huge. There is no guessing, is the data in the "customer" field in this table supposed to be the same as the "customer name" in different table.  The answer was simple, a "customer" really is something different from a "Customer Name", so go back and figure out what the difference is.

manfredi.leto say:

It (almost) works!

This is how I did it after a lot of attempts. Please note that I cannot be sure it will work for you because during all the attempts I could have done some key passage without knowing it. Anyway, I could say that drivers in packages sp58000 and the newer sp66915 doesn't work in my laptop (Elitebook 8560w with VFS471 sensor). So I found a previous version of the driver and of the software WBF DDK. Below all the passages:

1) carefully unistall everything related with validity and biometric: go to control panel and from "uninstall a program" uninstall Validity WBF DDK or other validity software
2) in device manager uninstall the biometric driver (checking the box "delete the driver software for this device") if you have one
3) download the package sp56957 from here
4) extract the package and run "WBFDDKSetup.exe" (located in c:/swsetup/sp56957/) and install WBF DDK (version 4.3.304.0)  
5) go in device manager, identify the biometric sensor between the unknown devices (if you have more than one, right click on it, properties and go in the details tab and look for something like WBFPID=003c) and manually install the driver (version 4.3.304.0) picking it from the folder c:/swsetup/sp56957/
6) always in device manager go on the device installed, right click on it, properties, go in the power managment tab and uncheck the box "allow the computer to turn off this device to save power" (this passage was needed because the fingerprint reader didn't work after standby or WINKEY+L)
8) go on log-in settings and setup windows hello with your fingerprints, if it doesn't show, try to restart the laptop
7) Restart the laptop and try if it works

For me it worked, I say "almost" because when I leave the laptop in standby for a lot of time, like the entire night, it doesn't work on wake up...but I don't know why...
