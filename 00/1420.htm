tushar mehta say:
I am trying to compare sales by month and i am gettting an error message.  Example April Sales 20, May sales 0, i want to see the percentge of change.  what formula can i use to view this?


The way to compute percentage change is (new - old) / old.  So, if Apr sales were 20 and May were 0, then you would get (0-20)/20 = -1, i.e., -100%.

Of course, if the old were zero, then the result is undefined (or infinite depending on your preference).

Deal with this with a formula along the lines of =IF(old=0,NA(),(new-old)/old) or =IF(old=0,"",(new-old)/old).

joeu2004 say: Hilda wrote:
I am trying to compare sales by month and i am gettting an error message.  Example April Sales 20, May sales 0, i want to see the percentge of change.  what formula can i use to view this?

It would be helpful if you showed us the formula that is producing the #DIV/0 error.

if April sales is in B2 and May sales is in B3, the percentage change (May over April) should be calculated by:

=B3/B2-1

formatted as Percentage.

In that case, you would not get a #DIV/0 error.  I suspect you are calculating B2/B3-1, which is the percentage change of April over May, usually not the way we talk about change.

However, if May is 0 and June is 15 (B4), you might want to compute the percentage change of June over May.  Ostensibly, that would be:

=B4/B3-1

formatted as Percentage.

That will produce a #DIV/0 error.  There is no mathematically right answer in that case.  You need to make an arbitrary decision about how to reflect change from zero.  My choice:  100% with the sign of the direction of change.  In that case, returning to the May-over-April formula, the general formula is:

=IF(B2=0,SIGN(B3),B3/B2-1)

formatted as Percentage.

PS:  That assumes that no month will have negative sales(!).  As strange as that might sound, it can happen (and it has).  If you want a formula that works in that case as well, try:

=IF(B2=0,SIGN(B3),(B3-B2)/ABS(B2))

john w. vinson say:

Well... if you're trying to take the email address out of one table and store it in a different table, don't. Such information should be stored, once and once only, in the Contact table, and then looked up when you want it.

If you just want to display the email address on the form when you select a supplier, you can include the email field in the combo box's RowSource query, and display it on the form. Put a textbox on the form with a control source

=comboboxname.Column(n)

where n is the zero based index of the field in the rowsource; that is, if the combo cboSupplier has the email address in its third column, use

=cboSupplier.Column(2)

ken sheridan say: As you say 'I can click on as many as I want' it sounds like you are using a multi-valued field.  If you only need to select a single supplier I'd advise that you change this to a conventional foreign key column, e.g. SupplierID, of long integer number data type. You can then use a conventional combo box set up as follows:

Name:                 cboSupplier
ControlSource:    SupplierID

RowSource:     SELECT [CompanyID], [EmailAddress], [CompanyName] FROM [Companies] ORDER BY  [CompanyName];

BoundColumn:   1
ColumnCount:    3
ColumnWidths:  0cm;0cm;8cm

If your units of measurement are imperial rather than metric Access will automatically convert the last one to inches.  The important thing is that the first two dimensions are zero to hide the first two columns.

Add a text box to the form and set its ControlSource property to:

=[cboSupplier].Column(1)

The Column property is zero-based, so this will show the second column, the email address.

If you do need to select more than one supplier per record, then you should not have a SupplierID column in the form's underlying table (which I'll call Orders for this example, and assume it has an OrderID primary key column), but model the many-to-many relationship type by a third table like this:

OrderSuppliers
....OrderID  (FK)
....SupplierID  (FK)

The primary key of this table is a composite one made up of the two foreign key columns.  In your form you should now use a continuous subform based on the OrderSupplers table, linked on OrderID, and in it put a combo box and text box as described above in the subform, not the parent form.  You can then select as few or as many suppliers as required simply by inserting a row in the subform for each.  You'll find an example of this basic many-to-many relationship type and how to represent it in forms as StudentCoures.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

In this little demo file various interfaces are illustrated, but I'd recommend the conventional form/subform, which is very easy to set up and code free.

Several problems. First, your Companies table should probably be renamed Suppliers (or, better yet, tblSuppliers). It should have three fields:

SupplierID (PK autonumber)

SupplerName (Text)

ContactEMail (Text)

Second, what is the Recordsource of the form where you have the combobox to select the Supplier? That table should have a field: SupplierID (Number/Long Integer). Name: cboSupplier

Controlsource: SupplierID

RowSource: SELECT SupplierID, SupplierName, ContactEMail FROM tblSuppliers ORDER BY Suppliername;

Bound Column: 1

Column Count: 3

Column Widths: 0";2";0"

Next you should have another textbox on the form with a ControlSource of:

=cboSupplier.Column(2)

This will display the contact E-mail for the selected supplier. It will NOT store it in the table the form is bound to, which is as it should be as you can pull the e-mail anytime you want to.
