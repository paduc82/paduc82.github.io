  • Option + E (acute accent), followed by a vowel, will yield √°, √©, √≠, √≥, √∫, √Å, √â, √ç, √ì, √ö
  • Option c produces √ß and Option C gives √á
  • Option n gives √±, and √ë with Shift, but also √£, √É, √µ, √ï

There are other keyboard layouts in System Preferences that forego the need to press Option. You can enable more than one and select them through the menu bar's flag (or via shortcuts, as shown in the Input sources panel).

For the rest of them, see How to Type French Accents.

john mcghie say:

I doubt whether anything I say you will consider "helpful" :-)  However, I can try to give you an idea what is going on.


When Word opens a document, it also opens as many as 20 other support and scratchpad files.


Generally, the Finder tags the file, then requests OS X to lock it and pass it to Word along with an OPEN command.  If that chain fails, Word receives very little information other than "Open failed for ...".  Word then tries to guess the problem and issue a helpful error message.  The error messages are often anything but helpful.


The screenshots you show are from three different conditions.


The first is specific to the obsolete file format .doc.  There were three versions of .doc -- loosely the "8-bit", "16-bit", and "32-bit" versions.

*  Generally, the 8-bit files (word 2 and earlier on the PC) will not open at all.

* The 16-bit files (Word 95 and above, Word 5 on the Mac) will open via File>Open, but not from the Finder.

* The 32-bit files will open normally.

The problem is that the obsolete file types can contain viruses, and checking for these is technically very complex. Checking and disabling active content is easy in the 32-bit files.


Your second screenshot is a deadlock.  Unfortunately, it does not say which file of the 20-odd involved has been locked.  This is sometimes because the users are not signing on to the server, they are signing on to their local machine only.  Unless you go round and set a unique Unix User ID on each machine, the server cannot tell who is who, every users has the default UUID of "501" and the server can't tell them apart.  Making sure the users authenticate with their network user ID is usually the cure to this.


The other issue is that sometimes the server or workstation is "lazy" about clearing its lock files.  If all users log out at the end of the day, and shut down once a week, this problem often goes away. This is especially helpful when these problems occur on local hard disks: the local computer has run out of file handles, logging out or rebooting enables the system to perform its house-keeping and usually clears the problem.


Your third issue should be taken literally. The file might be damaged so Word can't read it.  This often happens to files that have been sent through web-based free mail services.  They a Word document attachment as a "Zip file" (which it is...) and helpfully un-zip it for you.  If they do that, Word can't read it.


The file may be locked by another user (see above, same issue).


Or somewhere in the entire path between the server name/hard disk name and the file extension, there may be a character that word considers "illegal". For more, see here: http://word.mvps.org/mac/cantsavefilenames.html

Have I fixed it for you?  No.  But now you know you have three different errors, and roughly what causes them. As to specific instructions on how to get rid of these errors in a networked environment, sorry, we have been trying to obtain this information for years.  We don't have it...

The first one is easy: tell all users that if they open any file, make sure they Save As into .docx.  This will release 75% of your server space (.docx is a lot smaller) and get rid of the error.

The second one should be easy for you now you know that it's a deadlock, usually caused by user authentication or access problems.

The third problem can often be caused when someone has an apostrophe (') which is one of the illegal characters in their hard disk name.  "Tom's Mac" will render all files on its hard disk unreadable.  Re-name the HDD to remove the apostrophe and everything will be fine.

Sorry to be no help at all.
