The dot operator has been the recommended convention for some years now.  It used to be the case that the exclamation mark (bang) was necessary only when referencing an object as a parameter in a query, but the dot operator now works perfectly well in that context too.

You'll often see the exclamation mark used in code posted here, sometimes inappropriately, e.g. when qualifying a column name by a table in a query.  This generally results from the use of the expression builder by inexperienced developers.

As Daniel says, in reality it's probably not going to make any significant difference.

albert d. kallal say:

Actually there is a reason to use one or the other.

I use ‚Äúdot‚Äù in code to refer to controls. However, if I want to reference the underlying reocrdset, then I use ! (bang).

The reason for above is if you ‚Äúchange‚Äù the forms recordset at runtime AND you do NOT have such controls on the screen, then ‚Äúdot‚Äù will fail. (Especially if new columns are in the query as opposed to when the form was created).

Also, by adopting this standard then you communicate to others reading the code that the intention here is to grab the column, and not the control.

Access ‚Äúgenerates‚Äù a set of properties to allow use of ‚Äúdot‚Äù based on the forms current record source. However if you change the forms record source at runtime, then dot can fail, but bang will not.

This case is especially noticeable by developers using ‚Äúdot‚Äù. You often see a whole dog pile of controls in some corner with visible = false to get around their code breaking.

So if one stuffs in ‚Äúsql‚Äù to a forms recordsource, then the ‚Äúlist‚Äù of fields by ‚Äúdot‚Äù is not updated ‚Äì and thus code can fail.

For this reason, when referencing a control, I use ‚Äúdot‚Äù. This will also catch compile errors if you delete the control on the form (especially for unbound forms).

However bang will continue to work with or without the control. (And you note the error message with! is ‚Äúaccess can‚Äôt find the field (does not say control!). So using "dot" suggests the devleoper wanted to grab values from a control, and ! means the forms undelying data set.

One can make the case that ‚Äúdot‚Äù will show compile errors, whereas bang will only give errors at runtime.

So this effect can be used to a developer‚Äôs advantage to catch errors at compile time. This applies doubly so to un-bound forms. You delete a control ‚Äì then compile code will catch any ‚Äúdot‚Äù referances to such controls.

As noted using ! to reference the underlying data source thus gets rid of the ‚Äúdog pile‚Äù of controls I often seen piled up in the corner by previous developers that used ‚Äúdot‚Äù. (and then they found their code breaking).

So the above tip allows developers to use compile time errors to one‚Äôs advantage by adopting the above tip.
