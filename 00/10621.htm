jay freedman say:

It isn't an accent, it's a dot (like a raised period/full stop).

In some of the Unicode fonts there is a "combining dot above" character at 0307 (enter the number and press Alt+X), but in combination with the letter i it just looks like the dot that's part of the i. The same is true if you use the equation editor (Alt+=) and insert the first-derivative dot on the Accents gallery and insert the letter i in the placeholder.

There may be a better way, but the only one I can think of is the EQ field with the \O switch, which overstrikes one character with another. Start by inserting the following field code:

{ EQ \O(i,.) }

You can use the Insert > Quick Part > Field dialog, or you can insert the field braces by pressing Ctrl+F9 and then typing the code inside them.

Format the i as italic. Then select the dot, open the Format Font dialog (Ctrl+Shift+F) and go to the Advanced tab, and set the Position boxes to Raised by 9 pt. (You may have to experiment with this setting to get the height you want.)

jay freedman say:

It's one of at least three common notations for the first derivative of a function. The others are the "prime" mark (U+2032) and the fraction notation such as dx/dy.

These notations were created, respectively, by Isaac Newton, Joseph Lagrange, and Gottfried Leibniz. The article https://en.wikipedia.org/wiki/Notation_for_differentiation also lists a fourth notation (Leonhard Euler's), but I haven't seen that in introductory textbooks.

karl timmermans [outlook mvp 2012-2018] say:

1) By "load", I do mean "import".

Why is "importing" a bad idea?

2) It has been *months* now - and the search is still not working.

In addition to what has already been stated about importing - some other factors that need to be taken into account when importing a PST file.

- <IF> the PST file being imported has underlying errors - that can cause the import process to take an exceedingly long time or fail altogether. ScanPST should be run on the PST file being imported to ensure that underlying errors are corrected

- it is quite possible for a PST file to have underlying errors contained within the file which would never show themselves in normal day to day operation unless a very specific set of conditions come into play.

- a PST2PST import is a "copy" of one item from one PST file to another so if a source item is corrupted - a very good chance that the destination may also be corrupted. This differs from an import from another file source (I.e. CSV) which actually "adds" (newly creates) the destination item.

There is no single right or wrong answer about whether something should be imported - depends on the circumstances, requirements, objectives but in this case, would certainly agree with Brian based on the information provided so far.
