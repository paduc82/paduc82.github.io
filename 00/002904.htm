The number in a cell does not match the number in the formula line - Microsoft Community

zukolame say:

The number in a cell does not match the number in the formula line

Hello. I'm having such issue in excel 2010:

typing in a cell 2047,83051 - formula line shows 2047,83051

typing in a cell 2048,83051 - formula line shows 2048,83050999999

This happens with all numbers between 2047 and 65532 with numbers 83051 after comma.

What may be the problem?

joeu2004 say:

ZukoLame wrote:

my main point is - that the issue has to have nothing to do with the formatting

I know.  I did not mean to imply that it has anything to do with the cell format.  Perhaps I should use the word "display" or "appearance".

It is a defect in how Excel displays the value.

Initially, (binary representation of) the value is indeed what we typed, for example 2048.83051, even though it looks like 2048.83050999999 in the Formula Bar, as well as in the cell if we choose a cell format that displays 11 decimal places or more.

If the actual value were always what we intended despite appearances, I'd suggest that we just ignore how it appears (due to the defect).

But besides the confusing appearance, the real problem is:  Excel -- which might be as simple as pressing f2, then Enter.

That can change the behavior of some formulas

That is why I suggest that whenever you notice that the displayed value is not equivalent to what you typed, enter VALUE("2048.83051") instead.

Unfortunately, that means changing the constant 2048.83051 to the formula =VALUE("2048.83051").  In rare circumstances, that might have undue consequences.  So obviously, you must use your own judgment about this.

-----

Although the Excel display defect is related to the internal binary representation of numbers, it is not a normal (expected) consequence of that representation.

To demonstrate, note that VBA has no problem displaying 2048.83051.  For example, the VBA statement

MsgBox 2048.83051 & vbNewLine & Format(2048.83051, "0.00000000000")

displays 2048.83051 and 2048.83051000000, which are correct,

But in Excel, if we enter =VALUE("2048.83051") into A1, the Excel formula =TEXT(A1,"0.00000000000") displays 2048.83050999999, which is not correct.

-----

By the way, it is very difficult to specify which constants are prey to this display defect.  If you are curious, the conditions that expose the defect are:

1. The integer part must be less than 65536.  That is, it can be represented in 16 bits.  And....

2. We must be able to represent the decimal fraction in 32 bits with no loss of precision.  That is, the remaining bits to the right must be zero.  And....

3. The 16th significant digit of the decimal value must be 5 or more.

I doubt that Condition #3 is actually explicit in the Excel implementation.  But the display defect will be evident (noticeable) only when Condition #3 is true, because that is when we would expect the decimal representation to be rounded up -- which is not happening due to the defect.

-----

Finally, in case you do not know already (but I suspect you do), most decimal fractions cannot be represented exactly in the internal binary form that Excel uses (64-bit binary floating point).

For example:

2048.83051 is exactly

2048.83050999999,977648258209228515625

2048.83050999999 is exactly

2048.83050999999,022678821347653865814208984375

Here, I use period for the decimal place and comma to demarcate the first 15 significant digits, which is all that Excel displays and formats, usually rounding the 16th significant digit.
