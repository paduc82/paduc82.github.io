john w. vinson say: Your database is corrupt, unfortunately! 

Is it a "split" database - with the tables in one (backend) database and the forms/reports/code in a frontend? If so the backend is corrupt; try opening it directly. If it lets you see the tables, open the frontend again and DELETE all the (links to the) tables, and use File... Get External Data... Link to relink to the tables.

If it's not split, or if the tables can't be read in the backend either, use Tools... Database utilities... Compact and Repair in the database containing the tables. 

If this doesn't help, breath a sigh of relief that you have a backup; create a new, empty database and import all the tables and other objects from a working database.

See http://www.granite.ab.ca/access/corruptmdbs.htm for a good discussion of database corruption.

Pricing formula for adding users? - Microsoft Community

a. user say:

Pricing formula for adding users?

We are about to move to Office365.  We have figured out how many users we need, and we intend to pay yearly. But we are also planning to add some users some time in the next year.  Our Microsoft Partner tells us that . when we add a user we pay the full yearly price for that user, but the expiration date for all users gets pushed out by some formula that depends on the percentage of users we are adding.

We'd kinda like to know that that formula is, but our Partner doesn't know it (at least the people I've contacted don't) and there appears to be no trace of it on the Web.

Can I haz formula?

chevia say:

Hi Jon,

 

The following information explains the formula:

=======================================

 

A customer will have the option to increase the seats on their prepaid subscription by choosing the Add option in the portal and adding a product key. Doing so results in a new user and an updated end date.

 

Each product key has a number of users and days for those users associated with it. A product key for one user for 1 year has a 365 days associated. A product key with five users for 1 year has 1825 days associated with it.

 

The end date calculation is performed whenever a customer adds users to an existing prepaid subscription.

 

The number of days being added to the subscription is added to the days currently on the subscription and divided by the number of users to give the days the subscription will be active.

 

If the customer uses a credit card to increase the number of seats, this will result in a new subscription with a different end date than their current users. The new anniversary day will be the day the customer purchased the additional seats with their credit card.

 

Example (The customer adds two users to an existing five-user prepaid subscription):

 

Customer has a prepaid subscription with five users and 300 days remaining on their subscription.

Customer adds two users and the two new product keys have 365 days.

(5 users * 300 days + 2 users * 365 days) / 7 users = 318.57 rounded up to 319 days for the subscription

 

In this example, the customer’s subscription’s end date will be moved out 19 days, because the additional time is spread among all users.

 

Example (The customer renews with a smaller license count product key):

• Customer has a prepaid five-seat E plan. The subscription end date is March 1, 2015. If the customer renews the subscription on February 1, 2015, using a one-seat product key, what happens?

The seat count is reduced to one on the day the customer renews with a one-seat product key, February 1, 2015. The remaining time of the five users is distributed to this one user, pushing the end date out 5 months (~June 21, 2016).

(5 * 28 days (time remaining on the subscription) + 365 days) / 1 user

Vivian
