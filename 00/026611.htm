diane poremsky mvp (slipstick.com) say:

Properly coded SMTP servers do dot stuffing - adding a second dot to lines with leading dots, but for most affected users, this problem is caused by antivirus scanner and disabling the scanner takes care of the problem with future messages. The reason for this is that most scanners work by dropping a small smtp server between outlook and the internet.

 

 If you use certain older smtp servers that don't properly escape a leading dot, the problem can occur. This is beyond the control of most people though - best you can do is tell your mail host to upgrade the smtp server. Using Quoted-printable encoding should avoid problems too.

dennis wi - msft say:

Hi Michael!,

 

The definition with TypeGuessRows may help clear up the confusion.

 

"The number of rows to be checked for the data type. The data type is determined given the maximum number of kinds of data found. If there is a tie, the data type is determined in the following order: Number, Currency, Date, Text, Boolean. If data is encountered that does not match the data type guessed for the column, it is returned as a Null value. On import, if a column has mixed data types, the entire column will be cast according to the ImportMixedTypes setting.

The default number of rows to be checked is 8. Values are of type REG_DWORD."

 

http://msdn.microsoft.com/en-us/library/bb177610(v=office.12).aspx

 

In summary, the TypeGuessRows will scan the first 8 records of your excel file. For creating linked tables; if the majority of those 8 records contain numbers, the number datatype will be used. However, if you were importing the Excel file and you had the ImportMixedTypes registry key set to Text; then if any of the first 8 values were Text, the column would be imported as Text.

I think a lot is going to depend on just how the interaction between the scanner and Excel works.  I presume the scan result is going into a TextBox control on the UserForm?  If so then you might look at some code for either that text box's _Change() or _AfterUpdate() event.  The _AfterUpdate() event would be best because then you'd know that all the data was in it.  The _Change() event for a text box fires with each character entered into it or deleted from it, so if you typed in 123[Enter] you'd get 3 _Change() events with 1 _AfterUpdate() event triggers.
