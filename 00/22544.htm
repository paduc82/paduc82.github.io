    Provided that the aggregation is per the same grouping in each case you can use any number of aggregation operators.  The following is an example from my DatabaseBasics demo which counts, sums and averages the transactions per customer:

    SELECT [FirstName] & " " & [LastName] AS Customer,
    COUNT(*) AS TransactionCount,
    SUM(TransactionAmount) AS TotalAmount,
    AVG(TransactionAmount) AS AverageAmount
    FROM Customers INNER JOIN Transactions
    ON Customers.CustomerID = Transactions.CustomerID
    GROUP BY Customers.CustomerID, [FirstName] & " " & [LastName],
    LastName, FirstName
    ORDER BY LastName, FirstName;

    ken sheridan say: Do you want to be able to enter and/or edit data in the form?  If you do, then a query like that from my demo which I posted earlier won't work as an aggregating query which uses the SQL aggregation operators is not updatable.  You would need to use the VBA domain aggregation functions DAvg, DMin and DMax in computed columns in the query.  The following is an example, using the tables from my DatabaseBasics demo, which, in addition to the base data, would return the Avg, Min and Max amounts per customer.  As the query is updatable, if a TransactionAmount value were changed in the query's result table the aggregated values in the computed columns would be recalculated automatically:

    SELECT FirstName, LastName, TransactionDate,TransactionAmount,
    DAvg("TransactionAmount","Transactions", "CustomerID = " & Customers.CustomerID) AS CustomerAverage,
    DMin("TransactionAmount","Transactions", "CustomerID = " & Customers.CustomerID) AS CustomerMinimum,
    DMax("TransactionAmount","Transactions", "CustomerID = " & Customers.CustomerID) AS CustomerMaximum
    FROM Customers INNER JOIN Transactions
    ON Customers.CustomerID = Transactions.CustomerID
    ORDER BY LastName, FirstName, TransactionDate;

    ken sheridan say: Then MIN and MAX Sodium values are each values which exist in the QueryTotalFoodIntake query's result table, so it is possible to restrict a query to a row where the Sodium values is either of those values.  

    The AVG Sodium value, on the other hand,  is not necessarily a value which exists in the query's result table, however, but the average of all the values in the Sodium column.  So, unless the average value fortuitously happens to be one of the values in the column, which is unlikely, you cannot return a row where the value is the average value, because no such row exists.

    What is possible would be to restrict a query's result table to a row where the Sodium value is not the average, but instead is the median, or more strictly speaking is the statistical median.  The statistical median, unlike the financial median, must be a member of the original set.  Where there are an even number of members of the set (rows in the table in database terms) there are two statistical medians, the upper statistical median and the lower statistical median.

    So the nearest you can get to restricting a query to the row where the sodium value is the average sodium value would be to restrict it to the row where the sodium value is the upper or lower sodium statistical median.  You'd need to decide which.

    For an illustration of computing medians take a look at Median.zip in my public databases folder at:

    https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

    Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.
