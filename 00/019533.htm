Hi Beth,

SharePoint on-prem uses your local time zone and SharePoint online uses its own time zone. When you migrate SP2013 to Office 365, it changes the Date Time value (in your local time zone) to O365 time zone and that’s the reason dates are off by one day.

In this case, you must check the regional settings for the affected site in SharePoint online.

ken sheridan say: I very strongly endorse what John has said about the design of your database.  To arbitrarily divide a table on the basis of the physical pages in your paperwork is at variance with the basic principles of the database relational model.

A relational database is a model of a part of the real world with which it is concerned in terms of the entity types and the relationships between them.  In a relational database each table represents an entity type, e.g. Contacts, Companies, Cities, States etc. and each column in a table represents an attribute type of the entity type, e.g. ContactID, FirstName and LastName might be attribute types of Contacts and hence columns of a Contacts table.  It's important that each attribute type must be specific to the entity type, so that each 'fact' is stored once only.  In the jargon it's said that the attribute type is 'functionally dependent' solely on the whole of the primary key of a table.  At each column position in each row in a table there is a value of the attribute type which the column represents, e.g. in a row in a Contacts table the value at the FirstName column position for a row which represents me as a contact would be Ken.

Data must be stored in one way only, as explicit values at column positions in tables.  It must never be encoded as column names.  In a Contacts table for instance it would be wrong to have columns named Business and Personal to record whether a contact is of one type or the other, or indeed of both types.  There should be separate ContactTypes table and a table to model the many-to-many relationship type between Contacts and ContactTypes (see the example of OrderDetails from Northwind described below for a table modelling a many-to-many relationship type).  Similarly data must never be encoded as table names, e.g. it would be wrong to have a separate table for sales in each accounting year.

To relate tables a 'referencing' table will have a foreign key column which makes the link to the 'referenced' table, e.g. A Contacts table might have a CompanyID column as a foreign key, while a Companies table has a CompanyID column as its primary key.  Consequently no data other than the CompanyID needs to be stored in a row in the Contacts table for all the company information for that contact to be known; its available via the relationship and can be returned in a query joining the two tables on the CompanyID columns.

Similarly the Companies table might have a CityID column and the Cities table a StateID column.  If it's an international database the States (or more generically Regions) table would have a CountryID referencing the primary key of a Countries table.  So via the relationships, simply by entering (in reality this would be selected from a list of Companies in a combo box, not typed in)a CompanyID in the Contacts table the location of the contact's company is also known.  Redundancy, and therefore repetitive data entry is avoided.

To see how a database is made up of related tables take a look at the sample Northwind database.  It's not completely normalized in fact (deliberately so for the sake of simplicity) but illustrates the main principles of how tables representing entity types relate to each other.  An example of its lack of proper normalization can be found in its Customers table.  You'll see that this has City, State/Province and Country/Region columns so we are told twice that  Los Angeles is in California  and three times that California is in the USA.  Not only does this require repetitive data entry, but more importantly it opens up the risk of inconsistent data, e.g. It would be perfectly possible to put Los Angeles in California in one row and in Massachusetts in another, and to put California in Ireland!  Proper normalization as I described above would prevent this as the fact that Los Angeles is in California would be stored only once in the database as would the fact that California is in the USA.

An example of what at first sight might seem to be redundancy, but in fact is not, can also be found in Northwind.  The Products table and the OrderDetails table both have unit price columns, UnitPrice in OrderDetails and ListPrice in Products.  It might be thought that the unit price of a product could always be looked up from the Products table, so it's unnecessary in Order Details.  However, the unit price of a product will change over time, but each order needs to retain the price in force at the time the order was created.  Consequently a  unit price column is needed in both tables; that in products holds the current price and is used to get the value for that in Order Details (code in the ProductID control's AfterUpdate event procedure in the Order Details Subform does this), which then remains static when the current price (in products) changes.  In each case the unit price is functionally dependent on the key of the table, so there is no redundancy

The OrderDetails table is of interest in another way because it illustrates a different type of relationship from those we discussed earlier where the relationship is modelled by a foreign key column in one table referencing the primary key of another.  This type of relationship is one-to-many.  If you consider the relationship between Orders and Products, however, you'll see that this is many-to-many, i.e. Each order might be for one or more products, and each product might be in one or more orders.  A many-to-many relationship type cannot be modelled by one table referencing another directly, but is always modelled by a third table.  OrderDetails is such a table as it models the many-to-many relationship type between orders and products.  It does this by resolving the many-to-many relationship type into two one-to-many relationship types, with OrderDetails containing foreign keys OrderID and ProductID referencing the primary keys of Orders and Products.  If we draw this out diagrammatically this part of the model looks like this:

Orders----<OrderDetails>----Products

Which brings me on to my final point.  When designing a database, before creating any tables draw out the model diagrammatically on paper as an 'entity relationship diagram', with boxes representing each entity type (and therefore table) and directional lines, from the one side to the many side, between them representing the relationships.  If you find you have a line which points in both directions you have a many-to-many relationship type which needs an extra table to model it.  A relationship type is really just a special kind of entity type, so can have its own attributes as with UnitPrice in OrderDetails as discussed above.

I cannot overemphasize the importance of getting the logical model right at the outset, so think very hard about how your paper model will handle the real life situation which it is intended to represent.  Ask questions of it and amend the model where necessary in the light of the answers.  Only when you are thoroughly satisfied with the model should you translate it into tables and relationships in the database.  If you achieve a solid and reliable model of the reality then the  design of the interface in the form of the forms, reports, queries etc should fall into place naturally.  Get the model wrong, however, and you'll find yourself constantly trying to work around its defects.

While I appreciate that you will have invested time and effort in designing your current application the adage that 'when in a hole the first thing to do is stop digging' is a sound one.  Sometimes bullets need to be bitten, and you'd be strongly advised to redesign your database from scratch along the established principles of the relational model as outlined above.  The prospect may not be an attractive one at present, but it will pay dividends in time and save you many future headaches.
