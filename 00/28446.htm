steve rindsberg say:

The object model for tables in PPT is a mess; it'll fight you tooth and nail.  And it'll win.

However, if you were to create your table (sans hyperlinks) then copy it to the clipboard, paste special/as EMF, that'd give you a graphic you could ungroup a couple of times.  Each of the original table cells will be a rectangle.

Now you can apply a Run Macro action setting to either the shape representing the cell or the text in it and have it run a macro like:

Sub RunMe(oSh as Shape)

   With oSh

      .Fill.ForeColor.RGB = RGB(255, 0, 0)

   End With

End Sub

Problem is, once you assign a Run Macro action setting, PPT no longer recognizes the hyperlink you might have originally assigned.

If the text in the shape is the number of the slide you want to link to, we can make that work.  Otherwise, you'll have to get trickier, eg. by adding a .Tag to the shape and picking that up

How to retrieve/show user input info on a report - Microsoft Community

dmkkg say:

How to retrieve/show user input info on a report

In a query user has to key in two dates for Date of Sale field criteria (Between [Include Sales FROM:] And [Sales TO:]).

I would like to show these dates on a report based on the query but not sure what to put in the Control Source of the text box which will show the dates.

I can create a text box on the report and ask the user to key in the dates in there but want to avoid asking for the same information twice.

Is there a way to point to the information keyed in the query?
0 people found this helpful

ken sheridan say: You can ether reference the parameters in the text boxes' ControlSource properties, but I would normally include the parameters in the query's select clause as well as its WHERE clause, e.g.

     PARAMETERS [Include Sales FROM:] DATETIME, [Sales TO:] DATETIME;
     SELECT *, [Include Sales FROM:] AS SalesFrom, [Sales TO:] AS SalesTo
     FROM [YourTable]
     WHERE [Date of Sale] BETWEEN [Include Sales FROM:] AND [Sales TO:];

You can then simply bind text box controls in the report to the SalesFrom and SalesTo columns, or you can reference them in a single computed text box control, with a ControlSource property such as.

="Sales from " & Format([SalesFrom],"dd mmmm yyyy") & " to " & Format([SalesTo],"dd mmmm yyyy")

ken sheridan say: What I posted is the SQL statement of a query.  You can see this with your query if you switch from design view to SQL view, though Access will have thrown in a lot of unnecessary garbage, which is harmless but distracting.

The PARAMETERS clause declares the parameters, which is not absolutely necessary but generally a good idea, particularly with parameters of date/time data type as it avoids a date entered in short date format at the prompt being misinterpreted as an arithmetical expression.  It also caters for regional variations in the date format.  Parameters other than those of date/time data type should not be declared, however, if the parameter is being examined for NULL in the query.

The SELECT clause lists the columns returned by the query.  These are mostly columns (fields) from the table(s) used, but can also be expressions, as in this case where the parameters are entered.  The column name returned is determined by the AS assignation.

The FROM clause lists the table or tables which the query uses.

The WHERE clause includes the criteria expressions which restrict the rows retuned by the query

You can either amend your query in SQL view by adding the PARAMETERS clause and adding the parameters to the SELECT clause, or you can do so in design view by declaring the parameters through the dialogue as you've done, and adding two additional columns to the design grid with the following in their 'field' rows respectively:

SalesFrom:[Include Sales FROM:]
SalesTo:[Sales TO:]
