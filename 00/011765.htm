jay freedman say:

The feature on smartphones is called "predictive text". There is -- so far -- no such feature in Word or in Windows generally. You can add your vote to the thousands at windows.uservoice.com.

There is also no direct way to add a "data base" to either AutoCorrect or AutoText, nor do I know of any such data base (at least, one that's publicly available) of common words. If you do come up with a list, though, I have a way that you can add it to AutoCorrect.

Visit http://jay-freedman.info/ and download the file AutoCorrect2007.zip. (Despite the name, it's compatible with all versions of Word from 2007 to 2016.) Extract the file AutoCorrect.dotm from it and store it in your Templates folder. Double-click it to open Word.

There will be a small dialog box containing buttons for Backup, Restore, and Cancel. Click the Backup button, which creates a document containing a table of all the existing AutoCorrect entries. Save that document.

Now you can add new entries to the table, one row per entry. The left column contains the "trigger", the thing you type to activate the entry, and the middle column contains the replacement that will be inserted in its place. The third column contains the word False if the entry is plain text, or True if it's a formatted entry.

Save and close the backup document. Again double-click the AutoCorrect.dotm template, and this time click the Restore button. Select the backup document in the Open dialog. The utility will add the new entries to the AutoCorrect feature, and it will redefine any entry whose definition has been changed. It won't delete any entries, even if you've removed them from the backup document.

For one thing you're operating in a non-standard environment. It is bad practice for a developer to make changes to a Production system while it is being used. Perhaps it is time to get a real development discipline. That's too big of a topic to discuss here in detail, but elements include:

Dev / Beta / Production are different stages of the development life cycle, and the application moves through these stages in a deliberate fashion.

Management decides what developer shall implement for the next version.

Developer implements this on his/her local machine, using local resources (e.g. copies of the Access databases and Excel worksheets)

Developer unit tests, increments version, and offers it to the Beta test team.

Beta testers exercise the new functionality (and some of the possibly affected existing functionality) using local resources, report bugs, and test some more until satisfied.

Developer deploys to Production.

sagarmatha77 say:

Thank you. Unfortunately this is wrong. I stated that I have done this before and it worked. Fortunately I figured it out. This approach successfully mail merges sharepoint files indefiintely. 

1. Set up a mail merge in the Word document with a local file.

2. Change the Word document extension to .zip from .doc or .docx

3. Extract the zip file into a new folder. 

4. Open the word\settings.xml file or word\settings\settings.xml file in notepad

5. In notepad, change the merge file source location and name information in both (two) places with the url of the sharepoint source file, and save (example: target=‘file:///c/documents/myfile.xlsx’ to target=‘https://sharepoint/folder/filename.xlsx’)

6. Repeat 4,5 for the file rels\setttings.xml.rels

7. Rezip the extracted folder and change the extension back to .doc or .docx OR

7b. replace the files in the zipped folder and change the extension back to .doc or .docx

8. Your word document is now successfully mailmerged with a sharepoint source file. And it works perfectly. 

It’s not ideal. But it’s a work around to Microsoft’s decade of unwillingness to address productivity and collaboratoin needs with its platforms.
