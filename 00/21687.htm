Fonts - Microsoft Community

dhnyc say:

Fonts

I've been using Microsoft Office for Macintosh for more than 30 years, ever since Word version 1.05, so I've seen a lot of changes over the years. There have been a lot of improvements and also some changes that have made things worse. Office 2016 is consistent with this pattern, though I'm sorry to say that the changes for the worse outweigh the changes for the better. One particularly frustrating downgrade for me has to do with how Microsoft chose to implement fonts in Office 2016.

Previous versions of Office automatically installed fonts into the Library folders where they belong under macOS. At least some versions of Office allowed users to chose whether to install only a basic set of fonts or also to install additional fonts for foreign languages.

Office 2016 made a big change by placing fonts in the applications themselves rather than in Library folders. Version 15.41 has 220 fonts in one location and another 30 in a second location within each application's package. The reason Microsoft did this most likely has to do with changes in macOS that discourage--but do not prevent--developers from putting fonts in Library folders. What's wrong with Microsoft's new approach? Several things that all represent changes for the worse:

  • Reducing user control--Users can no longer use standard font-management techniques to control which fonts are available to them. For example, fonts supplied with Office can't be deactivated with Font Book. With the Version 15 builds it was possible to override this by manually going into the application packages, removing fonts, and deleting and/or editing .plist files. Recent Version 16 builds made even that absurdly cumbersome and user-hostile process impossible.
  • Reducing value to users--Placing fonts in proper Library font folders makes them available to all applications. Hiding them inside application packages means that they can't be used by other applications. (Users can get around this by going into the packages, copying the fonts, and putting the copies in the proper folders, but why force them to jump through that hoop?)
  • Creating bloat--Apple's switch to solid-state drives reversed the trend toward ever-increasing storage space, and it now matters again how much drive space applications take up. The fonts inside each Office application take up almost 500 MB. If they were placed inside the proper Library folders, they'd occupy about 500 MB (assuming users didn't delete unwanted fonts). But because the same fonts are bundled inside Excel, Outlook, PowerPoint, and Word, they take up 4x500 MB, i.e., almost 2 GB. That's crazy. (OneNote has fewer fonts, so it doesn't add as much to the problem, but it also doesn't help.)
  • Redundancy and the potential for font conflicts--Let's say you have Calibri in one of the proper Library font folders (most of us do) and you're running an Office application. Two copies of Calibri are active at the same time. I haven't seen evidence that this causes problems for Office applications but font conflicts like this are notorious sources of trouble in general.
  • Preventing font version control--Some of the fonts in Office applications are older versions than those supplied with macOS or available elsewhere. By putting fonts inside the packages, Microsoft makes it difficult or impossible for users to choose the version of a given font that is active in Office applications.
  • Reducing work efficiency--There may be a handful of users who need each and every one of the 250 fonts (including a vast number of foreign-language fonts) supplied with Office. But for everyone else, cramming that many fonts into the applications means that every time users wants to select a font, they have to scroll through endless fonts that they will never, ever use. This makes font selection slower virtually every time. (And, actually, there probably are literally zero users who need all 250 fonts; Marlett? Bookshelf Symbol 7?) Over time, this makes Office applications significantly less efficient to use than they used to be.

What's all the more annoying about the way Office now does fonts is that it's completely avoidable. There are any number of alternative ways of supplying fonts that would avoid all this. For example, the installer could offer the option of whether or not to install foreign-language fonts, something that prior versions of Office offered. Or users could be given a folder full of fonts (helpfully organized) with easy instructions for how to install them with Font Book.

The fact that there are multiple font management programs in widespread use (e.g., Suitcase Fusion, Font Agent, and macOS's own Font Book) demonstrates that font management isn't some weird esoteric practice reserved only to publishers or design professionals. By going its own way and disregarding users, Microsoft has taken a big step backward. It is, sadly, consistent with other anti-user choices in Office 2016, such as eliminating toolbars.

What do you think? Do you agree? Do you disagree? Do you have ideas about how Microsoft could deal with the font situation in a more user-friendly way?

ossiemac say:

Do you mean that you want to sum column BD or column CG. If so then the following formula examples.

=SUM(INDIRECT(L10&":"&L10))             Formula to sum column BD

=SUM(INDIRECT(M10&":"&M10))          Formula to sum column CG

If you want to sum all of the contents from column BD to column CG then the following example.

=SUM(INDIRECT(L10&":"&M10))

More information on the Indirect function at the following link.

Excel Indirect function

If the above is not what you require then need more explanation.
