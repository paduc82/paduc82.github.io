AFAICS, as long as you can work with one of the Unicode 16-bit formats (Big- or Little-Endian) in the other applications you need to use, you can both save and re-open in Word 2011 successfully, via those tedious dialog boxes, and you may be able to use VBA to make your job slightly easier. Both Mac textEdit and Windows Word 2010 appear to open both 16-bit formats successfully. I haven't checked any other packages.

I do not think that you can do that with Word 2011 using the UTF-8 format (because there are clearly some faults in Word 2011 - see below - when it comes to opening UTF-8 format, and you cannot work around them using VBA)

It looks to me as if you could only work easily with the Japanese (Mac OS) format if you have used the Office Language Register to select Japanese (or at least, not English). If you have English selected as the language, you can, I think, use VBA to save in the Japanese (Mac OS) format, but you cannot specify an encoding when you re-open the file in VBA, and the wrong encoding is used. However, I do not know what other effects using the Japanese option in the Language Register might have on your workflow.

Here are some basic suggestions on the VBA front - I'd wrap them up in more useful routines, but don't have the time right now, and it may not be what you need.

You can save in the Unicode 16-bit format from VBA by specifying the file format as 

WdSaveFormat.wdFormatUnicodeText

(e.g.
ActiveDocument.SaveAs "Macintosh HD:auser:Documents:u16.txt", WdSaveFormat.wdFormatUnicodeText
)

This saves the file in the Big-Endian format. Whichever Unicode format you choose, Word always inserts a BOM (Byte Order Mark) at the beginning.

If you need to specify the format, you can use, e.g.

ActiveDocument.SaveAs "Macintosh HD:auser:Documents:u16.txt", WdSaveFormat.wdFormatUnicodeText, , , , , , , , , , MsoEncoding.msoEncodingUnicodeLittleEndian

AFAICS 

 a. you can open either Unicode 16-bit format using VBA just by specifying the file name, but only if the file has a BOM, e.g.

Documents.Open "Macintosh HD:auser:Documents:u16.txt"

 b. if you try to do it without VBA (i.e. through the user interface, you always see the file conversion dialog box, regardless of the "Confirm conversion at open" setting in Word->Preferences

 c. If "Confirm conversion at open" is checked, and you need to suppress the dialog, you can do it via

Documents.Open "Macintosh HD:auser:Documents:u16.txt", False

 d. If the file does not have a BOM, you might hope that you would be able to open at least the "default" format (Big-Endian) by specifying the file type e.g.

Documents.Open "Macintosh HD:auser:Documents:u16.txt", False, True, False, , , , , , WdOpenFormat.wdOpenFormatUnicodeText

There is no additional parameter (as there is in Windows Word) to let you specify the encoding as well.

However, that does not work with either Big-Endian or Little-Endian files with no BOM. Word clearly assumes some other encoding. So in that case, your only option seems to be to prepend a BOM, or open via the User Interface and specify the format (which seems to work OK).

Background:

There is undoubtedly at least one error in this area that needs to be fixed by Microsoft. Since they do not appear to have any two-way communications method for fault reporting, I can only suggest that you report via Word->Help->Send feedback about Word, and don't hold your breath for a response. I don't suppose Santa has started responding to software fix requests just yet, either :-)

I can see here (partly by comparing with what Windows Word 2010 does) that
 a. Word 2011 successfully saves in UTF-8 format if specified, at least in the dialog box. FWIW it puts the three-byte Byte Order Mark (BOM) at the beginning, which ought to be a very strong signal to any software that the file is highly likely to be UTF-8 encoded.
 b. Error 1 seems to be that when you try to open this file, Word does not even manage to select the "Other Encoding"" radio button in the encoding dialog box. 
 c. Even when you select that radio button, and select the UTF-8 encoding, if you wait before clicking OK, the dialog box reselects the Mac OS (Default) option. That might give the MS Dev people a clue.
 d. Even when you reselect the Other Encoding button, and proceed
    - the file loses Japanese character encoding, as you say.
    - in fact, it does not apear to open other UTF-8 files correctly, either, and even with a UTF-8 file with nothing but low-end characters (32-127) the opened file shows the 3 characters of the UTF-* BOM (which should be removed.

In contrast, Windows Word gets it all right.

If you save as Japanese (Mac OS) instead of UTF-8, then
 a. opening through the conversion dialog, you still have to select "Other Encoding," and you still have to select the encoding, but in this case the wrong radio button is not reselected as it is with UTF-8.
 b. opening with VBA does not help with either of the above format encodings...

unless you use the Office 2011 Language Register to switch to Japanese Language support, in which case
 a. the conversion dialog selects Japanese (Mac OS) format correctly
 b. using VBA to open the file works correctly with Japanese (Mac OS) format, even if you omit the wdOpenFormat parameter altogether.
 c. Word still doesn't open the UTF-8 format correctly whatever you do.

I don't use the Language Register (i.e. I don't change from the "English" setting) so do not know what other consequences working with the Japanese settings might have.

Incidentally, this does not appear to be a special problem with the English Language version of Word 2011 - I happen to have a German language copy here too, and the problem appears to be the same.

I wasn't able to see any user-editable settings in the various settings files maintained by Office/Word (or even the Language Register) that would help. However, I can't guarantee that I have looked everywhere.

linda wu msft say:

Hi JosueCastroSoto,

In the new experience for SharePoint document libraries, if users want to have the discard check out option, the permission level must include the Override List Behaviors - Discard or check in a document which is checked out to another user, and change or override settings which allow users to read/edit only their own item.

As a workaround, you can create a new site permission level including theOverride List Behaviors. Please follow the steps below:
1.  Navigate to the library, click gear > Site settings, in the User and Permissions section, click Site permissions.
2.  Click Permission Levels in the ribbon.

3.  In the opened page, click Contribute.
4.  Scroll down to the end of the page > click Copy Permission Level > type in the name, for example: discard check out > select the Discard or check in a document which is checked out to another user, and change or override settings which allow users to read/edit only their own item > Create.

5. Then you can assign your users the permission for the library. Go to the SharePoint library, click Library in ribbon > Library settings > Permissions for this document library > Grant Permissions.

6.    Then the user can use the discard check out option.

For your referenceï¼šEdit permissions for a list, library, or individual item.
