The indented information that follows is our response when a customer reports the above issue to us. Eugene already has this information because he wrote us last week about it (he's just hoping for another workaround, which is understandable, and this is a reasonable place to ask about that):

Unfortunately, this is an issue recently introduced by Microsoft in its 16.16 Office release for Mac. 
This issue happens with any image images internally inserted as PDF (like MathType formulas).

We honestly apologize for the inconvenience and we hope that Microsoft fixes the issue soon.

We encourage you to contact Microsoft regarding this issue hoping that it will speed up its resolution (https://support.microsoft.com/en-us/contactus/).

This is an issue with Retina displays, so that's one possible workaround: if you have a non-Retina display, use that when inserting or editing equations. The equations will be fine if merely viewed or printed when using a Retina display, but when you insert or edit one is when the sizing issue happens.

WIRIS America, makers of MathType

afwings say:

Sorry for the delay; I was out of the office last week.

It wasn't a MathType issue, so MathType 7.3.1 doesn't fix it. However, it was fixed by Microsoft in Word version 16.17. The current version is later than that (it's 16.18, with the insider version being 16.19), so if you upgrade Word/Office, you shouldn't see the incredible shrinking equation.

Beyond that, one thing we have done with MathType is to detect if Word is shrinking the equations. If it is, MathType corrects Word's error by presenting the equations at the proper size. This is better than actually resizing the equations because 1) this scenario will only happen if MS breaks Word again and 2) if MathType simply resizes the equations to a larger size, they'll then be too large when MS once again fixes their bug. In effect, MathType now makes it irrelevant if this bug is present in Word or not.

jan karel pieterse say:

You could use something like this macro:

Sub CheckAndClearCells()
    Dim oCell As Range
    Dim oCells2Clear As Range
    For Each oCell In Selection
        If Not oCell.Validation Is Nothing Then
            If oCell.valuedation.Value = False Then
                If oCells2Clear Is Nothing Then
                    Set oCells2Clear = oCell
                Else
                    Set oCells2Clear = Union(oCells2Clear, oCell)
                End If
            End If
        End If
    Next
    If Not oCells2Clear Is Nothing Then
        oCells2Clear.ClearContents
    End If
End Sub

charles kenyon say:

When you are working in the header/footer area the body of the document fades or becomes completely invisible. When you go back to the body, in print view, the header and footer appear faded. That is because it is a different document layer, not directly available for typing.

To see both at equal weight, you need to print. This happens when you press Ctrl+P, your preview there will show them equal weight. Likewise if you create a pdf or otherwise print, they will be equal.

If, within the header or footer you try to do a find or replace operation, the search area is the header/footer layer. Conversely, a find/replace operation in the body of the document will not find things in the header/footer layer.

Here is a (temporary) link to a sample: https://www.dropbox.com/s/0apkum64ep9ugek/Header%20and%20Footer%20Examples%202017-06-28.docx?dl=0

If viewed on the web rather than downloaded, the headers and footers will appear at the same weight as the body (i.e. will not be faded out).
