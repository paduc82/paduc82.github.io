This date is then converted to 201702 using the formula =IF(MONTH(cell ref)<10,YEAR(cell ref)&"0"&MONTH(cell ref),YEAR(cell ref)&MONTH(cell ref)).

The correct result to the SUMIFS formula is 12000. If the "<=" part of the formula is replaced with "=" or deleted completely (including deleting the &, the formula correctly calculates 12000. However, keeping the "<=" yields an answer of 0.00, which is incorrect. The cells used all appear to have the correct formatting so I don't understand why a 0.00 answer is provided when the 'less than or equal to' mathematical operator is included in the formula. Also, if I physically overwrite the 201702 (don't let the formula calculate this part) the formula will correctly calculate 12000 when using the ‚Äú<=‚Äù operator. Any thoughts on why this is happening?

arkadi_73 say:

A possible solution to your problem.... but I'm not sure if this is compatible with the rest of your uses for the data, all I know is that it would work.

Format your L4 cell as a number (no decimals), and change your date conversion formula for column Q to return a number:

=VALUE(IF(MONTH(cell ref)<10,YEAR(cell ref)&"0"&MONTH(cell ref),YEAR(cell ref)&MONTH(cell ref)))

This should then recognize the match and return the right values.

You can also simplify your date conversion to:

=YEAR(cell ref)&TEXT(MONTH(cell ref),"00")

or with new approach:

=value(YEAR(cell ref)&TEXT(MONTH(cell ref),"00"))

In Excel 2011 I have no problem getting the correct result. In this screen shot, column D contains the formula =IF(A2="yes",B2,"") copied down for B3, B4, etc. Cell D12 contains the formula =SUM(D2:D11) which returns a result of 70.

I believe HansV was drawing attention to the inclusion of a colon between the C & the 2 in the formula you wrote in the first paragraph of your message as being the possible cause of the error. However, Excel would not accept that without displaying an error message requiring a correction. As he suggested, though, there must be another reason for the error.

Further to that point, when a cell contains a formula the cell simply displays the result of that computation, the cell content is not replaced by that result. IOW, the formula or function, itself, still constitutes the content of the cell. E.g., In the example above, D12 displays 70 but still contains the same function... If you select the cell, that formula will be displayed in the Formula Bar as well as in the cell if you double-click it. The displayed result of 70 is not selectable. The same is true of the null result of your =IF() function.

Perhaps it would be a god idea to investigate a bit further to verify that the content of the sheet is what you expect it to be. As a side note, though: Unless you have a pressing need to display the result of the intermediate =IF() operation it would be far more efficient to omit it altogether. Use the =SUMIF() function, instead of =SUM(). For example: =SUMIF(A2:A11,"=yes",B2:B11)

tom sobell say:

Hi

Pardon the delay.

Ooo, this is tricky but a neat trick!  Yes works.  Documentation throws me off.   This is completely different than in the previous versons.   Congrats to the code-heads in Redmond!

1) Outlook > File > BCM  > synchronize  .... look at screen to make adjusts, if any

Here is how I tested.  Now that "synchronized"

a) made new (Test1) contact card inside the BCM only.    Now did a search on "test1" by 

          utilizing the new "Find a contact" top top upper left menu.    Results  =   Test1  comes up in both  the BCM and Outlook contacts. 

       The results are misleading.. I know this sounds confusing, but just humor me.    The new "test1" I put in the BCM side only APPEARS as though it's ALSO in the Outlook contacts too.  Wrong.   Coders set up an electronic mirror to bounce back over to the BCM side and edit the record there.  Exit after edit and you are back inside outlook.  No muss, no fuss.   It's a little like you took a ride in hyperspace then back to earth again.  but you never knew it.  

b) Did another (Test2) contact.   Put this ONLY in the outlook side.   Do the same

                     utilizing the new "Find a contact" top top upper left menu

          Well Mr. Test2 shows up ONLY in the outlook contacts as one would expect.   No hyperspace this time.  You stayed on earth all the time.

OK, now the big test.....  put PDA in the Activesync - let it do it's thing.  Vwalla !  Both the Test1 and Test2 show up on the contacts list of the PDA.   Yeah!

One final humor me...  But now there is a minor difference between this 2010 version and 2007 one needs to beware.   Here it is.  The 2007 PDA record of the BCM, had a tab that allowed you to look at all the histories,  phone logs etc  in addition to the regular notes area - just like you were using the BCM on your laptop.  No difference.     Ok, now the 2010 version, only shows you the NOTES area tab.  All those attached records like the email link histories stayed on the laptop.  Not synched Laptop <> PDA    So at the end of the day.  When you look at a record on the PDA, it's no longer clear to the user if you are looking at a record originated from BCM or Outlook contacts itself.      Hey, this is great if you are weaned on Office 2010.   However, makes your head spin a bit if you are weaned on Office 2007.  The synchronized records dont work the same !     This sure got me confused!

Ok, at the end of the day, it all works.   It's the documentation that is lacking.  Perhaps that was done for simplicity.   I guess it does.  But debugging as I tried, just got me more confused because the CHANGES were not documented.

Many thanks.  All set.  Cheers

Tom

oops, one last item.    ActiveSynch.

When using BCM / Outlook 2010   -  Do NOT check the box to synch with BCM inside activesync.    ActiveSynch > Tools > Options   = dont check the BCM synch here
