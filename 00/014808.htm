Short answer: put a call to MsgBox in the Form's On No Data event.

Probably better answer: Don't use the [Enter summons #] prompt at all! Instead, use an unbound combo box on the form itself as a criterion: =[Forms]![YourForm]![comboboxname]. It's user-hostile to make them enter a criterion and slap their hands if they choose wrong; instead, give them a list of all the valid choices and let them pick one.

ken sheridan say: Try putting the following code in the form's Load event procedure:

    Const MESSAGE_TEXT = "Summons not found.  Try typing the summons in again. " & _
          " If you still have trouble, the summons may be eligible for discount."

    With Me.Recordset
        If .BOF() And .EOF()
            MsgBox MESSAGE_TEXT, vbInformation, "Warning"
            DoCmd.Close acForm, Me.Name
       End If
    End With

If you are unfamiliar with entering code into a form's or control's event procedures, this is how it's done in form design view:

1.  Open the form's properties sheet if it's not already open.

2.  Select the On Load event property and select the 'build' button (the one on the right with 3 dots).

3.  Select Code Builder in the dialogue and click OK.  This step won't be necessary if you've set up Access to use event procedures by default.

4.  The VBA editor window will open at the event procedure with the first and last lines already in place.  Enter or paste in the above code as new lines between these.

However, a better approach would be, rather than using a crude system generated parameter prompt, to reference an unbound control in an unbound dialogue form as the parameter and open the bound form from the dialogue form by means of a command button.  The unbound control can then be a combo box which lists all relevant summons # values, with its LimitToList property set to True (Yes).  The user would then be able only to enter or select valid numbers.

ken sheridan say: An unbound form is one which has no RecordSource property, i.e. it does not display data from tables, but is merely for entering values for use in restricting the bound form, which is your current form, whose RecordSource is your query.  In the query, instead of a simple parameter prompt like [Enter summons number] you'd reference the combo box, which has no ControlSource property and is thus unbound, in the unbound form like this:

    Forms!frmSummonsNumberDlg!cboSummonsNumber

where frmSummonsNumberDlg is the name of the unbound form, and cboSummonsNumber
is the name of the unbound combo box in it.

The RowSource property for the combo box would be like this:

    SELECT DISTINCT [Summons #] FROM [2016 Amnesty Exclusions] ORDER BY [Summons #];

In the unbound form you'd then open your bound form with a command button by putting the following code in its Click event procedure:

    Const MESSAGE_TEXT = "Please select a summons number first."

    If Not IsNull(Me.cboSummonsNumber) Then
        DoCmd.Openform "YourFormNameGoesHere"
    Else
        MsgBox MESSAGE_TEXT, vbExcalamation, "Invalid Operation"
    End If

Instead of opening your bound from directly you'd open the frmSummonsNumberDlg form, select a summons number and then click the button.

If you want the unbound dialogue form to close automatically then put the following code in the Close event procedure of the bound form:

    On Error Resumer Next
    DoCmd.Close acForm, "frmSummonsNumberDlg"

A few supplementary comments:

1.  Using spaces or special characters like the # (hash) sign in object names is not recommended.  It's better to use CamelCase, e.g. SummonsNumber, or represent a space by an underscore character, e.g. Summons_Number.

2.  I notice that you have a table with the year in its name, which suggests that you have separate tables for each year.  This is bad relational database design as it encodes data as table names.  A fundamental principle of the database relational model is the Information Principle (Codd's Rule #1). This requires that all data be stored as values at column positions in rows in tables, and in no other way.  You should have a single table which includes a column such as SummonsYear (don't use Year as a column name as it is the name of a built in function) and restrict the query to the year in question.

lesgom say:

assuming you know how to enter the VBA editor and create a module... copy and paste

Sub ExportSheet()
Dim strMyFilename As String
Dim strMyDate As String
Dim strMyPath As String

    strMyDate = Day(Now()) & "_" & Month(Now()) & "_" & Year(Now)
    If Range("A1").Value = "" Then
        strMyFilename = InputBox("please input a customer name") & " CMAC Registration Form " & strMyDate & ".xlsx"
            Else
        strMyFilename = Range("A1").Value & " CMAC Registration Form " & strMyDate & ".xlsx"
    End If
    strMyPath = ActiveWorkbook.Path & "\"
   
    Sheets("form").Copy
    ActiveWorkbook.SaveAs Filename:=strMyPath & strMyFilename, _
        FileFormat:=xlOpenXMLWorkbook, CreateBackup:=False
    ActiveWindow.Close
   
    MsgBox "Form has succesfully been exported as..." & Chr(10) & strMyFilename
   
End Sub
