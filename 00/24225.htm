ken sheridan say:
.......therefore by the process of normalisation we end up with staff 1, staff 2 and staff 3 columns on the courses table

Quite the opposite I'm afraid; First Normal Form (1NF) requires that you store only one value of each attribute in each row.  Sometimes this is misinterpreted as meaning more than one value in a single column in each row; that's not the case, storing values in multiple columns representing multiple instances of the same attribute is barred.  The formal definition is:

First Normal Form: A relvar is in 1NF if and only if, in every legal value of that relvar, every tuple contains exactly one value for each attribute.

Loosely speaking, in the language of the relational model, a relvar (relation variable) equates to a table, a tuple to a row (record) and an attribute to a column (field).

The many-to-many relationship type between course and staff is modelled by a table which resolves the relationship type into two one-to-many relationship types.  So the tables, in outline, are:

Staff
....StaffID (PK)
....FirstName
....LastName
....etc

Courses
....CourseID (PK)
....CourseName
....etc

and to model the relationship type

StaffCourses
....StaffID (FK)
....CourseID (FK)
....etc

The last table might have other columns representing attributes of a staff member's relationship with a course, e.g. each staff member could have different date ranges in StartDate and EndDate columns, representing when they teach the course.

Similarly there is a many-to-many relationship type between students and courses, so the tables are:

Students
....StudentID (PK)
....FirstName
....LastName
....etc

and to model the relationship type with courses:

StudentCourses
....StudentID (FK)
....CourseID (FK)
....etc

However, this assumes that each student registered for a course can be taught by each member of staff who teaches the course.  It's possible a student may be taught by only one or two of the staff in question, in which case the table would be:

StudentCourses
....StudentID (FK)
....CourseID (FK)
....StaffID (FK)
....etc

In the last scenario the table is modelling a ternary (3-way) relationship type rather than a binary (2-way) relationship type.

For data entry in each case use a form in single form view with a subform in continuous forms view.  You'll find an example of this sort of many-to-many relationship type as SudentCourses.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo shows various ways of representing this in a form, but in your case use the conventional form/subform example as your model.  My demo does not include staff, but you'd do the same for courses, using a courses form with a staff subform based on the StaffCourses table within it.
