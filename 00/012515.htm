rohn007 say:

Back in early 90's the first exposure I had to computer virus's was the Word macro virus". Our Office was being re-infected daily by an outside partner (it drove the LAN admin nuts).  That problem was addressed and made way for other virus attacks.  Recently, macro virus's have been making a comeback.

MS introduced the new x-file formats in 2007.

There were several reasons for the new format:

  • X vs M file makes saving macros in a file very Obvious to users and creator
  • the new format is based on an "open standard" controlled by the OSF an entity external to MS, while the old DOC format was owned and "Proprietary" to MS
  • the X-file format is text based, making it easy to look into the "guts" of the file, while the old files were a "binary" format, you could compare them loosely to a mini-program that "ran" inside of the application making them effectively unreadable by users
  • The x-file text format allows you to easily extract elements like text or images from the file
  • the x-file text format in theory makes "file corruption" from external causes less likely
  • the text elements of the x-file format are saved inside of industry standard "ZIP" compression to reduce the text file size, usually substantially below the equivalent DOC file
  • using "Trusted Folders" is actually intended as a temp work around. The intent of the "Trust"design was to ultimately to have everyone "sign" their macro documents and use recognized "trusted Publishers" so that once you "recognize" a publisher as trusted, you no longer see the restriction no matter where the file is located

Some other side effects of the x-file formats:

  • the x-file text format in practice makes it easier for users to correct "internally created" file corruption, where the program messes up, ie the periodic XML "end tag" error we see. In the old binary format, this would have resulted in a totally unusable file
  • An unintended benefit of the x-file format is that other "open source programs" like LibreOffice are better at finding, working around and even fixing XML errors that the MS programs are creating <ironic grin>
  • an unintended deficit of the x-file format is the order content is saved. In Word files, the body text is unfortunately saved at the "bottom" of the XML file structure. This makes it susceptible to loss when USB drives are not removed properly from Windows computers. From a user point of view it would be preferable to lose almost EVERYTHING else before the body text

DOCX vs DOC Formats

DOCX and DOTX files cannot contain macros, so you know they are safe. DOCM and DOTM files can contain macros, so you know they may pose a risk. Their icons have a yellow script in the bottom left corner with an exclamation mark in it.

Word 2007 puts security warnings in a bar below the ribbon rather than in a modal dialog so they don’t interrupt your work.

Clicking the Options... button allows you to enable the macros.

The Trust Centre (Developer | Code | Macro Security) lets you change the security settings.

You should digitally sign your code so that you can be recognised as a Trusted Publisher. For testing purposes, on your PC only, you can create a digital certificate, with which to sign your code, using SELFCERT.EXE which you should find installed on your PC (probably in C:\Program Files\Microsoft Office\Office12). For distribution to other people, you should get a trusted code signing certificate from your IT department, if they have the facilities to create one, or from a 3rd party such as Thawte or Verisign.

You sign your code in the VBA Editor (Tools | Digital Signature...) and then save your document/template. When anyone opens that document for the first time they are shown that the code is signed, can examine the certificate and choose whether to trust it or not. The signature proves that the code came from you and has not been tampered with since you signed it. Any attempt to tamper with the code will break the signature and that will be reported to the user. The user can choose to trust all code signed with a particular certificate and that puts the certificate into the Trusted Publishers store. Certificates are valid for one or two years. To avoid signatures created with the certificate also becoming invalid at the end of the certificate’s life, you should timestamp the signature. (This prove the signature was created within the valid period of the certificate.) There is a registry setting which will cause this to happen automatically whenever you sign code.

I don’t like having Trusted Locations because that means that any document or template in those folders is not subject to security. I prefer to disable all Trusted Locations and insist on digital signatures on all documents or templates with macros.

ken sheridan say: The structure you've posted is really just a partition of a single table into various categories, not a modelling of the reality in terms of its 'relations'.

When considering your model its is important to understand how a relational database represents the reality which it is modelling.  It does so in terms of entity types and relationship types.  An entity type is a class of the reality, e.g.  each one of use is an entity of type people.  In a company each member of staff is an entity of type employees.  An entity type is not necessary a physical class, it could be a process for instance, e.g. Sales Orders.

Each entity type will have attributes, e.g. the Peoples entity type can have attributes such as gender, date of birth, nationalities (not just one, my family has both Irish and British nationality – I'll come back to this later) etc.

Entity types are related to each other in a number of ways.  The entity type Customers is related to the entity type SalesOrders in a one-to-many relationship type, i.e. each customer can be the subject of many orders, but each order can relate to one customer only.  In the case of People and Nationalities the relationship type is many-to-many, i.e. each person can be of one or more nationalies.  In the case of SalesOrders (or purchase orders) the relationship type with products is many-to-many, in that each order can be for one or more products and each product can be in one or more orders.  A many-to-many relationship type is modelled by a third table, e.g. OrderDetails which resolves the relationship type into two one-to-many relationship type's by having foreign key columns OrderID and ProductID.  Note that a relationship type is really just a special kind of entity type so will often have non-key attributes, e.g. an OrderDetails will have attributes Quantity and UnitPrice, each modelled by a column in the table.

A very important principle of the database relational model is the Information Principle.  This requires that data is stored at column positions in rows in table and in no other way.  Data must not be encoded as table names or column names.  For instance a table FruitStocked would not have columns named Apples, Oranges, Bananas, Pears etc.  It would have single column in which each fruit name would be a value in a separate row.

Another important mechanism of the database relational model is Normalization this is a formal process by which a non-normalized table (such as you'd have in a spreadsheet for instance) is decomposed into a set of related tables.  The key principle here is that each non-key column in a table is determined solely by the whole of the primary key of the table.  A table which satisfies this requirement is in Third Normal Form (3NF).  There are lower and higher normal forms, but 3NF is the one for relatively inexperienced developers to always bear in mind.

The relational database model is far more complex than the few points I've addressed above of course.  There is a vast amount of academic literature on the subject going back to Codd's original introduction of the model around 1970 in his seminal paper on the subject.  The pints I've addressed above are some of the key issues, however.

ken sheridan say:

You know your business better than anyone else, and certainly far better than we do, so you are in the best position to identify its entity types and their attributes.  For the moment forget about computers and databases and think about what makes up the model of your business in terms of the real world entity types and the real world relationship types between them.  Then sit down with a pencil and paper and draw out the model as an 'entity relationship diagram'.  This is similar to what you see in a database's relationship window in which each table is represented by a box, with directional lines indicating the relationships between them.

The most striking thing about your current tables is that, from what columns are visible, each table seems to have the same set of columns, which in database terms means the same attributes.  That would not be the case in a well structured database, where each table would have a distinct set of columns representing the attributes of the entity type which the table is modelling.

If you download my DatabaseBasics.zip file from my OneDrive folder at the link I posted earlier, and go to the section '4.  Bringing the model together' you'll see that each of the tables in this simple model of a contacts database has columns which represent attributes of the entity type modelled by the table:

The Contacts table for instance has FirstName, LastName, Address and CityID columns, the last being a foreign key, whereas the Cities table has City and RegionID columns.  Apart from the keys such as CityID and RegionID, which enable the tables to be related, there is no duplication of columns in different tables.  Note also how each table has a relatively small number of columns.  This is a sign of well-structured model, in which a table can have very many rows, but a limited number of columns.  Tables with very large numbers of columns, on the other hand, almost always indicate a poor design.  Where it is necessary to output data as a large number of columns this would be done by a query which joins multiple tables on the keys.
