Using Access with SharePoint - synch issues

I am developing a database with an Access 2007 front end that uses SharePoint 2010 lists as back end tables. The reason for this configuration is that users are able to take the Access database offline, and synch to SharePoint at a later time. SharePoint takes care of all the synch issues.

Since the database is for a bunch of engineers who are based in Australia but who do work in places like Chile, Africa, and Indonesia, this seems a reasonable approach.

One thing I dislike about the architecture is that whilst the tables are effectively managed on a server back end (i.e. SharePoint) the queries are stored in the Access front end. This means that most of the business logic is kept in the front end. I can probably live with this, but I don't like it much. I would much prefer the queries (or stored procedures) to be kept on the server.

Is there a way to use SharePoint to handle synchronization between the back end and front end and also store SQL queries on the server?
albert d. kallal say:

Unfortunately no, but on the other hand since you talking about an application that is going to on the user's desktop in disconnected mode, then that pretty much suggests that your queries,  program logic,  and most things have to run on the user's desktop anyway. 

 

In other words it kind of a contradiction in stating that you want your application to function offline, but then all of a sudden you want some queries and processing to occur on the server side?

 

In effect you're asking to duplicate the work and duplicate your designs on both ends of the spectrum (client side, and then again on the server side).

 

Keep in mind that if you using access on a traditional split network in which if you have 20,000 records, and access requests ONE record, then only the ONE record is pulled across the network pipe down to the client. This is EVEN the case when your use and build the query in the client side application. So even with program logic running on the user's desktop, not all data is pulled down.

 

If you were to move the back end database part to SQL server in the above then AGAIN when you build the program logic and even build a query in the front end part, then again the ONE record only travels down the network pipe. Thus again the query is still built in the front in access database.

 

And in the case of access to SharePoint? Well in theory no record need to be pulled down network pipe because you already have a local copy of the data. (I should point out that the setup works far better in access 2010 and sharepoint 2010 that it does the previous edition with 2007).

 

So at the end of the day from an architecture point of view, I can't really say there's going to be a lot of benefits to having these queries managed and created on the server side. The reason being is you would ALSO need a means to create and edit and maintain and a way to execute such queries on the client based application when you are disconnected anway.

 

Having said the above, access 2010 and sharepoint 2010 (Enterprise edition) does have the ability to publish queries that you build inside of the access application.  However once again because that access desktop client keeps a copy of the data local, then the local query system is used EVEN when you do this. So once again there's no need (nor advantage ) to having such queries run on the server side.

 

However the distinct advantage or significant difference in access 2010 is in fact that you can published web reports and web forms to SharePoint. In this case then honestly the web queries you build are going to run server side, and you don't need the client side application all, but you can use any web browser form a smartphone to an iPad.

 

So in this case you don't need Access client side.

 

However for your scenario of an offline disconnected mode, then such server side ability really doesn't help a whole lot unless there was some other particular issue are concerned that you've not clarified.

hansv mvp say:

Let's say you have the unrounded times in C2 and down.

In D2, enter the formula

=FLOOR(C2,1/(5*86400))

Format D2 as mm:ss.0 (time with tens of seconds) and fill down.

If you prefer a macro, you can select a range with unrounded times, then run the following macro:

Sub RoundToFifths()
    Dim c As Range
    For Each c In Selection
        c.Value = Application.Floor(c.Value, 1 / (5 * 86400))
    Next c
End Sub

daniel g. grau say:
Question-what is the best way to hide the numerical result of the sequences?
Format them with the Hidden attribute. Then you can click on the ¬∂ button in the Standard toolbar to toggle their display. Printing them or not is controlled through a Word > Print preference.

Why does the last bookmark show the total of all the numbers?
It's not the bookmark that shows it, but the SEQ field itself. The bookmark is just a label to that portion of the document, so the previous REF fields will know where to pick up their data.

If you have a total of "X" SEQ fields, you will get "X" numbers that are part of that sequence.

The formulas can be changed so that you won't need that last separate SEQ, but it's messier since the last combo field must contain the bookmark. This will yield 5, 4, 3, 2, 1:

   {= {REF Last} - {SEQ Num) + 1}
   {= {REF Last} - {SEQ Num) + 1}
   {= {REF Last} - {SEQ Num) + 1}
   {= {REF Last} - {SEQ Num) + 1} 
   {= {REF Last} - [{SEQ Num}] + 1}

where the brackets enclose the very last SEQ field (value = 5) with a bookmark called "Last." So the calculations performed in each case are:

   5 - 1 + 1
   5 - 2 + 1
   5 - 3 + 1
   5 - 4 + 1
   5 - [5] + 1

This can be done by using formulas or macro. I prefer macro. :-)

I assume that the PDF data is in column A and the data blocks are all equal in size.

Are they not, then you have to adjust the size. I can not that for you, unfortunately, you have not given enough information about your data structure.

Andreas.

Sub TransposeData()
  Dim FromR As Range, ToR As Range
  'First vertical data
  Set FromR = Range("A1:A6")
  'First cell for horizontal data
  Set ToR = Range("C1")
  Do
    'Copy vertikal data
    FromR.Copy
    'Transpose it to horizontal
    ToR.PasteSpecial Paste:=xlAll, Operation:=xlNone, SkipBlanks:=False, Transpose:=True
    'Setup Range to next data
    Set FromR = FromR.Offset(FromR.Rows.Count, 0)
    Set ToR = ToR.Offset(1, 0)
    'Loop until empty cell is found
  Loop Until IsEmpty(FromR(1, 1))
End Sub
