joeu2004 say: In reply to deleted message

Dana wrote:

It may have to do with the way ( ) holds off rounding when it thinks there's another calculation pending.  This is because ( ) has higher precedence.  I don't know for sure thou.  Similar to the following idea:

= 10.3 - 0.2 - 10.1
0

= (10.3 - 0.2 - 10.1)
1.77636 E - 15

Dana's characterization of the demonstrated behavior is incorrect.  Parentheses around an expression do not "hold off rounding" per se.

The only reason why =10.3 - 0.2 - 10.1 returns exactly zero (0.00E+00 when formatted as Scientific) is because Excel arbitrarily substitutes zero for the exact arithmetic result when Excel deems that the operands of the last subtraction (or last addition of operands with opposite signes) are "close enough".

This dubious heuristic is poorly described under the misleading title "Example When a Value Reaches Zero" in http://support.microsoft.com/kb/78113.

But the heuristic is applied inconsistently.  For example, IF(10.3 - 0.2 - 10.1 = 0, TRUE) returns FALSE(!).

Likewise, =10.3 - 0.2 - 10.1 - 0 returns about 1.78E-15.  The redundant -0 is sufficient to thwart the poorly implemented heuristic.

Similarly, =(10.3 - 0.2 - 10.1) returns about 1.78E-15 because the redundant parentheses are sufficient to thwart the heuristic.

There is no good reason for these inconsistencies.

And yes, that is why =(SUMSQ(C8:F8)-SUMSQ(G8:J8)) returns the exact arithmetic result, whereas =SUMSQ(C8:F8)-SUMSQ(G8:J8) might return exactly zero arbitrarily.

But bear mind that the latter form does not "always give the right answer".  For example, =10.1 - 10 - 0.1 does not result in exactly zero.

acg.chicago say:

Okay, let me see if I can clarify my previous reply; please correct me if I am misunderstanding the situation here.

The problem lies in the incorrect internal naming of the earlier release of the Arial Narrow fonts, such that any utility trying to work with them will be unable to tell them apart. The MS hotfix, as I understand it, would have delivered a corrected set of Arial Narrow fonts for MS utility use. (Word or whatever) However, Adobe InDesign may be configured to use its own font resources (either in place of or in addition to font resources elsewhere), and so it's those copies of the Arial Narrow that you need to track down and replace with the updated ones.

At this moment I'm not on an office machine, only a family PC with Adobe Reader on it, so I'm going to have to wing it on the following, but here's where I think you may need to look:

Adobe applications such as Reader, Acrobat and (I assume) InDesign keep their own font resources under the C:\Program Files\Adobe\... path, such as (on this machine) C:\Program Files\Adobe\Reader 9.0\Resource\Font for the font resources used by Adobe Reader here. I believe you should find a similar path for your InDesign installation, that will lead you down to a set of fonts including Arial Narrow. If you use the Microsoft Font Properties Extension to examine those fonts, I believe you'll find the "StyleName: Narrow" goofup there, and those are the fonts needing to be updated with newer copies. (You should also find an AdobeFnt*.lst font resource listing file in that folder, and browsing that in any text editor may also bring up the Arial Narrow font information showing the naming error.)

I'm afraid that's all I can offer on a Sunday night, but see if that helps, or if I'm going off in the wrong direction.

-- Andy
