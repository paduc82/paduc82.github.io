  • If using webmail, does the issue with sending emails also occur when trying to accessing the account from the same computer using a different browser? What about a different computer?

 

With regard to not receiving emails, you can check the mail filters of your account that could be causing the issue.

 

Check your Junk folder

1. Sign in to Hotmail.

2. Click on the Junk in the left pane.

3. Check to see if the missing mail is already in this folder.

4. If you see any messages that should have been delivered to the Inbox here, click to put a check in the box to the left of each message and then click the Not Junk button.

 

Check your Blocked sender list

1. From the Hotmail Options screen under Preventing junk email, click Safe and blocked senders.

2. Click Blocked senders.

3. In the Blocked senders list, select the name or domain (the part of an e-mail address that follows the @ sign, such as hotmail.com) that you now want to receive mail from, and then click << Remove from list.

 

Add the email address to your Safe senders list

Old thread, but hopefully others will find this useful.  We've noticed some of the same issues with certain email addresses.  The problem has something to do with how Outlook is reading the contact.  If the "display as" is only showing their name when you send an email, it will never reach the person.  To make things worse, it doesn't give a bounce back either.

The solution is pretty easy, but annoying:

1. Open the contact information card.

2. Delete the name below the email address and hit tab.  It should now show their name with their email address in parenthesis.

3. Save and close the contact.

4. Create a new email and begin typing their name.  When it shows up in the autofill field, make sure their name is highlighted and hit the DELETE key.  (This will only delete the autofill field name, not the contact).

5. Click TO: and find their contact.  Their name and email address should now be displayed in the new email.

The other alternative is to completely empty out your autofill list so you don't make any mistakes with old email addresses on file.  It will be annoying for a while though while you rebuild your autofill list.

This applies to Windows XP, 7, 8, and 8.1.  We are all using the new version of Outlook (2013) and are all on MS 365.

barry xie msft say:

Hello VuNguyen9,

 

Thanks for your information. Generally, this issue occurs if the SetRules call to the Mailbox Replication Proxy (MRSProxy) service fails. This issue may also occur if the mailbox that's being moved contains many junk-mail rules and user rules.

 

To resolve this issue, change the values of the MRSProxyHttpsBinding settings and of the MRSProxyWSSecurityBinding settings in the Web.config file on every Client Access server in the on-premises environment that's in the path through which the mailbox passes when it is moved. To do this, follow these steps:

  1. On the Client Access server, locate and then open the Web.config file.

    Note In Exchange Server 2010, the Web.config file is located in the following folder:

\Program Files\Microsoft\Exchange Server\V14\ClientAccess\exchweb\ews

 

  1. Change certain values of the MRSProxyHttpsBinding settings and of the MRSProxyWSSecurityBinding settings from 1048576 to 8388608. Then, save the file. 

    The following example shows what this section of the Web.config will look like after it's changed:

 

<binding name="MRSProxyHttpsBinding"> <reliableSession />
<textMessageEncoding>
<readerQuotas maxDepth="32" 
maxStringContentLength="8388608" 
maxArrayLength="8388608" 
maxBytesPerRead="4096" 
maxNameTableCharCount="16384" />
</textMessageEncoding>
<httpsTransport authenticationScheme="Negotiate"
maxReceivedMessageSize="8388608" />
</binding>
<binding name="MRSProxyWSSecurityBinding">
<reliableSession />
<textMessageEncoding>
<readerQuotas maxDepth="32" 
maxStringContentLength="8388608" 
maxArrayLength="8388608" 
maxBytesPerRead="4096" 
maxNameTableCharCount="16384" />
</textMessageEncoding>
<httpsTransport authenticationScheme="Anonymous"
maxReceivedMessageSize="8388608" />
</binding>

 

  1. Restart IIS by using the iisreset command.

 

Regards,

Barry

On the File tab, choose Open. Scroll down both the page and, if necessary, the file list (there are two scroll bars). At the bottom of the file list, click Recover Unsaved Documents.

FYI: I (Rohn) have not tested these tools in Windows 8 (just havent had an excuse to make/waste the time), but I dont expect any problem. The underlying mechanics of Windows 8 are essentially the same as Windows 7 if you ignore the Metro User Interface. Think of metro as lipstick on a pig. Normally you have several resources to find your Retail Product Key.

1.      Your original package or purchase confirmation e-mail

2.      Online download account

3.      You can extract the Product Key from the machine itself

4.      Image copy backups of system with Office installed

5.      Activation backup utilities

From what Ive read, MS uses a file called TOKENS.DAT to store encrypted product activation keys:

 

C:\ProgramData\Microsoft\OfficeSoftwareProtectionPlatform\Tokens.DAT

C:\Windows\ServiceProfiles\NetworkService\AppData\Roaming\Microsoft\SoftwareLicensing\Tokens.data

C:\Windows.Old\ProgramData\Microsoft\OfficeSoftwareProtectionPlatform\Tokens.DAT (if you did an in-place Windows Upgrade)

After you extract the Product Key, make sure to write it down and keep it in a safe place. Burning a copy of the installation file to a CD would also be a good idea if you dont have the installation disk.
