diane poremsky mvp (slipstick.com) say:
>> The Outlook 2016 launcher insists on connecting an email account before I can open Outlook 2016.
You need to set up a profile the first time you run it, before Outlook will open. It will add the accounts as Microsoft Exchange accounts and sync down the contents of your mailbox. It won't sync contents of other folders in Outlook desktop up to the server.

>> After I get Outlook 2016 set up and synchronizing with Outlook.com, I will have my various IMAP email accounts connected to both 2016 on my PC and Outlook.com. That seems to me at least a redundancy, and possibly a conflict problem. What is the recommended practice for dealing with that?

If you add the imap accounts to outlook and have them configured in Outlook.com as connected accounts, yes, it will be redundant and you'll have duplicates of all messages.

You won't be able to reply from Outlook desktop using the connected accounts currently - you'll need to log into outlook.com to reply. For this reason, my recommendation is to not use connected accounts for accounts you actively use to send email. Remove them as connected accounts and set them up in Outlook desktop. Connected accounts are great for addresses you need to monitor but not send mail to.

Good thing you haven't put any data intro tables yet, because your tables are not well designed. 

Having a separate table for each course is wrong design. It encodes data using the table, rather than a field in the table identifying the course.


let me explain that a database is a collection of tables. Each table defines a specific type of entity. So your point about not "having to do a database based on the trainings and a separate one based on the employees" doesn't make sense.

So, you appear to have 2 main entities here. Employees and Training courses. So you should, as Duane described, have a table of employees and a table of courses. An entity has attributes, these make up the fields in your tables. So attributes of the employee entity are things like first & last name, DOH, etc. Attributes of a Course would be the name, a description, ehter its required and, if so,  when it was required to be completed. 

The next thing you need is a record of which employee took which course and when. You have a many to many relationship between employees and courses (an employee can take multiple courses and a course can be given to multiple employees). So that's where the third table Duane mentioned comes in. That table Would contain the EmployeeID, the CourseID and an indication of when it was given.

With the above structure, you can easily do reports for an individual employee, individual course or specific requirement.

The links John gave go into much greater depth about entities, attributes, normalization etc. So I strongly suggest you stop any design until you understand better how to design.

A Many to Many relationship is just composed of two One to Many relationships. In your example, Duane's and Scott's suggested tables are Employees, Courses, and  Enrollment.

There's a one to many relationship from Employees to Enrollment - each employee can enroll in zero, one, or multiple Courses.

When you have all three tables populated (using Forms and Subforms as described in, among many other places, Crystal's tutorials in the link I posted), you can then create a Query joining all three tables to see who is enrolled where.

suzanne s. barnhill say:

The goal is to have the verses left-aligned but visually centered on an average-length line. I've never figured any way to do this other than to change the left indent for each poem. What I would suggest, however, is that you define a series of styles with progressively larger left indents: say 0.5", 1", 1.5", and so on. Then apply the style that works best for a given poem.

 

To be clear, you should not be justifying poetry; the lines should be ragged right.

 

Some people just center the whole thing, but that is lazy and incorrect, as the verses do need to have a uniform left margin.

galahad72 say:

Instead of taking the average line length, for typesetting poetry it seems more common to take the longest line for centering. That's more likely to convey a centered appearance for the poem as a whole.

Then, using the Left Indent control on Layout, keep trying point values until a second line is visually left-aligned with the centered line. Then apply that value to a selection of all lines above and below the centered line, at least in the current stanza. The value that seems to work can also be applied as the value of the indent for the centered line, once its centering is removed. That way the left alignment is exact for all lines.

What is not so great about this overall approach is that it's dependent on the initial line length between the margins. Change the margins or the paper width and you have to start again. It doesn't work for table cells.

When the poet has indented of the some lines him- or herself, I can only suggest uniform fixed spaces for their indents.

suzanne s. barnhill say:
As previously noted, individual lines should not be centered. The goal is to have the entire poem roughly centered, not necessarily on the longest line (which might even be run over) but on the general mass of the poem. Each stanza of the poem should be a single paragraph, so there will be a single left indent for the entire paragraph. As for runover lines, tabs are preferable to spaces.

galahadbr say:

First I should say that it's the poet's own indented lines (perhaps every 2nd line, or the last 2 in a Shakespearean sonnet) for which a fixed space seems to be in order. Whether runover lines should be indented to the same value is another matter. Twice the indent is one approach I've noted for runover lines.

Let me apply Suzanne's advice in my own words.

Don't center a poem based on the longest line. Rather, center it visually for the poem as a whole. You could start with a stanza, but it's not necessary.

Within a poem, the line endings should all be manual line breaks, except possibly for the last one. Use the Left Indent control on Layout to push the stanza into the visual center. Record the indent value.

Select the remaining stanzas. Apply the value in the Left Indent control to the other stanzas.

And that's all there is to it. So simple! And it works within a table cell, which is my present context. Thank you.

As for the poet's own indents: use a fixed space of your choice for the desired indent. Copy it to other lines that the poet indented.*

Alternatively, for lines that the poet has indented you could define a style with a hanging indent and no Spacing After and make each couplet a paragraph (adding Spacing After to the last paragraph in a stanza).

The main report's record source query should not have a join to the subreport's table.
