marshall barton say:

Your situation remains in the unusual category.  If the object names are unique and you are using them as foreign keys in another table, then they should be the primary key in the objects table and you have no use for the ID field.  If the object names might be edited to correct spelling or whatever, then the ID field servers as a never changing unique value that is better used as the primary key and therefore should be used as the foreign key in other tables.

In other words, make up your mind.  If the ID field is the primary key, then a simple select query should display the ID field instead of the name.  When you need to see the name in a report or as a read only field in a form, then use a query to Join the other table to the Objects table as the report/form's record source.  If the name field needs to be edited in a form, use a combo box as you had it before and it will display the name while saving the ID.

OR, if the name is the primary key, then you should delete the ID field because it serves no useful purpose and is confusing you.

Hi,

formula (in cell C3), returns 6, 5, 0
depend on values, from another cell
say, in cell A1..

assuming that data is on sheet1,

right click on sheet1 tab, select view code and paste in...

 

 

Private Sub Worksheet_Change(ByVal Target As Range)
Dim formulaCell As Range, myCell As Range
Set formulaCell = Range("C3")   '<<< change
Set myCell = Range("A1")    '<<< change
If Not Intersect(Target, myCell) Is Nothing Then
[formulaCell].Font.Color = 0
If [formulaCell].Value = 6 Then [formulaCell].Font.Color = vbRed
If [formulaCell].Value = 5 Then [formulaCell].Font.Color = vbGreen
If [formulaCell].Value = 0 Then [formulaCell].Font.Color = vbYellow

End If
End Sub
