msoisgood say:

My daughter needed the same thing. Here's a workaround I discovered for Word on our iPad2:

1) Open a New Template provided in Word

2) Create the Template you want to save by making changes to the selected Template

3) Click the circled Left Arrow to exit the document, then click Save

4) Name the file (ex. Template - Times New Roman)

5) Click iPad

6) Click Save

7) Go to Recent

8) Click the Pin to the right of the new Template you just created so it moves to the top of the file list under the heading PINNED

9) When you want to use that Template, open it, click the File Menu option (looks like a bent page with three dots on it), click Save a Copy, Rename the Template For the new file you're going to create, click iPad, click Save.

10) Click the circled arrow to exit the new Template 11) Go to Recent and select the renamed file, start typing!

12) Your original Template is still listed under Pinned Files and is ready for you the next time you need it.

13) Make as many Templates as you want using the same procedures above

TIP: If you name all the templates you create with the starting word Template, they will all be grouped together alphabetically for easy locating under the Open tab. This way of naming is especially useful if you have other non-Template files PINNED since PINNED files are listed in order of creation with most recent file PINNED listed first

Even Microsoft Visio (an enterprise software focused on flowcharting) doesn't auto-create a very nice shape with just your first layer (two entities, "co-owned" by three people on the same horizontal).

In general, I think connecting to different points on the box is fairly easy, and you've already intuitively layered your boxes so each one is below its "dependencies" on a new row. Actually programming the layout would be a significant chore (again, you'll have to do better than Visio's built-in logic). Your colored lines are one option, assuming none of your end users are colorblind, but it will still be a chore to autogenerate a clean layout from source data, IMHO.

Many thanks for your comprehensive reply - it sounds like I may have to accept an element of manual moving of boxes.

It would be useful to know how to work out the connection points - number and which ones so that they can be moved to make it clearer. Also if I went down this route is there any way not to lose the actual connection so that if some one manually moves the box the connectors continue to follow.

kergga say:

Great questions.

The shapes have their own connection points (using "connection point" as a technical term), and you can address those directly, but a rectangle will only have one connection point on the cetner of each side, so that isn't what you want. You will be identifying alternate locations to start/terminate your lines (more on that below). The downside is that only official connection points drag the lines when a connected shape is moved, you would have to re-run your code or re-layout your lines if you moved one of the rectangles.

I suppose it would be possible to store all the shape information, and when a shape is moved check each of your line termination points to see if it intersected with the border of a shape, and if so, identify that shape, find the shape current location, and replace the connector.

For actual seperation of the connection points; each of your rectangles will have the properties of .Left, .Top. .Width, and .Height. Your layout algorithm will have to tell you whether the connector is connecting to the bottom of a parent shape or the top of a child shape. Your "Real" connector will be at bottom center, which you can address using the connector property location, something like (Parent.Left + (Parent.Width *.4)) for a line a little to the left of the midpoint and (Parent.Left + (Parent.Width *.6)) for a little to the right.

Caveat: If it were me, I would use all fake connectors and ignore the actual shape connectors. Otherwise you'll have some lines that use the actual connector on one (or both) ends and will relocate automatically when the shape moves, and other lines that will have to be moved via your code, which means you would have to keep track of whether each line was liked to a connector. Easier in my mind just to treat them all the same and relocate them all the same way.

I haven't had a project where I looked at line crossings. I would be surprised if Excel pre-calculates or identifies those points automatically. You may have to write code/logic to figure out whether lines cross, and also whether the line crosses another shape (Excel doesn't auto-route around shapes if you are using the basic rectangle and lines from the shapes menu). You can adjust where the 90-degree changes take place by adjusting the location of the yellow box (shown when the line connector is selected) using something like Shaperange.Adjustments.Item(ItemNum) = .5  but play around with the shapes first - the yellow connector only allows you to adjust the line in one direction, and the direction varies based on however Excel calculated the orientation of the line. I suggest you drop a bunch of boxes and a few lines on the sheet, then try putting the connector ends in different locations, and see how you can adjust the lines manually so you understand the limitations (and maybe someone else on the forum will have insight into how Excel calculates which direction the line can be moved, and whether you can change that in code). Either way, the basic squared-off line isn't going to give you a complicated routing, it will still be pretty basic. Try recording short macros when you are moving the lines around to see how Excel generates the code.

One other thought- Maybe your code could connect all the lines to the rectangle connection points by default, so they move with the boxes (if that is needed), then separate code to move each line endpoint off the connector (left or right by a tiny amount). This would be easier to code, because you would just loop through each shape, then loop through each connector to find out if any lines are attached. If so, just shift each one a little to the left or right until they are all disconnected from the connector and separate from each other. To reconnect, use the coordinates of each line endpoint to find the nearest connector on the same plane (and maybe limited to horizontal plane), and reconnect them before the shapes are moved. That still leaves you with having to develop the line layout code, but might help if you think the box locations will change often.
