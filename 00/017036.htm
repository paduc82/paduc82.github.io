Dates are stored as serial numbers in Excel and displayed as per their cell format as Dates. Excel stores dates as sequential serial numbers so that they can be used in calculations. By default, January 1, 1900 is serial number 1, and January 1, 2008 is serial number 39448 because it is 39,447 days after January 1, 1900.

To enter a date of 12th feb 1956 you will need to type the serial number 20497 and format the cell as a date format. Instead it would be easier to type in 12 Feb 1956 in a cell which is formatted as a date or type in 2/12/1956 if the date format is yy/mm/yyyy.

ken sheridan say: You can remove all the quotes characters with an 'update' query like this:

UPDATE [YourTable]
SET [Field1] = REPLACE([Field1],"""",""),
[Field2] = REPLACE([Field2],"""","");

Changing the table and column (field) names to your own and adding further rows for each column as necessary.

In query design view you'd enter Replace([Field1],"""","") etc in the 'update to row of each column, changing the column name in each case.

The point Scott is making is not merely that the table is imported from Excel, but that a relational database is, ....well, relational, i.e. it models the underlying reality in terms of its entity types and the relationships between them, with each entity type being modelled by a table and the attributes of the entity type in question being modelled by the columns in the table.  There are formal rules which govern this to ensure that, as far as is possible, the database contains no redundancies which can undermine the integrity of the database by allowing update anomalies, i.e. each fact is stated once and once only.  At only one place in a database for instance would the fact that London is in England be stated.  If it were not, then there would be nothing to stop London being placed correctly in London in one row and erroneously in France in another. We know which is correct and which not, but a visitor from Mars would not.

The process by which redundancies are eliminated is known as 'normalization' and is achieved by decomposing a table into two or more related tables.  You'll find an example as DecomposerDemo.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo takes data from a simple Excel worksheet and decomposes it into correctly normalized related tables by executing a series of 'append' queries in a specific order.  The rule of thumb is that rows must be inserted into a referenced table (on the 'one' side of a one-to-many relationship type) before rows are inserted into a referencing table (on the 'many' side of the relationship type).
