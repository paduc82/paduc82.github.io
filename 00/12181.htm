tiparker say:

I managed to get it working in Visio, after attempting to enable tracing as defined in the URL that you provided. Enabling tracing from both the interface afforded by  "%WINDIR%\System32\odbcad32.exe" and via editing the registry was to no avail, i.e. in spite of tracing being enabled a log wouldn't generate.

This led me to explore the directory in which the ODBC 5.3 drivers were installed on my windows machine, in my case "C:\Program Files \MySQL\Connector ODBC 5.3". The "README.txt file" in this directory documents usage of the "myodbc-installer.exe" CLI tool.

Running said tool, with elevated privileges, without any parameters afforded exactly the details necessary to remedy my issue. In this case:

  • myodbc-installer.exe -d -l
    • MySQL ODBC 5.3 ANSI Driver
    • MySQL ODBC 5.3 Unicode Driver
  • myodbc-installer.exe -s -l
    • no output
  • myodbc-installer.exe -s -a -c2 -n "{name of DSN to create}" -t "DRIVER=MySQL ODBC 5.3 ANSI Driver;SERVER={IP of server};DATABASE={DB name};UID={DB User};PWD={DB Pass}
  • myodbc-installer.exe -s -l
    • Name of created DSN

I then selected this newly created DSN from within Visio and viola, it worked! I'm not sure why the other ODBC DSN defined via "%WINDIR%\System32\odbcad32.exe" worked just fine in other programs and not Visio, but doing the above steps did the trick. Thank you for helping me to get in the right direction Al.

One other important factor is the architecture of your app.  We had a 64 bit Win 7 system that we installed the MySQL 64 bit ODBC drivers, first for current ODBC 5.3 and then an older ODBC 5.1 version during troubleshooting.  No matter what we tried from above, the list of available connections in Vision 2010 was always blank, and there were no MySQL ODBC drivers available in Visio (when you "Setup" the Generic ODBC item).

We Googled for a lot of different things, but finally found another MSAnswers question where a MS Expert asked a user having a Visio ODBC problem if their MSOffice was 32 bit or 64 bit.  We checked her Visio 2K10 and it was 32 bit, so we downloaded the 32 bit version of MySQL ODBC 5.3 and installed it right over the top of the old one.  She deleted the User DSN and System DSN, she recreated the User DSN, and she restarted Visio.  When starting the Reverse Engineer wizard, now the MySQL ODBC drivers were available in the Setup screen, and the defined User DSN was visible.  There was a warning message complaining about something, but we knew from other articles found online that this was an expected warning (not a fatal error).  Then ODBC worked as expected and the Reverse Engineer function produced the desired ER Diagrams.

This is a bug in OneNote 2007 that has been fixed in 2010.  The issue is caused by multiple auto-sized table columns that get confused when they all try to fit on a PDF page.  If you encounter this bug, first try restarting OneNote to force it to recalculate table layout.  If that doesn't work, the recommended work-around is to manually resize the table columns to force them to use a fixed size.

Full Page View isn't the real culprit.  Auto-sized columns depend on the page size, which may depend on the window size.  Because Full Page View increases the page size it could affect column sizes, but not always.

That is by design.

You cannot control how Access chooses to run your query, but indeed it often, if not always, runs the query on a single cpu.

If you want more performance, you will need to upgrade to a server-based RDBMS such as SQL Server, which can, if the conditions are right, run your query across all CPUs.
