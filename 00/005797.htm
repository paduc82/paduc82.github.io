    Procedure: Account Hard-Matching - Microsoft Community

    alex rizeanu msft say:

    Procedure: Account Hard-Matching

    Technical Level : Intermediate

    Summary

    The following article will detail the procedure of performing a Hard-Match between an on-premises Active User and an AzureAD (Office365) user.

    For more details on when this procedure should be used please see: Mailboxes - AzureAD Account Matching - Source of Authority Transfer to On-Premises


    Details

    Step One: Pause directory synchronization

    We do not want a delta sync firing off in the middle of our work. You can either disable the scheduled task used trigger the sync in 3 hour intervals or you can disable the service itself.

    Step Two: Backup EXO account attributes

    Obtain all attribute information including a full list of email addresses from the cloud account. First, connect to Exchange Online using remote PowerShell. If the recipient type is a mailbox, run the following commands: 

    $FormatEnumerationLimit = -1 # so we don't truncate the email addresses output
    Get-mailbox <mailboxID> | fl

    Step Three: Prepare the On-Prem Account

    Proper account provisioning in an Exchange Hybrid scenario using dirsync requires that you have at least the on-premises Exchange management tools installed in your environment. This article will not deal with methods outside of using the Exchange management tools. If you don't have them installed, you need to install them. If you are linking an on-prem account to an EXO mailbox, then the proper procedure is to create an on-premises remote mailbox account. This can be done directly when you create the account or an existing AD account that is not yet mail-enabled.

    Scenario One: New Account 

    new-remotemailbox -Name -UserPrincipalName -Alias -DisplayName -FirstName -LastName -Initials -PrimarySmtpAddress -RemoteRoutingAddress
    #For room or equipment mailboxes, add the -room or -equipment switch.
    #Once the account is created, then you can add all of the additional email addresses that you need to 
    preserve
    set-remotemailbox <accountYouJustCreated> -emailaddresses #Verify that the UPN is set correctly on-premises

    Scenario Two: Existing Account 

    enable-remotemailbox <IDofExistingUser> -alias -primarySMTPAddress -remoteRoutingAddress
    #For room or equipment mailboxes, add the -room or -equipment switch
    #Once the account is mail-enabled, then you can add all of the additional email addresses that you need to 
    preserve
    #If you want to change attributes like UPN and displayname, do that here on this command.)
    set-remotemailbox <accountYouJustcreated> -emailaddresses 
    # Examples found here: https://technet.microsoft.com/en-us/library/ff607302(v=exchg.160).aspx

    Step Four: Convert On-premises AD ObjectGUID

    Use the Powershell Module for ADfor the following command on-premises: 

    # Gather the Object GUID of the on-premises AD object
    Get-ADUser <UserAlias> | Select UserPrincipalName, ObjectGUID

    Go to http://hassanu/cloudtools/Paste the GUID copied previously into the field under GUID to ImmutableID and click "Convert GUID to Immutable ID. (Copy the ImmutableID value reported)

    Step Five: Change the ImmutableID for the Azure AD object

    Connect to AzureAD Powershelland run the following commands: 

    Set-MSOLUserPrincipalName -UserPrincipalName <*** Email address is removed for privacy ***> -NewUserPrincipalName <*** Email address is removed for privacy ***> 
    # NOTE: You do not need to change UPN if Azure user UPN is already domain.onmicrosoft.com address
    Set-MSOLUser -UserPrincipalName <*** Email address is removed for privacy ***> -ImmutableID <Value copied earlier>

    Verify that the ImmutableID changed by running Get-MSOLUser -UserPrincipalName "*** Email address is removed for privacy ***" | FL ImmutableID , this should match what was copied previously.

    Final touches

    1. Change the password on-premises
    2. Change the UPN back to *** Email address is removed for privacy ***
      1. Set-MSOLUserPrincipalName -UserPrincipalName "*** Email address is removed for privacy ***" -NewUserPrincipalName "*** Email address is removed for privacy ***"
    3. Turn DirSync back on and trigger a delta synchronization.

    Validating the changes

    Login to Azure Portal and under Active Users search for the user in question. This should now give a status of "Synced with Active Directory"

    Back to Exchange Online support Corner
