jay freedman say:

Yes, you must unprotect the document in order to delete one or more sections, and reprotect it when the process is completed.

Between the Sub statement and the first Delete, insert the statement

ActiveDocument.Unprotect 

or, if you applied a password to the form protection, instead use

ActiveDocument.Unprotect Password:="whatever"

(of course, with the real password between the quotes).

At the end of the macro, just before the End Sub statement, include the corresponding one of these statements:

ActiveDocument.Protect Type:=wdAllowOnlyFormFields, NoReset:=True

ActiveDocument.Protect Type:=wdAllowOnlyFormFields, NoReset:=True, Password:="whatever"

    Eliminate Duplicate Records in a query in MS Access 2007 - Microsoft Community

    barbaraharrod say:

    Eliminate Duplicate Records in a query in MS Access 2007

    I have developed a query using 3 different tables.  The query contains duplicates in the Name field with different data in the other fields.  I am then using this query to prepare a summary report where the detail is hidden and only the totals show up.  I either need to find a way to eliminate the duplicates or not include them in my count for the summary report.  I have tried counting records and counting values but my total is the same. I want my total to show only 2 or eliminate the duplicates so it counts only 2.  Can anyone help me.

    Thanks,

    John Smith          Teacher                       

    John Smith          Parent

    Kay Ward            Student

    Kay Ward            Student Worker

    4        Count of Records

    ken sheridan say: 1.  Group the report by the name column.

    2.  In the group header add a text box with a Name property of txtNameCount, a ControlSource property of =1 and a RunningSum property of Over All.  Set the height of the text box and the group header section to zero to hide it.

    3.  In the report footer add a text box with a ControlSource property of =[txtNameCount].  This will show the count of distinct names.

    However, personal names can legitimately be duplicated, so you should have a distinct numeric PersonID or similar column as the primary key of a referenced People table and include this in your query.  You can then group the report on PersonID  not the name column to distinguish between two or more people of the same name.  Don't assume names will be distinct, even in small groups; I worked with two Maggie Taylors.
