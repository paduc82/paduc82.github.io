cmdrkeene say:

It doesn't, but it's user choice to do so. Windows will not delete a file that a user or another program created without your prior permission.  The automatic 3:00 AM maintenance window only cleans up junk related to servicing (updates) and a few other system-generated locations (old problem report logs, old blue screen logs, etc.), but it will not delete cached files.

The cleanmgr.exe program allows up to 999 combinations of cleanup schedules (admittedly more than any person would ever use).  Or you could even just schedule a command that is a single-line command "del c:\temp\*.*".  Although temp files are stored in hundreds of other places, it depends on whoever built the program. For example, most open source tools keep their temps on the same directory the tool is installed upon, probably for easy portability.

But before you go deleting these, consider that they serve a purpose.  You won't do any damage clearing your temp files out of your browser, for example. But then to access the same content again, it must be fetched from the server. Maybe you want this (don't want cached values, or maybe you don't care how much bandwidth you use) or maybe you don't (you want to read it later on a plane, or your bandwidth is very expensive per-kilobyte charges).

No one mentioned otherwise, and I'm not accusing anyone here of this attitude, but I have seen users that run "memory cleaner" programs, then wonder why their hard drive is thrashing loudly as they load programs and content. Well, duh, it's been forcibly removed from memory and now has to be reloaded from a slower source (a hard drive is hundreds of times slower than RAM, even for new solid state drives).

Also noteworthy: cached files will never slow down your computer. It's like a library of billions of books. It doesn't matter how many files you have, it matters how many are loaded into RAM and being processed simultaneously.  If you had two identical computers with the same programs, but added 100 GB of files to a \temp folder on one computer, it would not run any slower.  The only limit to this rule occurs when you have less than 5% free space and there is no longer sufficient buffer for paging file (virtual memory).

The bottom line from the system is to err on the side of caution and never delete something until the user says it's okay or requests it. The temp files can never do harm, but removing them could (or at least irritate the user).  So if you want it to be erased every week, you absolutely can, but you have to set it up the first time, either by using the built-in cleanup utility (which only checks that one folder), or deciding what you want the rule to be.

If you'd like I can post a batch file that scans for files modified within X days and moves them to the recycle bin (not deleting).

SELECT NameOfTheKeyColumn, C1 & C2 & C3 AS C4
