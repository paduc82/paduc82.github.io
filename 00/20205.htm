Verbose logging is a setting that exposes more information during the installation process. It will capture warning as well as error messages that provide us with clues to your problem.

To do onetime verbose logging:

Diagnosing When Setup Stops Responding

At times, Office Setup stops responding (hangs), and you do not receive any error message. The best thing to do in this situation is to restart your computer, and run Office Setup again with complete verbose logging turned on (with one additional option). To do this, start Office Setup. To do so, follow these steps:

  1. Click Start, and then click Run.
  2. In the Open box, type the following command line, and then click OK:

path\Setup.exe /L*v! C:\Verboselog.txt

Note that Path is the full path of your Office source location.

 

To enable Windows Installer logging yourself, open the registry with Regedit.exe and create the following path and keys:

HKEY_LOCAL_MACHINE\Software\Policies\Microsoft\Windows\Installer


Reg_SZ: Logging
Value: voicewarmupx

The letters in the value field can be in any order. Each letter turns on a different logging mode. Each letter's actual function is as follows for MSI version 1.1:

v - Verbose output
o - Out-of-disk-space messages
i - Status messages
c - Initial UI parameters
e - All error messages
w - Non-fatal warnings
a - Start up of actions
r - Action-specific records
m - Out-of-memory or fatal exit information
u - User requests
p - Terminal properties
+ - Append to existing file
! - Flush each line to the log
x - Extra debugging information. The "x" flag is available only on Windows Server 2003 and later operating systems, and on the MSI redistributable version 3.0, and on later versions of the MSI redistributable.

"*" - Wildcard, log all information except for the v and the x option. To include the v and the x option, specify "/l*vx".

Note This should be used only for troubleshooting purposes and should not be left on because it will have adverse effects on system performance and disk space. Each time you use the Add/Remove Programs tool in Control Panel, a new Msi*.log file is created.

When looking through the MSI logs we will typically want to look for a value 3 entry in the logs. Windows installer returns codes during the install which will indicate if a particular function was successful or not.

Value 1 = Success
Value 2 = Cancel
Value 3 = Error

failed to update headers in Outlook - Microsoft Community

acontrario say:

failed to update headers in Outlook

This is an old topic relating to IMAP syncing not working well enough.

The practical issue is that Outlook client stops synchronizing in the rights folders ("folders" for Outlook, "labels" for Gmail). But the label/folder "All messages" continues synchronizing. So, I should receive gmail messages in Inbox on the client. They arrive on my Android device. They don't show up in Outlook Inbox for the gmail a/c on some PCs.

I did not find my solution on any of MS forums, so I post it here as it may help others.

I have a stand alone Outlook 2010 client next to others at home - I am retired (Outlook 2013 & 2016).  So, this happens with a private Office install, not with a cy one nor with a cy server.  I use W10 on all PCs, uptodate. 

 

I got the nagging recurring issue: "failed to update headers" error message on the gmail private account. So, Outlook client does not work well but at least knows it doesn't and tell you in sibiline terms, not giving you a hint on the message where it encounters the problem.  Going into details, it elaborates further that "an IMAP command failed". thank you but this is too short to help. 

All other email accounts were working. I use Outlook client to consolidate, search and manage archives. It mostly works very well, even on very large PST files, much better than gmail products, when it works.

The connexion is IMAP. 

So, usual solutions:

Remove the account & recreate it. Usually it works but not always. 

I did this 7 times (number of versions I counted PSTs in email folder). I always use a dedicated email folder in a protected area not to mix things up in Microsoft technical folders. Microsoft by programming design philosophy does not help at all separating clearly program/application data and private user data. On the contrary. So, I manage this myself to facilitate backup (segregating what I can lose & my data) & for security purposes (trying to ensure all my data is in a place I can easily protect, including the backups). Just a reflex. But there are obvious limits to my approach. 

Of course, I also tried the old scanpst.exe to validate / repair if necessary the pst file if this useful or not. I saw it was still recommended on MS forums these days when looking for a solution. Of course, these are very basic advice with little use, and none to me.  

So, to no avail.

I don't know what exactly cured it but not willing to create an eighth version of gmail pst, here is what I tried in last resort. I tried to find another more reliable client than outlook also, but forgot my attempt. 

I went to gmail in cloud.

1) searching on all "MIME" or "failed receipt", I deleted all the messages containing MIME

2) I deleted all messages responses to emails sent with e-signature certificates attached (In Europe, it is easy to sign an email with an eID card (Belgium) or with a e-signature smartcard (Luxembourg) or with a certificate issued in compliance with EC e signature directive 1999 as reviewed by 2014 EC regulation). These are important because these messages receive a legal value similar to registered mail since 1999 or about.  There is a procedure to use a Belgian eID card with Outlook but it requires a small modification in the register, very easy but still a pain because not straightforward and no parameter (this is because the eID certificate is not linked with a unique email address).  But once it works, all registered mail can go through Outlook, esigned (but not encrypted),  instead of using physical registered mail.    

After doing the above deletion steps, my Outlook client started to receive again all the pending messages in all the right folders. Miracle! 

Why did I do this: on a previous occasion that I could not replicate here, the sync had also stopped. I looked where / at what time it had stopped. I deleted the "next message" in gmail cloud that should have come in. And the flow started again.  The next message was containing a response to an email with a e-signature certificate attached to it. So, I suspected those.  

Hope it helps as the topic is closed on most Outlook forums while still annoying.

I have not experienced it on Outlook 2016. So, probably closed with a fix for more recent versions.

Still, annoying for many people out there with older versions. 

But the approach may help with other type of messages that don't sync well, giving the same error message.
