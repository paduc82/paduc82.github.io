tom ogilvy say:

So I have to make a lot of assumptions. 

Assume your data is on a sheet named "data" and the data that you want to print is in the range

A6:M1000   45 rows will fit on each page.  

Assume the formatted sheet is named template and template has the start row in cell A1.  It has the five rows at the top in cells A2:M6. 

Then there are formulas in A8:M52 

In A54:M58 are your bottom rows that you want on every sheet.

You have set the print area on the Template sheet to print the range A2:M58

so the macro would look like this

Sub PrintData()

Dim sh as worksheet

Dim i as Long

set sh = worksheets("Template")

' now we loop starting at row 6 to row 1000 placing 45 rows in the

' template sheet on each loop (by updating cell A1)

for i = 6 to 1000 step 45    ' the step is the number of rows to print

  sh1.Range("A1").Value = i

  sh1.printout

Next

End Sub

If you want to number pages and so forth your would need to add that to the macro.  

Your formula in A8 would look like

=Indirect("Data!"&CHAR(COLUMN()+64)&$A$1+ROW(A1)-1)

You would then drag fill this in A8:M52

In the picture below I show the layout of the template sheet.  In A8 to M52 I show argument to Indirect showing how the address would appear based on the number displayed in A1.   So I have placed this formula ="Data!"&CHAR(COLUMN()+64)&$A$1+ROW(A1)-1 in A8 and drag filled it to A8 to M52, then hidden rows 11 to 49 to make the picture smaller.

Input Mask Rule

Also what is the character to use for 'any character'? I can use 0 for numbers ? for letters, what do I use for any symbol like "/"?

dirk goldgar say:

It may be possible to do this with an input mask, but I don't think I'd use an input mask for this.  Input masks are really only good when you can constrain the input to one simple, regular format.  I think instead I'd do two things:

  1. Restrict input into the text box to only the acceptable characters:  {0123456789'"/ }.  You can do this with code in the text box's KeyPress event, which will be called for each ASCII character typed in the box, and can cancel that character by setting the event procedure's KeyAscii argument to 0.  Note that the procedure will also receive the character (e.g., tab or carriage return) that brought the focus to the text box.

  2. In the text box's BeforeUpdate event, examine the value of the control (the user's entry) to make sure that it conforms to your allowed formats.  You'll find the Like operator useful for this, but you may need to use more than one Like expression;  e.g., if it's "Like this" then it's good, but if not then if it's "Like that" it's good, ... , else (if it's none of the valid forms) then show a message and set the event procedure's Cancel argument to True,

gustavbrock say:

And - further to Dirk's comment - users hate input masks because they are slow and restrictive.

So, accept any input. If the converter function fails on an input, cancel the update and pop a messagebox that also displays examples of typical acceptable input formats.

Users will quickly adopt the valid formats.

dirk goldgar say:

And - further to Dirk's comment - users hate input masks because they are slow and restrictive.

I'll qualify that a little bit, because people often forget a particular type of user.  Data entry clerks *love* input masks, in a form designed for rapid data entry, because they allow them to enter data rapidly, without error, and without lifting their hands from the keyboard.  Often these people are working from printed or manually-filled-out forms, and they get paid according to their throughput.

These days, this kind of data entry is much less common than it was 25 years ago, but it still exists.  That's what input masks are good for.

gustavbrock say:

Yes and no. For smooth operation - indeed for clerks - you most often also need a lot of code to make it work as expected by the user.

Two examples are here:

Entering ISO formatted date with input mask and full validation in Microsoft Access

Entering 24-hour time with input mask and full validation in Microsoft Access

dirk goldgar say: Agreed.  The critical phrase, as used in those article titles, is "with full validation".  Input masks can (in appropriate scenarios) ensure that the entry is syntactically valid.  It takes code to ensure that it's semantically valid.

dirk goldgar say:

Giving the problem a bit more thought, and considering you're probably going to want to  convert the feet/inches expression into a simple number of inches, it seems to me that you may want to use what's called a regular expression for this.  Regular expressions are not built into Access or VBA, but you can use them in your code by leveraging the "Microsoft VBScript Regular Expressions" object library -- you should be able to find some version of that library in the available references in the Tools -> References dialog in the VB  Editor.  Note: if I were doing it, I'd add the library reference while I was working on the code, and then when I was done I'd change the code to use late binding and remove the reference.

There are regular expressions for lots of different parsing problems published on the web, and I know there are some out there for this purpose.  On the other hand, regular expressions are potentially very complex and hard to understand, so you may not want to get involved with them.

scottgem say:

What Ken and I are saying is you should never have gotten to this point in the first place. And we don't understand how and why you got here.

First, let me say that if you delete a linked table you don't delete the table itself. The table will still remain in the file that was linked to. You just won't see it from the FE.

Second, what you have done is create a huge headache for yourself by keeping multiple copies of the same tables. As Ken said earlier, the backend should only have tables and the front end should have everything else. While you might have temp tables in the FE for processing and performance reasons, such tables would not keep data permanently. Also, while you might have copies of your BEs for test or development purposes, there should be one and only one production BE that your FEs link to (I'm not counting backup copies).

So, what I think you need to do here is determine which is your production BE. Make sure that has all your data in it. Then remove all other tables from your FE and relink to the what you have identified as production data. If you want to copy these tables into another file for safety you can do by exporting. 

Once this has been done, redeploy the FEs linked to the production tables and change the process that has been creating all these copies.
