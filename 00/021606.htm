The rest of the columns are dates that the vehicle is allocated - so in my query I have the date fields listed twice (one to sort the data and the other used as a column to display the data.

All works well, except the date displayed in the columns is not being displayed in ascending order (the rows are fine).  Could anyone advise what I need to do to force the column based data to be displayedin ascending order?

The SQL code is here:

TRANSFORM First(qryRosterView.fldFirstName) AS FirstOffldFirstName
SELECT qryRosterView.fldCarNo, qryRosterView.fldShiftType, qryRosterView.fldStartDate
FROM qryRosterView

ken sheridan say: To generate the SQL statement for the query you'd enter the start and end dates into two text boxes in an unbound dialogue form.   You can then build the string expression for the query in the click event of a button on the form, including a WHERE clause to restrict the results to the date range and the IN clause.  The following is untested air code, remember, and may well need some debugging.

First you'd declare the variables:

Dim strSQL As String
Dim strValList As String
Dim dtmDate As Date

You'd then start with the part of the SQL statement up to the WHERE clause and assign this to a variable:  

strSQL = "TRANSFORM First(qryRosterView.fldFirstName) AS FirstOffldFirstName " & _
    "SELECT qryRosterView.fldCarNo, qryRosterView.fldShiftType, qryRosterView.fldStartDate " & _
     "FROM qryRosterView"


Then you'd add the WHERE clause:

strSQL = strSQL & " WHERE fldStartDate BETWEEN #" & _
    Format(Me,txtStartDate,"yyyy-mm-dd") & "# AND #" & _
    Format(Me,txtEndDate,"yyyy-mm-dd") & "# "

Then the GROUP BY clause:

strSQL = strSQL & "GROUP BY qryRosterView.fldCarNo, " & _
    "qryRosterView.fldShiftType, qryRosterView.fldStartDate "

Then the PIVOT clause:

strSQL = stSQL & "PIVOT Format([fldStartDate],""Short Date"") "

And finally you'd build the value list for the IN clause:

For dtmDate = CDate(Me,txtStartDate) to CDate(Me,txtEndDate)
   strValList = strValList & ",""" & Format(dtmDate,"Short Date") & """"
Next dtmDate

strSQL = stSQL & "IN(" & Mid(strValList,2) & ");"

And assign it to the SQL property of the query:

CurrentDb.QueryDefs("YourCrosstabQuery").SQL = strSQL

You can then open the query:

DoCmd.OpenQuery "YourCrosstabQuery"

The problem with binding a form to the query is that the column names will change depending on whatever date range you specify, so you cannot just bind controls in a form to the columns as you would with a straightforward query which returns columns of fixed names.  You'd have to follow up the above with some more code which opens the form in design view transparently, amends the ControlSource properties of the relevant controls to the dates returned by the query and saves and closes the form.  If you are going to attempt this name the controls with names like Date1, Date2 etc, so with 42 dates spanning the 6 weeks the final control would be Date42.  You can then loop through the controls in the form's Controls collection For n = 1 to 42 indentifying each with Me,Controls("Date" & n) and assign a formatted date from txStartDate to txtEndDate as the ControlSource of the control in the same way as the value list was built in the above code.

Another possible way of presenting data like this, and one I prefer to use wherever possible, is not to use a crosstab query at all, but to use a report in which the column headings are supplied by one multi-column subreport and the values by another.  You'll find a simple example of this as Payments..zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

This little demo file shows customer payments for a week in this way, presenting the data in a crosstab type format by means of two subreports, one which supplies the dates as the column heads, the other the payments by each customer.  The parent report returns the customer names and total payments for the week, and the second subreport is linked to the parent report on Customer to restrict each instance of the subreport to the current customer's payments.  the first report, showing the dates, is unlinked, and restricted solely by parameters.
