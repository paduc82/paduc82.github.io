karl timmermans (outlook mvp 2012-2018) say:
The addition of the (1) has nothing to do with your email account. it has to do with the data file name associated with the account. For accuracy, you didn't "change" your email account from POP3 to Imap - what actually happened was that you configured the same email account using a different profile.

Also for accuracy, the Outlook Navigation Pane doesn't list <email accounts> but rather the data files configured within the profile. In the case of POP3, a PST file is associated with the account which in your case would also have been the "default data file" for the profile (assuming that you only have one email account configured in Outlook at the moment).

When a POP3 account is removed from the Outlook profile, it does NOT remove the data file from the profile (which in your case is also a good thing). The default data file is also where the default contact/calendar folders are stored.

When you configured the email account using IMAP, an .OST file got created but in this scenario, the folder tree for the .ost file (Imap account) would not have any contact/calendar folders since Outlook would use those in the "default data file" for the profile (or in your case, the original PST file). Contact/calendar folders in an OST file for an IMAP Account are local to the machine (do not sync with the server) and if the IMAP account gets deleted, so does the OST file and you would lose all data in the contact/calendar folders (which would have the additional folder description of "This Computer Only"

To begin with, using custom properties for anything more than just a few bits of data is kind of like playing Russian Roulette with five bullets loaded.  You *might* survive it but the odds are agin' ya.

PPT uses a limited data area to store custom properties and shares it with other things (links for one).  Put too much stuff in there and you start overwriting other stuff that's already there.  

Tags on the other hand are virtually unlimited.

Every presentation, slide, or shape can have pretty much as many tags as you like.

For example, assuming a reference to a shape in oSh

oSh.Tags.Add "Bookmark1", "Here's the text for bookmark 1"

Tags are a collection, first parm is the item's name, second is its value, both are Strings

' To see the tag value:
Debug.print oSh.Tags("Bookmark1")
