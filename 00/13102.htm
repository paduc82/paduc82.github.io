Those fax cover sheets are based on a template that contains two macros (CheckIt and UncheckIt) and two AutoText entries (Checked Box and Unchecked Box). As you can see by pressing Alt+F9 to reveal the underlying field code, each check box is actually a MacroButton field that runs one macro or the other depending on the current state; all the macro does is change the prompt text from the checked box to the unchecked box or vice versa). rob koch say:

You are mis-reading teh certificate information, since code signing certificates work differently than those for SSL at a web site.

 

You need to look at both the date range you mentioned and also the 'Signing time' within the Signer information section.  As long as the time at which the software package was signed is within the range of the Valid dates indicated in the certificate itself, the signature is valid.  If the certificate itself were 'bad', for example due to the mis-use of an organization's name by a third-party, the certificate itself would have been revoked, resulting in a failure of the certificate to be recognized as 'OK'.

 

If you thinnk for a moment you'll realize that some software may not be replaced for years, so if the certificate itself became invalid after a certain date, this would require that the software be updated for no other reason than to refresh the certificate which is obviously rather pointless.

 

On the other hand, an SSL certificate for a web site certifies that the ownership of the domain and web site are known and valid, which does have a time related component and so requires that the certificate be updated on at least some minimum term along with the domain itself.

The issue is related to the way that Calendar events are stored in Exchange and Outlook.  The calendar events are stored as an offset from GMT.  So each device that displays an appointment must be in the same timezone with the same offset in order to see the appointments exactly the same.
