Sub DeleteEmpties()

Dim ws As Worksheet, wb As Workbook

Application.DisplayAlerts = False

For Each wb In Application.Workbooks
    For Each ws In wb.Worksheets
        If WorksheetFunction.CountBlank(ws.Range("A1", ws.Cells.SpecialCells(xlCellTypeLastCell))) = _
        ws.Range("A1", ws.Cells.SpecialCells(xlCellTypeLastCell)).Cells.Count Then
            ws.Delete
        End If
    Next
Next

Application.DisplayAlerts = True

End Sub

Delete empties is a macro, not a function.  Put it in a new module and press "run."

It deletes empty spreadsheets, not worksheets without code.  I tested it to this end and it works as advertised.  So if you click on the 79 tabs in your file when it is open, any that don't have data or formulas in them anywhere would be deleted from the workbook upon executing this macro.

As Rick has said, the sheets in your VBA code editor are just representations of the sheets in your actual workbook.  If you're seeing sheets in your Project Editor that aren't also tabs in your workbook, it is because those worksheets are hidden.  Simply right click on a worksheet and press unhide.  This will unhide all worksheets in the workbook.  If you want to delete only sheets that you see on your Project Editor but don't see in your workbook, simply do

Sub DeleteEmpties()

Dim ws As Worksheet, wb As Workbook

Application.DisplayAlerts = False

For Each wb In Application.Workbooks
    For Each ws In wb.Worksheets
        If ws.Visible = xlHidden Or ws.Visible = xlVeryHidden Then
            ws.Delete
        End If
    Next
Next

Application.DisplayAlerts = True

End Sub

You can add the styles to the Quick Style Gallery on the Home tab. In the Styles pane, right-click on a style and choose "Add to Quick Style Gallery."

john w. vinson say:

No. Frontend-backend relationships are VERY demanding in terms of bandwidth and stability. Connecting over the internet is neither fast enough, nor stable enough; you'll get dreadful performance and run a major risk of database corruption.

 

2010 and 2013 support "web databases" but they are different in user interface (they use web pages with code rather than the familiar desktop-database forms and reports).

tom van stiphout say:

No, Google docs and equivalents like dropbox and skydrive are accessed over the HTTP protocol which does not support opening files and keeping them open.

 

One option is to upsize your database to SQL Server, and host the server at some ISP. You can still access the back-end using linked tables and a standard ODBC connection. We have several clients running in this configuration. There are a few gotchas but by and large it works well.

 

The other way is to go all web-based. If you are thinking of using Access for that, A2010 and A2013 both support creating BASIC web apps. You will want to use A2013 in this case because A2010 is technologically a dead end and A2013 is a restart on a completely different technology platform.

 

One other option of course is to use Remote Desktop Connection to run a session on a computer on the LAN. Especially if you have just a few remote users this is a great option because it requires zero changes to your app. If you have more than a few remote users, you need Terminal Server licenses and an IT firm can set this up for you.
