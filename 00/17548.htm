The forum thread Arshad pointed you to makes more or less the same suggestion I was going to:  That PPT creates a temporary file in the same folder as the file it's opening and if the user's permission levels don't permit creating and writing to a new file in that same folder, the file won't open.

scottgem say:

There are several ways to maintain an audit trail of data. My blog has an article of one way to do it using Data macros.

I also use a VBA procedure to do it:

Public Sub AuditTrail(strTable As String, strField As String, lngRecord As Long, varOld As Variant, varNew As Variant)

Dim strSQL As String, strUser As String
Dim intCompany As Integer

strUser = DLookup("[EmpID]", "tblParameters")
    
strSQL = "INSERT INTO tblAuditTrail (Tablename, Fieldname, RecordID, OldValue, NewValue, ChangeDate, ChangeBy) " & _
            "VALUES('" & strTable & "', '" & strField & "', " & lngRecord & ", '" & varOld & "', '" & varNew & "', #" & _
            Now() & "#, '" & strUser & "');"
CurrentDb.Execute strSQL, dbFailOnError
End Sub

Both methods require a table like so:

tblAuditTrail

AuditTrailID (PK Autonumber)

Tablename

Fieldname

RecordID

OldValue

NewValue

ChangeDate

ChangeBy

To use the VBA proc, you would put code in the After Update event of any control on a form that you want to audit:

Call AuditTrail(tablename, fieldname, RecordID, Oldvalue, NewValue)

ken sheridan say: It looks to me like you are merely trying to record each employee's pay rate history, rather than create an audit trail.  In a relational database this is a many-to-many relationship type between employees and pay rates.  If pay rates are a fixed set of rates, e.g. a spinal column salary structure, then these would be modelled by a table.  The tables would thus, in broad outline, be as follows:

Employees
â€¦.EmployeeID  (PK)
â€¦.FirstName
â€¦.LastName
â€¦.DateApppointed

PayRates
â€¦.PayRate  (PK)

And to mdel the relationship type betewwn the above two:

EmployeePayRates
....EmployeeID  (FK)
â€¦.PayRate  (FK)
â€¦.DateEffective

The EmployeePayRates table is 'all key', i.e. its primary key is a composite one of all three columns.  Note that I have not given PayRates a surrogate primary key such as an autonumber PayRateID. To do so would be unnecessary, and indeed counter-productive as it would require the PayRates table to be included in any query which returns an employee's pay rate, whereas using a 'natural' key requires only the EmployeePayRates table to be included in such a query.

To return an employee's current pay rate the Employees table would be joined to the EmployeePayRates table, with the query being restricted to returning the rate with the latest (MAX) DateEffective value.

If there is no fixed set of pay rates, but the rate is assigned on a purely ad hoc basis per employee, then the PayRates table can be disposed of.  In either case data input would usually be by means of a subform based on EmployeePayRates, with a form based on Employees.
