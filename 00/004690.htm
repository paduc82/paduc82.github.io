Cross Forest Full Access permissions in a hybrid Exchange deployment - Microsoft Community

alex rizeanu msft say:

Cross Forest Full Access permissions in a hybrid Exchange deployment with Office 365 not working correctly

  • Some background:

More often we get calls related to the following scenario: One Office 365 mailbox, who has Full Access permissions on a shared mailbox located in on premise, is not working correctly. The below article will explain the solution found in one of these situations:

By not working correctly, Im referring to the infamous error in Outlook desktop client Cannot open the set of folders...

First thing that we have checked, was, of course the on-premise mailbox permissions, using the Exchange Management Shell:

Get-MailboxPermission *** Email address is removed for privacy *** | fl

The permissions where there, so now it came the question why they are not working properly for these users when for other users, cross forest Full Access permissions, were working correctly.

So, this was a case of one on premise mailbox not being accessible by 2 cloud users who have Full Access permissions.

  • Troubleshooting the issue:

Next thing we have done, was to run the Outlook Connectivity test on https://testconnectivity.microsoft.com, in this, particular way:

- go to testconnectivity.microsoft.com

- select the Office 365 tab

- select the Outlook Connectivity test (not Outlook Autodiscover)

- input the user name of the cloud mailbox

- input the email address of the on-premise mailbox

- input the password of the cloud mailbox

- confirm the password of the cloud mailbox

- perform the test

The result of the test was very interesting, as apparently it was green everywhere, meaning that our cloud user could logon on the on-premise mailbox store with no issues.

However, Outlook was still unable to open the on-premise mailbox, so we carried on.

We have looked at the properties in the on-premises and Office 365 environment on a working user and compared it to the non-working one. The main difference was that the non-working users mailbox had the following property RemoteRecipientType stamped with the following value ProvisionMailbox the correct value should have been Migrated

ProvisionMailbox means that we create the mailbox directly in Office 365 with one of the following methods:

-          Assign a license to a synchronized user in O365 to create a mailbox for him

-          Go to on premise Exchange Management Shell and run Enable-RemoteMailbox *** Email address is removed for privacy *** -RemoteRoutingAddress *** Email address is removed for privacy ***

OR

-          Go to on premise Exchange Management Shell and run New-RemoteMailbox -Name "User User" -Password $Credentials.Password -UserPrincipalName *** Email address is removed for privacy *** -RemoteRoutingAddress *** Email address is removed for privacy ***

-          Go to Office 365 after the user has been synced (New-RemoteMailbox will create an AD user in on premise to be synced to cloud) and assign him an Exchange Online license

Also, the following attributes, on the Get-RemoteMailbox, in the on-premises Exchange organization were blank, we have populated them:

  • ExchangeGuid (had a value of 000.), populated with the value of the same attribute in Office 365.

From https://technet.microsoft.com/en-us/library/ff607302(v=exchg.160).aspx:

The ExchangeGuid parameter specifies the ExchangeGuid property value of the mail user, which should match the ExchangeGuid value of the corresponding cloud mailbox.

  • An X500 in proxy addresses, with the value of the cloud LegacyExchangeDN

When having a hybrid deployment, the on-premise mailboxes that are synced as Mail User objects in Office 365, will have stamped on their cloud account, an X500 address containing the LegacyExchangeDn of the on-premise mailbox.

Similarly, when having a cloud mailbox, the Remote Mailbox object in on premise, must have an X500 address containing the LegacyExchangeDn of the cloud mailbox.

Since the cloud user was never migrated, but directly created in Office 365 and then matched with the on-premise environment, by running the Enable-RemoteMailbox command, the cloud LegacyExchangeDn was not stamped as an X500 address on the on-premise Mail User (Remote Mailbox).

An example of the importance that the X500 has, can be seen in this article: https://support.microsoft.com/en-us/help/2807779/imceaex-non-delivery-report-when-you-send-email-messages-to-an-interna

A similar thread can be found here as well: https://answers.microsoft.com/en-us/msoffice/forum/msoffice_outlook-mso_winother/exchange-hybrid-new-cloud-users-cannot-access-on/370c402e-927b-4c66-bc6d-a26f59608285

  • The Solution:

The solution was easy, after having figured out all the above. We had to go to the on-premise Exchange Management Shell and execute this command:

Set-RemoteMailbox *** Email address is removed for privacy *** -ExchangeGuid value from the same attribute in cloud -EmailAddresses @{add= "X500://o=ExchangeLabs/ou=Exchange Administrative Group (FYDIBOHF23SPDLT)/cn=Recipients/cn=375834c6f449463694bc7fdb15e23aa4-User"}

Pay a great deal of attention when adding the X500, not to catch an empty space in the string, otherwise, other issues can arise.

After we made these changes on the on premise remote mailbox, restarting Outlook and trying to open again the on premise shared mailbox, resulted in a success.

  • Conclusion:

When dealing with hybrid deployments and issues of Full Access permissions cross forest, not working, make sure that you have these attributes in order:

On the on premise Remote Mailbox object:

-          The ExchangeGuid must be populated with the same value that we see on the associated cloud mailbox

-          In Proxy Addresses, on the same Remote Mailbox, we must have an X500 address containing the LegacyExchangeDn of the cloud mailbox

Same way goes if we speak about an on-premise user that needs to access a cloud mailbox. The above 2 attribute conditions must be met.

Behind the scenes explanation:

Well the error Cannot open the set of folders is quite generic, but what happens is that on the set of MAPI connections that Outlook was attempting, when trying to open the on premise shared mailbox it got the following response:

 Result Description The connection attempt has failed. The connection goal will be set to disconnected and the retry logic will attempt to get us connected later.

Result Error ecUnknownUser

But what ecUnknownUser: really means, by searching our MSDN library, we can find the following explanation:

https://msdn.microsoft.com/en-us/library/ee179302(v=exchg.80).aspx

https://msdn.microsoft.com/en-us/library/ee219330(v=exchg.80).aspx

Look up the ESSDN (specified in the Essdn field of the request) in the global directory to get that user's configuration information. If lookup fails specifically because the ESSDN could not be found, the server MUST fail the operation with a ReturnValue of ecUnknownUser (section 3.2.5.1.3). If lookup fails for any other reason, the server MUST fail the operation with a ReturnValue of ecLoginFailure, as specified in section 3.2.5.1.3.

enterprise/site/server distinguished name (ESSDN): An X500 DN that identifies an entry in an abstract naming scheme that is separate from an address book. The naming scheme defines enterprises, which contain sites, and sites contain servers and users. There is no concrete data structure that embodies an ESSDN. Instead, an address book entry can contain an ESSDN as a property of the entry.
