Unfortunately, Access is not able to open or import Works database files.  In order to get this data into Access, you will need to open your Works database file and export it to a file format that Access can read from.  Once you have the data exported, you should then be able to import that data into Access.

 

The article below discusses this and has you export your Works database to a dBASE file.  If a dBASE file doesn’t work, I would recommend exporting it to a .csv file.

 

327147 – HOW TO: Import a Works Database into Microsoft Access 2002

http://support.microsoft.com/default.aspx?scid=kb;EN-US;327147

 

 

 Steps:

1) Open the database by using Microsoft Works.

2) On the File menu, click Save as.

3) In the Save as type box, select Comma Delimited (*.csv), and then click OK.

Note: If *.csv isn’t an option, you could try exporting to a dbase file type.  But I would try save to a .csv file first if available.

4) Import the comma delimited text file (*.csv) that you created in step 3 into Microsoft Access.

 

This article discussed how to import objects into Access:

http://office.microsoft.com/en-us/access-help/import-or-link-data-and-objects-HP005187610.aspx

Hi all –

So there continues to be some confusion on this issue and I want to try and resolve it in one comprehensive response that hopefully covers all sides of this and addresses all your questions. This response will be marked as the new Answer to this thread and the thread will then be locked.  We are happy to address any questions or issues you have but these new questions or comments are best addressed in new issue threads you can create at answers.microsoft.com.  Once again we apologize for any inconvenience this particular issue may have caused you. 

To summarize..the issue is that some of you have been unable to edit workbooks on your phone in Excel for Windows phone after a ‘save’ of the file in Skydrive or Office Web Apps server.  The 'bug' was introduced in Skydrive on Jan 24 and in Sharepoint/ Office Web Apps server as part of the March Public Update.

If you experience the issue with files in Skydrive:
This issue has been fixed for Skydrive.  Any ‘new’ Excel files created in Excel Web App as of today (or later than February 19, 2013 when we shipped the fix for this) can now be successfully edited in Excel for Windows Phone.  For files that were created or last saved before February 19, you will first need to open those files in Excel Web App (or Excel desktop client), make a change to the file (edit a single cell for instance) and then save the file back to Skydrive for the file to be editable again on the phone.  Excel Web App has ‘autosave’ so simply closing the file is sufficient after making the change. 

If you experience the issue with files in Sharepoint 2013/Office Web Apps 2013 (on premise deployments)
If you are experiencing the issue described in this thread and it is related to your own (on premise) deployment of Sharepoint and Office Web Apps Server, you can address it by installing a couple updates from the April Public Update.  The specific updates you would need are:


·         Security Update for Microsoft SharePoint Enterprise Server 2013 (KB2737969) 64-Bit Edition
·         Update for Microsoft Office Web Apps Server 2013 (KB2810007)


Please remember that as in the case of Skydrive, you will need to do a 'Save' on the affected file using Excel Web App or Excel desktop client to allow the file to be editable on Excel for mobile phone. 

If you still cannot edit workbooks on your phone after following above steps:
If you are still having problems editing workbooks on your phone after following these above steps, it is an issue that is (unfortunately) By Design, meaning you are hitting an inherent limitation of the product.  There are some features (‘unsupported features’) that can block an Excel file from being viewed or edited in Excel Web App, and in Excel for Mobile.   These are inherent limitations of these products.  We are working to remove many of these blocking issues..it’s a priority for our teams. But it’s important to understand that these limitations do exist currently.  You won’t unfortunately be able to view or edit 100% of all features in the web app or phone.   

·         For help understanding issues with unsupported features in Excel for Mobile phone:
To know what features may cause your workbook to not be editable in Excel for Windows Phone, please refer to to see the “Can’t Edit Workbook” message at the top of the view when viewing on the phone.  

·         For help understanding issues with unsupported features in Excel Web App:
To understand the story around unsupported features that may block viewing or editing in the Excel Web App, please refer to  Some features only block editing of the file in the browser. 

Once you understand what features are blocking,  you can then consider whether it is an option for you to remove these features from your workbook. 

Thank you for your feedback and patience while we work through this issue. And thanks for using Excel Web App!

-Rob

The problem is well determined but no practical solution in the interface, so I went ahead and wrote this code to fix all my pictures

Public Sub CorrectAllPicsBrightness()
  Dim sld As Slide
  Dim i As Long
  For Each sld In ActivePresentation.Slides
   For i = sld.Shapes.Count To 1 Step -1
     If sld.Shapes(i).Type = msoPicture Then
       sld.Shapes(i).PictureFormat.Brightness = 0.5  'magic number after some trials.
     End If
   Next
  Next
End Sub

jeanette cunningham say:

Create a new subform with a record source different from the failing subform (no code) and test.

If the new subform doesn't error, we know it is a problem with the failing subform.

Take the failing subform and comment out all code. If the error goes away, we know the code has a problem.

If it is still failing, change the record source of the failing subform to something simple and see what happens.

If the above doesn't work, then do the suggestions to recover from corruption.

It is possible for a single form to become corrupted, but it is often a good idea to do the recovery corruption steps when this happens.
