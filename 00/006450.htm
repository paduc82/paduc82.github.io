In Office 365, only Password admins, User management admins and Global admins can receive the password reset request notification emails.

Word Mail Merge Data Issue - Odd Reaction to Ampersand - Microsoft Community

jefleexs say:

Word Mail Merge Data Issue - Odd Reaction to Ampersand

I have a simple name field and have, as mail merge data, "John & Jill Smith".

If I do not quote this field in the data source (CSV), EG:
    John & Jill Smith
then the & character causes a problem. It makes Word ask for the delimiter character.
If I quote this name, EG:
    "John & Jill Smith"
then Word accepts all the data and proceeds as normal.

Why is that? And are there other odd characters like the & I need to handle?

Thanks,

Jef.

a. user say:

Well, I do not think the following answers your question either, but I thought it was worth trying to pin a little bit more down,a nd perhaps others can add to it or argue with it.

1. The only way I could /easily/ get Word to see a problem with unquoted &, but not with quoted &, was to use a contrived data source like this:

columna,colu&mnb,columnc
valuea1,valu&eb1,valuec1

(Word pops the question)

columna,colu&mnb,columnc
valuea1,"valu&eb1",valuec1

(Word doesn't pop the question).

2. AFAIK the list of candidate delimiter characters when Word is opening using its text converter is as follows - this is nothing more than the two lists you see when Word pops up the relevant dialog box.

For record delimiters, it's 

<tab>,<enter> and
.!#$%&()*+-/:;<=>?@[]^_`{}|~

(I hope that lot gets through!)

That's every ASCII/ANSI character from 0x20 to 0x7F except the digits 0-9, letters A-Z and a-z and

0x20 (space)
0x22 (double-quote)
0x27 (single-quote)
0x5c (\)
0x7f (rubout, or whatever it's called)

For "<Enter>", Word 2010 recognises at least 0A (LF), 0D0A (CRLF) and 0A0D (LFCR, a sequence I have never seen in the wild), but doesn't like 0D on its own. I haven't checked stuff like FF/VT

For field delimiters, there is an additional possibility, "<none>"

However, off the top of my head I don't know whether Word does additional stuff with "similar" characters, e.g. treats curly double quotes like straight ones, and so on.

Double-quotes are ruled out because of their role in quoting text. However, single quotes are AFAIK always treated as data. If you put curly double quotes in the file instead of straight ones, then the chances are you will see a character encoding dialog box when you try to select the source, but then the curl quotes will be treated as text delimiter characters. So it is possible that the same thing applies to candidate delimiter characters - e.g. other types of "hyphen" might be treated the same as the minus sign. I don't know.

Backslash is probably ruled out because there's a per-document compatibility option to 'treat \" as "" in Mail merge data sources'

3. I have had another look at the whole business of what happens when you connect using OLE DB, and AFAICS the list of candidate delimiter characters when Word is opening a text file using its OLE DB provider is exactly one: the comma. That's despite the fact that the tab is listed first, as if it's the default. Also, the record delimiter seems to have to be CRLF, LF, or (unlike the text converter) CR. Again, there may be other possibilities.

The following doesn't entirely agree with what I remember from the past but is what happens here, now:
 a. if you don't specify the "Confirm file format conversion on open" option when you first connect to the data source, Word will use the text converter, not the OLE DB option.
 b. if you do specify the "Confirm file format conversion on open" option, and select the OLE DB option (the only one that is presented by default), then Word will connect via OLE DB if it can, and will prompt for the delimiter. I have not checked what it will accept as a delimiter, but it is not constrained by the list I have provided above. You can for example specify "x" as a delimiter if you want. In this case, the OLE DB provider does not appear to get delimiter info. from any schema.ini file.
 c. When saving such a document as .docx, Word does save the delimiter in settings.xml. But when you re-open the document, it will reprompt for the value. Haven't checked for .doc.

OTOH

 d. If you macro record what Word does during (b) and run the macro, although Word reports a Connectstring and Querystring that looks as if it is using an OLE DB connection, it is in fact connecting via the text converter. It seems to do that as long as you specify a text file in the Name: parameter of OpenDataSource. 
 e. In order to get Word to connect programmatically via OLE DB I had to create a .odc file for the relevant folder and use that. In that case,
  -  Word will use "comma" as the delimiter unless you have a suitable schema.ini and have specified something else. 
  - the settings.xml stores 9 (i.e. "tab")
  - the user is not reprompted for the delimiter when re-opening the .docx
