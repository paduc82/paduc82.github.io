For 25 out of every 30 days ... this is flawless.

The remaining 5 display at random (or perhaps, a pattern that has eluded me) the "Microsoft Access has stopped working" error message, requiring a user intervention to press OK to close the application.  The application doesn't grow and it only runs stored procedures returning data from an Oracle database.  I could have used Visual Basic in reality but the client doesn't have it and Office is there and licensed ... you know how it is.

I've exhausted ways to overcome this random failure.  Everything is up to date, all unnecessary add-ins are disabled, there is a pro-active compress & repair routine that runs weekly - (app 1 loads app 2 that has only 1 function - to reload app 1 and shut itself down) and I used the Task Scheduler to Open - Run - Close/Compress at 6am instead of running 24/7.  I still get sporadic, intermittent, periodic "Microsoft Access has stopped working" responses.

My next plan of attack is a 2nd application that monitors the status of the 1st and only runs if it detects there is something wrong.  I could do something simple like update a flag on a local, linked, table if app 1 completes successfully - app 2 would check for the flag around 6:20am and only run if the app 1 flag is missing. 

I was wondering, however, if there is a more elegant way to detect that an application has hung and has displayed a "Not Working" messagebox.  Is there a VBA method where app 2 could detect a hung or broken process, close it down and start it up again?  My plan A should work but it will still require the client to press OK from time to time and compact / compress app 1 to keep it all maintained.  The client is cool and can handle it, but I don't feel the job is complete without a maintenance free set up.
