Private Function GetAutoCorrectEntries()

Dim x As Long
Dim TotalACEntries As Long
Dim oRow As Row, MyRange As Range, oTable As Table, TestRange As Range

    TotalACEntries = Word.Application.Autocorrect.Entries.Count

    Set oTable = Word.Application.ActiveDocument.Tables.Add(Range:=Word.Application.Selection.Range, _
                                                            NumRows:=(TotalACEntries + 1), NumColumns:=3)

    Set MyRange = oTable.Cell(1, 1).Range
    MyRange.Collapse
    MyRange.Text = "Name"
    Set MyRange = MyRange.Next(wdCell)
    MyRange.Collapse
    MyRange.Text = "Value"
    Set MyRange = MyRange.Next(wdCell)
    MyRange.Collapse
    MyRange.Text = "RTF"
    Set MyRange = MyRange.Next(wdCell)
    MyRange.Collapse


    Set ofrm = New frmProgress
    ofrm.Show vbModeless
    ofrm.Caption = "Backing up autocorrect entries"

    For x = 1 To TotalACEntries
        PortionDone = x / TotalACEntries
        ofrm.lblProgress.Width = ofrm.fmeProgress.Width * PortionDone
        MyRange.Text = Word.Application.Autocorrect.Entries(x).Name

        Set MyRange = MyRange.Next(wdCell)
        MyRange.Collapse

        'check for formatting and insert RTF text if necessary
        If Word.Application.Autocorrect.Entries(x).RichText = True Then
            Word.Application.Autocorrect.Entries(x).Apply Range:=MyRange
        Else
            MyRange.Text = Word.Application.Autocorrect.Entries(x).Value
        End If

        If MyRange.Tables(1).NestingLevel > 1 Then
            'if the autocorrect entry was a table, the nesting level will be 2 or higher
            Set MyRange = MyRange.Tables(1).Range
            MyRange.Collapse wdCollapseEnd
        End If

        Set MyRange = MyRange.Next(wdCell)
        MyRange.Collapse

        MyRange.Text = Word.Application.Autocorrect.Entries(x).RichText

        If x < TotalACEntries Then
            Set MyRange = MyRange.Next(wdCell)
            MyRange.Collapse
        End If

        'update status bar
        'Word.Application.StatusBar = "Adding AutoCorrect Entry: " & x & " of " & TotalACEntries
        DoEvents
    Next x
    Unload ofrm
    Set MyRange = Nothing
    Word.Application.ActiveDocument.Range(0, 0).Select

End Function

According to https://support.microsoft.com/en-au/kb/926927 unformatted AutoCorrect entries in 2007 Microsoft Office programs are stored in an AutoCorrect list (.acl) file.

The .acl file for the 2007 Office programs is located in the profile for the user.

Note The default location of the .acl file for the 2007 Office programs is as follows:

C:\Documents and Settings\username\Application Data\Microsoft\Office

Formatted AutoCorrect entries in Word 2007 are stored in the default template file for the user. The default template file in Word 2007 is named Normal.dotm.

The linked kb article contains instructions for moving the entries from one computer to another.

A person's name is - sorry - an absoulutely lousy primary key.

A primary key should meet three conditions: it must be unique (that's part of the definition, it uniquely identifies a record); it should be short (for computer efficiency); it should be stable (not subject to change). Names fail on all three counts! I know three guys named Fred Brown here in the little town of Parma; my friend Christina Harris used to be Christine Simmons; any of these names are much larger than a 4-byte Long Integer.

A person's name should exist in a People table, and noplace else.

ken sheridan say: Firstly let's get the terminology right; we are talking about relating tables here, not linking them, which is something entirely different.  Let's look at a simple hypothetical model, using medical patients, which, while not very realistic, illustrates different relationship types.

Say we start with a table of Patients which records each patient's names and  date of birth, and has an autonumber primary key PatientID.

Now, for clinical reasons which we won't go into to, we want to record different attributes about male and female patients.  This means that we have two sub-types of Patients, Males and Females, so we have two tables Males and Females, each of which will again have a PatientID primary key, but in this case not an autonumber, as it is also a foreign key referencing the primary key of Patients.  The relationship type is therefore one-to-one in each case, from Patients to Males and from Patients to Females.  In addition to the PatientID column the Males table will have columns representing those attributes specific to male patients, and the Females table will have columns representing those attributes specific to female patients.  Any attributes, like names and date of birth, which are common to both males and females are represented by column in the Patients table.

We also want to record each patients addresses, so we have an Addresses table with an autonumber AddressID column as its primary key, and columns AddressLine1, AdressLine2 etc.

However, two or more patients could shared the same address, and each patient could have more than one address.  So we can't simply have an AddressID column in the patients table, or a PatientID column in the Addresses table.  The relationship type here is many-to-many.  However, its not just a 2-way (binary) many-to-many relationship type but a 3-way (ternary) one as we also want to record the type of address in each case, e.g. home or business, so we also need an AddtreesTypes table, with an autonumber AddressTypeID primary key, and a text AddressType column.  This table forms the third leg of the 3-way relationship.

A many-to-many relationship type is always represented by another table, sometimes called a 'junction' table, which sits between the tables in the many-to-many relationship.  It models the many-to-many relationship type by resolving it into two or more one-to-many relationship types (3 in this case), which it does by having foreign key columns each referencing the primary key of one of the 'referenced' tables.  So to model the relationship type we have the following table:

PatientAddresses
....PatientID
....AddressID
....AddressTypeID

In this case none of these should be autonumbers, however.  The primary key of this table is a little unusual in that it is a composite one made up of all three columns.  There is a mnay-to-one relationship from this table to each of Patients, Addresses and AddressTypes.  One thing to note here is that AddressTypeID is an attribute of this table, not of the Addresses table as the same address could be a different address type for different patients, or even for the same patient if their business address and home address are the same for instance, as is the case with me.

Returning now to the Addresses table we'd obviously want to record the city, so we'd include a CityID foreign key column, which would reference a Cities table with columns CityID (autonumber primary key), City and StateID (in the US, Region ID more generically).  There is therefore a one-to-many relationship from Cities to addresses.  Postal codes like Zip codes would usually go in the Addresses table, though I have seen examples where these are in a related table, but differences in how postal codes work internationally sometimes make that difficult.

The StateID column in Cities would reference the autonumber primary key column of a States table in another many-to-one relationship ( a one-to-many relationship from States to Cities looking at it from the other direction).  Note that the Addresses table does not have a StateID column; its unnecessary as we know the sate from the city via the relationships, but more importantly it would be redundant and leave the Addresses table open to the risk of update anomalies as there would be nothing to stop two addresses having the same city, but in different states, incorrectly in at least one case.  By avoiding this redundancy the table is correctly 'normalized'.

Hopefully the above will give you an idea of how relationships work, and you'll be able to apply the underlying principles to your own set of tables.  To get a better understanding if the above model I'd suggest you draw it out on paper as an 'entity relationship diagram', showing each table as a box , with the relationship types between them as directional arrows from the one-side to the many-size of each.

In a real clinical database the model would obviously be extended considerably, adding related tables to cover admissions, diagnoses, procedures, medication and so on, but the basic principles outlined above would still apply.  More advance principles of normalization would come into play, e.g. so that the wrong procedures or medication cannot be given in the case of particular diagnoses, but at this stage you don't need to concern yourself with that.

Ken Sheridan, Stafford, England
