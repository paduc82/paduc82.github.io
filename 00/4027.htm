ronrosenfeld say: If you are going to use merged cells, you need to be quite careful.

Examine your Step 4:  When you wrote "select A1:B1"  how did you do that?  I suspect that you really selected A1:C1 as I do not believe it is possible to select just a single cell of a merged range from the worksheet.  At least not in 2007.

Was the highlighting showing only over A1:B1 and NOT over C1??

Also, during the process of "selecting", before you either take your finger off the mouse button or <shift>, depending on the selection method you are using, you will see in the address block (just above A1), the notation 1R x 3C indicating you have selected 1 row and THREE (3) columns, not two.

So "fred" refers to B1:C1 and, in your set up, will return the array {12,0}.  So your results are expected.

That does not mean you "should not use a defined name for merged cells if one expects a single value to result!".  All it means is that since you cannot select a single cell of a merged range, you cannot use the "create from selection" wizard if you expect a name to refer to a single cell in a merged range.

But you can certainly define "fred" as referring to Sheet1!$B$1  if you want "fred" to refer to a single cell.

Why Outlook on WM10 needs to add tasks - Microsoft Community

ken forbus say:

Why Outlook on WM10 needs to add tasks

My friends from Microsoft tell me that tasks in Outlook do not get much attention from the Outlook team because only 25% of Outlook users make use of them, roughly.  I have no way to confirm that number, but for those of us who use them, they are an invaluable tool that nothing else comes close to.  Why?  Essentially, because tasks are best handled as database items.  The advantages of doing so include

1. You can add arbitrary tags to them (i.e. categories), and sort/filter/view accordingly.  Why this matters:

(a) You can implement GTD-style contexts easily (e.g. @hack, @web, @write, etc.  for contexts, with .projects for your project list, and anything not on those is essentially your someday/maybe list).

(b) You can tag the same task with multiple categories.  Being able to see, for instance, that there are a a lot of writing chores piling up (e.g., @write), or a lot of chores related to a specific grant piling up (e.g. SILC) helps one load-balance.  Personally, I find a Deadlines category, whose view is sorted by date, is extremely useful. 

2. You can easily reorganize them as priorities change, or as projects come and go. 

For example,  If your research trajectory is longer range than any specific grant, then there are things that you want to get done that you may not have funding for currently.  It's still important to track them, and sometimes free labor can be found (e.g. undergraduate independent study projects) that can let some of the non-urgent ones get done sooner than they would otherwise. 

3. The notes field of a task is a great place for contextual information

Examples include the email that spawned the task, URLs, pointers to OneNote pages, etc. 

The two common alternatives I've heard to using tasks are OneNote and 3rd party apps.  Neither are satisfactory, IMO.  I love OneNote and use it many times daily on all my machines.  But it fails all three criteria above.  A list with check-box tags cannot have cross-filed items, copy/paste is incredibly less efficient and more error-prone than flipping category tags, and there's no notes field.  I heard good things about 2Day, a 3rd party app, so I tried it.  Not bad, but it fails the first two criteria -- its model is folders, and it seems you can have a task in only one folder at a time. It tries on the third, but the editor is so lame that it can't preserve formatting and will remove it if you try to edit, so it fails on the third.  (Transparent round-trip of data is the absolute floor for any app, IMO.)  It has several UI issues (e.g. clumsy rendering of long task text, poor layout) and performance problems, i.e. sync is seriously slow (I have just under 2,500 items in my task database -- big someday/maybe list, to GTD'ers).

To summarize, this is something where Microsoft ought to step up and make tasks work for real on phones.  As part of the Outlook app would be terrific, since drag/drop of email is an extremely common way to create tasks for many.  If it were a separate app (why buy Wunderlist otherwise?), I'd be okay with that, but tight Outlook integration would be important.  Maybe add a Share option of Share to task?  Requires way more knowledge of the internals of W10 and WM10 than I want to have to weigh in on how, please view this as a suggestion about what, with common use-cases to help guide how.  Thanks for listening.

      Ken
