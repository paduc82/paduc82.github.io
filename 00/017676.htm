Right-click the merge field in the merge document.

Select 'Toggle Field Codes' from the context menu.

You didn't include any info about your hardware, so it doesn't seem as though you're looking for any help, but if I may, I'll offer a couple of suggestions that you might find helpful/beneficial:

The "lot of data" that you mention losing, should be in a separate partition from your OS partition, and redundantly  backed up; then, whatever build you have installed or in a state of failure at any given time won't matter as far as the data is concerned.

Further, a tester (not just a flounderer) maintains the capability to move back and forth between relevant builds easily and quickly by using a 3rd party program to make disk images (that won't require any OS to restore) of the OS partition, and saving them on a separate hard drive.

Example:

My accounting program produces data which is maintained in a separate data partition and redundantly backed up, including online. All of that data (banking and stock transactions) includes dates of actions, which must be up to date. Since my accounting program is always installed in whichever build of OS I'm running, I can, at any time, do banking and/or stock transactions, without confusing the dates of action status.

So, although I'm typically running the latest build in the fast ring, about once a month, I restore to the latest public release, get the updates for it, and apply any updates to the installed software programs, then make a new disk image of the OS partition, replacing the old one ... and while I'm there, I can make banking and/or stock transactions, because the relevant data is always current, regardless of which build I'm running.

I can then restore back to the latest build in the fast ring, and all data is still current, ready to be used or added to.

If the next build should happen to fail to install correctly, or to contain a major bug, no sweat ... I boot to my rescue disk and restore to the previous build (without needing to rely on any OS to do it) ... it takes about 10 minutes to do, and my data is still safe and current, ready to be used. 

By using disk image backups, one can devote more time to testing/using a build, and less time trying to recover from a calamity and reinstalling everything.

Now, just to touch on your issue of not being able to upgrade successfully beyond 16199, some are running into outdated hardware issues at that point, and will need to update their hardware to move ahead, or, to perhaps go back to the last public release to avoid being stranded by the expiration of their OS, and some are "crying foul".
