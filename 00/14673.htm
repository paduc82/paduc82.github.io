Hold CTRL key and click on Excel icon

quin810 say: My problem is that I did not realize that the cookies did have expires dates.  The expires date that I see in Windows explorer when viewing cookies is not what determines if this cookie is persistent or not.  When cookies were being added, they were actually just appending to the contents of the cookie that was visible through windows explorer.  If those cookies that append to the content have expires dates, then this cookie is persistent.  If no dates are set, then this cookie does not get created on disk.

john rubdy say:

Hi,

Cookies are removed after the browser is closed, however some cookies still remain.

For example:

Document Object Model (DOM) storage

The DOM storage is a kind of "super cookie" web developers can use to retain information. Like regular cookies, they are not kept after the window is closed.

There are many types of cookies a few of them are listed below:

Persistent cookies (or saved cookies) remain on your computer after you close Internet Explorer. Websites use them to store information, such as a sign-in name and password so that you don't have to sign in each time you go to a particular site. Persistent cookies can remain on your computer for days, months, or even years.

First-party cookies come from the website that you're viewing and can be either persistent or temporary. Websites might use these cookies to store information that they'll reuse the next time you go to that site.

Third-party cookies come from other websites' advertisements (such as pop-up or banner ads) on the website that you're viewing. Websites might use these cookies to track your web use for marketing purposes.

Entering 1234567890123 - Is that an eleven plus a two digit or a twelve and a one?

 

ken sheridan say: It can be done quite easily provided that the values in the unbound combo box include a token, e.g. a comma to separate the two numbers.  They cannot be an uninterrupted string of numbers as the length of each is variable.

So, the RowSource property of the combo box would be along these lines:

SELECT [Key1] & "," & [Key2]
FROM [YourTable]
ORDER BY [Key1], [Key2];

and the code in its AfterUpdate event procedure would be:

Dim lngKey1 As Long, lngKey2 As Long
Dim strCriteria As String
Dim ctrl As Control

set ctrl = Me.ActiveControl

lngKey1 = Val(Left(ctrl,Instr(ctrl,",")-1))
lngKey2 = Val(Mid(ctrl,Instr(ctrl,",")+1))

strCriteria = "[Key1] = " & lngKey1 & " And [Key2] = " & lngKey2

With Me.RecordsetClone
    .FindFirst strCriteria
    If Not .NoMatch Then
        Me.Bookmark = .Bookmark
    End If
End With

In the form's Current event procedure put the following to keep the combo box in sync with the current record should the user navigate other than via the combo box:

If Not Me.NewRecord Then
    Me.cboFindRecord = Me[Key1] & "," & Me.[Key2]
Else
    Me.cboFindRecord = Null
End If

and in the form's AfterUpdate event procedure put:

Me.cboFindRecord.Requery
Me.cboFindRecord = Me[Key1] & "," & Me.[Key2]

But I do share the others' reservations about the table definition.  A primary key comprising two numeric columns is common where the tables models a binary many-to-many relationship type between two other tables, e.g. an OrderDetails table with a key made up of OrderID and ProductID.  Even if an autonumber column is added as a surrogate key, these two columns are still a candidate key and must be indexed uniquely, s I don't see any real advantage in adding a surrogate key column to such a table.  And yes, I do know that the sample Northwind database does this does this, but I would never put forward Northwind as an example of good design.  Quite the contrary in fact.

In this example the value of the OrderID might be meaningful as order numbers often follow some structured format.  ProductID could also be meaningful if a structured product number is used, but is probably less likely to be so as the key of the referenced Products table will often be an autonumber, whose values are of course arbitrary, in which case the product would generally be identified by the user by the product name, possibly accompanied by one or more other values such as a size where the product is one of a range of the same named items in different sizes.  So, while what you ask can be done as described above, I'd be dubious about the wisdom of the methodology being employed here.
