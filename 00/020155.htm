scottgem say:

I missed this part in my first read. As Tom said you can create a database design and setup relationships without using a single line of code. But lets explain further about relationships. Creating relationships between tables is a DESIGN issue. Take, for example, your relationship of projects to contractors. A project has one contractor, therefore you include a field in the Projects table that indicates what contractor is associated with the project. This is referred to as a Foreign Key. iIt stores a value that uniquely identifies the contractor record. Boom you have now created a relationship between projects and contractors. You need do nothing more. However, Access has a way of (what I call) formalizing a relationship. This is done in the Relationships Window. Its done by dragging the key field in one table over the corresponding key field in the related table. Formalizing gives the advantage of being able to impose Referential Integrity on the relationship. RI allows you insure that there are no orphan records. That every Project that has a value in the Contractor FK has a corresponding value in the Contractor table. You can also turn on Cascading Deletes. This means if you delete a contractor, every project associated with that contractor is removed. It also allows for Cascading Updates. If you change the key value for a contractor, all FKs will be automatically updated. 

RI is a good thing, but it can also be managed by proper design of your user interface. So its not absolutely necessary to formalize relationships. You can create the same joins in your queries without formalizing the relationships.

albert d. kallal say:

Keep in mind there is now two editions of the venerable and longstanding Access example called the North Winds trader database.

 

The new version is designed as you note what is called a "services database".  In fact more specifically the correct term is "Access web services database".  So it called NorthWind Services.

So the term "services" is the term that suggests this is the web edition.

 

This so called Web services database example has more functionality than any version of the traders example, but it's been designed as a publishable web database. Being a "Services" database explains why you cannot put forms into design mode. 

 

Access 2007 introduced an ability to design forms in "live" preview, or what we call layout mode.  No doubt this new layout mode was in preparation for access 2010 and that of web forms design that was comming down the pipe.

 

It's not really that big of a deal here, but just keep in mind that for web based database design in access, you don't have form design mode anymore and you MUST use the new layout mode.  There are good number of reasons for this, and one of them is simply that in web based systems many restrictions such as controls can not overlap or be on top of each other. The older design mode does not reflect these new issues in regards to web browsers.

 

So you're forced to use these new design services because is a lot more restrictions in web based development.

 

The other interesting thing is I believe this is the first version of the traders database that has the code present to update and maintain inventory levels automatically. This ability is accomplished using the new database triggers feature.

 

You don't have to use the access web services example, and you can still download the previous 2007 version of the north wind traders database which of course is built as a classic non web publishable database (and the forms will have desing mode, but the over all applicaiton does not have the features we see in the 2010 version).

 

So you now have two edditions of this database. In fact in theory I believe there's a third edition which was designed for ADP, but that's not really relevant to this discussion. So there are several editions of this database over the years.

 

If you want the classic designs surfaces, you'll have to use an older edition of the north wind traders database.

 

Keep in mind the advantages of the new version is that you can publish this example to a web site, and your users can run the application with nothing more than a web browser, that includes launching it even on something like their smart phone or an iPad.

markbiallas say:

Emails still going to Junk Folder after being put on Safe Senders list

I've tried all the logical steps, still....emails are being sent to junk folder after I have put them on the safe sender list.  Our business relies heavily on email communication so this is a problem.  I use Outlook 2010.

brian tillman [outlook mvp 2007-2019] say:

There may be two completely separate safe lists at work here: the one in Outlook 2010, and one that may be on the server. If the server itself thinks the messages are junk, it may move them to the Junk E-mail folder before Outlook ever sees the message. You add the sender to Outlook 2010's Safe Sender list, but the server has no knowledge of this list (it doesn't sync to the IMAP server), so the next time a message arrives from the same sender, the server once again puts in in Junk.

Log into the mailbox using a web interface and see if there is a way to disable the server-side junk mail filter or see if you can add the sender to the server-side safe sender list. Then Outlook 2010's junk filter will be the sole arbiter of what is junk or not.
