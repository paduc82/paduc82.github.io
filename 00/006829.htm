marshall barton say:

I don't see how that count can be useful unless maybe if your table is not properly normalized, in which case you';; have bigger problems than this question.  Please reevaluate your table design to see if it follows the rules of Database Normalization.  I'm pretty sure your question will become moot when the tables are normalized.

 

But, to try to answer the question as asked, use an expression like:

 

In a query, the calculated column could look something like:

 

NotEmptyCount:  -(Nz(field1,"")<>"")+Nz(field2,"")<>"")+Nz(field3,"")<>"")+Nz(field4,"")<>""))

 

A form/report text box expression would then look like:

 

= -(Nz(field1,"")<>"")+Nz(field2,"")<>"")+Nz(field3,"")<>"")+Nz(field4,"")<>""))

 

It would be a slightly different expression if the fields were a number type.

This reinforces what Marshall said. Apparently you have multiple fields within the same record for scores. This is a denormalized structure. You have a horizontal structure when you should have a vertical one. Your scores should be stored in a child table like:

tblScores
ScoreID (PK Autonumber)
PersonID (Foreign Key)
ScoreTypeID (FK)
Score

So this table identifies who made the score (PersonID), what the score was for (ScoreTypeID) and what the score was. You only have records where a score was actually made. So your Average can be done very easily. A Group By query that groups by person and score type then averages the scores. 

As Marshall said, this eliminates your problem about counting the blank score fields.

marshall barton say:

For number fields, it would be (I think I corrected my earlier mismatched parenthesis):

 

   -((Nz(field1,0)<>0)+(Nz(field2,0)<>0)+(Nz(field3,0)<>0)+(Nz(field4,0)<>0))

 

To average the fields:

 

   -(Nz(field1,0)+Nz(field2,0)+Nz(field3,0)+Nz(field4,0)) / ((Nz(field1,0)<>0)+(Nz(field2,0)<>0)+(Nz(field3,0)<>0)+(Nz(field4,0)<>0))

 

You should be thinking that expression is unnecessarily complicated and you'd be right.  But using a simple expression such as Avg(field) in a Totals type query requires you to have properly normalized tables instead of tables that try to imitate a spreadsheet.

 

Please do yourself a big favor and google/bing Database Normalization and read at least one of the less theoretical ones diligently.  Once you get a grip on the first three Normal Forms, many of your problems like the one you asked here will disappear.

 

You should also read Access Basics at http://www.accessmvp.com/strive4peace/

 I've found the answer!

 The first time I upgraded from build 10041 to 10049 it took me a LLLLOOOONNNNGGGG 10 hours.  Since then I've read a number of posts suggesting that if you disconnect or otherwise disable your Internet connection, you can significantly reduce this time.  So yesterday I deleted my Windows 10 build 10049, installed 10041 from a DVD I made from the ISO, and tried it.  It didn't make any difference.  So today I decided to try it again, but this time I disconnected the network cable after the download was complete but BEFORE it rebooted.  VOILA!!  From the "Preparing to Install" message until I was up and running with 10049 took all of 62 minutes.  That's a 10:1 performance improvement just by disconnecting the network cable at the right time!  And I was doing all this on an ordinary several years old 320 GB SATA hard drive.  I have 6 GB RAM and an Intel 2.93 GHz E700 dual core CPU.

 So let me just sum it up.

1) Start the upgrade via the usual Settings (fast track) update.

2) Wait for the download to get to 100% and give the message "Preparing to install"

3) At this point you'll have roughly 10 minutes before the computer will reboot.  This is the time to unplug the network cable.
