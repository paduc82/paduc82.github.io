If JScript is detected, the XSS Filter searches evidence of reflection, information that would be returned to the attacking Web site if the attacking request were submitted unchanged. If reflection is detected, the XSS Filter sanitizes the original request so that the additional JScript cannot be executed.

Access navigation pane scroll jump - Microsoft Community

as_accessdev_2016 say:

Access navigation pane scroll jump

I know this is a subject that has been raised on several forums over many years covering versions of Access since 2007 and on many version of Windows.

Unfortunately, it is still a bug that Microsoft has ignored and it is worth raising again.

Overview: if you are working in an Access DB with a lot of objects and you drag the scroll bar down, it will jump back when you release it making it very difficult to scroll to an object and select it.  The farther down the list you go, the more it will jump to the point where you may need to keep dragging the scroll bar four or five times to even be able to select the last item.

Over the years, suggestions include installing a service pack (for Access 2007) that mentioned a similar problem, changing the Navigation Bar "View By" option to list, and finally, changing the Windows scaling factor to 100%.

At this point, I have testing this with Access 2010 and 2016 and cannot fix the problem with any of the standard suggestions.  Most people have had luck with the 100% scaling, but this is not working for me.  I have the same issue at any scaling factor.  I have also testing resolutions from 800x600 to 4K.

Theory: when I drag the scroll bar and it jumps, it also changes size.  I think what Access is doing behind the scenes is paging the object list.  These are all tables, forms, queries, etc in a local database on the C drive, so there is no latency issues at play.  I assume the page count is very small, so it has to page four or five times to get through around 50 objects.  I have looked through the registry in the obvious places, but not found anything that looks helpful.

Any suggestions or feedback would be appreciated.  I know Access devs often feel like second class citizens in the Microsoft world, and this is a good example of why.
