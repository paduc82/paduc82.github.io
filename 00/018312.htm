Hugh wrote:

I will add another column (criteria date) that rounds up the unique date, and use sumifs(uniquedate, "<"&criteriadate, values).

I'm not sure what you intend to do exactly or why you think it would remedy anything.  It might help if you provided a concrete example.

If C122 contains =42575 + 122/10000, =COUNTIF(C122,"=" & C122) returns zero, which demonstrates the problem.

=COUNTIF(C122,"=" & ROUNDUP(C122,4)) does not remedy the problem, if that's what you have in mind.

ROUNDUP(C122,4) returns the binary representation of 42575.0122, and Excel still formats that 42575.0121999999.

There is no getting around the formatting defect, AFAIK, as long as you intend to work with 42575.0122.

(It might be noted that "="& is redundant in my examples.  And if we write =COUNTIF(C122,C122), there is no problem because in that context, apparently Excel uses the actual binary value of C122 instead of converting it to text with just 15 significant digits.  However, you intend to use "<="&.  So I am using "="& to emulate Excel's behavior with "<="&.)

Generally, if you change the divisor, the problem simply moves to other rows.

I guess you reject the idea of adding 100000 in column C, which truly avoids the formatting defect.

I did stumble across another alternative that allows you to format column C as d/m/yyyy h:m:s, if that is your preference.

C122:  =A122 + ROW()/86400

E122:  =COUNTIF(C122, "=" & TEXT(C122, "d/m/yyyy h:m:s"))

However, that depends on the local date and time format.  For example, I would use TEXT(...,"m/d/yyyy...") even if column C is formatted d/m/yyyy.

That might not be an issue for you, if you intend for the Excel file to be used only in your locale.
