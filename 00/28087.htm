john - project say:

philcuddy,

I'm not a big fan of the Switch statement, nor am I a fan of complex formulas but I think the following formula will do what you want:

IIf([% Complete]=0 And [Scheduled Start]<[Current Date] Or [% Complete]<100 And [Scheduled Finish]<[Current Date],"overdue",IIf([% Complete]<100 And [Scheduled Start]<[Current Date],"in progress",IIf([% Complete]=0 And ([Scheduled Start]>=[Current Date] And [Scheduled Start]<=[Current Date]+7),"due soon","")))

However, I question your logic. First, just because a task hasn't started and it was scheduled to start before the current date, I wouldn't necessarily consider that task as overdue. It could be a fixed duration task wherein the resource won't actually start on it until a couple of days in. Second, in Project, formulas only apply to the current task line. Unless you specifically select to use formulas for  calculation of task and group summary rows (this is an option on the Custom Fields window), nothing will be show for summary lines.

Just for reference, have you looked at the Status field? It already does what you want although the criteria is slightly different.

John

john - project say:

philcuddy,

I can't explain why you had to set and unset the formula option for summary rows. It's possible your file has some corruption. You could try saving the file as XML and then re-opening in Project. That normally clears corruption but you will lose any custom formatting, so I'd only do it if you don't have custom formatting.

I'm not a fan of the Switch statement simply because I never use it. I find the "If" construct to be more useful but it's strictly a matter of preference. I also mentioned that I'm not a fan of complex formulas. They are difficult to enter correctly, difficult to troubleshoot when they don't work, and can be easily lost (i.e. not saved in the Project Global like a VBA module). Also, be advised that in Project nesting of "If" statements is limited to 15. I don't know if that also applies to the Switch statement.

I might not have made it clear that formulas in Project are much more limited than in Excel. As I said previously, a formula can only operate on data on the current task line. For example, you cannot take the Start date of task ID 3 and use it in a formula for Task ID 4. Some Project level data is available (e.g. Status Date, Project Start), but that's limited. That's why I use VBA much more than formulas in custom fields.

Did you take a look at the built-in Status field? It takes a little learning to understand exactly what it uses for criteria, but it does essentially provide the information you are attempting to get with your formula.

If this answered your question, please mark it as answered.

    You've learned a lot here already about the difficulties involved in the type of testing you are attempting, but so far no one has discussed the deeper issue of the ability to find zero-day malware that the AV under test is able to detect.

    The first thing needed is to fully understand the testing scenario, since even the few major AV testing groups have significant difficulty finding malware that bypasses the Windows 10 built-in protections, exactly as you've discovered.  From your descriptions so far, I'm assuming you have created a Windows 10 virtual machine with at least the Creators Update from spring of this year and patched it up to the current date.

    If so, you've already installed the most secure version of Windows ever created, which though few understand what's truly "under the covers" inherently contains many security protections that don't require the use of Windows Defender to enable.  For example, here's an excerpt from a Mitigate threats by using Windows 10 security features document that discusses the many built-in security features for that earlier Creators Update and which are enabled or require configuration in a business environment.

    In recognition of this landscape, Windows 10 Creator's Update (Windows 10, version 1703) includes multiple security features that were created to make it difficult (and costly) to find and exploit many software vulnerabilities. These features are designed to:+

    • Eliminate entire classes of vulnerabilities

    • Break exploitation techniques

    • Contain the damage and prevent persistence

    • Limit the window of opportunity to exploit

    The following sections provide more detail about security mitigations in Windows 10, version 1703.

    I won't try to cover these mitigations here, so brief the document for more, since what's already included within Windows 10 is far more extensive than most people understand.  Also note that the Fall Creators Update has additional improvements so deal specifically with the ransomeware and other classes of malware that have still proven difficult to stop.

    Stopping ransomware where it counts: Protecting your data with Controlled folder access

    On top of that, the core applications included in Windows 10 which are most often involved in malware attacks have been greatly improved as well, with the Edge browser security an excellent example of this.  Along with a much simpler interface and operation, which greatly reduces the attack surface, the security features built into Edge are again far more extensive than in the past.

    Microsoft Edge Building a safer browser - Microsoft Edge Dev BlogMicrosoft Edge Dev Blog

    So though the malware threats are still out there and changing more quickly then ever, the protections within the architecture of Windows 10 are far more resilient as well, with the ability for the malware to exploit any actual vulnerabilities much reduced.

    This doesn't mean that no malware can successfully get through, but it's highly unlikely to succeed without help from the user via social engineering.  What I've seen in my own use of  Windows 10 is that I've almost never seen any true malware detected by Windows Defender, since in all but one case of a browser hijack, the malware had either been blocked by the Internet Explorer browser I was using or by some of the later mitigations mentioned in the article above, which I have personally configured more securely than the Microsoft consumer defaults.

    In other words, the built-in protections now so often block the ability of most malware on a well configured system, that only with the direct aid of someone less capable are these typically able to enter a Windows 10 system.  If this is your problem then better detection by antivirus might reduce the potential for a successful attack, but that's only likely if the attack itself isn't a zero-day, since then the issue becomes one of whether your specific AV vendor happens to either already have a signature for that item or at least good enough heuristic or behavioral detections to block the attack.

    In most cases the better technique is to train your personnel, since with the speed at which new malware is created today, the greatest risk is these zero-day socially engineered attacks, so only by stopping the tendencies for this risky behavior is the real problem truly resolved.
