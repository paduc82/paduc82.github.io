Hi WlilHB,

I am an engineer on the OneNote team.  Sorry to hear you’re experiencing this error from OneNote's Recycling Bin.  For clarification, OneNote keeps its own Recycling Bin that is different from the Windows Recycling Bin and is Notebook specific.  So, if you have a OneDrive or O365 Notebook that is shared with others, the Recycling Bin is also shared, such that deleted Pages or Sections can be restored regardless of who deleted them.  Also, OneNote's Recycling Bin automatically cleans up Pages or Sections after 60 days assuming the Notebook is opened by at least one OneNote client.

In order to get out of the situation you described, you currently have two options:

  1. Wait until OneNote client detects the past due Recycling Bin contents, and automatically deletes them permanently
  2. Manually delete the Recycling Bin content (note: unfortunately, this functionality is only available on OneNote Windows Desktop).  To do this, you would:
    1. Open OneNote on Windows
    2. Click on "History"
    3. Click on "Notebook Recycle Bin"
    4. Select the Section or Pages you want to delete, Right Click, and choose "Delete"
    5. Optionally, you could choose to Empty Recycle Bin in Windows on Step 3, by using the Drop Down.

After the changes have automatically sync'ed to OneDrive or O365, you should no longer see the error.

You are probably misinterpreting your observations.  It is probably not that "recalcs do not redisplay", but instead that certain values in dependent cells cause the formula to have unexpected behavior.

Without the formula and example data, we can only guess.

But a common cause of observations like yours is mixing text that look and should be numeric with numeric data.

If that sounds plausible, you can check the dependent cell type using something like =TYPE(A1).  It returns 1 for numeric and 2 for text.  Unfortunately, Excel is inconsistent in its handling of text that looks numeric.  For example, "2"*4 yields the number 8 (not text); but "2"<4 yields FALSE, which might be unexpected.  For another example, LOOKUP(2,{1,2,3},{1,2,3}) yields 2 as expected, but LOOKUP("2",{1,2,3},{1,2,3}) yields a #N/A error.  Yet COUNTIF(A1:A3,"2") has the same intended result as COUNTIF(A1:A3,2).

A common source of mixing "text numbers" with "real numbers" is incorrect expressions like IF(A1,1,"2").  When A1 is TRUE, you get a numeric 1; but when A1 is FALSE, you get a text "2".

Another common cause of observations like yours is using the null string in arithmetic expressions.  It is not uncommon to write expressions like IF(A1="","",2) in B1, say.  Then B1 will __appear__ to be blank, but it does not behave the same as a truly empty cell (no constant and no formula).  Consequently, an expression like B1*B1 will yield 0 if B1 is truly empty, and it will yield 4 if A1 is nonblank.  But B1*B1 yields a #VALUE error if A1 appears blank (truly empty or contains a null string or string of spaces).

If none of this is helpful, you will probably need to post the formulas and data unless someone else gets lucky with his/her wild-a.s.s guesses.
