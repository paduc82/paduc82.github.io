christopher mauldin (bms) say:

Depending upon the complexity of the Access Database there are at least two viable alternatives:

1) If the tables are relatively simple - then you could build your tables as SharePoint Lists and then access these tables with a Desktop Access Database.  With this approach, each user would still have an MS Access DB on their desktop but they would all be updating the same set of tables in the cloud.

2) Create the Database in MS SQL in Azure.  Then use an MS Access desktop that uses the remote tables.

The only technical concern is if the amount of data being joined/digested in a single transaction is large then you need some of the heavy lifting done server side.  This either means being very careful with the Table design (SharePoint Lists) or writing some apps that exist in the Azure space in the cloud.

As long as your atomic transactions don't require to many table reads/row reads (such as reports) you should be OK.  Lists with a million rows are not a problem.

Hope this helps.

hansv mvp say:

Even with 2-3 users, I would recommend splitting the database, for the following reasons:

  • A form is more likely to get corrupted than a table. If you have a separate back end database with tables only, the data won't be affected if a form in the front end becomes corrupted.
  • If all users have their own copy of the front end, the probability of conflicts is smaller; and if the front end of one of the users develops a problem, you can simply replace it with a fresh copy. Since the front end doesn't contain data, nothing is lost when you replace it.
