Link removed. Link in post of January 16, 2016 is still valid. Code below is still valid for this answer.

You should be able to download the file, unzip and then insert your list of names with number of tickets into column B and C under the column headers.

Click the button to create a list. It will clear any existing list and prize winners and it creates a list of all the names repeated for the number of times shown in column C.

Click the Draw Raffle button and it will draw one name at a time and insert it against the prize. You can have as many prizes as you like; just add the prizes on the left. You might even want to edit 1st Prize, 2nd Prize etc with the name or description of the actual prize.

The code is designed to select a random row number from 2 to the bottom of the created list and correlates the row number to the name. (the row numbers are the ticket numbers). It cannot repeat a row number so no one gets multiple prizes with the same ticket but it will repeat the names up to the number of tickets for that name.

You can test the code in the workbook before using it with your real data. If you continue to click the Draw Raffle button then it will stop with a message after all tickets have been drawn.

The associated code is below but you can simply use the workbook that I uploaded.

You will need to enable macros in Options. See Help for how to do this. (The option to "Disable all with notification" should be OK.)

Sub CreateList()

    Dim rngNames As Range
    Dim rngCel As Range
    Dim rngDestin As Range
    Dim i As Long
   
    With Worksheets("Sheet1")
        .Range(.Cells(2, "E"), .Cells(.Rows.Count, "E").End(xlUp)).ClearContents
        .Range(.Cells(2, "I"), .Cells(.Rows.Count, "J")).ClearContents
        Set rngNames = .Range(.Cells(2, "B"), .Cells(.Rows.Count, "B").End(xlUp))
        .Cells(1, "E") = .Cells(1, "B")
        .Cells(1, "E").Font.Bold = True
        For Each rngCel In rngNames
            Set rngDestin = .Cells(.Rows.Count, "E").End(xlUp).Offset(1, 0)
            For i = 1 To rngCel.Offset(0, 1)
                rngDestin = rngCel.Value
                Set rngDestin = rngDestin.Offset(1, 0)
            Next i
        Next rngCel
        .Columns("E:E").AutoFit
    End With
           
End Sub


Sub DrawRaffle()
    Dim lngLastRow As Long
    Dim lngDrawRow As Long
    Dim strName As String
   
    With Worksheets("Sheet1")
        lngLastRow = .Cells(.Rows.Count, "E").End(xlUp).Row
        Do
            If WorksheetFunction.CountA(.Columns("E:E")) = WorksheetFunction.CountA(.Columns("I:I")) Then
                MsgBox "All names have been drawn."
                Exit Sub
            End If
       
            lngDrawRow = WorksheetFunction.RandBetween(2, lngLastRow)
            If WorksheetFunction.CountIf(.Columns("J:J"), lngDrawRow) = 0 Then
                Exit Do
            End If
        Loop
       
        .Cells(.Rows.Count, "I").End(xlUp).Offset(1, 0) = .Cells(lngDrawRow, "E").Value
        .Cells(.Rows.Count, "I").End(xlUp).Offset(0, 1) = lngDrawRow
        .Columns("I:J").AutoFit
       
       
    End With
   
End Sub

a. user say:

Hello - I have been working with Microsoft for a month on this issue.  It became a very high priority for our company.  We have spent a lot of time on it. 

I was able to to duplicate the problem internally and I was able to finally get Microsoft to duplicate the problem in their labs environment within the past week.  I am still waiting on an answer from them.

The majority of people I have worked with on this issue experience the problem in Outlook v14.0.6112.5000.   

My research found that the following Outlook versions did not experience the problem.

1. Outlook 2010 v14.0.06023.1000 (32 bit)
2. Outlook 2010 v14.0.4760.1000 (32 Bit)

We were able to track the problem down to Outlook v14.0.6112.5000 inability to properly decode the PDF's Base64 data in the email.  The example emails we used to duplicate the problem contained spaces right after the Base64 content ended in the MIME encoded file.  Outlook v14.0.6112.5000 grabs these spaces and tries to use them in the decoding process.  Spaces are not valid Base64 characters.  The question I posed to Microsoft is "why does Outlook v14.0.6112.500 include the spaces in the decoding process?"  Other email clients, as you mentioned (hotmail, gmail, older versions of Outlook, etc) don't have a problem with viewing the PDF. 

 

Hopefully they will find an answer and create a hotfix. I will post updates to this forum as I learn more. 

 

Keep reading if you want the gory details on our conclusions.

We determined the root cause of the problem via extensive testing.  This following focuses on Outlook 2010 v14.0.6112.5000 since this email client was used the most by people who reported the problem.   

1. Configure the registry of the machine that cannot view the email attachment to capture all of the MIME encoding in the email header. 

HKEY_CURRENT_USER\Software\Microsoft\Office\11.0\Outlook\Options\Mail
Parameter: SaveAllMIMENotJustHeaders
Type: REG_DWORD
Data Value: 0x00000001

2. Send an email from with a PDF attachment to a recipient who uses the Outlook 2010 email client. 

3. View and copy the “Internet Headers” into a text editor such as NotePad++.  Make sure to copy the entire header.  There will be a lot of Base64 data to copy. You can view the “Internet Headers” by clicking on the email, then “File > Info > Properties”. 

4. Decode the Base64 content of the email.  NotePad++ has Mime Decode “plugin” to do this.  This can be performed by highlighting the entire Base64 content of the email and clicking on “Plugins > MIME Tools > Base64 Decode”.  This will attempt to decode the Base64 content of the email. 

5. The decoding will fail with the following error message for emails that have spaces at the end of the Base64 content. The decoding is successful if the extra spaces do not exist after the end of the attachments Base64 data.  This can be proven by removing all remaining MIME information from the decoded Base64 file and then saving the file as PDF, i.e. file name of TEST.PDF.


This proves the email before Outlook 2010 contains all of the Base64 information it needs to properly decode the PDF attachment. 

 

The following actions were taken to try and prove the theory the problem exists in latest version of Outlook 2010: 

• Action Taken = Disabled all Outlook add-ins on v14.0.6112.5000 (32 Bit)
• Result = No impact on resolving the problem.  PDF still corrupt when downloaded fresh from POP3 account.

• Action Taken = Started Outlook v14.0.6112.5000 (32 Bit) in safe mode from command line using “outlook /safe” without any add-ins on.
• Result = No impact on resolving the problem.  PDF still corrupt when downloaded fresh from POP3 account.

• Action Taken = Created a brand new Outlook v14.0.6112.5000 (32 Bit) profile on Windows XP SP3 and configured the POP3 account. 
• Result = No impact to resolving the problem.  PDF still corrupt when downloaded fresh from POP3 account.

• Action Taken = Created a brand new Outlook on v14.0.4760.1000 (32 Bit) on Windows 2008 R2.
• Result = Did not experience the problem.  PDF was NOT corrupt when downloaded fresh from POP3 account.

• Action Taken = Created a brand new Outlook on v14.0.06023.1000 (32 bit) on Windows XP SP3.
• Result = Did not experience the problem.  PDF was NOT corrupt when downloaded fresh from POP3 account.

Next, I went through each Windows update to see if the decoding change was specific to a certain Windows update.  My tests concluded the update seems to be associated with KB2553310.  

Here are the steps STR Software took to come to this conclusion:

1. Created a new Outlook 2010 v14.0.4760.1000 profile.  Send an email to the client.  The result was the PDF file could be viewed properly. 
2. On the same machine, installed “update for Microsoft Office 2010 (KB2553310).  This KB updates Outlook 2010 to v14.0.6112.5000. 
3. Rebooted the machine.
4. Deleted the old profile to create the profile again without downloading the PDF file.
5. Sent the same email.  The end result was the PDF file could not be opened. 
6. Then, uninstalled KB2553310.  This reverted Outlook 2010 back to v14.0.4760.1000.
7. Rebooted machine.
8. Opened Outlook and tried to view the same email that was previously downloaded on Outlook v14.0.6112.5000 and were unable to open it.   This test supports STR Software’s early theory that the problem occurs during the downloading of the PDF file from the email server to the email client. 
9. Uninstalled Outlook v14.0.6112.5000 and installed Outlook on v14.0.4760.1000.  Sent the same email again to the older email client version and the PDF was viewable.

This test proves that something in the KB2553310, or an update released around the time this article was released, makes Outlook behave differently when downloading and decoding this PDF file.

You won't hear from a Publisher MVP. Though I almost never acknowledged it and preferred to help anonymously, I was a Publisher MVP for nine years until MSFT decided that there was no need and retired the expertise from the MVP program a couple years ago. I was the last Publisher MVP. Here is my opinion.

It is understandable why the message that MSFT is going to discontinue Publisher circulates. I read these reports in the NNTP newsgroups back before these forums were created. MSFT has always treated Publisher as the poor neglected step child and never devoted the resources to it that they did to the core Office programs and it has gotten worse since Pub 2010 was introduced.

Some long time Publisher users will argue that Pub 2007 was the best version of Publisher and that it has been downhill since. Back then MSFT provided free copies of Publisher to printers so users could send native Publisher files to them for printing. MSFT gave up, dropped the program after Pub 2007. MSFT introduced the ribbon menu as of Pub 2010 and also started removing functionality.

Web building was one example.

Another - they have stripped CMYK from Publisher 2013. 

As of Win8 and continuing with Win10 they changed the way text is rendered than with Pub files produced with Win7 or older. Text is rendered at about a 5% difference in size and thus users have found they cannot use older Pub files as templates without reformatting, changing the layout or installing Win7 fonts. MSFT has never acknowledged the issue let alone offered a fix.

A few other examples include MSFT removing the Fast Resize and Zoom option from the Graphics manager in Publisher 2013 which rendered a lower resolution version of the picture but printed a full resolution version with full accuracy. This changed the way pictures are rendered and probably contributed to Publisher freezing and choking on publications with large picture files which is a common problem. Now if a user wants to use Publisher to layout a publication such as a book with lots of high resolution pictures they risk the almost inevitable corrupted file that won't open. Back up your back up files...

Word wrap control and cropping has been changed to the negative.

MSFT decided the users no longer needed clip art and discontinued including it and now send users to Bing to find clipart. Oh, and they also discontinued the extensive on-line Publisher template collection while they were at it. Notice on the templates page the absence of any Publisher templates? There used to be hundreds.

Mail merge has been broken since Pub 2010 with many users finding after multiple pages print Publisher chokes and then only blank pages, never all merged pages. There is recent example of that above.

PDF output became blurred as of Pub 2013 and now there are many examples of Publisher choking even in the attempt to produce a PDF.  With no fixes forthcoming the best alternative has been to abandon MSFT PDF tools and switch to one of the freebies such as primopdf.

The Live Preview feature has been removed from Publisher 2016 without an explanation from MSFT. It still works just fine in Pub 2013...why remove such a valuable feature?

In the past you could run multiple versions of Publisher on the same computer but as of Pub 2016 you cannot run any Office 2013 products; they will automatically be uninstalled. Even more disconcerting is that Win10 patches are breaking the printing capabilities for Publisher. This is especially a problem for Pub 2007 since it is no longer 'supported' and thus users cannot expect a fix from MSFT other than using a workaround or forcing the user to abandon Pub 2007 and 'downgrade' to Publisher 2016. It is impossible to say that this was done on purpose but anecdotal evidence suggests that backward compatibility is no longer a priority for MSFT; they seemingly want users to buy or rent the newest version of Office.

These are just a few examples of Publisher bugs and program deprecation that MSFT has ignored over the years, so yes, it is understandable why the rumors of Publisher's demise prevail given the degradation of the program since Pub 2007 and seemingly the lack of interest in fixing anything by MSFT. And while there has been no official announcement from MSFT, I wouldn't be surprised to see Publisher disappear in the future. At best it certainly looks to be less functional and more buggy if history is to be believed.

OK, I am off my soapbox, but you did ask for a Publisher MVP to respond and I am the closest to that left in this forum. It is a bit scary that as I get older people need to be careful when asking my opinion...they are likely to get it. <g>
