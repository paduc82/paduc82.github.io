graham mayor say:
There are potentially four header and footer ranges for each section of the document (though they cannot all be used in the same document). If you use columns on some pages and not others then the pages with the columns will be separated into separate sections. If you want a header in the first page, then you appear to have set that up correctly, however if you want footers in all the pages, you will have to create them separately.

On the Page Layout tab > Page Setup > Layout > Check Different First Page and apply to whole document.

You then need to insert a page field - type CTRL+F9 and between the brackets write Page thus {Page} -  in each relevant footer that doesn't have one (click ) .

Select the first page page number and from the Insert tab  click the arrow by Page Number and select format page number. Check the box for continue from previous section, and if the start number is anything but empty or 1, change the start number to 1. Check that each section has the page number setting continue from previous section.

john w. vinson say:

Ordinarily one would not split the database until the structure of the tables is pretty well established. Adding new tables and new fields to tables should NOT be a routine, frequent operation.

It can happen, of course; and it's a bit more of a chore in a split database than in a single one. You need to take several steps:

MAKE BACKUPS OF THE FRONT AND BACK END DATABASES!

  1. Get all the users out of the database
  2. Open the backend exclusively
  3. Add any new tables, modify table structures, add fields, define relationships including referential integrity, etc.
  4. Close the backend
  5. Open the MASTER copy of the frontend (you MUST keep this in .accdb or .mdb format, and it should probably be kept offline and not used for anything but changes like this!)
  6. DELETE the links to any tables that have been changed in the backend (you may be able to just relink, but in my experience it's safer and just as quick to delete the table link in the frontend and make a fresh link
  7. Use External Data... Access... Link on the ribbon to create new links to any changed tables, plus any new tables. I'll generally just delete all the links and create new links to everything.
  8. Add any new forms, queries, reports etc. needed to work with the data in the new tables, and reassign the Recordsource properties of any forms or reports which reference changed tables<
