how do I program a cell to display a number associated with a name - Microsoft Community

justinnewton say:

how do I program a cell to display a number associated with a name picked form a combo box?

For example A1 has a combo box with four names.  I want B1 to display "1234" if the first name is displayed, "2345" if the second name is selected, "3456" if the third name is selected and "4567" if the fourth is selected.

ronaldoonenil say: Assuming you have a list as the source for the combo box values, then put your numeric values in the column immediately to the right of these against the relevant values.

e.g. Your source for the names in the combo is Y1 to Y4, then put your numeric values in Z1 to Z4

your formula in B1 can then be

=VLOOKUP(A1,Y1:Z4,2,0)

When you select a name in the combo box, the relevant numeric value will display in B1

Create a named range from your 4 names (select them then go upper left corner (left to formula bar, just below the ribbon) and write a name ex: MyNames  ... then enter

press ok

exit Design mode and try it

rohn007 say:
That is a common problem with dual installations. Windows and office work from the base assumption that there is no reason to have more than one version of office installed. Therefore, each time you start an app that is a different version than currently active in the registry, it must mean you just installed it and obviously want to replace the currently installed one. WRONG.  The fix is a simple registry "hack".

Here are links to the articles I've used to make multiple copies work reasonably well together:


Dual Install - Working with Multiple Versions of Office 

https://office.microsoft.com/en-us/word-help/keep-earlier-versions-of-office-programs-when-installing-office-2010-HA102597134.aspx

This tip describes the specific steps and choices during the 2010 custom install. After you have the multiple versions installed you still have some additional steps to do. They are described in this tip:

http://pubs.logicalexpressions.com/pub0009/LPMArticle.asp?ID=762

With multiple versions installed, there is one issue you can’t avoid, Default Windows File Associations. Only one app can be used to open the files by double clicking in Windows Explorer. You have to pick which one.

 

Personal Supplemental instructions

MS tends to put everything in separate folders and with long wordy names.  I decided to consolidate things somewhat.  So I navigated to the “Start” menu and edited folder and shortcut names to be more concise.  I created a single “Microsoft Office” folder and migrated all of the shortcuts into it.  I created shortcuts to the 3 “2003 to 2007 Interactive Command References”. Then I started renaming the shortcuts.  In my case I had Office 2003 and Office 2007, so I renamed them as follows:

 “2003 Excel”
“2003 PowerPoint”
“2003 Word”
“2007 Excel”
“2007 Excel 2003 to 2007 Cmd Ref”
“2007 PowerPoint”
“2007 PowerPoint 2003 to 2007 Cmd Ref”
“2007 Word”
“2007 Word 2003 to 2007 Cmd Ref”
etc…

Or

Word 2003
Word 2007
Word 2010
Word 2003 to 2010 Mapping
Excel 2003
Excel 2007
Excel 2010
Excel 2003 to 2010 Mapping
etc…

As well, I decided to keep both sets of ``Office Tools`, so I kept one folder and renamed the shortcuts using the same pattern, ie starting with the year.

I also do custom installations, specifying unique installation folders to further separate the installations. It isn't completely successful, some common folders are still used,  but it seems to help.

Microsoft's recommendations are incomplete regarding Word - see http://www.gmayor.com/Toolbars_in_word_2007.htm for further recommendations with regard to shared resources that are better not shared.

Office apps also work better also if their startup folders are not shared.

In the case of Excel I would relocate the contents of the (Excel 2003) XLSTART folder to a new sub folder of the existing XLSTART -The start-up folder location is stored in the registry at
HKEY_CURRENT_USER\Software\Microsoft\Office\11.0\Common\Xlstart

Set the value to match the sub new folder name e.g. Office11XLSTART. That should sort out Excel 2003. Create a similar sub folder and registry entries for Office 2010 i.e.Office14XLSTART and at HKEY_CURRENT_USER\Software\Microsoft\Office\14.0\Common\Xlstart

The same idea applies to Word’s startup and template folders.

ken sheridan say: Duplicate data should never be stored in more than one table; that is one of the basic rules of the database relational model.  Each table's columns must represent attributes of the entity type which the table is modelling, and only of that entity type.  So a Patients table would, in addition to a numeric PatientID or similar as its primary key, have columns such as FirstName, LastName, DateOfBirth etc.  In the language of the relational model these non-key columns are said to be functionally determined by the table's primary key.

I can't be categorical about what other tables you'd need without knowing a lot more about the real world situation being modelled by the database, but you almost certainly should not have separate tables for each locality of pain  being experienced by patients.  The locality of pain is data, and data is represented in one way only, as values at column positions in rows in tables.  Having a table for each locality is encoding data as table names, and is not legitimate within the rules of the relational model.  So you'd model this with a table  PainLocalities say, which would have column PainLocality and one row for each locality in which pain might be experienced.

A patient's experience of pain is a many-to-many relationship type between Patients and PainLocalities, and is modelled by a third table, PatientPain say (I'm sure you can think of more appropriate names for the tables which better describe the entity type being modelled).  This table is the database's representation of the packets being filled out by the patients, but while the physical packet will include their personal details the table does not as these are already stored in Patients.  It resolves the many-to-many relationship type into two one-to-many relationship types by having two foreign key columns, i.e.

PatientPain
....PatientID
....PainLocality

In addition it will have other columns which represent non-key attributes of the relationship type, i.e. the information a patient enters into each packet in relation to their experience of the pain at that locality.  So other columns in this table might be PacketDate (when they fill it in), and columns about the severity, duration etc of the pain.  Some of these columns may well be foreign keys referencing other tables.

For data entry in a case like this you would normally use a form in single form view based on the Patients table, and within that a subform based on the PatientPain table.  The subform would be linked to the parent form on PatientID, and can be in single or continuous forms view as desired.  The PatientID column's value is entered automatically by the linking mechanism, so you'd have controls in the form bound to the other columns only.  Where the column is a foreign key referencing another table its control would be a combo box, e.g. the control for PainLocality  would be combo box with a RowSource property of:

SELECT PainLocality FROM PainLocalities ORDER BY PainLocality;

For analysis of the data on the basis of a specific pain locality a simple query can be created which returns only the data for that locality, e.g.

PARAMETERS [Enter pain locality:] TEXT(255);
SELECT FirstName, LastName, DateOfBirth,
PacketDate, PainLocation, Severity, Duration
FROM Patients INNER JOIN PatientPain
ON Patients.PatientID = PatientPain.PatientID
WHERE PainLocation = [Enter pain locality:]
ORDER BY LastName,FirstName, PacketDate DESC;

This would return rows for all patient who have reported pain at the locality entered at the prompt, with rows ordered by patient, and for each, by the date of the reports with the latest first.  Opening a form based on this query for instance would prompt for the pain locality and open the form restricted to that entered at the prompt.

While a simple parameter prompt like that above is a cheap and cheerful way of restricting the rows returned, a better and more flexible way is by using controls in an unbound dialogue form to enter the parameters.  You'll find examples in MultiSelect.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo opens simple reports, but could equally well open forms or more complex reports which aggregate data over sets of rows.

scottgem say:
First, you should NEVER have to exit out of an app to allow a user to do data entry. That you had to do that indicates your environment is not designed properly. Your app should be split between a back end (the tables) and a front end (everything else). The back end is then stored on a network share while the front end is stored locally on each user's machine. At the least you have a production front end stored on the network. 

Then you have a development front end. THAT is where you make all your changes, test them, make sure they work properly and THEN put the new front end into production. 

Now to answer your question, did you try doing a compact and repair? Having a single file for an app is an invitation to corruption. The only thing I can tell from that info, is the crash probably occurred running some VBA code. Try the Compact and repair, then split the app and set it up properly.
