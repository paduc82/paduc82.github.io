albert d. kallal say:

Well, first there is quite a difference between the issue of how long a form is in use or open as opposed to how long one spends on a single record in that form.

So the above is two VERY separate issues and problems to deal with.

So for above, you could simply have some code in the forms on-load. It would set (grab) the time.

edit: ****** I see that is what you are already doing - my bad - I have re-edited this response as a result.

I would suggest that you in the forms on-load event set a stat time for the form tracking.

Then in the forms close, or un-load event, you now grab the time, take the difference and send it out to your log table.

For how long a user edits a record? Hum, that quite a bit more difficult. This suggests using the forms on-current event. However, a user re-fresh (or VBA code) can often occur and re-trigger the on-current event. So to really determine the time spent on one record, this suggests that the pk "ID" will also have to be tracked. I user might re-fresh the form, and that triggers the on-current. So the user may well NOT have moved off that record - hence tracking of PK is id required.

This suggests that your on-current event will have to track the “id” or primary key. When the PK changes, you set the start time, but would also have to check for a different “id”, and if yes, then you would have to send the time difference out to a log file (well, likely a table of “log events”.

So now that this problem is laid out, we have to cook up a design (the above is only the starting ideas required). Once we hammer out a conceptual design, then we can start the coding process but the design and how this will work has to be laid out, determined first. This does suggest that this information has to be sent out to a separate table.

ken sheridan say:

Rather than trying to log the elapsed times just log the time each event occurs:

1.  The date/time when the form is opened; easily done in the form's Load event procedure.

2.  The time when a user moves to a record.  This can be done in the form's Current event procedure, but for the reasons Albert has outlined, this should only be when the value of the form's recordset's immutable key has changed.  With a table this would in most cases be the value of its autonumber primary key; with a query it might be the composite value of multiple columns in more than one table.  To determine if the value of the key has changed, the value should be compared with that in a module level variable.  If the value of the key differs from that of the variable a row should be inserted into the log table, and the value of the variable set to that of the current key.

3.  The time when the form is closed; easily done in the form's Unload event procedure.


The values written to columns in the log table, by executing an INSERT INTO statement in code, would probably be along these lines:

1.  The name of the form.

2.  The value of the key.  In the case of an empty new record this will be Null, but the Nz function could be called to insert a value such as 'New Record' in this case.  If the user then begins to enter data another row, with the new key value, can be inserted in the form's Dirty event procedure.

3.  The current user's log-in name.

4.  The event, i.e. open form, move to record , close form.

5.  The date/time of the event as returned by the Now() function

The time spent on each record is simply the subtraction of the value from that following it in date/time sequence per user per form.  This can be obtained by a query which joins two instances of the log table, or more easily in a report, where it would also be easy to return the total time the form was open by subtracting the time opened from the time closed.

The user name can be obtained by calling the GetUser function from the following module:

' module basGetUser
Option Compare Database
Option Explicit

Declare Function GetUserName Lib "advapi32.dll" Alias "GetUserNameA" (ByVal _
   lpBuffer As String, nSize As Long) As Long

Public Function GetUser() As String

   Dim strBuffer As String
   Dim lngSize As Long, lngRetVal As Long

   lngSize = 199
   strBuffer = String$(200, 0)

   lngRetVal = GetUserName(strBuffer, lngSize)

   GetUser = Left$(strBuffer, lngSize - 1)

End Function
