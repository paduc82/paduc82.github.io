1. It's always worth checking that you can get the data from a particular source using Excel and MS Query - it can help verify that the DSN is OK, that the Query is valid, and that the users still have access to the relevant data.

2. Can you get the following information, if necessary by opening the Word document on the mail merge creator's machine (i.e. so the document connects to the data source correctly), going into the VBA Editor, enabling the Immediate window (e.g. via the View menu), and entering the following commands:

?activedocument.mailmerge.datasource.name

?activedocument.mailmerge.datasource.connectstring

?activedocument.mailmerge.datasource.querystring

2. If it's an ODBC connection, connectstring should either start like this:

DSN=

or like this:

FILEDSN=

3. If it starts with "DSN=", you need to ensure that a machine DSN ("user" or "system" DSN) with the specified name is visible from each machine that needs to open the data source. A user DSN is per-user on a particular machine, and a system DSN is for the machine. I don't know exactly how that translates into a Terminal Server environment, but I would guess that everyone shares a single set of System DSNs in that case. If so, and if one has been deleted or altered in some way, the connection wouldn't work, but it might from the developer's PC if they have a DSN with the correct name.

4. If it starts with FILEDSN=, you need to ensure that the specified file exists and is accessible to all the users.

5. Either way, you may also need to verify that

  a. the DRIVER specified in the DSN has exactly the same name as the driver actually present on the machine, as shown in the ODBC Administrator

  b. the rest of the DSN is correct (that may not be easy to check, and I would leave it till last)

6. Verify that the querystring is correct and that the users have access to the relevant data.

Not sure I can think of anything beyond that, other than the obvious question about "what changes have been applied to your systems?"

autoformat not adding dashes - Microsoft Community

c shep say:

autoformat not adding dashes

I have the autoformat option to replace a double hyphen (--) with a dash turned on, but Word is not doing it.  I type a double hyphen, and it just stays there as 2 hyphens.  What could cause this?
jay freedman say:
The replacement doesn't happen until you type the space or punctuation following the word to the right of the hyphens. For example, if you type "this should be a dash--after I type" then the hyphens should change to an em dash when you press the spacebar to the right of the word "after".

If you type a space to the left of the dashes, they're replaced by an en dash instead of an em dash, but the timing is still the same.

It may also help to remember that the default shortcut keystroke for an em dash is Ctrl+Alt+Numberpad Minus. The en dash doesn't have a default shortcut, but you can assign one if you need it; the combination Ctrl+Alt+Shift+Numberpad Minus is available.

daniel g. grau say:

I downloaded your sample file.

The problem appears to be that your tables are floating and cover the full width of the page. Thus, there is no room to place on Page 1 the bunch of blank ¶s below the A portion and, at the same time, the B portion of the table. In addition, the long merged cell at the left of table B can't split.

If you delete them (you may have to Cut the last remaining ¶), the tables will join.

To solve the problem, start by selecting the table, then do Table > Properties > Table > None (instead of Around). Although Around tables (floating) can be positioned anywhere on the page, they can be difficult to tame (just like floating pictures). Unless a table needs to be placed in parallel to some text or image, don't use Around, use None and position it by changing its left outside margins.

If you need edge-to-edge tables, using the None setting will require very small margins via Format > Document > Margins.
