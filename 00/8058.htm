The paragraph break is because you have included the whole document and that includes the final paragraph break. If you don't want the final paragraph break, select all of the text you want to include except for the last paragraph and bookmark it. charles kenyon say:

OK, now you are using a LINK rather than an INCLUDETEXT field. My personal preference would be to use an IncludeText field, simply because I am more familiar with it. The LINK field is designed to allow things other than Word documents to be used as a source.

Going back to your original IncludeText link:

instead of "\* MERGEFORMAT"  you would type the name of your bookmark in the source document.

When you have finished there would not be a "\*" switch marker.

davidf2 say:

Liz,

Before anything else I want to compliment you on your site. It is actually a site I had in my list of good examples of Publisher built sites but I didn't realize it was the one you were talking about. You have done a really good job but I also can see why you are asking about how big a site can get and now how to archive materials.

I can see upon looking at http://www.nflaace.org/index_files/Page456.htm why all this is coming to a head for you. You are running out of space in the text box for links to the archived newsletters. Your site is a good example how a Publisher site can eventually grow to have so many pages it becomes cumbersome to manage. Though you could continue to grow your site as it is currently organized, I think your best solution is to break up your site and build it with multiple Pubisher files so you don't have to upload the whole site each time you make a change only to the newsletters and meeetings content. I don't think that trying to convert your old newsletters or meeting pages to pdf files or .mht files is your best solution.

Like Geoffrey I primarily use Publisher 2000 to build my sites, and you can only build a 10 page site if you want to use the navbar wizard. This meant that very quickly I outgrew the navbar and had to change the way I approached my site structure and navigation system. David  Bartosik wrote an article about this many years ago and it helped be think through how I wanted to organize my site so I could grow it without it becoming too cumbersome to manage. I would suggest you read that article now.

Reference: Building a web site with multiple Publisher web publication files:
http://msmvps.com/blogs/dbartosik/pages/81264.aspx

The main thrust of that article was the idea of breaking up a site and building it with multiple Pub files. David illustrated a way to choose a different file name than index.htm and thus a different thicket folder name (ex. index_files) as a way to organize your files and folders on the server.

I prefer a different approach. I would rather let Publisher continue to generate the default index.htm file name and use subfolders to organize my site such as what Geoffrey was suggesting. As an example David Bartosik suggests creating the music section by creating a new Publisher file and when you Publish to the web you save it as music.htm. He then would upload the music.htm file and the music_file folder to the same root directory as the index.htm file and the index_files folder.  I would instead create a subfolder on the host called 'music' in the root directory or in other words at the same level as the index_files folder and let Publisher use the default index.htm file name and upload that index.htm file and index_files folder to the 'music' folder. So while the path to the first page of the music section would be written by David B. as http://www.yourdomain.com/music.htm I would write it as http://www.yourdomain.com/music/index.htm .

I decided to break up my site and produce each section of my site with a different Pub file. My home page is produced with one Pub file. Each main section such as your Meetings, News, Certification, Scholarships, etc would be produced with a seperate Publisher file and uploaded to an aptly named subfolder on my site..ie, Meetings, News, Certification, etc.  So if I wrote a link to the first page of the Meetings section it would look like: http://www.yourdomain.com/meetings/index.htm  My links in my main navbar would reflect this approach. By organizing it this way I needed only upload those parts of my site that changed.

Now in your case you may not need or want to reorganize your whole site, so perhaps follow Geoffrey's approach. Create a subfolder called 'archives'. Create a new Publisher file and call it Archive.pub. Put the links that you have on page: http://www.nflaace.org/index_files/Page456.htm on the first page of that new file. Perhaps provide an introductory paragraph, maybe even a short paragraph describing or outlining what was in the old newsletters under the links to them. Then add one page for each archived newsletter...don't put multiple newsletters on one long page and link to the home page of this new Pub file. Now when it comes time to produce a new newsletter you can just add a page to this new file, copy and paste the old newsletter from your main Pub file, and add the new newsletter as you have always done. Then you would Publish to the web the archive.pub file and upload that resulting index.htm file and index_files folder to the archive folder on your server.  Now when you write a link to your archived or old newsletters instead of http://www.nflaace.org/index_files/Page456.htm you will write http://www.nflaace.org/archives/index.htm .

Now you also can also add the archived meetings to the same archive.pub file. Just add two pages each month...one for the old newsletter page and one for the old meetings page and link from the home page just like you did with the newsletters. Or, you can create two archive subfolders. Call one 'newsletters_archives' and one 'meetings_archive' and create two new pub files for each.

Another possible approach is to create a new pub file for each newsletter and put the html file in a seperate folder. In other words instead of http://www.nflaace.org/index_files/Page540.htm create a pub file called newsletterOctNov2011.pub or newsletter10-11.pub and put the html output into a subfolder called 'newsletter10-11' and link to it from your main site and from your list of archived newsletters http://www.nflaace.org/newsletter10-11/index.htm. The advantage of this would be that you would never have to upload that content more than once.

Another alternative is to create a new pub file called 'news.pub' and produce the first page is http://www.nflaace.org/index_files/Page531.htm ...everything on your current news page...and then add pages to this pub file for all your newsletters and meetings both current and archived. Create a subfolder called 'news' and upload all web content form news.pub to that folder. So now your main menu would link to news via http://www.nflaace.org/news/index.htm By combining all the news, newsletters and meetings including archived ones, on one pub file would mean that you would only have to upload one section some months.

You see where I am going with this? Spend some time thinking about how to seperate out the dynamic portions of your site from the static. Think of how you can organize the content and build it using multiple pub files and subfolders on your site. Most of my sites have a home page built with one pub file and every section is built with a seperate pub file and uploaded to a corresponding subfolder in the root directory on my site. By organizing this way I can also add temporary content to a new folder built by a new pub file and then just link to that folder from the main site and delete it when I am done. The main advantage is I only upload portions of my sites and not the whole thing and it is much easier to manage.

That was a lot of words and examples. I hope it was more helpful than overwhelming.

roady [mvp] say:

When sending an attachment, always think about whether or not the recipient might be able to open it.

 

When you send an email from Outlook as an attachment, you are basically sending a msg-file which can only opened by Outlook. If the recipient doesn't have Outlook, he/she won't be able to open it. However, and this is specific to msg-files, during transmission, the msg-files are often extracted and converted to eml-files by the receiving web servers or at least for the web interface). Recipients will then be able to open it with various locally installed mail clients but sometimes even on the web. The web interfaces of Gmail and Outlook.com do not support opening them on-line so the recipient will have to download it first and open it locally.

 

So make sure you sending your attachments in a format that the recipient supports. To be safe, you can use the html- or pdf-format.
