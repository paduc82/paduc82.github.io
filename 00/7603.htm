henry habermacher say:

so then you have to tell us more about the used datatypes in your query. For me it looks like at least one date fields of the Top20Item table or query isn't a date but a string that represents a date and that this representation in the insert statement can't be converted back to the date datatype.

 

You can try following: remove step by step the date fields from your insert statement (in the Insert field list and in the Select field list) until the error doesn't occur anymore. Then you know which field makes trouble being converted into date.

You then can setup your regional settings in a way that the date representation is compliant with the string you have in the field that can't be converted.

Alternatively you will have to convert this field into a string that can be converted to date automatically with the current regional settings.

 

Typical problem btw is that SQL want's to see the date in the format "MM-DD-YYYY" but in most countries other than USA the standard date representation is "DD-MM-YYYY". If you have set your regional setting. So switching from Englisch (US) to English (UK) or vise versa could solve the problem for now. Better would be to represent the date in the format "YYYY-MM-DD" as this format is ISO compliant and can then be correctly interpreted independent from the regional settings.

 

HTH

xploitmachine say:

Get Rid Of The Flashing Envelope Icon

If you've just put your SIM card in your Nokia 5800, you may have noticed a flashing envelope icon at the top of the screen. This icon indicates that the SIM card's own text message storage is full. This doesn't actually cause problems on the 5800 because it stores its messages in the phone's own memory, but the envelope can look a bit annoying.

To get rid of the flashing envelope:
1. On the standby screen press the white menu button.
2. Select "Messaging"
3. Select "Options"
4. Select "SIM Messages"
5. Delete some or all of the messages from the SIM messages folder using the Options menu.

This should make the flashing envelope go away. You only have to do this once because when the SIM card is emptied it won't fill any more, as the 5800 does not store messages on SIM cards.



But why does the envelope appear in the first place?

The phone uses internal memory intead of the SIM card's memory. However, SIM card memory is very limited and gets full quite quickly. If a phone relies on the SIM card for storage and the SIM card is full, then it can no longer receive messages. The 5800 does NOT use SIM cards for message storage, so it doesn't actually make any practical difference whether the SIM card is full or not. However, the flashing envelope icon will still appear if the SIM card is full just to let you know about it.

The colours of the bars are determined by the bar styles, which can be edited. Format, and there is a button on the ribbon for bar styles. Make sure you drop the button to choose styles rather than formatting just the bar of the selected task.

You can also format the bars individually, but this is a waste of time and produces a mess.

If you switch to the tracking Gantt view, and baseline the project, you will see the grey bars which extend from the early start and early finish as baselined. Stacked on top of the baseline bars are the current as scheduled bars. The baseline bars don't move, but the others will move to either their actual start, actual duration and actual finish if they have actuals, or their scheduled start, duration and finish.

You can't retain the original dates other than having them baselined. The date is either the as scheduled date or the actual date if the task has one. This is the way it works and it makes very good sense.

If you are going to track progress, and you must, then you need to start with a status date. See project, project information. It is also a good idea to display the status date as a solid vertical red line on the chart. Right click on the chart, choose gridlines and status date is the second last one on the list. There should be nothing scheduled or planned on the left of the status date and no progress or actuals on the right of the status date. You are currently showing several tasks which have actual finish dates or progress in the future relative to now. That is just impossible.

All of your tasks are current manually scheduled and new tasks are manually scheduled. You will get a lot more use and value out of MSP if you work only with auto scheduling and makes ure that every task has at least one FS0 predecessor and at least one FS0 successor.

Although you say that you are interested in having particular custom colours for the bars, you are better off just allowing MSP to use the default colours and concentrating on other more important issues.

tom_bpc say:

MaverickMaycare,

Trevor has given some excellent advice.  In addition, I would highlight the fundamental issue at the heart of your question about Task 13.

Start, Finish, and Duration fields are automatically over-written when you enter progress.  When you enter Actuals, Project will NOT remember the previous values of these fields, as it is not relevant for the ongoing schedule.  To track such things, you need to set a Baseline for the project before entering any progress at all.  (Setting a Baseline tells project to remember the current values for Start, Finish, Duration, Work, Cost, etc.)  Then you can apply the Variance Table to show the differences between the Baseline and Current/Actual schedule.

Baselines are mandatory for Earned Value analysis and can be effective in project change-management.  They can also be dangerous, however, as the plentiful variance data can lead some project managers to focus on variance analysis (e.g. "why did this task start 3 days later than it was planned?") rather than managing the critical path to project completion.

Good luck, tom
