There is only one street address line for a given address set. The only reason that you see <....Street 2 / ....Street 3> is to allow you to import street address fields that are separated in the input data file. What Outlook does is combine these fields into one StreetAddress field (Home, Business, Other etc). Similarly, you also cannot export separate street address lines.

Generally when Outlook imports the separate address fields, it will insert a line break after each field when these are combined. However, if you also are mapping a <combined street address> field then that could over-ride what should be happening.

In short, if your Excel file contains 3 separate street address lines and you are not mapping anything to the <Business Address> field directly, then 3 street address lines should appear correctly after import.

karl timmermans [outlook mvp 2012-2018] say:

As stated in the original reply, there is only ONE Street Address field within a given Address Block (Business/Home/Other) in a contact item. The fields shown in the import map (Business Street, ...Street 2, ...Street 3) are there <ONLY> to enable importing a street address from a data file that has separate street address line fields for a "multi-line" street address. These fields are always combined into one field on import.

On export, Those same fields (..Street 2, ...Street 3) will always be blank using the Outlook export wizard. The entire <street address> will always be contained in the first street address field (for those who require it - ContactGenie QuickPort does provide the option of exporting a multi-line street address into multiple output fields (up to 4 street address lines) -http://www.contactgenie.com/quickpot_4_outlook.htm)

If, after importing your multi-line street address information, a multi-line street address appears as one contiguous line - then the issue is with the input data file in that there are no line breaks contained between the different street lines. To put this another way, after importing your information, if you open up a contact item that has a multi-line street address, clicking on the appropriate <Address> button and all the info is in one contiguous line in the street address field - you have an input

Importing any data into the <Business Address> field will result in Outlook <parsing> the information mapped to that field and making a best effort to determine what belongs in what specific field (i.e. City, State, Postal Code, Country etc) with varying degrees of success. In this case, frequently, the individual unerlying fields that make up the full address will not be populated correctly (not something that would be considered a defect of the Outlook import process having seen many examples of what people are trying to import). This issue is compounded when fields are mapped to include the individual address fields (i.e. one field is mapped to the <BusinessAddress> field and separate mappings are set for things like <City> etc). You can't do both - doesn't work.

Very simply put, if your input data file has a multi-line street address contained in <one> field (cell in Excel), the onus is on you to ensure that the individual street address lines contain the appropriate line breaks. There is nothing you can do via the import process that will otherwise magically split (display) a multi-line street address otherwise. It's not spaces that need to be added within this type of field but rather the equivalent of <Alt> <Enter> between the different street address lines in the file that is being imported.

As a general side note, there is absolutely no value in making the fields names in an input data source match Outlook field names. It is a practice that is a complete waste of time and prone to error(s). It only takes a few extra seconds to manually map all the individual fields (which can have any field name you want) to ensure everything is mapped correctly (or at a minimum , verify that whatever has been "auto-mapped" is both correct and complete). Errors that occur using the Outlook import wizard - in 99.9999% of the cases - occur for 1 of 2 (or both) reasons
     1) the data has an issue
     2) the data has not been mapped correctly
(would have said in 100% of the cases but need to leave open the possibility that there is some scenario in .0001% of the cases where a problem may exist - just haven't encountered one as yet).

per jessen say:

 

You are on the right track. Call this macro from a button on the sheet. It is most userfriendly!.

The macro belowinsert the first two formulas in column A:B. Change the column letters as required, and see if you can complete the macro with the missing formulas.

Sub InsertRowAndFormulas()
Dim DestRow As Long
Dim MyFormula As String

DestRow = ActiveCell.Row
msg = MsgBox("Do you want to insert a new row at row: " & DestRow, vbQuestion + vbYesNo, "New row")
If msg = vbNo Then Exit Sub
Rows(DestRow).Insert

MyFormula = "=F" & DestRow & "-C" & DestRow & "*E" & DestRow
Range("A" & DestRow).Formula = MyFormula
MyFormula = "=H" & DestRow & "-F" & DestRow & "*G" & DestRow
Range("B" & DestRow) = MyFormula

End Sub
