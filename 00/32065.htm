    Bug in renaming a file on exFAT - Microsoft Community

    thanatica say:

    Bug in renaming a file on exFAT

    There's a bug in renaming files on exFAT filesystems.

    When renaming a file only changing letters to capital letters, after submitting the new name it doesn't actually change the filename at all. So this is a silent fail (and that's really bad). Only when adding or removing letters, it successfully submits the new name.

    So this rename won't work:

    flowers.mp4 -> Flowers.mp4

    After hitting enter, it'll instantly change back to its old name.

    This rename won't work either:

    FLOWERS.mp4 -> Flowers.mp4

    Again, after hitting enter, it'll instantly change back to its old name.

    I don't understand why this is, because the F is a different character than f.

    Just as „Åç is different from „Ç≠ as well, and this change does get successfully applied in a rename. Even though they too, are phonetically identical, just like our lowercase & uppercase letters.

    This bug exists only on exFAT formatted disks. NTFS doesn't suffer from this bug. What type of disk it is, makes no difference. This is a pure software problem.

    Win7 hard drive enumeration different order from Bios.

    I installed The C drive into sata-1 on the mainboard and verified that the bios reflects this.   Installed Win7 and and confirmed the drive enumeration as drive 0 in the windows drive manager.   Then added 3 more drives.  Rebooted to find that Windows now thinks the C drive drive number 3 (should be 0).  If I attempt to force it back to drive zero by removing the extra drives, windows fails to boot ("missing boot drive") and requires a repair DVD boot to fix the drive path.  (if anyone else runs into this error --that is the fix).

    Additionally it appears that adding/removing a new USB drive in local mode can cause renumbering of the drives. (XP would give internal drives priority and they would always match the bios.)

    This does not affect actual Windows operation, but the reason it's important is to know which physical drive to replace if the event viewer should start calling out drive errors.  Any ideas?

    System consists of: Win7 on an I7 2.8 ghz processor on a P6T SE Asus Mb with 12g memory.  4 identical WD Velocirator hard drives set to sata IDE emulation (one designated as "active" C:).  Nvidia GTX 470 vid card.

    waseemulla shariff say:
    Hi realGWoodard,
     
    We would recommend you to check the boot sequence priority set in BIOS (Basic Input/Output System) and whether the hard drives has been correctly selected as Master and Slave.
     
    Incorrect settings of the Master and slave drive can cause the boot issues.
     
    Verify if there are any latest BIOS updates on the system manufacturer's website, if any we would recommend you to contact the manufacturer and install them as necessary.
     
    The event viewer will give the details of the drive that has failed. Cross checking the sequence of the drives in the BIOS will give us more information as to which drive is causing the issue.
    See What information appears in event logs (Event Viewer)?
