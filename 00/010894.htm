scottgem say:

The first thing you need to understand is that Access is NOT a spreadsheet. It is NOT an advanced version of Excel. It is a very different program. Excel is a spreadsheet that excels in doing calculations, what Ifs and data analysis. It has limited database functionality. Access is a full fledged relational database that excels in capturing, storing, querying and reporting on data. To approach converting data from Excel to Access, you have to drop the spreadsheet mentality. 

This mentality is evidenced by your questions.

1. Access doesn't not have any sort of concept of "inserting" rows of data. This is because Access doesn't store data in numbered rows. There is no inherent order in an Access table. Records can be entered in Access in any order you want. 

But if there is a need to sort data in a certain order, then that has to be planned for in the design of the tables. In Access you set a Sort order for the data you are displaying. Therefore you need a field to sort on. And,  if there is a possibility of needing to insert a record within a sort you need to provide for that. If you explain why you need to "insert" a record between other records, we can advise further.

2. See the above answer. Again a table in datasheet view, though it may resemble a spreadsheet is not a spreadsheet. So you can't copy and paste to expose a blank row. 

3. Tables in Access are live data. Anything you do in a table is immediately stored. So saving the file doesn't preserve an old set of data unless you save a copy of your accdb file with a different name or in a different folder. Or make a copy of your table with a different name. 

Hopefully, you still have the original Excel file so you can re import.

One last piece of advice. If you are simply going to import your Excel tables into Access don't bother. To effectively use Access, you are going to have to redesign your Excel data to "normalize" it before you do anything else. A relational database design has to follow certain rules called normalization to work properly. So the first thing you need to do is familiarize yourself with this set of rules, then redesign your data tables and re import your data into the new tables.

ken sheridan say:
Can I insert a row in the middle of the data? In Excel, I could just click "insert row"

Scott has explained how spreadsheets and databases differ conceptually, and Gina has given you some useful links, but taking the above point as an example, it may help give you an insight into how things are done in a database application if I show you how you can achieve what you are aiming at by taking a real example from one of my own databases.  This is a database of my own personal financial accounts in fact, and includes a form for entering data into a particular account represented by rows in a Transactions table.  The form is based on a query which returns the rows from the account, and the query orders the rows by TransactionDate.  Let's say I want to insert a new row for a transaction on 12 August 2014, to appear in the form between existing rows for transactions on 11 August 2014 and 13 August 2014.

As Scott has explained the rows in the table are in no order at all.  A table is a set, and by definition a set has no intrinsic order.  They are shown in date order in the form by virtue of the query on which the form is based.  To enter the new row this is done in the empty row at the bottom of the form, but then the row magically appears  in its correct position between the earlier and later dates.  So how does this happen?  the answer is that I've programmed it to happen by including some code in the form's AfterUpdate event procedure.  The code is as below, and I've added rather more detailed comment lines than I would normally to show how it works.

Private Sub Form_AfterUpdate()

    ' firstly a variable of long integer data type is declared
    Dim lngID As Long
    
    ' the value of the new record's primary key TransactionID
    ' column is stored in the variable
    lngID = Me.TransactionID
    ' the form is now requeried.  This reloads its recordset
    ' so the row will now appear in its correct position
    ' in the query's sort order.
    Me.Requery
    
    ' however, when a form is requeried the first record in
    ' the sort order becomes the current record, whereas we
    ' want to stay on the record we've just entered.
    ' To do this we use a clone of the form's recordset
    ' which is referenced by the RecordsetClone property
    With Me.RecordsetClone
        ' using the variable in which we stored the TransactionID
        ' value of the new record, we find that record in the clone
        .FindFirst "TransactionID = " & lngID
        ' we firstly check that the record has been found, which is
        ' not entirely necessary, but is good programming
        If Not .NoMatch Then
            ' the form's Bookmark is then set to the same as the
            ' clone's Bookmark.  This has the effect of making
            ' the form's current record the one we've just found
            ' in the clone, i.e. the new record we've just inserted
           ' QED!
            Me.Bookmark = .Bookmark
        End If
    End With
    
End Sub

I'm giving this example not to show you how to undertake this particular task, but to illustrate the fact that a relational database management system like Access is not an out-of-the-box application, but a development environment.  There is much you can do in Access by using the wizards provided, and there are many templates for common types of databases available on-line, but when you want to customize a database to your own requirements there is no escaping the fact that there is a learning curve to be climbed.  Don't be discouraged by this, however.  If you do the homework and try putting what you learn into effect gaining a good level of proficiency is within the grasp of any reasonably intelligent person.  You'll make mistakes, we all do, it's part of the learning process.

The example I've given you above is the use of VBA code to achieve a task, and this is an important part of database development (though in a web databse you can't use VBA, but only macros), but in many ways getting a good understanding of how a relational with this, and Scott alluded to the importance of this in his final paragraph.  An example of how data from Excel can be imported into Access and 'decomposed' into a set of properly structured and related tables is shown in DecomposerDemo.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

In this little demo, data from a simple Excel worksheet is imported, and by means of a series of 'append' queries is inserted into a set of correctly structured tables in Access.  Each query is explained as you step through the demo.  I realize the details of this will be difficult to grasp at this early stage in the learning process.  Don't let this discourage you.  I well remember when SQL (the language of queries) was a complete mystery to me.  Just try and get a flavour of what is being done.  Learning how to do tasks like this will take some time and effort; there is no escaping that fact.

Greetings Massifheed!  Welcome to the Windows Phone forums!

 

I would suggest a few quick checks to try-out.

 

First, I would go to settings > cellular > Data connection and turn-off your Data Connection. Wait a minute or two, then turn the Data Connection back on and try using Speech again.  If this fixes it, then your phone's software was simply acting glitchy.

If that doesn't work, then I would try turning-off the phone (press and hold the power button until the swipe down message appears, then swipe down to turn off phone), waiting a few minutes then power back up and try using Speech again.  If this solves the problem, then the phone's firmware/hardware was acting glitchy and simply needed to restart.  This could also signal a possible problem with your local cell tower.

Lastly, are you always testing this failure from a single geographic spot?   If so, the problem may be the cell tower serving your location.  If your local tower is mis-configured, it might make phones operating off of it seem disfunctional like yours.  Just for grins, I would consider trying to use the Speech function off of a different cell tower.  Since most cell towers are spaced about 1-2 miles apart, we can deduce that the typical range is about .5 to 1 mile, so going a few miles away ought to place you firmly in the range of a different cell tower.   If the problem goes away using this method, it's definitely a cell tower issue (something for Orange UK to fix).

 

I hope one of these solutions helps!
