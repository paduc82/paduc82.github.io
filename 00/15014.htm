toddallcock say:

Microsoft never created any Windows Mobile sync software for Mac OS, but a few third-party developers stepped up and filled the void.

 

The two I remembers off the top of my head are "Missing Sync" and "Sync Mate".

 

Having said that, if the only use of these devices is GPS data collection, you might not need any sync software at all. Unlike today's crippled mobile devices, Windows Mobile was a very PC-like OS, with a fully exposed file system. If you know the names of these eight files and where to find them, you can use the device's built-in File Explorer to beam them to your Mac via Bluetooth, upload them via the browser to a Dropbox, or even browse a network share on your work LAN via WiFi and copy the files across the network. You could also copy the files to device's SD card, and then pop out the card and plug it into the computer to copy them off, or add an email account to the device's email client and email the files to yourself from the device. If the eventual use of this data is to email it to the "GIS Pros" you could actually do it right from the device with no intermediate sync. (Obviously the device will need internet connectivity via WiFi for some of these suggestions.) 

 

The device is almost as flexible as a computer with regard to moving files around.  Having said that, it's running a *really* old version of the OS, first released in early 2005 so as powerful as it is, it's quite unpolished compared to today's devices!

 

FWIW, to clarify some of your comments, Windows Mobile Device Center is the sync software for Windows Vista, 7 and 8 PCs, Microsoft (not Google) ActiveSync is the sync software for Windows XP, and I believe Windows Download Management Center was a tool for corporations to deploy bunches of identically configured devices to workers (all with the same apps, etc.) but I've never personally used that one. The small companies I worked with just configured devices from our Exchange servers.

Run-time error 438:

Private Sub TB_Change()
    TB.Parent.AddTextBox TB

If you add a textbox to a frame, the parent of that textbox is the frame and a frame has no "AddTextBox" method. That's the reason for that error.

In your case the parent of the frame has this method, it's the Sub AddTextBox in your userform.

andreas killer say:

Yes, we can do this, but to add the textboxes is just the first step, to store all the values from the textboxes into the correct locations is much more complicated.

Create a new class module, name it "MyTextbox" and add this code

--- schnipp ---

Option Explicit

Public WithEvents TB As MSForms.TextBox

Private Sub TB_Change()
  'Try to add a textbox on every change
  TB.Parent.AddTextBox TB
End Sub
--- schnapp ---

Then add a userform, add the code below and run it.

Andreas.

 

Option Explicit

Dim MyTBs As New Collection
Dim LastTBPos As Single

Sub AddTextBox(Optional ByVal TB As MSForms.TextBox)
  Const Space = 3
  Dim C As Control, MyTB As MyTextbox
 
  'Add a textbox only if the last textbox calls this sub
  If Not TB Is Nothing Then If TB.Top + TB.Height < LastTBPos Then Exit Sub
 
  'Add the control
  Set C = Me.Controls.Add("Forms.TextBox.1")
  'Move it below the last control
  C.Top = LastTBPos + Space
  'Create a new instance of our class
  Set MyTB = New MyTextbox
  'Assign this control
  Set MyTB.TB = C
  'Keep it alive in our collection
  MyTBs.Add MyTB
  'Calculate the position of the last textbox
  LastTBPos = C.Top + C.Height
End Sub

Private Sub UserForm_Initialize()
  AddTextBox
End Sub
