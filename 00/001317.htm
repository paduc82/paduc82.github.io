Problem with tasks updating in Project 2016, was not a problem in 2010 - Microsoft Community

shift computer solutions say:

Problem with tasks updating in Project 2016, was not a problem in 2010

I have a situation where a client has a big project file setup, with heaps of tasks, that are all dependant on earlier tasks being completed.

If he sets an "actual start date" on a task that is LATER than the planned start date, then all the dates for the subsequent tasks are all updated to reflect the fact that they will have to start later as a consequence.

However, if he sets an "actual start date" EARLIER than planned, nothing is updated, not even the finish date for the task he set the date on.

This used to work fine in Project 2010.

Any ideas why this is and how to fix it?

Mathew.

tom_bpc say:

Mathew,

The behavior you describe seems normal for MSP 2010 the way I use it, so "This used to work fine in Project 2010" is puzzling.

By default, if you enter both an actual start AND an actual finish, then Project schedules all follow-on work from the actual dates (ignoring any prior dependencies that may have been violated.  This is a "progress override" option in other tools.)  There is no option to change this behavior up through 2010, but 2016 may have introduced one - I don't know.

If you enter an actual start and NO actual finish, then the task is in-progress, and Project will schedule remaining work according to the "Split in-progress tasks" checkbox under "Scheduling options for this project."  

  • If the box is unchecked, then "progress override" applies, and all prior dependencies are ignored.  The result is that an accelerated but out-of-sequence start of one task causes acceleration of all of its successors, so long as they are not delayed by other dependencies.  I suppose this is what you expected to see.
  • If the box is checked, then Project will take the all prior dependencies into account for the remaining work, splitting the task and delaying its remaining part as needed to satisfy its predecessors.  The result is that the accelerated but out-of-sequence start of one task (without actual completion) will NOT cause acceleration of its successors (so long as the associated dependencies are FS or FF.)  This seems to be what you are observing.

You should look at the current status of the checkbox in your project and see what happens when you change it.

"Progress override" behavior that you expect to see (i.e. un-checking the "split in-progress tasks" box) is frowned on by most serious professional planners/schedulers, and it would violate the conditions of many contracts that bother to address such things.  The professional approach is as Trevor has recommended:  Leave the box checked, then update and re-schedule the project.  Delete predecessor links for tasks with out-of-sequence progress ONLY after confirming that there is no legitimate reason for keeping the link and thereby delaying later tasks.  There usually is.

john korchok say:

In addition to Charles' suggestion, set the position of graphics and text to be floating rather than inline with text, and positioned relative to the page instead of the default 

You can recognize if a graphic is inline with text because you can't click on it and move it around on the page. To fix this, select the graphic and choose Picture Format>Position>More Layout Options>Text Wrapping and select either Behind text or In front of text.

To set the position relative to the page, select the graphic and choose Picture Format>Position>More Layout Options>Position. For both Horizontal and Vertical, choose the Absolute Position option if it isn't already selected. Then change to the right of from Column to Page and below from Paragraph to Page. The graphic will now stay in the same spot whatever other formatting changes you make.

Header text can be placed in a text box with similar settings to the position to keep it in the same spot.
