albert d. kallal say:

The problem is ‚Äúonce‚Äù you establish a connection, then that connection REMAINS cached until you quit Access.

 

There is no way around the above issue.

 

What above means when using a file DSN, then if you re-linked tables using say read only, the ‚Äúread‚Äù connection you had would remain in effect.

 

So ‚Äúonce‚Äù you linked, then that connection is cached by Access. If you (during the same session and do NOT exit Access) then re-link, then both connections will be available to Access, and you can‚Äôt really control which connection Access is going to use.

 

And you have to be ‚Äúreally‚Äù careful. Since if you by any means connect with a correct and legal UID/password, you can now actually re-link and connect with a WRONG UID/password. What access will do is NOT tell you the connection failed, but in fact use the known and correct working connection. You have to in this case actually ‚Äúexit‚Äù Access (clears the connection cache), and then re-launch to see the effect. And when you do this DOUBLE (triple) check the connection strings that Access actually saved. Access will ‚Äúflip‚Äù those connections to known working ones during the re-link process.

 

So in any and EVERY testing you do with a re-link, if you actually going to CHANGE the re-link to the same database? Well, you better exit access. Launch access (hold down shift key ‚Äì don‚Äôt let any connections or forms etc. with a connection occur).

 

At this point, you can safe re-link with your new connection, and since no previous connection occurred, then Access will not fool you and grab the previous working connection that you established.

 

While all of the above ‚Äúcontext‚Äù applies to a FILE dsn, I believe the same applies to any other kind.

 

The FILE DSN is preferred, since in ALL CASES Access automatic converts the FILE dsn to dsn-less for you. At this point, Access 100% ignores the FILE dsn. You can even delete the FILE dsn, and access will continue to work fine. This is ideal in most cases, since now you can re-distribute the front end to any workstation ‚Äì and not setup or need a DSN on each station.

 

So you don‚Äôt need VBA re-link code to create and use DSN-less connections ‚Äì Access does this by default (when using FILE dsn‚Äôs). Of course with a system DSN, those are stored in the registry ‚Äì and that can be messy.

 

To flip a read only connection to read, you don‚Äôt even have to re-link. All you have to do is ‚Äúlogon‚Äù to the database again, and all the links should now be read/write.

 

So you ‚Äúmight‚Äù be able to flip from read only (and leave the linked tables as read only). So to flip to read/write, then just execute a logon.

 

However, you will NOT be able to flip from a read/write to read only ‚Äì you have to exit access for this to occur.

 

So just remember during testing, you will OFTEN be fooled by this ‚Äúcached‚Äù behaviour. If you don‚Äôt take above into account (that Access caches a working connection), then what occurs will be ‚Äústrange‚Äù to you.

 

What this means when playing and linking to the SAME database, you must (I repeat MUST exit) access to be sure you re-linking with the connection you want.

 

Once a connection is made ‚Äì it remains intact for the session until you exit access.

 

Normally the re-link process is to a different database, so you don‚Äôt notice this issue much. However when re-linking to the SAME database, this this caching issue can drive you crazy ‚Äì rooms with padded walls!

 

So using linked table manager, or code ‚Äì all of the above will continue to occur and mess with you.

 

So you could link tables with the read only ID. Access launched should then be read only. If you simply execute a logon with uid/password (and not even re-link), then all tables and links should now be read/write. To flip back to read only, you have to exit Access.

Once you execute a logon, then all linked tables will work based on that uid/password that you ‚Äúintroduced‚Äù into the connection cache.
