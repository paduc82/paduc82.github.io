How did you reach your conclusion that the update had turned your computer off? If it was, in fact, off, you wouldn't need to unplug it for a while to get it to turn on ... you would only need to push the power button.

During an upgrade installation, when the computer is restarted, there is system processing going on that, for a time, doesn't provide entertainment on the screen for the overly anxious operator; but that doesn't mean that the computer is off.

The installation doesn't turn the computer off, it only restarts it. If the computer turns off instead of restarting, it's a fault in the computer that causes it ... you can easily verify whether or not it's capable of restarting.

You did turn the computer off, however, interrupting the installation process, when you unplugged the power. That meant that when you plugged the power back in, and hit the power button, it was a "cold boot" (which is the only time that the fast start feature is active) and that would obviously interfere with the continuation of the install process that you interrupted, so it would roll back to the previous build.

Microsoft set the fast startup to be on by default because it eliminates wasted time in a "cold boot". It isn't active during a restart of the computer, so has no role in the build upgrade process.

Assigning common write/install tasks to drive other than C: - Microsoft Community

jkotest say:

Assigning common write/install tasks to drive other than C:

It's becoming more common for users to set up their systems using a small boot drive (an SSD for instance) and a different, larger disk for general storage and program installation.  At least, that's their intention; but it's very difficult to configure a system to behave this way natively - it seems that Windows by default will always want everything to happen on the same drive that Windows boots up from (usually C:), and if you want anything stored on another drive you have to transfer it manually, or make sure to choose the correct non-C: drive letter during installation each time something is installed.  The alternative is trying to make potentially dangerous registry changes, using software to "spoof" drive letters, and other such nonsense.

I propose a setting, perhaps in the Disk Management (or some other appropriate) app, to allow users to assign a non-OS drive to be the default destination drive for all system write/storage/install tasks.  So that for example if a user installs Windows on drive C:, they can set the default directory for things like Documents, Downloads, and installed programs to be drive D: (or otherwise, as desired) if they so choose.
