Efficient read-only link to table in external Access DB in SQL, no VBA - Microsoft Community

paul domaskis say:

Efficient read-only link to table in external Access DB in SQL, no VBA

I am trying to develop an analysis process almost entirely with SQL statements, with absolutely minimal use of VBA in order to avoid handling countless SQL statements as VBA strings.

I recently got advice on how link to a a table in an external Access database without resorting to VBA (http://answers.microsoft.com/en-us/office/forum/office_2010-access/link-to-external-table-via-sql-statement/a2188369-de00-40d6-aecc-45a65891c3c6, thanks to John V).  Hoping to improve on this with the following two questions.

Question#1a:
The full path of the external database is needed in SELECT <whatever> FROM SomeTable IN "c:\Full\Path\To\Database.accdb", which makes for difficult to manage code.  Is there a way to avoid this?

Question#1b:
Thoughts: I was toyed with the idea of creating a view from the aforementioned select statement, so the full path is only used once.  However, for each statement in my SQL script that uses the view, the view must be reconstructed. Is this an accurate view of the penalty?

Question#1c:
Related to this, if the same view is joined multiple times in the same SQL statement (with different join conditions, of course), does the view have to be reconstructed for each instance?  The context of this is not that the SQL statements make up query objects in the Access GUI.  Rather, each SQL statement is read from a file using a VBA parser and executed using CurrentProject.Connection.Execute.  Outside of this parser, the VBA is minimized so that most of the anlaysis process is in the SQL script.

Question#2:
Now I was wondering if the external table linkage can be done so that the linked table is read-only?  The SELECT statement above, as well the creation of a view, might be a solution, but there are the caveats asked about above.  My web surfing says that pass-through queries to SQL Server are the solution.  But having web searched SQL Server (proper noun), it doesn't represent my situation.  I am only trying to link to an Access database.
tom van stiphout say:

General comment: the IN clause is nice for temporary tables, but I would not build the whole app on it. That's why Access has linked tables, which work just fine in 99.9% of the cases. Linked tables do not require VBA.

1a: If you don't specify the path, Access will choose one; probably the current directory. Experiment a bit with that, but it may not get you what you want because the BE is unlikely to be in the current dir.

1b: Yes, but Access will optimize the statements so the penalty may not be as severe as you think. Also see my general comment.

1c: See 1b.

2: Linked tables cannot be made read-only. Some queries are always read-only (distinct, group by, union), others depend on finer points. For example this is read-write:

select * from sometable in "c:\Full\Path\To\Database.accdb"

while this is read-only:

select distinct * from sometable in "c:\Full\Path\To\Database.accdb"

talking moose say:

Since this is a new computer, I suspect your data was migrated but you may be experiencing permissions issues because of the unable to access the database message.

You should be able to follow these instructions safely without negatively impacting anything:

  1. Quit Outlook and all other applications.
  2. Click on a blank area of your Desktop to ensure youre in the Finder.
  3. Choose Go menu > Home.
  4. Note the name of your home folder either at the top of the window or in the sidebar to the left. Ill use bill for my example below.
  5. Launch the Terminal application found in Applications > Utilities folder. If youre the only user of your computer then youre an admin. You can run this command in the Terminal to correct ownership of your home folder files:

    sudo chown -R bill /Users/bill
  6. Be sure to replace bill with the name of your home folder.
  7. Press the return key.
  8. Youll need to enter your password. You wont see the cursor move. Just press the return key again when done. Allow a little time for this command to run. Its finished when you see the cursor return to the Terminal window.
  9. Try launching Outlook now.

Hope this helps!

steve rindsberg say:

Whether embedding an audio file from someplace on the internet is legal or not, that's up to you to determine.

Using Audacity (a free sound recording/editing program) you can record pretty much anything that your computer is playing.

You'd need to download and install Audacity and also the LAME encoder (which makes it possible for Audacity to export MP3 files).

In Audacity, choose your computer's internal sound system as the input device, start recording, then start the audio playing from {wherever}. 

You may find that some services pop up ads in the middle of the music; don't worry about  it, let it keep recording, note the time where the ad's occurred (Audacity's timeline will give you this info).  Afterwards, it's simple to go back and edit the ad out.
