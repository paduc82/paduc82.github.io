Try using the IE9 Developer Tools, Network Capture (F12, CTrl-Backtab, F5) to trace your request.   Often requests for that host name result in redirects to a national host name, so that may be the one that is really causing the problem symptom.

 
Also, even if that is the real host name that you are ultimately going to end up with, it is often an alias name for a caching server, so your problem symptom could have more to do with an inability to get a lookup for it that with the name that you are using.

 

Finally, you can use a cmd window and ping and ipconfig /displaydns  commands to prepare a lookup for IE to use.    E.g.  first do:

    ipconfig /flushdns

    ping  -n 1  -w 1   www.google.com

then

    ipconfig  /displaydns

 

Hmm...  so in this case that looks like there is no alias involved but if we then use telnet to simulate what will happen when IE makes its request with that

 

    telnet  www.google.com  80

GET  /

 

I can see that there will be an immediate redirect.   So, if IE is having trouble it would then be a good idea to help it by doing the same kind of preparation with it.   E.g. another  ping  -n 1  -w 1   with the new target host name (but without the  ipconfig  /flushdns  <w>).
