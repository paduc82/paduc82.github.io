boblarson say:

The normal way to do this is to have the Applications or Resources (depending on which you want to treat as the parent item) be the main form record source and then you make a subform using the crossRef_BusOwner table and you would not need to have the one ID on the form but then you would have a combo to select the other. 

 

So for example, if you made the main form your Applications then you would have the Applications table as the main form's record source and then you would have a subform (linked on Master/Child Links to AppID) with the crossRef_BusOwner table as the record source for the subform and on that subform you would have one control - a combo box which has the ROW SOURCE of the Resources table and with RID as the bound column and the control source bound to RID in the crossRef_BusOwner table.  And then there is NO VBA required whatsoever.

Using Access VBA to "Unlink" all Linked Tables - Microsoft Community

florian wilber say:

Using Access VBA to "Unlink" all Linked Tables

I'm looking for a solution where I can use VBA code to only "Unlink" all linked tables and make the table a local one.

I've searched around and most codes only point to deleting the actual linked table.

But I would like to still retain the table, and if possible, even copy all data from the link before unlinking it.

Usage Scenario: Where a previous Access Database was split into FE and BE, and I'd like to combine it back into one.

Thanks a lot :)

hansv mvp say:

DoCmd.TransferDatabase will indeed import the tables into the current database. It would be much easier if you ran the code from the frontend database.

If that is not feasible, you'll have to open a second instance of Access and open the frontend database in that instance to do what you want. I haven't tested the following code, hopefully it'll work:

Sub UnlinkTables()
    Const FEPath = "C:\Users\My-PC\Desktop\DB\file.mdb"
    Dim dbs As DAO.Database
    Dim tdf As DAO.TableDef
    Dim i As Long
    Dim n As Long
    Dim arr() As String
    Dim app As Application
    ' Create a list of linked tables and delete them
    Set dbs = OpenDatabase(FEPath)
    For i = dbs.TableDefs.Count - 1 To 0 Step -1
        Set tdf = dbs.TableDefs(i)
        ' Is it a linked table?
        If tdf.Connect <> "" Then
            ' Add info to the array
            n = n + 1
            ReDim Preserve arr(1 To 3, 1 To n)
            ' Path of the source database
            arr(1, n) = Mid(tdf.Connect, InStr(tdf.Connect, "=") + 1)
            ' Name of the source table
            arr(2, n) = tdf.SourceTableName
            ' Name of the linked table
            arr(3, n) = tdf.Name
            ' Delete the linked table
            dbs.TableDefs.Delete tdf.Name
        End If
    Next i
    dbs.Close
    ' Import the tables
    Set app = New Application
    app.OpenCurrentDatabase FEPath
    For i = 1 To n
        app.DoCmd.TransferDatabase _
            TransferType:=acImport, _
            DatabaseType:="Microsoft Access", _
            DatabaseName:=arr(1, i), _
            ObjectType:=acTable, _
            Source:=arr(2, i), _
            Destination:=arr(3, i)
    Next i
    app.Quit
End Sub

charles kenyon say:

First, if it is not in a Startup folder (or loaded by a macro in a template that is in such a folder) it will not load automatically. It must be in one of the two StartUp folders.

Try saving as a .dotx file in the Word Startup folder. (I know, your Add-In won't work, try anyway.)

If it loads as an Add-In, then your macro security settings are blocking the macro-enabled version. Check your trust center trusted locations to make sure that the folder is a trusted location.

You should not need to use the Office Startup folder. If you do use it, it, too, must be a trusted location. It will be in your Program Files (or Program Files (x86)) folder on your C: drive

Here is the location of mine for Word 2010:

C:\Program Files (x86)\Microsoft Office\Office14\STARTUP

I stand corrected. Word will load a .docm as an Add-In through the dialog. It will not do so simply by placement in the Startup folder. I had never tried loading one through the dialog box.

By the way, when I want to store my Add-Ins elsewhere and still want them loaded upon startup, I put a shortcut to the Add-In in the startup folder. That works so long as the actual storage location is a trusted location.
