Okay, let's summarize.

1. Access will give you a #Name error if it

    a. Can't find a field with that name either because it is misspelled, is not available, or has been removed.

    b. If you have something else named the same it can't figure out which you want to use.  So, this is the field the same name as the control.

    c. If you have a procedure somewhere which uses the same name, in a calculated control it can't tell what you want - the procedure or the other object.

    d. It is missing the equals sign before the calculation.

   

2. What we know -

   a. You have checked the spelling and the fields are spelled correctly.  Unit Cost and SumOfNew Amt (with the space in between New and Amt) are correct.

   b.  You might have something else conflicting with the name as we haven't checked there (that I know).

   c.  Similar to B, there aren't any procedures in your database (public functions/subs) which have the same name as one of the fields or new name ExtCost, correct?

   d. You showed the equals sign so that is there.

 

If all of that comes up empty then it could be corruption - the report is corrupt and you might need to rebuild the report.  And failing that, I think the only thing left is to actually see and play with a copy of your database so we can try looking around and maybe spotting something we haven't thought about.

hansv mvp say:
Create the following function in a module in the Visual Basic Editor:

Function LookupName(s As String) As Variant
    Dim r As Range
    Set r = Worksheets("Sheet1").Range("A2:K78").Find(What:=s, LookAt:=xlWhole)
    If r Is Nothing Then
        LookupName = CVErr(xlErrNA)
    Else
        LookupName = r.Offset(ColumnOffset:=11).Value
    End If
End Function

Change Sheet1 to the name of the sheet containing the data.

Let's say you have a part number in A100.
The formula to retrieve the corresponding part name is

=LookupName(A100)
