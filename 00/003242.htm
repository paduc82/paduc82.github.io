    There are a number of templates available from Microsoft which deal with various business accounting tasks, but they tend to be fairly simple in structure.  You may find examples in them of routines which will provide you with a basis for developing your own routines. On the whole though, if your want a customized solution you need to build it yourself.  The reason I first learnt to develop databases some 15 years ago was that I found the available solutions did not satisfy the requirements of my own, admittedly quite specialized, line of work.


    Provided you have the base data as values in properly structured and related tables, almost certainly yes.  The set of tables and the relationships between them must accurately model the real world situation, however, and be in accord with the principles of the database relational model.  It is especially important that the tables be normalized to whatever normal form is required to avoid redundancy and the consequent risk of update anomalies which might otherwise arise.  You'll sometimes see it said that that normalizing to Third Normal Form is usually sufficient.  What that really means, however, is that normalization to 3NF will in most cases mean that a table is normalized to the higher normal forms without further decomposition.  It does not mean that normalization beyond 3NF is unnecessary.  If it's needed it's needed and should be undertaken.  You should be comfortable with the concepts of normalization and functional dependency (on which the normal forms are based)  if you are going to develop an application of this nature which is fit for purpose in a real world working environment.

    However, your question raises other issues.  Firstly, do you need to create a separate table at all to store the invoices, or can these be computed at any time from the base data?  Normally the latter would be possible if the base tables include the necessary columns for this to be done.  Moreover it is preferable as to store data which can be derived from other data is generally not a good idea, there being nothing to stop updates to one set of the data or both being made so that they become inconsistent with each other.

    When you join tables in a query you get the product of the join, so if there are 2 rows in Table2 which match each row in Table1, there will be two rows returned for each row in for Table1.  If you group the query by a column or columns in Table1 and COUNT the rows returned per group then the count would be 2, so if each row in Table1 represents some event which you want to count only once you'll get the wrong result.  If there are 3 matches in Table2 the count will be 3 of course.  If you then join another table in the query to the join of the first two, and this has multiple matches with rows in Table2 the count will increase again.  So it's a matter of knowing how to write the queries to return the correct results.  This might involve including subqueries in the query, or it might involve creating several queries and joining them


    If you want the restricted functionality of a billing system then it can be done if you are confident that you have the necessary competence to develop a robust application in the light of my comments above.  On the other hand it will be time consuming and likely to take you more time than it would a seasoned professional developer, so would it be cost effective to do it yourself?  If you want a fully functional business accounting system, then it makes far more sense to purchase a commercial accounting package.  Such systems are complex, and you will not be able to reproduce one with MS Access which approaches the functionality of a purpose designed product.
