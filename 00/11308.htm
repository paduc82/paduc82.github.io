I suspect you're going about this the wrong way.  That code looks like it comes from a form's KeyDown event.  Is that right?  If so, it appears to me that mouse clicks are not presented to that event at all.  I know that the constant vbKeyLButton is defined in the vbKeyCode constants, but as far as I know the KeyDown event will not fire when a mouse button is clicked.  The MouseDown and MouseUp events will fire, and the event procedure signature for those events includes a Button argument that can be checked to see which button was clicked, but those events don't let you override the processing by setting the Button argument to 0, the way you can with the KeyCode argument in the KeyDown event.

What you might do, to achieve the same effect, is to use the form's MouseDown event to check whether multiple rows have been selected and, if they have, deselect them.  Code might look like this:

Private Sub Form_MouseDown(Button As Integer, Shift As Integer, X As Single, Y As Single)

    If Me.SelHeight > 1 Then Me.SelHeight = 1

End Sub

ken sheridan say: If you look at this expression:

Count: IIF([total1 is null,0,1)+IIF([total 4]0,1)

you'll see that you are missing a closing bracket, there is an unwanted space in [total 4], and the second IIF function call is wrong as it does not examine the control for Null, and only has two arguments, not 3.  Also Count is a bad choice of column name as it is the name of a built in SQL operator, so, as a 'reserved word', should not be used as an object name.  Use something like TotalCount instead.  So, taking account of those points:

TotalCount: IIF([total1] IS NULL,0,1)+IIF([total4] IS NULL, 0,1)

However, this would return 1, whereas you say you are expecting 2 as the result.  That is not a count of the column positions (not cells, that's a spreadsheet concept) which contain values but the addition of the values at the column positions, ignoring Nulls.  To get that the expression would be like this:

GrandTotal: NZ([total1],0)+NZ([total4],0)
