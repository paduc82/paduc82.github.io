I have a table with a number of yes/no fields. Is it possible to - Microsoft Community

bethparks say:

I have a table with a number of yes/no fields. Is it possible to use a parameter to return a yes/no field in a table?

I have a number of different fields in a table set to data type yes/no.  I would like to build a query that will request which field to return.  Is this even possible?
ken sheridan say:
or if I was clear enough.

You were perfectly clear, but unfortunately what is clear is that your database design is badly flawed.  In the language of the relational model it is 'encoding data as column headings'.  A fundamental principle of the database relational model is the Information Principle.  This requires that all data be stored as values at column positions in rows in tables, and in no other way.  As the others have said the first thing you need to do is get your data into a properly normalized set of related tables.  What you have here is a many-to-many relationship type between people and activities.  To model this you need three tables:

People
....PersonID (PK)
....FurstName
....Lastname
....etc

Activities
....ActivityID (PK)
....Activity

and to model the relationship between them:

PeopleActivities
....PersonID (FK)
....ActivityID (FK)

You'll find an example of exactly this sort of relationship type and how to represent it in a form/subform as ParentActivities.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

More pertinent to your question, however, is the file MultiSelect.zip in the same SkyDrive folder.  This has an option to filter an employee's report by one or more projects in a related table,  In this employees are analogous to your people and projects to your activities.  This little demo illustrates how to use a multi-select list box to select one or more projects and return the employees who have been assigned to any of the selected projects.

To restructure your database first create the Activities and people activities tables and set up enforced relationships between them (see the relationships window in my ParentActivities demo for the model).  Firstly you need to insert rows into Activities, which you have to do manually, one row per activity, inserting the activity name into the Activity column; the ActivityID as an autonumber will automatically be assigned values.

If your existing table doesn't already have a PersonID autonumber primary key column give it one; it's values will be assigned automatically.

Next you create a UNION based on your current table and the new Activities table:

SELECT PersonID, ActivityID
FROM [YourCurrentTableName], Activities
WHERE [Greeter] = TRUE
AND Activity = "Greeter"
UNION ALL
SELECT PersonID, ActivityID
FROM [YourCurrentTableName], Activities
WHERE [Meal Care] = TRUE
AND Activity = "Meal Care"
UNION ALL
SELECT PersonID, ActivityID
FROM [YourCurrentTableName], Activities
WHERE [Driver] = TRUE
AND Activity = "Driver";

Include separate SELECT statements for each of your activities along the above pattern.  Save this query and then base an 'append' query on it to insert rows into the new PeopleActivities table.  You should now find that you can join all three table in a query and return the activities per person.  A simple parameter on the Activity column would enable you to select people by activity, though a dialogue form with a multi-select list box as in my demo is more flexible of course.

Once you are happy that the tables are correctly populated with data the Boolean (Yes/No) activity columns from your original table can be removed from the table design and the union query deleted.   I would advise that you back up the database securely before doing the above, however.

You do not have to use the table and column names I've used, of course, but remember that if you use names which include spaces or other special characters (not recommended) they must be wrapped in square brackets [like this] in a query.  If in doubt include the brackets.
