doug robbins - mvp office apps & services (word) say:

Use a macro with the following code, inserting the keyword in place of keyword in the code (retain the quotes

 

Selection.HomeKey wdStory
Selection.ClearFormatting
Dim rngfound As Range
With Selection.Find
    Do While .Execute(FindText:="keyword", Forward:=True, _
    MatchWildcards:=False, Wrap:=wdFindStop, MatchCase:=True, MatchWholeWord:=True) = True
        Set rngfound = Selection.Bookmarks("\line").Range
        rngfound.Font.Bold = True
        Selection.Collapse wdCollapseEnd
    Loop
End With

Note that after the bold font has been applied, the emboldened text may flow over to the next line.

C:\Program Files\Microsoft Office\Office14\STARTUP

Look in each place and delete the AVWiz file wherever you find it. Then restart Word.

jim gordon mvp say:

Enabling connections and refreshing data are independent actions.

Closing a workbook closes the data connection.

When you open a workbook that has a built-in connection you get the yellow Security Warning band about Enabling data connections to alert you that the workbook has a query that wants to connect to an external data source. Because Microsoft has been criticized in the past regarding security issues, they now display this warning and there's nothing you can do or say that will get them to change this behavior.

If you want to maintain the data connection, don't close the workbook. The connection will stay open as long as the workbook is open, at least as far as Excel is concerned. Network or other policies on your data source can terminate the connection.

Refreshing the data on open is controlled in the Data Connection properties. There's a Properties button in the dialog that asks where on the worksheet you want to place the data when you first bring your data to the worksheet.

If your data is in an Excel Table, convert it to a range. Then, right-click on your data set and choose connection Properties. You'll get a dialog where you can specify to refresh the data on open, set a schedule for refreshing the data, or choose to not refresh data on open.

ken sheridan say: I think your model is wrong.  I think the GRNDetail table should model a many-to-many relationship type between PODetail and GRNEntry like this:

POEntry---<PoDetail----<GRNDetail>---GRNEntry

The definitions of the latter two tables would be:

GRNEntry
….PO  (FK)
….GRNDate

The table is 'all key', i.e. its primary is a composite one of the PO and GRNDate columns.  You might wish to add an additional candidate key of GRNNumber or similar for administrative purposes.  If so this column should be uniquely indexed.

GRNDetail
….PO  (FK)
….GRNDate  (FK)
….POD  (FK)
….QtyRec

The primary key of the table is a composite one of the three columns made up of the two foreign keys, PO/GRNDate and POD, the former being a composite foreign key referencing the composite primary key of GRNEntry .

A GRN can thus be raised in relation to quantities of one or more products per purchase order received on the same date.  For data entry into GRNEntry and GRNDetail I would recommend that it be done independently of the purchase order form by means of a parent from based on GRNEntry, within which is a subform based onGRNDetails, linked to the parent form on PO;GRNDate.  The controls in the subform would be a combo box bound to the POD column, set up to hide the bound column  and show the Product name in the usual way, and a text box bound to the QTRyRec column.

To raise a new GRN in relation to a purchase order a button could be added to the POEntry form to open the GRNEntry form at an empty new record, passing the current PO value to it as the OpenArgs argument of the OpenForm method.  In the GRNEntry form's BeforeInsert event procedure the value of its OpenArgs property would then be assigned to a control bound to the PO column.
