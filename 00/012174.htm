jasles say:

To keep only the "primary" worksheet visible upon opening, and keep the input box visible until the correct password is entered, replace your code with this:

Private Sub Workbook_Open()

    On Error Resume Next
   
    Dim MyPassword As String
    Dim WS As Excel.Worksheet
   
    For Each WS In ThisWorkbook.Worksheets
        If WS.Name <> "primary" Then WS.Visible = xlSheetHidden
    Next
   
    Do Until MyPassword = "pass"
        MyPassword = Application.InputBox(prompt:="Enter password", Type:=2)
    Loop
   
    For Each WS In ThisWorkbook.Worksheets
        WS.Visible = xlSheetVisible
    Next
   
    Set WS = Nothing

End Sub

    ken sheridan say: Are the iodinemax and iodinemin values in the referenced Output table time-independently determined by the foreign key ACCFSN column?  If so the table is not normalized to Third Normal form, which requires that all non-key columns be functionally determined solely by the whole of the primary key.  In your case the two columns would be transitively dependant on the key.  The table would thus be wide open to the risk of update anomalies, making the database unfit for purpose.  The columns must be deleted from the Output table to rectify this.

    If, on the other hand, the two columns are not time-independently determined by the foreign key ACCFSN column, then the table is normalized to 3NF.  I very much doubt that this is the case, however, but if this were really to be the case then the values would need to be assigned to the columns in the referenced Output table when a row is inserted.  This can be done by means of code in the subform's BeforeInsert event procedure, calling the DLookup function to return the relevant values from the referenced production Information table, with the  value as the criterion.

    add numbers together from different columns in access - Microsoft Community

    patrickjudd say:

    add numbers together from different columns in access

    I have a table with field names of dates, 1/6/15, 1/13/15,/1/20/15,/1/27/15.  data type = numbers.  I want to add the sum of numbers from each field together.  patrick
    dhookom say:

    I agree your table structure is wrong. However, you can create a couple queries to normalize and then sum:

    Create a union query to normalize your data:
    [quniDateValues]
    SELECT [PrimaryKeyFieldName], #1/6/15# as TheDate, [1/6/15] As TheValue
    FROM [YourTableName]
    UNION ALL
    SELECT [PrimaryKeyFieldName], #1/13/15#, [1/13/15]
    FROM [YourTableName]
    UNION ALL
    SELECT [PrimaryKeyFieldName], #1/20/15#, [1/20/15]
    FROM [YourTableName]
    UNION ALL
    SELECT [PrimaryKeyFieldName], #1/27/15#, [1/27/15]
    FROM [YourTableName];

    Then create a totals query:
    SELECT [PrimaryKeyFieldName], Sum(TheValue) as TheSum
    FROM [quniDateValues]
    GROUP BY [PrimaryKeyFieldName];

    Not sorry, because this gives me the chance to educate you on proper Database design. So I hope you aren't sorry that you allowed me to do that.

    1) Don't accept the default name for Autonumber ID fields. Use ClientID for example.

    2) While you have a ClientID field, you don't have a Foreign Key for to relate the client to the visits table.

    3) Your client table should be broken down into at least 2 tables: tblHousehold and tblMembers:

    tblHousehold

    HouseholdID (PK autonumber)

    HouseholdName

    StreetAddress

    City, State, Zip, Phone etc.

    Dateofmembership

    tblMember

    MemberID (PK Autonumber)

    HouselholdID (Foreign Key)

    Lastname

    Firstname

    DOB

    SSN (Social?)

    I'm not sure if fields like WIC, FoodStamps (notice no spaces in object names), Veteran, ChurchAffiliation, etc. belong in the Household or member table.

    Then your Visits:

    tblVisit

    VisitID (PK Autonumber)

    ClientID (FK)

    MemberID (FK)

    VisitDate,

    Allotment

    tblAttachment

    AttachmentID (PK Autonumber)

    ClientID

    MemberID

    AttachmentType
