bernie deitrick say:

Areas ignores hidden rows - so your code is putting formulas into hidden cells.  Try it like this, using visible cells:

Sub TestMacro()
    Dim Area1 As Range
    Dim Area2 As Range
    Dim MyColumn As String
    
    MyColumn = "O"
    Set Area1 = Cells(Rows.Count, MyColumn).End(xlUp)
    While Area1.Row <> 1
        Set Area2 = Range(Area1, Area1.End(xlUp))
        If Application.CountA(Area2) <> Area2.SpecialCells(xlCellTypeVisible).Count Then Set Area2 = Area1
        Area1.Offset(1, 1).Resize(100, 1).SpecialCells(xlCellTypeVisible)(1).Formula = "=SUBTOTAL(109," & Area2.Address & ")"
        Set Area1 = Area2(1).End(xlUp)
    Wend
End Sub

The solution I discovered when VBE7.dll is crashing is to "Convert Form's Macros to Visual Basic" (or any other macros for that matter) whenever clicking a form and the database freezes, which prompts the user and forces the database to either check for online solution or immediately close.  I discovered this when adding a new button feature to all my forms; however, the crash would only occur when clicking on a couple of the forms even though I installed the same exact button (copy and pasted) onto each form.  I recall a similar problem in the past, which the solution was to convert the macros into Visual Basic.

Regardless, I converted the buttons just like everything else is converted to Visual Basics and the crash problem was resolved.  In addition, in my opinion it is better not to run macros as some workstations/networks block or disable it by default because malicious code is executed and/or gaining a foothold in a system/network is achieved using the macro feature.

dirk goldgar say:

I haven't experienced the issue you described, VBE7.dll crashing.  If it happens with a particular database, but not others, I'd guess it's caused by corruption of the VB project.  If it happens with all databases, then I'd think there must be something wrong with the installation of Access or Office.

In the former case, probable corruption of the VB project, I would first try decompiling the database, compacting it, and recompiling, and see if the problem goes away.  In the latter case, possible problem with the installation of Access/Office, I would try repairing the installation or reinstalling.

It's interesting (and surprising to me) that a conversion of macros to VBA would fix a problem with the VBE7 DLL, since you would be adding to the VB project by doing so.  I wonder if that effectively causes a decompile of the database, or the replacement of some part of the VB project.

I'm not sure I understand what you mean by, "in my opinion it is better not to run macros as some workstations/networks block or disable it by default because malicious code is executed and/or gaining a foothold in a system/network is achieved using the macro feature. "  One of the main reasons for  using macros instead of VBA code is to eliminate the security threat that VBA presents.  VBA code can do anything, while macros can execute those things for which defined macro actions have been built.  As a result, high-security environments often prohibit VBA code, but permit macros.
