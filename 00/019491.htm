jay freedman say:
You shouldn't need the first part about selecting the figure title, because the Insert
Cross-Reference dialog presents a list of the things it can refer to, and you have to
choose from that list. The following macro does what you requested:
 
Sub InsertXRef()
    Dim dlg As Dialog
    With ActiveDocument
        If .ProtectionType <> wdNoProtection Then
           .Unprotect  ' Password:="myPassword"
        End If
       
        Set dlg = Dialogs(wdDialogInsertCrossReference)
        dlg.InsertAsHyperlink = msoCTrue
        dlg.Show
       
        .Protect Type:=wdAllowOnlyFormFields, NoReset:=True ' ,Password:="myPassword"
    End With
End Sub
 
If there is a password applied to the forms protection, you'll need to uncomment the
Password parameters in the Unprotect and Protect statements and insert the password in
place of myPassword.
 
One caution: Although this will let you set a cross-reference to a figure, table, etc. in
a protected section, clicking the hyperlinked cross-reference will  *not* jump the cursor
onto that caption, or even necessarily anywhere near it. The cursor can't go anywhere
inside a protected section except into a form field, so Word will force the cursor to the
"nearest" form field (but Word's idea of "nearest" may be the first form field in the
document, or somewhere else).

The "accent 3" part refers to one of the colors that are part of a "theme", which is a set of colors, fonts, and text effects. You choose a theme from the Theme button on the Design tab in Word 2013 (it's on the Page Layout tab in Word 2010). Each theme has a different selection of colors, as shown by the little swatches in each theme's thumbnail picture. Just to complicate things, Word 2013 has an entirely different set of themes from those in Word 2010.

The color that Word 2010 shows as orange 50% darker can be created in the More Colors dialog's Custom tab by setting Red = 127, Green = 51, and Blue = 0.

rob koch say:

What's going on is exactly what the notification from CFA told you, the applications involved are attempting to make some sort of change in the Windows Desktop folder of the user starting the program.

The problem with this is that it's the same type of operations that might be seen when a ransomware attack attempts to modify items in the Desktop Folder in order to make your system non-functional or encrypt whatever personal files you may have placed there.

The decision you (and the developers of those badly behaving programs) need to make is whether the actions being attempted by these otherwise valid programs truly need to occur, since these are why Microsoft can't simply block all access to folders like the Desktop in order to avoid ransomware attacks altogether.

Since the most common reason that older (e.g. outdated design) applications made such access to the Desktop was simply to add an icon, which is something you could do yourself with a simple set of mouse clicks.  By automating this in either their installation programs or worse yet when you're simply starting the program itself, they create issues for Microsoft when trying to protect you from programs with more malicious intent.

If all you see when using these programs is such notifications and they otherwise seem to operate properly, then you can simply ignore the message(s) and continue using the programs with CFA fully intact.  If they don't work properly, then you'll need to allow those programs the rights within CFA to make changes to those folders or better yet, update to more current versions of these programs that no longer require this access.

In short, it's your trashy outdated software that's making it impossible for Microsoft to easily protect you from ransomware, so either get rid of them or live with it.

Rob

Set rngStart = ActiveDocument.Bookmarks("bkmark1").Range
Set rngEnd = ActiveDocument.Bookmarks("bkmark2").Range

    ActiveDocument.Range(rngStart.Start, rngEnd.End).Copy

jay freedman say:

Use a Range object. Set its .Start and .End properties to the .End of bkmark1 and the .Start of bkmark2, respectively. Then check the length of the result of applying the Trim function to the range's .Text property. (The Trim function removes space characters from the left and right ends of the string.) If the length is zero, then there were only spaces, or nothing, to begin with.

Sub sample()
    Dim rg As Range
    
    With ActiveDocument
        Set rg = .Range
        rg.Start = .Bookmarks("bkmark1").Range.End
        rg.End = .Bookmarks("bkmark2").Range.Start
        
        If Len(Trim(rg.Text)) = 0 Then
            MsgBox "Range consists of spaces or nothing."
        Else
            MsgBox "Range:" & vbCr & rg.Text
        End If
    End With
End Sub
