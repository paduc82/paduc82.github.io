This isn't specifically about read-only files, but the symptoms are similar, so I'm posting in case it helps. By default, PowerPoint 2010 sees a network share as an Internet location and puts the presentation into Protected View. Usually, you'll see a yellow-orange bar under the Ribbon with the text:

Protected View This file originated from an Internet location and might be unsafe. Click for more details.

To the right of the text is the Enable Editing button, clicking on that unlocks the file for editing.

If the above describes the conditions you see, you can turn off Protected View:

  1. In PowerPoint, click on File, then on Options.
  2. Select the Trust Center pane, then click on the Trust Center Settings button.
  3. Select the Protected View pane, then uncheck at least options 2 and 3, or all of them, if you prefer. OK out.

scottgem say:

 I believe that if I use a form to edit data when using a query as a source, the table will not be updated ... Im thinking that I can just use the table as the source all of my reports.....maybe?


First, your belief is partially incorrect. A query is simply a view of the data in tables. The advantage of using a query is the ability to filter data. On the other hand, a multi-table query may not be editable. Generally, I will use a table as the Recordsource of my forms, though sometimes using a single table query where I want to display a limited number of records in the form. I will then use subforms to allow for viewing/editing/inputting of data into related tables.

On the other hand, I will, almost always, use a query as the Recordsource of a report. This is because, in a well designed database, data is spread out over many tables and you need to join those tables in a query to pull in all the data you need. If you have designed a database where all the data you need for a report is in one table, you probably have not properly designed the database. 

To illustrate this I'll use the fairly standard example of an order entry application. Such an application will have, at the least, 4 tables:

Customers: Information about your customers, names, address, etc.

Products: info about your products, name, unitprice, etc.

Orders: Info about your orders, order date, customer, shipping address etc.

OrderDetails: info about the items orders: Product, quantity, unitprice, etc.

So you would have an Order form bound to the Orders table. On that form would be a combobox to select the CustomerID (and display customer info) as well as controls to enter the date, etc. Also on that form would be a subform bound to the OrderDetails table and linked to the main form on OrderID. On the subform, you would have a combobox to select the ProductID (by ProductName) and enter quantity. Note that your Orders table would NOT have fields for Customer name since that is stored in the Customers table. Since that would be redundant, Similarly, the Product name would not be stored with the Details, only the ProductID as a foreign key.

Conversely, if you were creating a report of Orders, You would create a query that joins the all four of those tables. You would pull customer info from Customers, Order info from Orders, Product info from Products and Detail info form OrderDetails.

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to amend the form design accordingly.  

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo file illustrates in simple terms some of the basic concepts underlying the database relational model and how these are implemented in Access.

In my file you'll see that there is mention of the important concept of functional dependency.  In your case it seems that PostCode determines both AreaNumber and Price so this would be represented by a referenced table like this:

PostCodes
….PostCode
….AreaNumber
….Price

Postcode is a candidate key of this table, so can be made its primary key.

In the referencing table, the one in which you are entering data via a form, PostCode is a foreign key column referencing the primary key of PostCodes, but as AreaNumber is time independently determined by PostCode there should be no AreaNumber column in this table as this would introduce redundancy and the risk of update anomalies.  Price, on the other hand, is not time independently determined by PostCode as prices will change over time, so a Price column in the referencing table is legitimate as it is functionally determined by the key.

The AreaNumber can be shown in the form by various means.  One would be to have a combo box bound to the PostCode column set up as follows:

Name:               cboPostCode
ControlSource:  PostCode
RowSource:      SELECT PostCode, AreaNumber, Price  FROM PostCodes ORDER BY PostCode;
ColumnCount:   2

For the AreaNumber add an unbound text box to the form, with a ControlSource property of:

=[cboPostCode].[Column](1)

The Column property is zero-based, so this will show the second column, AreaCode.

For the price add a bound text box, named txtPrice, to the form with Price as its ControlSource property.  To insert the current price from the PostCodes table into this as the default value, which can then be edited if necessary, put the following code in the AfterUpdate event procedure of the cboPostCode combo box:

Me.txtPrice = Me.cboPostCode.Column(2)

If you are unfamiliar with entering code into a form's or control's event procedures, this is how it's done in form design view:

1.  Select the form or control as appropriate and open its properties sheet if it's not already open.

2.  Select the relevant event property and select the 'build' button (the one on the right with 3 dots).

3.  Select Code Builder in the dialogue and click OK.  This step won't be necessary if you've set up Access to use event procedures by default.

4.  The VBA editor window will open at the event procedure with the first and last lines already in place.  Enter or paste in the code as new lines between these.

ken sheridan say: PS.  It's possible that Price might be determined by AreaNumber rather than PostCode if there are multiple post codes per service area, in which case the PostCodes table would need to be decomposed into two tables as follows:

PostCodes
….PostCode  (PK)
….AreaNumber  (FK)

ServiceAreas
….AreaNumber  (PK)
….Price

The RowSource property for the combo box would then be:

SELECT PostCode, ServiceAreas.AreaNumber, Price  
FROM PostCodes INNER JOIN ServiceAreas
ON ServiceAreas.AreaNumber = PostCodes.AreaNumber
ORDER BY PostCode;

ken sheridan say: This probabaly arises from differences between the way US Zip codes and UK postcodes work, but Scott is assuming that there are multiple service areas per postcode,  whereas I, on the basis that a UK postcode defines a very small area (one side of my street of 72 houses for example is defined by ST17 4PF), am assuming that there are multiple postcodes per service area.  We both assume that price is determined by the service area.  If I'm right then your tables are correct; if not Scott's model is the correct one.

If we assume that I'm right for the moment, you need to create relationship between the two tables on AreaNumber, and enforce referential integrity.  That protects the integrity of the data.

The interface for data entry into the two tables would be a ServiceAreas form, in single form view, and within it a PostCodes form in continuous forms view, linked to the parent form on AreaNumber.  The subform would contain a single control, bound to the PostCode column.  The correct value will be automatically inserted into the AreaNumber foreign key column by the linking mechanism, so there is no need for a control bound to it in the subform.

BTW, do not include spaces or other special characters in table or column names.  I know that Access allows it and it's often done in Access templates, but it's a bad idea.  Either use CamelCase, e.g. ServiceArea, or represent a space by an underscore character, e.g. Service_Area.
