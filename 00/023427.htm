Word doesn't track changes within a SmartArt object (such as adding/deleting shapes or modifying text).

why do I get an incorrect result displayed when multiplying - Microsoft Community

carolwebster say:

why do I get an incorrect result displayed when multiplying

why do I get an incorrect result displayed when multiplying 111,111,111 by 111,111,111

the answer is 12345678987654321

varying the decimal places makes no difference since the result is an integer anyway

joeu2004 say: Carol wrote:
why do I get an incorrect result displayed when multiplying 111,111,111 by 111,111,111
the answer is 12345678987654321
but excel, in at least two versions, provides a result of 12345678987654300

People might say "Excel stores numbers with a precision of 15 significant digits" and " Excel can only calculate to 15 digits".  But that is wrong.

In fact, if A1 contains 111,111,111 and A2 contains =A1*A1, Excel computes exactly 12345678987654320, which is only off by one.  You can confrim that by computing =A2-12345678987654300.  Note that the result is 20.

The correct answer to your question ("why...?") is two-fold.

First, it is true that Excel will format only the first 15 significant digits (i.e. not counting any leading zeros).  That is why it displays 12345678987654300 instead of 12345678987654320.

(Excel has a similar limitation on entering numbers.)

Second, it is true that all computer arithmetic, including calculator arithmetic, is limited by the finite precision of the internal representation and computation of numbers.  But there are choices that can be made.

In the case of Excel, it is limited by the 53 bits (binary digits) of precision in the 64-bit floating-point representation.  Consequently, the largest integer that Excel can represent exactly is 9,007,199,254,740,992.  (Although Excel will display 9,007,199,254,740,990.)

(By the way, note that that is 16 digits, not 15.)

For more details (albeit flawed in some respects), see http://support.microsoft.com/kb/78113,

kristine arp say:

Hi Roger,

 

 

Text-To-Speech is a feature designed to read aloud digital text like the ones you’ve typed in Word using a digital voice. However, unlike other accessibility tools, it was only designed to read up to 4,096 characters including paragraph marks and other hidden formatting characters. This is the reason why the process would stop at some point and the user would have to restart it all over again. As a workaround, you can break your document into smaller sections to enable Speak to read them.

 

Though it’s limited, you can download other languages for Text-To-Speech feature. Check this link for the list of available languages.

 

Since Text-To-Speech is using a digital voice, it is bound to some reading limitations like incorrect pronunciations, unnecessary pauses, incorrect tone or emotion. Though it was designed to as an accessibility tool, it is not recommended to be used to create an audiobook recording. You might want to check third-party accessibility tools that can achieve your goal.

mike h.. say:

Hi,


What is CurrentMonth? I have assumed it's a named range that contains a month number. Try this




=SUMPRODUCT((MONTH(B7:B41)=CurrentMonth)*(B7:B41<>"")*(H7:H41))


You need the additions check that B7:B41 isn't empty because an empty cell would evaluate as the month of January.

mike h.. say:

Hi,



You said


<If I understand correctly, you are building an array of 0 if a condition is false and 1 if it is true, then multiplying each element in the first array by the array I want to sum and finally summing the new array


That's nearly what the formula does, what it exactly does is build an array of TRUE and FALSE for the month range not 1 and zero. The TRUE and FALSE are coerced into 1 and zero when we do the multiplication.


You said


< Can you elaborate on what (B7:B41<>"") means?


Yes I can. If any of the cells in column B are empty then that is the same as zero. If you put a zero in a cell and format as date then you'll see it evaluates as 1/1/1900 so if we are summing for January we need that bit of the formula to stop blank cells evaluating as January.


You said


<Is it not possible to evaluate the MONTH() function within the range of SumIf?


It is but it's messy because SUMIF and SUMIFS don't accept a month argument for the range for example =sumif(month(B1:B20),1,H1:H20)) doesn't work. However we can do this with sumifs:-


=SUMIFS(H1:H20,B1:B20,">" & A1,B1:B20,"<"&A2)


In the above formula A1 contains the date 31/12/2012 and A2 contains the date 1/2/2013 so the formula sums January 2013 which isn't really what you asked for you wanted to sum by MONTH and didn't specify year.


You also said my original sumproduct formula returned a #VALUE! error. The reason for that is almost certainly you have text value in either column B or column H that may look like numbers.


Try this. Enter this in 2 empty cells in the same row


=isnumber(b7)

=isnumber(h7)


Drag both down and they should return TRUE if any return false then you have text. You may also have formula in the range returning "" or null strings.

daniel pineault say:

Firstlt, it is a horrible idea to develop using a more recent version of Office than your end-users.  You must always do your development using the least common denominator, so in your case Office 2003.

 

As for your problem, I suspect either you recently added an ActiveX control or a library that does not exist on earlier versions.  Open the database on a computer with the problem.  Go into the VBE and select Tools -> References.  One should be marked as being MISSING, you will need to select the library version on that PC, maybe version9.0 or 8.0.  Then try again.

If you don't see any MISSING references, try compiling your code, does it bring up any errors?

Your true solution however is to switch all your code from early binding to late binding.  This avoids such issues since you do not include references in your database, they are called at runtime based on the version available on the local machine.
