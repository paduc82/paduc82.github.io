PS:

boblarson say:

Your SQL should be:

 

SELECT subcategories.name, subcategories.ID
FROM subcategories
WHERE subcategories.category_ID = [Forms]![YourFormNameHere]![categoryCombo] OR [Forms]![YourFormNameHere]![categoryCombo] IS NULL;

boblarson say:

And if it is on a subform, then

 

SELECT subcategories.name, subcategories.ID
FROM subcategories
WHERE subcategories.category_ID = [Forms]![YourFormNameHere]![SubFormControlNameHere].[Form]![categoryCombo] OR [Forms]![YourFormNameHere]![SubFormControlNameHere].[Form]![categoryCombo] IS NULL;

 

Where SubFormControlNameHere is the name of the control on the parent form which houses/displays the subform and NOT the subform name itself UNLESS they both share the same exact name.  The .Form! part would remain as shown and the YourFormNameHere is the name of the parent form.

ken sheridan say: One further point.  You should also set the subcategory combo box to Null in the AfterUpdate event procedure of the category combo box.  If a user changes their mind after selecting a subcategory, and goes back and selects a different category in the first combo box the subcategory combo box will appear to have no value.  However, its real value will be the hidden subcategoryID value originally selected.  Consequently, if the user does not select a subcategory the underlying table will contain conflicting values in the two columns.

This raises another issue.  By having columns for both the category and subcategory in the table it is not normalized to Third Normal Form (3NF)  because subcategory determines category, so the latter is transitively functionally dependant on the key of the table.  All non-key columns should be determined solely by the whole of the key for the table to be normalized to 3NF.  A table not in 3NF allows the possibility of the sort of update anomalies which I've described above.

You might be prepared to live with this degree of non-normalization - many people are.  If not, you'll find examples of how to deal with this sort of thing in the context of a normalized table in the file ComboDemo.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

This little file illustrates a number of approaches, including the use of hybrid controls to allow the top-down selection of hierarchical values by means of correlated combo boxes.  This does of course require the use of continuous forms view, not datasheet view.

E51 Dropped Bluetooth Connection/Calls SOLUTION for BMW - Microsoft Community

bmwe51fix say:

E51 Dropped Bluetooth Connection/Calls SOLUTION for BMW

After several months of pure frustration and getting no where with Nokia or BMW (both blaming the other) trying to fix my dropped bluetooth connection every 2mins during calls in my 2007 BMW 335i, I came to these forums for help. Unfortunately I haven't really come across a thread that seemed to offer a clear cut solution but there was a reference in one of the myriad of "contact transfer" threads that helped me figure out a solution that worked.

Problem: my E51's Bluetooth connection will establish but disconnect/drop every 2mins, as other have stated, making calls virtually impossible.

Solution: The problems seems to be that that phone is attempting to unsuccessfully transfer contact information to the car (though there is no sign or message stating that it is trying to do this). It seems to drop the connection after the failed attempts...for my E51, this usually was close to 2mins.

Another forum member mentioned that this might be the problem and to "unsynch" your contacts, and that seemed to work for them.

I did this in "PC Suite" and voila! the damn thing actually stayed connected indefinitely!! Amazing...details of my phone/ PC Suite setup are below: THIS SOLUTION WILL NOT ALLOW ANY CONTACT TRANSFER TO OCCUR TO THE CAR, ONLY ALLOW BLUETOOTH CONNECTION FOR MAKING AND RECEIVING CALLS

Some of these steps may be redundant, but i'm not changing a damn thing since this has worked without a dropped call in over a month!

PC SUITE: access from menu, then "Tools" folder
then click the "Sync" icon
--you will then see PC Suite, click options and "edit sync profile"
--click "Applications" and click the first setting "Contacts"
-- change "include in sync" to "no"
(the other two settings: remote database is set as "contacts" and synch type "both ways", but this is not relevant since you are removing the ability to synch your contacts)
-- now go back to main PC Suite page and click the "connection settings" (don't know if you need to change these, but i'll tell you what mine are)
Server Version : 1.1
Data Bearer: Bluetooth
Host Address: PC Suite
User name: none
psswd:**** (whatever you have selected)
Allow sync requests: NO

HOPE THIS WORKS FOR YOU

Again this is only a solution for dropped connection problems, I have not been able to successfully transfer a single contact to my car (though my wife's motorola and my old nokia 6131 did so without issue). I've tried the contact download manager and mutliple steps/suggestions on the forums to no avail. Nokia has blamed BMW and BMW dealer says it's a Nokia problem. Personally I believe it's a Nokia problem since other nokia phones work and this phone will connect and stay connected if you "unsync" your contacts.

At this point, since i love the phone, I'm happy with just using the hands free connection in my car without any issues. I'm not a business person, so the speed dials work for me for calling. Hope this helps.

I probably won't check back on this forum much, just felt obliged to help those who are having the same frustrating problem as I am. If you have any questions, just contact me.
