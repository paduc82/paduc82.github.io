Combo box updating based on results of another - Microsoft Community

sirtruman say:

Combo box updating based on results of another

I have read quite a few examples on updating one combo box with the results from another but I have a little different scenario that I cannot find an answer to. I have two tables, Project and Permits. Fields in Projects are ProjectID, ProjectNumber, ProjectName, and PermitID thrown in because I am a beginner and trying to figure out table relationships. Anyway, Permits has PermitID, PermitNumber, ProjectNumber, and ProjectName.

The Projects have to be entered into the Projects table first with a ProjectNumber or ProjectName and sometimes we actually have both at the time of entry.

So then in the Permits table we can have a number of permits to a project. I have a combo box which displays the available projectnumbers from the projects table and the ProjectNumber combo box displays the available ProjectNames from the projects table. The problem is that for each permit you can select a projectnumber from the combo box and also select a project name that does match the number from the project table.

I want to pick a project number from the combo box in the permit form and have the projectName combo box autofill with the matching name. I know this is the standard cascading combo box, but I want the reverse to also happen. If there is not a projectnumber, I want to be able to select a projectname and have the project number autofill even if it's a null value or not. Is this possible and how? Thanks

ken sheridan say:
I know this is the standard cascading combo box, but I want the reverse to also happen. If there is not a projectnumber, I want to be able to select a projectname and have the project number autofill even if it's a null value or not. Is this possible and how?

Actually, this has nothing to do with correlated ( aka 'cascading') combo boxes.  Correlated combo boxes are where the values available for selection in one are restricted to a subset of the values of the attribute in question as determined by the selection in another, e.g. selecting a state in one determines what cities are available for selection in another.  One does not update the other, which is what you want to happen.

As the Projects table allows either the ProjectNumber or ProjectName to be Null, neither of these can be the primary key of the table, so the first thing you need to do is add a ProjectID autonumber column to the table as its 'surrogate' primary key.  You should not have a PermitID column in the Projects table as this is the referenced table in the relationship.  So the structure of the Projects table should be:

Projects
....ProjectID  (PK)
....ProjectNumber
....ProjectName

Bill has suggested that there is a many-to-many relationship type between projects and permits which would be modelled by a third table which resolves the relationship type into two one-to-many relationship types.  Your description is unclear as to whether this is the case or not.  It would be so if each project can have one or more permits, and each permit can apply to one or more projects.  However, your description also allows for the interpretation of the relationship type being one-to-many from projects to permits, i.e. each project can have one or more permits, but each permit applies to one project only.  It may be the case that there are categories of permits, and if so, the Permits table would be modelling a many-to-many relationship type between projects and permit categories.

If we assume that each project can have one or more permits, but each permit applies to one project only, i.e. the relationship type is one-to-many, then the structure of the Permits table would be:

Permits
....PermitID (PK)
....PermitNumber
....ProjectID (FK)

Note that there is neither a ProjectNumber nor ProjectName column in Permits, only the ProjectID  foreign key column.  Consequently all redundancy is eliminated from the table.  A relationship between the tables on ProjectID should be created, and referential integrity enforced.

When it comes to the permits data entry form you need two combo boxes, both of which have ProjectID  as the ControlSource property.  The first would be set up like this:

RowSource:     SELECT ProjectID, ProjectNumber FROM Projects ORDER BY ProjectNumber;

BoundColumn:   1
ColumnCount:    2
ColumnWidths:  0cm

If your units of measurement are imperial rather than metric Access will automatically convert the unit of the last one to inches.  The important thing is that the dimension is zero to hide the first column.

The second combo box would be set up like this:

RowSource:     SELECT ProjectID, ProjectName FROM Projects ORDER BY ProjectName;

BoundColumn:   1
ColumnCount:    2
ColumnWidths:  0cm

You can select a value from either, and the other will automatically show the corresponding value, i.e. selecting a number shows the name in the other, and vice versa. QED!

Each row in the Permits table models a permit which specifically applies to one project only.  What Bill is suggesting is that each represents an instance of a type of permit, what I've called PermitCategories above.  If this is the case then the Permits table would have another column, PermitCategoryID say, which would be a foreign key referencing a table set up as follows:

PermitCategories
....PermitCategoryID (PK)
....PermitCategory

and the permits form would gave a combo box bound to the PermitCategoryID  column and set up in the same way as those I described above.
