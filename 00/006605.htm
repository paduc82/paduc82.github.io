a. user say:

Word MailMerge doesn't support XML format data directly as the data source for a MailMerge. You would need a mechanism to transform your XML into a data source that Word can work with.

Further, MailMerge expects a data source to be tabular in the sense that it contains rows and columns, and each row has the same number of columns. XML is a very flexible format which allows all kinds of hierarchical "data shapes". So in the general case, a transform would need to "flatten" your data into a simple, tabular format. Even if you have a simple XML format like this

<books>
  <book id='1'>
    <some book info>
 </book>
 <book id='2'>
    <some other book info>
 </book>
...etc.
</books>

ie. where every child element of <books> is a <book>, you would need to end up with a data source format where each different possible child element (and attribute) of <book> was a separate column in your data source. So if for example you allowed multiple <author> elements per book in your XML, you would have to decide how to transform that in your data source:
 -  you could combine them into one long text column with a separator such as ";" between each author
 -  you could define columns called author1, author2, author3, and perhaps expand the number depending on the maximum number of authors per book you had in your XML file
 - you could create one record for each author/book combination. Then you would have to decide how to deal with that in the merge.

So although it's not impossible to do something useful, if you plan to use MailMerge, it's probably better to keep things simple and think in terms of what you need to achieve, and how to represent the data in a simple table. If you are more ambitious and are willing to get to grips with the XSL transform language XSLT, you could in theory transform any XML directly into a Word "Flat OPC" format XML file which Word could read and print. But it's not easy, in my view. I certainly don't think these groups would be the right place to find out more info. about how to do that. And in any case, you might be better off transforming into some other format (for example HTML5 or Microsoft's XPS format) rather than going via Word.

You could write a book about this. But... 

.csv in itself should be OK, but only if
 - there is a header row with column names
 - each data row has the same number of columns
 - Word understands where each column begins and ends (i.e. the column and row delimiters are unambiguous, and coumn and row delimiters that occur in the data are suitable wrapped or escaped
 - there is no confusion between the column delimiter (i.e. "," for a true csv file) and, say, decimal point characters
 - there is no confusion as a result of ambiguous character encoding (Word uses some heuristics to determine encoding and does not always guess correctly)

For example, Word would have no special problem with the following simple source

k,t1,t2
1,text11,text12
2,text21,text22
3,text31,text32

It wouldn't like this

k,t1,t2
1,text11,text12
2,text21,text22
3,text31,text32,text33

It probably wouldn't like this

k,t1,t2
1,text11,text12
2,text21,text22
3,text31,tex,t32

but would probably be OK with this

k,t1,t2
1,text11,text12
2,text21,text22
3,text31,"tex,t32"

and it probably wouldn't like this

k,t1,t2
1,text11,text12
2,text21,text22
3,text31,name
address1
address2

but might manage this

k,t1,t2
1,text11,text12
2,text21,text22
3,text31,"name
address1
address2"

etc.

There is another complication. Word reads csv type files using one of two main techniques:
 a. an internal or external text file converter
 b. an OLE DB provider (or ODBC driver, but let's stick to OLE DB)

In case (a), once Word has determined the column and row delimiters, it will pretty much leave the data as is. In case (b), Word uses the Windows/Office Jet/ACE OLE DB provider to read the data. It wil only do (b) if the column count is 255 or lower. Then, it will look for a file called schema.ini in the folder that contains the data source file, and look for entries that describe that file's character encoding, delimiter characters, column names and data types. It will also try to determine a data type for each column based on the specified data types and /or what it finds in the first few rows in each column. It will then interpret what it finds in each row accordingly. So if (say) the first 8 rows contain numbers, Word will decidethat the column is nummeric and a value such as "abc" will be returned as "0"

In other words, if your data is "simple", 99 times out of 100 you will be OK with a simple csv source, but if you want certainty, make sure your csv works no matter what it contains, or use another format (and there are no especially good choices in that case)

a. user say:

From your description, MailMerge should be a good fit. I had assumed you had tried it and come up against a problem with your data source, which is why I provided a lot of detail - probably far more than you need.

In my view, the best way to get going with mail merge is to start using it -  it should only take a few minutes to create a new document, attach the data source, insert a few merge fields and perform a test merge to a new document. If your tests indicate that you need VBA after that, I doubt you will have wasted much time.

For your "optional" fields, 
 A. To test a value of 'abc' and do something different you use an IF field, like this

{ IF "{ MERGEFIELD myfield }" = "abc" "insert text and fields for the case where it is abc" "insert text and fields for the case where it is not abc" }

All the {} have to be special "field brace pairs" that you can insert using ctrl-F9. There are plenty of examples of more complex IF field constructions in these groups, although they may be rather hard to discover - searching for the word "MERGEFIELD" is probably the best starting point. 

 B. there is no good way to test for field existence - I.e. Word will normally object if you have a field like { MERGEFIELD myfield } but there is no column called "myfield" in your data source. You can test an empty MERGEFIELD using an IF field (compare with "" ) or in some cases, by using an \f or \b switch in the MERGEFIELD. If you need to distinguish between an empty field and a missing field, you'll probably need another column that indicates whether "myfield" ishould be regarded as missing or not.
