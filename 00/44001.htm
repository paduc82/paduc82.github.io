a. user say:

Errors like that (24) indicate some (usually minor) corruption in the NT File System (NTFS) and that can happen if the system has a power failure, power interruption, ungraceful shutdown, aborted restart, plug pulling, using the reset button, removing the battery (for laptops) or sometimes system crashes.

When that happens sometimes the "dirty bit" will be set on a volume/drive to signal the Windows startup that something is wrong and if the dirty bit is set on a volume Windows will automatically run a chkdsk with error correction (chkdsk  /r) on the afflicted volume to try to fix the problem. 

Usually one pass is all that is needed but it can take a long time - several to many hours depending on the size of the volume, the speed of your system, the amount of data on the volume and what the chkdsk finds to do.

Why don't you learn now how to check the dirty bit on a volume so you know how to do it and know what it looks like:

You can query the dirty bit on a volume from the command prompt window with Administrator privileges.

Click the Start orb/button and in the box enter the following:

cmd.exe

Right click cmd.exe and choose to Run as Administrator:


To query the dirty bit on drive C, in the Command Prompt window enter the following command:

fsutil  dirty  query  C:

Sample output should be one or the other of:

Volume C: is dirty
Volume C is NOT dirty

You should check each of your volumes or each of your hard drives (C, D, E, etc.).

Are any volumes dirty?

If a volume is dirty the only thing that can clear it is a chkdsk with error correction (chkdsk  /r).

If your chkdsk is only taking 10 minutes it sounds like it is not a chkdsk with error correction but just a plain chkdsk which reports results but does not actually fix anything.  That would not be logged.  If a chkdsk with error correction runs on startup those results would be in the Event Viewer logs.

If a volume is dirty Windows will automatically try to fix it so you really don't really need to run the chkdsk on every reboot if you don't wan to waste that time.   And if it is just a plain "chkdsk  c:" that will not fix anything.

You don't know or remember how your system was set up to run the chkdsk on every reboot?  Seems like a waste of a good 10 minutes and just a "chkdsk  c:" will only report and not actually fix anything but maybe you like it that way.

Sometimes it might take more than one pass to clear the dirty bit and if the drive is not healthy it may never clear the dirty bit.  In those cases there are some workarounds.

Why not configure your system to generate crash dump files so you don't have to do all that renaming after a crash.  Crash dumps will then automatically accumulate in the Minidump folder and you can upload the most recent ones analysis as needed.

Here's how:

On your Desktop right click Computer, Properties, Advanced system settings and make it look like this:

There is no need to create the Minidump folder as Window will create one if it needs to.

Then click OK enough times to close all the dialog boxes and restart your system for the changes to take effect.  After rebooting you can go back and check the settings again to be sure the adjustments "stick".

a. user say:

If the drive is dirty Windows is going to try to fix it by running a chkdsk with error correction (chkdsk  /r) when Windows starts.  It could be that is why chkdsk runs on every reboot.  Maybe it is not something you have configured somewhere.

When the chkdsk is done check the dirty bit again to see if it is clear.

If it is clear and becomes dirty again something is marginal or you have had something like a power failure (the usual cause).

The only thing that can clear the dirty bit is chkdsk /r.

Sometimes no matter how may times you run chkdsk /r the dirty bit will not clear and that could be a sign of a drive getting ready to fail...   or it might last a long time. 

I have an external drive where the dirty bit will not clear (for many years) and have informed Windows not to check it on restart because it takes too long and I don't think it will ever clear the dirty bit but the drive still works and if it someday fails it won't bother me too much.

If the dirty bit just will not clear and you want to live with it (keeping your fingers crossed) you can tell Windows not to check the dirty bit on a volume with the following command using the C drive as an example:

chkntfs  /x  c:

To reset/undo that action and set things back to normal/default issue this command:

chkntfs  /d

========== About MSDN or TechNet Product Keys - Credit Andre Da Costa ==========


"About MSDN or TechNet Product Keys
They are genuine Microsoft product keys, they are actually retail licenses, but it is intended for a particular product channel either the Microsoft Software Developer Network (MSDN) or TechNet for IT Professionals who pay a subscription fee. The main purpose is for evaluation purposes. The great thing about them, unlike trial Microsoft software, MSDN or TechNet keys don't expire. Because the agreement under which the subscription is provided is a single license, none of the software should be distributed outside of it. Even though its $50,000 worth of licenses, it is for one person only to use and no one else. Unfortunately, regardless of the licensing terms, persons still abuse the program, either giveaway product keys or resell it on auction sites. Microsoft licenses it in good faith that customers won't do so, but I guess human nature wins out.


In your case, what probably happened is, you bought an MSDN licensed key, which carries up to 10 activations unlike full packaged retail licenses which only carry 1 activation. The person who sold it to you probably sold it to 10 other persons. Somewhere along the way, one of those persons might have installed it on a second system, activated it, because it went past 10 activation threshold, Microsoft detected it that it was being abused and blocked the key from further use."

Additional information:

Computers, which are built by large manufactures that come with Windows Pre-Installed, come with two (2) Product Keys:

 

 OEM SLP: This key comes pre-installed in Windows, when it comes from the Factory. This key is geared to work with the special instructions found only on that Manufacturer's computer hardware. So when Windows was installed using the OEM SLP key (at the factory) Windows 7 looks at the motherboard and sees the special instructions and Self-Activates. (that's why you did not need to Activate your computer after you brought it home)

 

 COA SLP: This is the Product key that you see on the sticker on the side (or bottom,[ or inside the battery compartment]) of your computer. It is a valid product key, but should only be used in limited situations (such as if the OEM SLP key stops self-activating for whatever reason). The key must be activated by Phone. (Note: All manufacturers that use the OEM SLP system are required by contract to include a Certificate of Authenticity (COA) sticker, that has a COA SLP key, on the computer)
