ken sheridan say:

Are you suggesting a two step process, that is run the cross tab query then use your suggested query to create the average?

No, I'm suggesting you use the query instead of a crosstab query.  A crosstab query returns values at the intersection of rows and columns, the column headings being values from the table or tables on which it is based.  You, on the other hand, simply want to return a single row of aggregated values in this case, so there is no need to use a crosstab query at all.

Usually when returning aggregated values from one column grouped on the basis of values in one or more other columns, the results are in one column on separate rows, so a simple aggregating query in your case would return two rows.  However, by using the IIF function to return a value of 1 or 0 depending on the value of the 'group' (Advertising or Maintenance as defined by the column number range in your case) the aggregated values per group can be returned in separate columns in the same row.  The sum of the value in the column multiplied by 1 or 0 gives the sum for the group in question.  The sum of the 1 or 0 values gives the count of the rows for the group.  So, by dividing the former by the latter the average for the group is returned.

The query is based directly on your table as you posted, so is completely independent of any other queries and can simply be pasted into the SQL window of the query designer and opened.  It will prompt you to enter the line number parameters on which the result will be based.  The SQL statement is derived from the table structure you posted, however, so is reliant on that being the actual structure of your main table.  The only point of doubt is the data type of the two columns which could be text or an integer number formatted to show leading zeros.  I've assumed they are text, but the query would only need small amendments if not.

ken sheridan say: The AVG operator does what you say, viz 'count the number of entries in a given field and then divide by the number to get an average'.  What you actually want is 'count the number of distinct values in a given field and then divide by the number to get an average'.  That's not so straightforward as there is no COUNT DISTINCT operator in the flavour of SQL which Access uses.  The way it is done is to count the number of rows returned by a subquery in which the SELECT DISTINCT predicate is used to return one row per distinct value.  This is simple enough in itself, e.g. using Northwind as an example:

SELECT COUNT(*) AS CustomerCount
FROM (SELECT DISTINCT [Customer ID]
             FROM Orders
             WHERE [Ship State/Province] = "FL");

would return the number of customers who have placed orders where the shipping state is Florida.

The following would give the total value of all orders where the shipping state is Florida:

SELECT  SUM(Quantity*[Unit price]) AS TotalValue
FROM Orders INNER JOIN [Order Details]
ON [Order Details].[Order ID] = Orders.[Order ID]
WHERE [Ship State/Province] = "FL";

So the average value of orders by customer is the result of the second divided by the result of the first.

SELECT
    (SELECT  SUM(Quantity*[Unit price]) AS TotalValue
     FROM Orders INNER JOIN [Order Details]
     ON [Order Details].[Order ID] = Orders.[Order ID]
     WHERE [Ship State/Province] = "FL")/
COUNT(*) AS AverageValuePerCustomer
FROM (SELECT DISTINCT [Customer ID]
           FROM Orders
           WHERE [Ship State/Province] = "FL");

Rather than restricting the results to a specific state the total value of orders for each state can be returned by a conventional aggregating query:

SELECT  [Ship State/Province],SUM(Quantity*[Unit price]) AS TotalValue
FROM Orders INNER JOIN [Order Details]
ON [Order Details].[Order ID] = Orders.[Order ID]
GROUP BY [Ship State/Province];

The total value per state could then be divided by the number of distinct customers per state by means of a subquery which is correlated with the outer query on Ship State/Province:

SELECT  [Ship State/Province],SUM(Quantity*[Unit price])/
    (SELECT COUNT(*) AS CustomerCount
     FROM (SELECT DISTINCT [Customer ID]
                FROM Orders AS O2
                WHERE O2.[Ship State/Province] = O1.[Ship State/Province]))
AS AverageValuePerCustomer
FROM Orders AS O1 INNER JOIN [Order Details]
ON [Order Details].[Order ID] = O1.[Order ID]
GROUP BY [Ship State/Province];

Theoretically you should be able do the same by returning the sum of expenses per department and then dividing that by the count of distinct companies per department.  However, while this is possible in Northwind with its well-structured related tables, I'm not so sure it would be possible with your rather loosely structured tables.  Even if it were possible the task of transforming the conventional aggregating query into a crosstab query could be tricky.  What this shows is how important it is for the tables and relationships in a database to strictly adhere to the formal principles of the database relational model.

cody c. say:
Hi Bahaa,

There are two separate 14.2.4 installation issues that have caused this error.

 

The first issue (and most common in this thread) stems from re-installing on top of existing installations. In general, don’t do this. Re-installing on top of existing installations can re-introduce files that should not be present. If you need to re-install, please be sure to remove Office 2011 as described on the following links:

 

Video | Uninstall Office 2011

Step-by-Step Instructions | How to completely remove Office for Mac 2011

 

One workaround for the 14.2.4 update that has helped some users is to temporarily move or rename the /Library/Automator folder. You can do this by following the steps below:

 

  1. With Finder open, choose Go > Go to Folder...

  2. Type in the following path and then click Go:
    /Library

  3. Find the Automator folder, select it, and then click the name of the folder again to Rename the folder. Name the folder Automator_Old

    Note    :     OS X may prompt you for your password. This is normal.

  4. Attempt to install the 14.2.4 update again.

  5. If the update installs correctly, rename the Automator_Old folder back to its original name: Automator

 

If that solution does not work and you know you have re-installed Office 2011 on top of an existing Office 2011 installation, you’ll need to remove Office 2011 properly and re-install. 

 

The second issue was some users changed the size of the /tmp/ directory on their machine which was too small for the update to fully install. Re-sizing the /tmp/ directory resolved the install issue for these users.
