https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

The model for this little demo file is as below:



As you see the StudentCourses table models a ternary (3-way) relationship type between Students, Courses and Statuses, with the primary key of the table being a composite one of the StudentID and CourseID columns.

The demo illustrates a number of interfaces, but that of a conventional form/subform is the relevant one to look at in your case.  As you'll see the LinkMasterFields and LinkChildFields properties of the subform control are each StudentID, i.e. the primary key of Students and the corresponding foreign key in StudentCourses.  The latter is also part of the composite primary key of StudentCourses.  the other part of the key, CourseID is inserted in the subform via the bound CourseID combo box control.  A value is inserted into StudentID automatically by virtue of this being the LinkChildFields property, but if this were not the case then that part of the primary key would be left Null, raising the error which you are experiencing.

While my demo's form includes only the one subform, it could, like yours, equally well include others for inserting data into other tables of course.

So, if your relationships are similar, the first thing to be checked is that the subform's LinkChildFields property is the name of the relevant foreign key column, and that there is a control or controls in the subform for inserting data into any other column or columns which form part of the relevant table's primary key.

If you are unfamiliar with the LinkMasterFields and LinkChildFields properties of a subform control, as you mentioned in your reply to Duane, that is not surprising.  It is these properties which restrict each instance of the subform to the rows which relate to the parent form's current record.  The LinkMasterFields will normally be the name of the parent form's table's primary key column, while the LinkChildFields property will normally be the name of the corresponding foreign key column in the subform's table.  By linking the subform to the parent form on these columns the former's recordset is restricted to those rows where the foreign key column's value is that of the parent form's current record's primary key value.

In your case, therefore, the LinkMasterFields and LinkChildFields properties of the subform control are likely to be StudentID or similar.
